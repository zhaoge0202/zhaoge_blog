# 进程间通信有哪些方式？Java 后端常遇到哪些？

> IPC 的核心问题是：进程地址空间相互隔离，想交换数据就要借助内核或显式共享区域。

## 为什么线程能直接共享，进程不行？

同一个进程里的线程共享堆、方法区、打开文件等进程资源，所以一个线程写入堆对象，另一个线程能通过同一个对象引用看到变化。不同进程默认拥有独立虚拟地址空间，进程 A 的地址 `0x1000` 和进程 B 的地址 `0x1000` 通常不是同一块物理内存。

这种隔离提升了安全性和稳定性，也带来一个直接问题：两个进程不能像两个线程一样“传一个对象引用”就完成通信。进程间交换数据，要么经过内核中转，要么把同一块物理页映射到不同进程，要么把通信抽象成文件描述符、信号或 Socket。

## 常见 IPC 怎么选？

可以先按“数据怎么走”分成四类：

| 类型       | 代表方式                   | 适合场景                                   | 主要代价                       |
| ---------- | -------------------------- | ------------------------------------------ | ------------------------------ |
| 内核中转   | 匿名管道、FIFO、消息队列   | 简单命令串联、小消息传递、父子进程通信     | 容量有限，读写要经过内核       |
| 共享映射   | 共享内存、`mmap`           | 大块数据交换、低延迟队列、文件映射读写     | 只解决共享，不解决同步         |
| 控制与同步 | 信号、信号量、文件锁       | 优雅停机、互斥、通知某个事件已经发生       | 信息量少，业务协议要另行设计   |
| 通用连接   | Unix domain socket、Socket | 本机代理、Sidecar、RPC、数据库和缓存客户端 | 需要协议、序列化和连接生命周期 |

管道和消息队列像“把数据交给内核转发”；共享内存像“两个进程看同一块公告板”；信号像“敲门提醒”；Socket 则把本机和跨机器通信统一到一套接口上。

## 管道为什么简单，但容易卡住？

Shell 里的 `A | B` 就是匿名管道的典型场景。管道本质上是一段内核缓冲区：A 往写端写，B 从读端读。匿名管道没有文件系统里的名字，通常依赖父进程创建后再 `fork`，让父子或兄弟子进程继承同一组文件描述符。

命名管道 FIFO 则会在文件系统里出现一个管道文件，不要求两个进程有亲缘关系：

```bash
mkfifo /tmp/demo.pipe
echo "hello" > /tmp/demo.pipe
cat < /tmp/demo.pipe
```

管道适合流式小数据，不适合复杂协议。它有两个常见边界：

1. **单向为主**：要双向通信通常要建两条管道。
2. **缓冲有限**：写端持续写、读端不消费，写端会阻塞。

Java 里最常见的管道坑是 `ProcessBuilder`。应用启动外部命令后，如果不持续消费子进程的 stdout/stderr，子进程可能写满管道缓冲区后卡住，表现为“命令明明没报错，但 Java 一直等不到结束”。

排查时可以先看文件描述符和进程状态：

```bash
ls -l /proc/<pid>/fd
lsof -p <pid>
strace -f -e trace=read,write,pipe,wait4 -p <pid>
```

## Java 后端最常碰到哪些？

**第一，Socket。** RPC、HTTP、MySQL 协议、Redis 协议，本质都是进程间通信，只是很多时候跨机器。Unix domain socket 还可以用于同机高效通信，例如本机代理、Sidecar、数据库本地连接。

**第二，管道。** Java 进程调用外部命令时，`ProcessBuilder` 的标准输入、标准输出、标准错误流背后就和管道模型有关。标准错误流不要忽略，很多命令主要把错误和进度输出写到 stderr。

**第三，共享内存和 mmap。** Java 里 `MappedByteBuffer` 可以把文件映射到进程地址空间，常用于大文件读写、索引文件、队列存储。它不是普通堆对象，排查 RSS、文件刷盘和容器内存时要单独考虑。

**第四，信号。** `kill -15` 给进程发 `SIGTERM`，应用可以做优雅停机；`kill -9` 是 `SIGKILL`，进程不能捕获，清理逻辑也来不及执行。Kubernetes 终止 Pod 时也会先给容器主进程终止信号，再等待宽限期。

## 消息队列和共享内存差在哪？

消息队列把消息放在内核维护的队列里。发送方写入消息后可以返回，接收方按规则读取。它比管道更像“有边界的一条条消息”，但数据仍然要在用户态和内核态之间拷贝，而且消息大小、队列总长度都有限制。

共享内存把同一块物理内存映射到多个进程的虚拟地址空间。一个进程写入后，另一个进程可以直接从自己的虚拟地址读到，少了“用户态 -> 内核 -> 用户态”的数据搬运，所以适合大块数据和低延迟场景。

但共享内存只解决“都能看到同一块数据”，不解决“谁先写、谁后读、写到一半能不能读”。比如一个进程把订单写入共享内存，另一个进程读取；如果没有同步机制，读进程可能看到半条订单。工程上通常要配合信号量、文件锁、环形队列序号、内存屏障等机制。

这和多线程共享堆很像：共享数据越方便，越要认真处理一致性。

## 信号和信号量为什么不是一回事？

信号是异步通知，适合表达“发生了一个控制事件”。比如：

- `SIGTERM`：请求进程终止，应用可以捕获后做优雅停机。
- `SIGKILL`：强制终止，不能捕获、不能忽略。
- `SIGCHLD`：子进程状态变化，父进程可以据此回收子进程。
- `SIGPIPE`：向已经关闭读端的管道或 Socket 写数据时可能触发。

信号量是计数器，主要做同步和互斥，不负责传业务数据。计数为 1 时可以当互斥锁用；计数为 0 时可以让消费者等待生产者通知。

这两个名字相近，但语义完全不同。面试时可以这样区分：信号告诉你“有事发生了”，信号量约束“现在能不能进入临界区”。

## Socket 为什么也算 IPC？

Socket 不只是跨机器网络通信。Unix domain socket 使用本机文件路径或抽象命名空间通信，不经过 IP 路由，适合同机进程之间传输结构化请求。

典型场景包括：

- Nginx、Envoy、本机 Sidecar 和业务进程通信。
- 应用通过本地 socket 访问数据库或代理。
- 容器运行时、监控 Agent、日志采集进程暴露本机控制接口。

排查本机 socket 时可以看：

```bash
ss -xl
ss -xlp
lsof -U -p <pid>
```

如果是 TCP/UDP Socket，就要继续看监听端口、连接状态、收发队列和协议层问题，这部分会和网络专题衔接。

## 容易踩的坑

- 管道不是无限缓冲，子进程 stdout/stderr 不消费会导致阻塞。
- 消息队列适合小消息，不适合搬大块数据。
- 信号量不是“传消息”的队列，它主要做同步和互斥。
- 信号处理函数里不能随便做复杂逻辑，异步场景下要特别小心可重入问题。
- Socket 不只用于跨机器，本机进程也可以使用 Unix domain socket。
- `kill -9` 不等于优雅停机，线上优先用 `SIGTERM` 配合应用 shutdown hook。

## 小结

- IPC 的前提是进程地址空间隔离，通信需要内核中转或显式共享。
- 管道简单但能力有限，消息队列适合小消息，Socket 最通用。
- 共享内存性能高，但必须配合同步机制，否则会读到不一致数据。
- 信号是异步控制通知，信号量是同步和互斥工具，两者不是一类东西。
- Java 后端常见 IPC 落点包括 RPC、外部命令、mmap 文件、本机 socket 和进程信号。
- 排障时要关注文件描述符、管道消费、优雅停机信号和本机 socket。
