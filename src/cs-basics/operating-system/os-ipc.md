# 进程间通信有哪些方式？Java 后端常遇到哪些？

> IPC 的核心问题是：进程地址空间相互隔离，想交换数据就要借助内核或显式共享区域。

## 为什么线程能直接共享，进程不行？

同一个进程里的线程共享堆，所以一个线程写对象，另一个线程能看到。不同进程默认拥有独立虚拟地址空间，进程 A 的地址 `0x1000` 和进程 B 的地址 `0x1000` 通常不是同一块物理内存。

这种隔离提升了安全性和稳定性，但也让通信需要专门机制。

## 常见 IPC 怎么选？

| 方式          | 适合场景                       | 主要代价                      |
| ------------- | ------------------------------ | ----------------------------- | ------------------ |
| 匿名管道      | 父子进程单向流式通信，Shell `A | B`                            | 方向单一、容量有限 |
| 命名管道 FIFO | 无亲缘关系进程的简单通信       | 仍是流式，复杂协议不方便      |
| 消息队列      | 小消息异步传递                 | 有大小限制，用户态/内核态拷贝 |
| 共享内存      | 大量数据高速交换               | 必须额外做同步                |
| 信号量        | 进程间互斥与同步               | 不传输业务数据                |
| 信号          | 通知异常或控制事件             | 信息量小，异步处理要小心      |
| Socket        | 本机或跨机器进程通信           | 协议栈和序列化成本            |

管道和消息队列像“通过内核中转包裹”；共享内存像“两个进程映射同一块公告板”；信号像“发一个中断通知”；Socket 则是最通用的通信接口。

## Java 后端最常碰到哪些？

**第一，Socket。**
RPC、HTTP、MySQL 协议、Redis 协议，本质都是进程间通信，只是很多时候跨机器。Unix domain socket 还可以用于同机高效通信，例如本机代理、Sidecar、数据库本地连接。

**第二，管道。**
Java 进程调用外部命令时，`ProcessBuilder` 的标准输入、标准输出、标准错误流背后就和管道模型有关。如果子进程输出没人读，缓冲区写满后子进程可能卡住。

**第三，共享内存和 mmap。**
Java 里 `MappedByteBuffer` 可以把文件映射到内存，常用于大文件读写、索引文件、队列存储。它不是普通堆对象，排查 RSS 和文件刷盘时要单独考虑。

**第四，信号。**
`kill -15` 给进程发 `SIGTERM`，应用可以做优雅停机；`kill -9` 是 `SIGKILL`，进程不能捕获，清理逻辑也来不及执行。

## 共享内存为什么还需要锁？

共享内存只解决“两个进程都能看到同一块数据”，不解决“谁先写、谁后读、写到一半能不能读”的问题。

比如一个进程把订单写入共享内存，另一个进程读取。如果没有同步机制，读进程可能看到半条订单。工程上通常要配合信号量、文件锁、环形队列序号、内存屏障等机制。

这和多线程共享堆很像：共享数据越方便，越要认真处理一致性。

## 容易踩的坑

- 管道不是无限缓冲，子进程 stdout/stderr 不消费会导致阻塞。
- 消息队列适合小消息，不适合搬大块数据。
- 信号量不是“传消息”的队列，它主要做同步和互斥。
- Socket 不只用于跨机器，本机进程也可以使用 Unix domain socket。
- `kill -9` 不等于优雅停机，线上优先用 `SIGTERM` 配合应用 shutdown hook。

## 小结

- IPC 的前提是进程地址空间隔离，通信需要内核中转或显式共享。
- 管道简单但能力有限，消息队列适合小消息，Socket 最通用。
- 共享内存性能高，但必须配合同步机制。
- Java 后端常见 IPC 落点包括 RPC、外部命令、mmap 文件和进程信号。
- 排障时要关注管道阻塞、优雅停机信号和本地 socket。
