# 上下文切换为什么会影响性能？

> 上下文切换的成本不只是“换个线程”，还包括保存现场、恢复现场、调度决策以及 CPU 缓存命中下降。

## 什么是上下文？

CPU 执行一个线程时，需要知道当前指令执行到哪里、寄存器里有什么、栈在哪里。为了让线程被暂停后还能继续跑，操作系统必须保存这些运行现场。

上下文一般包含：

- CPU 寄存器和程序计数器。
- 线程栈位置和内核栈信息。
- 调度状态、优先级、等待原因。
- 进程切换时还涉及地址空间、页表和内存管理信息。

可以把它理解成打游戏存档：切出去前要存档，切回来要读档。存档和读档本身不产出业务结果，但会消耗 CPU。

更准确地说，CPU 上下文切换包括三个动作：

```text
保存当前任务的寄存器、程序计数器、栈指针等现场
  ↓
调度器选择下一个可运行任务
  ↓
恢复下一个任务的现场，并跳到它上次停下的位置继续执行
```

这里的“任务”可能是进程、线程，也可能是中断处理。它们都会打断当前 CPU 的连续执行，只是保存和恢复的内容不同。

## 哪些场景会触发切换？

常见触发点有几类：

1. 时间片用完，调度器让其他线程运行。
2. 线程发起阻塞 IO、等待锁、`sleep`、`park` 等操作。
3. 更高优先级任务被唤醒。
4. 中断到来，CPU 先处理中断上下文。

Java 服务里，请求线程阻塞数据库、等待 Redis、抢同一把锁，都可能把线程从运行态切到等待态，再由调度器切换别的线程。

从后端排障角度看，可以分成三类：

| 类型       | 典型触发                         | 常见表现                       |
| ---------- | -------------------------------- | ------------------------------ |
| 自愿切换   | 线程主动等待 IO、锁、sleep、park | `WAITING`/`TIMED_WAITING` 很多 |
| 非自愿切换 | 时间片用完、被更高优先级任务抢占 | 线程都在抢 CPU，延迟抖动       |
| 中断切换   | 网卡、磁盘、时钟等硬件中断       | `sys` 占比升高、软中断压力变大 |

自愿切换不一定坏，比如线程等待网络 IO 很正常；问题在于等待是否过多、是否集中在同一个下游、是否可以通过异步化、限流或超时治理。

## 为什么它会拖慢性能？

上下文切换主要有三类成本：

| 成本          | 说明                               | 后端表现             |
| ------------- | ---------------------------------- | -------------------- |
| 直接 CPU 成本 | 保存和恢复寄存器、栈、调度信息     | CPU sys 占比升高     |
| 缓存成本      | 新线程的数据不一定在 CPU cache 中  | 吞吐下降、延迟抖动   |
| 队列成本      | 就绪队列、等待队列、锁队列频繁变化 | 线程很多但有效工作少 |

进程切换通常比同进程线程切换更重，因为可能涉及地址空间和 TLB 影响。同进程线程切换少了一些资源切换，但仍然会打断 CPU 的连续执行。

这里要区分两种“切换”：

| 名称                  | 含义                           | 例子                       |
| --------------------- | ------------------------------ | -------------------------- |
| 用户态/内核态模式切换 | 同一个线程进入内核再返回用户态 | 系统调用、缺页、中断返回   |
| 调度上下文切换        | CPU 从一个任务切到另一个任务   | 线程阻塞、时间片用完、抢占 |

一次系统调用一定有用户态/内核态切换，但不一定发生调度上下文切换；只有线程阻塞、被抢占或内核决定运行别的任务时，才发生调度意义上的上下文切换。分析性能时要把这两个概念分开。

## 线程切换和进程切换差在哪？

同一进程内的线程切换通常主要保存/恢复线程私有现场：寄存器、栈指针、程序计数器、调度状态等。因为它们共享地址空间，通常不需要切换整套页表。

跨进程切换会更重一些，除了执行现场，还可能切换地址空间、页表和内存管理上下文，带来 TLB 刷新、缓存局部性下降等成本。

但不要把线程切换理解成“只保存几个寄存器所以很便宜”。如果线程数量远大于 CPU 核数，或者线程频繁在多个核心间迁移，缓存行、锁竞争、运行队列竞争也会把成本放大。

## 怎么观察上下文切换？

Linux 上可以先看系统级指标：

```bash
vmstat 1
pidstat -w -p <pid> 1
top -H -p <pid>
```

`vmstat` 里的 `cs` 表示每秒上下文切换次数；`pidstat -w` 可以看进程或线程的自愿、非自愿上下文切换。

还可以加几条辅助命令：

```bash
# 看每个 CPU 的中断和软中断
mpstat -P ALL 1

# 看进程线程数和每个线程 CPU
ps -eLf | grep <pid> | wc -l
top -H -p <pid>

# 看 Java 线程状态和栈顶
jcmd <pid> Thread.print > /tmp/thread.txt
```

Java 服务里，如果看到：

- 线程池开得很大，但 CPU 利用率不高。
- `jstack` 里大量线程处于 `WAITING`、`TIMED_WAITING`、`BLOCKED`。
- `pidstat -w` 的切换次数明显偏高。

就要怀疑线程模型、锁竞争或阻塞调用设计有问题。

## 怎么从指标走到根因？

可以按这条链路排查：

```text
vmstat 发现 cs 很高
  ↓
pidstat -w 定位是哪个进程/线程切换多
  ↓
top -H 看线程 CPU 和状态
  ↓
jstack / jcmd Thread.print 看线程卡在哪里
  ↓
结合业务日志、慢 SQL、RPC 超时、锁竞争指标定位根因
```

常见根因和处理方向：

| 现象                   | 常见根因                       | 处理方向                         |
| ---------------------- | ------------------------------ | -------------------------------- |
| 大量 `BLOCKED`         | 热点锁、锁粒度太大             | 缩小锁粒度、减少共享状态         |
| 大量等待连接池         | 下游慢、连接池太小或泄漏       | 超时、限流、扩池前先查下游       |
| 线程数很多但 CPU 不高  | 阻塞 IO 或等待队列堆积         | 控制线程池、异步化、设置超时     |
| CPU 高且非自愿切换多   | 可运行线程远多于 CPU 核数      | 降低线程数、隔离 CPU 密集任务    |
| `sys` 占比高、软中断高 | 网络包、磁盘 IO 或内核态开销高 | 看网卡、磁盘、系统调用和中断分布 |

不要只看 `cs` 的绝对值。高 QPS 网络服务上下文切换本来就会比空闲服务多，更重要的是它和延迟、CPU 利用率、线程状态、吞吐变化是否同时异常。

## 怎么减少不必要的切换？

思路不是“完全避免切换”，而是让线程数量和阻塞点可控：

- IO 密集型线程池根据下游延迟和 QPS 估算，不要拍脑袋开几千个。
- CPU 密集型任务线程数接近 CPU 核数，避免大量线程抢 CPU。
- 缩小锁粒度，减少热点锁；能用无锁队列、分段锁、局部变量就少共享。
- 网络服务优先使用 epoll/Reactor 这类事件模型，避免一个连接一个线程。
- 慢查询、慢 RPC 要设置超时，否则线程会长期挂在等待队列里。
- 公平锁、频繁唤醒、过小批量处理都可能增加切换，要结合吞吐和延迟目标取舍。

Java 里还要注意：把所有阻塞调用都丢到一个巨大线程池，不等于解决问题。它可能只是把下游慢、锁竞争或容量不足转化成更多线程、更多切换和更大的内存占用。

## 小结

- 上下文切换需要保存旧线程现场、恢复新线程现场，本身不产生业务价值。
- 切换会带来 CPU 直接成本，也会降低缓存和 TLB 的利用效率。
- Java 服务中线程过多、锁竞争、阻塞 IO 都会放大切换成本。
- `vmstat`、`pidstat -w`、`top -H`、`jstack` 可以联合观察，并要区分自愿/非自愿切换。
- 优化方向是控制线程数、减少阻塞、降低共享竞争。
