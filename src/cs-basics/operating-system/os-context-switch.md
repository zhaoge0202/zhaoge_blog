# 上下文切换为什么会影响性能？

> 上下文切换的成本不只是“换个线程”，还包括保存现场、恢复现场、调度决策以及 CPU 缓存命中下降。

## 什么是上下文？

CPU 执行一个线程时，需要知道当前指令执行到哪里、寄存器里有什么、栈在哪里。为了让线程被暂停后还能继续跑，操作系统必须保存这些运行现场。

上下文一般包含：

- CPU 寄存器和程序计数器。
- 线程栈位置和内核栈信息。
- 调度状态、优先级、等待原因。
- 进程切换时还涉及地址空间、页表和内存管理信息。

可以把它理解成打游戏存档：切出去前要存档，切回来要读档。存档和读档本身不产出业务结果，但会消耗 CPU。

## 哪些场景会触发切换？

常见触发点有几类：

1. 时间片用完，调度器让其他线程运行。
2. 线程发起阻塞 IO、等待锁、`sleep`、`park` 等操作。
3. 更高优先级任务被唤醒。
4. 中断到来，CPU 先处理中断上下文。

Java 服务里，请求线程阻塞数据库、等待 Redis、抢同一把锁，都可能把线程从运行态切到等待态，再由调度器切换别的线程。

## 为什么它会拖慢性能？

上下文切换主要有三类成本：

| 成本          | 说明                               | 后端表现             |
| ------------- | ---------------------------------- | -------------------- |
| 直接 CPU 成本 | 保存和恢复寄存器、栈、调度信息     | CPU sys 占比升高     |
| 缓存成本      | 新线程的数据不一定在 CPU cache 中  | 吞吐下降、延迟抖动   |
| 队列成本      | 就绪队列、等待队列、锁队列频繁变化 | 线程很多但有效工作少 |

进程切换通常比同进程线程切换更重，因为可能涉及地址空间和 TLB 影响。同进程线程切换少了一些资源切换，但仍然会打断 CPU 的连续执行。

## 怎么观察上下文切换？

Linux 上可以先看系统级指标：

```bash
vmstat 1
pidstat -w -p <pid> 1
top -H -p <pid>
```

`vmstat` 里的 `cs` 表示每秒上下文切换次数；`pidstat -w` 可以看进程或线程的自愿、非自愿上下文切换。

Java 服务里，如果看到：

- 线程池开得很大，但 CPU 利用率不高。
- `jstack` 里大量线程处于 `WAITING`、`TIMED_WAITING`、`BLOCKED`。
- `pidstat -w` 的切换次数明显偏高。

就要怀疑线程模型、锁竞争或阻塞调用设计有问题。

## 怎么减少不必要的切换？

思路不是“完全避免切换”，而是让线程数量和阻塞点可控：

- IO 密集型线程池根据下游延迟和 QPS 估算，不要拍脑袋开几千个。
- CPU 密集型任务线程数接近 CPU 核数，避免大量线程抢 CPU。
- 缩小锁粒度，减少热点锁；能用无锁队列、分段锁、局部变量就少共享。
- 网络服务优先使用 epoll/Reactor 这类事件模型，避免一个连接一个线程。
- 慢查询、慢 RPC 要设置超时，否则线程会长期挂在等待队列里。

## 小结

- 上下文切换需要保存旧线程现场、恢复新线程现场，本身不产生业务价值。
- 切换会带来 CPU 直接成本，也会降低缓存和 TLB 的利用效率。
- Java 服务中线程过多、锁竞争、阻塞 IO 都会放大切换成本。
- `vmstat`、`pidstat -w`、`top -H`、`jstack` 可以联合观察。
- 优化方向是控制线程数、减少阻塞、降低共享竞争。
