# 线程数量能无限增加吗？

> 线程不是只有一个对象那么简单，它要消耗栈空间、内核任务结构、PID 编号和调度能力。

## 创建一个线程要消耗什么？

在 Java 里写 `new Thread(...).start()` 很轻松，但平台线程背后通常对应一个操作系统线程。它至少需要：

- 线程栈，用来保存方法调用、局部变量和返回地址。
- 内核里的任务结构，用来参与调度。
- 线程 ID 或 PID 编号。
- 若干虚拟内存区域，用于栈、保护页和运行时数据。

所以线程数上限不是一个固定公式，而是多个约束一起决定。

可以把线程创建成本拆成两类：

| 成本类型 | 说明                                 | 常见后果                     |
| -------- | ------------------------------------ | ---------------------------- |
| 内存成本 | 栈、保护页、内核任务结构、TLS 等     | RSS 上涨、本地内存不足       |
| 调度成本 | 进入运行队列、阻塞队列、锁队列、唤醒 | 上下文切换增多，P99 延迟抖动 |

所以线程数不是“能创建就合理”。创建成功只是第一步，调度得动、响应稳定、下游扛得住才算合理。

## 线程栈为什么关键？

每个线程都有独立栈。Linux 上可以用下面命令看默认栈大小：

```bash
ulimit -s
```

Java 里可以通过 `-Xss` 调整单个线程栈大小。假设每个线程栈 1MB，创建 1000 个线程，光栈的虚拟地址空间就可能预留约 1GB。注意这里说的是虚拟地址空间，不一定马上占用同等物理内存。

栈调小可以让线程数上限变高，但不是越小越好。递归深、调用链长、JNI 调用复杂时，栈太小会触发 `StackOverflowError`。

线程栈有两个容易混淆的点：

1. 虚拟地址空间预留不等于立刻占用同等物理内存，栈页通常是按需提交。
2. 容器里看的是进程整体 RSS 和 cgroup 限制，不是只看 Java 堆。

比如容器限制 2G，如果 `-Xmx` 已经给到 1600M，再创建大量平台线程，线程栈、直接内存、元空间和 native 开销都可能把进程推到 OOMKilled。

## 还会受哪些 Linux 参数限制？

常见限制包括：

```bash
ulimit -u
cat /proc/sys/kernel/threads-max
cat /proc/sys/kernel/pid_max
cat /proc/sys/vm/max_map_count
cat /proc/<pid>/limits
```

含义可以这样理解：

| 参数            | 影响                            |
| --------------- | ------------------------------- |
| `ulimit -u`     | 当前用户最多能创建多少进程/线程 |
| `threads-max`   | 系统级最大线程数量              |
| `pid_max`       | 进程和线程 ID 的编号上限        |
| `max_map_count` | 单进程最多拥有多少虚拟内存区域  |

有的资料会用“32 位系统 3GB 用户空间 / 栈大小”估算线程数，这个估算只适合说明虚拟地址空间限制。64 位系统虚拟地址空间大很多，实际更常被内核参数、物理内存、调度开销和业务延迟限制。

如果跑在容器里，还要看 cgroup 限制：

```bash
cat /sys/fs/cgroup/pids.max 2>/dev/null
cat /sys/fs/cgroup/memory.max 2>/dev/null
cat /sys/fs/cgroup/memory.current 2>/dev/null
```

不同系统的 cgroup 路径可能不同，排查时以当前容器运行时挂载为准。核心思路是：线程既受 PID/任务数量限制，也受容器内存限制。

## Java 服务怎么判断线程过多？

典型信号有：

- 报错 `unable to create new native thread`。
- `top -H` 看到同一个进程下线程数非常多。
- `jstack` 中大量线程卡在同一类等待，例如连接池、锁、队列。
- CPU 不一定很高，但上下文切换次数高，响应时间抖动。

排查时先看线程来源：

```bash
jcmd <pid> Thread.print
jstack <pid>
ps -eLf | grep <pid> | wc -l
```

如果大量线程来自业务线程池，优先检查线程池是否无界、队列是否无界、下游超时是否缺失。如果来自 Netty 或调度任务，要看是否重复初始化了客户端或定时器。

也可以从线程名和栈顶归类：

| 线程来源        | 典型线索                      | 常见根因                       |
| --------------- | ----------------------------- | ------------------------------ |
| HTTP/RPC 线程池 | `http-nio-*`、`Dubbo*` 等     | 下游慢、队列堆积、超时缺失     |
| 业务异步线程池  | 自定义线程名前缀              | 无界线程池、任务生产速度过快   |
| 定时任务        | scheduler、timer              | 重复创建调度器、任务执行超时   |
| Netty EventLoop | `nioEventLoopGroup`、`epoll*` | 客户端重复初始化、IO 线程阻塞  |
| JVM 内部线程    | GC、JIT、Reference Handler    | 通常不是数量失控主因，先别误杀 |

排查顺序建议是：先看线程总数，再按线程名分类，再看栈顶阻塞点，最后回到线程池配置和下游耗时。

## 线程数该怎么设？

没有万能数字，但有基本方向：

- CPU 密集型：接近 CPU 核数，或略高一点。
- IO 密集型：结合下游平均耗时、目标 QPS 和可接受排队时间估算。
- Netty EventLoop：通常按 CPU 核数配置，不按连接数配置。
- 定时任务和异步任务：必须有边界，拒绝策略要能暴露问题。

不要靠“调大线程数”掩盖慢 SQL、慢 RPC 或锁竞争。线程数加大后，短期可能吞吐上升，长期常见结果是内存、上下文切换和下游压力一起失控。

一个简单估算思路是：

```text
需要并发数 ≈ 目标 QPS × 平均响应时间
```

比如目标 1000 QPS，下游平均耗时 100ms，只看等待并发就大约需要 100 个正在处理的请求槽位。实际线程池大小还要结合 CPU 时间、排队时间、超时、连接池大小和下游容量，不能只按这个公式机械设置。

CPU 密集型任务更应该接近 CPU 核数；IO 等待型任务可以多一些，但上限应该来自压测和监控，而不是“机器还没报错”。

## 虚拟线程能不能解决线程上限？

Java 虚拟线程能显著降低大量等待型任务的承载成本，因为虚拟线程不会长期独占 OS 线程。它适合很多“请求大部分时间在等 IO”的场景。

但它不是无限线程按钮：

- CPU 密集计算仍然受 CPU 核数限制。
- 下游数据库、Redis、HTTP 服务仍然有容量限制。
- JDK 21-23 中，虚拟线程在 `synchronized` 临界区里阻塞可能占住承载线程；JDK 24+ 已大幅改善这类 monitor pinning。
- native 调用、类加载初始化等少数无法挂起的阻塞点仍可能占住承载线程。
- 每个虚拟线程仍有对象、栈片段和调度管理开销。

所以虚拟线程能减少“平台线程太多”的问题，但不能替代限流、超时、连接池和下游容量治理。

## 事故里怎么止血？

看到线程暴涨时，先保服务，再修根因：

1. 先摘流、限流或降级，降低新任务进入速度。
2. 保存 `jstack`、线程数、`/proc/<pid>/limits`、容器 pids/memory 限制。
3. 如果是下游慢导致堆积，优先恢复下游或熔断依赖。
4. 如果是线程池无界，短期限制提交速率，长期改有界队列和拒绝策略。
5. 如果是重复初始化客户端或调度器，修代码生命周期，避免每次请求创建线程资源。

不要一上来只调大 `ulimit -u` 或 `-Xss`。这些动作可能让服务多撑一会儿，但根因如果是任务无限堆积，下次故障会更重。

## 小结

- 线程数不能无限增加，会受栈空间、内核结构、PID、VMA 和调度成本限制。
- `-Xss` 越大，单进程可承载的平台线程通常越少。
- `unable to create new native thread` 往往不是堆 OOM，而是本地线程资源、PID 或容器限制耗尽。
- 排查要结合 `jstack`、`ps -eLf`、`ulimit`、`/proc/<pid>/limits` 和 cgroup 限制。
- 线程池大小要服务于吞吐和延迟目标，而不是越大越好。
- 虚拟线程降低等待型任务的线程成本，但不能绕过 CPU 和下游容量限制。
