# 虚拟内存解决了什么问题？

> 虚拟内存给每个进程一套“看起来独占”的地址空间，再由操作系统和 MMU 映射到真实物理内存。

## 如果没有虚拟内存会怎样？

假设两个程序都直接写物理地址 `0x1000`。程序 A 写订单缓存，程序 B 写日志缓冲，它们可能直接互相覆盖。程序一多，内存保护、地址分配和程序装载都会变得混乱。

虚拟内存解决了三件事：

1. **隔离**：每个进程拥有独立虚拟地址空间。
2. **保护**：用户态不能随意访问内核空间，进程也不能随意读写别的进程。
3. **按需使用**：虚拟地址可以先分配，真正访问时再映射物理页。

有些资料会用“64 位 Linux 用户空间 128TB”来说明地址空间很大，这个说法要带边界：它常见于 x86-64 四级页表、48 位规范地址的布局。换成五级页表、不同 CPU 架构或不同内核配置，用户空间范围可能变化。

## 虚拟地址怎么变成物理地址？

CPU 访问的是虚拟地址，MMU 会根据页表把虚拟页号翻译成物理页号，再加上页内偏移得到物理地址。

```mermaid
flowchart LR
 A[虚拟地址] --> B[页号 + 页内偏移]
 B --> C[查询 TLB/页表]
 C --> D[物理页号 + 页内偏移]
 D --> E[访问物理内存]
```

页表属于进程上下文的一部分。切换进程时，地址空间也会跟着切换，这也是进程上下文切换比线程切换更重的原因之一。

## 分段和分页有什么区别？

分段按逻辑切：代码段、数据段、堆、栈。它符合程序员理解，但段大小不固定，容易产生外部碎片。

分页按固定大小切：常见页大小是 4KB。虚拟内存和物理内存都切成页，通过页表映射。分页减少了外部碎片，也让按需加载、换入换出更容易。

分页也有代价：页表会占内存，地址翻译会耗时。因此现代系统使用多级页表节省页表空间，并用 TLB 缓存常用翻译结果。

## 缺页异常是不是错误？

不一定。缺页异常只是 CPU 访问的虚拟页当前没有映射到物理内存，内核要介入处理。

常见情况：

- 第一次访问 `malloc` 得到的内存，内核才分配物理页。
- 访问 mmap 文件页，内核从文件加载到内存。
- 物理内存紧张时，被换出的页需要从磁盘换回。

前两类是正常机制；如果频繁 swap，延迟会明显上升，服务会表现为“CPU 不高但很慢”。

还要区分 minor fault 和 major fault：前者通常不需要真实磁盘 IO，比如页已经在内存里只是页表还没建立；后者需要从磁盘读取，成本高很多。非法地址访问也会触发缺页路径，但最终可能变成 `SIGSEGV`，它和正常按需调页不是一回事。

## 和 JVM、容器有什么关系？

Java 进程看到的堆、线程栈、直接内存、元空间都处在进程虚拟地址空间里。`-Xmx` 限制的是 Java 堆，不等于进程 RSS 上限。

容器里更容易踩坑：进程的虚拟地址空间可能很大，但 cgroup 限制了可用物理内存。直接内存、线程栈、Page Cache 都可能让 RSS 接近容器限制，最终被 OOM Killer 杀掉。

Redis 做 RDB 或 AOF rewrite 时会 `fork` 子进程，父子进程一开始通过写时复制共享物理页。父进程继续写入越多，被复制的页越多，内存峰值越高。这是虚拟内存和页表机制在后端组件里的典型落点。

排查时不要只看 heap dump，还要看：

```bash
cat /proc/<pid>/status
cat /proc/<pid>/smaps_rollup
jcmd <pid> VM.native_memory summary
```

## 小结

- 虚拟内存让每个进程拥有独立地址空间，提升隔离、保护和装载灵活性。
- MMU 通过页表把虚拟地址翻译成物理地址，TLB 用来加速翻译。
- 分页减少外部碎片，也支持按需加载和页面换入换出。
- 缺页异常不一定是错误，要区分正常按需调页、major fault 和非法访问。
- JVM 排查内存时要区分虚拟内存、RSS、Java 堆和本地内存。
