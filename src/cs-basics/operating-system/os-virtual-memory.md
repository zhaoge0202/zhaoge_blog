# 虚拟内存解决了什么问题？

> 虚拟内存给每个进程一套“看起来独占”的地址空间，再由操作系统和 MMU 映射到真实物理内存。

## 如果没有虚拟内存会怎样？

假设两个程序都直接写物理地址 `0x1000`。程序 A 写订单缓存，程序 B 写日志缓冲，它们可能直接互相覆盖。程序一多，内存保护、地址分配和程序装载都会变得混乱。

虚拟内存解决了三件事：

1. **隔离**：每个进程拥有独立虚拟地址空间。
2. **保护**：用户态不能随意访问内核空间，进程也不能随意读写别的进程。
3. **按需使用**：虚拟地址可以先分配，真正访问时再映射物理页。

有些资料会用“64 位 Linux 用户空间 128TB”来说明地址空间很大，这个说法要带边界：它常见于 x86-64 四级页表、48 位规范地址的布局。换成五级页表、不同 CPU 架构或不同内核配置，用户空间范围可能变化。

## 虚拟地址怎么变成物理地址？

CPU 访问的是虚拟地址，MMU 会根据页表把虚拟页号翻译成物理页号，再加上页内偏移得到物理地址。

```mermaid
flowchart LR
 A[虚拟地址] --> B[页号 + 页内偏移]
 B --> C[查询 TLB/页表]
 C --> D[物理页号 + 页内偏移]
 D --> E[访问物理内存]
```

页表属于进程上下文的一部分。切换进程时，地址空间也会跟着切换，这也是进程上下文切换比线程切换更重的原因之一。

页表项不只保存“物理页号”。它还会带一些控制位，比如：

| 字段             | 作用                                       |
| ---------------- | ------------------------------------------ |
| present          | 这个虚拟页当前是否已经映射到物理内存       |
| read/write       | 控制页面是否可写                           |
| user/supervisor  | 区分用户态能不能访问                       |
| accessed / dirty | 标记最近是否访问、是否被修改，辅助页面回收 |
| execute disable  | 禁止把数据页当代码执行，增强内存保护       |

所以页表不只是地址翻译表，也是内存隔离、权限保护、缺页处理和页面回收的基础。

## 单级页表为什么不够用？

如果只用一张单级页表，空间开销会很快膨胀。

以 32 位地址空间、4KB 页为例，一个进程有 4GB 虚拟地址空间，页大小是 4KB，就需要约 `2^20` 个页表项。如果一个页表项 4 字节，那么每个进程光页表就要约 4MB。100 个进程就是约 400MB，还没算业务数据。

更关键的是，单级页表往往要覆盖整个虚拟地址空间。哪怕进程实际只用了很小一段地址，也要给大量没用到的地址留页表项。

多级页表解决的是这个空间问题：上级页表覆盖大范围地址，下级页表按需创建。某段虚拟地址从来没用过，对应的下级页表就不用分配。

它不是为了让查询更快。多级页表反而会让一次地址翻译多查几层内存，速度问题主要靠 TLB 缓解。

## TLB 为什么重要？

TLB 可以理解成 CPU 里的“页表项缓存”。程序访问内存通常有局部性，短时间内会反复访问一批热点页。MMU 先查 TLB，命中就能直接得到物理页号；未命中才去走多级页表。

```mermaid
flowchart LR
 A[CPU 访问虚拟地址] --> B{TLB 命中?}
 B -- 是 --> C[直接得到物理页号]
 B -- 否 --> D[查询多级页表]
 D --> E[更新 TLB]
 E --> C
 C --> F[访问物理内存]
```

这也是进程切换比同进程线程切换更重的原因之一：切换地址空间可能导致 TLB 里旧进程的地址翻译失效，后续访问要重新建立热点。

## 分段和分页有什么区别？

分段按逻辑切：代码段、数据段、堆、栈。它符合程序员理解，但段大小不固定，容易产生外部碎片。

分页按固定大小切：常见页大小是 4KB。虚拟内存和物理内存都切成页，通过页表映射。分页减少了外部碎片，也让按需加载、换入换出更容易。

分页也有代价：页表会占内存，地址翻译会耗时。因此现代系统使用多级页表节省页表空间，并用 TLB 缓存常用翻译结果。

分段和分页还可以组合成段页式：先按逻辑段管理，再把段切成页。现代 Linux 主要依赖分页管理内存；在 x86 历史包袱下仍能看到分段概念，但常见实现会把段基址弱化处理，让分页承担主要的地址隔离和映射工作。

## 缺页异常是不是错误？

不一定。缺页异常只是 CPU 访问的虚拟页当前没有映射到物理内存，内核要介入处理。

常见情况：

- 第一次访问 `malloc` 得到的内存，内核才分配物理页。
- 访问 mmap 文件页，内核从文件加载到内存。
- 物理内存紧张时，被换出的页需要从磁盘换回。

前两类是正常机制；如果频繁 swap，延迟会明显上升，服务会表现为“CPU 不高但很慢”。

还要区分 minor fault 和 major fault：前者通常不需要真实磁盘 IO，比如页已经在内存里只是页表还没建立；后者需要从磁盘读取，成本高很多。非法地址访问也会触发缺页路径，但最终可能变成 `SIGSEGV`，它和正常按需调页不是一回事。

如果空闲物理页不够，内核会尝试回收内存：

| 页面类型   | 回收方式                                           | 代价                         |
| ---------- | -------------------------------------------------- | ---------------------------- |
| 干净文件页 | 直接丢弃，需要时再从文件读入                       | 代价相对低                   |
| 脏文件页   | 先写回磁盘，再释放                                 | 产生磁盘 IO                  |
| 匿名页     | 堆、栈这类无文件载体的页面，通常依赖 swap 才能换出 | 换入换出成本高，延迟抖动明显 |

所以“缺页”本身不一定严重，严重的是主缺页、频繁 swap、直接内存回收和页面抖动。

可以用这些命令观察：

```bash
vmstat 1
sar -B 1
cat /proc/<pid>/stat
cat /proc/<pid>/smaps_rollup
```

其中 `vmstat` 的 `si/so` 能看 swap in/out，`sar -B` 可以观察缺页和页面扫描趋势。

## 页面置换算法怎么理解？

内存不够时，系统要决定“换谁出去”。教材里常见几类算法：

| 算法  | 核心思路               | 优点               | 问题                       |
| ----- | ---------------------- | ------------------ | -------------------------- |
| OPT   | 换出未来最久不用的页   | 理论最优           | 需要预知未来，无法真实落地 |
| FIFO  | 最早进入内存的页先出去 | 实现简单           | 可能误伤热点页             |
| LRU   | 最久没访问的页先出去   | 符合局部性         | 精确实现成本高             |
| CLOCK | 用访问位近似 LRU       | 成本较低，工程可用 | 只是近似，极端情况会退化   |

面试里常问的 Belady 异常，就是 FIFO 这类策略可能出现“页帧变多，缺页次数反而更多”。OPT 和 LRU 有栈性质，不会出现这种异常；CLOCK 是 LRU 的近似，不能简单说它一定等同于 LRU。

真实 Linux 不会直接照搬某个教科书算法，而是用活跃/非活跃 LRU、workingset、refault、cgroup、NUMA 等机制综合判断。面试不用把内核细节背得很深，但要避免说“Linux 就是某个简单算法”。

## 和 JVM、容器有什么关系？

Java 进程看到的堆、线程栈、直接内存、元空间都处在进程虚拟地址空间里。`-Xmx` 限制的是 Java 堆，不等于进程 RSS 上限。

容器里更容易踩坑：进程的虚拟地址空间可能很大，但 cgroup 限制了可用物理内存。直接内存、线程栈、Page Cache 都可能让 RSS 接近容器限制，最终被 OOM Killer 杀掉。

Redis 做 RDB 或 AOF rewrite 时会 `fork` 子进程，父子进程一开始通过写时复制共享物理页。父进程继续写入越多，被复制的页越多，内存峰值越高。这是虚拟内存和页表机制在后端组件里的典型落点。

`mmap` 也建立在虚拟内存之上。建立文件映射不代表文件内容已经全部读进内存，通常是访问到具体页面时才触发缺页并加载。这也是很多大文件读取、索引文件、队列存储会用 `mmap` 的原因：地址空间先建好，物理页按需进入。

大堆 JVM 还会遇到 TLB 压力。HugePage 用更大的页减少页表项数量，可能降低 TLB miss，但它不是 GC 调优万能药，还要结合堆大小、GC 类型、容器限制和操作系统配置一起看。

排查时不要只看 heap dump，还要看：

```bash
cat /proc/<pid>/status
cat /proc/<pid>/smaps_rollup
jcmd <pid> VM.native_memory summary
```

## 小结

- 虚拟内存让每个进程拥有独立地址空间，提升隔离、保护和装载灵活性。
- MMU 通过页表把虚拟地址翻译成物理地址，页表项还承担权限保护和回收标记。
- 多级页表主要节省页表空间，TLB 用来缓解多级查表带来的时间开销。
- 缺页异常不一定是错误，要区分正常按需调页、major fault、swap 和非法访问。
- 页面置换要理解局部性和回收代价，真实 Linux 是复杂近似机制，不是单一教材算法。
- JVM 排查内存时要区分虚拟内存、RSS、Java 堆、本地内存和 Page Cache。
