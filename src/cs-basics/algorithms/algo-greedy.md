# 贪心什么时候能用？

> 贪心每一步只盯「当前看起来最好」的选择；难的不是写循环，而是证明这步不会把全局最优毁掉。

## 贪心在做什么？

贪心把问题拆成一串决策：在每一步，根据某个固定规则选出局部最优，选完就不再反悔。代码往往很短——排序 + 一遍扫描——但面试官真正想听的是：

1. **你贪的是什么**（结束最早？覆盖最远？差值最小？）；
2. **为什么这个选择不会让后面变差**。

如果只能说出「每次选最优」，却举不出理由或反例检验，说明策略还没站稳。

## 正确性怎么讲：交换与反证

正式证明不必写得很数学，但要有这两种直觉之一。

**交换论证**：假设存在一个最优解，它在某一步的选择和贪心不同。把那一步改成贪心的选择，答案不会变差，于是可以把任意最优解「改造」成贪心解，从而证明贪心也最优。

**反证 / 边界直觉**：如果某步不按贪心选，会严格损失后续空间或覆盖范围，因此任何更优解都不可能绕开这个选择。

以活动选择为例：若最优解里第一个活动不是「结束最早」的那个，用结束更早的活动替换它，其余活动仍可接上（因为结束更早只会留出更多空隙），最优性不降。反复替换，就得到贪心解。

证不出来时，先用小样例找反例。能找到反例，策略就该丢掉；找不到也不等于正确，但至少排除了明显错误。

## 什么时候值得一试？

出现下面信号，可以优先考虑贪心：

- **排序后决策变得清晰**。区间、分配、调度类题，先按一端排序，再线性扫。
- **只需维护一个「当前最优边界」**。例如最远可达位置、当前箭的坐标、当前区间右端点。
- **子问题之间没有复杂耦合**。当前决策一旦确定，后续只依赖一个标量状态，而不是一整张 DP 表。

常见题型对照：

| 题型        | 贪什么            | 典型题                       |
| ----------- | ----------------- | ---------------------------- |
| 区间调度    | 结束最早 / 右端点 | 活动选择、无重叠区间、射气球 |
| 覆盖 / 跳跃 | 当前能到的最远    | 跳跃游戏、跳跃游戏 II        |
| 分配        | 先满足最易满足者  | 分发饼干、分发糖果           |
| 重构 / 划分 | 维护覆盖区间      | 划分字母区间                 |

## 代表题 1：活动选择 / 区间调度

题意：有一堆会议，每个有开始、结束时间，同一时刻只能开一场，求最多能安排多少场。等价问题：无重叠区间最多留几个、至少删几个。

**策略：按结束时间升序排序，能选就选。**

```text
function maxActivities(intervals):
    if intervals 为空: return 0
    按 end 升序排序
    count = 1
    end = intervals[0].end
    for i = 1 .. n-1:
        if intervals[i].start >= end:   // 不重叠（边界按题意取 > 或 >=）
            count++
            end = intervals[i].end
    return count
```

为什么按结束时间而不是按时长或开始时间？

- 结束越早，留给后面的时间窗越大。
- 「每次选最短」看起来合理，反例：短区间卡在两个长区间中间，会挤掉本可兼容的两场。
- 「按开始时间」也会丢最优：一个很早开始却拖到很晚结束的活动，会堵死后面一串。

变体写法：

| 目标             | 排序键     | 扫描动作                   |
| ---------------- | ---------- | -------------------------- |
| 最多不重叠区间   | 右端点升序 | 能接就接，更新 end         |
| 最少删除使不重叠 | 同上       | 答案 = n − 最多保留        |
| 最少箭射爆气球   | 右端点升序 | 左端点超出当前箭则新开一箭 |
| 合并区间         | 左端点升序 | 右端取 max 做合并          |

边界是否算重叠要看题：`[1,2]` 与 `[2,3]` 在「点接触算不算冲突」上，不同题条件是 `>=` 还是 `>`。

## 代表题 2：跳跃游戏

题意：数组 `nums[i]` 表示在下标 i 最多能向前跳几步，问能否从 0 跳到最后一格。

**策略：维护「当前最远可达位置 farthest」，一遍扫过去。**

```text
function canJump(nums):
    farthest = 0
    for i = 0 .. n-1:
        if i > farthest:
            return false          // 走到了不可达格
        farthest = max(farthest, i + nums[i])
        if farthest >= n - 1:
            return true
    return true
```

贪心点在于：**不关心具体从哪一格跳到 i，只关心 i 是否落在已覆盖范围内**。只要 i 可达，就可以用 `i + nums[i]` 把覆盖右边界往外推。任何能到终点的路径，其每一步落点都不会超出某个时刻的 farthest；若 greedy 判定不可达，说明存在断层，真实路径也过不去。

扩展「最少跳跃次数」时，多维护一个 `curEnd`（当前步数覆盖的右边界）：扫到 `curEnd` 时步数 +1，并把 `curEnd` 更新为这段里推出来的 farthest。

## 复杂度

贪心题的时间几乎总是：

- **排序 O(n log n)** + **线性扫描 O(n)** → 整体 **O(n log n)**；
- 若不需要排序（如跳跃游戏只扫一遍）→ **O(n)**；
- 额外空间通常 **O(1)**（原地扫），若排序实现需要则看语言与是否原地。

面试时先说「瓶颈在排序」，再补「扫一遍维护边界」。

## 反例：什么时候贪心会翻车？

**普通硬币找零。**  
面额若是 `1, 5, 10, 25`，贪心「每次用最大面额」正确；若面额是 `1, 3, 4`，凑 `6`：贪心取 `4+1+1` 共 3 枚，最优是 `3+3` 共 2 枚。结论：**面额不具备「规范」结构时，找零必须 DP**。

**0-1 背包。**  
按「性价比」从高到低装，对分数背包（可拆）是对的，对 0-1（不可拆）会错：高性价比的小件可能挤掉总体更优的大件组合。

**局部最优 ≠ 全局最优的通用信号：**

- 当前选择会**永久排除**某些后续组合，且排除掉的可能整体更好；
- 需要在多个历史状态里做权衡，而不是只盯一个边界标量。

这时应改 DP 或搜索。

## 和 DP 的边界

| 维度 | 贪心                  | 动态规划                |
| ---- | --------------------- | ----------------------- |
| 决策 | 当前一步定死，不反悔  | 枚举多种前驱状态再取优  |
| 状态 | 通常一个边界 / 计数器 | 一维或二维表            |
| 证明 | 交换 / 反证           | 最优子结构 + 重叠子问题 |
| 适用 | 区间、跳跃、特殊分配  | 背包、子序列、路径计数  |

判断口诀：

1. 先试着说清贪心策略和正确性直觉；
2. 用 2～3 个小样例（含极端）砸一下；
3. 砸出反例 → 转 DP；砸不穿且能讲清交换论证 → 贪心可写。

同一题族也可能分叉：买卖股票「无限次交易」可贪心累加所有上涨段；「最多两次」就要 DP 状态机。

## 工程上的贪心

线上问题经常不要求严格最优，只要「足够好且快」：

- 负载调度里优先填最空的机器；
- 缓存淘汰近似 LRU 的各种变体；
- 压缩、编码里的 Huffman 本身就是可证明的贪心。

这时要在方案里写明：**贪心是近似 / 启发式**，并给出与最优的差距上界（若有）或线上指标对比。面试若被问「为什么不用最优算法」，答「规模下 DP 不可承受，贪心有可接受的近似比 / 实测收益」比空谈「贪心更快」更有说服力。

## 小结

1. 贪心 = 局部最优决策序列；开口先说「贪什么」和「为何不毁全局」。
2. 正确性靠交换论证或反证直觉；说不清就用小样例找反例。
3. 区间题按右端点、跳跃题维护 farthest，是最高频的两套模板。
4. 复杂度多为 O(n log n)（排序主导）；硬币找零、0-1 背包是经典反例。
5. 局部最优推不出全局时改 DP；工程上贪心常作可解释的近似策略。
