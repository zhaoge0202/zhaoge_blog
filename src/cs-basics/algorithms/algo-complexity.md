# 时间与空间复杂度怎么分析？

> 复杂度描述的是规模增长时的成本趋势，不是某台机器上的毫秒数。

面试官问复杂度，其实想听两件事：这段代码随 `n` 变大时跑多少轮，以及额外占了多少空间。先把量级说清，再补一句工程上的常数和缓存影响，就够用了。

## 大 O、Ω、Θ 分别在说什么

三种记号都是在描述「增长趋势」，不是精确计时：

| 记号      | 含义                                   | 口语               |
| --------- | -------------------------------------- | ------------------ |
| `O(f(n))` | 上界：不会比 `f(n)` 更差（渐近意义下） | 最坏别超过这个量级 |
| `Ω(f(n))` | 下界：至少也要 `f(n)` 量级             | 再优化也甩不掉这块 |
| `Θ(f(n))` | 上下界同阶                             | 紧的量级，就这么快 |

面试默认说的「复杂度」几乎都是 **Big O（最坏上界）**。只有在最好/平均差很多时，才主动拆开讲：

- **最好**：运气最好的输入，如有序数组上的插入排序 `O(n)`。
- **最坏**：对手构造的坏输入，如快排遇到已有序且 pivot 选得差 `O(n²)`。
- **平均**：随机输入的期望，快排平均 `O(n log n)`。

什么时候必须区分？算法对输入形态敏感时——快排、哈希表、某些启发式搜索。若最坏和平均同阶（归并、堆排序），报一个就行。

## 常见量级与工程对照

量级阶梯从慢到快（增长更猛）：

```text
O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(n³) < O(2ⁿ) < O(n!)
```

结合题面约束，可以快速排除暴力：

| 规模 `n`    | 通常可接受             | 直觉               |
| ----------- | ---------------------- | ------------------ |
| ≤ 20        | `O(2ⁿ)`、状态压缩      | 枚举子集还扛得住   |
| ≤ 100       | `O(n³)` 有时可以       | 三维 DP / 三重循环 |
| ≤ 1000      | `O(n²)`                | 两两枚举常见       |
| ≤ 1e5       | `O(n log n)` 或 `O(n)` | 排序 + 一遍扫      |
| ≥ 1e6 ~ 1e7 | 接近 `O(n)`            | 常数也要克制       |

这不是铁律：语言常数、是否卡常、机器性能都会动边界。但面试口述时，用「`n=1e5` 所以不能 `O(n²)`」这种判断，比空背表格更有说服力。

## 分析步骤模板

把「怎么数」固定成三步，不容易漏：

1. **定基本操作**：比较一次、赋值一次、哈希查询一次……选一种做计数单位。
2. **顺序结构相加**：先跑 A 再跑 B，总代价 `T = T_A + T_B`，取主项。
3. **嵌套结构相乘**：外层 `n` 次、内层每次 `m` 次 → `O(n·m)`；内层规模还随外层变，就先写出求和再估阶。

补充两条经验：

- 循环变量每次 **翻倍/减半** → 通常 `O(log n)`。
- 双指针虽然写在 `while` 里，但若左右只单调走、各最多 `n` 步 → 整体仍是 `O(n)`，别看成嵌套平方。

## 代表分析一：嵌套循环计数

```text
for i = 0 .. n-1:
    for j = i .. n-1:
        // O(1) 工作
```

内层次数是 `n + (n-1) + … + 1 = n(n+1)/2`，主项 `n²/2`，记 **`O(n²)`**，也是 `Θ(n²)`。

再看一个容易误判的：

```text
for i = 1; i < n; i *= 2:
    for j = 0 .. n-1:
        // O(1)
```

外层 `log n` 轮，每轮内层 `n`，总 **`O(n log n)`**。不能只数「两层 for」就报 `O(n²)`——关键是每层真实次数。

## 递归：递归树与主定理直觉

递归先问两个问题：

1. 每层拆出 **几个子问题**？
2. 本层除了递归，还做了多少 **额外工作**？

### 二分递归

每次只进一个子问题，规模减半，每层 `O(1)` 工作：

```text
T(n) = T(n/2) + O(1)  →  深度 log n  →  O(log n)
```

递归栈深度也是 `log n`，空间 **`O(log n)`**（迭代二分可压到 `O(1)`）。

### 归并分治

拆成 2 个子问题，规模各 `n/2`，合并代价 `O(n)`：

```text
T(n) = 2 T(n/2) + O(n)
```

画递归树：共 `log n` 层，每层合并总工作量都是 `O(n)`，所以 **`O(n log n)`**。额外需要合并缓冲区，空间通常 **`O(n)`**（另加递归栈 `O(log n)`，主项仍是 `O(n)`）。

### 主定理直觉（够面试用）

形如 `T(n) = a T(n/b) + O(n^d)`：

| 关系          | 结论直觉                      |
| ------------- | ----------------------------- |
| `log_b a > d` | 叶子占主导 → `O(n^{log_b a})` |
| `log_b a = d` | 每层同阶 → `O(n^d log n)`     |
| `log_b a < d` | 根层占主导 → `O(n^d)`         |

归并：`a=2,b=2,d=1`，`log_2 2 = 1 = d`，故 `O(n log n)`。二分：`a=1,b=2,d=0`，`log_2 1 = 0 = d`，故 `O(log n)`。

反例：朴素递归斐波那契 `T(n)=T(n-1)+T(n-2)+O(1)`，子问题重叠且树接近满二叉，约 **`O(2ⁿ)`**；加记忆化后每个状态算一次，回到 **`O(n)`**。

## 空间：额外空间 vs 输入，别忘递归栈

空间复杂度默认看 **算法额外开辟** 的空间，输入本身一般不算：

- 新建数组 / 哈希表 / 队列 / 栈
- **递归调用栈**（没写容器也算）
- 排序、归并的辅助缓冲
- 结果集是否计入：看题面；面试可主动说「若输出算额外，则……」

例子：迭代反转链表只要几个指针 → `O(1)`；递归反转虽然没有数组，栈深 `n` → **`O(n)`**。

## 常数因子与缓存局部性

渐近相同，工程上仍可能差几倍甚至一个数量级：

- **常数因子**：同样 `O(n)`，内层多一次哈希、多一次取模，`n=1e7` 时差距明显。
- **局部性**：连续数组比链表友好；列优先扫二维矩阵容易 cache miss。
- **`O(n)` vs `O(n log n)`**：`n` 很小时，带大常数的线性扫可能输给轻量的 `n log n`。

面试可以补一句：「渐近上 A 更优；若 `n` 固定且很小，要以实测为准。」这显得你不是只会背符号。

## 面试怎么口述复杂度

别只丢一个结论，按「做了什么 → 为何是这个量级」讲：

```text
两数之和哈希做法：扫一遍数组，每个元素对 HashMap 做查询和插入，
平均 O(1)，故时间 O(n)。最坏哈希退化到链表时可能更差，但面试默认平均。
额外 Map 最多存 n 个键，空间 O(n)。
```

口述检查清单：

1. 时间最坏（必要时补平均/最好）
2. 空间额外部分 + 是否含递归栈
3. 若先排序再处理，别漏掉排序的 `O(n log n)`
4. 二维输入写 `O(m·n)`，别顺手写成 `O(n)`

## 边界与易错

| 易错点           | 正确说法                                              |
| ---------------- | ----------------------------------------------------- |
| 忽略 `log`       | 二分、堆调整、平衡树查询常是 `O(log n)`，不要当成常数 |
| 把输入当额外空间 | 入参数组本身通常不计；另开 `O(n)` 拷贝才计            |
| 均摊 vs 最坏     | 动态数组扩容均摊 `O(1)`，单次扩容仍是 `O(n)`          |
| 哈希默认 `O(1)`  | 平均是；最坏冲突链可退化                              |
| 双指针当 `O(n²)` | 指针只前进不回头时，总移动次数线性                    |
| 递归「没开数组」 | 栈空间仍可能是 `O(n)` 或 `O(log n)`                   |

## 小结

1. 复杂度讲的是增长趋势；面试默认 Big O 最坏，敏感算法再拆最好/平均。
2. 分析三板斧：定基本操作、顺序相加、嵌套相乘；翻倍循环想对数。
3. 递归看分支数、子问题规模与每层额外工作；主定理直觉够应付分治。
4. 空间只算额外部分，递归栈不能漏；结果集是否计入要说清。
5. 口述带推导；工程再提常数与局部性，避免「`O(n)` 一定更快」这种绝对话。
