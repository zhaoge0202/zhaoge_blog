# 数据脱敏和字段加密怎么落地？

> 脱敏解决“展示时少暴露”，加密解决“存储和传输时看不懂”。两者目标不同，不能互相替代。

## 先判断数据属于哪一类

敏感数据保护不能靠“看到手机号就打星号”这种零散规则。更稳的做法是先分级，再决定展示、存储、查询、导出和日志策略。

常见数据可以粗分为四类：

| 类型           | 例子                         | 主要风险                     | 常见保护手段                      |
| -------------- | ---------------------------- | ---------------------------- | --------------------------------- |
| 身份识别信息   | 手机号、邮箱、身份证号、姓名 | 撞库、社工、隐私泄露         | 展示脱敏、字段加密、访问审计      |
| 金融与证照信息 | 银行卡、合同、证件照片       | 盗用、合规风险、批量泄露     | 强加密、最小权限、导出审批        |
| 凭据与密钥     | Token、验证码、API Secret    | 接管账号、伪造请求、横向移动 | 不落明文、短期有效、哈希存储、KMS |
| 业务敏感信息   | 报价、库存、诊断结果、风控分 | 商业泄露、越权访问、内部滥用 | 权限隔离、字段级策略、审计和告警  |

不同类型的目标不一样：

- 手机号可能需要“能展示后四位、能等值查询、不能批量明文导出”；
- 身份证号可能需要“平时不可见、授权后短时查看、数据库密文保存”；
- API Secret 可能只允许创建时展示一次，后续只保存哈希用于校验；
- 测试环境可能不需要真实数据，应该使用不可逆脱敏或合成数据。

所以落地前先问五个问题：

1. 这个字段是否必须保存明文语义？
2. 是否需要反查、等值查询、模糊查询或排序？
3. 哪些角色、接口、任务可以看到明文？
4. 明文会不会进入日志、缓存、消息、搜索索引和导出文件？
5. 密钥轮换、人员离职、数据泄露时怎么止损？

## 脱敏、加密、哈希分别管什么？

这三类手段经常被混用，但它们的能力边界完全不同：

| 手段      | 是否可逆 | 主要用途                   | 典型例子                   |
| --------- | -------- | -------------------------- | -------------------------- |
| 展示脱敏  | 否       | 减少页面、日志、导出的暴露 | `138****8888`              |
| 字段加密  | 是       | 存储后仍可按权限解密使用   | 身份证号密文入库           |
| 哈希/摘要 | 否       | 校验、去重、等值匹配       | 手机号标准化后做 HMAC 索引 |
| Token 化  | 可控可逆 | 用替代值隔离真实敏感数据   | 支付卡号换成内部 token     |

一个容易误导的说法是“用 MD5 或 SHA-256 对银行卡号加密”。严格说，这不是加密，而是摘要或哈希；它不能解密回原文，只能用于匹配或校验。如果业务还要展示完整银行卡号，就必须另存可解密密文；如果只需要判断“是不是同一张卡”，摘要列可能就够了。

## 展示脱敏怎么设计？

展示脱敏的目标是“当前场景不需要完整明文，就不要暴露完整明文”。它不是数据库安全边界，也不能替代后端权限校验。

常见规则可以统一成策略表：

| 字段       | 默认展示              | 高权限展示           | 备注                       |
| ---------- | --------------------- | -------------------- | -------------------------- |
| 手机号     | `138****8888`         | 审批后短时完整展示   | 查看明文要记录原因和操作人 |
| 身份证号   | `110***********12`    | 原则上不展示完整明文 | 导出场景更要收紧           |
| 邮箱       | `zh***@a.com`         | 按业务需要展示       | 注意企业邮箱也可能是隐私   |
| 银行卡号   | `6222 **** **** 1234` | 极少数场景完整展示   | 不要落入普通业务日志       |
| 地址       | 省市区可见            | 详细地址按权限查看   | 配送、售后场景要单独授权   |
| Token/密钥 | 永不完整展示          | 创建时最多展示一次   | 后续只能重置，不能找回     |

工程上不要让每个 Controller 自己 `substring`。更稳的是集中到序列化层、DTO 转换层或字段策略组件：

```text
Entity 密文字段
  ↓ 解密服务按权限取明文
Domain / DTO
  ↓ 脱敏策略按场景处理
Response / Export / Log
```

需要注意两个边界：

1. **前端脱敏不可信**：后端返回了完整明文，前端再打星号，只能改善展示，不能防泄露。
2. **脱敏规则要按场景切换**：列表页、详情页、客服页、后台审核页、导出文件的规则不应该完全一样。

完整明文查看应该是独立敏感权限，不要默认跟“详情查看”绑定。普通查询返回脱敏值；完整明文需要角色权限、数据范围、查看原因、审批单或短时授权同时满足。客服、审核、财务、运维和导出任务看到的字段范围也应该分开设计。

## 字段加密怎么落表？

字段加密不是把 `phone` 改成 `phone_encrypt` 就结束了。一个可维护的设计通常要同时保存算法、密钥版本、随机数和密文：

```text
user_profile
├── id
├── phone_cipher
├── phone_key_id
├── phone_nonce
├── phone_tag
├── phone_index
└── phone_last4
```

各字段的作用：

| 字段           | 作用                                    |
| -------------- | --------------------------------------- |
| `phone_cipher` | 加密后的密文                            |
| `phone_key_id` | 使用哪一个密钥版本，支持轮换            |
| `phone_nonce`  | 加密随机数或 IV，通常不保密但不能乱复用 |
| `phone_tag`    | AEAD 模式的认证标签，用于发现篡改       |
| `phone_index`  | 等值查询索引，通常用 HMAC 标准化值      |
| `phone_last4`  | 展示后四位或辅助人工核对                |

如果使用 AES-GCM 这类 AEAD 模式，也可以把认证标签和密文放在同一个二进制字段里，但逻辑上仍要知道它们存在。

写入流程可以简化成：

```text
原始手机号
  ↓ 标准化：去空格、去分隔符、统一国家码
normalizedPhone
  ↓ HMAC(indexKey, normalizedPhone)
phone_index
  ↓ AES-GCM(dataKey, normalizedPhone, aad, nonce)
phone_cipher + tag + nonce + keyId
```

这里的 `aad` 可以放业务上下文，例如租户 ID、用户 ID、字段名。这样密文被挪到其他租户或其他字段时，解密校验会失败。

`phone_index` 只是等值查询索引，不是字段加密本身。`indexKey` 要和 `dataKey` 分离，并按环境、租户或用途隔离。低基数字段不适合建 HMAC 查询列，比如性别、状态这类值域太小的字段，即使不可逆也会泄露明显模式。

## 加密后还能查询吗？

这是字段加密最容易低估的问题。加密会破坏数据库原本的索引、排序和模糊查询能力。

常见方案如下：

| 查询需求      | 可选方案                           | 代价和边界                         |
| ------------- | ---------------------------------- | ---------------------------------- |
| 等值查询      | 标准化后存 HMAC 索引列             | 只能查完全一致，密钥泄露会影响索引 |
| 后四位查询    | 单独保存 `last4`                   | 只能辅助筛选，不能作为唯一身份依据 |
| 前缀/模糊查询 | 尽量改业务流程，或引入受控搜索字段 | 泄露模式更多，设计要非常谨慎       |
| 排序/范围查询 | 不建议对强敏字段直接支持           | 会泄露顺序关系，复杂方案成本很高   |
| 去重          | HMAC/摘要列                        | 要先标准化，否则同一值可能多种写法 |

例如手机号登录需要按手机号查用户，就不能只保存随机加密后的手机号密文。随机加密每次结果都不同，数据库没法直接用明文条件命中。比较常见的设计是：

```sql
-- 伪代码：应用侧先算 phone_index，再查库
select id, phone_cipher, phone_key_id
from user_profile
where tenant_id = ?
  and phone_index = ?
limit 1;
```

查到后再按权限决定是否解密展示。这个设计把“查询”和“展示”拆开：查询靠不可逆索引列，展示靠密文字段。

手机号、邮箱这类字段虽然看起来空间很大，但现实里仍然可能被枚举。一旦索引密钥泄露，攻击者可以离线枚举常见号码段或邮箱字典。所以 HMAC 索引要和访问控制、密钥隔离、审计一起设计，不能当成万能匿名化。

## 密钥怎么管，怎么轮换？

字段加密最核心的不是算法，而是密钥生命周期。推荐把密钥分层：

```text
KMS / 主密钥
  ↓ 包裹
数据密钥 dataKey
  ↓ 加密
业务字段密文
```

落地要求：

1. **密钥不进代码仓库**：本地配置、单测样例、Docker 镜像都不能带生产密钥。
2. **按环境和用途隔离**：生产、预发、测试密钥分开；字段加密、JWT 签名、接口 HMAC 密钥分开。
3. **密文保存 keyId**：没有 keyId，就很难知道历史数据该用哪把密钥解。
4. **轮换支持双读单写**：新数据用新密钥，旧数据按 keyId 读取，后台逐步重加密。
5. **密钥访问要审计**：解密失败率、明文查看次数、批量导出行为都要能追踪。

轮换流程可以这样做：

```text
发布新 keyId
  ↓
写入新数据使用新密钥
  ↓
读取旧数据仍按旧 keyId 解密
  ↓
后台批量重加密旧数据
  ↓
确认无旧 keyId 访问后停用旧密钥
```

重加密任务要限速、可恢复、幂等，并记录迁移进度；旧 key 停用前要确认没有存量密文依赖。密钥删除、禁用、导出都应该有审批和回滚窗口，否则一次误操作就可能让历史数据不可读。

不要把轮换设计成“一改配置，全量数据立刻不可读”。字段加密的迁移通常是灰度过程。

## 日志、缓存、消息和导出更容易漏

很多项目数据库字段加密做了，最后却在其他链路泄露明文。

重点检查这些位置：

| 位置     | 常见泄露方式                         | 防护方式                         |
| -------- | ------------------------------------ | -------------------------------- |
| 日志     | 请求参数、异常堆栈、SQL 参数完整打印 | 日志拦截脱敏，敏感字段禁止打印   |
| 缓存     | Redis 里保存完整用户资料             | 缓存值脱敏或加密，设置合理 TTL   |
| 消息队列 | 事件消息携带完整证件号               | 只传 ID，消费端按权限查询        |
| 搜索索引 | 同步完整手机号、地址                 | 只同步必要字段，敏感字段单独治理 |
| 导出文件 | Excel 批量导出明文                   | 审批、水印、限量、过期下载       |
| 链路追踪 | span tag 记录请求体                  | 参数采样和字段清洗               |
| 测试环境 | 生产数据直接导入                     | 不可逆脱敏或合成数据             |

尤其是日志。不要只依赖开发人员“注意不要打印”。更稳的是在日志框架、网关、统一异常处理和审计组件里做字段清洗。

明文查看和解密也要形成证据链。审计字段至少包括操作人、目标用户或数据 ID、字段名、查看原因、审批单号、调用来源、时间、IP/设备、结果和批量规模。短时间大量解密、非工作时间查看、导出量异常、解密失败率异常，都应该触发告警。

## 测试数据脱敏怎么做？

生产数据下发到开发、测试、外包、数据分析环境前，应优先考虑不可逆脱敏或合成数据。

常见处理方式：

| 方式     | 适合场景             | 注意点                       |
| -------- | -------------------- | ---------------------------- |
| 掩码     | 展示演示、人工核验   | 不能保留可还原信息           |
| 替换     | 姓名、地址、邮箱     | 要保持格式合法，避免测试失真 |
| 重排     | 批量统计、训练样本   | 单字段重排仍可能被关联还原   |
| 加噪     | 金额、位置、评分     | 要评估对统计结果的影响       |
| 合成数据 | 新系统测试、压测     | 更安全，但需要构造真实分布   |
| Token 化 | 仍需跨系统关联的场景 | token 映射表本身要严格保护   |

测试数据治理的目标不是“看起来打码了”，而是让非生产环境无法还原真实个人或业务敏感信息，同时尽量保留测试所需的数据形态。

单字段脱敏不代表整个数据集安全。手机号、地址、订单时间、地理位置、金额这些字段组合在一起，仍可能重新识别个人或客户。生产数据下发前要按数据集整体评估重识别风险，必要时优先使用合成数据。

## 容易踩的坑

1. **把脱敏当权限控制**：后端已经返回完整明文，只靠前端打码没有安全意义。
2. **把哈希叫加密**：哈希不可逆，适合匹配或校验；需要展示原文时还要保存密文。
3. **加密后才发现不能查**：等值查询、模糊查询、排序都要在建模时提前设计。
4. **密钥和密文放在同一个风险面**：数据库泄露时，如果配置表里也有密钥，加密基本失效。
5. **日志里打印明文**：日志、链路追踪、告警平台经常比数据库权限更松。
6. **所有场景一套脱敏规则**：客服、审核、导出、普通用户页面的可见范围应该不同。
7. **轮换没有 keyId**：历史数据无法平滑迁移，只能停机或写复杂兼容逻辑。
8. **生产数据直接进测试环境**：这是非常常见的数据泄露入口。

## 小结

1. 脱敏、加密、哈希解决的问题不同：展示少暴露、存储可解密保护、不可逆匹配校验。
2. 敏感字段要先分级，再分别设计展示、查询、存储、导出、日志和测试数据策略。
3. 字段加密要保存 keyId、nonce/iv、密文和认证信息，并配套 KMS、轮换和审计。
4. 等值查询通常用标准化后的 HMAC 索引列，密文字段只负责按权限解密展示。
5. 数据保护不能只盯数据库，日志、缓存、消息、搜索索引、导出文件和测试环境都要治理。
