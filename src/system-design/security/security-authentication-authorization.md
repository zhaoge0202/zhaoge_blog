# 认证和授权有什么区别？

> 认证回答“你是谁”，授权回答“你能做什么”。很多安全设计混乱，都是因为把这两件事揉成了一步。

## 先看一个接口请求

用户访问后台删除订单接口时，服务端至少要做两层判断：

1. 认证：请求里带的登录态能不能证明这个人是 `userId=1001`。
2. 授权：`userId=1001` 有没有删除这笔订单的权限。

```text
请求 -> 解析登录态 -> 得到用户身份 -> 查询角色/权限 -> 判断资源动作是否允许
```

如果没有认证，系统不知道是谁在访问；如果没有授权，任何登录用户都可能越权访问。

## 认证通常怎么做？

认证的输入是凭据，输出是身份。

常见凭据包括：

| 凭据          | 说明                     | 风险点                 |
| ------------- | ------------------------ | ---------------------- |
| 用户名 + 密码 | 最常见登录方式           | 弱密码、撞库、明文传输 |
| 手机验证码    | 短期动态凭据             | 短信轰炸、验证码泄露   |
| 第三方登录    | 由外部身份提供方确认身份 | 回调校验、账号绑定     |
| 客户端证书    | 强身份校验               | 证书管理成本           |

认证成功后，服务端通常会生成 Session 或 Token，让后续请求不必每次都输入密码。

## 授权通常怎么做？

授权的输入是身份、资源和动作，输出是允许或拒绝。

```text
用户 userId=1001
资源 orderId=9001
动作 delete
结果 allow / deny
```

授权不能只看“是否登录”，还要看资源归属和权限范围。比如：

- 普通用户只能查看自己的订单。
- 客服只能查看负责租户下的订单。
- 管理员也不一定能绕过审批删除关键数据。

## 容易踩的坑

**只在前端隐藏按钮。** 前端隐藏只能改善体验，不能当权限控制。后端接口必须重新校验。

**只按角色判断，不看数据归属。** `ROLE_USER` 可以访问订单详情，不代表能访问别人的订单详情。

**认证信息信任过头。** Token 里的用户 ID 只能说明请求声明了身份，还要校验签名、过期时间和吊销状态。

## 小结

1. 认证解决身份问题，授权解决权限问题。
2. 认证成功后才进入授权，但两者不能合并。
3. 授权要同时考虑角色、动作、资源和数据归属。
4. 前端权限只是体验控制，后端接口必须做最终校验。
