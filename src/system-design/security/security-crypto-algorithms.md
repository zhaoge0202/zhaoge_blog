# 常见加密算法在后端系统里怎么用？

> 后端安全里最常见的错误，是把哈希、加密、签名混着用。它们解决的问题完全不同。

## 先把几个概念拆开

很多安全事故不是算法本身选错，而是把几个概念混着用：

| 能力         | 是否可逆 | 是否需要密钥               | 解决的问题                   | 常见机制                         |
| ------------ | -------- | -------------------------- | ---------------------------- | -------------------------------- |
| 摘要/哈希    | 否       | 通常不需要                 | 完整性校验、指纹、去重       | SHA-256、SHA-3、SM3              |
| 密码哈希/KDF | 否       | salt 必须随机，可加 pepper | 密码存储、对抗离线爆破       | Argon2id、bcrypt、scrypt、PBKDF2 |
| MAC/HMAC     | 否       | 需要共享密钥               | 防篡改、证明消息来自持密钥方 | HMAC-SHA256、CMAC                |
| 对称加密     | 是       | 加解密使用同一把密钥       | 保护数据机密性               | AES-GCM、ChaCha20-Poly1305、SM4  |
| 非对称加密   | 是       | 公钥加密、私钥解密         | 密钥交换、小数据加密         | RSA、ECIES、SM2                  |
| 数字签名     | 否       | 私钥签名、公钥验签         | 身份确认、不可否认、防篡改   | RSA-PSS、ECDSA、EdDSA、SM2       |

一句话区分：

- **哈希**回答“内容有没有变”；
- **密码哈希/KDF**回答“这个密码是否匹配，同时尽量拖慢爆破”；
- **HMAC/MAC**回答“消息有没有被不知道密钥的人篡改”；
- **加密**回答“别人能不能看懂内容”；
- **签名**回答“是不是某个私钥持有人发出的内容”。

所以不要说“用 MD5 加密密码”，也不要说“用 RSA 签名后别人就看不到内容”。这些说法都把目标混在一起了。

## 哈希能不能用来存密码？

普通哈希不适合直接存密码。原因很简单：SHA-256、SM3 这类摘要函数设计目标是快，适合做文件校验、内容指纹、数据去重；但密码存储最怕的就是“攻击者拿到哈希后可以高速离线猜密码”。

密码存储应该使用密码哈希/KDF：

```text
用户密码 + 随机 salt
  ↓
Argon2id / bcrypt / scrypt / PBKDF2
  ↓
保存：算法版本 + 参数 + salt + hash
```

几个工程要点：

1. 每个用户都要有独立随机 `salt`，不要全站共用固定盐。
2. `cost`、内存成本、并行度这些参数要能随硬件升级调整。
3. 只保存哈希结果，不保存明文密码，也不保存可逆密文。
4. 需要额外保护时，可以把 `pepper` 放在 KMS 或密钥管理系统里，而不是和用户表放在一起。

如果只是文件校验或幂等指纹，SHA-256/SM3 可以用；如果是密码，直接上密码哈希/KDF。

## 接口签名为什么常用 HMAC？

接口签名要解决的不是“内容不被看见”，而是“内容不能被篡改、请求确实来自持有密钥的一方”。

典型流程是：

```text
method + path + query + bodyHash + timestamp + nonce
  ↓ 规范化拼接
canonical request
  ↓ HMAC-SHA256(secret, canonical request)
signature
```

服务端收到请求后，用同样方式重新计算签名。两边一致，说明请求内容没有被不知道密钥的人改过。

这里有几个细节很容易踩坑：

1. **必须规范化参数**：排序、大小写、空格、编码方式要固定，否则客户端和服务端算出来不一致。
2. **要防重放**：只签名不够，还要校验 `timestamp`、`nonce` 或请求序列号。
3. **不要把 secret 放前端**：浏览器、小程序、移动端都不是保密环境，前端签名只能防误用，不能当强安全边界。
4. **HMAC 不是加密**：payload 仍然是明文，机密性要靠 HTTPS 或额外加密。
5. **比较签名要用常量时间比较**：不要用会提前返回的字符串比较，避免把匹配前缀长度变成侧信道。

比起“把参数拼起来做 MD5”，HMAC 的优势是引入了密钥，不知道密钥的人不能伪造合法摘要。

## 对称加密适合保护哪些数据？

只要数据量较大，并且需要加密后还能解密，通常优先考虑对称加密。比如：

- 数据库里的身份证号、银行卡号、手机号等敏感字段；
- 对象存储里的合同、证件照片、报表文件；
- 消息队列中的敏感业务消息；
- 备份文件、导出文件、离线数据包。

现代后端更推荐使用 AEAD 模式，例如 AES-GCM 或 ChaCha20-Poly1305。它们不只加密，还会给密文做完整性校验，能发现密文被篡改。

不推荐再用 ECB；CBC/CTR 这类模式即使用得对，也要额外处理完整性校验。新系统优先选成熟库里的 AEAD 模式，少自己组合“加密 + 摘要 + 拼接格式”。

一条记录加密时，至少要保存这些信息：

```text
ciphertext = Encrypt(dataKey, plaintext, aad, nonce)

保存字段：
algorithmVersion
keyId
nonce / iv
ciphertext
authTag
```

注意 `nonce`/`iv` 通常不需要保密，但必须满足算法要求，尤其不能在同一把密钥下重复。AES-GCM 里重复 nonce 会严重破坏安全性。

随机数也要用密码学安全随机源，例如 Java 里的 `SecureRandom`。不要用时间戳、自增 ID、普通 `Random` 拼 nonce、密钥或 token。`aad` 可以放租户 ID、业务 ID、字段名这类上下文，让密文被挪到其他上下文时无法通过认证。

## 非对称加密是不是更安全，所以都用 RSA？

不是。非对称算法解决的是“没有共享密钥时怎么建立信任或交换密钥”，不是拿来加密大数据的通用方案。

RSA、ECC、SM2 这类算法有几个工程边界：

1. 计算成本比对称加密高很多。
2. 能直接加密的数据长度有限。
3. 实际系统里常用于密钥交换、数字签名、证书体系，而不是加密整份文件。
4. RSA 加密要选安全填充，例如 OAEP，不要裸 RSA 或继续沿用不合适的旧填充。

更常见的做法是“混合加密”：

```text
随机生成 dataKey
  ↓
用 dataKey 加密大数据
  ↓
用公钥或 KMS 加密 dataKey
  ↓
密文 + 加密后的 dataKey 一起保存
```

这就是信封加密的基本思路。大数据交给对称加密，密钥保护交给非对称算法或 KMS，各自做擅长的事。

签名也不要口头说成“用私钥加密”。更准确的是：先对待签内容做规范化和摘要，再用私钥生成签名，接收方用公钥验签。签名保证来源和完整性，不保证内容保密。

## HTTPS 里这些算法怎么配合？

HTTPS/TLS 不是“只用了 RSA”或者“只用了 AES”。它是一套协议，里面同时用到了多类能力：

```text
证书链校验
  ↓ 确认服务端身份
密钥协商
  ↓ 生成会话密钥
对称加密 + 完整性保护
  ↓ 保护后续 HTTP 数据
```

可以这样理解：

| 阶段     | 主要目标       | 常见机制                        |
| -------- | -------------- | ------------------------------- |
| 证书校验 | 确认服务端身份 | X.509 证书、CA 信任链、数字签名 |
| 密钥协商 | 生成会话密钥   | ECDHE 等                        |
| 数据传输 | 机密性和完整性 | AES-GCM、ChaCha20-Poly1305      |

早期资料常把 HTTPS 讲成“浏览器用服务器 RSA 公钥加密随机数”，这个说法只覆盖了旧式 RSA 密钥交换的一部分。现代 TLS 更常见的是基于 ECDHE 的临时密钥协商，核心收益是前向安全：即使服务器长期私钥未来泄露，历史会话也不应该被直接解开。

## 密钥管理比算法选择更容易翻车

算法选对只是第一步。更多线上事故来自密钥管理：

```text
业务数据
  ↓ dataKey 加密
密文数据 + keyId
  ↓
dataKey 由 KMS / masterKey 保护
```

落地时至少要做到：

1. **密钥不进代码仓库**：配置文件、测试脚本、前端包里都不能硬编码生产密钥。
2. **按环境隔离**：开发、测试、预发、生产不能共用密钥。
3. **按用途隔离**：JWT 签名密钥、字段加密密钥、接口 HMAC 密钥不要混用。
4. **有 keyId 和版本**：密文里记录密钥版本，才能支持轮换和灰度迁移。
5. **轮换要能双读单写**：新数据用新密钥写，旧数据读取时按 keyId 解密，再逐步迁移。
6. **日志不能泄露**：不要打印明文、密钥、完整 Token、完整签名串和重置链接。
7. **操作要审计**：谁使用了密钥、何时解密、失败率是否异常，都应该能追踪。

密钥轮换不是“改一个配置重启”。只要历史数据还用旧密钥加密，系统就必须知道旧密钥版本和迁移策略。

密钥还要分清“数据加密密钥”和“密钥加密密钥”。业务数据通常由 dataKey 加密，dataKey 再由 KMS 或 masterKey 保护。这样可以按租户、字段或文件粒度轮换 dataKey，而不必把所有明文都暴露给应用配置。

## 典型场景怎么选？

可以按目标反推算法：

| 场景             | 推荐做法                           | 不推荐做法                      |
| ---------------- | ---------------------------------- | ------------------------------- |
| 登录密码存储     | Argon2id / bcrypt / scrypt + salt  | MD5、SHA-256、AES 可逆加密      |
| 文件完整性校验   | SHA-256 / SHA-512 / SM3            | 用加密结果当文件指纹            |
| 开放接口防篡改   | HMAC-SHA256 + timestamp + nonce    | 参数拼接后 MD5，且不防重放      |
| 敏感字段可逆保护 | AES-GCM / SM4 + KMS / 信封加密     | ECB 模式、固定 IV、密钥写代码   |
| 大文件加密       | 随机 dataKey + 对称加密 + 信封加密 | RSA 直接加密整个文件            |
| JWT 防篡改       | 固定允许算法，使用 JWS 签名        | 接受 `none` 或从 token 里信算法 |
| HTTPS 传输保护   | TLS 1.2+ / TLS 1.3，正确证书配置   | 自己在 HTTP 上拼一套加密流程    |

这个表的核心是：先问风险是什么，再选机制。想保密就加密，想防篡改就 MAC/签名，想存密码就密码哈希，想校验完整性就摘要。

选型还要看生命周期：是否需要轮换密钥、是否需要审计解密、是否需要跨语言互通、是否有合规要求、是否能接受历史数据重加密成本。算法只是起点，密钥、参数、模式和协议组合才决定最终安全性。

## 容易踩的坑

1. **把 MD5/SHA-256 当密码存储方案**：普通摘要太快，泄露后很容易被离线爆破。
2. **把哈希叫加密**：哈希不可逆，不能解密；加密可逆，必须保护密钥。
3. **使用 ECB 模式**：相同明文块会得到相同密文块，容易泄露数据模式。
4. **固定 IV 或重复 nonce**：尤其在 GCM 这类模式下，重复 nonce 会带来严重风险。
5. **自己设计协议**：密码协议组合很容易出错，应优先用成熟库和标准协议。
6. **密钥进代码库或日志**：算法再强，密钥泄露也会失去保护意义。
7. **用 RSA 加密大文件**：性能差、长度受限，应改成混合加密或信封加密。
8. **只加密不鉴别完整性**：密文被篡改时无法可靠发现，优先使用 AEAD 模式。

## 小结

1. 哈希、密码哈希、HMAC、加密、签名解决的是不同问题，不能混用概念。
2. 密码存储要用 Argon2id、bcrypt、scrypt 这类密码哈希/KDF，不要直接用普通摘要。
3. 敏感字段和文件加密优先用对称加密，并配合 KMS、keyId、轮换和审计。
4. 非对称算法更适合密钥交换和签名，大数据加密通常用混合加密或信封加密。
5. HTTPS/TLS 是证书认证、密钥协商、对称加密和完整性保护的组合，不是一种单独算法。
