# 常见加密算法在后端系统里怎么用？

> 后端安全里最常见的错误，是把哈希、加密、签名混着用。它们解决的问题完全不同。

## 四类能力怎么分工？

| 类型            | 解决什么                 | 常见算法/机制          |
| --------------- | ------------------------ | ---------------------- |
| 摘要/哈希       | 校验完整性，不能还原     | SHA-256、SM3           |
| 密码哈希        | 存密码，抗爆破           | bcrypt、scrypt、Argon2 |
| 对称加密        | 加解密同一把密钥         | AES、SM4               |
| 非对称加密/签名 | 密钥交换、验签、身份确认 | RSA、ECDSA、SM2        |

不要用普通哈希存密码，不要用加密替代签名，也不要自己设计加密协议。

## 典型场景怎么选？

- 密码存储：慢哈希 + salt。
- 接口防篡改：请求参数 canonicalize 后做 HMAC 或签名。
- 敏感字段存储：信封加密，业务密钥再由 KMS 管。
- 文件完整性：SHA-256 摘要。
- HTTPS：证书身份认证、密钥协商、对称加密分工协作。

## 密钥比算法更容易出问题

算法选对只是第一步。更常见的事故来自密钥硬编码、日志打印密钥、多人共享密钥、长期不轮换。

工程上至少要做到：

1. 密钥不进代码仓库。
2. 密钥按环境隔离。
3. 关键密钥有轮换机制。
4. 加解密失败有审计，不打印明文。

## 小结

1. 哈希不可逆，加密可逆，签名用于证明来源和防篡改。
2. 密码存储要用慢哈希，不要用 MD5/SHA 系列直接存。
3. 敏感字段加密要同时设计密钥管理和轮换。
4. 不要自研密码协议，优先使用成熟库和官方推荐模式。
