# RESTful 接口怎么设计才不容易返工？

> 接口设计翻车基本不会翻在"这个 URL 该不该用复数"这种细节上，会翻在没约定错误码、没考虑幂等、字段改动导致老客户端崩溃这几件大事上。这篇按会返工的顺序讲。

先说清楚一件事：REST 不是宗教，它是一套让 API 好猜、好扩展的**命名和语义约定**。项目里的内部服务、RPC 调用、消息队列消费者，完全没必要削足适履套 REST 的壳——那是给"服务和服务之间隔着团队、隔着文档"的场景准备的。下面讲的规则，套用到什么程度，看你的 API 面向谁：对外开放平台严格套，公司内部两个团队之间的服务可以就着团队习惯来，团队内部一个服务自己调自己甚至直接上 RPC 框架（Dubbo、gRPC）都行，没人逼你写 `GET /users/{id}`。

## 资源命名：先想清楚"这是一个什么东西"

RESTful 的核心思路是把 API 路径当成"名词 + 层级"，而不是"动词 + 参数"。这句话听起来简单，实际最容易在命名时露怯。

| 场景             | 差的写法                                  | 好的写法                                                        | 为什么                                                                            |
| ---------------- | ----------------------------------------- | --------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| 查询所有订单     | `GET /getOrderList`                       | `GET /orders`                                                   | 路径里出现动词，说明你在设计"函数调用"而不是"资源"；HTTP 方法本身已经是动词了     |
| 创建订单         | `POST /createOrder`                       | `POST /orders`                                                  | 同上，`POST` 已经表达了"创建"的语义                                               |
| 查询某用户的订单 | `GET /orders?userId=xxx` 单独作为一等路径 | `GET /users/{userId}/orders`                                    | 有明确从属关系时用嵌套路径，语义上更直接；但嵌套不要超过 2-3 层，否则改用查询参数 |
| 禁用某用户       | `POST /disableUser?id=1`                  | `PATCH /users/1 {"status":"disabled"}` 或 `PUT /users/1/status` | 状态变更本质是资源属性的更新，不是一个新动作                                      |
| 批量删除         | `GET /deleteAll?ids=1,2,3`                | `DELETE /orders?ids=1,2,3` 或 `POST /orders/batch-delete`       | 用 GET 做删除是硬伤：GET 语义上不该有副作用，浏览器/CDN/爬虫可能主动重放 GET 请求 |

有两类操作是真正的例外，允许出现"动词感"的路径：**计算类和触发类操作**，比如 `POST /orders/{id}/cancel`、`POST /files/{id}/archive`、`POST /reports/export`。这些操作本身不是对某个资源做增删改查，而是触发一个业务动作，用 `POST` + 动词末段路径比硬套 PUT/PATCH 更直白，没必要为了"纯洁性"硬拗成 `PATCH /orders/{id} {"action":"cancel"}`。

命名上还有几条低成本但经常被忽略的规矩：

- 集合用复数：`/orders` 不是 `/order`，因为它对应数据库里的一张表、一堆记录。
- 大小写和分隔符统一用小写中划线：`invitation-code`，不要用下划线或驼峰，浏览器地址栏和大部分网关对中划线更友好。
- 资源层级要反映真实的从属关系，不要瞎嵌套。`/companies/{id}/departments/{id}/employees/{id}/tasks` 这种四层嵌套基本没人愿意维护，超过两层就该把后面几层拍平成独立资源加查询参数。

## HTTP 动词：语义比你想的更重要

动词选错，最直接的后果是**语义和实际行为对不上**，导致网关、缓存、重试机制按错误的假设去处理你的请求。

| 方法   | 语义                | 是否安全（不改数据） | 是否天然幂等         | 常见误用                                                                                                                                                                       |
| ------ | ------------------- | -------------------- | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| GET    | 查询                | 是                   | 是                   | 拿 GET 做删除/修改（如上面提到的 `GET /deleteAll`）                                                                                                                            |
| POST   | 创建资源 / 触发动作 | 否                   | **否**               | 把"全量更新"也用 POST 做，导致重复提交产生多条重复数据                                                                                                                         |
| PUT    | 整体替换资源        | 否                   | 是                   | 只传了部分字段却用 PUT，导致没传的字段被置空——这其实是把 PUT 当 PATCH 用了                                                                                                     |
| PATCH  | 局部更新资源        | 否                   | 不保证（取决于实现） | 把"数值 +1"这种操作也放进 PATCH body，两次重放结果不一样，这种 PATCH 其实不幂等                                                                                                |
| DELETE | 删除资源            | 否                   | 是                   | 认为"资源已经不存在了所以第二次删除该报错"——不对，幂等的意思是重复执行的**结果一致**，第二次删除一个不存在的资源，返回 204 或 404 都算符合幂等语义，只要不报 500、不产生副作用 |

"安全"和"幂等"是两个不同维度：安全指不改变服务器状态（只有 GET 和 HEAD 满足），幂等指多次执行效果和执行一次相同。GET 又安全又幂等，PUT/DELETE 不安全但幂等，POST 两者都不占——这也是幂等要单独拎出来讲的原因。

## 错误处理：业务码和 HTTP 状态码是两套东西，别混用

这是新手最容易踩的坑：**要么完全依赖 HTTP 状态码表达业务语义，要么反过来所有响应都返回 200，把状态码当摆设。**

先说 HTTP 状态码该干什么：它表达的是"这次 HTTP 请求本身"的结果——网关层、监控系统、客户端框架都靠它做统一判断（比如 4xx 不重试、5xx 触发熔断降级）。所以它的颗粒度必须粗，够用就行。常见的一套：

| 状态码 | 含义                       | 用在什么场景                                            |
| ------ | -------------------------- | ------------------------------------------------------- |
| 200    | 成功                       | 查询/更新/删除成功                                      |
| 201    | 已创建                     | POST 创建资源成功，通常配合 `Location` 响应头指向新资源 |
| 204    | 成功但无内容               | DELETE 成功、或不需要返回 body 的更新                   |
| 400    | 请求参数不合法             | 缺字段、类型错、格式错                                  |
| 401    | 未认证                     | token 缺失或过期                                        |
| 403    | 已认证但无权限             | 登录了但没权限访问这个资源                              |
| 404    | 资源不存在                 | id 查不到                                               |
| 409    | 状态冲突                   | 并发修改冲突、重复创建唯一资源                          |
| 422    | 参数格式对但业务校验不通过 | 比如手机号格式对，但已被注册                            |
| 429    | 请求过于频繁               | 限流                                                    |
| 500    | 服务器内部错误             | 未捕获异常                                              |

顺带说一句：查资料时经常看到只列 400/401/403/404/405/500/502/504 这几个的表，把 409、422、429 漏掉了——这几个在实际项目里出现频率其实很高（并发冲突、限流、业务校验失败），漏了就只能全部塞进 400 或 500，导致客户端没法区分"要不要重试""要不要提示用户"。

再说业务码该干什么：一次请求里可能有几十种业务失败原因，但 HTTP 状态码就那么几个粗粒度桶，装不下。所以真正定位问题靠的是响应体里的业务码，而不是状态码本身。统一错误体大概长这样：

```json
{
  "code": "ORDER_ALREADY_PAID",
  "message": "订单已支付，不能重复扣款",
  "requestId": "a1b2c3d4",
  "timestamp": "2026-07-14T10:20:30Z"
}
```

`code` 给客户端程序判断分支用（不能用 `message` 做判断，文案随时可能改），`message` 给人看，`requestId` 用来在日志系统里串联排查。这里有个常见分歧：**这类业务失败，HTTP 状态码到底该返回 200 还是对应的 4xx？** 两种做法都有团队在用，各有取舍：

- 状态码统一 200，靠 `code` 字段判断业务结果：好处是前端只用处理一套"成功/失败"分支逻辑，不用同时兼容 HTTP 层和业务层两套异常；坏处是网关、CDN、APM 监控没法直接从状态码分布看出错误率，都得解析 body。
- 状态码按语义返回真实值（4xx/5xx），`code` 只是细分：好处是运维监控、网关熔断策略能直接吃状态码；坏处是前端要同时处理 HTTP 异常分支和业务失败分支，容易漏。

没有标准答案，团队定了就别来回换，最忌讳的是同一个系统里有的接口这么做、有的接口那么做。

## 幂等性：PUT/DELETE 免费得到，POST 得自己造

幂等的价值在哪？**网络会抖动、前端会手抖点两次、重试机制会重发请求**——如果接口不幂等，这些场景都可能导致数据重复或状态错乱。

PUT 和 DELETE 天然幂等，原因很朴素：PUT 传的是资源的完整目标状态，"把它设成这样"这个操作执行几次结果都一样；DELETE 传的是"让它不存在"，执行几次结果也是不存在。这是操作语义本身带来的，不需要额外设计。

POST 不天然幂等，因为它的语义是"创建一个新的"，默认情况下调用两次就是创建两个。最典型的例子是下单接口：用户网络慢点了两次"提交订单"，如果没有防护，后台可能生成两笔订单、扣两次款。

标准解法是**幂等键（Idempotency Key）**：客户端在发起请求前生成一个唯一 ID（通常是 UUID），放在请求头（如 `Idempotency-Key`）里，服务端收到请求后：

1. 先查这个 key 是否已经处理过（存 Redis 或者一张幂等表，key 加短期过期时间）。
2. 没处理过：正常执行业务逻辑，把结果和 key 绑定存下来，返回结果。
3. 已经处理过：直接返回上次存下来的结果，不重复执行业务逻辑。

这套机制的关键是"查询是否处理过"和"执行业务逻辑"必须是原子的，不能查完发现没处理、还没来得及写标记就被第二个并发请求也查到"没处理过"——常见实现是用数据库唯一索引兜底（幂等 key 建唯一约束，插入失败就说明已经处理过），或者用 Redis 的 `SETNX` 抢锁。

幂等键由谁生成也有讲究：如果让服务端生成，客户端断网重连后请求会换一个新 key，等于没做防护；所以幂等键必须**客户端生成**，同一次用户操作，无论重试多少次都带着同一个 key。

## 版本管理：URL 里放版本号，还是塞进 Header？

API 版本升级早晚要面对——字段结构变了、语义变了、老客户端还没升级完，新老版本得共存一段时间。放哪最合适，两种做法各有阵营：

| 方案         | 示例                                                              | 优点                                                                         | 缺点                                                                                                            |
| ------------ | ----------------------------------------------------------------- | ---------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| URL 路径版本 | `/v1/orders`、`/v2/orders`                                        | 一眼看出版本，浏览器直接能访问调试，网关按路径分流最简单，日志天然带版本信息 | 违背"同一个资源应该有唯一 URI"的 REST 原则；版本一多，路由规则、Nginx 配置跟着膨胀                              |
| Header 版本  | `Accept: application/vnd.api.v2+json` 或自定义 `X-API-Version: 2` | URL 保持干净、语义纯粹，符合"资源只有一个地址"的理念                         | 调试麻烦（浏览器地址栏测不了，得靠 Postman/curl 带头），网关做路由分流要解析 header，容易被中间代理层丢弃或改写 |

实际项目里 URL 版本明显更主流，原因很现实：调试成本低、网关分流简单、出问题时看日志一眼知道是哪个版本的流量。Header 版本更多出现在强调"API 即产品"、对外提供 SDK 且严格遵循 REST 理念的开放平台（比如 Stripe 用的是日期版本 + Header）。

还有一种折中：**只在破坏性变更时升版本号，兼容性变更（加字段）不升版本**。这样版本号增长得慢，`v1` 能撑很久，比每次改动都升版本靠谱得多——版本号本身也是一种承诺，升得太频繁等于告诉调用方"我们的接口不稳定"。

## 分页、过滤、排序：参数名统一，比多聪明更重要

这一节没什么深奥的道理，纯粹是"团队内部要统一，别各写各的"。

| 用途     | 推荐参数                                                       | 说明                                                                                                                 |
| -------- | -------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| 分页     | `page` + `size`（页码分页），或 `cursor` + `limit`（游标分页） | 页码分页简单直观但深分页性能差（`OFFSET 100000` 数据库要扫描前面所有行）；游标分页性能稳定但不支持跳页，按业务场景选 |
| 过滤     | `?status=active&type=vip`                                      | 直接用字段名做 query key，不要发明一套自己的 DSL，除非过滤条件复杂到需要专门的查询语言                               |
| 排序     | `?sort=createTime,-price`                                      | 字段名前加 `-` 表示降序，逗号分隔支持多字段排序，这是社区里最常见的约定（JSON API 规范也这么定）                     |
| 字段裁剪 | `?fields=id,name,status`                                       | 列表接口默认别把所有字段都吐出来，允许调用方按需要选字段，减少带宽和序列化开销                                       |

分页接口的响应体也要统一结构，不要有的接口直接返回数组、有的接口包一层 `data`：

```json
{
  "data": [
    /* 当前页数据 */
  ],
  "page": 1,
  "size": 20,
  "total": 358
}
```

深分页是真实会踩的坑：如果一个列表接口允许 `page=5000`，数据库执行 `LIMIT 20 OFFSET 100000` 要先扫描并跳过前 10 万行，页码越往后越慢。日志类、订单类这种数据量大且用户很少翻到后面页的场景，游标分页（返回"下一页的起始标记"而不是页码）是更合适的方案。

## 兼容性：字段只增不删，是接口能长期活下去的底线

接口一旦对外发布，就有别人在依赖它——这句话说起来简单，真正执行起来最容易被"图省事"打破。核心原则就一条：**已发布的字段，改名可以（配合废弃期），直接删除不行；新增字段永远安全，前提是老客户端要能忽略未知字段。**

具体到操作上：

- **加字段不算破坏性变更**：只要客户端解析 JSON 时对未知字段是"忽略"而不是"报错"（大部分 JSON 库默认行为就是忽略），加字段随时可以做，不需要升版本号。
- **删字段、改字段类型、改字段语义，都算破坏性变更**：哪怕这个字段"看起来没人在用了"，只要没法确认所有调用方都不依赖它，就不能直接删——你能看到的调用方不代表全部调用方，尤其是给外部三方开放的接口。
- **废弃一个字段/接口的标准流程**：先标记废弃（响应体里加 `deprecated: true`，或者用 `Deprecation` / `Sunset` 响应头告知下线时间），观察一段时间的调用量，调用量降到可接受范围（或者主动通知剩余调用方迁移）之后再真正下线。直接一次发版就删，等于给自己埋雷。
- **删枚举值也是破坏性变更**：客户端很可能写死了 `switch(status)` 覆盖已知的几个值，突然出现一个没见过的值，处理不好就走进默认分支炸掉——所以枚举新增值本身也不是 100% 安全的，最好让客户端对未知枚举值有兜底逻辑（比如统一显示"未知状态"而不是抛异常）。

## 容易踩的坑

- **把 GET 请求做成有副作用的操作**（比如点击链接就能触发删除），CDN、浏览器预加载、爬虫都可能无意中重放 GET，这类接口迟早出线上事故。
- **PUT 只传部分字段**：PUT 语义是整体替换，没传的字段会被当成"客户端想清空"处理。只想改一个字段就该用 PATCH，图省事把 PUT 当 PATCH 用，是最容易被忽略的语义错配。
- **状态码和业务码混着用又没有统一规范**：一部分接口用状态码表达业务失败、一部分接口无论如何都返回 200，前端要为同一个系统写两套异常处理逻辑。
- **幂等键设计成服务端生成**：这样起不到防重复提交的作用，因为客户端断网重试时根本拿不到同一个 key。
- **分页接口允许无限制翻页**又用的是 offset 分页，数据量一大直接拖垮数据库。

## 小结

- 资源命名用名词、复数、层级；触发类动作允许 `POST /resource/{id}/action` 这种例外，不用为了"纯洁"硬套 PUT/PATCH。
- HTTP 方法要注意"安全"和"幂等"是两个维度：GET 两者都占，PUT/DELETE 幂等不安全，POST 两者都不占。
- HTTP 状态码表达"这次请求"的通用结果（供网关/监控用），业务码放统一错误体里表达具体失败原因，两者别混为一谈，团队要统一 200-always 还是状态码语义化，定了别来回换。
- PUT/DELETE 幂等是操作语义自带的；POST 的幂等要靠客户端生成的幂等键 + 服务端原子查重实现。
- 版本号放 URL 简单直接、适合大多数团队；放 Header 更"纯粹"但调试和网关分流成本更高，选哪个看团队规模和对外开放程度。
- 接口兼容性的底线是字段只增不删，删除前先走废弃期通知，别指望"看起来没人用了"就能直接删。
- 这套约定不是必须无条件套用——内部 RPC 服务、微服务间调用完全可以用更简单直接的方式，不必为了合规而合规。
