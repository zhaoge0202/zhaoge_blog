# 面试里最容易答飘的点有哪些？

有一类错误特别隐蔽：你说出来的话不算错，但说得太满，面试官只要往下追一句，你就露怯了。比如“CAP 三选二”“Redis 分布式锁绝对安全”“Kafka 默认不丢消息”，这些话背下来都顺口，可它们背后都藏着一个前提条件，一旦被问“那如果 XX 呢”，答案立刻站不住。

这篇不重新讲原理，只挑出最容易说过头的十六个点，把“大家常见的答法”和“更经得起追问的答法”放在一起对比，再补两三句解释为什么要这么改。每个点后面配了站内的详细文章，原理没吃透的话可以点进去回炉。

## 1. CAP 三选二，随便挑两个都行

| 错答                                                           | 更稳答法                                                                           |
| -------------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| CAP 三个特性里，工程上随便选两个组合，比如选 CP 就是放弃可用性 | 只要发生网络分区，你就必须在 C 和 A 之间选，P 不是可选项，是分布式系统躲不开的现实 |

很多人把 CAP 讲成“三选二的自由组合”，但网络分区（P）几乎不可能靠设计规避掉，真正能选的只有分区发生时是要一致性还是要可用性。而且这个选择不是系统级的一刀切，同一个系统里不同接口完全可以做不同取舍，比如支付走强一致，商品详情走可用优先。详见[CAP 和 BASE 怎么理解？为什么不是简单三选二？](../distributed-system/distributed-cap-base.html)。

## 2. Redis 分布式锁，加上就绝对安全

| 错答                                                    | 更稳答法                                                                                                                                     |
| ------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| 用 `SET NX PX` 或者 Redisson 加锁，业务就不会重复执行了 | 单节点场景基本够用，但主从异步复制下，主节点没同步就宕机，新主上锁可能被别人重新拿到；跨机房、强一致场景要考虑 Redlock 或转用 ZooKeeper/etcd |

单实例锁最大的坑不在“怎么加锁”，而在“锁过期时间和业务执行时间对不上”以及“主从failover 导致锁丢失”。如果只答出 `SET key value NX PX 30000` 就收尾，面试官大概率会问“如果业务没执行完锁过期了怎么办”，这时候得能讲出续约（看门狗）和高可用代价的取舍。详见[Redis 分布式锁到底能不能用？](../database/redis/redis-distributed-lock.html)。

## 3. Kafka 默认配置不丢消息

| 错答                                             | 更稳答法                                                                                                                                             |
| ------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| Kafka 是消息队列，天生可靠，不用调配置就不丢消息 | 默认 `acks=1` 只保证 leader 落盘就返回成功，leader 挂了且未同步给 follower 时照样丢；要不丢至少得 `acks=all` 配合 ISR 和合理的 `min.insync.replicas` |

“Kafka 不丢消息”这句话背后其实是一整套需要显式配置的条件：生产者 acks 级别、副本同步策略、消费者手动提交位移的时机。只要有一环用了默认值或者图省事关掉了同步确认，消息就可能丢。回答时最好直接把“不丢”拆成生产端、broker 端、消费端三段来说。详见[Kafka 副本、ISR 和 ACK 如何影响消息可靠性？](../high-performance/high-performance-kafka-replica-isr.html)。

## 4. `@Transactional` 加了就一定生效

| 错答                                            | 更稳答法                                                                                                 |
| ----------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| 方法上加了 `@Transactional`，事务就会按预期回滚 | 同类内自调用会绕过代理直接走 this 调用，注解直接失效；私有方法、非 public 方法、跨线程调用同样拿不到事务 |

Spring 事务靠 AOP 代理实现，本质上是“调用方要经过代理对象”这个前提。类内部 `this.otherMethod()` 这种自调用根本没经过代理，注解形同虚设，这是最常被问到、也最容易被现场证伪的坑。回答时最好直接举一个自调用失效的例子，再说怎么绕过去（抽到别的 Bean、用 `AopContext.currentProxy()`）。详见[Spring 事务为什么会失效？](../system-design/framework/spring-transaction-failure-cases.html)。

## 5. HashMap 线程不安全就是会死循环

| 错答                                     | 更稳答法                                                                                                             |
| ---------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| HashMap 并发扩容会死循环，所以线程不安全 | JDK 8 之后扩容死循环已经修复了，但并发 put 仍然可能导致数据覆盖、size 计数错乱，本质是“没有同步保护”，不是特指死循环 |

死循环是 JDK 7 头插法扩容时的经典问题，很多人一提 HashMap 线程不安全就只会讲这一个，但 JDK 8 改成尾插法后死循环基本不会出现了。真正持续存在的问题是并发写入时的数据丢失和状态不一致，这才是“线程不安全”更准确的落点。详见[HashMap 为什么线程不安全？JDK 8 后还会死循环吗？](../java/collection/java-collection-hashmap-thread-safety.html)。

## 6. 虚拟线程用了就能解决所有并发问题

| 错答                                                     | 更稳答法                                                                                                                             |
| -------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| 上了虚拟线程，线程池、锁这些都可以不用管了，性能直接起飞 | 虚拟线程解决的是“IO 密集型场景的线程创建成本”，遇到 `synchronized` 同步块或者本地方法调用还是会被 Pinning 钉住，退化成平台线程的开销 |

虚拟线程的收益集中在大量阻塞 IO 等待的场景，比如高并发网络调用；如果业务本身是 CPU 密集型，或者代码里大量用 `synchronized`，虚拟线程不但没有收益，还可能因为 Pinning 问题拖慢吞吐。回答时最好补一句“它不是免费午餐，得看阻塞点在哪”。详见[虚拟线程解决了什么问题？和平台线程什么关系？](../java/concurrent/java-concurrency-virtual-thread.html)和[Java 21 虚拟线程对传统线程池有什么影响？](../java/new-features/java-new-features-java21-virtual-threads.html)。

## 7. 加了限流系统就高可用了

| 错答                           | 更稳答法                                                                                         |
| ------------------------------ | ------------------------------------------------------------------------------------------------ |
| 系统做了限流，高可用就算搞定了 | 限流只解决“防止被打爆”这一层问题，高可用还需要熔断、降级、超时控制、多副本部署和故障转移一起配合 |

限流是高可用体系里的一个组件，作用是保护系统不被超出承载能力的流量打垮，但它管不了下游超时、也管不了单点故障。如果面试官问“高可用怎么做”，只答限流会显得理解得很单薄，最好把限流放进“流量防护、故障隔离、快速恢复”这个更大的框架里讲。详见[限流算法怎么选：固定窗口、滑动窗口、漏桶、令牌桶？](../high-availability/high-availability-rate-limiting.html)。

## 8. 数据量大了先分库分表准没错

| 错答                               | 更稳答法                                                                                                           |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------ |
| 表大了就应该分库分表，越早做越省事 | 分库分表会让跨分片查询、分布式事务、扩容迁移全部变复杂，能靠索引优化、读写分离、冷热分离撑住的场景，优先别引入分片 |

分库分表不是免费的性能提升，而是拿“单库容量”换来“跨库协调成本”，尤其是分片键选不好，后续跨库 join 和二次扩容会非常痛苦。回答时最好先说清楚“什么量级、什么场景才需要分片”，而不是一上来就讲怎么分。详见[分库分表之后查询、事务、扩容会变复杂在哪里？](../high-performance/high-performance-sharding-tradeoffs.html)。

## 9. JWT 无状态，所以比 Session 更安全更省事

| 错答                                                 | 更稳答法                                                                                                                                  |
| ---------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| JWT 不用存服务端，无状态、扩展性好，全面替代 Session | JWT 无状态换来的代价是主动失效很麻烦，令牌一旦签发，服务端很难在过期前强制让它失效，权限变更、踢下线都得额外设计黑名单或短有效期+刷新令牌 |

无状态是 JWT 的优点，也正是它的短板来源：Session 天然支持服务端主动删除会话，JWT 要做到同样的效果，得自己维护黑名单或者靠短 TTL 换刷新令牌，等于又把状态放回了服务端。回答时要把这个代价讲出来，不能只夸优点。详见[JWT 为什么不能无脑替代 Session？](../system-design/security/security-jwt-vs-session.html)。

## 10. 索引越多，查询越快

| 错答                           | 更稳答法                                                                                                                                  |
| ------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------- |
| 给字段都加上索引，查询肯定更快 | 索引会拖慢写入（每次 insert/update 都要维护索引结构），还占用额外存储，而且索引选得不对根本用不上，比如最左前缀没对齐、字段上做了函数计算 |

索引是拿写性能和存储空间换查询性能，不是越多越好；而且很多索引因为查询写法不对根本没生效，比如 `WHERE YEAR(create_time) = 2024` 这种在索引列上做函数计算就会导致索引失效。回答“怎么设计索引”时，最好同时讲清楚“什么情况索引会失效”，这两块是配套的。详见[索引是怎么设计和使用的？](../database/mysql/mysql-index-design.html)和[索引为什么会失效？](../database/mysql/mysql-index-invalidation.html)。

## 11. 缓存穿透、击穿、雪崩说的是一回事

| 错答                                               | 更稳答法                                                                                                                                              |
| -------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- |
| 反正都是缓存失效导致数据库被打爆，统称缓存雪崩就行 | 穿透是查询根本不存在的数据、缓存和数据库都没有；击穿是某个热点 key 突然过期；雪崩是大批 key 同时失效或者 Redis 整体不可用，三者成因和治理手段都不一样 |

这三个词经常被混着用，但对应的修复方式完全不同：穿透靠布隆过滤器或者缓存空值，击穿靠热点 key 不过期或者加互斥锁重建，雪崩靠给过期时间加随机值、做多级缓存或者提升 Redis 自身的高可用。混为一谈的回答会让面试官觉得你没有真正拆解过这几个故障场景。详见[缓存雪崩、击穿、穿透怎么治理？](../database/redis/redis-cache-problems.html)。

## 12. 先更新数据库再删缓存，就等于强一致

| 错答                                                     | 更稳答法                                                                                                                   |
| -------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| Cache Aside 模式下先更新数据库再删缓存，就能保证数据一致 | 这只是工程上风险最低的最终一致方案，读请求和写请求交叉时仍然存在短暂不一致窗口，删除失败还得靠重试或者 MQ、binlog 订阅补偿 |

这条经常被当成“标准答案”背下来，但如果面试官追问“那它是不是绝对安全”，答“是”就直接翻车了。准确的说法是它把不一致的窗口压到最小，且比“先删缓存再更新数据库”更不容易让脏数据长期存在，但依然不是强一致，删除失败之后的补偿机制才是真正的重点。详见[如何保证缓存和数据库一致性？](../database/redis/redis-cache-consistency.html)。

## 13. 分布式事务随便选一个方案就行，反正都能保证一致

| 错答                                               | 更稳答法                                                                                                                                                                            |
| -------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 2PC、TCC、Saga、本地消息表随便挑一个用，效果差不多 | 2PC 是强一致但阻塞成本高、单点风险大；TCC 侵入业务代码但实时性好；Saga 靠补偿事务、适合长流程；本地消息表实现简单但依赖轮询，几种方案没有互相替代关系，得按一致性要求和业务侵入度选 |

分布式事务没有万能解，2PC 依赖协调者、容易长时间阻塞资源；TCC 需要业务自己实现 Try/Confirm/Cancel，改造成本高；本地消息表几乎是最简单的方案，但只适合最终一致要求不那么苛刻的场景。回答时最好按“这几种方案的取舍维度”来组织，而不是背下每种的流程就完事。详见[分布式事务怎么选：2PC、TCC、Saga、本地消息表？](../distributed-system/distributed-transaction-selection.html)。

## 14. 重试次数越多，系统就越稳

| 错答                                       | 更稳答法                                                                                                                       |
| ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------ |
| 调用失败就重试，多试几次总能成功，系统更稳 | 下游本来就快扛不住了，大量客户端同时重试会形成重试风暴，把本已脆弱的服务彻底打垮；重试要配合退避、次数上限、超时预算和幂等设计 |

重试是把“单次失败”摊到多次请求上去博概率，如果下游故障的根因是过载而不是偶发抖动，重试只会让请求量雪上加霜，形成级联雪崩。真正稳妥的做法是指数退避加抖动、给重试设置整体超时预算，并且确保被重试的操作是幂等的。详见[重试为什么可能放大故障？](../high-availability/high-availability-retry-storm.html)。

## 15. ConcurrentHashMap JDK 8 之后完全无锁

| 错答                                                   | 更稳答法                                                                                                                                          |
| ------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| JDK 8 的 ConcurrentHashMap 全部用 CAS 实现，不再加锁了 | 无冲突的 `put` 确实靠 CAS 完成，但一旦发生哈希冲突需要在链表或红黑树节点上操作，还是会对头节点加 `synchronized`，只是锁粒度从分段锁细化到了单个桶 |

JDK 7 用 Segment 分段锁，粒度是若干桶共享一把锁；JDK 8 改成了对单个桶头节点加 `synchronized`，配合 CAS 处理无冲突场景，锁的粒度更细但并没有完全消失。回答“为什么并发性能更好”时，讲清楚这一点比笼统说“不加锁了”更站得住。详见[ConcurrentHashMap 是怎么从分段锁演进到 CAS + synchronized 的？](../java/collection/java-collection-concurrenthashmap.html)。

## 16. G1 垃圾回收器没有停顿

| 错答                              | 更稳答法                                                                                                                        |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| G1 是低延迟回收器，几乎不会有停顿 | G1 只是能按目标停顿时间去规划回收范围，本质仍然是有 STW 的分代收集器，Mixed GC 选错 Region 或者跨代引用处理不好，停顿照样能拉长 |

G1 的核心卖点是“可预测的停顿时间模型”，通过把堆划分成多个 Region、按回收收益优先挑选 Region 来控制单次停顿在目标范围内，但它并不是无停顿回收器，遇到大对象分配失败、并发标记跟不上垃圾产生速度，一样会退化成长时间的 Full GC。把它说成“零停顿”会被立刻问出破绽。详见[G1 相比 CMS 改进了什么？](../java/jvm/jvm-g1-vs-cms.html)。

## 小结

这十六个点有个共同规律：答案本身没错，错在“说得太绝对”，少了一个前提条件或者一个例外场景。面试时与其把话说满，不如主动补一句边界——“这个方案在什么情况下会失效”，反而更显出你真正理解了原理，而不是背下了结论。真遇到拿不准的点，直接点上面的链接回去补细节，比临场硬撑一个满分答案要稳得多。
