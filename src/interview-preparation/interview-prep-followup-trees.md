# 高频题怎么准备追问树？

> 很多人栽的不是不会答，是只会答第一层。面试官问完定义，接着往下一层问，答案就开始编了。这篇不重复讲知识点本身，只画“问到第几层算真的懂”，配上站内能查的原文。

## 为什么要按"树"准备，不是按"知识点"准备

背知识点是背一句话，比如"线程池有 7 个参数"。但面试官不会满足于这一句，他会顺着往下追：参数怎么配、执行顺序是什么、拒绝了怎么办、项目里怎么调优。这是一条**追问链**，每往下一层，考察的深度都不一样。

所以准备的单位不该是"知识点"，而应该是"这条追问链我能接到第几层"。下面 6 棵树，每棵给 3-4 层，每层写清楚：面试官大概率会问什么、你答到哪一层算过关、原文链接在哪。

---

## 树 1：线程池 7 个参数

```
线程池怎么理解？
├─ 第1层：7个参数分别是什么，各自作用
├─ 第2层：任务提交后的执行流程（核心线程→队列→非核心线程→拒绝策略）
├─ 第3层：拒绝策略怎么选、为什么不推荐 Executors 工具类
└─ 第4层：项目里怎么估算核心线程数、怎么监控、Spring 里怎么配置
```

| 层  | 面试官可能问什么                                       | 答到哪一层算过关                                                                                                                                     | 链接                                                                              |
| --- | ------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| 1   | "线程池的 7 个参数是什么？"                            | 说全 7 个，且能讲出 `corePoolSize`/`maximumPoolSize`/`workQueue` 三个的作用，不只是背名字                                                            | [线程池 7 个参数怎么理解？](../java/concurrent/java-concurrency-thread-pool.html) |
| 2   | "任务进来之后怎么走的？"                               | 讲出"核心线程 → 队列 → 非核心线程 → 拒绝策略"这个顺序，并说清楚非核心线程执行完初始任务后为什么不会立刻销毁（还在循环 `getTask()`）                  | 同上                                                                              |
| 3   | "队列满了、线程也到上限了怎么办？4 种拒绝策略怎么选？" | 能按业务场景挑：核心业务用 `AbortPolicy`，允许反压用 `CallerRunsPolicy`；并能说出为什么不用 `Executors.newFixedThreadPool` 这类（无界队列 OOM 风险） | 同上                                                                              |
| 4   | "生产环境线程池怎么配、怎么监控？"                     | 讲出核心线程数按 CPU/IO 密集型估算或用 Little's Law，加上活跃线程数、队列堆积、拒绝次数这几个监控指标，还有业务隔离（不同业务用不同池）              | 同上                                                                              |

> 追问到第 4 层，其实已经绕到了"这道题在你项目里是怎么落地的"，这是拉开候选人差距的地方。

## 树 2：MVCC 与隔离级别

这两个话题经常绑在一起问，因为 MVCC 就是隔离级别的实现手段之一。

```
事务隔离级别怎么理解？
├─ 第1层：脏读/不可重复读/幻读的区别，四个隔离级别分别解决了什么
├─ 第2层：为什么 MySQL 默认是 RR 而不是标准的 RC
├─ 第3层：RR 下幻读是"很大程度解决"而非"彻底解决"，靠什么机制
└─ 第4层：MVCC 具体怎么实现（版本链 + ReadView + 可见性判断）
```

| 层  | 面试官可能问什么                          | 答到哪一层算过关                                                                                                                                                                   | 链接                                                                           |
| --- | ----------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| 1   | "不可重复读和幻读有什么区别？"            | 说清楚一个是"内容变了"（同一行 UPDATE），一个是"行数变了"（INSERT/DELETE），别混着说                                                                                               | [事务隔离级别怎么理解？](../database/mysql/mysql-transaction-isolation.html)   |
| 2   | "MySQL 为什么默认用 RR？"                 | 说出历史原因是主从复制在 `STATEMENT` binlog 下靠间隙锁保证一致性，同时补一句"现在 `ROW` 格式下这个包袱已经缓解，但默认值出于兼容性没变"                                            | 同上                                                                           |
| 3   | "RR 解决幻读了吗？"                       | 不能只说"解决了"，要拆成快照读（MVCC 挡住）和当前读（next-key lock 挡住），并点出这不是 100% 消灭，存在边界场景                                                                    | 同上                                                                           |
| 4   | "MVCC 具体怎么判断一行数据能不能被看见？" | 讲出三个隐藏字段（`DB_TRX_ID`/`DB_ROLL_PTR`/`DB_ROW_ID`）、ReadView 四要素（`m_ids`/`min_trx_id`/`max_trx_id`/`creator_trx_id`），并能带一个具体事务 ID 走一遍可见性判断的三种情况 | [MVCC 和 ReadView 是怎么工作的？](../database/mysql/mysql-mvcc-read-view.html) |

> 常见翻车点：把"RC 和 RR 的区别"答成"锁的多少"，其实唯一区别是 **ReadView 的生成时机**——RC 每次快照读都重新生成，RR 只在事务第一次快照读时生成一次。这一句话答出来，说明真的懂了，不是背的。

## 树 3：缓存与数据库一致性

```
如何保证缓存和数据库一致性？
├─ 第1层：目标是最终一致还是强一致，先分场景
├─ 第2层：为什么删缓存而不是更新缓存，为什么不能先删缓存再更新数据库
├─ 第3层："先更新数据库再删缓存"是否绝对安全，删除失败怎么补
└─ 第4层：延迟双删的局限、不同业务怎么分层落地
```

| 层  | 面试官可能问什么                                         | 答到哪一层算过关                                                                                                                         | 链接                                                                           |
| --- | -------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| 1   | "缓存一致性要做到什么程度？"                             | 先分场景：商品详情这类追求最终一致，库存余额这类要收敛缓存参与、别指望缓存兜底强一致                                                     | [如何保证缓存和数据库一致性？](../database/redis/redis-cache-consistency.html) |
| 2   | "为什么是删缓存而不是更新缓存？为什么不能先删再更新库？" | 能举反例：并发写场景下"更新缓存"谁后落地不确定，容易把新值覆盖成旧值；"先删缓存再更新库"会被并发读请求在中间窗口把旧值重新写回缓存       | 同上                                                                           |
| 3   | "先更新数据库再删缓存，就绝对安全了吗？"                 | 答"不是"，要说出仍存在的窗口（读请求在库已更新、缓存未删之间把旧值写回），以及删除失败的补偿手段：重试告警、MQ 异步补偿、binlog 订阅修正 | 同上                                                                           |
| 4   | "延迟双删了解吗？项目里具体怎么分层处理？"               | 知道延迟双删只是"压低概率"不是银弹（睡多久没有标准答案），并能按"读多写少/写多命中率优先/强一致场景"分别给出不同策略                     | 同上                                                                           |

> 面试里最容易被抓的坑：把"先更新数据库再删缓存"说成"能保证一致性"。准确说法是"比另一种顺序风险更低的最终一致方案"，这个措辞上的差别就是第 3 层的过关线。

## 树 4：MQ 不丢不重

不丢和不重复经常连着问，因为工程上这两者是互相牵制的：为了不丢，只能接受"至少一次投递"；接受了至少一次投递，就必然要处理重复。

```
MQ 如何保证消息不丢、不重复？
├─ 第1层：消息可能在哪三段丢（生产者/Broker/消费者）
├─ 第2层：生产者侧怎么保证发出去（发送确认、本地消息表/Outbox、事务消息的局限）
├─ 第3层：Broker 和消费者侧怎么保证（持久化刷盘、多副本确认、业务成功后再 ack）
└─ 第4层：不丢和不重复本质冲突，怎么用幂等兜住重复消费
```

| 层  | 面试官可能问什么                               | 答到哪一层算过关                                                                                                                                    | 链接                                                                                         |
| --- | ---------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| 1   | "MQ 怎么保证消息不丢？"                        | 不能只说"持久化、ack、重试"，要按生产者→Broker→消费者拆开说，否则面试官会立刻追问"那消费者那段呢"                                                   | [MQ 如何保证消息不丢？](../high-performance/high-performance-message-reliability.html)       |
| 2   | "业务已经落库了，消息还没发出去就宕机怎么办？" | 讲出本地消息表（Outbox）思路：业务表和待发送消息表同一个本地事务提交，后台任务扫描补发；顺带说清事务消息只解决"生产端一致"，不保证消费端一定成功    | 同上                                                                                         |
| 3   | "Broker 收到了就不会丢了吗？消费者呢？"        | 讲刷盘策略（同步/异步）和副本确认条件；消费者侧说清"业务处理成功后再 ack"这个顺序，反过来先 ack 再处理是最隐蔽的丢失                                | 同上                                                                                         |
| 4   | "那消息会不会重复消费？怎么办？"               | 承认重复消费是默认要接受的现实（rebalance、ack 丢失都会导致），转向业务幂等：区分消息级重复和业务级重复，只靠消息 ID 去重通常不够，要回到业务唯一键 | [MQ 如何处理重复消费和幂等？](../high-performance/high-performance-message-idempotency.html) |

> 这棵树答到第 4 层，其实已经自然接上了下一棵树（幂等设计），面试官很可能顺势往那边追问，提前把两篇内容串起来记，效率更高。

## 树 5：限流、熔断、重试怎么组合

这是最容易被问"背了一堆名词但讲不清楚顺序"的题。

```
限流、熔断、降级、重试怎么组合？
├─ 第1层：限流先定目标（保护什么、允不允许突发），4种算法怎么选
├─ 第2层：超时/重试/隔离/熔断/降级在调用链里各自的分工和顺序
├─ 第3层：重试为什么会放大故障（乘积效应），重试应该收敛在哪一层
└─ 第4层：重试和幂等必须一起设计，不然"重试"本身就是新的故障源
```

| 层  | 面试官可能问什么                                     | 答到哪一层算过关                                                                                                                                                                   | 链接                                                                                                         |
| --- | ---------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------ |
| 1   | "限流算法怎么选？"                                   | 先说清楚要保护的是入口、下游还是某个稀缺资源，再挑算法：粗粒度用固定窗口，接口限频用滑动窗口，对接外部配额用漏桶，大多数业务入口用令牌桶（允许突发）                               | [限流算法怎么选？](../high-availability/high-availability-rate-limiting.html)                                |
| 2   | "熔断、降级、隔离、超时、重试，这几个到底谁管什么？" | 一句话说清分工：超时负责及时松手、重试负责有限度再试、隔离负责圈住影响范围、熔断负责止血、降级负责给用户兜底；并能画出"入口限流→超时预算→隔离→调用→失败重试/熔断→降级"这条链的顺序 | [熔断、降级、隔离、超时、重试怎么组合？](../high-availability/high-availability-resilience-composition.html) |
| 3   | "为什么说重试可能把故障放大？"                       | 讲出乘积效应：三层链路每层都重试 3 次，最坏情况到最底层是 27 次调用；结论是重试要收敛在最懂失败语义的一层，不能每层都开                                                            | [重试为什么可能放大故障？](../high-availability/high-availability-retry-storm.html)                          |
| 4   | "重试的时候怎么保证不会把状态搞乱？"                 | 说出重试的前提是这个操作可幂等，否则重试等于对同一业务意图重复执行；进一步展开幂等键设计、去重表、状态机                                                                           | [接口幂等怎么设计？](../high-availability/high-availability-idempotency-design.html)                         |

> 第 2 层是这棵树的核心分水岭：很多人只会罗列名词，说不出"谁在前谁在后"。能画出那条调用链顺序图，基本就说明真的在项目里用过，不是背概念。

## 树 6（可选）：服务注册发现

网关和注册发现经常被放在一起问，但注册发现的追问链更深、更容易考出层次，这里展开注册发现，网关部分放在树末尾简单带一句。

```
服务注册发现解决什么问题？
├─ 第1层：硬编码 IP 的坑，注册中心解决的核心问题是什么
├─ 第2层：谁来判断实例"死了"（客户端心跳上报 vs 服务端主动探测）
├─ 第3层：摘除策略和保护阈值——为什么不能"心跳超时就立刻摘除"
└─ 第4层：AP 型 vs CP 型怎么选，网络分区下不同注册中心的真实表现
```

| 层  | 面试官可能问什么                                         | 答到哪一层算过关                                                                                                                                  | 链接                                                                                   |
| --- | -------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| 1   | "为什么不能配置文件里写死服务地址？"                     | 说出扩容、发布、宕机、迁移这几种场景下硬编码都会出问题，本质是"谁在提供服务"是动态的，注册中心把它变成了自动更新的目录                            | [服务注册发现解决什么问题？](../distributed-system/distributed-service-discovery.html) |
| 2   | "注册中心怎么知道一个实例挂了？"                         | 区分客户端心跳上报（Eureka/Nacos 临时实例）和服务端主动探测两种模式，并能举例说明业务线程卡死但心跳线程独立时会出现"假活"                         | 同上                                                                                   |
| 3   | "网络抖动的时候，是不是应该赶紧把心跳超时的实例都摘掉？" | 说出保护阈值（自我保护）机制：心跳数骤降时暂停摘除，代价是可能保留一些真正死掉的实例，靠负载均衡+重试兜底                                         | 同上                                                                                   |
| 4   | "Eureka、ZooKeeper、Nacos 怎么选？"                      | 能用"短暂返回过期数据"和"选主期间直接不可写"这两种代价做取舍判断，说出服务发现场景多数偏向 AP，同时点出 Eureka 已进入维护模式，新项目更常见 Nacos | 同上                                                                                   |

> 网关那条线（[API 网关做了什么？](../distributed-system/distributed-api-gateway.html)）更适合单独准备：核心是分清"路由转发"和"请求过滤"两类职责，以及为什么限流鉴权要放在网关而不是各业务服务里自己写一遍。如果面试官把网关和注册发现放一起问，通常是想看你是否知道网关会动态拉取注册中心的服务列表，两者不是割裂的。

## 怎么用这几棵树

- 复习顺序不是从上往下背，而是先确认自己能不能独立说出第 1、2 层，再对着表格检查第 3、4 层有没有漏。
- 每棵树的最后一层通常都指向"这东西在真实项目里怎么配、怎么出问题、怎么补"，这是最容易被追问倒、也最容易和别人拉开差距的地方，值得多花时间。
- 树与树之间是有关联的：MQ 不丢不重接幂等设计，重试放大故障接幂等设计，缓存一致性也会被追问到"删除失败要不要重试"——把这些链路串起来记，比孤立背 6 篇文章效率更高。

## 小结

1. 高频题要准备成 3-4 层追问树，而不是只会开场白。
2. 第 1-2 层过关靠定义和主流程，第 3-4 层靠边界、配置和线上事故。
3. 树与树会交叉：重试接幂等，MQ 接一致性，网关接限流。
4. 复习时先自测能否独立讲前两层，再补后面的链路。
5. 最后一层尽量落到“你们线上怎么配、出过什么问题”。
