# 项目经验怎么用 STAR 讲清楚？

> 同一个项目，有人讲三分钟就让面试官追着问，有人讲十分钟对方还是没听出重点在哪。差别不在项目本身，在讲法。

## STAR 不是格式，是"面试官到底想验证什么"

很多人把 STAR 理解成一个填空模板：情境写一段、任务写一段、行动写一段、结果写一段，四段拼起来就是"用了 STAR"。这么理解会导致一个后果——你讲得很完整，但面试官听完还是不知道该往哪追问，因为每一段都是平的，没有能抓住的点。

真正有用的理解是：**STAR 的每一步都对应面试官心里的一个疑问**。

| 阶段              | 面试官在想什么                             | 你要交代什么                                      | 常见漏洞                                               |
| ----------------- | ------------------------------------------ | ------------------------------------------------- | ------------------------------------------------------ |
| Situation（情境） | 这个背景是真的吗？规模够不够看出你的水平？ | 业务场景、系统规模、当时的痛点是什么触发的        | 讲成"我们公司是做电商的"，没有任何量级信息             |
| Task（任务）      | 这件事具体是谁的责任？边界在哪？           | 你被分配/主动认领的具体目标，和别人的任务怎么分工 | 说"我们要优化性能"，听不出这是不是你负责的那部分       |
| Action（行动）    | 你是怎么想的？为什么选这条路不选那条？     | 你做的判断、取舍、排除掉的方案、遇到的具体困难    | 直接跳到"然后我们用了 XX 中间件"，中间的推理过程全没了 |
| Result（结果）    | 有没有验证？效果能不能量化？               | 具体指标的前后对比，以及这个结果怎么衡量出来的    | "效果很好""性能提升明显"，没有一个数字                 |

这张表里最容易被忽略的是 Result 那一行的最后一列——**没有指标的 Result 等于没讲**。"响应时间从 800ms 降到 120ms""P99 从 2.3s 降到 400ms""故障恢复时间从 40 分钟压缩到 8 分钟"，这些才是能被记住、能被追问的句子。面试官听到"性能提升明显"，唯一能问的下一句就是"提升了多少"，而你答不上来的瞬间，前面三段的可信度会一起打折扣。

如果实在拿不到精确数字（比如项目已经过去很久，或者当时没做压测留档），也不要空着，退而求其次给一个有依据的区间，并且诚实地说明这个数字是怎么估的："大概是 3-4 倍，这是根据当时慢查询日志里 TP99 的变化估算的，不是精确压测数据"。这比编一个假装精确的数字更安全——面试官一追问数据来源，编造的马上露馅。

## 3-5 年经验，讲的是"你的判断"，不是"我们团队"

工作 1-2 年的候选人讲项目，讲"团队做了什么"没问题，因为面试官对这个阶段的预期就是"参与、执行、学习"。但到了 3-5 年，预期完全变了：面试官想确认的是**你能不能在没人告诉你怎么做的情况下，自己拿主意**。这时候如果通篇都是"我们团队""我们讨论后决定"，面试官听完唯一的感觉是——这人可能只是执行者，独立判断能力没验证过。

对比着看差别在哪：

| 场景     | 团队背景型讲法（弱）                  | 主导判断型讲法（强）                                                                                                                                                                          |
| -------- | ------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 缓存选型 | "我们团队讨论后决定引入 Redis 做缓存" | "我评估了本地缓存和 Redis 两个方向，因为这个数据要跨 3 个实例共享，本地缓存会有一致性问题，所以选了 Redis；但为了防止 Redis 挂了导致穿透，我又加了一层本地缓存兜底"                           |
| 接口超时 | "我们发现接口很慢，就加了索引"        | "我先用慢查询日志定位到是这条 SQL 没走索引，加索引前我评估了写入频率——这张表每天有 200 万次写入，加复合索引会增加写放大，所以我选了覆盖索引而不是普通索引，把回表的开销也省了"                |
| 服务拆分 | "我们把单体拆成了微服务"              | "我判断这个模块该不该拆的标准是发布频率和故障半径——这个模块每周都要单独发版，还老因为改动影响到订单模块，所以我主导把它拆出来，代价是多了一次网络调用，我们评估过延迟增加在 5ms 以内可以接受" |

看出规律了吗？弱的讲法里，主语永远是"我们"，动词永远是"发现问题、做了决定"这种笼统词；强的讲法里，主语是"我"，而且交代了**排除掉的选项**和**为什么排除**。这才是"取舍"两个字的真正含义——取舍不是"选了 A"，而是"A 和 B 我都想过，B 在什么条件下会更差，所以我选了 A"。

如果你的项目里确实有一段是纯执行角色（比如刚入职时接手的老代码），坦率地说清楚"这部分是我接手维护，做的判断集中在后续的这次改造上"，好过硬把执行经历包装成主导经历——面试官问两个追问就能听出来是不是真主导过。

## 每个项目要准备的六块内容

面试前不要临场组织语言，提前把每个项目按下面六块整理成文字稿，逼自己想清楚，讲的时候才不会卡壳：

| 块           | 要写清楚什么                                              | 检验标准                               |
| ------------ | --------------------------------------------------------- | -------------------------------------- |
| 背景         | 业务是做什么的、系统大概什么规模（QPS、数据量、团队人数） | 别人问"多大规模"你能立刻报数字         |
| 你的职责     | 你在这个项目里具体负责哪一块，边界在哪                    | 能说清楚"这块是我做的，那块是别人做的" |
| 技术选型     | 关键组件为什么选它，不选别的会怎样                        | 能讲出至少一个被你否掉的备选方案       |
| 难点（2 个） | 遇到的最有含金量的两个技术难题，每个都要能画图讲清楚      | 面试官打断问细节，你能立刻往下展开     |
| 指标         | 这个项目/这次改造前后的量化对比                           | 数字精确到具体数量级，不是"提升了很多" |
| 复盘         | 现在回头看，哪里做得不够好，如果重来会怎么改              | 敢说出项目里真实存在的缺陷             |

这里额外说一下"难点（2 个）"和"复盘"这两块，因为最容易被略过。

**为什么是 2 个难点，不是列一堆**：面试时间有限，你如果罗列 5 个"难点"，每个都讲得很浅，面试官记不住任何一个；只挑 2 个真正有技术含量的，讲透——包括当时的错误尝试、卡住的地方、最后怎么解决的——反而更容易让人记住"这人是真做过"。选难点的标准很简单：这个问题网上搜不到现成答案，需要你自己分析定位的，才算数。"配置了一下线程池参数"不算难点，"线上偶发超时，最后定位到是线程池队列积压导致的雪崩，还得排除是不是下游慢了"才算。

**为什么一定要有复盘**：讲项目时通篇"我们做得很好"，反而会让面试官起疑——真实的工程项目不可能没有遗憾。主动说出"如果重来我会怎么改"，一是显得诚实、有反思能力，二是能提前把面试官想问的"这方案有什么问题"堵掉。比如："这次缓存一致性方案用的是延迟双删，能覆盖大部分场景，但极端情况下还是有短暂不一致窗口，如果重做我会考虑接入 binlog 订阅来兜底，当时没做是因为工期紧、这个窗口业务能接受。"

## 反例 vs 正例：技术选型到底怎么讲

这是最容易翻车的地方。几乎所有人简历上都写着"熟悉 Redis""使用 Kafka 处理异步消息"，但一问深入就现原形。看下面这组对比：

| 维度     | 反例（只说用了什么）                  | 正例（讲透为什么、边界、坑、数据）                                                                                                                                                                                                                                                                                                                           |
| -------- | ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 缓存     | "我们用 Redis 做缓存，提升了系统性能" | "订单详情页读多写少，DB 直接扛不住大促流量，我们加了 Redis 缓存，TTL 设了 30 分钟；踩过的坑是大促当天 QPS 突增，缓存过期时间集中在同一批，导致缓存雪崩打穿 DB，后来给 TTL 加了随机偏移量；缓存命中率从没有缓存时的 0 提到了 96%，DB 的 QPS 从 1.2 万降到 800"                                                                                                |
| 消息队列 | "我们用 Kafka 处理异步消息"           | "订单创建后要同步给库存、积分、短信三个下游，同步调用一个变慢会拖垮整体，所以改成发 Kafka 消息异步处理；选 Kafka 不选 RabbitMQ 是因为这条链路对吞吐要求高、对顺序没有强要求；踩过的坑是消费者重启时会重复消费，后来在消费端加了基于订单号的幂等表；改造后下单接口的 P99 从 900ms 降到 180ms"                                                                 |
| 限流     | "我们做了限流保护接口"                | "大促前压测发现下游库存服务在 QPS 超过 3000 后响应时间陡增，所以在网关层加了令牌桶限流，阈值按照压测出的拐点打了 8 折设为 2400；限流之后要考虑限流不生效时熔断怎么兜底，这块可以参考[限流算法怎么选](/high-availability/high-availability-rate-limiting.html)里讲的固定窗口和令牌桶的差别——我们一开始用固定窗口，边界突刺的问题很明显，后来换成令牌桶才解决" |

正例和反例的本质区别，是正例里永远有"为什么选它、它的边界在哪、踩过什么坑、数字是多少"这四样东西中的至少三样。反例只回答了"用了什么"，正例回答的是"这个技术在这个场景下具体解决了什么问题、付出了什么代价"。

如果准备项目时发现自己对某个组件只能讲出"用了什么"，说明这块知识没吃透，建议回去补对应的原理文章——比如讲 MQ 选型和积压排查可以看[消息队列进阶怎么处理重复消费](/high-performance/high-performance-message-backlog.html)，讲限流和熔断降级组合可以看[限流算法怎么选](/high-availability/high-availability-rate-limiting.html)，把原理和自己项目里的具体配置对上号，讲的时候才有底气往深处答。

## 准备 3 个故事：性能优化、稳定性事故、架构改造

面试官问项目时，几乎离不开这三类。提前把每一类准备一个能完整讲 5 分钟的故事，比临场从项目里现挑要稳得多。

### 故事一：性能优化类

| STAR | 内容示例                                                                                                                                                                                                                                                                         |
| ---- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| S    | 商品详情页大促预热期间，接口平均响应时间从 200ms 涨到 1.5s，用户投诉页面卡顿                                                                                                                                                                                                     |
| T    | 我负责把详情页接口的 P99 压回 500ms 以内，不能新增机器（预算已锁定）                                                                                                                                                                                                             |
| A    | 先用链路追踪定位到耗时集中在一次跨库的商品属性联表查询；分析后发现属性数据基本只读、变更频率极低，适合做成本地缓存 + Redis 两级缓存；本地缓存解决大促期间 Redis 网络抖动的风险，Redis 解决多实例间的一致性；缓存更新用的是数据变更时发消息主动失效，而不是定时刷新，避免过期窗口 |
| R    | P99 从 1.5s 降到 380ms，详情页接口的 DB 查询量下降了 92%，大促期间没有再出现因为这个接口导致的报警                                                                                                                                                                               |

讲这类故事时，重点不在"加了什么缓存"，而在**你怎么定位到瓶颈**——是靠链路追踪、慢查询日志、还是压测火焰图，这部分体现的是排障方法论，比结果本身更值钱。这块建议提前吃透可观测性怎么帮你定位问题，比如[日志、指标、链路追踪怎么分工](/high-availability/high-availability-observability-pillars.html)里讲的三支柱怎么配合，讲的时候能顺带带出你对可观测体系的理解，加分不少。

### 故事二：稳定性事故类

| STAR | 内容示例                                                                                                                                                                                                                                                                                                          |
| ---- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| S    | 凌晨 2 点线上支付回调接口大面积超时，故障持续了 25 分钟，期间订单状态更新延迟，客服接到大量催单                                                                                                                                                                                                                   |
| T    | 作为值班兼这块代码的主要维护者，我要先止血恢复服务，再定位根因并给出根治方案                                                                                                                                                                                                                                      |
| A    | 先看监控确认是下游银行回调接口变慢，不是自己服务的问题；当时的重试策略是失败后固定间隔重试 3 次，下游一慢就触发大量重试，形成重试风暴反而加重下游压力；先临时调大超时阈值+暂停自动重试止血，服务恢复后复盘发现根因是重试没有退避策略、也没有熔断保护；后续加了指数退避 + 熔断器，下游异常时快速失败而不是攒着重试 |
| R    | 故障平均恢复时间（MTTR）从这次的 25 分钟，通过后续预案演练压缩到后续同类故障的 6 分钟以内；重试风暴类告警上线半年内没有再触发                                                                                                                                                                                     |

这类故事的核心是"止血和根治要分开讲"——先讲你怎么快速恢复服务（这是应急能力），再讲怎么找到根因并且从根上改掉（这是深度）。很多人只讲了"我们重启了服务就好了"，这只是止血，没有根治，面试官会追问"那后来还犯吗"。重试策略这块背后的原理，建议对照[重试为什么可能放大故障](/high-availability/high-availability-retry-storm.html)讲清楚指数退避和熔断怎么配合，讲事故故事时顺带体现你对这块原理的掌握。

### 故事三：架构改造类

| STAR | 内容示例                                                                                                                                                                                                                                                                                                                                               |
| ---- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| S    | 单体应用里的营销模块因为活动频繁，每周都要单独发版，但和主流程耦合在一个进程里，发版一次要影响整个下单链路重启                                                                                                                                                                                                                                         |
| T    | 我主导评估并推动把营销模块从单体里拆出来，减少发版对主流程的影响，同时不能引入过多的分布式复杂度                                                                                                                                                                                                                                                       |
| A    | 先判断这个模块该不该拆，用的标准是发布频率和故障半径而不是"微服务更先进"这种理由；确认要拆之后，评估了拆分后的服务间调用方式，本地方法调用改成 RPC 会增加一次网络往返，做了压测确认延迟增加在可接受范围（15ms 以内）；数据一致性上，原来是本地事务，拆开后跨服务操作改成了本地消息表保证最终一致；上线用的是灰度，先切 5% 流量观察一周没问题再逐步放量 |
| R    | 营销模块的发版频率从原来拖累主流程一起重启，变成了独立发版，主流程的月度发版次数减少了 60%；一次营销活动的紧急下线，从原来要等主流程发版窗口，压缩到 10 分钟内独立完成                                                                                                                                                                                 |

架构改造类最容易讲成"我们把系统拆成了微服务"这种空话。真正有含金量的是**你判断该不该拆的标准**、以及**拆分后新增的复杂度你怎么解决**——网络调用、数据一致性、灰度发布，这几个点面试官几乎必问，提前想清楚就不会被问懵。

## 容易踩的坑

- **只背故事的"结果"，中间的判断过程一带而过**：面试官一句"为什么不用另一个方案"就能戳破，讲故事前先把"排除掉的选项"也想好台词。
- **指标编得太圆，经不起追问**：如果说"提升了 3 倍"，要能答上来"这个 3 倍是怎么测出来的、测试环境还是生产环境、样本量多大"，答不上来就老实说是估算。
- **难点选得没有技术含量**：把"写了个定时任务"当难点讲，面试官不会觉得你有深度，反而觉得你对自己项目的技术含量判断力不够。
- **复盘只讲优点不讲缺陷**：完全正面的复盘等于没复盘，真实项目一定有遗憾，讲出来才可信。
- **三个故事讲成一个类型**：如果性能优化、事故、架构改造三类故事都长得差不多（都是"发现问题-加个东西-变快了"），说明故事库单一，面试官换个问法就会发现你没有别的素材可讲。

## 小结

1. STAR 的灵魂是可验证的 Result，以及你做的判断，而不是组件名单。
2. 每个重点项目准备一页卡：职责、取舍、两个难点、指标、追问点。
3. 尽量覆盖性能、稳定性、架构改造中的至少两类故事。
4. 口述把“我”放回主语，并主动讲代价与边界。
5. 用项目反推知识复习，比按目录盲刷更有效。
