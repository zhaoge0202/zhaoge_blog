# 行锁到底怎么加？间隙锁和临键锁怎么理解？

> 行锁加在索引上，基本单位是「前开后闭」的 next-key lock；当能精确定位、不会引发幻读时，它会退化成记录锁或间隙锁。把这条主线抓住，剩下的全是推导。

这篇是 InnoDB 加锁规则的集大成篇，规则确实多，但背后只有一条逻辑：**RR 隔离级别下加锁的唯一目的是防幻读，能用更小的锁防住，就用更小的锁。** 下面所有规则都从这条推。

本文所有讨论都基于 **InnoDB 引擎 + 可重复读（RR）隔离级别**。RC 下没有间隙锁，行锁只有记录锁一种，要简单得多。不同小版本规则可能有细微差别，本文以 MySQL 8.0 的行为为准。

## 先立三个基本认知

这三点是后面一切推导的地基，必须先吃透。

**① 行锁锁的是索引记录，不是数据行。** 你 `update ... where id = 1`，锁不是盖在「那一行数据」上，而是盖在主键索引 B+ 树里 `id = 1` 这个索引项上。这就解释了一个反直觉的现象：where 不走索引时，会变成对所有索引项加锁，等于锁全表（后面专门讲）。

**② 加锁的基本单位是 next-key lock（临键锁），它是前开后闭区间 `(a, b]`。** 临键锁 = 记录锁 + 间隙锁的组合，既锁住区间右端那条记录本身，又锁住它前面的间隙。这里要分清三种锁的边界：

| 锁类型               | 区间              | 锁住什么             | 防什么           |
| -------------------- | ----------------- | -------------------- | ---------------- |
| 记录锁 Record Lock   | 单点 `[b]`        | 一条记录本身         | 防这条被改/删    |
| 间隙锁 Gap Lock      | 前开后开 `(a, b)` | 中间的缝隙，不含两端 | 防往缝里插入     |
| 临键锁 Next-Key Lock | 前开后闭 `(a, b]` | 缝隙 + 右端记录      | 又防插入又防改删 |

**③ 在能「精确定位」时，next-key lock 会退化为记录锁或间隙锁。** 这是整篇文章的核心钩子。所谓退化，就是 MySQL 判断「只用更小的锁就足以防住幻读」时，主动把临键锁缩小成记录锁（去掉间隙部分）或间隙锁（去掉记录部分）。后面每条规则我都会顺手说一句它为什么退化，你就不用死记了。

另外补一个常被忽略的兼容性事实：**间隙锁之间是相互兼容的**。两个事务可以同时持有相同甚至重叠的间隙锁，互不阻塞，因为间隙锁的唯一目的就是「禁止插入」，纯防御性，不存在冲突。但插入意向锁会和间隙锁冲突——这是死锁的一大根源，本篇末尾点一下，细节留给下一篇。

## 实验用的表和数据

后面所有例子都用这张表，请记住它：

```sql
CREATE TABLE `user` (
  `id`   bigint      NOT NULL AUTO_INCREMENT,
  `name` varchar(30) NOT NULL,
  `age`  int         NOT NULL,
  PRIMARY KEY (`id`),
  KEY `index_age` (`age`) USING BTREE
) ENGINE=InnoDB;
```

- `id`：主键索引（唯一索引）
- `age`：普通索引（非唯一索引，二级索引）
- `name`：普通列，没索引

表里的数据（按 id 排好）：

| id  | name   | age |
| --- | ------ | --- |
| 1   | 路飞   | 19  |
| 5   | 索隆   | 21  |
| 10  | 山治   | 22  |
| 15  | 乌索普 | 20  |
| 20  | 香克斯 | 39  |

注意：**id 不连续**（中间缺 2、3、4、6…），这正是间隙锁有戏可演的地方。另外 age 列如果按值排序是 19、20、21、22、39，二级索引树是按这个顺序存的。

## 唯一索引等值查询

唯一索引（这里就是主键 id）做等值查询，只看一件事：**这条记录在不在表里。**

**记录存在 → 退化为记录锁（只锁这一条）。**

```sql
begin;
select * from user where id = 1 for update;
```

id = 1 存在。事务在主键索引 `id = 1` 上加的是 **X 型记录锁**，锁住范围就是 `id = 1` 这一个点。其他事务改它、删它都会被阻塞；但别人插入 id = 1 不会被这把锁阻塞，而是直接撞主键冲突报错。

为什么能退化成记录锁、不需要间隙？因为唯一索引保证 id = 1 只有一条，别人插不进第二条 id = 1（主键冲突），删又被记录锁挡着，前后两次查 id = 1 结果集不会变，幻读自然防住了，要间隙锁干嘛。

**记录不存在 → 退化为间隙锁（锁住它本该在的那个间隙）。**

```sql
begin;
select * from user where id = 2 for update;  -- 表里没有 id=2
```

查不到。这时在「第一条大于 2 的记录」即 id = 5 的索引上，加一把**间隙锁，范围 `(1, 5)`**（左边界是 5 的上一条记录 id=1，右边界是 5）。

| 别人想干的事          | 结果                          |
| --------------------- | ----------------------------- |
| 插入 id = 2、3、4     | 阻塞（落在间隙里）            |
| 插入 id = 1 或 id = 5 | 主键冲突报错（不是被锁挡）    |
| 删除 / 修改 id = 5    | 不阻塞（间隙锁不含 5 这个点） |

为什么是间隙锁不是临键锁？因为你查的是 id = 2，只要保证「id = 2 这个位置不会被别人插进来」就够了，id = 5 这条记录被删被改跟你的结果集没关系，没必要把 5 也锁死。所以只锁缝、不锁点，退化成间隙锁。也别想着给「不存在的 id = 2」加记录锁——锁是加在索引上的，不存在的记录在索引里没有对应项，无处可加。

## 唯一索引范围查询

范围查询的总规则是：**沿索引扫描，每扫到一条就加 next-key 锁，但首尾两端可能退化。** 把 `>`、`>=`、`<`、`<=` 拆开看。

### 大于 / 大于等于

`select * from user where id > 15 for update;`

扫描从第一条大于 15 的记录 id = 20 开始：

1. id = 20：不是等值命中，加临键锁 `(15, 20]`
2. 继续往后到「最大记录之后」的特殊记录 supremum，加临键锁 `(20, +∞]`
3. 停止

结果：锁住 `(15, 20]` 和 `(20, +∞]`。别人插 16/17/18/19、改删 20、插任何大于 20 的值，全被挡。`> 15` 这里没有等值条件，所以第一条 20 不退化。

`select * from user where id >= 15 for update;`

`>=` 多了个「等于 15」的等值味道，差别就在第一条：

1. id = 15：等值命中，临键锁**退化成记录锁**，只锁 id=15 这一点
2. id = 20：临键锁 `(15, 20]`
3. supremum：临键锁 `(20, +∞]`
4. 停止

记一句话：**`>=` 时，作为下界的那条等值记录退化为记录锁**（因为它要被包含在结果里，但它前面那段间隙不需要锁，别人在 15 之前插入不影响 `id >= 15` 的结果集）。

### 小于 / 小于等于

这边的退化发生在**扫描终止的那条记录**（第一条不满足条件的记录）上，要分「条件值在不在表里」。

`select * from user where id < 6 for update;`（6 不在表里）

1. id = 1：临键锁 `(-∞, 1]`
2. id = 5：临键锁 `(1, 5]`（5 满足 <6，继续）
3. id = 10：第一条不满足 <6 的记录，临键锁**退化成间隙锁** `(5, 10)`，停止

结果：`(-∞, 1]`、`(1, 5]`、`(5, 10)`。终止记录 10 退化成间隙锁——因为你查的是 < 6，10 这条记录本身改不改、删不删跟结果集无关，只需要挡住 6/7/8/9 别插进 (5,10) 这段缝就行。

注意：6 既然不在表里，那 `< 6` 和 `<= 6` 加的锁**完全一样**（都扫到 10 退化成间隙锁），因为有没有「等于 6」对实际扫描没影响。

`select * from user where id <= 5 for update;`（5 在表里）

1. id = 1：临键锁 `(-∞, 1]`
2. id = 5：临键锁 `(1, 5]`
3. 唯一索引不会有第二个 5，停止

结果：`(-∞, 1]`、`(1, 5]`。这里终止记录 5 的临键锁**不退化**——因为是 `<=`，5 本身在结果集里，要保护它不被删改，必须连记录带间隙一起锁。

`select * from user where id < 5 for update;`（5 在表里）

1. id = 1：临键锁 `(-∞, 1]`
2. id = 5：第一条不满足 <5 的记录，临键锁**退化成间隙锁** `(1, 5)`，停止

结果：`(-∞, 1]`、`(1, 5)`。`< 5` 不包含 5，5 这条记录跟结果集无关，只锁缝。

把 `<` / `<=` 的退化规律总结成一张表：

| 查询   | 条件值在表中？ | 终止那条记录的锁                 |
| ------ | -------------- | -------------------------------- |
| `< 5`  | 在             | 退化成间隙锁 `(1, 5)`            |
| `<= 5` | 在             | **不退化**，临键锁 `(1, 5]`      |
| `< 6`  | 不在           | 退化成间隙锁 `(5, 10)`           |
| `<= 6` | 不在           | 退化成间隙锁 `(5, 10)`，同 `< 6` |

一句话记忆：**`<` 的终止记录总退化、`<=` 仅在条件值不在表里时退化；其余被扫到的中间记录一律保持临键锁。**

## 非唯一索引等值查询

普通索引（age）做等值查询，跟唯一索引有两个关键不同：

1. **不具唯一性**，所以同一个 age 值可能有多条，必须一路扫描到「第一条不满足条件的记录」才停。
2. **二级索引和主键索引都会加锁**：扫到的二级索引记录加 next-key 锁，对应到满足条件那行的主键索引上加记录锁（回表时锁主键）。

`select * from user where age = 22 for update;`（age=22 存在，是 id=10 那行）

- 命中 age = 22：在二级索引上加临键锁 `(21, 22]`；同时回表给 id = 10 的主键索引加**记录锁**
- 继续扫到 age = 39（第一条不满足 =22 的记录）：临键锁**退化成间隙锁** `(22, 39)`，停止

| 索引         | 锁                | 含义                      |
| ------------ | ----------------- | ------------------------- |
| 二级索引 age | 临键锁 `(21, 22]` | 防改删 age=22、防部分插入 |
| 二级索引 age | 间隙锁 `(22, 39)` | 防插入 age=23~38          |
| 主键索引 id  | 记录锁 id=10      | 防改删这一行              |

关键点：**末尾那条 age=39 的临键锁退化成间隙锁**。为什么不退化到只剩 `(21,22]` 的临键锁就够？因为二级索引上 `(21, 22]` 这把锁挡不住「age=22, id=12」这种新记录的插入（二级索引按 age 排，age 相同再按 id 排，id=12 会插到 id=10 之后、id=20(age=39) 之前那个缝里）。只有再补上 `(22, 39)` 这把间隙锁，才能把 age=22 后面的缝堵死，彻底防住幻读。这就是为什么非唯一索引等值查询一定要扫到下一条并加间隙锁。

如果 age = 25 这种**不存在**的值：直接扫到第一条不满足的 age=39，退化成间隙锁 `(22, 39)`，因为没有命中行，所以**不给任何主键索引加锁**。

> 补充一个易错细节：performance_schema 里二级索引间隙锁的 `LOCK_DATA` 会显示两个值，如 `39, 20`，第一个是 age 右边界，第二个是对应的 id。能不能往 `(22,39)` 里插 age=22 或 age=39 的新记录，还得看新记录的 id 落在哪个缝——这是二级索引「age + id」共同定位带来的复杂度。面试一般不要求算到这个粒度，知道「插入成败取决于插入位置的下一条记录上有没有间隙锁」即可。

## 非唯一索引范围查询

规则比等值简单：**非唯一索引范围查询，next-key 锁不退化，扫到的每条二级索引记录都老老实实加临键锁。**

`select * from user where age >= 22 for update;`

- age = 22：临键锁 `(21, 22]`（**不退化成记录锁**，注意和唯一索引 `>=` 的区别）；回表锁 id=10 主键记录锁
- age = 39：临键锁 `(22, 39]`；回表锁 id=20 主键记录锁
- supremum：临键锁 `(39, +∞]`
- 停止

为什么 age=22 这条等值命中却不退化成记录锁（唯一索引 `>=15` 时退化了）？因为 age 不唯一，记录锁只能防改删、防不住插入，别人完全可以再插一条 age=22 的新记录造成幻读。所以必须保留间隙部分，用临键锁连缝带点一起锁。**唯一性是退化的前提，非唯一索引没这个前提，自然不退化。**

## 没有索引的查询：锁全表的元凶

前面都走了索引。如果 where 条件的列没索引（比如 `where name = 'xx'`），或者优化器算下来走全表扫描更划算，那就是全表扫描。

**全表扫描时，锁是在「遍历索引的过程中」逐条加的，不是只对返回结果加。** 既然要把整棵聚簇索引从头扫到尾，那每一条记录的索引项都会被加上 next-key 锁，连带 supremum，最终等于**把整张表锁死**。这期间除了普通 select（快照读不加锁），其他事务的增删改全部阻塞。

这正是 `update_index.md` 里那个线上事故：一条 `update ... where 某无索引列 = x` 直接锁全表，业务停摆。要强调两点：

- 别被「加了表锁」的说法误导。InnoDB 全程都是按索引项加行锁，只是把所有索引项都锁了，**效果等同表锁，但本质仍是 next-key 行锁**。源码层面没有真的去加一把表级写锁。
- where 带了索引列也不保证安全。**最终走不走索引由优化器决定**，如果它选了全表扫描，照样锁全表。

防护手段：

```sql
-- 1. 开启安全更新模式，拦截不带索引/limit 的危险 update/delete
SET sql_safe_updates = 1;
-- 此后 update 必须满足：where 带索引列，或带 limit，或两者都有
-- delete 必须同时带 where 和 limit

-- 2. 怀疑优化器选错执行计划时，强制走指定索引
update user force index(index_age) set name = 'x' where age = 22;
```

线上执行 update / delete / select ... for update 前，**先 EXPLAIN 确认走的是索引扫描而不是全表扫描**，这是基本纪律。

## 间隙锁 / 临键锁怎么防幻读

把上面的规则串起来看防幻读的逻辑就很清楚了：

- 幻读 = 同一事务前后两次查同样条件，结果集行数变了（多了或少了一行）。
- 当前读（`select ... for update`、update、delete）靠 **记录锁挡住「改/删」，间隙锁挡住「插入」**，两者合成的 next-key 锁就能把一个范围彻底冻住，让别的事务既删不掉里面的行、也插不进新行，结果集自然稳定。

`lock_phantom.md` 那个例子很典型：表里只有主键索引，事务 A 执行 `select ... where age > 20 for update`，因为 age 没索引走全表扫描，结果在主键索引上从 `(-∞,1]` 一路加到 `(9,+∞]`，整表冻结，事务 B 想删 id=2 都被阻塞——记录锁 + 间隙锁确实能防住「删除型」和「插入型」幻读。代价就是锁粒度大到离谱，反过来也印证了「给 where 列建索引」有多重要：建了 age 索引后，同样的查询只锁 age 索引上相关的几段，不会冻全表。

## RR 下幻读其实没有 100% 解决

很多人背「RR 用间隙锁解决了幻读」，这话不严谨。RR **很大程度上**避免了幻读，但有两个反例（来自 `phantom.md`），面试问深一点必须能说清。

**反例 (a)：快照读 + 对「本不可见」的记录做当前读写。**

```text
事务A: begin; select * from t where id=5;   -- 空，看不到（此时表里没有 id=5）
事务B: begin; insert id=5; commit;          -- 插入并提交
事务A: update t set name='x' where id=5;    -- 居然更新成功，影响 1 行
事务A: select * from t where id=5;          -- 现在看得见 id=5 了！幻读
```

原理：A 第一次快照读建了 Read View，本来看不到 B 后插的 id=5。但 A 的 update 是**当前读**，它把这条记录的隐藏列 trx_id 改成了 A 自己的事务 id，于是这条记录对 A「转正」可见，后续快照读就看见了。MVCC 的可见性在这里被自己的写操作绕过了。

**反例 (b)：先快照读 N 行，中途别人插入提交，再当前读变 N+1 行。**

```text
T1 事务A: select * from t where id>100;            -- 快照读，3 行
T2 事务B: insert id=200; commit;                   -- 插入并提交
T3 事务A: select * from t where id>100 for update; -- 当前读，4 行！幻读
```

原理：T1 的快照读靠 Read View 看不到 B 的插入；但 T3 的 `for update` 是当前读，读的是最新数据，于是多出一行。两次查询结果集不同，就是幻读。

**统一的规避办法**：事务一开始就用当前读把要操作的范围锁住——别先快照读再当前读。比如开事务后立刻 `select ... for update` 把范围锁死，间隙锁会挡住其他事务的插入，后面就不会出现「凭空多一行」。

## 用 data_locks 看实际加了什么锁

排障时最实在的手段，是直接查 MySQL 加了什么锁：

```sql
select * from performance_schema.data_locks\G;
```

看输出抓三个字段：

- `LOCK_TYPE`：`RECORD` 表示行级锁（注意，这里的 RECORD 是「行锁」的意思，不是「记录锁」），`TABLE` 表示表级锁（通常是意向锁）。
- `LOCK_MODE`：
  - `X` → next-key 锁（临键锁）
  - `X, REC_NOT_GAP` → 记录锁
  - `X, GAP` → 间隙锁
  - `X, INSERT_INTENTION` → 插入意向锁
- `LOCK_DATA`：临键锁/间隙锁时，它是锁范围的**右边界**；左边界要看 B+ 树里它的上一条记录。二级索引上可能显示两个值（如 `39, 20`），第二个是 id。

养成「先猜规则、再用 data_locks 验证」的习惯，加锁规则很快就内化了。

## 容易踩的坑

- **以为行锁锁的是数据行**。行锁锁的是索引项，这是「无索引查询会锁全表」的根本原因。
- **把退化规则记反**。唯一索引等值：存在→记录锁，不存在→间隙锁。`<` 终止记录总退化、`<=` 仅条件值不在表中时退化。
- **以为非唯一索引等值命中会像唯一索引那样退化成记录锁**。不会，它保留临键锁还要再加一段间隙锁，因为非唯一索引防不住「插入同值新行」。
- **以为非唯一索引范围查询也会退化**。不会退化，每条都是临键锁。退化是唯一索引的「特权」，前提是唯一性。
- **以为 where 带了索引就一定不锁全表**。优化器可能弃用索引走全表扫描，照样锁全表，要 EXPLAIN 确认。
- **以为间隙锁之间会互斥**。间隙锁纯防御、相互兼容，多个事务能同时持有同一间隙锁。真正冲突的是「间隙锁 vs 插入意向锁」，这才是死锁源头。
- **以为 RR 彻底没有幻读**。还有上面两个反例的漏洞，靠「事务开头就用当前读锁范围」规避。

## 小结

1. 行锁锁的是**索引记录**；基本单位是**前开后闭**的 next-key lock（记录锁 + 间隙锁）；能精确定位、不引发幻读时会**退化**为记录锁或间隙锁。
2. 唯一索引等值：**记录在→记录锁，记录不在→间隙锁**。范围查询里 `>=` 的下界退化为记录锁、`<` 的终止记录退化为间隙锁、`<=` 仅条件值不在表中才退化。
3. 非唯一索引等值：扫到第一条不满足的记录为止，末尾退化为间隙锁，命中行的主键索引另加记录锁；非唯一索引**范围查询不退化**。
4. where **不走索引（含优化器选全表）就锁全表**，用 `sql_safe_updates`、`force index`、上线前 EXPLAIN 防护。
5. RR 并未 100% 解决幻读：快照读后对不可见记录做当前读写、或先快照读再当前读，都会幻读；规避法是事务一开始就用当前读把范围锁死。
