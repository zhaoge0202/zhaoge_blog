# 表结构和字段设计怎么做？

> 表结构是数据库的地基。建表那一刻定下来的字段类型、是否可空、主键怎么选、用什么字符集，会跟着这张表的一生，事后想改往往要锁表、要迁数据，代价远比一开始想清楚大。

网上有一大堆「MySQL 设计规范」，动辄三四十条。我自己的看法是：这些条目不能当圣旨一条条背，得分清楚它们到底是「硬道理」还是「团队口味」。

- **硬道理**：违反了会真的影响性能或正确性。比如金额用 FLOAT 存会算错钱、字符集用 `utf8` 存不了 emoji、列上套函数会让索引失效。这些不分团队，谁踩谁倒霉。
- **团队口味**：主要是为了协作和维护统一，比如表名要不要小写下划线、临时表要不要加 `tmp_` 前缀。这些没有对错，团队定了就照做，但它们跟性能基本没关系，面试时也别把它们说得像性能优化。

下面我按主题来讲，重点说清楚每条背后的「为什么」。

## 字段类型：能小别大，能数值别字符串

字段类型的核心原则就一句话：**在能装下数据的前提下，选最小的那个类型**。

为什么纠结这点？因为 InnoDB 是按页（默认 16KB）来存数据的，字段越小，一页能塞下的行就越多，同样一次磁盘 IO 读进来的有效数据就越多，缓冲池（Buffer Pool）的命中率也越高。索引也一样，二级索引的叶子节点里会带主键，主键越小，每个索引项越小，整棵 B+ 树就越矮越紧凑。

具体几个常见选择：

**整型别动不动就上 BIGINT。** 状态位、是否启用这种 0/1 的字段，TINYINT（1 字节）足够；普通的计数、ID 用 INT（4 字节，无符号能到 42 亿）通常也够。只有当你确信行数会冲到几十亿、INT 装不下时才用 BIGINT（8 字节）。我见过不少表所有整型一律 BIGINT，单看一个字段差 4 字节没感觉，但一张几十列、几千万行的表累计起来，浪费的空间和 IO 是实打实的。

**非负数据用 UNSIGNED。** 自增 ID、年龄、IP 这些不可能为负的，加上 `UNSIGNED` 能把正数范围翻一倍（INT 从 21 亿变 42 亿）。不过这条要提醒一句：用了 UNSIGNED 之后，两个无符号数相减如果结果为负会直接溢出报错，业务上做差值计算要留意。所以它是个「锦上添花」的优化，不是必须。

**IP 能用整型就别用字符串。** `'255.255.255.255'` 用 VARCHAR 存要 15 个字节，转成整型只要 4 字节（INT UNSIGNED）。MySQL 自带 `INET_ATON()` / `INET_NTOA()` 做转换，存进去前转成整数、查出来再转回字符串。整数还连续、好比较、好建索引。补一句：这套函数只支持 IPv4，IPv6 得用 `INET6_ATON()` / `INET6_NTOA()` 配 `VARBINARY(16)`，现在不少系统得考虑 IPv6 了。

**金额必须用 DECIMAL，绝对不要用 FLOAT/DOUBLE。** 这是字段设计里最典型的「硬道理」。FLOAT、DOUBLE 是二进制浮点数，很多十进制小数（比如 0.1）在二进制里压根没法精确表示，存进去就是个近似值。最经典的现象：

```sql
-- 用 DOUBLE 算金额，会冒出诡异的尾差
mysql> SELECT 0.1 + 0.2;
+---------------------+
| 0.1 + 0.2           |
+---------------------+
| 0.30000000000000004 |
+---------------------+
```

对账时多出来个 0.0000...04，财务能跟你急。DECIMAL 是定点数，按十进制精确存储，`DECIMAL(10,2)` 就是最多 10 位、小数 2 位，算多少是多少。代价是它占空间、算起来比浮点慢一点，所以只在需要精确计算的地方（金额、汇率、计费）用，普通的物理测量值（温度、重量）不要求绝对精确，用浮点反而合适。

**VARCHAR 长度按实际需要给，别图省事一律 255。** VARCHAR 是变长的，实际占多少看内容，所以「定大了不浪费磁盘」这个直觉部分成立。但有个坑：MySQL 在做排序、建临时表、用内存做中间结果时，会按声明的**最大长度**预分配内存。你把一个其实最多 20 字符的昵称声明成 VARCHAR(255)，排序时它就按 255 来占内存，白白拉高内存消耗。所以长度该给多少给多少，手机号就 11、20，别拍脑袋 255。

**TEXT/BLOB 尽量拆出去或者干脆别用。** TEXT 能存到 64KB，BLOB 存二进制。它们的麻烦在于：内存临时表不支持这两种类型，一旦查询里带上它们做排序、分组，MySQL 只能落到磁盘临时表，性能直接掉一个台阶；而且 TEXT 列不能有默认值、只能建前缀索引。实务上的做法是把大文本垂直拆到单独的扩展表，主表只留常用字段，需要大文本时再按主键去关联查。

> 原文有一条「禁止在数据库存图片这类二进制大文件」，这个我同意——图片、视频该放对象存储（OSS、S3）或文件服务器，数据库只存个 URL。但要注意它说的是「大文件」，不是说 BLOB 完全不能用。存个几十字节的加密密钥、小的二进制指纹，BLOB 是合理的。别把「别存大文件」误读成「BLOB 是禁区」。

## NOT NULL：默认就该加上

> 除非确实有「这个值就是不知道」的业务语义，否则字段一律 `NOT NULL` 并给个 `DEFAULT`。

这是另一条很值得讲清楚的硬道理，但它常被讲得很玄，我说几个实在的点：

1. **NULL 会让你的判断逻辑变复杂。** SQL 里 NULL 不等于任何东西，连 `NULL = NULL` 都返回 NULL（不是 true）。判空得用 `IS NULL` / `IS NOT NULL`，写 `WHERE col != 1` 时还会把 NULL 的行漏掉——因为 `NULL != 1` 结果是 NULL 而非 true。这种「漏数据」的 bug 极其隐蔽。

2. **聚合函数对 NULL 的行为容易踩坑。** 最典型的是 `COUNT(列)` 会跳过该列为 NULL 的行，而 `COUNT(*)` 不会。`SUM`、`AVG` 也会忽略 NULL，导致 AVG 的分母变小、算出来偏大。这些不是 bug，是 SQL 标准定义的语义，但不熟的人很容易算错。

3. **存储和索引上有额外开销。** InnoDB 里可空字段需要额外的标记位记录「这一行该列是不是 NULL」，含 NULL 的列建索引也要多存信息。不过说句公道话——这点空间开销在现代硬件下其实很小，**真正的理由是前两条（逻辑正确性和可维护性），而不是省那几个字节**。网上很多规范把「省空间」吹成 NOT NULL 的主要好处，我觉得有点本末倒置。

实操建议：字符串字段默认 `''`，数值字段默认 `0`，状态字段给个明确的默认状态。这样应用层永远不用处理 NULL 分支，代码干净很多。

## 主键：老老实实用自增整型

InnoDB 是「索引组织表」，整张表的数据物理上就是按主键顺序排列、挂在聚簇索引的叶子节点上的。这个特性决定了主键的选法非常关键。

**为什么推荐自增整型（`BIGINT UNSIGNED AUTO_INCREMENT` 或 `INT`）：**

- 自增意味着**新数据总是往 B+ 树的最右边追加**，页基本是顺序写满再开新页，很少触发页分裂，写入效率高、空间紧凑。
- 主键值小且固定，所有二级索引的叶子节点里都带着主键值，主键越小，所有二级索引就越省空间。

**为什么不推荐 UUID / 字符串做主键：**

- UUID 是随机的，新插入的主键值东一个西一个，会插到 B+ 树中间，频繁触发**页分裂**和数据移动，写性能差、还会让页变得稀疏（碎片化）。
- UUID 是字符串，又长（36 字符）又占地方，每个二级索引都要背着这么一大坨，整体膨胀得厉害。

这条要补充一点：现在确实有「分布式 ID」的需求（分库分表后不能依赖单库自增），这时候会用雪花算法（Snowflake）这类**趋势递增**的方案。它的关键就在于「趋势递增」——保留了顺序性，规避了 UUID 随机插入的毛病。所以核心矛盾不是「数字 vs 字符串」，而是「有序 vs 随机」。MySQL 8 也支持有序的 UUID（`UUID_TO_BIN(uuid, 1)` 重排时间位），实在要用 UUID 也有补救办法。

另外，原文说的「不要用更新频繁的列做主键」「别用多列联合主键」都对——主键一旦变更，所有二级索引都要跟着改，代价很大。

## 字符集：直接上 utf8mb4

> 这条几乎没有讨论余地：新表的字符集统一用 `utf8mb4`，排序规则用 `utf8mb4_0900_ai_ci`（MySQL 8）或 `utf8mb4_unicode_ci`。

很多人以为 MySQL 里的 `utf8` 就是标准的 UTF-8，**这是个大坑**。MySQL 的 `utf8`（全名 `utf8mb3`）每个字符最多只存 3 个字节，而标准 UTF-8 里 emoji、部分生僻字、一些 CJK 扩展字符是要占 4 个字节的。结果就是：你用 `utf8` 的表，用户发个 😂 进来，轻则报错插入失败，重则被截断存成乱码。`utf8mb4` 才是「真·UTF-8」，最多 4 字节，emoji 和生僻字都能存。

代价几乎可以忽略——只有当字符真的需要 4 字节时才占 4 字节，ASCII 字符还是 1 字节。所以没有理由再用老的 `utf8`。

> 原文写的是「字符集统一使用 UTF8，需要存 emoji 时用 utf8mb4」。我得纠正一下：这个说法过时了。现在的默认建议就是**无脑 utf8mb4**，不用等到「需要 emoji」才换——因为你很难预判用户哪天会输入什么，而事后改字符集是要重建表的麻烦事。MySQL 8 的默认字符集本身就已经是 utf8mb4 了。

## 索引规范：少而精，优先联合索引

索引这块的规范，背后逻辑高度一致：**索引不是越多越好，加索引是拿写入和空间换查询。**

每加一个索引，所有的 INSERT / UPDATE / DELETE 都要顺带维护这个索引的 B+ 树，写入变慢、空间变大。而且查询时优化器要在一堆候选索引里挑，索引太多还会增加优化器选执行计划的开销。所以几条规范都是这个意思：

- **单表索引别太多。** 阿里规范给的是「单表不超过 5 个」，这个 5 是经验值不是硬上限，意思是「别滥建」，不是说第 6 个索引就一定错。该建还得建，关键是每个索引都得有真实的查询在用它。
- **优先建联合索引，而不是给每列单独建索引。** MySQL 一条查询绝大多数情况只用得上一个索引（虽然 5.6 后有「索引合并」优化，但效果远不如一个设计良好的联合索引）。所以与其给 `a`、`b`、`c` 各建一个单列索引，不如根据查询条件建一个 `(a, b, c)` 联合索引，配合最左前缀原则，一个顶仨。
- **区分度低的列别单独建索引。** 区分度 = `count(distinct col) / count(*)`。像「性别」这种就两三个值的列，单独建索引基本没用——优化器一看走索引还不如全表扫，干脆不用它。这种列适合放在联合索引里跟别的列搭着用，不适合挑大梁。
- **避免冗余索引和重复索引。** 已经有了 `(a, b, c)`，再单独建 `(a)` 或 `(a, b)` 就是冗余的——因为最左前缀原则下，`(a, b, c)` 已经覆盖了 `(a)` 和 `(a, b)` 的查询能力。重复索引（比如对主键又建了个普通索引）更是纯浪费。

联合索引的列序也有讲究，一般把**区分度高、最常用作等值查询条件**的列放左边。需要范围查询的列尽量靠右，因为范围条件之后的列用不上索引了（这部分展开是「索引失效」那篇的内容，这里不细讲）。

## 库表设计：单表别太宽，该冗余就冗余

**单表字段别太多。** 一张表几十上百个字段，问题是：一行数据太大，一页能装的行就少，IO 效率低；而且很多字段其实不常一起用。这时候可以做**垂直拆分**——把不常用的、或者特别大的字段（比如大段描述、配置 JSON）拆到副表，主表只留高频访问的核心字段，用主键关联。

**该反范式就反范式，别被三范式绑死。** 教科书讲数据库设计要满足第三范式，避免数据冗余。但严格遵守三范式意味着数据拆得很散，查个东西要 join 一堆表。实际开发里我们经常**故意冗余一点字段来换查询性能**——比如订单表里直接冗余一个「商品名称快照」，而不是每次都去 join 商品表。这不光是性能考虑，还有业务语义：商品名后来改了，订单里也该保留下单那一刻的名字。

冗余的代价是要自己保证一致性（商品改名了，要不要同步订单里的快照？通常不同步，因为那本来就是快照）。所以反范式要适度、要想清楚一致性怎么维护，不是随便复制字段。

**要不要分库分表，看单表行数。** 当单表数据涨到很大（常说几百万到上千万这个量级开始要警惕，但具体阈值跟行宽、查询模式强相关，没有死数字），DDL（改表结构）、备份、恢复都会变得很慢很痛苦。手段一般是：日志型数据做历史归档（冷数据搬走），业务数据做分库分表。这块跟「单表多少行算大」那篇是呼应的，这里不展开。

> 原文有「单表建议控制在 500 万行以内」。这个 500 万是个流传很广的经验数字，但别当成铁律——它源自早年硬件和某些场景的经验。一张窄表（几个 INT 字段）几千万行照样跑得很顺，一张宽表可能几百万就吃力了。真正的判断依据是 B+ 树高度、单行大小和你的查询模式，而不是一个固定的行数。面试时把它当「经验参考值」说，别说成「超过 500 万就必须分表」。

## SQL 写法里真正影响性能的几条

设计规范里夹了一堆 SQL 写法的约定，我挑出真正有性能/正确性影响的：

**别用 `SELECT *`，把要的字段列出来。** 理由实在：① 多读了用不上的列，浪费网络带宽和 IO，碰上 TEXT/BLOB 大字段尤其糟；② **用不上覆盖索引**——如果你只 select 索引里已有的列，查询可以只扫索引不回表（覆盖索引），但 `SELECT *` 强迫每次都回表拿全部列，速度差很多；③ 表加了字段后，`SELECT *` 返回的列会悄悄变化，可能搞坏上层代码。

**WHERE 里别对列做函数或运算。** 比如 `WHERE DATE(create_time) = '2026-01-01'`，对列套了函数，索引直接失效，变全表扫。改成范围写法 `WHERE create_time >= '2026-01-01' AND create_time < '2026-01-02'` 就能走索引。同理，隐式类型转换（`WHERE id = '111'`，id 是数字却传字符串）也会让索引失效。

**大批量写操作要分批。** 一次 UPDATE / DELETE 几十上百万行，会：① 在主从架构下造成严重主从延迟（从库要等主库整条执行完才开始追）；② 锁住大量行、撑出一个大事务，长时间阻塞别的请求，甚至占满连接数把整个库拖垮；③ row 格式 binlog 会因为这一大波改动产生海量日志。所以大批量操作要拆成小批次循环执行，每批几千行，给从库和其他事务喘息空间。

**join 别关联太多表。** MySQL 每多 join 一张表就多分配一块 join buffer，关联太多内存吃紧、还容易产生临时表。阿里规范建议 **join 不超过 3 张表**（原文写的是「最多 61 张、建议不超过 5 张」，这两个数字口径不同，3 是更保守的业务建议，记 3 这个数更稳）。关联列两边类型要一致，否则隐式转换又会让索引失效。

**用 EXPLAIN 验证，别凭感觉。** 觉得某条 SQL 慢，先 `EXPLAIN` 看它走没走索引、扫了多少行、有没有用临时表和文件排序（Extra 里的 `Using temporary` / `Using filesort` 是危险信号）。这是判断 SQL 好坏最直接的手段，比任何规范都管用。

**其他几条顺带提：** `IN` 代替 `OR`（OR 常用不上索引，但 IN 里别塞几百上千个值）；明确无重复时用 `UNION ALL` 而不是 `UNION`（省掉一次去重）；`ORDER BY RAND()` 别用（会把全表加载进内存排序，极耗资源）；子查询能改 join 就改（子查询的中间结果集没索引）。这些都是有实际依据的，不是凭空规定。

## 容易踩的坑

- **金额用了 FLOAT/DOUBLE。** 这是新手最常踩的坑，平时小金额看不出来，一对账就发现差了几分钱。建表时金额、价格、汇率一律 DECIMAL。
- **字符集用了 `utf8` 而不是 `utf8mb4`。** 上线后用户发 emoji 报错或乱码，再改就得重建表、迁数据。新表直接 utf8mb4。
- **拿 UUID/字符串当主键还以为没事。** 数据量小的时候没感觉，量一大写入性能和空间问题集中爆发，而且主键已经被外部引用，改起来要命。
- **把所有规范当圣旨。** 比如硬抠「单表必须 5 个索引以内」「必须 500 万行以内」，反而该建的索引不敢建、该归档的没归档。规范是参考，要理解背后的 why，结合自己的数据量和查询模式判断。
- **VARCHAR 一律 255。** 看着没浪费磁盘，但排序和临时表会按最大长度吃内存，宽表大查询时拖性能。
- **该加 NOT NULL 的字段留了可空。** 然后 `COUNT(列)` 漏数、`!=` 查询漏行，排查半天发现是 NULL 在捣鬼。

## 小结

- **字段类型选择是硬道理**：能小别大、能数值别字符串、金额必须 DECIMAL、字符集必须 utf8mb4——这几条踩了会真出问题，不是口味。
- **NOT NULL 默认就该加**，主要理由是逻辑正确性和可维护性（避免 NULL 在比较、聚合里捣乱），省空间只是顺带。
- **主键用自增整型**，本质是要「有序」以贴合聚簇索引、规避页分裂；分布式场景用趋势递增的雪花 ID，别用随机 UUID。
- **索引少而精、优先联合索引**，每个索引都要有真实查询在用；分库分表、单表行数上限这些都是经验参考值，不是死规定。
- **分清硬道理和团队口味**：命名风格、前缀约定这些跟着团队走就行，别当性能优化讲；真正影响性能的是类型、索引、NOT NULL、避免 SELECT \* 和列上套函数这些。
