# MySQL 有哪些锁？

> MySQL 的锁按"锁住多大范围"分成三层：全局锁锁整个库、表级锁锁一张表、行级锁锁一条记录（或一段区间）。把这三层搞清楚，锁这块就有了骨架。

很多人一上来就背"记录锁、间隙锁、临键锁、意向锁……"一堆名词，背完还是一团乱。其实只要抓住一条主线就清晰了：**锁是按粒度分层的**。粒度从大到小是全局 → 表 → 行，粒度越细，并发性越好，但实现也越复杂。InnoDB 比 MyISAM 强的一个关键点，就是它把锁做到了行级。

下面就按这三层来讲。

## 三层锁全景

| 层级   | 典型锁                                | 谁加           | 影响范围                   |
| ------ | ------------------------------------- | -------------- | -------------------------- |
| 全局锁 | FTWRL                                 | 手动执行       | 整个实例（所有库表）变只读 |
| 表级锁 | 表锁 / MDL / 意向锁 / AUTO-INC 锁     | 手动或自动     | 一整张表                   |
| 行级锁 | 记录锁 / 间隙锁 / 临键锁 / 插入意向锁 | 自动（InnoDB） | 一条或一段记录             |

记住这张表，后面所有细节都是往里面填东西。

## 全局锁

全局锁就是把**整个数据库实例锁成只读**。命令是：

```sql
flush tables with read lock;  -- 简称 FTWRL
```

执行之后，别的线程做这些事都会被堵住：

- 增删改（insert / update / delete）；
- 改表结构（alter table / drop table 之类）。

要释放就 `unlock tables;`，或者直接断开会话，全局锁会自动放掉。

**它主要用在全库逻辑备份上。** 道理也好理解：备份是一张表一张表地导出去的，如果导的过程中数据还在变，就可能导出一份"自相矛盾"的快照。

举个经典例子。备份正好导到一半，有人下了个单：

1. 用户表先被备份了（余额还没扣）；
2. 这时用户下单，余额扣了、库存减了；
3. 接着才备份商品表（库存已经是减过的）。

结果备份文件里：用户余额没少，但库存少了。拿这份文件恢复，等于用户白嫖一件商品。加了全局锁，备份期间没人能改数据，就不会有这种错位。

**缺点也很直白：** 整库只读，备份期间业务不能写。库一大，备份慢，业务就得停摆很久，这在线上基本没法接受。

那有没有不锁表的备份办法？有——**用 MVCC 的一致性快照**。如果存储引擎支持可重复读隔离级别，备份前先开个事务，事务一开就生成 Read View，整个备份过程都用这一个 Read View 读数据。期间别的事务怎么改都行，因为可重复读下你看到的始终是开事务那一刻的数据快照。

`mysqldump --single-transaction` 干的就是这件事：备份前先开事务，靠 MVCC 拿到一致性快照，**全程不锁表，业务照常读写**。InnoDB 默认就是可重复读，所以这招对 InnoDB 完全适用。

但要注意一个前提：**这招只对支持事务的引擎有效**。像 MyISAM 不支持事务，没有 MVCC 这套东西，备份它就只能老老实实上全局锁。所以"用 `--single-transaction` 代替 FTWRL"是有条件的，别一概而论。

## 表级锁

表级锁锁的是一整张表，InnoDB 里有四种：表锁、元数据锁（MDL）、意向锁、AUTO-INC 锁。前两种和并发控制关系大，后两种更偏"幕后辅助"。

### 表锁（LOCK TABLES）

显式的表锁分读锁和写锁：

```sql
lock tables t_student read;   -- 表级共享锁（读锁）
lock tables t_student write;  -- 表级独占锁（写锁）
```

有个容易忽略的点：**表锁不光限制别的线程，连本线程都一起限制。** 比如你给表加了读锁，那你自己接下来想往这张表写，也会被挡住，直到锁释放。释放用 `unlock tables;`，或者会话退出自动释放。

实际开发里手动加表锁很少见。InnoDB 有更细的行锁，没必要用粒度这么粗的表锁去拖垮并发。所以这个了解概念就行。

### 元数据锁（MDL）

MDL 不用你手动加，是 InnoDB **自动加**的，规则很简单：

- 对表做增删改查（CRUD）→ 自动加 **MDL 读锁**；
- 对表做结构变更（改字段、加索引等 DDL）→ 自动加 **MDL 写锁**。

它的兼容性是：**读读兼容、读写互斥、写写互斥**。也就是说多个查询可以同时读（读锁之间不冲突），但只要有人在改表结构，查询就得等；反过来有人在查，改表结构也得等。它的作用就是保证你在用一张表的时候，别人不能把表结构给你换掉。

MDL 还有个关键性质：**它在事务提交后才释放**，意味着整个事务期间 MDL 一直被持有。这就埋下了一个很经典的坑——

**长事务 + 改表结构会把整张表堵死。** 场景是这样：

1. 线程 A 开了事务但一直不提交，执行了一条 select，拿着 MDL 读锁不放；
2. 线程 B 也来 select，不冲突（读读兼容），正常执行；
3. 线程 C 要改表字段，申请 MDL 写锁——但 A 的读锁还占着，C 拿不到，**阻塞**。

问题是 C 一旦在那儿排队等写锁，**后面所有对这张表的查询也跟着被堵住了**。因为 MDL 锁请求是排队的，而且**写锁优先级高于读锁**：一旦队列里出现等待中的写锁，它后面的读锁请求就全得排在写锁后面。结果就是大量请求堆积、连接数飙升，数据库线程很快爆满。

所以线上**改表结构前一定要先查有没有长事务**（可以看 `information_schema.innodb_trx`）。如果有事务正持着这张表的 MDL 读锁，要么等它结束，要么把它 kill 掉，再做 DDL，否则一不小心就把库搞挂。

### 意向锁（IS / IX）

意向锁是个"表级的标记位"，用来**快速判断这张表里是不是已经有行被加锁了**。

规则：

- 给某些行加共享锁（S）之前，先在表级加**意向共享锁（IS）**；
- 给某些行加独占锁（X）之前，先在表级加**意向独占锁（IX）**。

比如 `insert / update / delete`，以及 `select ... for update`，都会先给表加 IX，再去给具体行加 X 锁；`select ... lock in share mode` 会先加 IS，再给行加 S 锁。注意普通的 `select` 走 MVCC 快照读，是不加行锁的，自然也不涉及意向锁。

那它有啥用？想象你要加一个"独占表锁"，得确认表里没有任何一行正被独占。**没有意向锁的话，你只能一行一行遍历去查**，表一大就慢得离谱。有了意向锁，只要看表上有没有 IX 标记——有就说明里面已经有行被独占了，直接判断冲突，不用遍历。这就是它存在的全部意义：**用一个表级标记，省掉逐行扫描。**

还有个特点：**意向锁之间互相不冲突**（IS 和 IX 彼此兼容），它也不和行级的 S/X 锁冲突，它真正参与冲突判断的对象是显式表锁（`lock tables ... read/write`）。所以你可以理解成，意向锁是为表锁服务的一个"提示"。

### AUTO-INC 锁

自增主键（`AUTO_INCREMENT`）靠它保证连续递增。

最原始的 AUTO-INC 锁是一种特殊表锁，但它有个不一样的地方：**不是等事务提交才释放，而是插入语句一执行完就立刻放**。一个事务持着它的时候，别的事务往这张表插数据会被挡住，这样就保证了自增值连续。代价是批量插入时并发会受影响。

从 MySQL 5.1.22 起，InnoDB 提供了更轻的方案：申请到自增值就马上释放，不等整条语句执行完。用 `innodb_autoinc_lock_mode` 这个变量控制：

| 取值 | 行为                                                                               |
| ---- | ---------------------------------------------------------------------------------- |
| 0    | 老的 AUTO-INC 锁，整条语句执行完才释放，最稳但并发差                               |
| 1    | 默认折中：普通 insert 申请到就放；`insert ... select` 这类批量插入仍等语句结束才放 |
| 2    | 申请到自增值就放，并发最高                                                         |

注意 mode = 2 虽然最快，但**搭配 `binlog_format = statement` 在主从复制下可能数据不一致**：两个 session 并发插入时，主库生成的自增 id 可能是交错的，而从库按串行重放 statement，生成的 id 是连续的，两边就对不上了。解决办法是把 binlog 格式设成 `row`——row 格式记录的是主库实际分配的自增值，从库照搬即可。所以生产上推荐 **mode = 2 + binlog_format = row** 的组合，既快又一致。

## 行级锁

行锁是 InnoDB 的招牌，MyISAM 没有。前面说过普通 select 是快照读、不加锁；要在查询时加行锁，得用**锁定读**：

```sql
select ... lock in share mode;  -- 加 S 锁（共享）
select ... for update;          -- 加 X 锁（独占）
```

这两条要放在事务里用（`begin` / `start transaction` / `set autocommit=0`），因为**事务一提交锁就放了**。

两种基本锁的兼容关系：

- **S 锁（共享锁）**：读读共享，读写互斥。多个事务可以同时持 S 锁，但有人持 S 锁时别人不能加 X 锁。
- **X 锁（独占锁）**：写写互斥、读写互斥。一旦有人持 X 锁，别人 S/X 都加不了。

在此基础上，InnoDB 行锁按"锁住什么形状"细分成下面几种。**这里只讲它们各自是什么概念，至于具体在什么 SQL、什么索引、什么隔离级别下到底会加哪种锁、间隙锁/临键锁怎么圈定边界、幻读边界怎么处理——这些加锁规则放在下一篇 [《行锁到底怎么加？间隙锁和临键锁怎么理解？》](/database/mysql/mysql-lock-rules.html) 细讲。**

### 记录锁（Record Lock）

最直白的一种：**只锁住一条记录**。也分 S 型和 X 型，兼容关系和上面 S/X 一样。

```sql
begin;
select * from t_test where id = 1 for update;
```

这就给 id = 1 这行加了 X 型记录锁，别的事务改不了它，直到本事务 commit。

### 间隙锁（Gap Lock）

间隙锁锁的不是某条记录，而是**记录之间的一段开区间**，目的是**防止往这段区间里插入新行**，从而解决可重复读下的幻读。它只在可重复读隔离级别存在。

比如有一个 (3, 5) 的间隙锁，别的事务就插不进 id = 4。注意区间是开的，端点本身不在锁定范围里。

间隙锁还有个反直觉的点：**间隙锁之间是互相兼容的**。两个事务可以同时持有有重叠区间的间隙锁，不会互斥。因为它的目的只是"不让你插进来"，而不是"独占这段空间"，所以多个事务一起守着同一段间隙，逻辑上不矛盾。

### 临键锁（Next-Key Lock）

临键锁 = **记录锁 + 间隙锁**，锁的是一个**前开后闭区间**，既锁住区间又锁住右端点那条记录。它是 InnoDB 可重复读下默认的行锁形态。

比如 (3, 5] 的临键锁，既不让别人插 id = 4（间隙部分管的），又不让别人改 id = 5（记录部分管的）。

兼容性要拆开看：它里面的**间隙部分仍然兼容**，但**记录部分要按 S/X 算账**。所以两个事务想拿同一范围的 X 型临键锁时，会因为记录锁那部分冲突而互相阻塞。

### 插入意向锁（Insert Intention Lock）

名字里带"意向"，但它**不是表级意向锁，而是一种特殊的间隙锁，属于行级锁**，别被名字骗了。

它出现在这种时刻：一个事务想插入新行时，得先看插入位置有没有被别人加了间隙锁（临键锁也含间隙部分）。如果有，插入就得等，等待期间会生成一个插入意向锁，表示"我想往这个位置插，但现在在排队"，直到持有间隙锁的事务提交、释放间隙锁。

可以这么理解：**间隙锁锁的是一段区间，插入意向锁锁的是区间里的一个点。** 关键差别在于——同一时刻，不能一个事务持着某段间隙锁、另一个事务又在这段间隙内持插入意向锁，这俩是冲突的（点落在被锁的区间里就冲突，落在区间外才行）。这正是间隙锁挡住并发插入的实现机制。

## 行锁是加在索引上的

最后补一个很关键、面试也爱问的点：**InnoDB 的行锁实际是加在索引记录上的，不是加在"数据行"这个抽象概念上**。

这带来一个直接后果：**如果你的查询条件没走索引（比如对没建索引的列做条件，或者写法导致索引失效），InnoDB 没法精确定位要锁哪几行，就会退化成给扫描到的所有记录加锁，效果近似锁全表**，并发直接崩掉。

所以"加了行锁就高枕无忧"是错觉——前提是**查询得走索引**。这背后的具体加锁路径同样在下一篇展开。

## 容易踩的坑

- **以为 `mysqldump --single-transaction` 万能**：它依赖 MVCC，只对 InnoDB 这类支持事务的引擎有效，MyISAM 备份还得靠全局锁。
- **长事务不提交 + 改表结构**：长事务死攥着 MDL 读锁，DDL 申请写锁失败排队，连带后面所有查询全堵死，是线上 DDL 翻车的高频原因。
- **mode = 2 配 statement binlog**：主从复制会出现自增 id 不一致，要么用 row 格式，要么换 mode。
- **把插入意向锁当成意向锁**：它其实是行级的特殊间隙锁，和表级意向锁（IS/IX）完全不是一回事。
- **以为加了 `for update` 就只锁一行**：查询不走索引时会退化成近似锁全表。

## 小结

- 锁按粒度分三层：**全局锁（整库只读，用于全库逻辑备份）→ 表级锁 → 行级锁**，粒度越细并发越好。
- 全局锁备份会锁死写操作，InnoDB 更推荐 `mysqldump --single-transaction`，靠 MVCC 一致性快照做到不锁表备份。
- 表级锁里最该记住 **MDL** 和**意向锁**：MDL 自动加、事务提交才放，长事务会把 DDL 和后续查询一起堵死；意向锁是表级标记，用来快速判断表里有没有行被锁。
- 行级锁是 InnoDB 招牌，分**记录锁 / 间隙锁 / 临键锁 / 插入意向锁**；间隙锁和临键锁专门防幻读，临键锁是可重复读下的默认形态。
- **行锁加在索引上**，查询不走索引会退化成近似锁全表——具体加锁规则见下一篇。
