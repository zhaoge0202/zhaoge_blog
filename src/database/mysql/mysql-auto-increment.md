# 自增主键为什么不一定连续？

> 自增主键只承诺两件事：值是递增的、值是唯一的。它从来没承诺过"连续"。一旦你默认它连续，就掉坑里了。

很多人对 `AUTO_INCREMENT` 有个想当然的印象：一行接一行插进去，id 就是 1、2、3、4、5……中间不会断。

实际不是这样。生产库里翻翻数据，经常会看到 id 是 1、2、4、7、8 这种带"窟窿"的序列。这不是 bug，是设计如此。这篇就把"为什么会断"讲清楚，顺便把面试里常被追问的几个点（自增值存哪、版本差异、申请策略、为什么不回退）都串起来。

先把结论钉死：

- **递增**：新申请的自增值一定比上一次大（默认步长下）。
- **唯一**：同一张表里自增主键不会重复（正常运行时）。
- **不保证连续**：中间允许有缺口。

## 自增值到底存在哪

要理解为什么会断，得先知道"下一个该发多少"这个数存在什么地方。

你 `show create table` 的时候会看到表定义里有个 `AUTO_INCREMENT=2`，意思是下次自动生成 id 会从 2 开始。但要注意，**这个值并不是真的"存在表结构文件里"**，它只是 MySQL 把当前自增计数器顺手显示给你看了一眼。真正的存储策略，不同引擎不一样：

- **MyISAM**：自增值写在数据文件里，重启不丢。
- **InnoDB（MySQL 8.0 之前）**：自增计数器只活在**内存**里，没持久化。每次"第一次打开这张表"时，InnoDB 会去扫一下当前最大的 id，拿 `max(id) + 1` 作为本次的自增起点。

这个内存计数器的特性，会带来一个很反直觉的现象。举个例子：

```sql
-- 表里现在最大 id = 1，AUTO_INCREMENT = 2
DELETE FROM t WHERE id = 1;
-- 删掉之后 AUTO_INCREMENT 还是 2，下一条会插成 id = 2
```

删了 id=1 这行，内存里的计数器并不会跟着退回去，还是 2。但**如果这时候重启 MySQL**，8.0 之前的 InnoDB 重启后会重新扫表算 `max(id)+1`，而表里已经空了（或者最大 id 变小了），于是 `AUTO_INCREMENT` 被算回成 1。下次插入又会得到 id=1。

这就埋了个隐患：重启后自增值可能"回退"，导致它去复用历史上已经被删掉的旧 id。如果有别的表存了"曾经的 id=1 指向某条业务数据"，重启后新插入的 id=1 含义就完全变了。这是 8.0 之前一个真实存在的坑。

**到了 MySQL 8.0，这个行为改了**：自增计数器的变更被记进了 redo log，做了持久化。重启后能从 redo log 恢复到崩溃/关闭前的值，不会再回退。所以上面那个例子在 8.0 里，重启后 `AUTO_INCREMENT` 仍然是 2。

> 面试里这个版本差异是高频追问点：**8.0 之前自增值在内存、靠 max(id)+1 重算、重启可能回退；8.0 起持久化到 redo log、重启不回退**。能把这句说清楚，基本就过关了。

## 自增值什么时候被改

插入时自增字段的取值规则其实很简单：

- id 字段填了 `0`、`NULL` 或干脆没写 —— 用当前的 `AUTO_INCREMENT` 值。
- id 字段明确写了一个具体值 —— 就用你写的那个。

而计数器要不要往上调，取决于这次实际用到的值和当前计数器谁大：你这次用的值 `>=` 当前计数器，计数器就被顶上去；小于就不动。

关键在于：**计数器是"申请的瞬间"就被改了，而不是等数据真正插入成功之后才改。** 这一条是后面所有"窟窿"的总根源。下面分场景看。

## 不连续的几种常见成因

### 1. 唯一键冲突：申请了，但插失败了

假设表里有个唯一索引列 `a`，现在 `AUTO_INCREMENT = 2`：

```sql
INSERT INTO t VALUES (NULL, 1, 1);  -- a=1 这行已经存在了，会报 Duplicate entry
```

执行流程是这样的：

1. InnoDB 发现没指定 id，取当前自增值 2；
2. 把记录改写成 `(2, 1, 1)`；
3. **把 `AUTO_INCREMENT` 改成 3**（注意，这步在真正写数据之前就干了）；
4. 真正去插的时候，撞上唯一键 `a=1` 冲突，报错返回。

结果就是：这条数据没插进去，但 id=2 这个值已经被"消费"掉了，计数器停在 3。下次插入直接拿 3，id=2 永远空着。

### 2. 事务回滚：申请了，但事务不要了

```sql
BEGIN;
INSERT INTO t VALUES (NULL, 3, 3);  -- 拿到 id=3，AUTO_INCREMENT 变 4
ROLLBACK;                            -- 回滚，这行没了
```

回滚之后表里确实没有这条记录，但**自增值不会跟着回滚**——它还是固执地停在 4。下次插入就是 id=4，id=3 这个号被浪费了。

### 3. 批量插入：一次申请一批，用不完就浪费

这里说的"批量插入"指的是 `INSERT ... SELECT`、`REPLACE ... SELECT`、`LOAD DATA` 这类 **MySQL 事先不知道到底要插多少行** 的语句。注意它**不是**指普通的 `INSERT INTO t VALUES (...),(...),(...)` 多 value——后者能精确算出要几个 id，一次性申请刚好够用，不会浪费。

对于不知道行数的批量语句，一个一个去申请自增 id 太慢，于是 InnoDB 用了"翻倍预分配"策略：

1. 第一次申请，给 1 个；
2. 不够了，第二次申请，给 2 个；
3. 再不够，第三次给 4 个；
4. 以此类推，每次是上次的两倍。

举个例子，用 `INSERT ... SELECT` 往新表灌 5 行数据：

- 第一次申请到 1 个：id=1
- 第二次申请到 2 个：id=2、id=3
- 第三次申请到 4 个：id=4、id=5、id=6、id=7

5 行只用掉 id=1~5，**id=6 和 id=7 是这次多申请、没用上的，直接作废**。之后再插一条普通数据，拿到的就是 id=8，中间 6、7 永远缺着。

### 4. 分布式场景下的步长设置

如果你为了双库错开主键，把 `auto_increment_increment`（步长）设成 2、`auto_increment_offset`（起点）分别设成 1 和 2，让一个库出奇数、另一个出偶数——那单看任意一个库，id 本来就是 1、3、5、7 这种，按定义就"不连续"。这其实是有意为之，不算异常，但面试答题时值得一提：**连续与否还受这两个参数影响**。

### 一句话串起来

唯一键冲突、回滚、批量预分配，本质都是同一件事：**自增值是"先申请、后使用"，申请走了就不还。** 用没用上、插没插成功，它都不管。

## 为什么申请了就坚决不回退

很自然会想：既然冲突/回滚时这行没插成功，把自增值改回去不就连续了吗？InnoDB 偏不这么做，原因是**回退的代价太大，得不偿失**。

用反证法走一遍：假设回滚时允许把自增值退回去。现在两个并发事务 A、B：

1. A 申请到 id=1，B 申请到 id=2，此时计数器是 3。
2. B 先提交成功，表里有了 id=2 这行；A 却因为唯一键冲突失败了。如果允许 A 把计数器退回 1——
3. 那么后面的事务又会申请到 id=2，可表里**已经有 id=2 了**，于是新插入直接报主键冲突。

要堵上这个新窟窿，只有两条路：要么每次申请前都去查一遍"这个 id 是不是已经被占了，占了就跳过"，要么把自增锁的范围扩大到"必须等一个事务彻底提交完，下一个才能申请"。前者每次插入都多一次检查、并发下满地是锁竞争；后者直接把并发插入串行化了。两种都会严重拖垮性能。

所以 InnoDB 干脆放弃"回退"这个设计：**语句失败也好、事务回滚也好，自增值一律不退。** 用一点点 id 的浪费（不连续），换来了申请逻辑的简单和高并发下的性能。这是一笔很划算的交易——id 是 BIGINT，几乎用不完，但锁竞争是实打实拖慢吞吐的。

## 顺带说两个相关点

**自增值会用完吗？** 会，但基本不用担心。`INT UNSIGNED` 上限约 42.9 亿，`BIGINT UNSIGNED` 大约 1844 亿亿，正常业务穷其一生也到不了。真要到了上限还继续插，会报主键冲突错误（达到上限后自增值不再增长，继续申请会拿到同一个最大值而冲突）。线上建议主键直接上 `BIGINT`，别图省空间用 `INT` 给自己埋雷。

**业务上千万别依赖主键连续。** 比如别拿主键 id 当"订单总数"用 `max(id)` 去统计、别假设 id 之间没有空隙去做分页或对账。要计数就老老实实 `count(*)`，要连续编号就单独维护一个业务序列号字段，别和主键混为一谈。

## 容易踩的坑

- **以为删一行后 id 会被复用** —— 默认不会，计数器只增不减，删了照样往后发。8.0 之前重启是个例外（可能回退去复用），8.0 修了。
- **把"连续"当成自增的承诺** —— 它只保证递增唯一。任何依赖连续性的业务逻辑（统计、对账、分页）都可能在冲突/回滚/批量插入后翻车。
- **分不清两种"批量插入"** —— 普通多 value 的 `INSERT INTO t VALUES (...),(...)` 是精确申请、不浪费；`INSERT...SELECT` / `LOAD DATA` 才走翻倍预分配、会留缺口。
- **以为回滚会还自增值** —— 不会。这是性能取舍，不是 bug。
- **生产用 INT 做主键还不监控** —— 增长快的表有撑爆 INT 的风险，到顶后插入直接报错，BIGINT 才是稳妥默认。

## 小结

1. 自增主键只保证**递增 + 唯一**，**不保证连续**，中间出现缺口是正常现象。
2. 不连续的根因是"**先申请、后使用，申请走了就不还**"：唯一键冲突、事务回滚、批量翻倍预分配都会留下被浪费的 id。
3. 自增值存储有版本差异：**8.0 之前在内存、靠 `max(id)+1` 重算、重启可能回退；8.0 起持久化进 redo log、重启不回退。**
4. 失败/回滚不退自增值，是为了避开"查重或扩大锁范围"的高昂代价，用 id 浪费换性能，是划算的取舍。
5. 业务上**不要依赖主键连续**；主键用 `BIGINT` 更稳妥，避免上限风险。
