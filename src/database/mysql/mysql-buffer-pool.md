# Buffer Pool 是怎么提速的？

> 一句话：数据真正躺在磁盘上，但 InnoDB 几乎所有读写都先在内存里的 Buffer Pool 转一圈，把"慢得离谱的随机磁盘 IO"尽量摊薄掉。

## 先想清楚：为什么非得有它

我们先做个量级对比，这是理解一切的前提。

读一次内存大概是几十到上百纳秒，读一次机械硬盘的随机 IO 是毫秒级——差了大概 10 万倍。就算换成 SSD，随机读也是几十微秒，比内存慢几百倍。数据库每天要扛成千上万次查询，如果每条查询都老老实实去磁盘上捞数据，那性能根本没法看。

所以思路特别朴素：**加缓存**。数据第一次从磁盘读出来之后，留一份在内存里，下次再查同样的数据就直接从内存拿，不碰磁盘。InnoDB 把这块专门用来缓存数据的内存区域叫 **Buffer Pool（缓冲池）**，它是 InnoDB 性能的命根子。

有了它之后，读写的路径变成这样：

- **读数据**：先看 Buffer Pool 里有没有，有就直接返回（命中）；没有才去磁盘读，读完顺手缓存进来。
- **写数据**：不直接改磁盘，而是改 Buffer Pool 里对应的那一页，把这页标记成"脏页"，然后由后台线程挑合适的时机批量刷回磁盘。

注意写这一步——**改完内存就算改完了**，磁盘是后面慢慢同步的。这就是为什么 InnoDB 的写也很快。当然这里马上会冒出一个疑问：内存改了磁盘还没改，万一宕机不就丢了？这个靠 redo log 兜底，后面刷盘那节会点一下，细节留给日志篇。

## 它到底有多大

Buffer Pool 是 MySQL 启动时一次性向操作系统申请的一大块连续内存。**默认只有 128MB**，这个值小得可怜，基本只够开发测试用。

生产环境一定要调大，参数是 `innodb_buffer_pool_size`，**经验值是物理内存的 60%~80%**。比如一台 64GB 内存、专门跑 MySQL 的机器，给 Buffer Pool 分个 40GB 左右是常见做法。

为什么不直接给 100%？因为机器上不止 MySQL 自己——操作系统要内存，MySQL 还有别的内存结构（连接缓冲、排序缓冲、临时表等），都得留出余量。给满了反而会触发 swap，性能断崖式下跌，得不偿失。

有个有意思的现象：MySQL 刚启动时你用 `top` 看，会发现它申请的**虚拟内存很大，实际占用的物理内存却很小**。这是因为申请来的内存只是"画了个地"，操作系统并不会立刻真的分配物理页，只有当这块内存被真正访问到时，才会触发缺页中断、把虚拟地址映射到物理地址。所以 Buffer Pool 的物理内存是随着数据被缓存进来一点点涨上去的。

## 里面缓存了哪些东西

Buffer Pool 不是简单一个 KV 缓存，它缓存的是 InnoDB 的各种"页"。

InnoDB 跟磁盘打交道的最小单位是**页，默认 16KB**。注意这个"最小单位"——哪怕你只查一行记录，InnoDB 也是把这行所在的整个 16KB 页一次性读进内存。因为索引只能帮你定位到"哪一页"，定位不到"页里第几行"，得把页加载进来再用页内的页目录去找具体那一行。

Buffer Pool 也是按 16KB 切成一个个槽位，每个槽位叫一个**缓存页**。它缓存的内容包括：

- **数据页**：存放表里实际行记录的页。
- **索引页**：B+ 树的索引节点页。
- **undo 页**：存放 undo log，用于事务回滚和 MVCC 的多版本读。
- **change buffer（插入缓存/写缓冲）**：对非唯一二级索引的写操作先缓存起来，攒一波再合并到索引页，省掉一堆随机 IO。
- **自适应哈希索引（AHI）**：InnoDB 发现某些页被频繁按等值查询访问时，会自动给它们建一个哈希索引，把 B+ 树的多层查找变成一次哈希定位。
- **锁信息**：行锁等锁结构也放在这里。

每个缓存页还配了一个叫**控制块**的小结构，记着这页属于哪个表空间、页号是多少、缓存页地址在哪、在各个链表里的位置等等。控制块放在 Buffer Pool 最前面，后面才是一片片缓存页。控制块和缓存页配对分配，最后可能剩一点点不够再凑一对的空间，那就是所谓的"碎片"，无伤大雅。

## 三个链表：怎么管这一堆页

Buffer Pool 是一整块连续内存，里面的缓存页有的空着、有的在用、有的还被改脏了。要是每次都从头遍历去找一个合适的页，效率太低。所以 InnoDB 用**三个链表**来分门别类管理，链表节点就是前面说的控制块。

**Free List（空闲链表）** —— 管空闲页。所有还没被用的缓存页挂在这条链表上。需要加载一个新页进来时，直接从 Free List 摘一个空闲页用就行，O(1) 拿到，不用遍历。这页用上之后就从 Free List 移除。

**Flush List（脏页链表）** —— 管脏页。只要一个缓存页被改过、跟磁盘不一致了，它就被挂到 Flush List 上。后台刷盘线程要干活时，遍历这条链表就知道哪些页需要写回磁盘，不用去全量扫描所有缓存页找脏的。

**LRU List** —— 管"正在用"的页（干净页 + 脏页），负责决定**内存满了该淘汰谁**。这是三个链表里最有讲究的一个，下一节专门说。

这里要厘清三种页的状态，别绕晕：

| 页状态               | 含义                       | 在哪些链表                 |
| -------------------- | -------------------------- | -------------------------- |
| Free Page（空闲页）  | 没被用                     | Free List                  |
| Clean Page（干净页） | 用了，但没改过，和磁盘一致 | LRU List                   |
| Dirty Page（脏页）   | 用了且改过，和磁盘不一致   | LRU List **和** Flush List |

关键点：**脏页是同时挂在 LRU List 和 Flush List 上的**。LRU 管它的冷热、决定淘不淘汰；Flush 管它的脏、决定什么时候刷。脏页一旦被刷回磁盘，数据就跟磁盘一致了，它从 Flush List 摘掉、退回成干净页，但还留在 LRU List 里。

## LRU 为什么不能用最朴素那版

淘汰策略最直觉的选择就是 LRU（Least Recently Used，最近最少使用）：链表头是刚用过的，链表尾是最久没碰的；内存满了就淘汰链表尾那个。访问命中就把这页挪到头部，未命中就从磁盘读进来塞到头部、同时踢掉尾部。

逻辑很顺，但 InnoDB **没有**直接用这版朴素 LRU，因为它扛不住两种场景。

### 坑一：预读失效

InnoDB 有**预读**机制：基于"空间局部性"——你访问了某页，它旁边的页很可能马上也要用——所以加载一页时会顺手把相邻的几页也提前读进来，想着减少后面的磁盘 IO。

问题是，这些**提前读进来的页未必真会被访问**。如果用朴素 LRU，这些预读页一进来就被放到链表头部，占着最"显眼"的位置。要是它们压根没人访问，又因为在头部迟迟不被淘汰，反而把尾部那些真正被频繁访问的热点页给挤掉了。预读本想帮忙，结果帮了倒忙，这叫**预读失效**。

### 坑二：Buffer Pool 污染

更严重的是大批量扫描。一条 SQL 扫了大量数据（典型就是**全表扫描**），这些页会被逐个读进来、逐个访问。朴素 LRU 下它们全被塞到链表头部，于是**Buffer Pool 里原本的热点页被成批挤出去**。等下次真正的热点查询来了，发现缓存全没了，只能重新去磁盘读，磁盘 IO 暴涨、性能骤降。这叫 **Buffer Pool 污染**。

特别坑的一点：**污染跟结果集大小无关**。哪怕一条 SQL 最后只返回几行，只要它走了全表扫描，照样要把一大堆页加载并访问一遍。比如：

```sql
select * from t_user where name like "%alice%";
```

前缀带 `%` 的模糊匹配用不上索引，整张表所有页都得读进来逐行比对，结果可能就匹配出三五条，但 Buffer Pool 已经被搅得天翻地覆了。

## InnoDB 怎么改进 LRU

InnoDB 的招数是把 LRU List **切成两段**：**young 区域**（前半段，热数据）和 **old 区域**（后半段，刚进来还没站稳脚跟的页）。

两段的比例由 `innodb_old_blocks_pct` 控制，**默认 37**，意思是 old 区域占整个 LRU 链表的 37%，young 区域占 63%。换算成约数大概就是 **young 占 5/8、old 占 3/8**。

> 这里提醒一句：网上常有人把比例写成"young 占 5/8、old 占 3/8"当成精确值，其实 5/8 = 62.5%、3/8 = 37.5%，跟默认的 63/37 只是约等于，不是真正的八分之几。记成"大约六成 young、四成 old"更稳妥，面试时说清楚是约数就行。

改进后的两条规则：

**第一条，新页先进 old，不直接进 young。** 不管是预读进来的页还是查询要用的页，刚加载进 Buffer Pool 时一律先塞到 **old 区域的头部**。如果这页从此再没人访问，它就在 old 区域里慢慢往尾巴挪，最后被淘汰——全程没碰 young 区域，热点数据安然无恙。这就治住了**预读失效**：预读进来的页就算白读了，也只占 old 的地，挤不掉 young 里的热点。

**第二条，升入 young 要"够格"。** 光把"被访问"当成升级条件还不够——全表扫描里每页都会被访问一次，要是访问一次就升 young，污染照样发生。所以 InnoDB 又加了一道**停留时间**门槛：

- 页第一次进 old 区域被访问时，控制块里记下这个访问时间。
- 之后再被访问时，比较两次访问的时间间隔：
  - 间隔**没超过**阈值 → 这页**不升 young**，留在 old。
  - 间隔**超过了**阈值 → 才把它升到 **young 区域头部**。

这个阈值是 `innodb_old_blocks_time`，**默认 1000ms（1 秒）**。

说白了，**一个页要升进 young 必须同时满足两个条件：被再次访问 + 在 old 区域已经待了超过 1 秒**。全表扫描的特点是"每页飞快地访问一两次就过去了"，两次访问间隔远小于 1 秒，所以这些页根本没机会升 young，扫完就待在 old 里等着被淘汰，young 区域的热点数据稳稳的。污染问题就这么解决了。

还有个小优化顺便提一下：为了避免 young 区域里的页因为被频繁访问而老是往头部挪（挪动本身也有开销），InnoDB 规定 young 区域**前 1/4 的页被访问时不挪到头部**，只有后 3/4 的页被访问才挪。这样 young 区里最热的那批就不用反复折腾了。

## 脏页什么时候刷回磁盘

写操作只改了 Buffer Pool 里的页，磁盘上还是旧数据，所以脏页迟早得刷回去保证一致。但绝不能每改一次就刷一次——那等于退化成同步写磁盘，前面的优化全白费。InnoDB 的做法是**攒着批量刷**，在这几个时机触发：

1. **redo log 快写满了**：redo log 是固定大小循环写的，写满了就没法记新日志，必须先把对应的脏页刷下去、腾出空间。这是比较"急"的一种刷盘。
2. **空闲页不够了**：要加载新页但 Free List 空了，得淘汰 LRU 尾部的页。如果要淘汰的恰好是脏页，必须先把它刷回磁盘才能复用这个槽位。
3. **系统空闲时**：后台线程见缝插针，定期刷一批适量的脏页，平摊压力，别让脏页攒成山。
4. **MySQL 正常关闭时**：会把所有脏页一次性刷干净再退出，保证下次启动数据是全的。

至于"内存改了磁盘没改、宕机怎么办"——InnoDB 用的是 **WAL（Write-Ahead Logging，预写日志）** 思想：**改数据之前先把 redo log 写下去**。redo log 是顺序追加写（比随机写数据页快得多），只要日志落盘了，就算脏页还没刷、机器突然崩了，重启时也能靠 redo log 把没刷的修改重放回来，数据不丢。WAL 的细节这里就点到为止，留给日志篇展开。

## 容易踩的坑

- **以为只缓存数据**：很多人以为 Buffer Pool 就是缓存表数据，其实索引页、undo 页、change buffer、自适应哈希、锁信息都在里面，它是 InnoDB 内存的大本营。
- **young/old 比例死记 5/8 和 3/8**：那只是 63/37 的约数，说成精确分数会被追问翻车，强调是约数即可。
- **把升 young 的条件只说成"被访问"**：必须强调"被访问 **且** 在 old 停留超过 1 秒"两个条件缺一不可，否则解释不了为什么能防污染。
- **以为污染只发生在大结果集**：跟结果集大小无关，只要走全表扫描、加载了大量页，哪怕只返回几行也会污染。
- **`innodb_buffer_pool_size` 设成物理内存 100%**：会挤掉操作系统和 MySQL 其他内存结构，触发 swap 反而更慢，留 20%~40% 余量。

## 小结

- Buffer Pool 是 InnoDB 在内存里的缓冲池，读写都先走它，目的是把慢几个数量级的随机磁盘 IO 摊薄；默认 128MB，生产建议设为物理内存的 60%~80%（`innodb_buffer_pool_size`）。
- 它以 16KB 的页为单位缓存数据页、索引页、undo 页、change buffer、自适应哈希、锁信息等，每个缓存页配一个控制块。
- 三个链表分工：Free List 管空闲页、Flush List 管脏页、LRU List 管在用页的冷热与淘汰；脏页同时挂在 LRU 和 Flush 上。
- InnoDB 改进 LRU 防两件事：新页先进 old 防"预读失效"；升 young 要满足"被访问 + 在 old 待够 `innodb_old_blocks_time`（默认 1 秒）"防"Buffer Pool 污染"。
- 脏页在 redo log 快满、空闲页不足、后台空闲、正常关闭时批量刷盘；靠 WAL（先写 redo log）保证宕机不丢数据。
