# 时间字段和主键怎么选？

> 时间字段挑的是「要不要让数据库帮你管时区」，主键挑的是「写入是不是顺序追加」——两件事看着小，建表时定错了后面很难改。

这篇是建表阶段最常被随手带过、面试又特别爱抠细节的两个点。我把它们放一起讲，因为它们都属于「字段类型一旦选定、上线后改起来要命」的那类决定。

## 先说一句：别用字符串存时间

很多人刚学数据库时会用 `VARCHAR` 存 `'2024-05-01 09:00:00'`，觉得直观。但这是反模式，原因有三个：

- **占空间大**：一个完整的日期时间字符串要 19 个字符以上，而原生时间类型最多 8 字节。
- **比较是按字典序**：`'2024-05-01'` 在字符串比较里会小于 `'2024-1-10'`（因为第六位 `0` < `1`），范围查询的结果可能完全是错的。
- **用不了日期函数**：算两个时间的间隔、做加减、`DATE_FORMAT` 这些都得先转换，麻烦又慢。

所以下面讨论的都是「正经的时间类型」：`DATETIME`、`TIMESTAMP`，以及用整型存的 Unix 时间戳。

## DATETIME vs TIMESTAMP：核心就四个差异

这俩长得几乎一样，查出来都是 `YYYY-MM-DD HH:MM:SS`，区别全在「底层怎么存、能存多大、带不带时区」上。

| 维度       | DATETIME                | TIMESTAMP                                |
| ---------- | ----------------------- | ---------------------------------------- |
| 占用空间   | 5~8 字节                | 4~7 字节（更省）                         |
| 存的是什么 | 字面值，不带时区        | 内部按 UTC 存                            |
| 时区转换   | 不转换，存啥读啥        | 写时按会话时区转 UTC，读时再转回会话时区 |
| 表示范围   | 1000-01-01 ~ 9999-12-31 | 1970-01-01 ~ **2038-01-19**              |
| 自动维护   | 需手动配                | 天然支持 `CURRENT_TIMESTAMP`             |

下面逐个说清楚。

### 差异一：时区

这是两者最本质的区别，也是面试官最爱追问的点。

`DATETIME` 存的是「字面值」。你写进去 `09:00:00`，它就老老实实存 `09:00:00`，**不管你的会话时区是几区**。读出来还是 `09:00:00`。它压根不知道这个时间属于哪个时区，相当于一张「没贴时区标签的纸条」。

`TIMESTAMP` 不一样。写入时，MySQL 会把你给的值（按当前会话时区理解）转成 UTC 存到磁盘；读出来时，再按当前会话时区转回去。所以同一条记录，在不同时区的会话里查，看到的「本地时间」不同，但它们指向的是同一个绝对时间点。

举个直观的例子：先在 UTC+0 的会话里插一条，两个字段都查到 `09:53:32`。然后把会话时区改成东八区：

```sql
SET time_zone = '+8:00';
```

再查，`DATETIME` 还是 `09:53:32`（雷打不动），而 `TIMESTAMP` 变成了 `17:53:32`（自动 +8）。

```text
+---------------------+---------------------+
| date_time           | time_stamp          |
+---------------------+---------------------+
| 2020-01-11 09:53:32 | 2020-01-11 17:53:32 |
+---------------------+---------------------+
```

**一句话记忆**：`TIMESTAMP` 自带时区转换，跨时区应用很省心；`DATETIME` 存的是死值，时区逻辑得你自己在应用层兜底。

### 差异二：占用空间

在 MySQL 5.6.4 之前是固定的：`DATETIME` 8 字节、`TIMESTAMP` 4 字节。5.6.4 之后引入了小数秒（fractional seconds），空间随精度变化：`DATETIME` 5~8 字节，`TIMESTAMP` 4~7 字节。

整体看 `TIMESTAMP` 更省 1 字节左右。但说实话，单条记录省这点空间在绝大多数业务里无关紧要——除非你这张表是亿级别且时间字段还进了组合索引，否则别为了这 1 字节去选 `TIMESTAMP`。

### 差异三：表示范围（2038 问题）

这是 `TIMESTAMP` 的硬伤：

- `DATETIME`：`1000-01-01` 到 `9999-12-31`，基本可以认为「这辈子够用」。
- `TIMESTAMP`：`1970-01-01 00:00:01` 到 **`2038-01-19 03:14:07`** UTC。

为什么卡在 2038？因为传统实现把它当成 32 位有符号整数存（秒数），到 2038-01-19 那一刻会溢出，俗称「2038 年问题」（Y2K38）。

这意味着：**任何可能落在 2038 年之后的时间，绝对不能用 `TIMESTAMP`**。比如保险到期日、贷款还款计划、长期合同结束时间——这些一旦超界，存进去要么报错要么数据错乱。

> 补一个资料里没强调、但很关键的点：`TIMESTAMP` 的范围下限是 1970，所以 **1970 年之前的时间也存不了**。需要记录历史事件（比如出生年份在 1970 前），同样别用 `TIMESTAMP`。

### 差异四：自动更新

`TIMESTAMP` 天然支持两个很方便的默认值：

```sql
`created_at` timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP,
`updated_at` timestamp NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
```

`DEFAULT CURRENT_TIMESTAMP` 是插入时自动填当前时间，`ON UPDATE CURRENT_TIMESTAMP` 是每次更新行时自动刷新时间。

这里有个常见误解需要纠正：**`DATETIME` 在 MySQL 5.6.5 之后也支持 `DEFAULT CURRENT_TIMESTAMP` 和 `ON UPDATE CURRENT_TIMESTAMP` 了**，并不是 `TIMESTAMP` 独有的能力。所以「想要自动维护时间就只能用 `TIMESTAMP`」这种说法是过时的。早期版本确实是这样，但现在两者都能做。

## 用数值存 Unix 时间戳，行不行？

实践中还有第三种玩法：用 `INT` 或 `BIGINT` 存 Unix 时间戳（从 1970-01-01 00:00:00 UTC 起的秒数或毫秒数）。

它的好处很实在：

- **天然无时区**：存的就是一个绝对数值，全球任何地方生成的同一时刻都是同一个数，系统间传递不用做任何转换，只在最后展示给用户时才转成本地时间字符串。
- **比较和排序快**：本质就是整数比大小，比时间类型的比较还直接。
- **跨语言跨系统方便**：前后端、不同数据库之间传一个数字，没有格式歧义。

坏处也很明显：

- **可读性极差**：你直接看 `1578707612` 完全不知道是几号，运维排查、人工核对数据时很难受，必须用 `FROM_UNIXTIME()` 转一下。
- **范围要注意**：用 `INT`（4 字节、有符号）存秒数，一样会撞 2038 问题。想避开就得用 `BIGINT`，或者存毫秒数时直接上 `BIGINT`。

> 这里要修正资料里的一处不严谨：它在总结表格里写「数值型时间戳 4 字节」。这只是用 `INT` 存秒数的情况，**而 `INT` 存秒会和 `TIMESTAMP` 一样栽在 2038**。真要用数值存、又想躲开 2038，应该用 8 字节的 `BIGINT`。所以「数值时间戳就一定省空间」这个印象是片面的。

MySQL 里互转用这两个函数：

```sql
-- 时间字符串 -> Unix 时间戳（秒）
SELECT UNIX_TIMESTAMP('2020-01-11 09:53:32');  -- 1578707612
-- Unix 时间戳（秒）-> 时间字符串
SELECT FROM_UNIXTIME(1578707612);              -- 2020-01-11 09:53:32
```

## 那到底选哪个？

没有银弹，看场景。我的实践建议是：

- **大多数场景：`DATETIME`**。范围大、没有 2038 隐患，时区统一在应用层用 UTC 处理就行（现在主流框架、ORM 都能配会话时区）。这是最稳、最不容易踩坑的默认选择。
- **需要数据库自动管时区、且时间确定在 2038 前：`TIMESTAMP`**。典型就是 `created_at` / `updated_at` 这种「记录发生时刻、且就是当下」的字段——它们永远是「现在」，2038 前肯定够用，又能蹭 `ON UPDATE` 的自动刷新。
- **极度看重比较性能、或频繁跨系统传时间、能接受可读性差：数值时间戳（`BIGINT`）**。高并发、分布式、需要做时间运算密集排序的场景可以考虑。

顺带一提：《高性能 MySQL》的作者更倾向 `TIMESTAMP`，理由是「数值表示时间不够直观」。这个观点可以参考，但别当圣旨——它更多是在「`TIMESTAMP` vs 数值」之间表态，而 2038 问题在那个年代还没这么紧迫。今天我个人更推 `DATETIME` 兜底。

## 主键怎么选：优先自增整型

主键这块详细原理在[索引设计篇](/database/mysql/mysql-index-design.html)讲聚簇索引时会展开，这里只讲结论和「为什么」。

**首选 `BIGINT AUTO_INCREMENT`（自增整型）**，原因都和 InnoDB 的聚簇索引有关：

- InnoDB 的数据是按主键顺序物理存放的（聚簇索引，叶子节点就是整行数据）。自增主键意味着每次插入都是「往最后追加」，新行总是落在 B+ 树最右边，**不会去中间挤已有的页**，几乎不触发页分裂。
- 主键本身占用小（`BIGINT` 8 字节），而且 **InnoDB 的二级索引叶子节点存的是主键值**——主键越小，所有二级索引就越小、越省内存和磁盘。

**为什么不推荐 UUID / 随机字符串做主键？**

| 对比项           | 自增整型       | UUID / 随机字符串        |
| ---------------- | -------------- | ------------------------ |
| 插入位置         | 顺序追加到末尾 | 随机散落在各页中间       |
| 页分裂           | 几乎没有       | 频繁，性能抖动大         |
| 页利用率         | 高、紧凑       | 碎片多、利用率低         |
| 主键大小         | 小（8 字节）   | 大（如 36 字符的字符串） |
| 二级索引连带影响 | 小             | 全部跟着膨胀             |

简单说：UUID 是无序的，插入时新行可能要塞进 B+ 树中间已经写满的页里，导致页分裂、产生碎片、写入变慢；而且它本身又长，二级索引都被它拖大。所以除非有特殊需求，别拿随机字符串当主键。

**分布式场景的折中**：当单库自增满足不了全局唯一、或者不想暴露业务量时，可以用雪花算法（Snowflake）这类**趋势递增**的 ID——它整体随时间递增，保留了「顺序写入、不易页分裂」的优点，同时具备分布式唯一性，是常见的折中方案。

## 容易踩的坑

- **以为 `DATETIME` 不能自动填时间**：5.6.5 之后 `DATETIME` 也支持 `DEFAULT CURRENT_TIMESTAMP` 和 `ON UPDATE`，别再以为这是 `TIMESTAMP` 专属。
- **给未来时间用了 `TIMESTAMP`**：到期日、合同结束、还款计划这类可能跨过 2038 的字段，用了 `TIMESTAMP` 是定时炸弹。
- **以为数值时间戳一定省空间**：`INT` 存秒省是省，但照样 2038 溢出；真要避开得上 `BIGINT`，就不省了。
- **`TIMESTAMP` 存历史时间**：1970 年之前的时间存不进 `TIMESTAMP`，记录出生年份、历史事件要小心。
- **拿 UUID 当主键还纳闷写入慢**：无序主键的页分裂是性能杀手，业务主键和聚簇主键最好分开（聚簇用自增、UUID 单独建唯一索引）。
- **混用时区不自知**：`TIMESTAMP` 的显示值依赖会话时区，如果应用各节点会话时区不统一，同一条记录看到的时间会不一样，排查问题时容易被绕进去。

## 小结

- 别用字符串存时间；时间类型在 `DATETIME`、`TIMESTAMP`、数值时间戳里选。
- 四个核心差异：空间（`TIMESTAMP` 略省）、时区（`TIMESTAMP` 自动转）、范围（`TIMESTAMP` 有 2038 上限和 1970 下限）、自动维护（如今两者都支持）。
- 默认推荐 `DATETIME` 兜底；`created_at`/`updated_at` 这类「当下、确定在 2038 前」的字段可以用 `TIMESTAMP`；极致性能/跨系统场景考虑 `BIGINT` 数值时间戳。
- 主键优先自增整型：顺序写入不页分裂、占用小、二级索引也小；别用无序的 UUID 做主键。
- 分布式要全局唯一就用雪花算法这类趋势递增 ID，兼顾顺序性和唯一性。
