# 事务隔离级别怎么理解？

> 隔离级别本质是一道选择题：你愿意为了"数据看起来更干净"付出多少并发性能。脏读、不可重复读、幻读是要规避的三种现象，四个隔离级别就是四档不同的取舍。

这道题几乎是 MySQL 面试的必考点。很多人能背出"四个级别、三个问题"，但一被追问"不可重复读和幻读到底差在哪""为什么 MySQL 默认是 RR 而不是标准的 RC"就卡住了。这篇就把这几层讲透。

## 先说事务的 ACID

要聊隔离级别，得先知道它在 ACID 里处于什么位置。事务有四个特性：

- **原子性（Atomicity）**：一个事务里的操作要么全做，要么全不做，中途出错就回滚到事务开始前的样子，就像没执行过一样。转账扣了 A 的钱却没加到 B 头上，这种半截状态是绝对不允许的。
- **一致性（Consistency）**：事务执行前后，数据都满足完整性约束。比如 A 和 B 共 1400 元，转账完还是 1400 元，不会凭空多出或蒸发。
- **隔离性（Isolation）**：多个事务并发跑的时候，互相不干扰，每个事务感觉自己在独占数据库。
- **持久性（Durability）**：事务一旦提交，改动就是永久的，哪怕之后断电、宕机也丢不了。

那 InnoDB 是靠什么实现这四个特性的？记住这张对应关系，面试基本能直接背：

| 特性   | InnoDB 的实现手段                                          |
| ------ | ---------------------------------------------------------- |
| 原子性 | **undo log**（回滚日志，出错时用它把数据还原）             |
| 持久性 | **redo log**（重做日志，崩溃后用它把已提交的改动重放回来） |
| 隔离性 | **MVCC + 锁**                                              |
| 一致性 | 由上面三者**共同保证**                                     |

这里有个容易被忽略的点要说清楚：**一致性是目的，原子性、持久性、隔离性是手段**。换句话说，一致性不是靠某一种单独机制实现的，而是前三个特性都做到了，数据自然就一致了。面试时如果你能主动点出这层关系，是个加分项。

这篇我们重点展开的是隔离性，因为它直接关联到隔离级别这个高频考点。

## 并发事务会带来哪三个问题

为什么需要隔离性？因为多个事务同时读写同一份数据时，会冒出一些"读得不对劲"的现象。一共三种，从轻到重排：脏读、不可重复读、幻读。

### 脏读：读到了别人没提交的数据

**一个事务读到了另一个事务"还没提交"的修改，就叫脏读。**

举个例子：

1. 事务 A 把用户余额从 100 万改成了 200 万，但**还没提交**。
2. 这时事务 B 来查这个用户的余额，读到的是 200 万。
3. 结果事务 A 反悔了，回滚！余额又变回 100 万。

那事务 B 手里那个 200 万就是个根本不存在过的"幽灵数据"——这就是脏读。问题的根源在于 B 读到了一笔可能被回滚的、尚未确定的数据。

### 不可重复读：同一行，两次读结果不一样

**在一个事务里多次读同一行数据，前后读到的结果不一样，就叫不可重复读。**

举个例子：

1. 事务 A 第一次读这个用户的余额，是 100 万。
2. A 还没结束，事务 B 把余额改成 200 万并**提交了**。
3. 事务 A 再读同一行，变成了 200 万。

注意它和脏读的区别：脏读是读到了"没提交"的数据，不可重复读读到的是"已经提交"的数据——问题不在于数据脏，而在于**在同一个事务内，同一行数据居然会变**，这破坏了事务读取的稳定性。

### 幻读：同一个条件，两次查到的行数不一样

**在一个事务里，按同一个条件查"符合条件的记录数量"，前后两次查到的行数不一样，就叫幻读。**

举个例子：

1. 事务 A 查"余额大于 100 万"的账户，查到 5 条。
2. 事务 B 插入了一条余额 120 万的新账户并**提交**。
3. 事务 A 再用同样的条件查，变成了 6 条，凭空多出一行，像见了鬼——这就是幻读。

### 不可重复读 vs 幻读，到底差在哪

这是面试官最爱追问的点，一句话区分：

- **不可重复读关注的是"某一行数据变了"**——还是那几行，但某行的值被改了（UPDATE）。
- **幻读关注的是"行数变了"**——符合条件的记录多了或少了（INSERT / DELETE）。

一个是"内容变了"，一个是"数量变了"。所以解决思路也不同：不可重复读靠锁住已读到的那几行就够了；幻读还得防住别人往这个范围里"插队"，光锁现有的行不行，得把行与行之间的空隙也锁上。

## 四个隔离级别分别解决了什么

SQL 标准定义了四个隔离级别，级别越高越干净，但并发性能越差：

- **读未提交（READ UNCOMMITTED，RU）**：别人没提交的改动你也能看见。最弱，脏读、不可重复读、幻读全都可能发生，实际几乎没人用。
- **读已提交（READ COMMITTED，RC）**：只能看到别人**已提交**的改动，挡住了脏读，但不可重复读和幻读还在。Oracle、SQL Server 默认就是这个级别。
- **可重复读（REPEATABLE READ，RR）**：一个事务执行期间，看到的数据始终和事务启动那一刻一致，挡住了脏读和不可重复读。**按 SQL 标准，幻读仍然可能发生**——但 InnoDB 这里做了加强，下一节细说。**MySQL InnoDB 的默认级别就是它**。
- **串行化（SERIALIZABLE）**：最强，事务一个一个排队执行，靠读写锁把并发冲突全挡掉，脏读、不可重复读、幻读统统没有。代价是并发性能最差。

一张对照表，谁能解决谁一目了然（√ 表示该问题会发生，× 表示不会）：

| 隔离级别            | 脏读 | 不可重复读 | 幻读                        |
| ------------------- | ---- | ---------- | --------------------------- |
| 读未提交 RU         | √    | √          | √                           |
| 读已提交 RC         | ×    | √          | √                           |
| 可重复读 RR         | ×    | ×          | √（标准）/ 大体 ×（InnoDB） |
| 串行化 SERIALIZABLE | ×    | ×          | ×                           |

记忆方法：从上往下，每降一个级别就多解决一个问题。RC 解决脏读，RR 再解决不可重复读，SERIALIZABLE 把幻读也干掉。

## 为什么 MySQL 默认是 RR，而不是标准的 RC

这是个很能体现你是否真懂的点。SQL 标准里大多数数据库默认用 RC，但 **MySQL InnoDB 偏偏默认 RR**，这是 MySQL 的一个特色，不是随便定的。

主流的说法是和**主从复制的数据一致性**有关。早期 MySQL 的 binlog 只有 `STATEMENT` 格式（按 SQL 语句记录），在 RC 级别下，主库和从库回放语句时，因为没有间隙锁的约束，事务执行顺序和可见性可能产生差异，导致主从数据不一致。RR 级别下的间隙锁能保证语句执行的确定性，主从复制才稳。所以 MySQL 把默认级别定成了 RR。

> 补充一句：如今 binlog 普遍用 `ROW` 格式（按行记录变更），这个历史包袱其实已经缓解了不少，RC 在很多生产环境也用得很安心。但 MySQL 出于兼容性考虑，默认值一直保持着 RR。面试时点明"这是历史原因 + 主从一致性考量"即可。

## RR 下的幻读：InnoDB 怎么"很大程度"解决的

前面表格里写 RR 对幻读是"标准会发生、InnoDB 大体不发生"，这正是 InnoDB 的精彩之处，也是高频考点。

InnoDB 在 RR 级别下用了两套机制，分别对付两种读：

- **快照读（普通 SELECT）→ 用 MVCC 解决。** 事务启动时拍一张数据快照（ReadView），之后整个事务期间的普通查询都读这张快照。哪怕别人中途插入了新行并提交，你的快照里也没有它，自然就查不出来，幻读被挡住了。
- **当前读（`SELECT ... FOR UPDATE`、`LOCK IN SHARE MODE`、`UPDATE`、`DELETE`）→ 用 next-key lock 解决。** 这类语句读的是最新数据，MVCC 帮不上忙。InnoDB 给它们加 **next-key lock**（行锁 Record Lock + 间隙锁 Gap Lock 的组合），不光锁住扫到的行，连行与行之间的"空隙"也锁上，别人就没法往这个范围里插入新行，幻读也被挡住了。

但请务必注意标题里的**"很大程度"**三个字——InnoDB 在 RR 下并**没有 100% 消灭幻读**。在一些快照读和当前读混用的边界场景里（比如先快照读、中间别人插入提交、再用当前读或更新），幻读依然可能露头。这部分涉及具体的加锁规则，细节我们留到[加锁规则那篇](/database/mysql/mysql-locks.html)展开，这里点到为止。面试时千万别把话说死成"InnoDB 在 RR 下彻底解决了幻读"，正确说法是"很大程度上避免了"。

## 怎么查看和设置隔离级别

查看当前隔离级别（MySQL 8.0 及以后用 `transaction_isolation`，8.0 之前用 `tx_isolation`）：

```sql
-- MySQL 8.0+
SELECT @@transaction_isolation;
-- MySQL 8.0 之前
SELECT @@tx_isolation;
```

正常会看到默认值：

```
+-------------------------+
| @@transaction_isolation |
+-------------------------+
| REPEATABLE-READ         |
+-------------------------+
```

设置隔离级别，`SESSION` 只对当前连接生效，`GLOBAL` 对之后新建的连接生效：

```sql
SET [SESSION | GLOBAL] TRANSACTION ISOLATION LEVEL
  { READ UNCOMMITTED | READ COMMITTED | REPEATABLE READ | SERIALIZABLE };
```

如果想自己动手验证脏读、不可重复读这些现象，开两个命令行窗口，各自 `SET SESSION` 不同级别，再用 `START TRANSACTION` 显式开事务对比着看就行（MySQL 默认是自动提交的，不显式开事务每条语句都会立刻提交）。

## 容易踩的坑

- **别把"一致性"当成某个机制实现的**。一致性是目的，靠原子性 + 持久性 + 隔离性共同达成，单独问"一致性靠什么保证"要答"前三者"。
- **不可重复读和幻读别混**。一个是行的内容变了（UPDATE），一个是行的数量变了（INSERT/DELETE）。
- **别说"InnoDB 在 RR 下彻底解决了幻读"**。标准说法是"很大程度上避免"，存在边界情况。
- **redo log 和 undo log 别记反**：undo 管原子性（回滚），redo 管持久性（崩溃恢复）。一个往回退，一个往前补。
- **隔离级别不是越高越好**。串行化能根治所有问题，但几乎等于把并发关了，生产上极少用，绝大多数场景 RR 或 RC 足够。

## 小结

- 事务的 ACID 里，InnoDB 用 undo log 保原子性、redo log 保持久性、MVCC + 锁保隔离性，一致性由这三者共同保证（一致性是目的，其余是手段）。
- 并发事务有三个问题：脏读（读到未提交数据）、不可重复读（同一行两次读结果不同）、幻读（同一条件两次查行数不同）。不可重复读是"数据变了"，幻读是"行数变了"。
- 四个隔离级别从弱到强：RU → RC → RR → SERIALIZABLE，每升一级多解决一个问题。RC 挡脏读，RR 再挡不可重复读，SERIALIZABLE 全挡。
- MySQL InnoDB 默认是 RR（区别于标准默认的 RC），主要出于主从复制数据一致性的历史考量。
- 标准里 RR 不解决幻读，但 InnoDB 在 RR 下用 MVCC（快照读）+ next-key lock（当前读）**很大程度上**解决了幻读——注意不是 100%。
