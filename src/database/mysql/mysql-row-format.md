# 一行记录在 InnoDB 里是怎么存的？

> 一条记录不是孤零零躺在磁盘上的，它被一层层包在「段 → 区 → 页 → 行」里，而行本身又被切成「额外信息 + 真实数据」两大块。把这套结构看明白，varchar 最大长度、NULL 占不占空间、行溢出这些面试题就全串起来了。

很多人对一行数据的理解停留在「就是表里的一行嘛」，但面试官一旦往下问——`varchar(n)` 的 n 最大能填多少、NULL 值占不占空间、一行太大放不下一页会怎样——就答不上来了。这些问题看着零散，其实背后是同一个知识点：**InnoDB 一行记录的存储结构**。这篇我们从外往里一层层扒开看。

## 数据到底存在哪个文件里

先定位一下物理文件。MySQL 的数据放在磁盘上，具体目录由 `datadir` 决定（典型是 `/var/lib/mysql/`）。每建一个 database 就在这个目录下生成一个同名子目录，表的结构和数据文件都在里面。

以一张 `t_order` 表为例，老版本里你会看到这几个文件：

- `db.opt`：存这个库的默认字符集和校验规则。
- `t_order.frm`：存表结构（表的元数据定义）。
- `t_order.ibd`：存表数据和索引，这就是所谓的**独立表空间文件**。

这里要提醒一句：上面这套文件结构是 **MySQL 5.x 时代**的。从 **8.0 开始 `.frm` 已经被废弃**了，表结构信息被收进了数据字典（存在 `mysql.ibd` 里），`.frm` 和 `db.opt` 这两个文件你在 8.0 上是看不到的。所以如果你装的是 8.0+，目录里基本只剩下 `t_order.ibd`。原理面试照着讲 `.ibd` 即可，别把 `.frm` 当成现在还存在的东西。

是不是每张表一个 `.ibd`，由参数 `innodb_file_per_table` 控制。它从 5.6.6 起默认就是 1（每表独立表空间），所以现在默认情况下每张表的数据都在自己的 `.ibd` 里。

## 表空间的四层结构：段 → 区 → 页 → 行

`.ibd` 这个表空间文件内部不是一锅粥，而是有层次的。从大到小是：**段（segment）→ 区（extent）→ 页（page）→ 行（row）**。倒过来从小往大讲更好理解。

### 行（row）

表里的记录按「行」存放。每行根据**行格式**的不同，内部结构也不同——这正是本文重点，后面细讲。

### 页（page）：16KB，读写的最小单位

记录虽然按行存，但 InnoDB **读写的最小单位是「页」，不是「行」**。原因很直白：如果一次 I/O 只能搬一行，效率会低到没法看。所以 InnoDB 把数据按页打包，要读一条记录时，是把它所在的整页（默认 **16KB**）一次性load进内存。

- 页是磁盘管理的最小单元，每次读写至少 16KB。
- 页有很多类型：数据页、undo 页、溢出页等。我们平时说的「行记录存哪」，存的就是**数据页**。

你可以把页理解成「快递箱」：哪怕只取一件小东西，也是整箱搬，省得来回跑。

### 区（extent）：1MB，64 个连续页

为什么在页之上还要有「区」？因为 InnoDB 用 B+ 树组织数据，B+ 树叶子层是用双向链表串起来的。如果按页一个一个分配空间，链表上逻辑相邻的两个页，物理位置可能离得很远，**范围扫描时就变成大量随机 I/O，很慢**。

解决办法是：数据量大时，不再按页分配，而是**按区分配，一个区 1MB = 64 个连续的 16KB 页**。一整块连着划出去，逻辑相邻的页物理上也挨着，范围查询就能走顺序 I/O，快得多。

### 段（segment）：按用途把区分组

最上层是段。表空间由若干段组成，段由若干区组成。段是按**用途**分的，常见三种：

- **数据段**：存 B+ 树叶子节点（也就是真实数据行）的那些区。
- **索引段**：存 B+ 树非叶子节点的那些区。
- **回滚段**：存回滚（undo）数据的区，MVCC 的多版本查询就是靠它实现的。

绕这一大圈才讲到行，是为了让你心里有张「从文件 → 表空间 → 段 → 区 → 页 → 行」的地图，不至于一上来对着行格式干瞪眼。

## InnoDB 有哪几种行格式

行格式（`row_format`）就是「一条记录的存储结构」。InnoDB 提供 4 种：

| 行格式     | 状态                                                  |
| ---------- | ----------------------------------------------------- |
| Redundant  | 5.0 之前的老古董，不紧凑，现在基本没人用              |
| Compact    | 5.1 起的默认格式，紧凑，一页能塞更多行                |
| Dynamic    | **5.7 起的默认格式**，在 Compact 基础上改了行溢出策略 |
| Compressed | 在 Dynamic 基础上再对页做压缩                         |

Dynamic 和 Compressed 都是从 Compact 改一点点来的，所以**把 Compact 吃透，其它的看一眼就懂**。下面重点讲 Compact。

## Compact 行格式长什么样

一条 Compact 记录分成两大块：**记录的额外信息** + **记录的真实数据**。

```
┌─────────────────── 额外信息 ───────────────────┐ ┌──── 真实数据 ────┐
变长字段长度列表 | NULL 值列表 | 记录头信息(5字节)    隐藏列 | 你定义的列...
                                              ↑
                                   next_record 指针指向这里
```

注意那个箭头：记录头里的 `next_record` 指针，指向的是**下一条记录「头信息」和「真实数据」之间的分界点**。这样设计的好处是——从这个点**向左读就是头信息，向右读就是真实数据**，两边都好定位。这个细节等下解释「逆序」时还会用到。

### 额外信息之一：变长字段长度列表

`char(n)` 是定长，`varchar(n)`、`TEXT`、`BLOB` 是变长——实际存多长不固定。既然不固定，读的时候怎么知道该读几个字节？答案就是：**把每个变长字段实际占用的字节数，记在「变长字段长度列表」里**。

举个例子，建一张字符集为 ascii（1 字符 = 1 字节）的表：

```sql
CREATE TABLE t_user (
  id    INT(11)      NOT NULL,
  name  VARCHAR(20)  DEFAULT NULL,
  phone VARCHAR(20)  DEFAULT NULL,
  age   INT(11)      DEFAULT NULL,
  PRIMARY KEY (id)
) ENGINE=InnoDB DEFAULT CHARSET=ascii ROW_FORMAT=COMPACT;
```

假设第一条记录 `name='a'`、`phone='123'`：

- `name` 占 1 字节，十六进制 `0x01`；
- `phone` 占 3 字节，十六进制 `0x03`；
- `id`、`age` 是 int 定长，不进这个列表。

关键点：这些长度是**按列的逆序存放**的。所以列表里是 `03 01` 而不是 `01 03`。

如果某条记录里某个变长列是 NULL（比如第三条 `phone` 为 NULL），那它**根本不出现在真实数据里**，自然也**不会在变长字段长度列表里占位**——列表里只记非 NULL 的变长字段。

> 为什么要逆序存？

结合前面那个 `next_record` 指针看就懂了。指针指向「头信息和真实数据的分界点」，真实数据是**从分界点向右**铺开的：第一个变长列的数据离分界点最近、第二个其次……而变长字段长度列表在分界点**左边**。如果让长度列表也逆序（第一个变长列的长度离分界点最近），那么「第一个变长列的长度」和「第一个变长列的数据」在内存里就**贴在分界点两侧**，更可能落进**同一条 CPU Cache Line**，缓存命中率更高，读取更快。这是个性能上的小心机。

> 是不是每张表都有这个列表？

**不是**。如果一张表里全是定长字段（比如全是 int），就没有变长字段，这个列表干脆不存在，省空间。

### 额外信息之二：NULL 值列表

如果把 NULL 老老实实塞进真实数据区，挺浪费的。Compact 的做法是：用一串二进制位（bit）来标记哪些列是 NULL。

规则：

- **只有允许为 NULL 的列**才参与（NOT NULL 的列不占 bit）；
- 每个可空列对应 1 个 bit，**按列逆序**排列；
- bit 为 `1` 表示该列是 NULL，`0` 表示非 NULL；
- 必须凑成**整数个字节**，位数不够就在高位补 `0`。

还是 `t_user` 三条记录（可空列是 name、phone、age）：

| 记录    | 哪些列为 NULL | bit（age phone name，逆序） | 补满 1 字节 | 十六进制 |
| ------- | ------------- | --------------------------- | ----------- | -------- |
| 第 1 条 | 无            | `000`                       | `00000000`  | `0x00`   |
| 第 2 条 | age           | `100`                       | `00000100`  | `0x04`   |
| 第 3 条 | phone、age    | `110`                       | `00000110`  | `0x06`   |

> 这个列表也不是必须的吗？

对。**当所有字段都是 NOT NULL 时，NULL 值列表整个消失**。所以设计表时尽量加 NOT NULL，至少能省 1 字节（NULL 值列表最少占 1 字节）。

> 固定 1 字节吗？9 个字段都是 NULL 怎么办？

不固定。9 个可空列就需要 2 字节（9 bit 凑不下 1 字节），以此类推，按需扩展。

### 额外信息之三：记录头信息

记录头是**固定 5 字节**，里面字段很多，挑几个面试常考的：

- `delete_mask`：标记这条记录是否被删除。注意——执行 `DELETE` 时并**不是真的擦掉数据**，只是把这个标志位置 1（后续由 purge 线程清理）。
- `next_record`：指向下一条记录的位置，记录之间是用单向链表串起来的。前面反复提到，它指的是下一条记录「头信息和真实数据的分界点」。
- `record_type`：记录类型。`0` 普通记录、`1` B+ 树非叶子节点记录、`2` 最小记录（Infimum）、`3` 最大记录（Supremum）。

### 真实数据：除了你定义的列，还有隐藏列

真实数据区里，除了你建表时写的那些列，InnoDB 还会偷偷加 3 个**隐藏列**：

| 隐藏列         | 字节 | 作用                                             | 是否必有                        |
| -------------- | ---- | ------------------------------------------------ | ------------------------------- |
| `row_id`       | 6    | 没主键也没唯一非空索引时，InnoDB 拿它当内部主键  | **否**，有主键/唯一非空键就不加 |
| `trx_id`       | 6    | 事务 id，标记这行是哪个事务改的                  | 是                              |
| `roll_pointer` | 7    | 回滚指针，指向这行的上一个版本（在 undo log 里） | 是                              |

`trx_id` 和 `roll_pointer` 就是 **MVCC 的命根子**：靠 `roll_pointer` 把每一行的历史版本串成一条「版本链」，再配合 Read View 判断哪个版本对当前事务可见。这里先点到为止，细节我们放到 MVCC 那篇专门讲。

补一句源文里没强调的：`row_id` 的触发条件准确说法是「**既没有主键、也没有非空唯一索引**」。光有一个允许 NULL 的唯一索引是不够的，InnoDB 还是会补 `row_id`。

## varchar(n) 里这个 n，到底是多少

先记住一条铁律：**除了 TEXT、BLOB 这类大对象，一行里所有列（不含隐藏列和记录头）加起来不能超过 65535 字节**。注意是**一行的总和**，不是单列。

然后澄清一个常见误区：`varchar(n)` 的 **n 是字符数，不是字节数**。具体占多少字节，取决于字符集：

- ascii：1 字符 1 字节，`varchar(100)` 最多 100 字节；
- utf8（mysql 的 `utf8mb3`）：1 字符最多 3 字节；
- **utf8mb4**：1 字符最多 4 字节（这才是真正的完整 UTF-8，能存 emoji，8.0 默认字符集）。

### 只有一个 varchar 列时，n 最大能填多少

直觉上会觉得「一行上限 65535，那 `varchar(65535)` 总行了吧」。建一张试试：

```sql
CREATE TABLE test (
  name VARCHAR(65535) NULL
) ENGINE=InnoDB DEFAULT CHARSET=ascii ROW_FORMAT=COMPACT;
```

**报错，建不出来**。报错信息会告诉你：一行上限 65535 是**包含 storage overhead** 的。这个 overhead 就是前面讲的「变长字段长度列表」+「NULL 值列表」。也就是说 65535 这个额度，得先把这两个列表的开销扣掉。

算一下这张表的开销：

- **NULL 值列表**：name 允许为 NULL，占 **1 字节**。
- **变长字段长度列表**：规则是——变长字段最大字节数 ≤ 255 用 1 字节记长度，> 255 用 2 字节。这里最大 65535 > 255，所以用 **2 字节**。

于是：`n 最大 = 65535 - 2 - 1 = 65532`。实测 `varchar(65532)` 能建成功，`65533` 不行，推论得证。

如果换成 utf8mb3（1 字符最多 3 字节），那 n 还要再除以 3：`65532 / 3 = 21844`。换 utf8mb4 就是 `65532 / 4 ≈ 16383`。所以「varchar(n) 的 n 上限」这种题，**一定要先问字符集**，没字符集这题无解。

### 多个字段时

公式变成：**所有列的数据长度 + 变长字段长度列表 + NULL 值列表 ≤ 65535**。本质没变，只是把单列换成了所有列求和。设计宽表时容易在这撞墙，尤其是一堆大 varchar 堆在一起。

## 行溢出：一行太大，一页放不下怎么办

页才 16KB（16384 字节），可一个 `varchar` 能存到 65532 字节，TEXT/BLOB 更夸张。一行明显塞不进一页，这时就发生**行溢出**，多出来的数据被甩到**溢出页**里。

不同行格式甩数据的策略不一样：

| 行格式                   | 真实数据区里留什么                                | 溢出页里放什么 |
| ------------------------ | ------------------------------------------------- | -------------- |
| **Compact / Redundant**  | 该列的**前 768 字节**前缀数据 + 20 字节溢出页指针 | 剩余数据       |
| **Dynamic / Compressed** | **几乎不留数据**，只放 20 字节溢出页指针          | 整列数据       |

这里要纠正源文一个含糊的地方：Compact 留下的「一部分数据」**不是随便一部分，而是固定的前 768 字节前缀**（再跟一个 20 字节指针）。源文只说「保存一部分」，没点明 768 这个数。

Dynamic 的策略（完全溢出、只留指针）其实**更聪明**：真实数据区不被大字段的前缀撑爆，一页能塞下更多行记录，B+ 树更矮、查询更快。这也是 5.7 之后默认换成 Dynamic 的原因之一。Compressed 在 Dynamic 基础上又对页做了压缩，进一步省空间，但压缩/解压有 CPU 开销，要权衡。

## 容易踩的坑

- **`.frm` 文件早没了**：8.0 起表结构进了数据字典，目录里只剩 `.ibd`。别在 8.0 环境里还去找 `.frm`。
- **`varchar(n)` 的 n 是字符不是字节**：算上限必须结合字符集，utf8mb4 下 1 字符吃 4 字节，n 会被压得很小。
- **65535 是「一行总和」且含 overhead**：不是单列上限，也不是纯数据额度，得扣掉变长长度列表和 NULL 值列表。
- **NULL 不是不占空间**：值本身不进真实数据，但只要有可空列，就有一份 NULL 值列表（最少 1 字节）。全 NOT NULL 才能彻底省掉它。
- **行溢出留多少跟行格式有关**：Compact 留 768 字节前缀，Dynamic 几乎只留 20 字节指针，别记串了。
- **DELETE 不是立刻物理删除**：只是把 `delete_mask` 置 1，空间后续才回收。

## 小结

- 表空间从大到小是**段 → 区 → 页 → 行**：区 1MB（64 个连续页）保证顺序 I/O，页 16KB 是读写最小单位。
- Compact 行格式 = **额外信息（变长字段长度列表 + NULL 值列表 + 5 字节记录头）+ 真实数据（含 row_id/trx_id/roll_pointer 三个隐藏列）**。
- 变长长度列表和 NULL 值列表都**逆序存放**，目的是让字段长度和数据更可能落进同一条 CPU Cache Line，提升缓存命中率。
- **一行上限 65535 字节**（不含大对象），且包含 storage overhead；`varchar` 的 n 是字符数，单列上限要结合字符集计算（ascii 下最大 65532）。
- **行溢出**时 Compact 留 768 字节前缀 + 20 字节指针，Dynamic/Compressed 几乎全溢出只留 20 字节指针；隐藏列 `trx_id`、`roll_pointer` 是 MVCC 的基础。
