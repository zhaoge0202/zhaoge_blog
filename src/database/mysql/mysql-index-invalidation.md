# 索引为什么会失效？

> 建了索引不等于一定会走索引——很多"失效"其实是优化器算下来全表扫描更划算，真正的判断标准是 `EXPLAIN`，不是死背结论。

我先把一个最容易被误会的总纲摆在最前面：**"索引失效"这个词其实把两类完全不同的事情混在一起了**。

一类是**语法/写法上真的没法用索引**——比如你在索引列上套了函数，B+ 树存的是原始值，套了函数之后的值在树里压根没有顺序，想二分查找都没法查，这是硬失效。

另一类是**优化器主动放弃**——索引能用，但优化器估算了一下，发现"走索引 + 大量回表"的随机 IO 还不如"全表顺序扫描"快，于是它理性地选了全表扫描。这压根不是 bug，是它干得对。

为什么要先强调这个？因为面试里你如果张口就背"用了 OR 就失效""IN 太多就失效"，碰到深一点的面试官追问"那为什么"，你就露馅了。下面每个场景我都会区分它到底属于哪一类。

下文我用一张统一的表来举例，省得每节重新交代：

```sql
CREATE TABLE t_user (
  id          INT PRIMARY KEY,           -- 主键（聚簇索引）
  name        VARCHAR(50),               -- 二级索引 idx_name
  age         INT,                       -- 无索引
  phone       VARCHAR(20),               -- 二级索引 idx_phone（注意是字符串类型）
  create_time DATETIME,                  -- 二级索引 idx_ct
  a INT, b INT, c INT,
  KEY idx_name (name),
  KEY idx_phone (phone),
  KEY idx_ct (create_time),
  KEY idx_a_b_c (a, b, c)                -- 联合索引，下面讲最左前缀用
);
```

## 联合索引不满足最左前缀

联合索引 `(a, b, c)` 你可以理解成"先按 a 排序，a 相同再按 b 排序，b 再相同才按 c 排序"——就像电话簿先按姓排、姓一样再按名排。所以你要用它，必须**从最左边那列开始、连续地用**。

| 查询条件                    | 能走 `idx_a_b_c` 吗    | 说明                           |
| --------------------------- | ---------------------- | ------------------------------ |
| `where a=1`                 | 能                     | 用了最左列                     |
| `where a=1 and b=2`         | 能                     | 连续命中 a、b                  |
| `where a=1 and b=2 and c=3` | 能                     | 全部命中                       |
| `where b=2`                 | 不能                   | 跳过了 a，b 在整棵树里是无序的 |
| `where c=3`                 | 不能                   | 同上                           |
| `where a=1 and c=3`         | a 走索引，c 不参与定位 | 索引截断，见下                 |

```sql
-- 跳过最左列 a，没法用 idx_a_b_c
SELECT * FROM t_user WHERE b = 2 AND c = 3;

-- 正确：从 a 开始连续命中
SELECT * FROM t_user WHERE a = 1 AND b = 2;
```

有两个细节值得多说一句：

**第一，`where a=1 and c=3` 不是简单的"c 失效"。** a 能用索引定位，c 因为中间断了 b，没法用来精确定位。但从 MySQL 5.6 起有**索引下推（ICP）**：c 这个条件会被推到存储引擎层，在扫描 a=1 的那一段记录时顺手把不满足 c=3 的过滤掉，减少回表。`EXPLAIN` 的 `Extra` 里会看到 `Using index condition`。所以说它"半失效"更准确。

**第二，where 里条件的书写顺序不影响。** `where b=2 and a=1` 和 `where a=1 and b=2` 完全一样，优化器会自己调整顺序去匹配索引。最左前缀说的是**索引定义里的列顺序**，不是你 SQL 里写的顺序，这点别搞混。

还有个反直觉的彩蛋后面"容易踩的坑"里会讲：如果表里所有列都在某个索引里，即使不满足最左前缀，也可能走 `type=index` 的全索引扫描，并不是全表扫描。

## 在索引列上做函数或表达式计算

这是硬失效，原因很简单：**B+ 树里存的是列的原始值，不是你算完之后的值。**

```sql
-- 失效：对 create_time 套了 YEAR()
SELECT * FROM t_user WHERE YEAR(create_time) = 2024;

-- 失效：对 id 做了表达式计算
SELECT * FROM t_user WHERE id + 1 = 10;
```

`YEAR(create_time)` 这种，索引树里存的是 `2024-03-01 12:00:00` 这样的完整时间，按时间先后排好序的；而 `YEAR(...)` 算出来的 2024 这个值在树里根本没有对应的有序结构，优化器没法二分定位，只能把每一行都取出来挨个算一遍 `YEAR()`，这就是全表扫描。

`id + 1 = 10` 同理。你可能会说"这数学上不就等于 id = 9 吗，MySQL 自己变换一下不就行了"——理论上能做，但 MySQL 偷懒没做，表达式形态太多它懒得逐个适配，干脆甩锅给程序员：你自己别在索引列上算。

改法就是把计算挪到等号右边，让索引列保持"裸露"：

```sql
-- 表达式移到右边
SELECT * FROM t_user WHERE id = 10 - 1;

-- 时间范围代替 YEAR()，强烈推荐这种写法
SELECT * FROM t_user
WHERE create_time >= '2024-01-01 00:00:00'
  AND create_time <  '2025-01-01 00:00:00';
```

补充一点：MySQL 8.0 支持**函数索引**，可以直接对 `YEAR(create_time)` 这种计算结果建索引。但这是补救手段，能改 SQL 写法就优先改写法，别动不动加函数索引。

## 隐式类型转换（转换方向决定生死）

这是最隐蔽、也是面试最爱挖的坑，核心就一句话：**比较两边类型不一致时 MySQL 会做隐式转换，转换落在谁身上谁就遭殃——落在索引列上就失效，落在常量上就没事。**

先记住 MySQL 的转换规则（来自官方文档比较规则第 7 条）：**字符串和数字比较时，统一把字符串转成数字（DOUBLE）再比。** 你可以用 `SELECT '10' > 9` 验证，结果是 `1`，说明 `'10'` 被转成了数字 10 来比，而不是按字符串比。

然后看两个方向：

| 写法                 | 列类型           | 谁被转换                              | 索引 |
| -------------------- | ---------------- | ------------------------------------- | ---- |
| `phone = 1300000001` | phone 是 varchar | **列被转**（`CAST(phone AS DOUBLE)`） | 失效 |
| `id = '1'`           | id 是 int        | **常量被转**（`CAST('1' AS DOUBLE)`） | 有效 |

```sql
-- 失效：phone 是字符串列，却拿数字去比
-- 等价于 WHERE CAST(phone AS DOUBLE) = 1300000001 —— 相当于对索引列套了函数
SELECT * FROM t_user WHERE phone = 1300000001;

-- 有效：id 是整型列，拿字符串去比，转的是常量，列没动
-- 等价于 WHERE id = CAST('1' AS DOUBLE)
SELECT * FROM t_user WHERE id = '1';
```

为什么"字符串列 = 数字"会失效，背后还有更深一层原因：字符串转数字**不是一一对应的**。`'10000a'`、`'010000'`、`' 10000'` 转成数字都是 10000（规则是：不以数字开头的转成 0；以数字开头的截取到第一个非数字为止）。也就是说一个数字 10000 对应着一大堆不同的字符串，索引树是按字符串原文排序的，这些"长得不一样但转完相等"的值散落在树的各处，自然没法用索引定位。

> 一句话记牢：**字符串列千万别拿数字去查，等号右边老老实实加引号。** 数字列拿字符串查虽然不失效，但也别这么写，类型对齐是基本素养。

## like 以 % 开头（左模糊）

`like 'xxx%'`（右模糊）能走索引，因为前缀确定，B+ 树能从"xxx"这个前缀定位到起始点往后扫；而 `like '%xxx'`（左模糊）前缀不确定，可能是"陈林、张林、周林"任何东西，树里找不到从哪开始扫，所以**通常**会失效。

```sql
SELECT * FROM t_user WHERE name LIKE '林%';    -- 走索引，type=range
SELECT * FROM t_user WHERE name LIKE '%林';    -- 通常失效
SELECT * FROM t_user WHERE name LIKE '%林%';   -- 通常失效
```

**但是！这里有个反直觉的真相，很多人背的"like %开头一定全表扫描"是不严谨的。**

到底走不走全表扫描，关键看**查询要的数据能不能被索引覆盖**。可以看下面这个典型对比：

- **表里有非索引字段**（比如 `t_user` 还有 age、phone 这些不在 idx_name 里的列），`select * ... like '%林'`：要查的数据二级索引里没有，得逐行回表才能拿全，左模糊又没法用有序性，优化器一算成本太高，直接 `type=ALL` 全表扫描。
- **表里只有主键 + 该二级索引**（即只有 `name` 列加上主键 `id`），`select * ... like '%林'`：此时 `select *` 等价于 `select id, name`，**要的数据全在二级索引的 B+ 树叶子节点里（索引值 + 主键值），不用回表**。这就是覆盖索引。这时候 `EXPLAIN` 显示的是 `type=index`，也就是**全扫描二级索引树**，而不是 `type=ALL`。

为什么覆盖索引这种情况会扫二级索引而不是聚簇索引？因为二级索引叶子节点只存"索引列 + 主键值"，又瘦又小；聚簇索引叶子节点存的是整行（主键、事务 id、回滚指针、所有列），又胖。优化器算下来，遍历瘦的那棵树成本更低，而且不用回表，所以选了它。

| 场景                                         | 执行计划 type | 是不是全表扫描     | 快慢                     |
| -------------------------------------------- | ------------- | ------------------ | ------------------------ |
| `like '林%'`                                 | range         | 否                 | 最快，能定位起始点       |
| `like '%林'` + 表有非索引字段                | ALL           | 是，全表扫         | 最慢                     |
| `like '%林'` + 表只有主键+该索引（覆盖索引） | index         | 否，全扫二级索引树 | 比 ALL 快，但比 range 慢 |

所以严谨的说法是：**左模糊匹配不一定走全表扫描，关键看能不能覆盖索引。** 这是个能让面试官眼前一亮的细节。

不过实战里也别钻牛角尖：真要做大规模模糊搜索，老老实实上 Elasticsearch，别指望 MySQL 的 like。

## OR 连接里有一边没索引

`OR` 的语义是"满足任一条件即可"，所以只要其中**有一个条件列没索引**，那一边就只能全表扫描去找，既然都要全表扫了，另一边的索引也就没意义了，整体退化成全表扫描。

```sql
-- age 没索引，整条语句全表扫描（即使 id 是主键）
SELECT * FROM t_user WHERE id = 1 OR age = 18;
```

解决办法有两个：

**办法一：给两边都建索引。** 两边都有索引时，MySQL 可能触发**索引合并（Index Merge）**——分别扫两个索引再把结果取并集，`EXPLAIN` 会看到 `type=index_merge`。

```sql
-- 给 age 也建索引后，可能走 index_merge
SELECT * FROM t_user WHERE id = 1 OR name = '林';
```

**办法二：改写成 UNION。** 把 OR 拆成两条独立查询，各自走各自的索引，还规避了优化器对 OR 成本估不准的问题：

```sql
SELECT * FROM t_user WHERE id = 1
UNION
SELECT * FROM t_user WHERE name = '林';
```

这里有个细节要注意：用 `UNION` 还是 `UNION ALL`？`UNION ALL` 不去重、更快，但如果两段结果有重叠（比如某行 id=1 且 name 也='林'），就会出现重复行。**只有确定结果不会重复时才用 `UNION ALL`，否则用 `UNION`**（代价是临时表去重）。我看有的资料直接推荐 `UNION ALL`，这是不严谨的，得看业务上会不会重。

## 其他几个别绝对化的场景

下面这几个，结论都不是"一定失效"，而是"看优化器成本怎么算"，别背死。

**IN / NOT IN：**

`IN (1, 2, 3)` 这种短列表正常走索引。但列表特别长时（超过 `eq_range_index_dive_limit`，默认 200），优化器会从精确的 Index Dive 探测切换成基于统计信息的粗略估算，万一统计信息陈旧，可能误判成本而放弃索引。这不是 IN 本身的锅，是估算策略变了。

`NOT IN (1, 2, 3)` 常量列表通常全表扫描，因为要遍历整棵树证明"不在集合里"。推荐用 `NOT EXISTS` 或 `LEFT JOIN ... IS NULL` 替代。

```sql
SELECT * FROM t_user WHERE age IN (1, 2, 3);        -- 短列表，正常
SELECT * FROM t_user WHERE id NOT IN (1, 2, 3);     -- 常量 NOT IN，倾向全表扫
```

**`!=` / `<>` 和 `is null` / `is not null`：**

这些经常被列进"失效清单"，但其实**不绝对**。它们走不走索引完全取决于优化器估算的选择率——如果某个值很罕见，`!= 它` 命中的行数也很少时不一定失效；`is null` 在某些情况下也能走索引（取决于该列 null 值的占比）。

所以与其背"`!=` 一定失效"，不如记住总纲：**这类场景能不能走索引由优化器按成本决定，唯一可靠的判断方式是 `EXPLAIN` 实测。**

## 容易踩的坑

- **"索引建了就一定会用"——错。** 优化器会按成本选执行计划，命中数据量太大（经验阈值 20%~30% 以上）时，它宁可全表顺序扫，也不愿大量随机回表。这时 `type=ALL` 是它的正确决策，不是性能问题。

- **"`type=ALL` 就是出 bug 了"——不一定。** 小表、或者要返回大部分数据时，全表扫描本来就是最优解。看到 ALL 先想想数据量，别条件反射地去加索引。

- **"like %开头一定全表扫描"——不严谨。** 上面讲过，覆盖索引场景下走的是 `type=index`（全扫二级索引树），比 ALL 快。

- **"联合索引不满足最左前缀就全表扫描"——也不一定。** 同样的彩蛋：如果查询涉及的列恰好全在某个索引里（覆盖索引），即使没按最左前缀，也可能走 `type=index` 全索引扫描，而不是 ALL。本质还是"扫瘦索引树比扫胖聚簇索引树便宜"。

- **隐式转换最坑在于它悄无声息。** SQL 语法完全正确、能跑出正确结果，只是慢——数据量小时根本发现不了，等数据涨上来才暴雷。字符串列查询右边一定加引号。

- **`UNION` 和 `UNION ALL` 别乱用。** 结果可能重复时用 `UNION ALL` 会多出重复行，得用 `UNION` 去重。

- **`select *` 本身不直接导致失效**，但它让覆盖索引失效、强制回表，间接抬高成本让优化器更容易放弃索引。能只查需要的列就只查需要的列。

## 小结

- 把"索引失效"拆成两类看：一类是**写法破坏了 B+ 树有序性**（函数、表达式、隐式转换、左模糊），属于硬失效；另一类是**优化器按成本主动放弃**（OR、IN 过长、命中数据量大），人家是干得对。

- **隐式类型转换记方向**：字符串列 = 数字 → 转的是列 → 失效；数字列 = 字符串 → 转的是常量 → 有效。字符串列查询右边永远加引号。

- **左模糊不一定全表扫描**：能被覆盖索引覆盖时走 `type=index`（全扫二级索引树），比 `type=ALL` 快。同理，覆盖索引下不满足最左前缀也可能走 index 而非 ALL。

- **联合索引认最左前缀**，认的是索引定义的列顺序，不是 where 里写的顺序；中间断列后续列可靠 ICP 索引下推补救。

- **不要背死结论**，`!=`、`is null`、`NOT IN` 这些都由优化器按成本定，唯一可靠的验证手段是 `EXPLAIN`——重点看 `type`、`key`、`rows`、`Extra` 四个字段。
