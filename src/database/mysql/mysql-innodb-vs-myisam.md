# InnoDB 和 MyISAM 有什么区别？

> 一句话：InnoDB 支持事务、行锁、外键、MVCC、崩溃后能安全恢复，是现在的默认引擎；MyISAM 只支持表锁、没有事务、崩溃容易丢数据，基本只在历史项目和极少数纯只读场景里还能见到。

这道题几乎是 MySQL 存储引擎部分的必问题。它考的不只是“背几条区别”，而是看你能不能把这些区别串起来——为什么 InnoDB 能扛并发写、为什么它崩溃能恢复、为什么现在几乎没人选 MyISAM。下面我按维度一个个拆。

## 先说背景：默认引擎换过

MySQL 是**插件式存储引擎**架构，存储引擎是挂在「表」这一层的，不是挂在数据库上的——同一个库里，理论上不同表可以用不同引擎（虽然实际很少这么干）。

时间线上有个分水岭：

- **MySQL 5.5 之前**，默认引擎是 MyISAM；
- **MySQL 5.5.5 开始**，默认引擎换成了 InnoDB，一直到现在。

这个“换默认”本身就说明问题了：官方用脚投票，把更稳、更能扛事务和并发的 InnoDB 推到了前台。所以下面所有对比，你可以理解成“为什么会换”的展开。

## 维度一：事务支持

**InnoDB 支持事务，MyISAM 不支持。**

这是两者最根本的区别，也是 InnoDB 最核心的价值。InnoDB 完整实现了 SQL 标准的四个隔离级别，能 commit、能 rollback，默认隔离级别是「可重复读（REPEATABLE READ）」。

MyISAM 完全没有事务这个概念。你执行一批 SQL，中间断电了，前面成功的、后面没成功的就那样了，没法回滚到一个干净的状态。对任何涉及钱、库存、订单的业务来说，这是不可接受的。

## 维度二：锁粒度（行锁 vs 表锁）

**InnoDB 支持行级锁（默认就是行锁），MyISAM 只有表级锁。**

这个差别在并发写场景下是天壤之别：

- MyISAM 一写就锁整张表，同一时刻只能有一个写操作进行，其它写请求只能排队。表一大、并发一高，直接卡死。
- InnoDB 默认对命中的行加行锁，只锁住相关的那几行记录，其它行照常读写，并发写能力强很多。

这里有个**容易被忽略的细节**，值得点明：InnoDB 的行锁是**加在索引上**的，不是直接加在“数据行”这个物理概念上。如果你的 `UPDATE` / `DELETE` 的 `WHERE` 条件没命中索引，或者索引失效了，InnoDB 会退化成扫全表并把扫到的行都锁上，效果上接近“锁全表”。所以“用了 InnoDB 就有行锁”不是无条件的，**前提是查询走了索引**。这是面试里能体现深度的一句话。

另外补一句：表级锁其实跟引擎无关，InnoDB 和 MyISAM 都支持表锁；真正只有 InnoDB 才有的是**行锁**。

## 维度三：外键

**InnoDB 支持外键，MyISAM 不支持。**

外键能在数据库层面帮你维护引用完整性，但代价是写入时要做额外的约束检查，对性能有损耗，而且会让表之间强耦合，给分库分表、批量数据操作带来麻烦。

所以现实里有个有意思的现象：虽然 InnoDB 支持外键，但很多团队（包括阿里《Java 开发手册》）反而明确**禁止用外键**，把约束放到应用层去做。也就是说，“支持外键”是 InnoDB 的能力，但用不用是另一回事。面试时可以顺带提一句这个取舍，会显得你有实战意识。

## 维度四：MVCC

**InnoDB 支持 MVCC（多版本并发控制），MyISAM 不支持。**

MVCC 的作用是让「读不加锁、读写不互相阻塞」：读操作读的是某个版本的快照，写操作生成新版本，两边不打架，大幅减少加锁带来的等待。InnoDB 正是靠 MVCC + Next-Key Lock，在可重复读隔离级别下很大程度上压住了幻读。

说句实话，拿 MVCC 来对比这两个引擎其实有点“凑数”——MyISAM 连行锁都没有，谈 MVCC 本就无从谈起。MVCC 可以理解成行级并发控制的进阶玩法，是建立在 InnoDB 这套行级机制之上的，MyISAM 自然全都没有。

## 维度五：崩溃恢复

**InnoDB 支持崩溃后安全恢复，MyISAM 不支持——这是 MyISAM 最致命的缺陷。**

InnoDB 有 **redo log（重做日志）**。事务对数据页的修改会先写 redo log（WAL，预写日志思想），即使数据还没真正落盘就断电了，重启时 InnoDB 也能拿 redo log 把已提交的修改重做出来，把数据库恢复到崩溃前的一致状态。

MyISAM 没有这套机制。一旦异常崩溃，数据文件和索引文件很可能对不上、损坏，得靠 `myisamchk` 之类的工具去修复，而且不保证修得回来，丢数据是常事。对生产数据库来说，这一条基本就能直接判 MyISAM 出局了。

## 维度六：索引实现（聚簇 vs 非聚簇）

两者底层都用 **B+Tree** 做索引结构，但叶子节点存什么不一样，这是它们组织数据方式上的根本差异。

- **InnoDB 是聚簇索引**：表数据本身就是按主键组织成的一棵 B+Tree，**数据就存在主键索引的叶子节点里**——索引即数据，数据即索引，二者是一体的。走二级（非主键）索引查询时，叶子节点存的是主键值，需要再拿主键回主键树查一次完整行，这一步叫**回表**。
- **MyISAM 是非聚簇索引**：**索引文件和数据文件是分开的**（物理上对应 `.MYI` 索引文件和 `.MYD` 数据文件）。索引的叶子节点存的是数据行的物理地址（行指针），查到地址后再去数据文件里捞行。它的主键索引和二级索引在这一点上是一样的，都是“索引指向数据地址”，没有聚簇/回表的概念。

一句话记：**InnoDB 索引和数据长在一起，MyISAM 索引和数据分两摊。**

## 维度七：count(\*) 快不快

这是个高频追问点。

- **MyISAM** 用一个变量**单独维护了表的总行数**，执行不带任何 `WHERE` 条件的 `SELECT COUNT(*)` 时直接把这个数字读出来返回，是 O(1) 的，非常快。
- **InnoDB 不维护这个精确总行数**。因为有 MVCC，不同事务在同一时刻看到的“有效行数”可能不一样（有的行对你可见、对别人不可见），根本没法用一个全局变量表示。所以 InnoDB 的 `COUNT(*)` 需要实际去扫描（通常会选一个最小的二级索引来扫），数据量大时明显慢。

要提醒的是：**只有不带 `WHERE` 条件时 MyISAM 才有这个优势**。一旦加了过滤条件，MyISAM 也得老老实实去算，那个缓存的总数就用不上了。这也是 MyISAM 唯一还能拿出来吹一吹的点，但分量很轻。

## 维度八：全文索引及版本变化

老资料常说“全文索引、空间函数是 MyISAM 的特色”，这话**在今天已经过时了，照搬会踩坑**：

- **InnoDB 从 MySQL 5.6 起就支持全文索引（FULLTEXT）了**，5.7 还加上了对中文等 CJK 的 ngram 分词支持；
- 空间索引（GIS / R-Tree）InnoDB 也在 5.7 之后补齐了。

所以“想用全文检索就得选 MyISAM”这个理由现在已经不成立。如果资料里把全文索引当作 MyISAM 的独占优势，那是 5.5 时代的旧说法。当然，真要做认真的全文/中文检索，业界更多还是上 Elasticsearch，而不是纠结 MySQL 哪个引擎。

## 维度九：缓存机制

顺带一提，两者缓存策略也不同：

- **InnoDB 用 Buffer Pool**，**同时缓存数据页和索引页**，所以热点数据能整体留在内存里，读得很快。
- **MyISAM 用 Key Cache**，**只缓存索引页，不缓存数据页**，数据还得靠操作系统的文件缓存兜底。

这也是“数据能放进内存时 InnoDB 往往更快”的一个原因——它把数据也一起缓存了。

## 一张表收尾核心差异

| 对比维度             | InnoDB                                           | MyISAM                           |
| -------------------- | ------------------------------------------------ | -------------------------------- |
| 事务                 | 支持，四个隔离级别，可 commit/rollback           | 不支持                           |
| 锁粒度               | 行锁（默认）+ 表锁                               | 只有表锁                         |
| 外键                 | 支持                                             | 不支持                           |
| MVCC                 | 支持                                             | 不支持                           |
| 崩溃恢复             | 支持，靠 redo log 安全恢复                       | 不支持，易损坏丢数据             |
| 索引与数据           | 聚簇索引，数据存在主键索引叶子上，二级索引需回表 | 索引与数据分离，叶子存数据行地址 |
| `COUNT(*)`（无条件） | 不维护精确行数，需实际扫描                       | 维护总行数，O(1) 直接返回        |
| 全文索引             | 5.6+ 支持 FULLTEXT                               | 支持（早期是其优势）             |
| 缓存                 | Buffer Pool，缓存数据页+索引页                   | Key Cache，只缓存索引页          |
| 默认引擎             | 5.5.5 起至今的默认                               | 5.5 之前的默认                   |

## 为什么现在基本都用 InnoDB

把上面几条收一收就明白了：事务、行锁、崩溃恢复、外键、MVCC，这些撑起一个可靠业务系统的能力，**InnoDB 全有，MyISAM 几乎全没有**。而 MyISAM 当年引以为傲的优势（全文索引、count 快、读性能好），要么已经被 InnoDB 追平甚至反超，要么只在很窄的场景才成立。

别轻信“MyISAM 比 InnoDB 快”这种经验之谈，这个结论往往不成立——尤其当查询用到聚簇索引、或者数据能整体放进内存时，InnoDB 的速度常常让 MyISAM 望尘莫及。再叠加 InnoDB 的读写并发能力随 CPU 核数近乎线性增长（MyISAM 因为读写不能并发，加核数也没用），结论就很清楚了：**日常业务系统，老老实实用默认的 InnoDB 就对了。**

## MyISAM 还有意义吗

说人话：**基本没有了。** 但为了答全，可以提一下它历史上还能勉强站住脚的场景：

- 纯只读、几乎不更新的数据（比如离线导入后只查不改的字典表、日志归档表），不在乎事务和崩溃恢复；
- 频繁执行不带条件的 `COUNT(*)`，想吃 MyISAM 那个 O(1) 计数的红利；
- 对单表存储体积比较敏感（MyISAM 表通常更紧凑一些）。

但这些场景今天都有更好的替代：要快速计数可以维护一张计数表或用缓存；要省空间 InnoDB 也有压缩行格式。所以这些“理由”更多是用来讲清 MyISAM 的历史定位，而不是真的建议你去用它。

## 容易踩的坑

- **“InnoDB 行锁”不是无条件的**：WHERE 没走索引会退化成全表加锁，效果接近表锁。别一口咬定“用了 InnoDB 就一定是行锁”。
- **全文索引≠MyISAM 专属**：InnoDB 从 5.6 起就支持，照搬旧资料说“要全文索引就选 MyISAM”是错的。
- **count(\*) 的快只限无条件查询**：MyISAM 缓存的是“整表总行数”，一加 WHERE 就失效。
- **表锁不是 MyISAM 独有**：InnoDB 也支持表锁，真正独有的是行锁；别把“表锁”当成区分两者的标志。
- **别只背“InnoDB 性能更强”**：性能高低和场景强相关，要能说出“因为聚簇索引/Buffer Pool 缓存数据页”这种原因，而不是一句空话。

## 小结

- InnoDB 支持**事务、行锁、外键、MVCC、崩溃恢复（redo log）**，MyISAM 这些几乎都没有——这是两者最核心的分野。
- 索引实现不同：**InnoDB 是聚簇索引，数据和主键索引长在一起，二级索引要回表**；MyISAM 索引与数据文件分离，叶子节点存数据行地址。
- `COUNT(*)`：MyISAM 维护了精确总行数，无条件计数 O(1) 很快；InnoDB 因 MVCC 不维护精确行数，需要实际扫描。
- 全文索引等老优势已被 InnoDB 追平（5.6+ 支持 FULLTEXT），“要全文检索就用 MyISAM”是过时说法。
- 结论：**默认且推荐 InnoDB**，MyISAM 在现代业务里基本没有使用理由，了解其历史定位即可。
