# count(\*)、count(1)、count(字段) 有什么区别？

> 它们的语义先不一样，性能再不一样；而且 `count(*)` 并不是你想象中那个"读所有字段"的慢家伙。

很多人第一次看到这道题，下意识会觉得 `count(*)` 最慢——因为脑子里把它跟 `select *` 联想到一起了，`select *` 确实要把每行所有字段都读出来。但 `count(*)` 里的那个 `*` 跟 `select *` 完全是两码事，它根本不读字段值。这道题考的就是你有没有真的理解 count 在底层干了什么，而不是望文生义。

下面分两层讲：先讲**语义**（它到底在数什么），再讲**性能**（哪种快、为什么）。

## 先搞清楚 count() 在数什么

count() 是个聚合函数，括号里可以放字段名，也可以放任意表达式（比如数字 `1`、`0`）。它的规则只有一句话：

> 统计"符合查询条件的记录里，括号内这个东西**不为 NULL**"的行数。

关键就在"不为 NULL"这四个字上，几种写法的语义差别全从这里来。

**`count(字段)`** —— 数的是"该字段不为 NULL 的行"。

```sql
select count(name) from t_order;
```

如果某行的 `name` 是 NULL，这行**不算**。所以 `count(name)` 和总行数不一定相等，这是它和其他几种写法最本质的区别，不只是性能问题。

**`count(主键)`** —— 主键不允许为 NULL，所以等于在数总行数。

**`count(1)`** —— 括号里是常量 `1`，它永远不是 NULL，所以也是在数总行数。

```sql
select count(1) from t_order;
```

可以理解成：每扫到一行，就拿"1 是不是 NULL"问一句，答案永远是"不是"，于是计数加一。等价于数总行数。

**`count(*)`** —— 数总行数。注意它不是"把所有字段取出来再数"，MySQL 会把 `*` 这个参数特殊处理掉，等价于 `count(0)`，同样是一个永不为 NULL 的常量。

所以一句话总结语义：**只有 `count(字段)` 会因为 NULL 漏掉行，`count(主键)`/`count(1)`/`count(*)` 都是在老老实实数总行数。**

## 几种写法的性能差在哪

先把结论摆出来（通常情况下，从快到慢）：

```
count(*) ≈ count(1)  >  count(主键)  >  count(普通字段)
```

理解了执行过程，这个排序就很自然了。MySQL 的 server 层会维护一个 count 变量，循环向 InnoDB 要一条条记录，每拿到一条就判断"括号里的东西是不是 NULL"，不是就把变量加一，扫完返回。差别就在**每扫一行要做多少额外的活**。

### count(主键)：要把主键值取出来

server 层向 InnoDB 要记录，InnoDB 把记录返回，server 层得**读出主键的值**，再判断它是不是 NULL（其实主键不可能是 NULL，但这个判断动作还是走了），不是 NULL 就加一。

多了"取出主键值"这一步，所以它比 `count(*)`/`count(1)` 略慢。

### count(1) 和 count(\*)：连字段值都不用取

`count(1)` 里参数是常量 1，不是字段，所以 InnoDB 把记录交给 server 层后，**根本不需要去读记录里的任何字段值**，直接判定"1 不为 NULL"，加一。比 `count(主键)` 省掉了取值那一步，所以更快。

`count(*)` 前面说了，被优化成 `count(0)`，执行路径跟 `count(1)` 基本一模一样，**两者没有性能差异**。MySQL 5.7 官方手册原话就是：

> InnoDB handles SELECT COUNT(`*`) and SELECT COUNT(`1`) operations in the same way. There is no performance difference.

所以那种"`count(*)` 比 `count(1)` 慢"或者反过来的说法，都是没根据的，别背反了。

### 优化器会挑最小的那棵索引树来扫

这里有个容易被忽略但很加分的点：执行 `count(*)`/`count(1)`/`count(主键)` 时，InnoDB **并不一定去扫聚簇索引**。

InnoDB 用 B+ 树存数据，聚簇索引的叶子节点放的是整行数据，而二级索引的叶子节点只放索引列加主键值。因为二级索引"瘦"，同样多的记录占的空间更小，一个数据页能塞下更多条，于是树更矮、扫起来要读的页（IO）更少。

所以只要表上有二级索引，优化器就会去扫 **key_len 最小的那棵二级索引**，而不是又大又重的聚簇索引；只有当表上只有主键、没有任何二级索引时，才退而扫聚簇索引。这也是为什么经常建议：要频繁 count 的表，最好建一个窄的二级索引。

### count(字段)：最惨的那个

```sql
-- name 是普通字段，没建索引
select count(name) from t_order;
```

`name` 上没有索引，优化器没有"瘦索引"可选，只能**全表扫描**（走聚簇索引把每行都读一遍）；而且因为语义是"数 name 不为 NULL 的行"，每行还得真的把 `name` 取出来判断一次是否为 NULL。又是全表扫、又要逐行取值判空，所以它最慢。

> 顺带提一句：如果你确实业务上要"统计某字段非 NULL 的条数"，那就给这个字段建个二级索引，`count(字段)` 也能用上索引扫描，快很多。这时 `count(字段)` 是有它存在意义的，并不是说它一无是处——只是当你的真实意图只是"数总行数"时，没必要用它。

## 为什么 InnoDB 的 count(\*) 不能像 MyISAM 那样秒回

经常有人问：MyISAM 数一张大表的总行数贼快，InnoDB 怎么就得一行行数？

**MyISAM** 不支持事务，它在表的 meta 信息里直接存了一个 `row_count`，由表级锁保证一致。所以**不带 where 的 `count(*)`，MyISAM 直接把这个数读出来返回，是 O(1)**，不用扫表。

**InnoDB** 不行，根子在于它支持事务 + MVCC（多版本并发控制）。同一个时刻，不同事务"应该看到多少行"是不一样的：

举个例子，表里现在 100 条记录。事务 A 刚插入了一行还没提交，事务 B 此刻来数总数——按隔离级别，B 可能看到 100，A 自己看到 101。**同一瞬间，两个事务数出来的总行数就是不同的**。这种情况下根本没法用一个全局计数器来代表"总行数"，因为总行数本身就不是一个全局唯一的值。所以 InnoDB 只能在查询的时候，按当前事务的可见性，实时一行行地数。

补一个细节：**带上 where 条件之后，MyISAM 和 InnoDB 就拉平了**，因为 MyISAM 那个 `row_count` 只对应"整表行数"，一旦有过滤条件，它也得乖乖扫表统计。所以 MyISAM 的优势仅限于"裸的、无条件的 `count(*)`"这一种场景。

## 大表 count 很慢怎么办

一张一千多万行的表，哪怕建了二级索引，`select count(*)` 也可能要好几秒。如果业务里频繁要这个数，几种思路：

**1. 不要精确值，用近似值。** 像搜索结果"约 1,230,000 条"这种场景，根本不需要准。

- `explain select count(*) from t_order;` 看 `rows` 字段——它是优化器的**估算值**，不真正查数据，所以极快。
- 或者 `show table status` 也能拿到一个估算的行数。

注意这俩都是估算，InnoDB 的统计信息是采样得来的，可能跟真实值有不小偏差，只能当"量级参考"。

**2. 要精确值，单独维护一张计数表。** 建一张表专门存这个总数，每次往业务表插入就把计数 +1，删除就 -1。

这里有个**最关键的坑**：这个加减一定要和业务的增删操作**放在同一个事务里**。否则插入成功了、计数没加上（或者反过来），数据就对不上了。放同一事务能保证要么都成功要么都回滚。

**3. 用 Redis 之类的计数器。** 性能最好，但要明确：**Redis 和 MySQL 不在一个事务里，没法保证强一致**。可能出现 MySQL 插入成功而 Redis 加一失败（或者宕机丢了），导致计数偏差。所以它适合"能容忍少量误差"的场景（比如点赞数、阅读量），不适合对账类必须精确的数据。

## 容易踩的坑

- **把 `count(字段)` 当成数总行数。** 它会漏掉该字段为 NULL 的行，语义就不对，不是慢不慢的问题。要数总行数请用 `count(*)`。
- **以为 `count(*)` 会读所有字段所以最慢。** 恰恰相反，它被优化成 `count(0)`，连字段值都不取，是最快的一档。
- **背反了 `count(*)` 和 `count(1)` 谁快。** 它俩在 InnoDB 里没区别，官方手册都明说了，纠结这个没意义。
- **以为有了二级索引 InnoDB 也能 O(1) 数总数。** 二级索引只是让扫描更省 IO，并没有把"逐行数"变成"读一个变量"，本质还是扫。
- **计数表 / Redis 的一致性。** 计数表必须和业务操作同事务；Redis 方案要接受可能不一致。

## 小结

1. **语义上**：`count(字段)` 只数该字段非 NULL 的行；`count(主键)`/`count(1)`/`count(*)` 都是数总行数。
2. **性能上**（通常）：`count(*) ≈ count(1) > count(主键) > count(普通字段)`，差别在于每行要不要取字段值、有没有索引可走。
3. `count(*)` 等价于 `count(0)`，被优化器特殊处理，不读字段、和 `count(1)` 一样快；有二级索引时优化器会挑 key_len 最小的那棵来扫。
4. **InnoDB 数得慢**是因为 MVCC——同一时刻不同事务看到的行数不同，无法用全局计数器，只能实时扫；MyISAM 无事务、靠 `row_count` 变量，无条件 `count(*)` 才能 O(1)，带 where 后两者一样。
5. **大表优化**：能近似就用 `explain` 的 rows 或 `show table status`；要精确就维护计数表（务必同事务）或上 Redis（接受可能不一致）。
