# 死锁是怎么产生的，怎么排查和避免？

> 死锁就是俩事务互相攥着对方想要的锁，谁也不松手，于是一起卡死。MySQL 不会让它们卡到天荒地老，而是会主动挑一个回滚掉。

死锁这个词听着吓人，其实拆开看很简单：事务 A 拿着锁 1 等锁 2，事务 B 拿着锁 2 等锁 1，俩人就这么僵住了。下面我先把概念讲清楚，再用一个 InnoDB 里特别经典、面试也特别爱问的间隙锁死锁例子走一遍时间线，最后说怎么排查和怎么躲开。

## 先说清楚：死锁的四个必要条件

操作系统课本里讲死锁有四个必要条件，背是能背下来，但放到数据库语境里更重要的是理解它们对应什么：

- **互斥**：锁本身就是互斥的，X 锁同一时刻只能被一个事务持有。这条是锁的天性，没法破坏。
- **占有且等待**：事务已经拿了一些锁，还想再拿别的，拿不到就不放手里已有的。对应到 SQL 就是：一个事务里连续执行多条加锁语句，前面拿的锁要等 commit 才释放。
- **不可剥夺**：一个事务持有的锁，别人不能硬抢走，只能等它自己释放。
- **循环等待**：A 等 B、B 等 A（或者更多事务连成一个环），这是死锁的最终形态。

这四条同时成立才会死锁，所以破坏任意一条就能避免。数据库里能动手脚的主要是「循环等待」——这也是后面讲避免方法的核心思路。

## 一个最经典的 InnoDB 死锁：间隙锁 + 插入意向锁

这个例子来自一个很常见的业务：**新增订单前先做幂等校验**。为了防止重复下单，插入之前先查一下这个订单号在不在，不在才插。查的时候用了 `select ... for update`（当前读 + 加锁），目的是防止两个相同订单号的请求同时进来都查到"不存在"然后都插进去。

隔离级别是默认的**可重复读（RR）**，存储引擎是 InnoDB。表长这样：

```sql
CREATE TABLE `t_order` (
  `id` int NOT NULL AUTO_INCREMENT,
  `order_no` int DEFAULT NULL,
  `create_date` datetime DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `index_order` (`order_no`) USING BTREE   -- 普通索引，非唯一
) ENGINE=InnoDB;
```

注意 `order_no` 是**普通（非唯一）索引**，这点很关键，后面会解释为什么换成唯一索引就能避坑。

假设表里现在 `order_no` 最大的记录是 1006。现在两个事务同时进来，事务 A 要插订单 1007，事务 B 要插订单 1008，两边都先做幂等校验。时间线如下：

| 时刻 | 事务 A                                                                        | 事务 B                                                                        |
| ---- | ----------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| T1   | `begin;`                                                                      | `begin;`                                                                      |
| T2   | `select id from t_order where order_no=1007 for update;`（查不到，加锁）      |                                                                               |
| T3   |                                                                               | `select id from t_order where order_no=1008 for update;`（查不到，加锁）      |
| T4   | `insert into t_order(order_no, create_date) values(1007, now());` —— **阻塞** |                                                                               |
| T5   |                                                                               | `insert into t_order(order_no, create_date) values(1008, now());` —— **阻塞** |
| 结果 | 互相等待，死锁                                                                | 互相等待，死锁                                                                |

走一遍发生了什么：

**T2：事务 A 查 1007。** 1007 不存在，且它大于表里最大的 1006，所以在二级索引 `index_order` 上加了一个 **X 型 next-key 锁，范围是 `(1006, +∞]`**。

**T3：事务 B 查 1008。** 1008 也不存在，也大于 1006，所以它也想在 `(1006, +∞]` 这个范围加 X 型 next-key 锁。**关键来了：这一步事务 B 没有被阻塞，加锁成功了。** 为什么？因为 next-key 锁 = 间隙锁 + 记录锁，而这个范围的右边界是 `+∞`（supremum，一个伪记录，不是真实的行），所以实际上这里只有间隙锁部分在起作用，没有真实记录上的记录锁要去争。**而间隙锁之间是相互兼容的**——两个事务可以同时持有覆盖同一段间隙的间隙锁。所以 A 和 B 现在都握着 `(1006, +∞]` 的间隙锁，相安无事。

**T4：事务 A 想插入 1007。** 插入操作在插进去之前，需要先在目标间隙上拿一个**插入意向锁（insert intention lock）**。插入意向锁本质上是一种特殊的间隙锁，但它有个脾气：**它要等到这段间隙上别的事务持有的间隙锁释放之后才能拿到**。现在事务 B 在 `(1006, +∞]` 上还持着间隙锁没释放（B 没 commit），所以事务 A 拿不到插入意向锁，**被阻塞，开始等 B 释放间隙锁**。

**T5：事务 B 想插入 1008。** 同理，B 要在 `(1006, +∞]` 上拿插入意向锁，但事务 A 在这段间隙上的间隙锁也没释放，**B 也被阻塞，开始等 A 释放间隙锁**。

到这里就成环了：A 等 B 的间隙锁，B 等 A 的间隙锁，谁都不会先 commit，**死锁形成**。

这个例子的精髓，我把它压成一句话记住：**间隙锁之间不冲突（能共存），但插入意向锁和间隙锁冲突（要排队）**。正是这个"进的时候大家都能进，但要插入时谁也插不动"的错位，造出了死锁。

## 把根因再抠细一点

很多人到这里会迷糊：既然 A 和 B 都拿到了同一段间隙的间隙锁，那为啥插入又互相挡住了？这里要分清两类锁的关系。

**间隙锁 vs 间隙锁：兼容。** MySQL 官方文档说得很直白，间隙锁是"纯抑制性的"（purely inhibitive），它唯一的作用就是**阻止别人往这段间隙里插数据**，而不是阻止别人也来观察这段间隙。所以共享间隙锁和排他间隙锁没有区别，多个事务可以同时持有覆盖同一间隙的间隙锁。

**间隙锁 vs 插入意向锁：冲突。** 插入意向锁虽然名字里带"意向"，但它不是表级意向锁，而是一种特殊的间隙锁，专门给插入操作用。它的规则是：当一个事务想往某个间隙插入时，如果这段间隙上已经有别的事务的间隙锁，它就得等那个间隙锁释放，才能拿到插入意向锁。

所以这类死锁的标准配方是：**两个（或多个）事务先各自在同一段间隙上拿到兼容的间隙锁，然后又都想往这段间隙里 insert，于是都卡在"等对方释放间隙锁"上**。

补充一个容易被忽略的点（资料里讲了，但值得单独强调）：**`(1006, +∞]` 这种以 `+∞` 结尾的 next-key 锁，两个事务能同时持有，不冲突**。因为 `+∞` 不是真实记录，没有记录锁要争。但如果是 `(1, 10]` 这种右边界是真实记录的 next-key 锁，两个事务想同时拿同范围的 X 型 next-key 锁就会冲突了——因为记录 10 上的 X 记录锁是互斥的。这也是为什么上面那个例子能"两边都先加锁成功"再一起卡在插入上。

## InnoDB 自己怎么处理死锁

死锁如果不管，事务就会永远卡着。InnoDB 有两道防线：

**第一道：等待超时。** 参数 `innodb_lock_wait_timeout`，**默认 50 秒**。一个事务等锁超过这个时间，就会被回滚，报 `Lock wait timeout exceeded`，它持有的锁随之释放，对方就能继续了。这道防线很被动——要傻等 50 秒，线上基本不能靠它兜底，通常会把它调小（比如几秒）。

**第二道：主动死锁检测。** 参数 `innodb_deadlock_detect`，**默认就是开启（on）**。InnoDB 内部维护一张等待关系图（wait-for graph），每当有事务要等锁，就检查会不会形成环，一旦发现环，立刻挑其中一个事务回滚掉，让其他事务继续。被选中的通常是**代价小（修改行数少、回滚成本低，也就是权重小）的那个事务**。报错信息是：

```
ERROR 1213 (40001): Deadlock found when trying to get lock; try restarting transaction
```

注意一点：死锁检测本身是有开销的，每个新来等锁的事务都要去扫一遍等待图。在**热点行高并发更新**的场景下（成千上万个事务等同一行），这个检测会消耗大量 CPU，反而成了瓶颈。这种极端场景下，有人会选择关掉死锁检测、靠超时兜底，或者用队列/中间件把对热点行的更新串行化。这不是常规操作，知道有这么回事就行。

## 怎么排查死锁

死锁发生后（哪怕已经被自动回滚），现场是能查到的。常用三招：

**1）`show engine innodb status`。** 输出里有一段 `LATEST DETECTED DEADLOCK`，记录了**最近一次**死锁的详细现场：涉及哪两个事务、各自执行的 SQL、持有什么锁、在等什么锁、最后回滚了谁。这是排查死锁最直接的手段。注意它只保留最近一次，所以生产上最好开启 `innodb_print_all_deadlocks=on`，把每次死锁都打进 error log，免得现场被覆盖。

**2）`performance_schema.data_locks`。** 这张表能看到当前每个事务持有/请求了哪些锁、锁在哪个索引上、锁的模式（next-key / gap / record / insert intention）。配合 `LOCK_MODE` 字段判断锁类型：

- `X`：X 型 next-key 锁
- `X, REC_NOT_GAP`：X 型记录锁
- `X, GAP`：X 型间隙锁
- `X, INSERT_INTENTION`：X 型插入意向锁

（顺便纠个常见的误读：`LOCK_TYPE` 里的 `RECORD` 指的是"行级锁"，不是"记录锁"，别看到 RECORD 就以为是记录锁，具体是哪种行锁要看 `LOCK_MODE`。）

**3）`performance_schema.data_lock_waits`。** 这张表直接给出"谁在等谁"的阻塞关系，配合 `data_locks` 一起看，能快速定位等待环。这是死锁正在发生（还没被回滚）时定位现场的利器。

## 怎么避免死锁

死锁的检测和超时都是"出事之后补救"，更重要的是从源头降低概率。几个实用方向：

- **让所有事务按相同顺序访问资源。** 这是破坏"循环等待"最直接的办法。如果所有事务都先锁 A 再锁 B，就不会出现 A 等 B、B 等 A 的环。
- **缩短事务、尽早提交。** 事务越短，持锁时间越短，撞上的概率越低。别在事务里夹长耗时操作（远程调用、大计算）。
- **加锁查询一定要走索引。** 如果 `where` 没命中索引，会全表扫描，给扫到的每一行加行锁，连行与行之间的间隙也锁上，相当于锁了整张表，死锁和阻塞概率飙升。线上尤其别执行不带索引条件的 `update`。
- **用唯一索引代替"先查后插"的幂等校验。** 回到开头那个订单例子：与其用 `select ... for update` 占着间隙锁去校验，不如直接把 `order_no` 设成**唯一索引**，靠它的唯一性保证不重复。重复插入时数据库会直接报唯一键冲突，业务捕获异常即可。这样就不会去抢那段 `(1006, +∞]` 的间隙锁了，从根上消除了这类死锁。代价是要处理插入异常，但比死锁好得多。
- **必要时把隔离级别降到 RC（读已提交）。** RC 下**没有间隙锁**（除了外键检查、唯一键冲突等少数场景），这类"间隙锁互等"的死锁基本就消失了。很多互联网公司线上默认就用 RC + row 格式 binlog，正是出于减少锁范围、提升并发的考虑。当然降隔离级别要评估业务能不能接受不可重复读/幻读。
- **让应用层有重试机制。** 死锁被回滚后，那条报错明确说了 "try restarting transaction"，应用捕获到 1213 错误就重试一次，通常就过去了。死锁无法 100% 杜绝，能优雅重试也是一种"避免影响"。

## 容易踩的坑

- **以为"两个查询都不冲突就不会死锁"。** 恰恰相反，这个经典例子里两个 `select ... for update` 都加锁成功、互不阻塞，死锁是卡在后面的 `insert` 上的。间隙锁兼容、插入意向锁不兼容，是这类死锁最反直觉的地方。
- **把 `data_locks` 里的 `RECORD` 当成记录锁。** `LOCK_TYPE=RECORD` 只表示"行级锁"，到底是 next-key、gap 还是 record，得看 `LOCK_MODE`。
- **以为死锁检测默认是关的。** `innodb_deadlock_detect` 默认开启，绝大多数死锁会被秒级回滚并报 1213，而不是傻等 50 秒超时。会傻等超时的，往往是"锁等待"而不是真正的"死锁"，别把两者搞混。
- **`order_no` 是非唯一索引这件事被忽略。** 正因为是非唯一索引，查不存在的值才会留下间隙锁；换成唯一索引，加锁行为和避坑方式都不一样。讲这个例子时一定要交代清楚索引类型。
- **RR 下才有这类间隙锁死锁，RC 下基本没有。** 讨论死锁一定要先问清楚隔离级别，脱离隔离级别谈间隙锁是没意义的。

## 小结

- 死锁 = 多个事务互相持有对方想要的锁、形成循环等待而僵死；四个必要条件中数据库主要从"破坏循环等待"入手。
- 最经典的 InnoDB 死锁：RR 下两个事务各用 `select ... for update` 锁住同一段间隙（间隙锁兼容，都能成功），随后都想往里 `insert`，而插入意向锁与间隙锁冲突，于是互相等待成环。
- InnoDB 两道防线：`innodb_lock_wait_timeout`（默认 50 秒，被动超时）和 `innodb_deadlock_detect`（默认开启，主动检测后回滚代价小的事务，报 1213）。
- 排查靠 `show engine innodb status` 的 `LATEST DETECTED DEADLOCK` 段，以及 `performance_schema.data_locks` / `data_lock_waits`；生产建议开 `innodb_print_all_deadlocks`。
- 避免靠：统一加锁顺序、缩短事务、加锁查询走索引、用唯一索引替代"先查后插"、必要时降到 RC，并在应用层做重试。
