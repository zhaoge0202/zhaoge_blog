# MySQL 主从复制是怎么工作的，为什么会延迟？

> 上一篇讲了 binlog 是 Server 层的归档日志，这一篇接着说它最重的用途之一：主从复制。复制能扩读、能备份、能容灾，但它「能保证什么、不能保证什么」是面试最容易被追问的地方。

很多人答主从复制就停在「主库写 binlog，从库回放」，一追问「从库具体靠哪几个线程」「为什么会有延迟」「半同步到底同步了什么」就接不住。这篇把复制的线程模型、三种方式、GTID 和延迟成因一次讲透。

## 复制是建立在 binlog 上的

主从复制的本质是：**主库把变更写进 binlog，从库把 binlog 拿过来重新执行一遍**。所以 binlog 是复制的唯一数据载体——这也是为什么[日志篇](./mysql-logs.html)里强调 binlog 的职责是「归档和复制」，而 redo log / undo log 都不会跨实例传。

binlog 的三种格式（STATEMENT / ROW / MIXED）对复制影响很大，后面专门讲。这里先记住：复制走的是 binlog，不是 redo log。

## 复制的三个线程

复制靠三个线程协作完成，主库一个、从库两个：

```mermaid
flowchart LR
    M[主库 binlog] -->|1. dump thread 读取并发送| N[网络]
    N -->|2. IO thread 接收| R[从库 relay log 中继日志]
    R -->|3. SQL thread 读取并回放| D[从库数据]
```

- **主库的 dump 线程**：监听 binlog 的更新，有新事件就读出来发给从库。它读 binlog 是「读出去就记下位点」，不会阻塞主库事务提交。
- **从库的 IO 线程**：接收主库发来的 binlog 事件，写到本地的 **relay log（中继日志）**。relay log 的格式和 binlog 一样，相当于从库这边的一份 binlog 副本。
- **从库的 SQL 线程**：读 relay log，把里面的事件在从库上重新执行一遍，于是从库的数据就跟着主库变了。

注意分工：**IO 线程只负责「拉日志落盘」，SQL 线程才负责「回放」**。这两步是异步解耦的——relay log 落盘了，不代表数据已经追上；要等 SQL 线程把 relay log 回放完，从库数据才真正一致。这也是主从延迟的根源。

## 三种复制方式：异步、半同步、组复制

按「主库提交事务时，要不要等从库确认」分三种：

| 方式                    | 主库提交时等不等从库                                           | 一致性 / 代价                                    |
| ----------------------- | -------------------------------------------------------------- | ------------------------------------------------ |
| 异步复制（默认）        | 不等，提交完就返回                                             | 性能最好；主库挂了且没传出去的 binlog 会丢       |
| 半同步复制（semi-sync） | 至少等一个从库收到 binlog（收到 relay log 即可，不要求回放完） | 主库挂了数据更安全；多一次网络 RTT，且超时会降级 |
| 组复制（MGR）           | 多数节点达成共识（Paxos 类协议）                               | 准强一致、能自动选主；部署复杂、写性能有损耗     |

几个边界要点：

- **异步是默认行为**，开箱即用，但「主库提交成功 ≠ 从库一定收到」。主库突然宕机，还在主库 binlog 缓存里、没传给从库的事务就丢了。这是「异步复制会丢数据」的真正含义。
- **半同步复制「同步」的是「binlog 传到从库的 relay log」，不是「从库回放完成」**。所以半同步也不能算强一致——从库收到日志但还没回放完时，主库挂了，从库虽然能靠 relay log 补上，但回放期间数据有短暂不一致。
- **半同步有超时降级**：如果等从库 ACK 超时（`rpl_semi_sync_master_timeout`，默认 10 秒），主库会自动降级回异步，避免一直卡住。所以半同步是「尽量同步，必要时退让」，不是绝对保证。
- 组复制（MGR）是 MySQL 5.7.17 引入、5.7.17+ / 8.0 成熟的方案，靠共识协议做到多数节点一致，是这三种里最接近强一致的，但也最重。

## binlog 格式对复制的影响

binlog 三种格式对复制的安全性和体积影响不同（格式的完整对比见[日志篇](./mysql-logs.html)，这里只讲复制视角）：

- **STATEMENT**：记原始 SQL。日志小，但用了 `now()`、`uuid()`、`user()` 这类**不确定函数**时，主从执行结果会不一致，导致主从数据漂移。所以生产基本不用它。
- **ROW**：记每行被改成了什么样。安全、确定，是 5.7.7 之后的默认值；缺点是批量 update 会让 binlog 暴涨。
- **MIXED**：平时用 STATEMENT，遇到不确定函数自动切 ROW，是个折中。

> 一句话：复制场景下优先 **ROW**。资料里偶尔还能看到推荐 STATEMENT 的旧说法，那是早期默认值还没切换时的建议，现在照搬会踩数据不一致的坑。

## GTID：用全局事务 ID 替代位点

传统复制靠「文件名 + 位点」（`master_log_file` / `master_log_pos`）告诉从库从哪开始拉，主库一切换、binlog 文件一滚动，位点就对不上了，运维很痛。

**GTID（Global Transaction Identifier）** 给每个事务分配一个全局唯一 ID（`server_uuid:事务序号`），从库只要告诉主库「我已经执行到哪个 GTID」，主库就自动把后面没传的发过来。

- 5.6 引入，5.7 完善，8.0 推荐默认开启。
- 好处：自动定位复制位点、故障切换时不容易丢事务或重放、复制拓扑变更更安全。
- 注意：开了 GTID 后，有些用法受限（比如 `create table ... select` 在某些场景被禁），升级时要先评估。

## 主从延迟是怎么来的，怎么治

主从延迟的根因就一句：**SQL 线程回放的速度，跟不上主库产生 binlog 的速度**。具体拆开：

- **大事务**：主库一条 update 改了百万行，binlog 巨大，从库 SQL 线程要回放很久。
- **SQL 线程单线程回放**：早期从库只有一个 SQL 线程串行回放，主库可是多线程并发写的，天然追不上。
- **网络**：主从之间带宽不足或延迟高，IO 线程拉日志就慢。
- **从库自身慢**：从库硬件差、有慢查询、被大查询占着资源，回放就慢。
- **锁冲突**：从库上跑的读业务持有锁，阻塞了 SQL 线程的回放。

治理手段：

- **并行复制（multi-threaded slave, MTS）**：让从库用多个 worker 线程并发回放。5.6 的并行是「按库并行」，粒度太粗；**5.7 引入基于组提交的 logical clock 并行**，同一组里提交的事务可以并发回放，大幅提升回放速度；8.0 默认 `WRITESET` 模式进一步放宽并行度。这是治延迟最有效的手段。
- **拆大事务**：把百万行的批量操作切成小批次，避免单事务 binlog 过大。
- **降低从库读压力**：从库别扛重查询，或者多挂几个从库分担。
- **网络与硬件**：保证主从带宽、从库 IO 能力别成为瓶颈。

> 纠正一个常见说法：「开了并行复制就没有主从延迟了」。不对。并行复制提升的是回放吞吐，但如果主库写入持续高于从库回放上限、或有无法并行的大事务，延迟照样会有。它「缓解」而不是「消除」延迟。

## 复制能保证什么，不能保证什么

这是面试最容易答模糊的地方，必须讲清边界：

- **异步复制**：最终一致。主库提交即返回，不保证从库已收到。主库宕机可能丢数据。**读从库可能读到旧数据**，这就是「主从读写分离后的一致性问题」。
- **半同步复制**：保证 binlog 至少到了一个从库的 relay log，比异步安全，但不是强一致（没等回放完、超时会降级）。
- **组复制**：多数节点共识，接近强一致，但代价是复杂度和写性能。

所以「主从复制 = 高可用 + 数据安全」这个说法是有条件的：异步复制提供的是**最终一致**，不是强一致。要做强一致读写，要么走组复制，要么在应用层做「关键写读走主库」的分流——这部分属于读写分离的工程实践，不在这篇展开。

## 容易踩的坑

- **把半同步当成强一致**：半同步只保证「日志到了从库」，不保证「回放完成」，还会超时降级回异步。
- **以为从库只有一个线程**：从库有 IO 和 SQL 两个线程；SQL 线程在 5.7+ 还能并行（MTS）。
- **用 STATEMENT 格式还期望主从一致**：`now()`/`uuid()` 这类函数会让主从结果分叉，复制场景用 ROW。
- **以为并行复制消除延迟**：它只是提升回放吞吐，大事务和高写入下延迟仍可能存在。
- **开 GTID 不评估兼容性**：GTID 会禁用部分语法（如某些 `create table ... select`），升级前要验证。
- **读写分离直接读从库不判断延迟**：写完立刻读从库可能读到旧值，关键链路要读主库或校验延迟。

## 小结

- 复制建立在 **binlog** 上：主库 dump 线程发 binlog，从库 IO 线程写 relay log，SQL 线程回放。
- 三种方式：**异步（默认，可能丢数据）**、**半同步（保证日志到达，非强一致，会降级）**、**组复制（共识，准强一致）**。
- binlog 格式影响复制：**ROW** 最安全，STATEMENT 有不确定函数会主从不一致。
- **GTID** 用全局事务 ID 替代位点，5.7+ 完善，故障切换更安全，但有语法限制。
- 主从延迟根因是「SQL 线程回放追不上主库写入」；**5.7 基于组提交的并行复制**是最有效的治理手段，但只能缓解不能消除。
- 异步复制是**最终一致**不是强一致；读写分离读从库要警惕读到旧数据。
