# 分页查询怎么写？深度分页怎么优化？

> 分页是几乎所有列表页都有的需求，`LIMIT` 写起来简单，但翻到深页时性能会断崖式下跌——`LIMIT 1000000, 10` 这种"深度分页"是线上慢查询的常客。怎么写分页、为什么深了会慢、有哪几种解法，是 SQL 工程化的必考题。

## 基础分页：LIMIT 的两种写法

`LIMIT` 限制返回行数，分页靠它配 `OFFSET` 实现。两种等价写法：

```sql
-- 写法一：LIMIT 偏移量, 行数
SELECT * FROM user_info ORDER BY register_time LIMIT 6, 3;

-- 写法二：LIMIT 行数 OFFSET 偏移量
SELECT * FROM user_info ORDER BY register_time LIMIT 3 OFFSET 6;
```

每页 3 条、取第 3 页（页码从 1 开始），偏移量 = (页码-1) × 每页条数 = `6`。`ORDER BY` 保证顺序稳定，分页必须带排序，否则翻页时数据可能错乱、重复。

这套写法在浅页（前几页）毫无问题，问题出在"深页"。

## 深度分页为什么慢

`LIMIT 1000000, 10` 的意思不是"直接跳到第 100 万行取 10 条"，而是**先扫描并丢弃前 100 万行，再返回接下来的 10 条**。

这个"扫描并丢弃"的代价很大：

- 即使有索引，MySQL 也得沿着索引**遍历 100 万行**，而且每行还要**回表**（回到主表取 `SELECT *` 要的字段），回表完才发现这行要被丢弃。
- 100 万次回表后，真正返回的只有 10 行——绝大部分 IO 都白做了。
- 优化器一看回表成本太高，可能干脆放弃索引转全表扫描，更慢。

所以深分页慢的根子是：**OFFSET 越大，要"白白扫描并丢弃"的行越多，回表开销越大。** 这和有没有索引关系不大，索引救不了"跳过大量行"这件事。

## 解法一：游标分页（范围查询）

最彻底的解法是**不用 OFFSET**，改用"上一页最后一条的值"当条件，直接定位下一页：

```sql
-- 第一页
SELECT id, name, register_time FROM user_info
ORDER BY id LIMIT 10;

-- 后续页：拿上一页最后一条的 id 当游标
SELECT id, name, register_time FROM user_info
WHERE id > #{last_id}        -- 上一页最后一条的 id
ORDER BY id LIMIT 10;
```

这样无论翻到第几页，`WHERE id > last_id LIMIT 10` 都只扫描 10 行，性能恒定，和页深无关。

代价是**不能跳页**——你只能"下一页""上一页"地翻，没法直接跳到第 1000 页，因为不知道那页的游标值。所以它适合 feed 流、无限滚动这类"只往下翻"的场景，不适合后台管理那种"跳到任意页"的需求。

> 多列排序时游标要带上全部排序列，比如 `WHERE (create_time, id) < (last_time, last_id)`，否则游标定位不准。

## 解法二：延迟关联

需要支持传统 `LIMIT offset, size` 跳页时，用"延迟关联"减少回表。思路是：**先在索引上只查主键，拿到这页的主键，再拿主键去关联原表取字段。**

```sql
SELECT u.*
FROM   user_info u
INNER JOIN (
    SELECT id FROM user_info ORDER BY register_time LIMIT 1000000, 10
) t ON u.id = t.id;
```

关键是子查询里只 `SELECT id`——如果 `id` 是主键、`register_time` 上有索引，这个子查询能走**覆盖索引**（只读索引不回表），快速定位到这 10 个主键；然后再用这 10 个主键去原表精确取行，回表只有 10 次而不是 100 万次。

对比一下：

|          | 直接 `LIMIT offset, n`    | 延迟关联             |
| -------- | ------------------------- | -------------------- |
| 回表次数 | offset + n 次（大量丢弃） | n 次（只回表目标行） |
| 适用     | 浅页                      | 需要传统跳页的深页   |
| 复杂度   | 简单                      | 多一层子查询 JOIN    |

延迟关联保住了"能跳页"的体验，又把回表成本压到最小，是大多数后台分页的首选。

## 两种解法怎么选

| 方案     | 核心思路                     | 适用场景          | 限制                     |
| -------- | ---------------------------- | ----------------- | ------------------------ |
| 游标分页 | `WHERE id > last_id LIMIT n` | feed 流、无限滚动 | 不能跳页                 |
| 延迟关联 | 子查询取主键再 JOIN 取字段   | 后台、需要跳页    | SQL 稍复杂，依赖覆盖索引 |

一句话：**能接受"只往下翻"就用游标分页（性能最好且恒定）；必须支持跳页就用延迟关联。**

## 业务层先想清楚：真需要深分页吗

技术方案之外，先问业务：用户真的会翻到第 10 万页吗？很多场景下深分页本身就是伪需求，几个常见处理：

- **限制最大翻页数**：只允许翻前 100 页，更深的让用户加筛选条件缩小范围。
- **改用搜索引擎**：海量数据的检索分页本就不该压在关系库上，Elasticsearch 这类更适合。
- **强制带筛选条件**：不限条件的全表分页最容易出深分页，加上时间范围、分类等过滤能极大缩小结果集。

## 容易踩的坑

- **分页不带 ORDER BY**：顺序不稳定，翻页会丢数据、出重复，分页必须排序。
- **OFFSET 越大越慢**：不是"直接跳到第 N 行"，而是"扫描丢弃前 N 行"，索引救不了。
- **游标分页多列排序要带全排序列**：只带 `id` 不带 `create_time` 会定位错。
- **延迟关联依赖覆盖索引**：子查询能走覆盖索引才省，否则退化，用 [EXPLAIN](../mysql/mysql-explain.html) 验证。
- **深分页先评估业务必要性**：可能限制翻页深度或转搜索引擎更合理。

## 小结

- 基础分页：`ORDER BY ... LIMIT offset, size`，浅页没问题。
- 深度分页慢的根因：OFFSET 要扫描并丢弃大量行，回表开销随页深线性增长，索引也救不了。
- 游标分页（`WHERE id > last_id LIMIT n`）性能恒定但不能跳页，适合 feed 流。
- 延迟关联（子查询取主键再 JOIN）保住跳页能力、把回表压到目标行数，适合后台。
- 深分页先从业务层评估：限制翻页深度、加筛选、转搜索引擎往往比硬优化 SQL 更划算。
