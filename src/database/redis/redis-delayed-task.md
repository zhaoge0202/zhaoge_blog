# Redis 延时任务怎么做更稳？

> Redis 能做延时任务，但真正要答好的不是“能不能做”，而是“哪种做法有什么坑，为什么很多项目最后会转向更稳的方案”。

延时任务特别常见，比如：

- 订单超时未支付自动取消
- 一段时间未接单自动关闭
- 优惠券到点失效后触发后续动作

很多人第一反应都是：“给 key 设个 TTL，不就行了？”  
这只能算开了个头，因为延时任务最怕的不是“不会触发”，而是：

- 触发不准时
- 消息丢了
- 多实例重复消费

## 先说结论：Redis 里常见有三条路

| 方案                  | 思路                               | 主要问题                        |
| --------------------- | ---------------------------------- | ------------------------------- |
| 过期事件              | key 到期后监听事件                 | 时效性和可靠性都一般            |
| ZSet 扫描             | score 存触发时间，轮询拉取到期任务 | 要自己处理轮询、并发和幂等      |
| Redisson DelayedQueue | 基于 ZSet 和就绪队列封装           | 比手写稳，但本质仍受 Redis 约束 |

如果你只是问“Redis 能不能做”，答案是能。  
如果你问“核心业务建议怎么做”，答案往往会变成：**能不用 Redis 单独硬扛，就别让它单独硬扛。**

## 第一种：基于过期事件监听

这是最容易想到的方案。

做法大概是：

1. 往 Redis 里写一个带 TTL 的 key
2. 订阅过期事件频道
3. key 过期时消费对应事件

看起来很顺，问题也很致命。

## 为什么过期事件方案不稳

### 1. 它不是“到点必触发”

这是最关键的坑。

Redis 的过期删除不是精确定时器模型，而是：

- 惰性删除
- 定期删除

所以“key 到期”不等于“这一秒就立刻删掉并发事件”。  
也就是说，**它天然不适合对触发时效要求很强的延时任务。**

### 2. 基于 Pub/Sub，消息可能丢

过期事件通知底层依赖的是发布订阅。  
如果当时消费者不在线，或者网络抖动，这条事件就过去了，不会帮你补。

这就决定了它更适合：

- 提醒型
- 弱可靠型

而不适合强依赖执行成功的链路。

### 3. 多实例下容易重复处理

如果多个服务实例都在订阅同一个事件通道，那谁来保证只处理一次？  
你可以自己补锁、幂等、去重，但补着补着就会发现复杂度上来了。

## 第二种：基于 ZSet 手写延时队列

这条路在工程里更常见一点。

思路是：

1. 用 `ZSet` 存任务
2. `score` 存“应该触发的时间戳”
3. 后台线程定期扫 `score <= now` 的任务
4. 抢到任务后再执行业务

它比过期事件好的地方是：

- 触发时间由你自己控制
- 不依赖 Pub/Sub 的瞬时投递
- 队列数据本身可以持久化在 Redis 里

## 但 ZSet 方案也不是免费午餐

### 1. 你要自己处理轮询

你得决定：

- 多久扫一次
- 一次扫多少
- 扫太勤会不会空转浪费
- 扫太慢会不会延迟触发

### 2. 你要自己处理并发抢占

多个实例一起扫时，谁拿到任务？  
你要补：

- 原子抢占
- 去重
- 幂等

否则就容易出现同一个任务被多个实例同时消费。

### 3. 你还要自己处理补偿

比如任务拿到了，但业务执行失败怎么办？  
这时你要决定：

- 重试
- 回队列
- 记录失败状态

这已经不是一个简单的数据结构问题，而是在手搓任务系统。

## 第三种：用 Redisson DelayedQueue

这类方案本质上还是基于 Redis，但比自己手写更稳一点。

它通常会把：

- 延时集合
- 到期搬运
- 就绪队列

这一整套机制封起来，你不必自己从零拼。

相比过期事件监听，它的优势更明显：

- 不依赖过期事件即时触发
- 不那么容易丢消息
- 多实例消费模型更顺手

但这里也要把边界说清：

**它只是把 Redis 延时队列做得更工程化，不是把 Redis 变成了专业延时消息系统。**

## 那为什么很多项目最后还是会选专业 MQ

因为延时任务一旦进入核心业务，就会很在意这些问题：

- 任务绝不能漏
- 重试要可控
- 消费失败要可追踪
- 堆积时要能扛
- 多实例消费要稳定

这时专业 MQ 的延时消息、重试、死信、消费治理能力，通常会比 Redis 单独实现更稳。

所以更务实的判断是：

- 轻量、低成本、已有 Redis：可以考虑 Redis 方案
- 核心链路、可靠性要求高：优先考虑 MQ 的延时消息能力

## 一个比较稳的面试回答方式

如果面试官问“Redis 怎么做延时任务”，比较稳的回答通常是：

1. 先说常见有过期事件和 ZSet 两类方案。
2. 过期事件实现简单，但时效性和可靠性都一般，不适合严肃业务。
3. ZSet 更常见，可以自己控制到期扫描，但要补并发抢占、幂等和失败补偿。
4. 实战里更倾向用 Redisson 这类成熟封装，而不是手写一整套。
5. 如果业务对可靠性要求更高，最终还是会优先用专业 MQ 的延时消息。

## 容易踩的坑

### “TTL 到了，任务就会准时触发”

不对。Redis 过期机制不是精确定时器。

### “ZSet 做延时任务已经足够等于 MQ 延时消息”

不成立。ZSet 只是存储和排序手段，不天然等于完整的任务执行保障体系。

### “用了 Redisson 就没有边界了”

也不对。它只是帮你封掉了一部分复杂度，底层还是 Redis 的能力模型。

## 小结

- Redis 可以做延时任务，但不同方案稳定性差很多。
- 过期事件方案最大的问题是触发不准时、消息可能丢、还容易重复消费。
- ZSet 是更常见的实现路径，但你要自己处理扫描、抢占、幂等和补偿。
- Redisson DelayedQueue 比手写稳，但本质上仍是 Redis 方案，不等于专业延时消息系统。
- 轻量场景可以用 Redis，核心链路更稳妥的选择通常还是专业 MQ。
