# 过期删除和内存淘汰到底有什么区别？

> 很多人把“key 过期了”和“Redis 内存满了所以把 key 踢掉了”混成一回事，但这其实是两套完全不同的机制。

这道题之所以常考，是因为它看起来像概念题，实际上很容易连到线上故障：

- 为什么明明设置了 TTL，内存还是不下来？
- 为什么有些 key 没过期却被 Redis 干掉了？
- 为什么缓存说是“永久有效”，结果线上还是读不到？

想把这些问题讲清楚，先要把两个概念拆开。

## 先把一句话记死

| 机制     | 触发条件                        | 解决的问题         |
| -------- | ------------------------------- | ------------------ |
| 过期删除 | key 到了过期时间                | 删除已经失效的数据 |
| 内存淘汰 | Redis 内存达到 `maxmemory` 上限 | 腾内存继续服务     |

所以：

- 过期删除是在管“这个 key 还该不该活着”
- 内存淘汰是在管“内存不够了，先牺牲谁”

这俩不是一回事。

## 过期删除：key 到点了，Redis 怎么处理

Redis 支持给 key 设置 TTL，比如：

- `EXPIRE`
- `PEXPIRE`
- `SET key value EX 60`

一旦你给 key 设置了过期时间，Redis 就得回答一个问题：

**这个 key 到时间后，什么时候真正删？**

## 为什么 Redis 不会用“到点就立刻删”的最理想方案

如果每个 key 都在设置过期时挂一个定时器，到点立刻删，看起来最干净。

但这套方案的问题是：

- key 多的时候，定时器管理成本高
- 大量过期时间接近的 key 会突然吃掉 CPU

所以 Redis 没选“纯定时删除”，而是走折中路线。

## Redis 的过期删除，核心是“惰性删除 + 定期删除”

### 1. 惰性删除

意思是：**你来访问这个 key 时，我才顺手检查它是不是已经过期。**

如果过期了，就删掉，再返回不存在。

优点：

- 对 CPU 友好

缺点：

- 如果一个 key 早就过期了，但一直没人访问，它还会继续占着内存

### 2. 定期删除

意思是：Redis 会周期性地抽样检查一批带过期时间的 key，把其中已经过期的删掉。

优点：

- 能主动清理一部分没人访问但已经过期的 key

缺点：

- 它不是全量扫，而是抽样扫，所以不会保证“到点立刻清空”

所以 Redis 的思路很现实：

- 访问时顺手删一波
- 平时后台再抽样清一波

这样能在 CPU 和内存之间找平衡。

## 为什么“key 过期了”不等于“它已经从内存里消失了”

这就是线上最容易误解的地方。

比如一个 key 设置了 10 分钟 TTL，10 分钟一到，它在逻辑上已经失效了，但：

- 如果没人访问它
- 且后台抽样又暂时没扫到它

那它可能还会在内存里待一会儿。

所以“过期”更准确地说是：

**这个 key 已经没有业务价值了，但不一定在物理上立刻被清掉。**

这就是为什么有些人看到 TTL 到了，内存却没立刻掉下来，会误以为 Redis 出 bug。

其实不是 bug，是策略。

## 内存淘汰：不是 key 过期了，而是 Redis 内存顶不住了

如果你配置了 `maxmemory`，那么 Redis 到达这个上限后，就得决定：

**是报错拒绝写入，还是踢掉一部分 key 腾位置？**

这时启动的就是内存淘汰策略。

注意，这和过期删除完全不是同一个触发条件。

### 一个 key 没过期，也可能被淘汰

这是最该记住的点。

比如你有个热点缓存，TTL 设置的是 24 小时，本来还远没到期，但 Redis 内存打满了，淘汰策略又刚好选中了它，那它照样会被踢掉。

所以业务上如果你把“有 TTL”理解成“在 TTL 到之前一定还在”，这个假设是不成立的。

## 常见淘汰策略该怎么理解

真正面试里不一定会让你把所有策略都一字不差背全，但至少要知道大方向。

### 1. `noeviction`

不淘汰，内存满了之后新的写请求直接报错。

适合：

- 你宁可失败，也不想静默丢缓存

### 2. 只在带 TTL 的 key 里淘汰

这类策略的思路是：

- 没设置过期时间的 key 不动
- 只从可过期 key 里挑

适合：

- 你把有些长期 key 看得更重，不希望它们被轻易踢掉

### 3. 在所有 key 里淘汰

这类策略不管 key 有没有过期时间，都会整体考虑。

适合：

- Redis 更像一个纯缓存池，而不是“有些 key 特别重要”

### 4. LRU / LFU / Random 这类选择方式

- `LRU`：最近最少使用
- `LFU`：最不经常使用
- `random`：随机挑

工程上更常见的是：

- 读热点明显时，优先考虑 LRU / LFU
- Redis 更像纯缓存时，常配在 allkeys 维度上

## 过期删除和淘汰策略是怎么配合的

可以这么理解：

### 第一层：先看它该不该活

如果 key 已经过期，Redis 会优先把它作为无效数据处理掉。

### 第二层：如果内存还是不够

那就按淘汰策略，从还在“有效期内”的 key 里再挑一部分牺牲掉。

所以从系统视角看，顺序通常是：

1. 尽量先删掉已经过期的数据
2. 如果内存依然紧张，再淘汰别的 key

## 一个特别容易踩坑的场景

有些团队会说：“这个缓存我们不设过期时间，靠后台主动刷新，保证命中率。”

听起来很稳，但这里有个坑：

- 不设过期，不代表它永远在
- 一旦 Redis 内存紧张，它照样可能被淘汰

所以如果业务线程读不到这个 key，不能直接得出“后台刷新失效了”的结论，也可能是它刚刚被淘汰了。

这就是为什么很多“永久缓存”方案，最后还是要配：

- 定期巡检
- 读 miss 后触发补载
- 或消息通知后台重建

## 面试里容易答错的几个点

### “过期 key 会立刻删掉”

不严谨。Redis 走的是惰性删除 + 定期删除，不保证物理上立刻清理。

### “设置了 TTL 的 key 到期前一定存在”

不对。内存淘汰可能提前把它踢掉。

### “内存淘汰只会淘汰过期 key”

不对。淘汰是为了解决内存不足，不是专门清理过期数据。

## 小结

- 过期删除和内存淘汰是两套机制：前者解决失效数据，后者解决内存不足。
- Redis 处理过期 key 的主流方式是惰性删除 + 定期删除，而不是到点立刻删。
- key 逻辑过期了，不代表它会立刻从物理内存消失。
- key 没过期，也可能因为达到 `maxmemory` 而被淘汰。
- 设计缓存时，不能把 TTL 理解成“存活硬承诺”，要考虑淘汰和补载。
