# Redis 分布式锁到底能不能用？

> Redis 分布式锁能不能用，答案不是简单的“能”或“不能”，而是：能用，但你必须知道它解决了什么问题，又解决不了什么问题。

这道题特别容易答成“我会写一条命令”：

```text
SET lock_key unique_value NX PX 10000
```

如果只答到这里，基本等于刚开了个头。因为面试官下一步几乎一定会问：

- 为什么要 `NX`？
- 为什么要带过期时间？
- 解锁为什么不能直接 `DEL`？
- 如果业务执行超过 10 秒怎么办？
- Redis 主从切换时会不会出现两个客户端都拿到锁？

真正想答好这题，得把“写法”“边界”“可靠性”这三层都讲出来。

## 先说结论：Redis 锁适合什么，不适合什么

Redis 分布式锁最适合的是：

- 抢一个任务执行权
- 控制同一资源短时间内只被一个实例处理
- 对极端一致性不是金融级别强要求的场景

比如：

- 定时任务防重复执行
- 批处理互斥
- 幂等控制的附加保护

但如果你要拿它去保护那种“绝不能双写、绝不能重复扣减”的核心资源，那就要特别谨慎，因为 Redis 锁不是天然强一致。

## 一个可用的单节点 Redis 锁，至少要满足三件事

### 1. 加锁要原子

不能先判断 key 在不在，再单独 set，那中间会被并发插队。

所以要用：

```text
SET lock_key unique_value NX PX 10000
```

这里：

- `NX` 表示 key 不存在才设置
- `PX 10000` 表示 10 秒后自动过期

这一条命令完成“判断 + 设置 + 过期时间”，才算原子。

### 2. 锁必须有过期时间

否则客户端拿到锁后如果挂了，锁可能永远不释放，后面所有人都卡死。

所以过期时间不是优化项，而是必选项。

### 3. 锁值要能标识持有者

不能所有客户端都写同一个值，比如都写 `1`。

正确做法是每个客户端生成唯一值，比如 UUID，请求结束时只释放属于自己的锁。

## 为什么解锁不能直接 `DEL`

这是 Redis 锁最经典的坑之一。

假设这样一个时序：

1. 线程 A 拿到锁，锁 TTL 是 10 秒
2. A 业务执行太慢，10 秒到了，锁自动过期
3. 线程 B 拿到了新的锁
4. 线程 A 这时才执行 `DEL lock_key`

结果 A 把 B 的锁删了。

所以解锁不能只是“删 key”，而是要：

1. 先判断当前 value 还是不是自己
2. 是自己再删

并且这两步必须是原子操作，通常用 Lua 脚本。

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

## 最常见的线上问题：业务没执行完，锁先过期了

这比“不会写加锁命令”更常见。

假设你给锁设了 10 秒，但某次业务跑了 15 秒，那么第 10 秒锁就自动失效，别的线程可能重新拿到锁，导致两个线程并发操作同一资源。

### 这说明什么

Redis 锁不是加上就万事大吉，它特别依赖 TTL 是否合理。

### TTL 为什么难配

- 设太短：业务可能还没跑完，锁先过期
- 设太长：异常情况下后继请求等太久

这就是 Redis 锁的一个核心难点：**业务执行时间往往不是严格稳定的。**

## 续约机制是怎么来的

为了解决“业务没跑完锁先过期”，很多实现会做续约。

思路是：

1. 先给锁一个初始 TTL
2. 启动守护线程
3. 在锁快过期时，确认锁还属于自己，就给它续命

这就是常说的 watchdog 思路，像 Redisson 就帮你把这层封装了。

所以如果面试官问“线上一般自己手写吗”，更稳妥的回答通常是：

**能不用手写就尽量别手写，常见项目里更愿意用 Redisson 这类成熟实现，因为它把续约、Lua 原子解锁这些细节都封掉了。**

## 单节点 Redis 锁解决了什么，没解决什么

### 解决了

- 大多数普通分布式互斥需求
- 性能好，命令简单
- 实现成本低

### 没解决

- 锁超时和业务时长不稳定的问题
- Redis 主从切换造成的锁丢失问题
- 强一致场景下的绝对可靠性

后两点才是真正该重点讲的。

## 为什么 Redis 主从复制会让分布式锁不够可靠

假设是主从架构：

1. 客户端 A 在主节点上加锁成功
2. 这条锁数据还没来得及同步到从节点
3. 主节点挂了
4. 一个没拿到这把锁的从节点被提升成新主
5. 客户端 B 又在新主上加锁成功

于是两个客户端都以为自己持有锁。

这不是代码写错，而是**主从复制本身是异步的**。

所以一句很关键的话是：

**单 Redis 主从架构下，分布式锁天然存在主从切换窗口风险。**

这也是为什么 Redis 锁能用，但别轻易吹成“绝对可靠”。

## 那 Redlock 要怎么理解

Redis 官方提出过 Redlock，思路是别只靠一个 Redis 节点，而是：

- 准备多个相互独立的 Redis 主节点
- 客户端依次去这些节点尝试加锁
- 如果超过半数成功，并且总耗时没超过锁有效期，就算成功

它想解决的就是单节点或主从切换带来的锁可靠性问题。

### 但面试里怎么说更稳

不要把 Redlock 说成“完美解决方案”。

更稳妥的表达是：

- 它是 Redis 官方给出的提高锁可靠性的思路
- 通过多个独立节点降低单点故障影响
- 但实现复杂度和运维成本明显更高

很多实际项目里，大家并不会真的自建一套 Redlock，而是：

- 普通互斥场景用成熟 Redis 锁实现
- 核心强一致场景改用数据库锁、ZooKeeper 或业务幂等方案

这才是工程取舍。

## Redis 锁到底值不值得用

可以按业务风险来判断。

### 适合用 Redis 锁的场景

- 定时任务只能跑一次
- 某个资源短时间内只允许一个实例处理
- 即使极端情况下偶发重复，也有业务幂等兜底

### 不适合只靠 Redis 锁的场景

- 余额扣减
- 严格库存一致性
- 一次重复执行就会造成严重资金或状态错误的场景

这些场景更应该先考虑：

- 数据库行锁
- 乐观锁版本号
- 幂等键
- 消息去重

而不是把所有希望都压在 Redis 锁上。

## 面试里最容易答错的点

### “Redis 锁就是 `SETNX`”

太简化了。至少还要讲过期时间、唯一值和 Lua 解锁。

### “有 TTL 就安全了”

不够。业务超过 TTL 时，锁可能提前失效。

### “有主从和哨兵，所以锁更可靠”

恰恰不是。主从复制异步，主从切换反而可能让锁丢失。

## 小结

- Redis 分布式锁能用，但要知道它适合的是普通分布式互斥，不是强一致银弹。
- 单节点可用写法通常是 `SET key unique_value NX PX ttl`，解锁必须用 Lua 脚本做“比对再删除”。
- 锁的 TTL 很难一次配准，所以成熟实现通常会做续约。
- Redis 主从复制是异步的，主从切换会带来锁丢失和双持有风险。
- 对一致性要求极高的场景，不该只把希望压在 Redis 锁上。
