# Redis 为什么会有内存碎片？

> Redis 内存碎片最容易被答成一个纯定义题，但线上真正关心的是：它为什么长出来、什么时候值得处理、处理会不会反过来影响性能。

很多人第一次看到 `mem_fragmentation_ratio` 大于 1，就会下意识觉得 Redis 出问题了。  
其实没这么简单。

Redis 的内存碎片，说白了是在问：**Redis 为了存这些数据，向操作系统要来的物理内存，和真正有效承载数据的那部分内存，差了多少。**

## 先把概念说准

可以先记住这个近似关系：

`mem_fragmentation_ratio = used_memory_rss / used_memory`

它表达的是：

- `used_memory`：Redis 自己觉得真正用来存数据的那部分内存
- `used_memory_rss`：操作系统层面实际给这个进程分配出去的物理内存

所以碎片率高，往往意味着：

- Redis 进程实际占了很多内存
- 但其中有一部分没真正变成有效数据

## 为什么会出现碎片

### 1. 内存分配不是按你想要的精确字节来切

Redis 底层会依赖分配器来申请内存，常见默认是 `jemalloc`。

而分配器做内存分配时，通常不会真的按“你要 17 字节，就只给 17 字节”这么细。

更常见的是：

- 申请 17 字节
- 实际按某个内存档位给你 32 字节

这样做的好处是分配效率更高，但副作用就是会有零碎浪费。

### 2. 数据频繁修改和删除

这才是线上更常见的碎片来源。

如果 Redis 里经常发生：

- 大量对象创建
- 大量对象删除
- 大量对象大小变化

那内存块就会变得越来越零散。  
即使你删除了很多 key，也不代表这部分空间会立刻以最理想的方式重新整理好。

## 为什么碎片率高不一定等于故障

这是特别容易说错的点。

碎片率高，确实代表内存利用不够漂亮，但它不一定立刻导致故障。  
真正要看的是两个问题：

1. 你的实例是不是已经接近内存上限
2. 这部分碎片会不会进一步影响 fork、扩容和容量规划

也就是说，碎片问题很多时候不是“马上挂”，而是：

- 机器看起来还有数据只占 2 GB
- 进程 RSS 却吃到 3 GB、4 GB
- 结果容量估算失真，甚至推高后续风险

## 内存碎片会带来什么影响

### 1. 同样的数据，占更多内存

这是最直观的。  
本来业务以为 Redis 还能装，结果实际物理内存先顶满了。

### 2. fork 成本变高

这点和持久化、高可用直接相关。

Redis 做 `bgsave` 或 AOF rewrite 时要 fork 子进程。  
如果实例 RSS 已经被碎片顶得很大：

- fork 的页表复制成本会更高
- 写时复制期间的额外内存压力也会更重

所以碎片问题不是孤立的，它会和持久化抖动、内存风险绑在一起。

### 3. 容量规划变形

比如你以为一台 8 GB 机器能舒服跑 5 GB 数据，结果因为碎片率太高，RSS 很早就逼近上限，那这台实例就会比预期更早触到风险线。

## 怎么看碎片情况

最常用的还是看 `INFO memory`。

重点不是死背所有字段，而是先看：

- `used_memory`
- `used_memory_rss`
- `mem_fragmentation_ratio`

### 这里要补一个边界

不要把：

`used_memory_rss - used_memory`

简单粗暴等同成“碎片大小”。

因为这中间还可能混着：

- 进程额外开销
- 共享库
- 栈空间
- 其他系统层面的内存占用

所以这个差值只能给你一个信号，不是精确账本。

## 多高才算值得处理

工程上常见的经验线通常是：

- 低一点的波动先观察
- 明显高于 1 且长期维持，再重点关注
- 如果已经接近 1.5 甚至更高，同时实例又逼近内存边界，就该认真处理

这不是绝对红线，但表达的意思很清楚：  
**不是所有碎片都值得动手，关键看它有没有开始影响容量和稳定性。**

## 怎么处理内存碎片

### 1. 先别急着“立刻整理”

如果实例很平稳、内存也还宽松，碎片率稍高一点不一定值得立刻动它。

因为整理动作本身也可能带来额外 CPU 消耗。

### 2. 用主动碎片整理

Redis 后续版本支持主动碎片整理，也就是常说的 `activedefrag`。

它的思路不是“重启清零”，而是：

- 在运行期逐步整理
- 尽量在可控成本内把碎片收回来

这个能力很有用，但同样有边界：  
它不是没有副作用，而是在“节省内存”和“多用一点 CPU”之间做交换。

### 3. 重启整理

最直接的方式当然是重启实例。  
但这不是说看到碎片率高就重启，而是要结合架构：

- 单机实例重启风险大
- 有主从或集群时，可以通过切换角色、滚动重启来更稳地处理

真正工程上看重的是：**怎么在不影响业务的前提下完成整理。**

## 比起整理，更重要的是预防

如果 Redis 持续在长碎片，往往说明上游用法本身就不够健康。

常见预防方向包括：

- 避免大量大对象反复改写
- 避免 big key
- 合理设置 `maxmemory`
- 不让实例长期贴着内存上限跑
- 让写入模式尽量平滑，别频繁大起大落

这类动作看起来不像“处理碎片”，但对长期稳定更重要。

## 容易踩的坑

### “碎片率大于 1 就说明 Redis 异常”

不对。它只是提示你存在额外内存开销，不等于立刻有故障。

### “碎片率高一定会让 Redis 变慢”

也不准确。碎片更直接的问题通常是多占内存，而不是每次请求都必然变慢。  
但它会间接放大 fork、容量、持久化相关风险。

### “整理碎片肯定越快越好”

不对。主动整理也要吃 CPU，不能脱离实例负载去谈。

## 小结

- Redis 内存碎片本质上是实际占用的物理内存，比有效数据内存更大。
- 碎片常见来源是分配器档位分配，以及对象频繁创建、修改、删除。
- 碎片率高不一定立刻是故障，但会推高容量压力和 fork 风险。
- 观察时重点看 `used_memory`、`used_memory_rss`、`mem_fragmentation_ratio`，别把差值机械理解成精确碎片大小。
- 处理上优先讲权衡：能运行期整理就运行期整理，必要时结合高可用架构滚动重启，更重要的是从建模和内存管理上预防。
