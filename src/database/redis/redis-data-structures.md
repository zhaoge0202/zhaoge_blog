# Redis 常见数据类型怎么选？

> Redis 数据类型这道题表面上在问“你背没背过 5 种类型”，其实更想听的是：面对计数器、排行榜、对象缓存、去重、消息流这些场景，你到底知不知道该落在哪个结构上。

很多人一说 Redis 数据类型，就是 `String、List、Hash、Set、ZSet` 一把梭，再补一句“还有 Bitmap、HyperLogLog、GEO、Stream”。这只能算点名，没有回答“为什么用它，不用别的”。

面试里真正高频的，是下面几类判断：

- 对象缓存用 `String` 还是 `Hash`？
- 排行榜为什么一般用 `ZSet`？
- 去重为什么用 `Set`，什么时候又会改用布隆过滤器？
- 消息队列为什么不太建议用 `List`？
- Redis 为什么明明叫 Key-Value，却能扛这么多业务场景？

## 先抓主线：数据类型不是语法题，是建模题

Redis 的值不是只有“字符串”一种。它之所以能做这么多事，是因为不同 value 可以承载不同的访问模式：

| 需求                           | 更适合的数据类型 | 核心原因                     |
| ------------------------------ | ---------------- | ---------------------------- |
| 缓存整个对象、计数器、分布式锁 | `String`         | 简单、快、命令丰富           |
| 缓存对象的多个字段             | `Hash`           | 能按字段读写，不必整对象覆盖 |
| 最新列表、简单队列             | `List`           | 天然有序，支持头尾操作       |
| 去重、共同好友、标签集合       | `Set`            | 无序唯一，交并差操作方便     |
| 排行榜、按分数排序             | `ZSet`           | 有序且可按 score 排名        |

所以选 Redis 类型时，别先想“我会哪个命令”，要先想三个问题：

1. 这个值要不要按字段改？
2. 要不要保证唯一？
3. 要不要排序，排序依据是什么？

把这三个问题想清楚，八成就不会选错。

## String：最常用，但别把它理解成“只能存字符串”

`String` 是 Redis 里最通用的类型。它既能存文本，也能存数字，还能存序列化后的 JSON、二进制内容、token、session。

它适合的场景主要有四类：

- 缓存整个对象：比如商品详情 JSON、用户资料 JSON
- 计数：阅读数、点赞数、库存余量
- 分布式锁：`SET key value NX PX`
- 会话状态：登录 token、验证码

### String 最常见的两个用法

#### 1. 整对象缓存

比如：

```text
SET product:1001 {"id":1001,"title":"机械键盘","price":399}
```

好处是简单，拿一次就是完整对象，适合“读多、整对象返回多、局部字段不常单独改”的场景。

#### 2. 计数器

比如阅读量、点赞数，用 `INCR`、`DECR` 一类命令就够了。

这是 Redis 一个特别强的点：单线程执行命令，让这种自增类操作天然适合高并发计数。

### String 的常见误区

#### 对象缓存一律用 String JSON

不一定错，但要看场景。

如果对象经常只改其中一个字段，比如用户昵称、头像、状态位，那每次都把整个 JSON 取出来、改一部分、再整包写回去，既浪费网络也容易产生并发覆盖问题。

这时往往 `Hash` 更合适。

## Hash：适合“对象字段经常单独读写”的场景

`Hash` 本质上是一个小对象容器。

比如用户信息：

```text
HSET user:1001 name alice age 28 city shanghai
```

之后你可以只改 `age`：

```text
HSET user:1001 age 29
```

不用把整个对象重新写回。

### Hash 适合什么场景

- 用户资料、配置对象、购物车摘要这类“一个对象多个字段”的数据
- 需要频繁单字段修改
- 需要只拿部分字段返回

### String 还是 Hash 存对象？

这是面试里很经典的一问。

可以这么答：

| 维度       | String                 | Hash           |
| ---------- | ---------------------- | -------------- |
| 存储方式   | 整对象序列化后一起存   | 按字段拆开存   |
| 读取方式   | 一次拿整个对象         | 可只拿需要字段 |
| 修改成本   | 改一个字段也得整包覆盖 | 可局部更新     |
| 使用复杂度 | 简单                   | 稍复杂         |

如果对象结构稳定、读多写少、通常整包返回，`String` 更省事。

如果字段经常局部修改，或者经常只读几个字段，`Hash` 更合适。

## List：适合“从两端进出”的有序数据，但别把它神化成消息队列银弹

`List` 适合“头尾操作多、顺序重要”的场景。

典型用法有：

- 最新动态列表
- 简单消息队列
- 简单栈/队列

比如你要做“最近 20 条消息”，就可以往左推：

```text
LPUSH feed:user:1001 item1 item2 item3
LRANGE feed:user:1001 0 19
```

### 为什么说它适合简单队列

因为 `LPUSH/RPOP` 或 `RPUSH/LPOP` 就能凑出 FIFO 队列。

但面试里最好顺便补一句边界：

- `List` 能做简单队列
- 但如果你要消费组、消息确认、重试、堆积治理，就不该停在 `List`
- Redis 里相对更像消息队列的是 `Stream`
- 真正重消息可靠性时，通常还是 Kafka / RabbitMQ 这类专业 MQ

这句话能明显拉开和“背命令”的差距。

## Set：适合去重和集合关系运算

`Set` 的关键词是两个字：**无序、唯一**。

所以它特别适合：

- 标签集合
- 去重名单
- 共同关注 / 共同好友
- 抽奖池、已读集合

比如一篇文章点赞过的用户：

```text
SADD article:1001:likes 12 18 25
```

判断某人点过赞：

```text
SISMEMBER article:1001:likes 18
```

### Set 的优势不只是去重

更重要的是它支持交并差：

- 共同好友：交集
- 推荐补集：差集
- 标签合并：并集

这类关系计算，`Set` 天生比自己在业务层遍历集合轻松很多。

## ZSet：排行榜几乎必考

`ZSet` 是 Redis 里特别有代表性的一个结构，因为它同时满足两件事：

- 元素唯一
- 支持按 score 排序

所以它特别适合：

- 排行榜
- 延时任务时间轮
- 按时间/权重排序的数据流

比如游戏积分榜：

```text
ZADD rank:game 1200 alice 980 bob 1500 carol
```

然后你可以：

- 查 topN
- 查某个用户当前排名
- 查某个分数区间的人

这就是 `ZSet` 比 `Set` 强的地方：它不是只有“在不在”，而是还能回答“排第几”。

### 为什么排行榜不用 List？

因为排行榜的核心不是“有顺序”，而是“顺序会随分数动态变化”。

`List` 更像手工排好的队列，插入中间、重排、按分数查范围都不顺手。

`ZSet` 才是为这种“按 score 排序”生的。

## 这几个特殊类型，面试里知道用途就够了

除了五大基础类型，Redis 还有几类常见扩展场景：

| 类型          | 典型场景           |
| ------------- | ------------------ |
| `Bitmap`      | 签到、活跃用户标记 |
| `HyperLogLog` | UV 近似去重统计    |
| `GEO`         | 附近的人、门店定位 |
| `Stream`      | 带消费组的消息流   |

这些不是每家公司都会用，但面试里知道“它们解决什么问题”会很加分。

## 选型时最容易踩的坑

### 1. 只会背类型，不会按场景选

这是最常见的问题。面试官一换成“购物车该怎么存”“排行榜该怎么做”，很多人就开始乱了。

### 2. 把 List 当通用消息队列

能做，但通常只适合很轻量、对可靠性要求不高的场景。

### 3. 把 String JSON 当万能对象结构

简单是真简单，但局部更新频繁时不一定合适。

### 4. 忽略数据结构背后的访问模式

真正该选什么，不取决于你喜不喜欢某个命令，而取决于：

- 读多还是写多
- 整体读还是局部改
- 是否唯一
- 是否排序

## 小结

- Redis 数据类型本质上是在回答“这个业务数据该怎么建模”。
- `String` 最通用，适合对象缓存、计数器、token、分布式锁。
- `Hash` 适合多字段对象，尤其是字段需要单独读写时。
- `List` 适合有序流和简单队列，但不该被当成消息队列银弹。
- `Set` 适合去重和集合关系计算，`ZSet` 适合排行榜和按分数排序的场景。
