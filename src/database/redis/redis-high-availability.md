# Redis 的高可用方案怎么理解？

> 很多人会背“主从复制、哨兵、集群”三个词，但一问各自解决什么问题、彼此是什么关系，就容易说乱。其实这三者完全可以顺着一个问题链讲下来。

最自然的切入方式是先问：

1. Redis 只有一台机器，会有什么风险？
2. 数据复制到别的机器后，主挂了怎么办？
3. 单机容量和吞吐扛不住了怎么办？

这三个问题，刚好对应：

- 主从复制
- 哨兵
- Redis Cluster

## 先抓主线：高可用不是一个点，是三个层次

| 层次     | 解决的问题           | 代表方案 |
| -------- | -------------------- | -------- |
| 数据备份 | 别只有一份数据       | 主从复制 |
| 故障转移 | 主挂了后自动恢复服务 | 哨兵     |
| 横向扩展 | 单机内存和吞吐不够   | Cluster  |

所以这三者不是互相替代，而是逐层升级。

## 主从复制：先解决“别只有一台 Redis”

如果 Redis 只有一个节点，它的风险很直接：

- 宕机时不可用
- 磁盘坏了或数据损坏时难恢复
- 读流量全压在一台机器上

主从复制的思路就是：

- 主节点负责写
- 从节点复制主节点数据
- 从节点通常承担读流量

这就是常说的读写分离。

### 第一次同步是怎么做的

第一次同步通常是全量同步：

1. 从节点向主节点发起同步请求
2. 主节点生成 RDB 快照
3. 把 RDB 发给从节点
4. 主节点再把这段时间新增的写命令补给从节点

所以第一次同步的成本通常不低，因为它涉及：

- fork 生成 RDB
- 传输全量数据

这也是为什么从节点很多时，主节点会有明显压力。

### 后续同步不是一直全量，而是命令传播 + 增量复制

第一次同步完成后，主从之间会保持长连接，主节点把后续写命令继续发给从节点。

如果网络短暂断开，再恢复后，Redis 会优先尝试增量复制，而不是重新全量同步。

这依赖两个关键概念：

- 复制偏移量 offset
- backlog 缓冲区

如果断线期间缺的那段命令还在 backlog 里，就能增量补回来；不在了，才退化成全量同步。

### 主从复制解决了什么，没解决什么

解决了：

- 数据多副本
- 读压力分担

没解决：

- 主节点挂了后谁来接管写
- 客户端怎么知道新主是谁

这就是哨兵要接的活。

## 哨兵：解决“主挂了以后谁来扶正”

单有主从复制还不够。

因为如果主节点挂了，哪怕从节点里已经有完整副本，也不会自己自动升主。这个时候还是要人工介入：

- 选一个从节点升成主
- 让其他从节点改为跟随它
- 通知客户端去连新主

哨兵就是把这套事自动化。

## 哨兵的三个职责

可以直接背成三个词：

- 监控
- 选主
- 通知

### 1. 监控

哨兵会持续 ping 主从节点，判断它们是否还活着。

### 2. 选主

如果多个哨兵都确认主节点客观下线，就会先选出一个 leader 哨兵，再由它负责做故障转移。

### 3. 通知

故障转移完成后，哨兵会把新主节点信息通知给客户端和其他从节点。

## 为什么哨兵至少一般配 3 个

因为单个哨兵太容易误判，比如：

- 它自己网络抖了
- 它和主节点之间链路断了

不代表主节点真挂了。

所以哨兵是集群部署的。一个哨兵判断主观下线后，还要拿到足够投票，才会把主节点判为客观下线。

通常会建议：

- 哨兵数量用奇数
- 常见是 3 个或 5 个
- `quorum` 一般配成半数以上

这样能兼顾容错和避免误判。

## 哨兵是怎么挑新主的

面试里这块经常会问。

挑新主不是随机选，而是会按一套顺序筛：

1. 先过滤掉明显状态不好的从节点
2. 比优先级 `replica-priority`
3. 优先级相同就比复制进度
4. 再相同就比节点 ID

所以一个更完整的表达是：

**新主节点通常优先选择网络状态稳定、优先级更高、复制进度更接近原主的那个从节点。**

## 哨兵解决了什么，没解决什么

解决了：

- 主节点故障自动转移
- 客户端自动发现新主

没解决：

- 单机内存上限
- 单机写吞吐上限

也就是说，哨兵让 Redis 更能“扛故障”，但没有让它变成“无限扩容”。

这时就轮到 Cluster。

## Cluster：解决“单机放不下、单机扛不住”

如果你的问题不再是“主挂了怎么办”，而是：

- 一台 Redis 内存不够
- 一台 Redis 写流量太高

那主从 + 哨兵就不够了。

因为主从本质还是一份数据的复制，容量并没有横向拆开。

Redis Cluster 的核心思路是：**分片**。

也就是把不同 key 分布到不同主节点上，每个主节点负责一部分数据，再为每个主节点配从节点做高可用。

### 你可以这样理解 Cluster

不是“一主多从”一套，而是“多组一主多从”并列存在：

- 主节点 A 负责一部分槽位
- 主节点 B 负责另一部分槽位
- 每个主节点各自有从节点

这样就同时有了：

- 容量扩展
- 吞吐扩展
- 节点级故障转移

## 什么时候该从哨兵走向 Cluster

如果你面试时能讲到这一层，会很像做过取舍。

### 还在哨兵层面的问题

- 主要矛盾是可用性
- 数据总量和写流量还没超出单机极限

### 该上 Cluster 的信号

- 单机内存快顶满
- 热 key 之外，整体写压力也在持续升高
- 业务规模要求横向扩容，而不是只做主备

## 高可用这条线要怎么一口气讲顺

比较自然的回答方式是：

1. 单机 Redis 有单点风险，所以先上主从复制，解决数据副本和读写分离问题。
2. 但主从复制本身不能自动切主，所以需要哨兵做监控、选主和故障转移。
3. 如果问题进一步发展成单机容量和吞吐不够，就要用 Redis Cluster 做分片，把数据和流量拆到多个主节点上。

这比背三个名词要强很多，因为你是在讲“演进路径”。

## 面试里容易答错的点

### “哨兵就是 Redis 集群”

不准确。哨兵重点解决故障转移，不是分片扩容。

### “主从复制完成后就高可用了”

不完整。只有副本还不够，主挂了仍然需要切主能力。

### “Cluster 只是多机版哨兵”

不对。Cluster 的关键是分片，它解决的是容量与吞吐横向扩展问题。

## 小结

- 主从复制先解决数据多副本和读写分离，但不负责自动切主。
- 哨兵在主从之上补齐监控、故障发现、选主和通知客户端。
- Redis Cluster 再往上一步，通过分片解决单机容量和吞吐瓶颈。
- 所以这三者不是互斥关系，而是一条从备份、切主到扩容的演进路径。
