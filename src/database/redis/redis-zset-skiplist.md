# ZSet 为什么用跳表？和红黑树有什么取舍？

> ZSet 跳表题的重点不是背“跳表复杂度 O(logN)”，而是解释 Redis 为什么需要同时满足查分数、查范围、查排名这几类访问模式。

排行榜是最容易理解 ZSet 的场景。

假设你有一个游戏积分榜：

```text
alice -> 1200
bob -> 980
cindy -> 1500
```

业务不是只问“bob 的分数是多少”，还会问：

- 前 100 名是谁？
- 分数在 1000 到 1500 之间的人是谁？
- bob 当前排第几？
- alice 分数更新后排名怎么变？

这就决定了 ZSet 不能只用一种普通集合结构。

## ZSet 要同时解决两类问题

ZSet 的核心需求可以拆成两条线：

| 需求                 | 需要什么能力           |
| -------------------- | ---------------------- |
| 根据 member 找 score | 快速精确查找           |
| 根据 score 排序查询  | 有序结构、范围和排名   |
| 更新 member 的 score | 快速定位，再调整有序性 |

哈希表擅长第一件事，但不擅长第二件事。

跳表擅长第二件事，但单靠它根据 member 查 score 不够直接。

所以 Redis 的大规模 ZSet 通常是组合结构：

```text
ZSet
├── dict：member -> score
└── skiplist：按 score 排序，score 相同再按 member 排序
```

这也是为什么“ZSet 底层是跳表”这个说法不够完整。

## 跳表到底是什么

跳表可以理解成带多级索引的有序链表。

普通有序链表查一个分数，可能要从头扫到尾。跳表给链表加了多层“快车道”：

```text
第 3 层: 10 ---------------- 80
第 2 层: 10 -------- 50 ---- 80
第 1 层: 10 -- 30 -- 50 -- 80 -- 100
```

查找时先从高层跳，跳过一大段，再逐层下沉。

所以跳表能把查找、插入、删除的期望复杂度做到 O(logN)，同时又保留链表天然适合顺序遍历的特点。

## 为什么跳表适合 ZSet

### 1. 范围查询很自然

ZSet 高频命令里有很多范围类操作，比如按分数范围查、按排名范围查。

跳表找到范围起点后，可以沿着底层链表继续向后扫，非常自然。

红黑树也能做范围查询，但遍历时要做中序遍历或维护额外指针，工程实现上没有跳表这么直接。

### 2. 排名计算可以通过 span 做出来

Redis 的跳表节点里不仅有 forward 指针，还维护了 span。

span 可以粗略理解成：这一层指针跨过了多少个底层节点。

有了 span，Redis 在查找过程中就能累加跨度，算出某个元素的排名。这对 `ZRANK`、`ZREVRANK` 这类命令很关键。

### 3. 实现和调试相对简单

红黑树要靠旋转维持平衡。插入、删除时有颜色调整和旋转逻辑，正确实现并不轻。

跳表靠随机层高维持概率平衡，代码结构更直观。对 Redis 这种强调工程可控性的系统来说，这一点很重要。

### 4. 范围遍历和局部移动更友好

排行榜经常不是只查一个点，而是查一段。

跳表底层本来就是链表，找到起点后向后遍历成本很清晰，也方便支持 score 相同情况下按 member 排序。

## 那红黑树是不是不行

不是。

红黑树也是成熟的平衡树，查找、插入、删除都是 O(logN)，在很多系统里非常合适。

但放到 Redis ZSet 这个场景，跳表有几个更贴近需求的优势：

| 对比项   | 跳表                         | 红黑树                         |
| -------- | ---------------------------- | ------------------------------ |
| 范围遍历 | 找到起点后沿底层链表顺序扫   | 需要中序遍历或额外维护遍历关系 |
| 排名支持 | 可以用 span 辅助累计排名     | 需要额外维护子树大小           |
| 实现复杂 | 随机层高，逻辑相对直观       | 插入删除旋转和颜色调整更复杂   |
| 性能稳定 | 期望 O(logN)，依赖随机化平衡 | 严格 O(logN)                   |

所以答案不是“红黑树不好”，而是：**跳表在 Redis ZSet 的范围查询、排名和实现复杂度之间更均衡。**

## 为什么还要 dict

如果只有跳表，按 score 范围查没问题，但根据 member 查 score 就不够直接。

比如执行：

```text
ZSCORE rank bob
```

Redis 希望快速知道 bob 的分数，而不是在有序结构里扫。

dict 正好解决这个问题：

- `ZSCORE`：走 dict。
- `ZADD` 更新已有 member：先走 dict 找旧 score，再调整跳表。
- `ZRANGE/ZRANGEBYSCORE`：走 skiplist。

这就是 dict + skiplist 协作的意义。

## 小集合为什么不一定用跳表

Redis 不是所有 ZSet 都直接上 dict + skiplist。

当 ZSet 元素少、元素也比较小的时候，用紧凑编码更省内存。数据规模上来后，再切换成更适合查询和排序的大结构。

这和 Hash、List 的策略一致：小对象优先省内存，大对象优先访问效率。

## 面试里怎么答更稳

可以按这条线说：

1. ZSet 要同时支持 member 查 score、score 排序、范围查询和排名。
2. dict 负责 member 到 score 的快速定位。
3. skiplist 负责按 score/member 维护有序性，天然适合范围遍历。
4. 跳表通过多层索引让查找、插入、删除保持期望 O(logN)。
5. 相比红黑树，跳表在范围遍历、排名辅助和实现复杂度上更贴近 Redis 的工程取舍。

## 容易踩的坑

### “ZSet 只有跳表”

不完整。大对象 ZSet 通常是 dict + skiplist。

### “跳表一定比红黑树快”

不能这么说。两者都有 O(logN) 能力，Redis 选择跳表更多是场景和实现取舍。

### “score 相同就随便排”

不对。ZSet 里 score 相同时还会按 member 做顺序比较，这能保证排序结果稳定。

## 小结

- ZSet 的难点在于同时支持精确查找、排序、范围和排名。
- dict 负责 member 查 score，skiplist 负责有序范围和排名。
- 跳表是带多级索引的有序链表，适合找到起点后连续遍历。
- 和红黑树相比，跳表更容易配合范围查询、span 排名和简洁实现。
- “ZSet 底层是跳表”只是简化说法，面试里要补上 dict 协作。
