# Redis 做分布式限流为什么经常配 Lua？

> Redis 限流题如果只会说“用计数器就行”，通常会被继续追问：那自增和过期怎么原子处理？多个实例怎么共享？滑动窗口和令牌桶怎么表达？

限流本身不是 Redis 独占的话题，但 Redis 很适合做：

- 分布式共享计数
- 低延迟状态判断
- 简单而高并发的限流判定

所以面试里 Redis 限流很常见。

## 先说为什么 Redis 适合做分布式限流

因为限流本质上就是在做一件事：

- 记录某个维度下的请求次数或令牌状态
- 在很短时间内快速判断“还能不能放行”

Redis 在这里的优势很直接：

- 访问快
- 可多实例共享
- 自增、过期、排序类命令都顺手

所以它天然比单机内存计数器更适合分布式服务。

## 最简单的做法：固定窗口计数

比如限制：

- 每个 IP 每秒最多 10 次请求

最直接的思路是：

1. 以 `ip + 秒级时间窗` 组成 key
2. 每次请求对这个 key 做 `INCR`
3. 超过阈值就拒绝
4. 给这个 key 配一个短 TTL

这个方案简单、便宜、很好理解。

## 那为什么还要 Lua

关键就在这里：

你如果把逻辑拆成两步：

1. `INCR`
2. `EXPIRE`

中间是有窗口的。  
如果你想把“自增 + 设置过期时间”合成一个不可被打断的动作，Lua 就很自然地进来了。

Redis 官方关于 rate limiter 模式和 `INCR` 文档也明确给了类似思路：  
窗口计数器经常需要把计数和过期处理放到同一个原子动作里。

所以 Lua 在限流里的价值，不是“炫技”，而是：

**把多步限流判定封成一次 Redis 内部原子执行。**

## 固定窗口的优点和缺点

### 优点

- 实现最简单
- 成本最低
- 很适合很多普通接口限流

### 缺点

窗口边界不够平滑。

比如一个用户在某秒最后时刻打满 10 次，下一秒刚开始又打 10 次，表面看每秒都没超，但瞬时两秒交界处其实冲了 20 次。

这就是固定窗口的典型边界。

## 滑动窗口为什么更细

滑动窗口的思路是：

- 不按整块时间窗粗暴切
- 而是基于更连续的时间范围判断

Redis 里常见做法一般会用：

- `ZSet`

把请求时间戳放进去，然后：

1. 删掉窗口外的旧请求
2. 统计窗口内还有多少
3. 决定是否放行

它更平滑，但代价也更高：

- 结构更复杂
- 操作更重
- 高并发下更需要 Lua 把多步动作包起来

## 令牌桶为什么经常配 Redis

令牌桶的思路不是“数请求”，而是：

- 系统按固定速率补充令牌
- 请求来了先尝试拿令牌
- 有令牌就放行，没令牌就拒绝

它比固定窗口更适合：

- 平滑放流
- 容忍短时间突发

而 Redis 适合做这件事，是因为：

- 令牌数状态可以共享
- 计算和更新可以放进 Lua

很多现成实现，比如一些客户端库里的 `RRateLimiter`，底层也就是这条思路。

## Redis 限流最适合哪些场景

- 接口级限流
- IP / 用户 / 资源级频控
- 多实例共享限流状态
- 对低延迟判定要求高的网关前置逻辑

## 它不适合替代什么

不要把 Redis 限流理解成“所有流量治理都交给 Redis”。

更准确的说法是：

- Redis 适合做共享判定层
- 但真正完整的流量治理，通常还会叠加网关、熔断、降级、黑名单等机制

所以它是限流体系的一环，不是全部。

## 一个更稳的答法

如果面试官问“为什么 Redis 限流经常配 Lua”，比较稳的回答是：

1. Redis 很适合做分布式共享限流状态。
2. 最简单的固定窗口方案通常基于计数器和过期时间。
3. 但计数和过期、滑动窗口里的删除和统计、令牌桶里的状态更新，都往往是多步操作。
4. Lua 可以把这些多步逻辑封到 Redis 内部原子执行，避免并发下判定不一致。
5. 所以 Redis + Lua 常常一起出现，不是巧合，而是因为结构上天然互补。

## 容易踩的坑

### “限流就是 `INCR` 一下”

太浅了。真正问题在于窗口边界、过期处理和并发原子性。

### “Lua 只是性能优化”

不止。它更重要的价值是把多步限流判定变成一次原子执行。

### “Redis 限流能替代网关治理”

不对。它适合做共享判定，但不是完整流量治理系统。

## 小结

- Redis 适合做分布式限流，因为它能低延迟共享计数和状态。
- 固定窗口最简单，滑动窗口更平滑，令牌桶更适合控制平均速率和短时突发。
- Lua 在限流里最重要的作用，是把多步逻辑封成 Redis 内部一次原子执行。
- Redis 限流适合做判定层，但通常还要和网关、熔断、降级一起配合。
- 真正答好这题的关键不是背算法名字，而是说清为什么这些算法落到 Redis 时常常需要 Lua。
