# Redis 能当消息队列吗？

> Redis 可以做消息队列，但你得先想清楚：你要的是“顺手做一个轻量队列”，还是“拿它替代专业 MQ 扛核心链路”。

这个问题特别容易答成一句空话：

- 能
- 但不建议

这两句都不算错，但还不够。面试里真正要讲清的是：

1. Redis 有哪些实现路径
2. 它们分别缺什么
3. 什么场景够用，什么场景不该硬上

## 先立一个判断标准：像样的 MQ 至少要有什么

如果你不先把标准立住，后面就很容易把“能发消息”误当成“就是 MQ”。

一个生产级消息队列，通常至少要考虑这些能力：

| 能力     | 你真正关心什么           |
| -------- | ------------------------ |
| 持久化   | 宕机后会不会丢消息       |
| 消费确认 | 消费者处理成功后怎么确认 |
| 重试机制 | 失败后能不能重新投递     |
| 消费组   | 多消费者怎么协作分工     |
| 堆积能力 | 消费慢时能不能扛住积压   |
| 扩展能力 | 吞吐上来后能不能横向扩   |

所以 Redis 能不能做 MQ，不在于它有没有队列结构，而在于它能覆盖这张表里的几项。

## 第一种：用 List 做最简单的队列

最原始的方式就是：

- 生产者 `LPUSH` / `RPUSH`
- 消费者 `RPOP` / `LPOP`

它的优点很直接：

- 实现简单
- 性能高
- 有天然顺序

如果再进一步，用阻塞读：

- `BLPOP`
- `BRPOP`

还能避免消费者空轮询。

### 但 List 队列的问题也很直接

#### 1. 消息取走就没了

消费者把消息 pop 出来之后，如果业务处理失败，这条消息就已经不在队列里了。

也就是说，**没有天然 ACK 机制。**

#### 2. 广播能力差

一个消息通常只能被一个消费者取走。  
如果你想让多个消费者都收到同一条消息，List 这条路就不顺了。

#### 3. 重试和死信要自己补

这不是不能做，而是做着做着你会发现：  
自己已经开始手搓一套不完整的 MQ 了。

## 第二种：用 Pub/Sub 做发布订阅

Pub/Sub 的长处在于它天然支持“一发多收”。

适合这种场景：

- 实时通知
- 广播消息
- 对丢失不敏感的事件分发

### 但 Pub/Sub 有个致命边界

**它是发后即忘的。**

如果消息发出去时，订阅者不在线，或者网络抖一下，这条消息就没了。

这意味着它缺少几样关键能力：

- 没有持久化
- 没有 ACK
- 没有可靠重试
- 没法优雅处理消息堆积

所以它更像“通知通道”，而不是严肃业务消息队列。

## 第三种：用 Stream 做轻量级 MQ

Redis 5.0 之后，真正像样能拿来讲 MQ 的，是 `Stream`。

它相对于前两种方式，最大的提升在于：

- 支持消费者组
- 支持 ACK
- 支持 Pending 视角下的未确认消息管理
- 支持按 ID 范围读取和回溯

也就是说，Stream 已经不只是“能排队”，而是开始具备一部分专业 MQ 的骨架。

## Stream 解决了什么问题

### 1. 有了消费者组

多个消费者可以归到同一个组里协作消费，而不是所有人都盯着同一条 List 去抢。

### 2. 有了 ACK

消费者处理完之后，通过 `XACK` 确认消息。  
如果消费者挂了，消息会留在 pending 状态，而不是直接无声丢失。

### 3. 能做未确认消息转移

借助 `XPENDING`、`XCLAIM` 一类机制，可以把长时间没确认的消息转给别的消费者处理。

这意味着它已经具备“至少一次投递”的基础能力。

## 但 Stream 也不是 Kafka 或 RabbitMQ

这里一定要把边界说清。

### 1. 持久化强度有限

Stream 的消息还是落在 Redis 这套持久化体系上。  
如果你配的是 AOF `everysec`，那本质上还是接受“极端情况下丢最近一小段消息”的风险。

所以它不是那种默认奔着金融级别可靠性去设计的。

### 2. 堆积能力受内存约束

Kafka 这类专业 MQ 很强的一点，是它天然适合磁盘堆积。  
而 Redis 终归是以内存为主。

这意味着：

- 少量堆积可以扛
- 中量堆积要谨慎
- 海量长时间堆积不该指望 Redis

### 3. 扩展能力不是它的强项

如果你的核心诉求是：

- 超高吞吐
- 海量日志流
- 超长堆积
- 强副本可靠性

那专业 MQ 还是更合适。

## 什么时候 Redis 做 MQ 是合理的

### 适合的场景

- 你本来就已经有 Redis
- 消息量不算特别夸张
- 更看重低延迟和实现简单
- 能接受轻量级可靠性，而不是极致可靠性

比如：

- 站内异步通知
- 轻量任务分发
- 中小规模事件流
- 对延迟敏感但消息量不巨大的链路

### 不适合的场景

- 金融级核心账务
- 海量日志采集
- 长时间大规模堆积
- 强依赖重试、死信、回溯治理的复杂消费体系

这时硬拿 Redis 顶，会越来越像“自己补专业 MQ 缺失的能力”。

## 一个更稳的答法

如果面试官问“Redis 能做消息队列吗”，比较稳的答法通常是：

1. 先说能，但要看实现方式。
2. List 只能做最简队列，缺 ACK 和可靠重试。
3. Pub/Sub 更像通知通道，不适合严肃业务队列。
4. 真正能拿来做轻量 MQ 的是 Stream，它有消费者组和 ACK。
5. 但 Stream 仍然受 Redis 持久化和内存模型约束，不该和 Kafka、RabbitMQ 混为一谈。

## 容易踩的坑

### “Redis 既然有队列结构，就等于 MQ”

不对。能存消息，不等于具备生产级消息系统能力。

### “Pub/Sub 最适合做 MQ，因为支持广播”

这句话很危险。广播没问题，但它没有可靠性保障，严肃业务队列不能只看广播能力。

### “Stream 已经足够替代所有 MQ”

不成立。它解决了很多 Redis 早期做队列的缺陷，但不代表它就能覆盖专业 MQ 的全部场景。

## 小结

- Redis 可以做消息队列，但不同实现路径能力差异很大。
- List 适合最简单的单消费队列，Pub/Sub 更像广播通知，Stream 才算轻量级 MQ。
- Stream 的优势是消费者组、ACK 和未确认消息管理，但可靠性和堆积能力仍受 Redis 本身约束。
- 如果场景是轻量、低延迟、已有 Redis，Stream 很有性价比。
- 如果场景是海量堆积、高可靠、复杂消费治理，专业 MQ 更合适。
