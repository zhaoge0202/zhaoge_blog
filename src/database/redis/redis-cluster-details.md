# Redis Cluster 为什么是 16384 个槽？扩缩容和路由怎么理解？

> Redis Cluster 这道题如果只会背“16384 个槽”，其实还没讲到重点。真正想听的是：键怎么映射、客户端怎么路由、迁槽时为什么会有 MOVED 和 ASK。

高可用那篇已经把 Cluster 放进了大图景里：  
它解决的是单机容量和吞吐不够的问题。

但面试官继续追问时，通常不会停在“它是分片集群”，而是会往下问：

- 为什么不是一致性哈希？
- 16384 个槽是怎么来的？
- 客户端怎么知道 key 该打到哪台机器？
- 扩缩容时能不能不停机？
- MOVED 和 ASK 到底差在哪？

这一篇就专门把这些工程细节拆开。

## 先抓主线：Cluster 的核心不是“多台机器”，而是“槽”

Redis Cluster 并不是直接把 key 映射到节点，而是先映射到 **hash slot**。

它的基本过程是：

1. 先根据 key 算出一个槽号
2. 再看这个槽当前归哪个主节点负责
3. 请求打到对应节点

所以 Cluster 真正的核心单位不是节点，而是 **槽位**。

## 为什么是 16384 个槽

Redis 官方规格文档给出的规则是：

`HASH_SLOT = CRC16(key) mod 16384`

也就是说：

- 先对 key 做 `CRC16`
- 再对 `16384` 取模

于是整个集群的 key space 被切成了 16384 个槽。

### 那为什么不是 10000，也不是更多

从工程角度看，这个数是折中结果。

槽位太少的问题是：

- 分片颗粒度不够细
- 扩缩容时不够灵活

槽位太多的问题是：

- 元数据更重
- 节点间传播和客户端维护映射表的成本更高

Redis 最后选了 16384，也就是 `2^14`，本质上是在：

- 分片颗粒度
- 元数据成本
- 实现复杂度

之间做平衡。

所以面试里别把它答成“这是一个神奇常数”，更准确的说法是：  
**它是 Redis Cluster 设计里在路由精度和管理成本之间做的工程折中。**

## 客户端怎么路由

Cluster 有个很重要的设计：  
**节点不做代理转发。**

这意味着如果你把请求打错了节点，它不会像某些代理型中间件一样帮你悄悄转发，而是直接告诉你：

- 你找错地方了
- 正确节点在那边

这就是 `MOVED` 重定向的来源。

成熟客户端通常会缓存一份“槽 -> 主节点”的路由表。初始化时可以通过 `CLUSTER SLOTS` 或 `CLUSTER SHARDS` 获取拓扑；收到 `MOVED` 后刷新对应槽位，必要时刷新整张槽表。

### MOVED 是什么意思

如果某个槽已经稳定归另一台节点负责，而你还把请求发到旧节点，旧节点会回你一个 `MOVED`。

这个信号表达的是：

- 这个槽的归属已经变了
- 后续请求应该改打新节点

所以 `MOVED` 更像“永久路由更新信号”。

## ASK 又是什么

`ASK` 最容易和 `MOVED` 混。

它通常出现在 **迁槽进行中**。

这时候槽位还没完全稳定迁完，可能会出现这种状态：

- 老节点还保留着部分 key
- 新节点已经开始接这部分槽的迁移流量

所以此时如果客户端只根据“以后都去新节点”来处理，就会出错。  
Redis 才会设计 `ASK` 这种更临时的重定向。

### ASK 的语义

它表达的是：

- 这次请求你先去新节点试
- 但别立刻把本地路由表永久改掉

也就是说：

- `MOVED` 是稳定迁移后的正式搬家
- `ASK` 是迁移过程中的临时指路

这个区别是 Cluster 面试里特别高频的细节题。

实际协议上，客户端收到 `ASK` 后，下一跳请求前需要先向目标节点发送一次 `ASKING`，表示“我知道这是迁移中的临时访问”。这也是为什么 `ASK` 不是简单换地址重试。

## 为什么扩缩容时还能继续服务

Cluster 的一个很强的点，就是 **可以在线迁槽**。

比如扩容时：

1. 加入新节点
2. 把一部分槽从老节点迁到新节点
3. 客户端在迁移过程中根据 `ASK` / `MOVED` 调整请求
4. 全部迁完后，路由稳定

所以它不是那种“停机后整体重分片”的模型，而是渐进式迁移。

这也是为什么 Redis Cluster 比“自己在客户端做简单分片”更像一个完整集群方案。

## 那多 key 操作怎么办

这正是 Cluster 的一个经典边界。

Redis 官方文档说得很清楚：  
复杂的多 key 操作，只有在这些 key 都落在同一个槽时才好做。

否则你会面临：

- 不同节点各持一部分 key
- 没法像单机那样天然原子执行

这时候就会引出另一个高频概念：**hash tag**。

## hash tag 是干什么的

Redis Cluster 允许你在 key 里用 `{...}` 指定一段“共同哈希片段”。

例如：

- `{user1000}.profile`
- `{user1000}.orders`

它们虽然 key 名不同，但因为花括号里的部分相同，就会落到同一个槽。

这样做的目的非常实际：

- 把相关 key 放到同一槽
- 让多 key 操作有机会继续成立

所以 hash tag 不是语法技巧，而是 **多 key 设计在 Cluster 下的工程补丁**。

如果多 key 命令里的 key 不在同一个槽，客户端或节点会报类似 `CROSSSLOT` 的错误。比如 `MGET user:1:name user:2:name` 可能跨槽失败；而 `MGET {user:1}:name {user:1}:orders` 可以落到同一槽。hash tag 要谨慎使用，滥用会把大量 key 压到一个槽，反而制造热点。

## 节点之间怎么知道槽位变化？

Cluster 节点之间会通过集群总线交换节点状态和槽位信息。客户端不需要连接所有节点才能知道完整状态，但它需要能处理重定向和拓扑刷新。

这带来两个工程要求：

- 客户端必须使用支持 Cluster 的连接方式，而不是普通单机连接池。
- 网络和安全组要允许节点之间的集群通信，否则节点互相看不到状态，故障检测和迁槽都会异常。

## Cluster 的一致性边界也要说清

很多人一听分片集群，就会默认“更强大”，但这里有个必须讲的边界：

Redis Cluster 的复制依然是 **异步复制**。

这意味着：

- 高性能和可扩展性很强
- 但不是强一致系统

官方规格文档也明确提到，确认过的写在某些窗口下仍可能丢失，尤其是在故障切换和网络分区时。

所以如果面试官问“Cluster 能不能保证绝对不丢写”，稳妥答案应该是：

**不能，它追求的是高性能、可扩展和可接受的写安全，不是强一致复制。**

还有一个可用性边界：Cluster 是按槽提供服务的。如果某个槽所在主节点以及它可用的副本都不可用，那么这个槽的数据就不可服务；默认情况下，集群可能进入不可用状态，避免继续提供不完整的 key space。也就是说，Cluster 提升了横向扩展能力，但每个分片仍然要靠主从副本保障可用性。

## 一个更稳的答法

如果面试里被追问 Cluster 细节，比较顺的答法是：

1. Cluster 不是直接 key 到节点，而是 key 先映射到 16384 个槽。
2. 每个主节点负责一部分槽，所以扩缩容本质是迁槽，而不是整体重建。
3. 客户端打错节点时，稳定状态会收到 `MOVED`，迁槽中间态会收到 `ASK`。
4. 多 key 操作要求 key 落在同一槽，工程上经常用 hash tag 保证。
5. Cluster 提供的是高性能分片和可接受的可用性，不是强一致系统。
6. Cluster 客户端要维护槽位路由表，并正确处理 `MOVED`、`ASK`、`ASKING` 和拓扑刷新。

## 容易踩的坑

### “Redis Cluster 用的是一致性哈希”

不对。Redis Cluster 的官方实现是哈希槽模型，不是一致性哈希。

### “MOVED 和 ASK 都是重定向，没本质区别”

不对。`MOVED` 更像正式路由更新，`ASK` 更像迁移过程中的临时指路。

### “上了 Cluster 就能随便做多 key 原子操作”

不成立。多 key 操作依然受限于是否在同一槽。

### “hash tag 用得越多越好”

不对。hash tag 是为了让强相关 key 同槽，不是为了绕过分片。把大量 key 固定到同一 tag，会制造热点槽和单节点压力。

## 小结

- Redis Cluster 的核心不是“多节点”，而是“16384 个哈希槽”。
- key 先映射到槽，再由槽映射到主节点，所以扩缩容本质上是迁槽。
- `MOVED` 表示稳定路由已经变更，`ASK` 表示迁槽过程中的临时重定向。
- 多 key 操作要尽量落在同一槽，常见办法是用 hash tag。
- Cluster 追求的是高性能和线性扩展，不是强一致分布式数据库。
- Cluster 客户端要维护槽表并处理重定向；跨槽多 key 操作会受限，滥用 hash tag 会造成热点。
- 某个槽的主从都不可用时，这个槽无法服务，Cluster 不是“任意节点挂了都无感”。
