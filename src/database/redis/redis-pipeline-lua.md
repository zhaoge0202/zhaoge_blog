# Redis 里的 Pipeline 和 Lua 脚本该怎么选？

> Pipeline 和 Lua 脚本都会被拿来做“批量操作”，但它们解决的不是同一个问题。一个主要是省 RTT，一个主要是把多步逻辑压成一次原子执行。

很多人一说 Redis 性能优化，就会把这几个词混着背：

- 批量操作
- 事务
- Pipeline
- Lua

如果不拆开，很容易答出几句彼此打架的话。  
更清楚的切法应该是：

- **Pipeline** 重点在减少网络往返
- **Lua 脚本** 重点在把多步逻辑放到 Redis 内部原子执行

它们都能减少交互成本，但关注点不一样。

## 先说 Pipeline：它到底优化了什么

Redis 官方文档里对 pipelining 的定义很直接：

- 一次发多个命令
- 不用每条都等响应回来
- 最后再统一把响应读回来

它主要优化的是 **RTT**，也就是请求从客户端到服务端再回来的往返时间。

### 为什么 RTT 会成为瓶颈

假设你连续做 1000 次 `INCR`。

如果每次都走一轮：

1. 发命令
2. 服务端执行
3. 回响应
4. 再发下一条

那 Redis 即使本身执行很快，来回网络和 socket I/O 也会累出一大笔额外成本。

Pipeline 的意义就在于：

- 先把一批命令尽量连续发出去
- 服务端连续处理
- 客户端再一次性收回结果

所以它不是在让单条命令更快，而是在减少 **一批命令的交互损耗**。

## Pipeline 最适合什么场景

最适合的是这种：

- 一次要发很多条命令
- 这些命令彼此没有前后依赖
- 你最在意的是吞吐和网络往返开销

比如：

- 批量写缓存
- 批量预热
- 批量计数更新
- 批量读取分散 key

## Pipeline 不解决什么

### 1. 它不保证原子性

这是最容易答错的一点。

Pipeline 的多个命令只是“打包发送”，不是“打包成一个不可分割事务”。

也就是说：

- Redis 会按顺序处理你这一批命令
- 但其他客户端的命令仍可能在执行层面和你交错

所以它的价值在于性能，不在于一致性保障。

### 2. 它不适合有依赖的多步逻辑

比如：

1. 先 `GET` 一个值
2. 根据这个值判断要不要 `SET`
3. 再决定是否删除另一个 key

这种逻辑如果后一步依赖前一步结果，Pipeline 就不适合，因为它不是让服务端帮你“边算边决定”，它只是减少来回通信。

## Lua 脚本解决的又是什么

Lua 的核心价值，不是“也能批量发命令”，而是：

**把一段带判断、带多步读写的逻辑，放进 Redis 内部一次执行。**

Redis 官方文档明确强调了脚本的阻塞式原子执行语义。  
也就是说，脚本执行期间，别的命令不会插进来把这段逻辑打断。

所以 Lua 特别适合这种场景：

- 先读再判断
- 再做多步更新
- 中间不想被别人插队

最经典的例子就是分布式锁解锁：

1. 先判断 value 是不是自己
2. 是自己再删

这两步分开做不安全，用 Lua 脚本就能封成一次原子执行。

## Lua 和事务也不是一回事

很多人会顺手把 Lua 说成“Redis 真事务”，这也不严谨。

更准确地说：

- Lua 脚本执行是原子的
- 但 Redis 不是关系型数据库那种回滚事务模型

如果脚本中途执行到一半后发生错误，前面已经做掉的写操作不会自动回滚。

所以这里最好把“原子执行”和“数据库式回滚事务”分开说。

## Lua 还有两个很容易漏掉的边界

### 1. key 参数不能乱来

Redis 官方文档对脚本参数有个很重要的约束：

- 脚本访问到的 key，应该显式通过 key 参数传进去
- 不应该在脚本里随意拼接程序化 key 名再乱访问

这个约束在 Cluster 下尤其重要，因为脚本执行也要考虑 key 所在槽位。

### 2. 脚本缓存不是持久资产

官方文档还特别提醒了一点：  
脚本缓存是易失的。

也就是说：

- 重启可能没了
- failover 后可能没了
- `SCRIPT FLUSH` 后也可能没了

所以业务不能假设“脚本一旦加载过，就永远都在”。

## 那 Pipeline 和 Lua 怎么选

可以直接按问题性质来分。

### 如果你只是想减少网络往返

优先看：

- 原生批量命令，比如 `MGET`、`MSET`
- Pipeline

因为你的主要矛盾是 RTT 和 socket I/O。

### 如果你要在 Redis 内部做多步判断和原子更新

优先看 Lua。  
因为这时你的主要矛盾已经不是交互次数，而是：

- 多步逻辑不能被打断
- 结果依赖前面读出来的值

## 在 Cluster 下还要再补一句

无论是 Pipeline 还是 Lua，在 Cluster 里都不能忘记槽位问题。

### Pipeline 的边界

如果这批 key 分散在不同槽上：

- 客户端通常要拆分请求
- 不一定还能享受到理想中的单节点批处理收益

### Lua 的边界

Lua 脚本也不能假装自己是“跨集群全局事务”。  
如果脚本涉及的 key 不在同一槽，Cluster 下就会出问题。

所以面试里这句很重要：

**在 Redis Cluster 里，性能优化和原子执行都仍然受槽位分布约束。**

## 一个更稳的答法

如果面试官问“Pipeline 和 Lua 怎么选”，比较顺的回答可以是：

1. Pipeline 主要优化 RTT，适合大量互不依赖的命令批量发送。
2. 它不保证原子性，也不适合有前后依赖的逻辑。
3. Lua 脚本适合把读判断写这类多步逻辑放到 Redis 内部原子执行。
4. 但 Lua 也不是数据库事务，不提供失败回滚。
5. 在 Cluster 下，两者都还要考虑 key 是否落在同一槽。

## 容易踩的坑

### “Pipeline 就是批量事务”

不对。它优化的是交互次数，不是事务语义。

### “Lua 就等于数据库式原子事务”

不严谨。它保证脚本执行期间不被别的命令插入，但不是失败自动回滚。

### “只要用了 Pipeline，吞吐就一定无限上升”

不对。Pipeline 也要控制批次大小，因为服务端需要暂存响应，批太大反而会吃额外内存。

## 小结

- Pipeline 的核心价值是减少 RTT 和 socket I/O，适合大量互不依赖的命令。
- Lua 的核心价值是把多步逻辑放进 Redis 内部原子执行，适合读判断写一体化场景。
- Pipeline 不是事务，Lua 也不是关系型数据库那种可回滚事务。
- 在 Redis Cluster 下，两者都要受槽位约束，不能假装跨槽无成本工作。
- 真正选型时先看主要矛盾：是网络往返太多，还是多步逻辑不能被打断。
