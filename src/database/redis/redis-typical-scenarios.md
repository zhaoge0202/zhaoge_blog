# Redis 里的排行榜、计数器、签到这些场景该怎么设计？

> Redis 面试最怕的不是不会背数据结构，而是面试官问你“点赞数怎么做”“排行榜怎么做”时，你只能回一句“用 Redis 很快”。

这一篇不讲抽象定义，直接看最常见的业务场景和结构落地。

## 计数器：最常见，也最容易被答浅

### 典型场景

- 阅读量
- 点赞数
- 请求次数
- 库存预占数量

### 最常见做法

一般直接用 `String + INCR/DECR/INCRBY`。

它为什么顺手：

- 命令简单
- 并发下原子自增方便
- 性能很好

### 但边界要说清

如果这个计数要求的是：

- 财务级精确
- 和数据库强一致

那就不能只靠 Redis 计数。  
Redis 计数特别适合“高并发快速累加”，不适合独自扛最终正确性。

## 排行榜：`ZSet` 几乎是默认答案

### 典型场景

- 积分榜
- 销量榜
- 热门帖子榜
- 游戏战力榜

### 为什么一般用 `ZSet`

因为排行榜通常同时需要两件事：

- 一个成员唯一标识
- 一个可排序分值

`ZSet` 天然就提供：

- member
- score

而且还能方便做：

- Top N
- 指定成员排名
- 按分数区间拉取

### 常见误区

如果你只会说“排行榜用 ZSet”，还不够。  
更完整的表达应该带上：

- 排名是不是实时更新
- score 是不是会频繁变
- 要不要定期落库
- 数据量很大时是否要分榜

## 签到和活跃统计：Bitmap 很顺手

### 典型场景

- 用户某天是否签到
- 某月签到天数
- 某天活跃用户标记

### 为什么 Bitmap 合适

因为这类场景本质上就是：

- 某个对象
- 在某个偏移位上
- 只有 0 和 1 两种状态

这正适合按位存。

### 一个很典型的好处

比如一个用户一个月签到 31 天，只需要 31 bit。  
如果你用普通结构去存，会浪费很多空间。

## UV 和独立访客：HyperLogLog 很省

### 典型场景

- 每日 UV
- 每周独立访问设备数
- 独立 IP 粗略统计

### 为什么不用 `Set`

`Set` 当然能做精确去重，但元素一多，内存压力就很明显。  
而 UV 这类场景通常更看重：

- 趋势
- 数量级
- 大盘统计

而不是每一个精确对象集合。

这时 HyperLogLog 的“低内存 + 近似统计”就很有价值。

## 附近的人 / 门店：GEO 够轻

### 典型场景

- 附近的人
- 附近门店
- 附近骑手

### 什么时候适合 Redis GEO

如果你的诉求主要是：

- 算距离
- 找附近若干点
- 做轻量排序

那 Redis GEO 很合适。

但如果已经涉及复杂地图能力，就别硬拿 Redis 做 GIS 系统。

## Session / Token：Redis 适合做共享登录态

### 典型场景

- 分布式 Session
- 登录 token
- 验证码

### 为什么适合

因为这类数据通常：

- 读多写少
- 有过期时间
- 需要多实例共享

所以 Redis 非常顺手。  
一般会用：

- `String`
- 或 `Hash`

再配 TTL 管理生命周期。

### 但边界也别忘

如果你把 Redis 当成唯一登录态真相，那它的持久化和高可用边界就必须认真设计。  
否则登录态丢失会直接打用户体验。

## 限流：本质上也是计数场景

虽然限流可以单独展开，但从结构视角看，它最常见的底座还是：

- 计数器
- 过期时间
- Lua 脚本

所以限流不只是“算法题”，它本质也是 Redis 典型业务场景的一部分。

## 一个更像项目实战的答法

如果面试官问“你项目里 Redis 一般怎么用”，比起背一串名词，更好的回答是按场景说：

1. 计数器用 `String`
2. 排行榜用 `ZSet`
3. 签到和活跃标记用 `Bitmap`
4. UV 统计用 `HyperLogLog`
5. 附近检索用 `GEO`
6. Session / Token 用 `String` 或 `Hash`

然后再补一句：

**结构不是重点，真正重点是知道每种场景追求的是精确、空间、排序、共享还是低延迟。**

## 容易踩的坑

### “能用 Redis 就都塞 Redis”

不对。Redis 适合做高频访问和高并发加速，不适合独自扛所有最终正确性。

### “UV 统计一定要精确”

很多业务不需要精确到绝对值，趋势和量级更重要，这时 HyperLogLog 更合适。

### “排行榜就是把数据都放进一个 ZSet”

不一定。榜单很大、更新很频繁时，还要考虑分榜、分时间窗和冷热拆分。

## 小结

- 计数器一般用 `String`，排行榜一般用 `ZSet`，签到和活跃标记一般用 `Bitmap`。
- UV 这种海量去重计数更适合 `HyperLogLog`，附近检索更适合 `GEO`。
- Session / Token 适合放 Redis，是因为它们天然符合“共享、过期、读多写少”的模型。
- 真正好的回答不是背结构名，而是把“业务目标”和“结构特性”对应起来。
- Redis 适合做高并发场景加速层，但很多场景仍要补数据库、一致性和高可用设计。
