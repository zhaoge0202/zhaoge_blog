# Redis 的大 key 和热 key 怎么发现和处理？

> big key 和 hot key 都会把 Redis 拖慢，但一个更偏“单个 key 太大”，一个更偏“单个 key 太热”，排查和治理思路并不是一套。

很多人一提 Redis 线上问题，就会把 big key 和 hot key 混着说。其实这两个词虽然经常一起出现，但本质完全不同：

| 问题    | 本质                                | 典型症状                         |
| ------- | ----------------------------------- | -------------------------------- |
| big key | 单个 key 占用内存太大，或者元素太多 | 读写慢、删除卡顿、主从同步压力大 |
| hot key | 单个 key 访问量远高于其他 key       | CPU 飙高、带宽吃满、单点被打爆   |

所以第一步不是背定义，而是先分清：你面对的是“太大”，还是“太热”。

## big key 到底大在哪

所谓 big key，说的不是 key 名字长，而是 **key 对应的 value 太大**。

工程里常见的经验阈值通常是：

- `String` 的 value 达到几百 KB 到 1 MB 以上，就该警惕
- `Hash`、`List`、`Set`、`ZSet` 这类复合结构，成员数达到几千甚至上万，就该怀疑

这不是官方硬门槛，更像排查时的经验线。因为：

- 成员多，不一定真正占内存最多
- 成员少，如果每个元素都很大，也可能已经是 big key

所以真正要看的是：**这个 key 会不会明显拖垮内存、网络、主线程和复制链路。**

## big key 为什么危险

### 1. 单次读写会变重

比如你把一个 2 MB 的商品详情 JSON 整个塞进 `String`，那每次 `GET` 都要把这 2 MB 从 Redis 拉出来。

这会直接带来两层成本：

- Redis 自己要处理更大的对象
- 网络要搬运更大的响应体

如果这个 key 访问还很频繁，问题会被放大得很快。

### 2. 删除时容易把主线程卡住

这是线上特别常见的坑。

很多人觉得删除一个 key 就是一条 `DEL`，很轻。问题在于，如果这个 key 底下挂了非常多成员，删除动作本身就可能很重。尤其是：

- 大 `Hash`
- 大 `List`
- 大 `Set`
- 大 `ZSet`

这时候 `DEL` 可能导致明显卡顿。

### 3. 主从复制和集群迁移会更痛

big key 不只是“本机读写慢”，它还会影响复制和迁移：

- 主从同步时数据更重
- 集群扩缩容迁 slot 时迁移更慢
- 故障恢复时加载成本更高

所以 big key 经常不是一个点的问题，而是会把整个 Redis 架构都拖沉。

## 怎么发现 big key

### 1. 用 `redis-cli --bigkeys`

这是最常见的第一步。

它的优点是上手快，能快速扫出每种类型里最可疑的大 key。

但要注意两个边界：

- 它更适合低峰期执行
- 最好优先在从节点执行，别直接在繁忙主节点上硬扫

因为它本质上还是在扫 keyspace，不是零成本操作。

### 2. 用 `SCAN` 配合长度类命令

如果你想更细地查，就不要上来用 `KEYS *`，而是：

1. 用 `SCAN` 分批扫 key
2. 根据类型配合 `STRLEN`、`HLEN`、`LLEN`、`SCARD`、`ZCARD`
3. 必要时再用 `MEMORY USAGE`

这样虽然麻烦一点，但对线上更友好，也更容易按业务前缀做针对性排查。

### 3. 分析 RDB 或云厂商统计

如果实例已经比较大，直接在线扫不一定合适，这时更实用的办法反而是：

- 分析 RDB 快照
- 用云厂商提供的 key 分析能力

这类方案的好处是：**更适合做离线体检，而不是线上硬扫。**

## big key 怎么治理

### 先说最核心的一句

**big key 最好的处理方式，不是删除，而是从建模阶段避免它长出来。**

### 1. 拆 key

这是最根本的手段。

比如一个超大 `Hash` 存整个用户画像，不如按业务维度拆开：

- `user:1:profile`
- `user:1:tags`
- `user:1:settings`

或者按分片规则拆成多段，而不是把所有字段都堆进一个桶里。

### 2. 换数据结构

有些 big key 是因为结构选错了。

比如：

- 二进制大对象不该直接塞进 Redis `String`
- UV 统计不该用巨大 `Set` 硬去重，可以考虑 `HyperLogLog`
- 状态位不该用一堆散 key，可以考虑 `Bitmap`

这类优化本质不是“调 Redis”，而是“重做建模”。

### 3. 删除时别一把梭

如果已经长成了 big key，删除时要特别小心。

常见做法有两种：

- 分批删除
- 用 `UNLINK` 异步删除

这里有个面试里很容易答太绝对的点：  
`UNLINK` 不是说删除就完全没成本了，而是把真正的内存释放工作尽量挪到后台线程，减少主线程阻塞风险。它比 `DEL` 更稳，但不是无代价魔法。

## hot key 又是什么

hot key 不是“特别大”，而是“特别忙”。

比如某个商品详情在大促时每秒被读几万次，哪怕 value 只有几 KB，它照样可能成为 hot key。

所以 hot key 的核心矛盾不是内存，而是：

- CPU
- 带宽
- 单点承压

## hot key 会造成什么问题

### 1. 单个实例局部过载

Redis 即使整体 QPS 还没到极限，也可能因为某一个 key 被集中访问，导致某个分片先扛不住。

### 2. 带宽被打满

如果这个 hot key 返回值还不小，问题会更明显。  
也就是说，**big key + hot key 叠在一起，是特别危险的组合。**

### 3. Redis 顶不住后，流量继续打穿下游

如果这个热点 key 又刚好失效，或者 Redis 实例因过载抖了，请求就会继续回源数据库，最后把问题从缓存层放大到存储层。

## 怎么发现 hot key

### 1. 用 `--hotkeys`

Redis 提供了 `redis-cli --hotkeys` 这种手段，但它有明显边界：

- 不是所有场景都适用
- 本身会增加扫描成本

所以它可以用来辅助，不适合当成常驻方案。

### 2. 短时用 `MONITOR`

`MONITOR` 能看到实时命令流，但它对性能影响不小，只适合：

- 故障窗口短时抓现场
- 抓完立刻关

不能把它当成常驻监控。

### 3. 从业务侧和监控侧识别

线上更靠谱的办法通常不是完全依赖 Redis 自身命令，而是综合：

- 业务侧埋点
- 接入层热点统计
- 云厂商 key 分析
- 本地缓存命中结构

真正的 hot key，很多时候是可以结合业务提前预判的，比如：

- 秒杀商品
- 热门榜单
- 爆款直播间

## hot key 怎么治理

### 1. 本地缓存兜一层

如果某些数据极热、更新又不频繁，最常见的办法就是：

- Redis 做共享缓存
- JVM 本地缓存再挡一层

这样可以把最极端的热点从 Redis 身上卸掉。

但代价也很明确：一致性会更复杂，别见热点就条件反射上多级缓存。

### 2. 读写分离或代理层分担

如果热点以读为主，可以考虑：

- 主从分担读
- 云厂商代理层热点缓存

它解决的是“同一个 key 太多人来读”，不是数据建模问题。

### 3. 热点拆散

有些热点问题，本质上是访问被集中到单个 key。  
这时可以考虑把读流量拆散，比如分片缓存、热点副本、分桶计数。

不过要注意：  
这类方案经常会引入汇总和一致性问题，不能只看到 QPS 下降就以为结束了。

## 容易踩的坑

### “成员数多就是 big key”

不严谨。成员数只是一个很粗的信号，不等于真实内存占用。

### “hot key 只能靠扩容解决”

不对。扩容能缓解整体容量，但未必能解决单热点集中打到同一 slot、同一分片的问题。

### “查 big key / hot key 就直接在线全量扫”

风险很大。排查手段本身也会带来额外压力，线上优先考虑低峰、从库、离线分析。

## 小结

- big key 是“单个 key 太大”，hot key 是“单个 key 太热”，两个问题不要混着答。
- big key 主要拖慢读写、删除、复制和迁移，治理重点在拆分建模和安全删除。
- hot key 主要打 CPU、带宽和单分片承载，治理重点在削峰、分流和本地缓存。
- `--bigkeys`、`--hotkeys`、`MONITOR` 都能用，但都有线上成本，不能无脑常开。
- 最好的治理永远不是“线上救火”，而是业务建模时就避免把数据做成又大又热。
