# Redis 主从复制延迟和全量同步该怎么排查？

> 主从复制题如果只会背“异步复制”，其实只答到了开头。线上真正麻烦的是：为什么从库突然跟不上、为什么又退化成全量同步了、为什么主库被拖慢了。

高可用和复制原理已经讲过主线了，这一篇只盯排障视角。

## 先抓主线：复制问题通常有三种表现

| 表现               | 你先怀疑什么                           |
| ------------------ | -------------------------------------- |
| 从库延迟持续升高   | 从库处理不过来、网络慢、主库写太猛     |
| 经常退化成全量同步 | backlog 太小、断线时间太长、重连恢复差 |
| 主库突然抖动       | fork、RDB 生成、全量同步传输开销       |

所以别只说“主从延迟”，要先看它是哪一类问题。

## 为什么主从延迟本来就不是 0

Redis 复制是异步的。  
也就是说主库处理完写请求后，不会等从库确认落地才给客户端返回。

所以：

- 短暂不一致是常态
- 关键在于延迟窗口多大、是否可控

这也是为什么 Redis 主从架构追求的是高性能和可接受的一致性，而不是强一致。

## 一旦延迟上来，先看哪几类原因

### 1. 主库写入太快

如果主库短时间写流量陡增，而从库消费命令流跟不上，延迟就会上来。

### 2. 网络链路抖动

复制本质上也是网络传输。  
带宽、丢包、跨机房抖动，都会直接反映成复制延迟。

### 3. 从库本身在忙

比如：

- 从库磁盘忙
- 从库在做加载
- 从库和别的业务混部竞争 CPU

这类问题容易被忽略，因为很多人只盯主库。

## 为什么会退化成全量同步

主从断线恢复后，Redis 会优先尝试增量同步。  
但不是每次都能成功。

关键就看一件事：

**断线期间缺掉的那段命令，还在不在 backlog 里。**

如果还在，就能补那一段。  
如果已经被覆盖了，就只能重新做全量同步。

## backlog 为什么这么关键

因为 backlog 就像“断线恢复缓冲带”。  
它越小，越容易出现这种情况：

- 主库写得很快
- 从库断了一小会儿
- backlog 里的旧命令已经被新命令覆盖

这时从库一回来，主库就只能走：

- 生成 RDB
- 传 RDB
- 回放后续写命令

这就是全量同步。

所以全量同步频繁出现时，特别要怀疑：

- `repl-backlog-size` 太小
- 网络断连恢复太慢
- 主库写流量峰值太高

## 为什么全量同步会拖主库

因为它不是简单“再传一份文件”。

全量同步背后通常意味着：

- 主库 fork 子进程
- 生成 RDB
- 传输大体量数据

如果实例本来就大，这一套动作足够让主库明显抖一下。

所以面试里要补一句：  
**复制问题不只是从库读到旧数据，严重时还会反过来拖慢主库。**

## 一个更实用的排查顺序

如果线上看到主从延迟或频繁全量同步，更像真实排障的顺序是：

1. 先确认是短暂毛刺，还是持续升高。
2. 看主库写入是否突然增大。
3. 看网络链路是否抖动、重连是否频繁。
4. 看 backlog 是否足够，是否经常被覆盖。
5. 看从库自身资源是不是吃紧。
6. 如果发生全量同步，再重点看主库 fork、RDB 和传输成本。

## 复制延迟大时，业务上要怎么想

这时别只想着“怎么让指标变小”，还要想业务影响：

- 从库还能不能继续承担读流量
- 某些强依赖实时一致读的请求是否该回主库
- 故障切换时会不会把一个落后过多的从库提成新主

也就是说，复制延迟不只是 Redis 指标，它直接关系到读写分离和故障切换策略。

## 容易踩的坑

### “主从延迟上来就一定是主库问题”

不一定。从库资源、网络链路同样很常见。

### “断线恢复默认都会走增量同步”

不对。只有缺失数据还在 backlog 里，才有机会增量恢复。

### “全量同步只是同步慢一点”

太轻了。全量同步会放大 fork、传输和主库负载问题。

## 小结

- Redis 主从复制天然是异步的，所以短暂延迟本来就不是异常。
- 复制延迟持续升高时，优先看主库写流量、网络链路和从库资源。
- 是否能增量恢复，关键看 backlog 里还有没有断线期间缺失的数据。
- backlog 太小、网络恢复太慢、写流量太猛，都会让恢复退化成全量同步。
- 全量同步不仅影响从库恢复，还会通过 fork 和 RDB 生成反过来拖主库。
