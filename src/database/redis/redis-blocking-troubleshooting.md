# Redis 卡顿了该从哪里开始排查？

> Redis 卡顿排查最怕的不是问题难，而是上来就盯着某一条命令看，结果把 CPU、持久化、网络、内存交换这些方向全漏了。

线上说“Redis 慢了”，通常不是一个标准化故障，而是一组表象：

- 请求超时
- RT 飙高
- QPS 掉下去
- 客户端连接堆积
- 主从延迟突然变大

这时候如果只会背“Redis 是单线程，所以被慢命令卡住了”，答得太薄。  
更完整的理解应该是：**Redis 的主线程很怕被重活打断，而这个重活不只来自业务命令，也可能来自持久化、删除、复制、网络和操作系统。**

## 先抓主线：排查时先分四类

| 方向         | 典型表现                    | 常见根因                               |
| ------------ | --------------------------- | -------------------------------------- |
| 命令层       | 某类请求 RT 明显升高        | 慢命令、范围过大、全量扫描             |
| 数据层       | 个别 key 操作特别重         | big key、批量删除、热点倾斜            |
| 持久化复制层 | fork 抖动、磁盘忙、主从延迟 | `bgsave`、AOF rewrite、fsync、全量同步 |
| 系统层       | 整体实例变慢                | 网络抖动、CPU 竞争、swap、内存压力     |

所以不要一上来就问“是不是有慢查询”，应该先问：**慢的是哪一层。**

## 第一类：先看是不是命令本身太重

Redis 的很多命令是轻的，但不是所有命令都轻。

最容易出事的一批通常是：

- `KEYS *`
- `HGETALL`
- `LRANGE` 大范围读取
- `SMEMBERS`
- `SUNION`、`SINTER`、`SDIFF`
- 大范围 `ZRANGE`、`ZREMRANGEBYSCORE`

问题不在命令名字，而在于它们对数据量很敏感。  
当 `N` 很大时，哪怕命令本身没写错，也会把主线程拖住。

### 面试里要补的一句

不是说这些命令一律不能用，而是：

- 要明确数据规模
- 要明确是否在线上高峰使用
- 能用增量遍历时尽量别用全量拉取

比如有遍历需求时，优先考虑 `SCAN`、`HSCAN`、`SSCAN`、`ZSCAN`，而不是直接全量扫。

## 第二类：看是不是 big key 或重删除

如果故障集中在某几个 key 上，就要立刻怀疑：

- 是不是 big key
- 是不是对 big key 做了重操作

常见场景包括：

- 读取超大对象
- `DEL` 大 `Hash` / `List` / `Set` / `ZSet`
- `FLUSHDB` / `FLUSHALL`

这里最容易答错的一点是：  
**删除不是天然轻操作。**

释放一个很大的对象，主线程照样可能被拖住，所以线上更稳的思路通常是：

- 分批删
- 优先 `UNLINK`
- 避免在高峰期做大范围清理

## 第三类：看是不是慢命令日志已经给你信号了

Redis 不是 MySQL，但它也有自己的慢日志：`SLOWLOG`。

这类工具的价值不是帮你“证明 Redis 变慢”，而是帮你定位：

- 到底是哪类命令慢
- 是偶发一次，还是持续性出现
- 是某个 key 模式，还是整个实例都在抖

要注意一个边界：

- `SLOWLOG` 记录的是命令执行耗时
- 它不等于完整的请求总耗时

比如网络排队、客户端连接池等待，并不都算在这条日志里。

所以如果应用侧 RT 很高，但 `SLOWLOG` 没明显异常，就别死盯命令执行，可能问题在网络、CPU 争抢或下游链路。

## 第四类：看持久化是不是在拖实例

### 1. `save`

这个最直接。  
同步做 RDB 快照，会阻塞主线程，生产里基本不该碰。

### 2. `bgsave`

很多人会说 `bgsave` 不阻塞，这句话不严谨。

更准确地说：

- 真正写 RDB 文件是在子进程做
- 但 fork 本身会带来抖动
- 后续写时复制还会带来额外内存压力

所以大实例、写流量高时，`bgsave` 照样可能让延迟变差。

### 3. AOF 刷盘

如果 AOF 打开了，排查时要看：

- `appendfsync` 配的是什么
- 磁盘是不是忙
- 后台 `fsync` 有没有把主线程反压住

尤其是磁盘抖的时候，AOF 这条链路经常会把问题放大。

### 4. AOF rewrite

它不是“后台重写就完全无感”。  
fork、重写缓冲区回放、新旧文件切换，这些阶段都可能带来抖动。

## 第五类：看主从复制和集群动作

Redis 卡顿有时候不是业务自己打出来的，而是复制链路拖出来的。

比如：

- 从节点重连导致全量同步
- 主节点突然 fork 生成 RDB
- 集群扩缩容时迁 slot
- big key 迁移时间过长

这类问题的特点是：

- 业务代码没改
- 但实例突然在某个时点抖一下

如果只盯请求日志，很容易误判成“流量波峰”。

## 第六类：看是不是 CPU 被抢了

Redis 虽然是内存型系统，但并不意味着只要内存够就行。  
它对 CPU 很敏感，尤其不适合和重 CPU 任务混跑。

典型问题有：

- 同机部署别的 CPU 密集型服务
- 容器限额太紧
- 热 key 导致单核打满

如果主线程所在核一直高占用，即使命令本身不重，排队也会明显变长。

## 第七类：看是不是 swap 了

这是 Redis 很怕、但很多人平时不会第一时间想到的方向。

Redis 的价值就在于“数据在内存里”。  
如果操作系统把一部分页换到磁盘，延迟会立刻变得很难看。

所以排查慢问题时，别只看 Redis 指标，还要看：

- 机器可用内存是不是过低
- `maxmemory` 有没有设
- 系统是不是发生了内存交换

一旦 swap 发生，很多“命令看起来没问题但实例就是很慢”的现象就解释通了。

## 第八类：看网络，而不是只看 Redis

有些“Redis 超时”其实不是 Redis 执行慢，而是网络链路有问题，比如：

- 网卡带宽打满
- 软中断高
- 突发丢包
- 跨机房延迟抖动

尤其是 big key、hot key 问题，会很容易把网络侧一起拖起来。

所以当你看到：

- 应用侧超时明显
- Redis CPU 却不算太高
- `SLOWLOG` 也没特别夸张

就该去怀疑网络了，而不是继续在命令上死磕。

## 一个更实用的排查顺序

如果面试官问“线上 Redis 卡顿怎么排”，比起散点罗列，下面这套顺序更像真做过：

1. 先确认是整体慢，还是某类请求慢，还是某几个 key 慢。
2. 看应用 RT、超时、连接池堆积，再对照 Redis 自身延迟和 QPS。
3. 查 `SLOWLOG`、命令分布、是否存在大范围扫描或重操作。
4. 排查 big key、hot key、批量删除、清库、迁移。
5. 看持久化：最近有没有 `bgsave`、AOF rewrite、磁盘刷盘抖动。
6. 看复制和集群：有没有全量同步、slot 迁移、主从延迟。
7. 最后看系统层：CPU 抢占、带宽、swap、内存压力。

这套顺序的核心价值是：**从最可能、最可观测的方向先收敛，而不是靠猜。**

## 容易踩的坑

### “Redis 慢就是单线程的问题”

太粗了。单线程只是背景，不是根因。真正的问题是主线程被谁拖住了。

### “没有慢日志，就说明 Redis 没问题”

不成立。网络、CPU 争抢、swap、复制链路都可能让请求变慢，但未必完整体现在慢日志里。

### “只要不用 `KEYS *` 就安全了”

远远不够。大范围 `ZRANGE`、`HGETALL`、大 key 删除、AOF 刷盘都可能把实例打抖。

## 小结

- Redis 卡顿排查不要只盯慢命令，要同时看命令层、数据层、持久化复制层和系统层。
- 最常见的业务侧根因是大范围扫描、big key、重删除、hot key。
- 最常见的基础设施侧根因是 fork、AOF 刷盘、全量同步、slot 迁移、CPU 竞争和 swap。
- `SLOWLOG` 很有用，但它不是完整请求耗时的全景图。
- 真正好的排查方式不是背故障词典，而是按“先收敛范围，再逐层定位”的路径走。
