# Redis Cluster 在工程上怎么部署和扩缩容？

> Cluster 细节那篇讲的是槽位和路由，这一篇更偏工程落地：节点怎么搭、扩缩容时业务会不会抖、什么情况下集群会被热点槽和大 key 拖歪。

很多人面试里一说 Cluster，就是：

- 16384 个槽
- 分片集群

这当然是基础，但如果你真在线上用过，更关心的问题通常是：

- 一组集群最小怎么搭
- 扩容怎么迁槽
- 缩容怎么收槽
- 迁槽时业务会不会中断
- 什么情况下某个分片会先被打爆

## 先说最常见的拓扑

工程上最常见的是：

- 多个主节点负责分槽
- 每个主节点至少配一个从节点

这样设计的目的很直接：

- 主节点扛写和主路由
- 从节点做故障切换备份

所以 Cluster 不是“多台机器一起存数据”这么简单，而是 **分片 + 每片高可用**。

## 扩容本质上是在做什么

不是把新节点加进来就完了。  
真正的扩容过程是：

1. 新节点加入集群
2. 从老主节点手里拿走一部分槽
3. 这些槽对应的 key 被迁到新节点
4. 客户端逐步感知新的槽位归属

也就是说，扩容的核心动作不是“加机器”，而是 **迁槽**。

## 迁槽时为什么还能继续服务

这是 Cluster 非常有价值的一点。  
它支持在线迁槽，所以理论上不需要像某些系统一样整体停机迁移。

但“能继续服务”不等于“毫无波动”。  
迁槽期间经常要付出这些代价：

- 请求重定向增加
- 某些 key 访问会变慢
- 大 key 迁移时间更长
- 热槽迁移时对业务更敏感

所以更准确的说法应该是：

**Redis Cluster 可以在线扩缩容，但迁槽过程依然可能带来局部抖动。**

## 缩容时为什么更要谨慎

扩容是把槽摊开，缩容是把槽收回来。  
缩容的风险通常更高一点，因为你是在把原本分散的压力重新汇聚。

这时如果你没看清：

- 哪些槽本来就热
- 哪些节点上有大 key

就很容易在缩容后把某个分片直接压歪。

## 热点槽为什么比热点 key 更难看

hot key 你已经知道了，它会打爆单个 key。  
但在 Cluster 里，还有一层更麻烦的概念：**热槽**。

也就是：

- 不止一个 key 热
- 而是恰好很多热 key 都落在同一槽或同一分片

这时候问题就不再是单 key，而是整个分片被打得更重。

所以 Cluster 场景下，除了看热 key，还要看：

- 槽位倾斜
- 节点 QPS 倾斜
- 节点内存倾斜

## 大 key 为什么会让迁槽更难受

大 key 在单机上已经麻烦，在 Cluster 里问题会更明显：

- 迁移更慢
- 网络压力更大
- 单次操作更重
- 更容易放大重定向窗口

所以 Cluster 工程里一个特别重要的经验是：  
**别等扩缩容时才发现大 key。**

应该在平时就把大 key 体检做掉。

## 一个更稳的扩容思路

如果你要回答“Cluster 扩容怎么做”，比较稳的表达可以是：

1. 先加新节点进集群。
2. 评估当前各主节点槽位、QPS、内存是否均衡。
3. 分批迁移一部分槽到新节点，而不是一次性暴力搬。
4. 重点关注热槽和大 key，避免迁槽过程放大抖动。
5. 迁完后继续观察是否出现新的分片倾斜。

注意这里的重点不是操作命令细节，而是：  
**扩容不是平均分槽就结束，还要观察流量和数据是否真的均衡。**

## Cluster 不是自动均衡魔法

这点特别值得单独说。

很多人默认只要用了 Cluster，数据和流量就自然均匀了。  
其实未必。

原因很简单：

- key 分布可能不均
- 业务热点可能集中
- hash tag 可能故意把相关 key 放同槽

所以 Cluster 解决的是“支持分片”，不是“自动消灭倾斜”。

## 容易踩的坑

### “扩容就是加节点，Redis 会自动搞平衡”

不严谨。加入新节点之后，真正的工作量在迁槽和均衡验证。

### “迁槽能在线做，所以不会影响业务”

不对。它可以在线做，但局部抖动、重定向和大 key 迁移成本依然存在。

### “只看槽位数量均衡就够了”

不够。真正要看的是槽位、QPS、内存和热点分布是否一起均衡。

## 小结

- Cluster 工程化的核心不是“多节点”，而是“分槽 + 迁槽 + 分片高可用”。
- 扩容本质是把部分槽迁到新节点，缩容本质是把槽收回旧节点。
- Cluster 支持在线迁槽，但不等于迁移过程完全无波动。
- 热槽和大 key 是扩缩容里最容易被低估的两个风险点。
- 用 Cluster 之后仍然要持续关注分片倾斜，别把它当自动均衡魔法。
