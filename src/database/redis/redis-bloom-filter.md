# Redis 里的布隆过滤器到底怎么用？

> 布隆过滤器最常见的出场方式是“防缓存穿透”，但如果只会背“它占内存小，有误判”，其实还没答到真正关键的边界。

很多人第一次接触布隆过滤器，是在缓存穿透场景里：

1. 请求先查 Redis，没有
2. 再查数据库，也没有
3. 攻击流量不断来，数据库被拖死

这时就会听到一句经典建议：  
**在 Redis 前面加一层布隆过滤器。**

这句话方向没错，但如果不把原理和边界讲清，很容易在面试或者设计里答得太绝对。

## 先说它到底解决什么问题

布隆过滤器不负责存完整数据，它更像一个很省内存的“存在性预检器”。

它最适合回答的问题是：

- 这个 key 大概率存在吗？
- 如果它一定不存在，我能不能立刻拦掉后续更贵的查询？

所以它特别适合：

- 缓存穿透预判
- 用户名、域名、券码是否已出现的快速预检
- 海量去重前的第一层快速过滤

## 它为什么省内存

布隆过滤器的核心不是“存元素集合”，而是：

- 一个位数组
- 多个哈希函数

当一个元素写入时，它会：

1. 经过多个哈希函数计算
2. 落到位数组里的多个位置
3. 把这些位置都标成 `1`

判断某个元素是否存在时，也走同样的哈希过程：

- 只要有一个位置还是 `0`，那它就一定不存在
- 如果所有位置都是 `1`，那它只能说明“可能存在”

这就是它省内存的本质：  
**它不保存原始元素，只保存哈希映射后的痕迹。**

## 最重要的一句边界

布隆过滤器的判断特性可以直接记成一句：

- 说“不存在”时，基本可以信
- 说“存在”时，只能说大概率

也就是常说的：

- **可以接受误判**
- **不能接受漏判**

更准确一点说：

- 会有假阳性，也就是“本来不存在，但它说可能存在”
- 不会有假阴性，也就是“本来存在，却说一定不存在”

这也是它能用于缓存穿透治理的根本原因。

## 为什么它特别适合挡穿透

缓存穿透最麻烦的点在于：

- Redis 没有
- 数据库也没有
- 但请求还会源源不断重复来

如果你在数据库外面先放一个布隆过滤器，流程就会变成：

1. 请求先查缓存，没有
2. 再查布隆过滤器
3. 如果布隆过滤器直接判定“不存在”，那就不用再查数据库

这样数据库就不会被一堆无效 key 拖住。

所以它的真正价值不是“让结果更准”，而是：  
**用极低的内存成本，把大量明显不存在的请求挡在数据库外面。**

## 但它不是银弹

### 1. 误判还是会发生

如果布隆过滤器判断“可能存在”，你通常还是得继续查：

- Redis
- 数据库

所以它不能完全消灭穿透流量，只能明显减少无效回源。

### 2. 删除不友好

经典布隆过滤器的一个硬伤是：  
它适合加，不适合删。

因为多个元素可能共享同一批 bit 位。  
你把某个位清掉，可能会误伤别的元素。

这也是为什么很多业务里，它更适合：

- 相对稳定的数据集
- 以新增为主的存在性判断

而不适合频繁删改的动态集合。

### 3. 数据同步不能忘

如果你用了布隆过滤器挡穿透，业务侧就必须考虑：

- 新数据写入数据库时，什么时候写过滤器
- 过滤器和数据库出现短暂不一致怎么办

所以它不是“配一个组件就完事”，而是你要把写路径也设计进去。

## Redis 里怎么做布隆过滤器

这里要先说清一个容易混淆的点：

**布隆过滤器不是 Redis 五大基础数据类型之一。**

如果你想在 Redis 里用得比较顺手，常见做法是基于 `RedisBloom` 这类模块能力。

官方文档里也把 Bloom Filter 放在概率型数据结构分类下，而不是基础类型里。

常见命令包括：

- `BF.RESERVE`
- `BF.ADD`
- `BF.EXISTS`
- `BF.MADD`
- `BF.MEXISTS`

这套命令的核心好处是：

- 直接可用
- 空间效率高
- 读写都快

## 一个更稳的业务用法

在缓存穿透场景里，更稳的思路通常是：

1. 业务写入数据库成功后，同时把主键标进布隆过滤器。
2. 读请求先查缓存。
3. 缓存没命中，再查布隆过滤器。
4. 过滤器判定一定不存在，就直接返回，不打数据库。
5. 过滤器判定可能存在，再继续查数据库并决定是否回填缓存。

这套流程的重点不是“精准”，而是“挡掉大部分无意义请求”。

## 那它和空值缓存怎么选

这两个方案经常一起出现。

### 空值缓存

优点：

- 实现简单
- 对单个不存在 key 很直接

缺点：

- 要为很多不存在的 key 占缓存空间
- 面对恶意海量乱 key 时，效果一般

### 布隆过滤器

优点：

- 更省空间
- 适合海量存在性预判

缺点：

- 有误判
- 删除和同步更麻烦

所以工程上更常见的判断是：

- key 空间不大、实现优先简单：空值缓存够用
- key 空间很大、穿透压力重：布隆过滤器更有价值

## 容易踩的坑

### “布隆过滤器能精确判断元素存在”

不对。它只能精确判断“不存在”，不能精确判断“存在”。

### “加了布隆过滤器就没有缓存穿透了”

不严谨。它会大幅减轻穿透，但误判流量还是会继续往后走。

### “Redis 原生五大类型里就有布隆过滤器”

不对。实际工程里通常依赖模块能力，比如 RedisBloom。

## 小结

- 布隆过滤器适合做存在性预检，最典型的应用就是防缓存穿透。
- 它的核心特点是：能确定“不存在”，但对“存在”只能给概率判断。
- 它省内存，是因为只保存哈希后的位信息，不保存原始元素。
- 它不适合频繁删除和强一致动态集合，使用时要考虑写路径同步。
- 在 Redis 里落地时，常见方案是基于 RedisBloom 一类概率型数据结构能力。
