# RDB、AOF 和混合持久化怎么选？

> Redis 快，不代表它不丢数据。持久化这道题真正想问的是：当性能、恢复速度和数据安全互相打架时，你知道自己在牺牲什么吗？

很多人对 Redis 的印象是“内存数据库，很快”。这句话没错，但只说这一句不够。因为只要 Redis 不只是临时缓存，而是承接了会话、计数、排行榜、消息流、热点配置这些业务，下一步就一定会被问：

- Redis 宕机了数据怎么办？
- RDB 和 AOF 有什么区别？
- 为什么 AOF 更安全但不一定是默认？
- `appendfsync everysec` 到底意味着什么？
- fork 为什么会把 Redis 打卡顿？

所以这道题的核心，不是背定义，而是讲清：**Redis 持久化到底在换什么。**

## 先立住一个大前提：Redis 的持久化不是 MySQL 那种强持久

Redis 支持持久化，但它的设计目标从来不是把自己做成“绝不丢一条”的事务型数据库。

也就是说，Redis 的持久化更多是在三个目标之间找平衡：

| 目标     | 代表问题                   |
| -------- | -------------------------- |
| 恢复速度 | 重启后能多快把数据拉起来   |
| 数据安全 | 宕机后最多丢多少数据       |
| 运行性能 | 平时写入时会不会拖慢主线程 |

后面你看到的 RDB、AOF、混合持久化，本质上都是这三个目标的不同取舍。

## RDB：存某个时刻的快照，恢复快，但可能丢一段时间的数据

RDB 的思路很好理解：在某个时间点，把 Redis 当前内存里的数据整体拍一张“快照”落到磁盘。

所以它记录的是**某一刻的数据结果**，不是每一条写命令。

### RDB 的优点

- 文件更紧凑，适合备份
- 恢复速度快，重启时直接把数据载入内存
- 对运行期写入开销相对可控

### RDB 的问题

它是“间隔性拍快照”，不是“每次写都记下来”。

这意味着如果 Redis 刚做完一次快照，之后又写了 5 分钟的数据，结果机器挂了，那这 5 分钟的数据可能就没了。

所以 RDB 的问题可以用一句话总结：

**恢复快，但数据新鲜度取决于快照频率。**

## RDB 为什么通常用 `bgsave`，而不是 `save`

`save` 是同步生成快照，会直接阻塞 Redis 主线程。

这在生产环境里基本等于自找麻烦，因为 Redis 正常就是靠主线程持续处理命令的。你把它停下来拍快照，线上延迟会立刻上来。

所以实际更常见的是 `bgsave`：fork 一个子进程去生成 RDB，主线程继续处理请求。

### 但 `bgsave` 也不是“完全没代价”

这点特别容易被说错。

`bgsave` 虽然不是在主线程里真正写 RDB 文件，但它前面有个很关键的动作：**fork 子进程**。

fork 本身就会带来两个代价：

- 复制页表，主线程会短暂阻塞
- 后续如果主线程继续写数据，会触发写时复制（COW），额外吃内存

所以 Redis 大数据量、写流量又高的时候，RDB 最大的风险不只是“丢几分钟数据”，还有：

- fork 延迟抖动
- 快照期间内存膨胀
- 大 key 修改导致 COW 更重

这也是为什么很多线上 Redis 卡顿，最后根因会查到 fork。

## AOF：把写命令追加到日志里，数据更完整，但恢复更慢

AOF 的思路和 RDB 不一样。

它不拍“结果图”，而是记“操作过程”。

比如你执行了：

```text
SET name alice
INCR counter
HSET user:1 age 28
```

这些写命令会被追加进 AOF 文件。Redis 重启时再把这些命令重新执行一遍，就把数据恢复回来了。

### AOF 的优点

- 粒度更细，通常比 RDB 丢数据少
- 可读性更强，排查时更直观
- 可以通过重写压缩历史命令

### AOF 的问题

- 文件会不断变大
- 恢复时要重放命令，通常比 RDB 慢
- 写入路径更容易受磁盘 IO 影响

所以 AOF 更像是在用**更多日志开销**换**更细粒度的数据保留**。

## AOF 的关键不只是开没开，而是怎么刷盘

面试里最容易被追问的，就是 `appendfsync`。

### 1. `always`

每次写命令后都尽量同步刷盘。

优点：

- 理论上数据最安全

缺点：

- 性能最差
- 主线程很容易被磁盘拖慢

### 2. `everysec`

每次写先进入 AOF 缓冲区，由后台线程大约每秒刷一次盘。

优点：

- 性能和安全性比较均衡
- 生产里最常见

缺点：

- 理论上会丢最近 1 秒左右的数据
- 磁盘压力大时，主线程仍可能被拖慢

### 3. `no`

由操作系统决定什么时候把缓冲区真正刷到磁盘。

优点：

- 性能最好

缺点：

- 丢多少数据不可控

所以这三者不是“谁先进”，而是谁更适合你的业务容忍度。

## 为什么很多场景会优先选 `everysec`

因为它通常是最现实的平衡点。

你可以这么理解：

- `always` 太贵，主线程容易被 IO 顶住
- `no` 太冒险，丢多少没谱
- `everysec` 大多数业务能接受

所以如果面试官问“线上一般怎么选”，一个比较稳的回答是：

**如果 Redis 不只是可有可无的临时缓存，而是承载了比较重要的业务状态，通常会倾向 AOF 的 `everysec`，在性能和数据安全之间取平衡。**

## AOF 为什么是在执行命令之后记日志

这个点很容易和 MySQL 搞混。

Redis AOF 通常是**命令执行成功之后**再把命令追加进日志，而不是像某些数据库那样先记日志再执行。

这样做的好处是：

- 不用为非法命令额外做复杂校验
- 当前命令不会先被日志写入动作阻塞

但代价也很直接：

- 命令执行成功但日志还没来得及落盘时，宕机就会丢

所以又回到了前面的结论：Redis 的持久化是尽量保，不是绝对不丢。

## AOF 重写为什么重要

如果 AOF 每次都只追加，文件会越来越大。

比如一个 key 被更新了 100 次，AOF 里可能会记 100 条命令，但恢复时你真正需要的，只是最后状态。

所以 Redis 会做 AOF rewrite，把当前内存状态重新组织成更紧凑的新 AOF 文件。

### 要注意的一点

“重写”不是在旧文件里逐行改，而是：

- 根据当前数据生成一个新的 AOF 文件
- 最后再替换旧文件

这样更安全，也更容易避免旧文件被污染。

## 混合持久化：想同时拿 RDB 的恢复速度和 AOF 的数据完整性

Redis 4.0 之后支持混合持久化。

它的思路是：

- AOF 重写时，前半段先写一份 RDB 风格的全量数据
- 后半段再追加增量 AOF 命令

这样做的好处是：

- 启动恢复时，先快速加载前面的 RDB 部分
- 再补后面的增量命令

于是它同时改善了两个问题：

- 纯 AOF 重放太慢
- 纯 RDB 容易丢较长时间窗口的数据

所以混合持久化本质上不是“第四种持久化”，而是把 RDB 和 AOF 拼到一起，拿两边的长处。

## 线上到底怎么选

### 场景一：Redis 只是普通缓存，丢了可以重建

比如商品详情、配置缓存、推荐结果缓存，这类数据本来就能从 MySQL 或其他存储重建。

这时很多团队甚至会弱化持久化要求，因为 Redis 真挂了也能慢慢回源重建，只是性能退化。

### 场景二：Redis 承载重要状态，但允许少量丢失

比如排行榜、计数、部分会话状态、延迟任务。

这类场景常见选择是：

- AOF `everysec`
- 或者混合持久化

因为这类业务通常既不能像纯缓存那样完全无所谓，又没到金融账务那种“绝不容错”的级别。

### 场景三：对正确性要求极高

这时要先提醒一句：**别把 Redis 持久化想成关系型数据库级别的强保障。**

如果业务真的强依赖不丢、不重、不乱序，那关键状态最好还是落在更强约束的存储上，Redis 做加速层，而不是单独扛最终正确性。

## 面试里最容易说错的点

### “RDB 不影响性能，因为它在子进程里做”

不严谨。真正的快照写文件在子进程里，但 fork 和 COW 都会产生代价。

### “AOF 一定比 RDB 好”

不成立。AOF 数据更细，但恢复更慢、运行期开销更高。没有绝对更好，只有取舍不同。

### “开了 AOF everysec 就只会丢 1 秒”

这是经验说法，不该讲成绝对承诺。磁盘抖动严重时，主线程也可能受影响。

## 小结

- RDB 记录的是某个时间点的全量快照，恢复快，但可能丢两次快照之间的数据。
- AOF 记录的是写命令，通常比 RDB 丢得少，但恢复慢、文件会变大、IO 开销更高。
- `appendfsync everysec` 往往是生产里更常见的折中选择。
- `bgsave` 和 AOF rewrite 虽然走子进程，但 fork 和写时复制依然会带来性能与内存代价。
- 混合持久化本质是用 RDB 加快恢复、用 AOF 保留增量，适合想兼顾两边优点的场景。
