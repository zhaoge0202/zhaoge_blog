# 缓存雪崩、击穿、穿透怎么治理？

> 这三兄弟看起来都像“缓存没命中，数据库扛不住了”，但真正面试时，面试官最想听的不是定义，而是你能不能分清它们为什么发生、该怎么治。

很多人会把这三种问题背成一句口诀：

- 雪崩：大量 key 同时失效
- 击穿：热点 key 失效
- 穿透：查的是不存在的数据

这三句没错，但太薄。线上处理问题时，关键不在背定义，而在于能不能判断：

- 这次数据库被打爆，到底是哪一种？
- 限流、互斥锁、布隆过滤器、空值缓存该放在哪用？

## 先把差异拉开

| 问题 | 本质                      | 典型现象                                 |
| ---- | ------------------------- | ---------------------------------------- |
| 雪崩 | 大量请求同时回源数据库    | 大量 key 同时过期，或 Redis 整体不可用   |
| 击穿 | 单个热点 key 回源把库打穿 | 某个超热点 key 刚好失效                  |
| 穿透 | 请求的数据本来就不存在    | 缓存查不到，数据库也查不到，还不断重复查 |

所以判断的关键是：

- 看是“一大片 key 出问题”还是“某个热点 key 出问题”
- 看数据库里到底有没有这条数据

## 缓存雪崩：大面积回源

雪崩最典型的两种触发方式：

### 1. 大量 key 在同一时间过期

比如你一次性给一批商品缓存统一设置了 30 分钟 TTL，30 分钟一到，所有请求一起回源数据库。

### 2. Redis 本身故障

比如 Redis 实例挂了、网络断了、机房抖了，结果本来应该被缓存挡住的请求一下全打到数据库。

### 雪崩怎么治

#### 1. 过期时间打散

这是最基础的一招。

不要让大量 key 共用同一个 TTL，而是加随机抖动，比如：

- 30 分钟 + 0~5 分钟随机值

这样它们不会同一秒集体过期。

#### 2. 热点数据预热

系统启动、活动开始前，提前把热点数据塞进缓存，别等第一波用户来时才触发回源。

#### 3. 回源时加互斥控制

避免所有请求一起重建缓存。可以让一个线程去查库重建，其他线程等待或降级。

#### 4. 限流 / 熔断 / 降级

如果 Redis 整体故障，真正救数据库的往往不是“赶紧查库”，而是：

- 入口限流
- 热门接口降级
- 非核心数据直接返回兜底值

这点很重要：**雪崩真正危险的不是缓存 miss，而是数据库被拖死。**

## 缓存击穿：一个热点 key 把数据库打穿

击穿可以看成雪崩的子集，但它更聚焦。

它的典型场景是：

- 某个热点 key 平时请求量非常大
- 它刚好失效
- 这一瞬间大量请求同时查库重建

比如秒杀商品详情、首页 banner、爆款活动库存展示这类数据。

### 为什么击穿比普通 miss 更危险

因为普通 key 就算失效，回源量通常也有限。

但热点 key 不一样，它的流量是瞬时集中的，一旦失效，数据库几乎是硬扛同一份查询的洪峰。

### 击穿怎么治

#### 1. 互斥锁 / 单飞

这是最经典的办法。

只有一个线程拿到锁去查数据库重建缓存，其他线程：

- 等待
- 重试
- 或者直接返回兜底值

#### 2. 热点 key 不主动过期

对于特别核心的热点数据，可以不依赖 TTL，而是：

- 后台定时刷新
- 快过期前异步续命

这样能显著降低热点 key 在高峰期刚好失效的概率。

#### 3. 本地缓存 + Redis 多级缓存

如果热点极高，还可以用本地缓存兜一层，进一步减轻 Redis 和数据库压力。

但这会引入一致性问题，所以别一听到“多级缓存”就条件反射地上。

## 缓存穿透：查的根本不是有效数据

穿透和前两种最大的区别是：

**数据库里本来就没有这条数据。**

所以请求路径会变成：

1. 查 Redis，没有
2. 查数据库，也没有
3. 由于没有结果，缓存没法正常建立
4. 下次请求再来，还是同样流程

如果恶意请求很多，数据库就会被一堆根本不存在的数据查询拖死。

### 穿透常见来源

#### 1. 参数本身非法

比如用户传了负数 ID、超长字符串、明显不存在的业务类型。

#### 2. 业务误删

缓存没了，数据库里这条记录也没了，结果不断被回源。

#### 3. 恶意攻击

有人专门枚举不存在的 key 或订单号来打接口。

## 穿透怎么治

### 1. 参数校验挡在最前面

这是成本最低的一层。

如果请求 ID 明显非法、格式不对、范围不可能存在，就别让它进缓存层和数据库层。

### 2. 缓存空值

如果数据库查出来确实没有，也可以把“没有”这个结果缓存起来，设置一个较短 TTL。

这样下一次同样的请求就不会再打数据库。

这招很实用，但要注意：

- TTL 不宜太长，避免把“刚创建出来的新数据”挡住太久

### 3. 布隆过滤器

这是面试里特别爱追问的一层。

它的作用不是替代数据库，而是在请求进数据库前先做一次“快速存在性判断”。

如果布隆过滤器判断“肯定不存在”，那就不用再查数据库了。

### 布隆过滤器该怎么描述

不用把数学原理讲得特别细，面试时一般这样说就够：

- 它用一组位图和多个哈希函数标记“可能存在的数据”
- 它的特点是：
  - 说“不存在”，通常就真不存在
  - 说“存在”，只是可能存在

所以它特别适合用来拦截不存在的数据请求，但不能把它当最终真相来源。

## 三类问题该怎么快速区分

如果线上报警是“数据库 QPS 暴涨”，你可以先按这个顺序判断：

### 看 Redis 是不是整体异常

如果 Redis 集体抖动、连接超时，那先怀疑雪崩。

### 看是不是某个热点接口突然暴涨

如果只有一个超热点业务回源，先怀疑击穿。

### 看数据库有没有大量“查无此数据”

如果大量 SQL 都是在查不存在的 id，那更像穿透。

## 实战里通常不是单招，而是组合拳

比如一个成熟点的方案，往往会同时有：

- TTL 打散
- 热点预热
- miss 回源加互斥锁
- 参数校验
- 空值缓存
- 布隆过滤器
- 限流熔断

也就是说，这三类问题不是“选一招秒杀”，而是**分层防守**。

## 面试里容易答错的点

### “击穿和雪崩就是一个东西”

不准确。击穿是热点 key 失效，雪崩是大面积失效或 Redis 整体不可用。

### “穿透就一定上布隆过滤器”

不一定。很多业务先靠参数校验和空值缓存就够了。布隆过滤器适合规模大、穿透风险高的场景。

### “有互斥锁就完了”

不够。互斥锁能挡住重复重建，但挡不住 Redis 整体故障，更挡不住恶意不存在请求。

## 小结

- 雪崩是大面积回源，击穿是热点 key 回源，穿透是请求的数据本来就不存在。
- 雪崩重点在打散 TTL、预热、限流和降级。
- 击穿重点在热点 key 的互斥重建和异步刷新。
- 穿透重点在参数校验、空值缓存和布隆过滤器。
- 实战里通常不是单个技巧，而是多层组合防守。
