# Redis 的特殊数据类型该怎么选？

> 很多人背 Redis 数据类型时，只会背 `Bitmap、HyperLogLog、GEO、Stream` 这几个名字，但面试真正想听的是：你什么时候该用它，什么时候别硬上。

五种常见类型之外，Redis 还有一批更偏“场景型”的结构。  
它们的共同特点是：

- 不是所有业务都需要
- 但一旦场景对上，性价比非常高

最典型的就是：

- `Bitmap`
- `HyperLogLog`
- `GEO`
- `Stream`

## 先抓主线：它们分别在解决什么问题

| 类型        | 适合解决的问题    | 核心优势                           |
| ----------- | ----------------- | ---------------------------------- |
| Bitmap      | 海量 0/1 状态记录 | 极省空间，按位统计快               |
| HyperLogLog | 海量去重计数      | 内存几乎固定，适合 UV 一类近似统计 |
| GEO         | 地理位置检索      | 内置距离和附近查询                 |
| Stream      | 轻量消息流        | 消费组、ACK、可回溯                |

所以别把它们看成“四个冷门知识点”，更应该看成“四类高频业务模型”。

## Bitmap：适合“只有 0/1 状态”的海量统计

### 它本质上是什么

官方文档说得很明确：  
Bitmap 不是一种独立的底层类型，而是在 `String` 之上定义出来的一组位操作。

也就是说，你可以把它理解成：

- 一个很长的位数组
- 每一位只表示一个状态

这就决定了它特别适合：

- 签到
- 某天是否登录
- 某用户是否做过某个动作
- 活跃标记

### 为什么它省空间

因为一个状态只占 1 bit。  
如果你用普通 `String`、`Set` 或 `Hash` 去存这种 0/1 状态，会浪费很多空间。

### 什么时候别用

如果状态不是简单 0/1，而是：

- 要存多个字段
- 要高频按对象维度查详情

那 Bitmap 就不合适了。它适合的是“批量状态位”，不是通用对象存储。

## HyperLogLog：适合“我只想知道大概有多少个不同元素”

### 它解决的是什么问题

最典型的是：

- UV
- 独立访客数
- 独立 IP 数
- 海量去重计数

这类场景你如果用 `Set` 去硬存，虽然精确，但内存会涨得很快。  
HyperLogLog 的思路是：

- 不追求绝对精确
- 用固定很小的内存，换一个误差可接受的近似计数

### 它最大的优点

Redis 官方文档和社区资料里都反复强调一点：  
它占用空间非常稳定，适合超大规模基数统计。

### 它的边界

它不是精确计数。  
如果你的业务要求：

- 财务对账
- 精确人数
- 不能接受近似误差

那就别用它。

所以 HyperLogLog 的本质不是“更强的集合”，而是：  
**用概率算法换空间效率。**

## GEO：适合“附近的人 / 附近的店 / 距离计算”

### 它本质上做了什么

GEO 让你直接把经纬度信息放进 Redis，并做：

- 两点距离计算
- 某点附近范围检索
- 指定成员附近检索

底层上，它并不是独立存储引擎，而是建立在有序集合能力之上。

### 它为什么好用

因为很多业务一开始并不需要完整 GIS 系统，只需要：

- 简单附近搜索
- 距离排序
- 附近若干候选点

这时 GEO 很省事。

### 它的边界

如果你的空间查询已经复杂到需要：

- 多边形范围
- 高级地图检索
- 复杂地理分析

那 Redis GEO 就不该硬扛了。

## Stream：适合“轻量级消息流”

你已经有单独的消息队列文章了，这里只抓它作为特殊类型最值得记的一点：

**Stream 不是简单 List 升级版，而是 Redis 在消息流场景下给出的专门数据结构。**

它比 List 更适合做轻量消息流，原因在于：

- 有消息 ID
- 有消费者组
- 有 ACK
- 支持回溯和挂起消息管理

但它仍然有边界：

- 堆积能力受内存限制
- 可靠性仍受 Redis 持久化模型约束

所以它是“轻量 MQ 数据结构”，不是“通吃所有消息场景的专业 MQ”。

## 怎么选：别背定义，先看业务模型

一个更实用的判断方式是：

### 如果你记录的是状态位

优先想 `Bitmap`。

### 如果你统计的是海量去重数量

优先想 `HyperLogLog`。

### 如果你查的是距离和附近范围

优先想 `GEO`。

### 如果你做的是事件流或轻量消息协作

优先想 `Stream`。

这比背“这是特殊数据类型”更接近真实工程判断。

## 容易踩的坑

### “Bitmap 是 Redis 独立类型”

不对。它本质上是基于 `String` 的按位操作能力。

### “HyperLogLog 能做精确去重统计”

不对。它是近似计数，不适合要求绝对精确的场景。

### “GEO 能替代专业地理位置系统”

不成立。它适合轻量附近检索，不适合复杂 GIS 场景。

### “Stream 既然有消费者组，就等于 Kafka”

不对。它只是 Redis 世界里的轻量消息流结构。

## 小结

- 特殊数据类型不是冷知识清单，而是典型业务模型的结构化答案。
- Bitmap 适合海量 0/1 状态，HyperLogLog 适合近似去重计数，GEO 适合附近检索，Stream 适合轻量消息流。
- 这些结构的最大价值都在“高性价比匹配特定场景”，而不是通用替代基础类型。
- 选型时先看业务问题，再决定结构，不要反过来先背命令再找场景。
- 真正答好这题的关键不是把名字背全，而是能说出每种结构的收益和边界。
