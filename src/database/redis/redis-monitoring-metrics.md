# Redis 线上监控到底该盯哪些指标？

> Redis 监控题不是让你背一堆 `INFO` 字段，而是让你能把“这个实例现在到底是慢、满、堵、抖，还是复制有问题”迅速判断出来。

很多人说 Redis 监控时，只会回一句：

- 看 CPU
- 看内存

这当然没错，但太粗了。  
真正实用的监控体系，至少要分成几层来看。

## 先抓主线：监控面板至少有四层

| 层次         | 重点问题                                 |
| ------------ | ---------------------------------------- |
| 业务层       | 缓存到底有没有帮我挡流量                 |
| Redis 实例层 | 它是不是在变慢、变堵、变满               |
| 复制持久化层 | 主从和后台任务是不是在抖                 |
| 系统层       | 机器是不是 CPU、内存、带宽或 swap 出问题 |

这四层一起看，才能定位“Redis 慢”到底慢在哪。

## 第一层：先看业务效果

### 命中率

缓存系统最先要看的，不是 Redis 自己有多忙，而是：

**它有没有真正挡住数据库流量。**

命中率低时，要先问：

- 是 key 设计有问题
- TTL 太短
- 缓存没预热
- 还是穿透、击穿、雪崩在发生

所以命中率是业务价值指标，不只是技术指标。

### 请求量和失败率

还要一起看：

- QPS
- 错误率
- 超时率

因为有些时候 Redis 自己并不满，但客户端超时已经先上来了。

## 第二层：看 Redis 实例自身状态

### 延迟 / RT

这是最直接的体感指标。  
如果 Redis 延迟上升，通常说明：

- 主线程被拖住
- 网络有问题
- 系统资源竞争变重

### 慢日志

`SLOWLOG` 不是常规业务指标，但它是排查异常时很关键的证据。

你至少要知道：

- 最近有没有重命令
- 是哪些命令慢
- 是偶发还是持续

### blocked clients

如果被阻塞的客户端数量持续升高，就说明：

- 主线程可能卡住了
- 某些命令或后台动作已经开始拖队列

这类指标非常适合和延迟一起看。

## 第三层：看内存

### `used_memory`

看 Redis 真正分配出去多少内存。

### `used_memory_rss`

看进程从操作系统层面实际吃掉多少物理内存。

### `mem_fragmentation_ratio`

看碎片问题是不是在恶化。

### 满内存接近度

也就是离 `maxmemory` 还有多远。  
这非常重要，因为一旦逼近上限：

- 淘汰开始变激进
- 写入可能报错
- 延迟和抖动风险会上升

## 第四层：看复制和高可用状态

### 主从延迟

如果主从延迟持续变大，往往意味着：

- 从库跟不上
- 网络链路有问题
- 主库写压力太大
- backlog 太小导致恢复成本上升

### 全量同步频率

如果你频繁看到全量同步，那就不是“小抖动”了，而是明显的稳定性问题。

因为全量同步意味着：

- fork
- RDB 生成
- 大量网络传输

它会明显放大实例压力。

## 第五层：看持久化和后台任务

### fork 延迟

这和：

- `bgsave`
- AOF rewrite
- 内存碎片
- 大实例

强相关。  
如果 fork 时间明显拉长，后台任务就已经开始威胁前台延迟了。

### AOF 和重写状态

要关注：

- 当前是否在 rewrite
- 最近刷盘是否抖动
- 磁盘 I/O 是否吃紧

因为很多 Redis “突然慢一下”的问题，根因不在命令，而在后台持久化动作。

## 第六层：看系统资源

Redis 是内存型服务，但它并不是只看内存就够了。

至少还要盯：

- CPU
- 网络带宽
- 磁盘 I/O
- swap

### CPU

CPU 高不一定是 Redis 本身命令太重，也可能是：

- 热 key
- 慢命令
- 混部竞争

### 网络

big key、热点流量、跨机房访问，都会让网络成为瓶颈。

### swap

这是 Redis 非常怕的。  
一旦发生 swap，延迟通常会很难看。

## 用 `INFO` 怎么看

Redis 官方文档里说得很明确，`INFO` 会返回多类统计信息。  
工程上更重要的不是把字段背全，而是知道重点看哪些 section：

- `memory`
- `stats`
- `replication`
- `cpu`
- `persistence`
- `commandstats`

这几个区块已经能覆盖大部分一线排障判断。

## 一个更实用的监控思路

如果让我做 Redis 面板，我会至少放这几组：

1. 命中率、QPS、错误率、超时率。
2. Redis RT、slowlog 计数、blocked clients。
3. `used_memory`、RSS、碎片率、离 `maxmemory` 的距离。
4. 主从延迟、全量同步次数、复制状态。
5. fork 时长、AOF rewrite 状态、磁盘 I/O。
6. CPU、网络带宽、swap。

这套指标一起看，基本能先判断 80% 的问题方向。

## 容易踩的坑

### “监控 Redis 就看内存和 CPU”

太粗。命中率、复制延迟、slowlog、fork 状态都可能更早暴露问题。

### “命中率高就一定没问题”

不对。命中率高也可能存在：

- 热 key 把单点打歪
- big key 把网络拖满
- 后台持久化抖动

### “只看 Redis 自己的监控”

不够。系统层和应用层指标不一起看，很容易误判。

## 小结

- Redis 监控至少要覆盖业务层、实例层、复制持久化层和系统层。
- 命中率、延迟、慢日志、blocked clients、内存占用、碎片率、主从延迟是最核心的一批指标。
- `INFO` 很重要，但重点不是背字段，而是知道从哪些 section 快速判断问题方向。
- 只看 CPU 和内存不够，网络、磁盘和 swap 经常才是隐藏根因。
- 好的监控面板不是指标越多越好，而是能帮你快速判断“是流量问题、命令问题、复制问题还是机器问题”。
