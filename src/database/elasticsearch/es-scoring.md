# 相关性打分是怎么回事？TF-IDF 和 BM25 有什么区别？

> 一句话点题：全文检索不像 SQL 那样按主键或时间排序，它要回答"哪篇文档和查询最相关"，这就是打分干的事。ES 现在用 BM25，而它取代的 TF-IDF 有个老毛病——词频线性增长，长文档天然占便宜。

上一篇讲到 Query context 会算相关性打分，但没说分怎么算。这篇就把打分讲透：TF-IDF 为什么不行、BM25 怎么改进、ES 从哪个版本默认换成了 BM25，以及什么时候根本不该打分。

## 为什么需要打分

`SELECT * FROM articles WHERE title LIKE '%mysql%'` 返回的结果，顺序由谁定？在 MySQL 里通常是主键或存储顺序，和"相关不相关"无关。但全文检索不一样——搜"mysql"，可能命中几千篇文档，用户期望**最相关的排前面**。这就需要一个函数给每篇命中文档算个分，按分排序。

这个分要反映"这篇文档和查询有多相关"。怎么衡量相关？直觉上有两条：

- 查询词在这篇文档里出现得越多，越相关（词频）；
- 查询词越"稀有"（在全体文档里出现得少），命中它越有价值（稀有度）。

TF-IDF 就是把这两条量化成公式，BM25 则是它的改进版。

## TF-IDF：两个因子相乘

TF-IDF 由两个因子相乘：

**TF（Term Frequency，词频）**：查询词在某文档里出现的次数。出现 5 次比出现 1 次更相关。朴素的 TF 就是出现次数本身（或取对数等变形）。

**IDF（Inverse Document Frequency，逆文档频率）**：衡量一个词有多稀有。公式大致是 `log(总文档数 / 包含该词的文档数)`。像"的""是""the"这种几乎所有文档都有的词，IDF 接近 0，命中它几乎不加分；而"BM25"这种只在少数文档出现的词，IDF 高，命中它价值大。

一篇文档对某个查询词的得分 ≈ `TF × IDF`。多个查询词的得分相加，就是文档对这个查询的总分。

这个思路对，但 TF-IDF 有个老毛病。

## TF-IDF 的问题：词频线性增长

问题出在 TF 上。朴素的 TF 里，**词频和得分成正比线性增长**——出现 10 次的得分是出现 1 次的 10 倍。这违反直觉：一篇文档提到"mysql"10 次，确实比提到 1 次更相关；但提到 100 次的，真比提到 10 次的相关 10 倍吗？显然不是，相关性增长到一定程度就该饱和，再多是冗余。

更严重的副作用是**长文档天然占便宜**：长文档字数多，同一个词出现的绝对次数自然多，TF 就高。结果是一篇又长又水、反复堆砌关键词的文档，得分压过一篇短小精悍、精准命中的文档。这正是 TF-IDF 被诟病的地方。

## BM25：给词频加饱和、给长度归一化

BM25（Okapi BM25）针对这两个毛病做了改进，核心是两刀：

**第一刀：词频饱和（参数 k1）。** BM25 的 TF 项不是线性增长，而是一个**饱和函数**——词频小时增长快，词频大了增长变缓，最终趋近一个上限。直观地说，"出现 1 次→2 次"加分明显，"出现 50 次→100 次"几乎不加分。`k1` 控制饱和速度，典型取 1.2；k1 越大饱和越慢（更接近线性），k1=0 则完全不看词频。

**第二刀：文档长度归一化（参数 b）。** BM25 引入文档长度因子，**惩罚比平均更长的文档**：同样出现 5 次"mysql"，在一篇短文档里比在一篇长文档里更值钱。`b` 控制归一化强度，取值 0~1，典型取 0.75；b=0 完全不考虑长度，b=1 全力惩罚长文档。

把这两刀合起来，BM25 的得分大致是：

> `score = IDF × (TF 饱和项) × (长度归一化项)`

相比 TF-IDF 的 `TF × IDF`，BM25 的 TF 不再线性，且多了长度惩罚。结果是：堆砌关键词的长文档不再占便宜，短小精准命中的文档得分更合理。这也是为什么 BM25 在信息检索领域长期是事实标准。

## 一个版本纠偏：默认打分什么时候换的

这是个容易被追问、也容易答错的版本点：**ES 从 5.0 开始，默认打分算法从 TF-IDF 切换成了 BM25。**

- 5.0 之前：默认用 Lucene 的 TF-IDF（实用评分变体）。
- 5.0 起：默认改为 BM25。

所以如果面试官问"ES 默认用什么打分"，答案是 **BM25**；但要知道早期是 TF-IDF，5.0 是分水岭。停留在"ES 用 TF-IDF"的回答，至少落后了 8 年。背后的实现是 Lucene 的 `BM25Similarity`，可以通过 `similarity` 参数给某个字段单独配别的算法。

## 用 explain 看打分明细

打分对不对、为什么这篇排前面，别靠猜，用 `explain` 直接看：

```json
GET /articles/_search
{
  "explain": true,
  "query": { "match": { "title": "mysql" } }
}
```

返回里会展开每个命中文档的打分细节：用了哪个 similarity、TF 是多少、IDF 是多少、长度归一化怎么算的、最终 `_score` 怎么来的。排查"为什么这篇相关性低"时，explain 是第一工具。

## 自定义打分：function_score 和 boosting

默认 BM25 不够用时，ES 提供了自定义打分能力：

- **`function_score`**：在 BM25 基础上叠加一个函数，按业务规则调整分数。比如"近 7 天发布的文章加权""浏览量高的加权"，可以用 `weight`、`field_value_factor`、`decay`（衰减函数）等。
- **`boosting`**：简单提升或降低某些条件的权重。`match` 里可加 `boost`；`bool` 的 `should` 子句天然就是加分。

典型场景：搜"mysql"时，希望近期文章排前，就在 `function_score` 里叠加一个时间衰减函数。打分从此不只是文本相关性，还能融合业务权重。

## 什么时候不该打分

最后回到一个常被忽略的点：**很多查询根本不需要打分，硬算分是浪费。** 上一节说过，纯过滤（状态、范围、存在性）放 Filter context 就不算分。判断标准很简单：

- 结果需要按"和查询多相关"排序 → 打分（Query context）；
- 结果只要"满足条件"，按业务字段（时间、销量）排序 → 不打分（Filter context + 业务字段排序）。

比如"查最近 7 天状态为已发布的文章，按发布时间倒序"，这里没有文本相关性，全程不该打分：过滤条件放 filter，排序用 `publish_at` 字段。无谓的打分既拖慢查询又吃 CPU，能避就避。

## 容易踩的坑

- **以为 ES 默认还是 TF-IDF**：5.0 起默认 BM25，答"TF-IDF"至少落后 8 年。
- **不理解 BM25 为什么比 TF-IDF 好**：核心是词频饱和（k1）和长度归一化（b），解决长文档堆砌占便宜的问题。
- **给纯过滤查询算分**：没有文本相关性的场景用 filter + 业务字段排序，别让 BM25 白算。
- **打分异常不查 explain**：`"explain": true` 能看到 TF/IDF/长度各项明细，是排查打分问题的第一工具。
- **把 k1、b 当玄学**：k1 典型 1.2、b 典型 0.75 是经验默认值，理解"k1 管词频饱和、b 管长度惩罚"比记数字重要。

## 小结

- 全文检索要按相关性排序，打分函数回答"哪篇最相关"；直觉是词频（出现多更相关）×稀有度（稀有词命中更值钱）。
- TF-IDF 的毛病是词频线性增长，长文档天然偏高、堆砌关键词占便宜。
- BM25 两刀改进：**词频饱和（k1，典型 1.2）让词频增长到顶后趋平；长度归一化（b，典型 0.75）惩罚长文档**，是信息检索事实标准。
- **ES 自 5.0 起默认 BM25**（此前 TF-IDF），可通过 `similarity` 参数改。
- 用 `explain` 看打分明细；纯过滤查询别打分，用 filter + 业务字段排序。
