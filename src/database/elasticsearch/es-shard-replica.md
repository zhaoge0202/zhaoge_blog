# 分片和副本是怎么回事？分片数怎么定？

> 一句话点题：ES 之所以能扛海量数据，靠的是把一个 index 切成多个分片散到多台机器。但分片数一旦定下就不能改——这不是设计的任性，而是路由公式的硬约束。所以"分片数怎么定"是个建索引时就必须想清楚的决策。

这篇讲 ES 分布式的地基：分片和副本。理解了它，下一篇的读写流程才有落脚点。

## 为什么需要分片

单台机器存不下、也扛不住并发，这是分布式的老问题。ES 的解法是把一个 index 的数据切成多份，每份叫一个 **shard（分片）**，分散到不同节点上。这样数据量和读写压力都能横向扩展。

关键认知：**一个 shard 本质上是一个独立的 Lucene 索引实例**。也就是说，ES 的一个 index，并不是一个"大索引"，而是"一群 Lucene 索引的逻辑组合"。每个 shard 各自有自己的倒排索引、segment、刷新机制，能独立完成搜索。ES 只是在外面把这群 shard 当成一个逻辑 index 暴露给你。

分片分两种：

| 类型                          | 作用                               |
| ----------------------------- | ---------------------------------- |
| **primary shard**（主分片）   | 原始数据，写入先落到主分片         |
| **replica shard**（副本分片） | 主分片的拷贝，用于容灾和分担读请求 |

一个 index 有 N 个主分片，每个主分片可以有 R 个副本。比如 3 主 1 副的配置：3 份原始数据 + 3 份拷贝，共 6 个 shard 实例散在集群里。

## 路由公式：为什么主分片数不能改

ES 靠一个固定公式决定一篇文档落在哪个主分片：

> `shard = hash(routing) % number_of_primary_shards`

`routing` 默认是文档的 `_id`（也可以自定义路由值）。这个公式有个关键性质：**主分片数 `number_of_primary_shards` 必须固定**。

为什么必须固定？因为取模的结果取决于除数。假设有 3 个主分片，某文档 `hash(id) % 3 = 1`，存在分片 1。如果你把主分片数改成 5，同样的文档 `hash(id) % 5` 可能变成 2，ES 就去分片 2 找，找不到——数据"丢"了（其实没丢，只是路由错位）。

所以**主分片数在建 index 时定下后就不能改**，改了所有文档的路由全乱。这也意味着选主分片数是个"一锤子买卖"，得在建索引前想好数据规模。

副本就不同了：副本数 `number_of_replicas` **可以动态调整**，因为它不参与文档路由，只是主分片的拷贝。读请求可以走副本，所以加副本能提升读吞吐；副本也提供容灾，主分片挂了副本能顶上（甚至提升为主）。

## 副本的两个作用

副本不是备胎，它干两件实事：

1. **高可用/容灾**：主分片所在节点挂了，副本可以提升成新的主分片，数据不丢、服务不停。前提是副本和主分片不在同一节点（ES 会自动保证这一点）。
2. **分担读请求**：读请求（搜索、聚合）可以在主分片或副本上任意执行，ES 会做负载均衡。副本越多，读吞吐越高。

但副本不是免费的：每个副本都是一份数据拷贝，**占存储、也增加写入成本**（写主分片后要同步给所有副本）。所以副本数要在"读吞吐/可用性"和"存储/写入开销"之间权衡。读多写少的场景副本可以多点，写入极重的场景副本不宜过多。

## 分片数怎么定

这是面试和工程实践都绕不开的问题。先给经验值，再讲怎么推。

**经验法则：单个分片建议控制在 30~50GB。** 这个数字不是硬规定，而是综合了几个约束的甜点：

- 分片太大：恢复/搬迁慢（节点故障时要把整个分片拷贝走），merge 也慢。
- 分片太小：segment 碎、节点开销大（每个 shard 都是独立 Lucene 实例，有固定内存和文件句柄开销）。

**推算主分片数**：用预估数据总量除以单分片目标大小。比如预计 200GB 数据，按单分片 40GB 算，大约需要 5~6 个主分片。再结合节点数（分片要能均匀分布）、未来增长留余量。

**副本数**：一般从 1 开始。要求高可用至少 1 副本（单副本意味着主挂了没备份）；读压力大的场景加到 2~3。日志类写入极重、又能容忍丢失的场景，甚至可以临时设 0 副本换写入速度。

## over-sharding：分片太多的害处

和"分片太少"相比，**分片太多（over-sharding）是更常见的坑**。很多人图省事一上来就开几十个分片，以为"将来好扩展"，结果踩了一堆坑：

- **每个 shard 都是独立 Lucene 实例**，有固定开销：内存、文件句柄、后台 merge 线程。分片太多，光是维持它们就吃掉大量资源。
- **查询要扇出**：一次搜索要发给所有相关分片，分片越多，协调节点的扇出和归并压力越大，反而拖慢查询。
- **集群状态膨胀**：master 要维护所有分片的元数据，分片数过多会让集群状态变大、传播变慢，影响稳定性。官方建议每 GB 堆内存对应的分片数控制在 20 以内（如 32GB 堆约 600 分片上限）。

所以**别为了"未来扩展"提前开太多分片**。真要扩展，可以用 split API（前提是原分片数是偶数倍关系）或 reindex 到新索引，而不是一开始就 over-shard。日志类按时间切索引（每天一个 index）的场景，尤其要控制每个 index 的分片数，否则索引一多分片爆炸。

## 容易踩的坑

- **以为主分片数能随时改**：路由公式决定它建索引时定死，改了路由全乱，只能 reindex 或 split。
- **一上来就开几十个分片**：over-sharding 让每个分片都碎小、开销叠加、查询扇出大，反而更慢更不稳。
- **副本设 0 图写入快但不顾可用性**：0 副本时主分片挂了数据就没了，只能临时用，不能当常态。
- **单分片撑到几百 GB**：恢复和 merge 都慢，故障时搬迁代价大，建议控制在 30~50GB。
- **主副本放同节点**：正常情况 ES 会自动避免，但自定义分配规则失误时可能让主副本同节点，失去容灾意义。

## 小结

- 一个 index 切成多个 shard 散到多台机器；**一个 shard 本质是一个独立 Lucene 索引实例**，index 是这群 shard 的逻辑组合。
- 主分片数靠 `shard = hash(routing) % 主分片数` 路由，**建索引时定死不可改**；副本数可动态调。
- 副本两作用：容灾（主挂副本顶上）和分担读（读可走副本），代价是占存储和增加写入同步成本。
- 分片数经验：**单分片 30~50GB**，按预估数据量反推主分片数；副本从 1 起，读压力大再加。
- **over-sharding 比分片太少更常见**：分片过多带来固定开销、查询扇出、集群状态膨胀，别为"未来扩展"提前开太多。
