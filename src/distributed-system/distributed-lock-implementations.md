# 分布式锁有哪些实现？Redis 锁的边界在哪里？

> 分布式锁最容易答歪的地方，不是“会不会背 `SET NX PX`”，而是你要先搞清：这把锁到底是为了**减少重复执行**，还是为了**绝对保护共享资源正确性**。这两个目标决定了 Redis、ZooKeeper、etcd 哪个方案能用，哪个方案只是看起来能用。

先看一个常见场景：

- 订单服务要避免同一订单被重复处理
- 多个实例会同时抢这份工作
- 有的场景只是“偶发重复执行也能接受”
- 有的场景一旦并发执行，就会造成库存、资金、状态机损坏

这两类场景不能用同一把锁心态去设计。

如果这篇只记一句话，我希望是：

**先分“效率型加锁”和“正确性型加锁”，再谈用什么实现。**

## 先抓总线：分布式锁要解决什么问题？

本地锁，比如 `synchronized` 或 `ReentrantLock`，解决的是：

- 同一个进程里多个线程互斥

一旦到了分布式系统，问题就变成：

- 多个进程
- 多台机器
- 甚至多个机房

它们都可能同时处理同一份共享资源。

这时你想要的通常是：

1. 同一时刻尽量只有一个执行者进入临界区
2. 持锁者挂了以后，锁不能永远不释放
3. 网络抖动、GC 停顿、主从切换时，不要轻易把旧持锁者误当成还有效

第三点恰恰是大多数“Redis 锁八股文”最容易略过去的地方。

## 先把场景分成两类

这是选型前最关键的一步。

### 1. 效率型加锁

这类锁的目标是：

**减少重复工作，提高吞吐或降低资源浪费。**

比如：

- 定时任务尽量只跑一份
- 缓存重建尽量别多个实例一起做
- 某个批处理任务最好别重复消费

这类场景里，即使极端情况下偶发并发执行一次，后果通常也还能接受。

### 2. 正确性型加锁

这类锁的目标是：

**一旦并发执行就会造成不可接受的数据错误。**

比如：

- 扣库存
- 扣余额
- 状态机只能按唯一顺序推进
- 外部设备控制不能同时下发相反命令

这时“偶尔失效一下也没事”的心态就完全不成立。

所以更稳的说法是：

**很多 Redis 锁方案在效率型场景很好用，但一旦你要保护真正的业务正确性，就必须把边界讲得更严。**

## 实现分布式锁，常见有哪几类？

从工程上看，常见方案基本就这几类：

| 方案         | 典型代表                       | 优势                  | 主要边界                     |
| ------------ | ------------------------------ | --------------------- | ---------------------------- |
| 数据库锁     | 行锁、悲观锁                   | 简单、现成            | 吞吐低、耦合业务库           |
| Redis 锁     | `SET NX PX`、Redisson、Redlock | 快、易接入            | 租约过期、主从切换、暂停窗口 |
| ZooKeeper 锁 | 临时顺序节点、Curator          | 一致性更强、顺序感强  | 性能更低、运维更重           |
| etcd 锁      | Lease + Txn/Mutex              | 强一致、天然 revision | 成本更高、通常用于云原生体系 |

所以真正的问题从来不是“哪一种最万能”，而是：

**你想换性能、换可用性、换正确性，还是换实现复杂度。**

## 基于 Redis 的最小实现长什么样？

最常见的思路是：

```text
SET key value NX PX 30000
```

意思是：

- `NX`：只有不存在时才设置
- `PX`：带过期时间

如果设置成功，说明你拿到了锁。

释放锁时不能直接 `DEL key`，而是通常配合唯一 value + Lua：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
 return redis.call("del", KEYS[1])
else
 return 0
end
```

因为否则会出现：

1. A 拿锁
2. A 执行太久，锁过期
3. B 拿到新锁
4. A 迟到执行 `DEL`
5. 把 B 的锁误删

所以：

**唯一 value + Lua 原子校验删除**
是 Redis 锁的最小正确姿势，不是可选增强。

## 为什么很多团队更愿意直接用 Redisson？

因为自己手写 Redis 锁，光做到“最小可用”还不够。

资料里讲得比较清楚，Redisson 的核心价值之一就是：

**Watch Dog 自动续期。**

也就是说：

- 线程还在执行
- 锁快过期了
- Redisson 会自动延长过期时间

这样能缓解一个高频问题：

> 业务执行时间比锁租约更长，结果锁提前过期，别的线程又进来了。

所以工程上更稳的说法是：

**如果你只是要用 Redis 锁，优先考虑成熟客户端实现，而不是从 `SET NX PX` 生写起。**

## 但为什么说 Redis 锁仍然有边界？

这是这篇最重要的部分。

### 边界一：租约不是所有权证明，只是时间窗口

Redis 锁的核心语义其实是：

**在一段租约时间内，别人最好别再拿到同一把锁。**

这和“系统绝不会出现第二个执行者”不是一个强度。

如果出现：

- 长 GC 停顿
- 网络卡顿
- 线程挂起
- 调度抖动

持锁者可能还以为自己持锁，但从 Redis 看，这把锁早就过期并被别人拿走了。

这就是所谓 lease-based locking 的天然边界。

### 边界二：Redis 主从异步复制会破坏单实例锁的互斥想象

Redis 官方分布式锁文档明确指出：

- 单主 + 异步复制场景下
- 如果主节点刚写入锁就宕机
- 从节点还没同步到这把锁
- 故障转移后，新主可能允许别人再次加锁

官方文档直接把这个当成单实例主从场景的安全问题来讲。
所以更稳的回答一定要明确：

**Redis 主从异步复制下，单实例写锁并不天然等于主从切换后也不会重复加锁。**

这也是为什么 Redis 官方后来提出了 Redlock。

## Redlock 在解决什么问题？

Redis 官方文档里给出的思路是：

- 不依赖一个 Redis 主实例
- 而是在多个互相独立的 master 上尝试加锁
- 只有在严格多数派实例上都成功，并且耗时还在租约窗口内，才算加锁成功

也就是说，Redlock 想解决的是：

**单个 Redis 主实例在异步复制 + 故障切换场景下太脆弱的问题。**

它本质上是把“单点锁”改成了“多数派租约”。

## 为什么 Redlock 依然有争议？

因为它并没有把锁从“租约模型”变成“绝对正确性证明”。

资料里已经把争议点讲得很清楚了，核心在于：

- 它依赖有界网络延迟
- 依赖有界时钟漂移
- 依赖进程暂停不会长到把租约语义彻底打穿

所以很多工程师的共识并不是：

> Redlock 完全不能用

而是：

> Redlock 更适合“我比单 Redis 主从更在意互斥”，但仍然不应该被误当成金融级绝对正确性锁。

## 为什么说“Redis 锁的边界”本质上是正确性边界？

因为 Redis 锁最容易被误用在这类场景：

- “我已经加了锁，所以绝不会重复扣库存”
- “我已经加了锁，所以旧客户端不可能再写进来”

但 lease-based lock 真正不能替你兜底的是：

**旧持锁者在自己看来还活着、还继续写，而资源端又无法分辨它是不是陈旧持锁者。**

这时问题就不在“锁还在不在”，而在：

**资源端有没有能力拒绝陈旧客户端的写入。**

这就是 fencing token 要解决的问题。

## Fencing Token 为什么这么关键？

你可以把它理解成：

**每次成功拿锁，都顺便拿到一个单调递增的令牌。**

然后资源端在写入时做校验：

- 只有 token 更大的请求才能写
- token 更旧的请求即使“自以为持锁”，也被拒绝

比如：

```text
客户端 A 拿到 token=10
客户端 B 拿到 token=11
```

如果 A 因为 GC 暂停迟到恢复后又来写，资源端看到它只有 `10`，而当前最新已经是 `11`，就直接拒绝。

这才是真正把“旧持锁者误操作资源”的问题卡在资源端。

所以可以把一句话立住：

**锁只能控制“谁先拿到进入资格”，fencing token 才更接近控制“谁最终能写成资源”。**

## ZooKeeper 锁为什么经常被说“更可靠”？

因为它的核心不是依赖租约过期，而是依赖：

- 临时节点
- 顺序节点
- watcher / 事件通知

资料里讲得比较直白：

**ZooKeeper 锁更适合强调正确性和顺序感的场景。**

最常见做法是：

1. 每个客户端创建一个临时顺序节点
2. 谁的序号最小，谁拿锁
3. 其他客户端监听前一个节点
4. 前一个节点消失后再继续竞争

它的好处在于：

- 锁释放和会话生命周期绑定
- 公平性更容易做
- 不靠“过期时间猜测持锁者是否还活着”

当然代价也很明确：

- 性能通常低于 Redis
- 对协调服务本身要求更高
- 运维和使用成本更重

## etcd 锁为什么在云原生体系里越来越常见？

因为 etcd 本身就是：

- 强一致 KV
- 基于 Raft
- 自带 Lease、Txn、revision 这些很适合做协调的原语

资料里提到的一个关键点是：

**etcd 天然更适合和 fencing token 一类资源端兜底机制一起用。**

尤其 `revision` 这种单调递增版本号，本身就很适合当 fencing 语义的基础。

所以在 Kubernetes / 云原生体系里，很多协调能力天然更偏 etcd 风格，而不是专门再引一个 ZooKeeper。

## 一个更实用的选型结论

如果把选型压缩成更工程的话，大概可以这样分：

| 场景                               | 更常见选择                 |
| ---------------------------------- | -------------------------- |
| 只是减少重复执行、偏效率           | Redis / Redisson           |
| 强调资源正确性、可接受更重协调成本 | ZooKeeper / etcd + fencing |
| 已经深度在云原生体系               | etcd 更顺手                |
| 已有稳定 ZooKeeper 体系            | Curator 锁会更自然         |

这比“Redis 快、ZooKeeper 准”这种过度简化更有用。

## 一个更稳的排障顺序

如果线上怀疑锁出了问题，我会先按这条线排：

```text
1. 这是效率型场景还是正确性型场景？
2. 当前锁是基于单 Redis、Redisson、Redlock，还是 ZooKeeper/etcd？
3. 失效点是加锁成功重复了，还是释放错了，还是旧客户端迟到继续写了？
4. 有没有 GC 停顿、网络分区、主从切换、时钟漂移？
5. 资源端有没有 fencing token 或版本校验兜底？
```

很多“分布式锁失效”其实不是锁没加住，而是：

- 锁过期了
- 旧客户端误操作资源
- 业务把效率型锁当成了正确性型锁

## 容易踩的坑

### 把 `SET NX PX` 当成最终答案

不对。
这只是 Redis 锁的最小实现起点，不是整个问题的终点。

### 觉得 Redisson 就等于绝对正确

也不对。
Watch Dog 解决的是租约提前过期问题，不是所有正确性问题。

### 觉得加了锁就不用资源端兜底

这恰恰是最危险的误区。
对正确性型场景，资源端最好还要有 fencing token 或版本校验。

## 小结

- 分布式锁先要区分“效率型加锁”和“正确性型加锁”，这一步比选 Redis 还是 ZooKeeper 更重要。
- Redis 锁常用、性能好，但本质上是租约模型，边界在于过期、暂停、异步复制和主从切换。
- Redlock 试图缓解单主 Redis 锁的安全问题，但它依然不是无限强的正确性证明。
- ZooKeeper 和 etcd 更适合正确性要求高的协调场景，尤其配合 fencing token 时更稳。
- 真正防住陈旧客户端误写资源的关键，往往不只是“有没有锁”，而是“资源端有没有基于版本/令牌做最终兜底”。
