# CAP 和 BASE 怎么理解？为什么不是简单三选二？

> CAP 真正想讨论的，不是“系统平时想要哪两个”，而是**一旦发生网络分区**，你到底优先保线性一致性，还是优先保证每个请求都尽快得到响应。

先看一个很典型的误解：

- “分布式系统要在一致性、可用性、分区容错性里三选二。”
- “互联网系统一般选 AP，数据库一般选 CP。”

这两句话都不算全错，但都太粗了。
真正往下问时，面试官想听的通常是：

1. CAP 里的 `C` 到底是什么一致性
2. 为什么 `P` 不是你想不要就不要
3. “三选二”为什么会误导人
4. BASE 跟 CAP 是对立关系，还是工程补充

## 先抓一句话：CAP 讨论的是“分区发生时”的取舍

CAP 最容易被答歪的地方，就是把它说成“系统日常长期属性三选二”。

更准确地说，CAP 关注的是：

**当分布式系统发生网络分区时，你还能不能同时对外保证强一致和高可用。**

所以这题一开口，最好先把“前提”说出来：

- 不是单机场景
- 不是网络永远稳定
- 重点是网络分区发生时

如果没有分区，很多系统平时当然可以既一致又可用。
真正困难的是：

**一旦节点之间消息丢了、延迟很高、链路断了，系统还怎么选。**

## CAP 里的 3 个字母，到底分别是什么意思？

### C：Consistency 不是泛泛的“数据看起来一致”

在 CAP 语境里，`C` 更接近：

**线性一致性（linearizability）**

翻成人话是：

- 一次写成功后
- 后续任何读都应该立刻读到最新值
- 所有节点对外表现得像只有一个最新真相

所以这里的 `C` 不是：

- “最终会一致”
- “业务上大差不差”
- “几秒后能同步过来”

这些都不算 CAP 里的那个 `C`。

这是第一层高频误区：
很多人把 CAP 的 `C` 和数据库事务里的“一致性”、或者业务语义上的“最终没问题”混成一回事。

### A：Availability 不是“服务不宕机”

CAP 里的 `A` 也不要答得太口语化。

更准确地说，它强调的是：

**每个到达非故障节点的请求，都能在有限时间内得到一个非错误响应。**

也就是说：

- 不是只要进程还活着就算可用
- 不是返回“系统繁忙稍后再试”也算真正意义上的 A
- 不是“P99 很低、1 秒内返回”这种工程 SLA

如果你为了保证一致性，一旦检测到分区就对一部分请求直接拒绝，那从 CAP 语境看，这部分可用性其实已经让掉了。
请求快不快当然很重要，但那是延迟目标，不是 CAP 里 `A` 的定义本身。

### P：Partition Tolerance 不是“我愿不愿意选”

`P` 讨论的是：

**节点之间发生网络分区后，系统还能不能继续工作。**

网络分区可以理解成：

- 节点互相连不上
- 消息大量丢失
- 延迟高到跟断了差不多

在真实分布式系统里，网络不可靠不是假设，而是现实。
所以这就是为什么很多人说：

**P 往往不是主动选择，而是系统必须面对的客观事实。**

## 为什么说“CAP 不是简单三选二”？

这是这篇最核心的点。

### 误区一：好像平时就能随手选掉 P

现实里只要是跨机器分布式系统，网络分区迟早会发生。
你没法说：

> 我这个系统不要 P。

你最多只能说：

> 一旦分区发生，我宁愿牺牲一部分可用，也要守住一致性。

或者：

> 一旦分区发生，我宁愿先让请求成功返回，后面再慢慢收敛一致性。

所以真正的取舍不是：

```text
C / A / P 任意选两个
```

而更接近：

```text
发生分区时：C 和 A 之间怎么取舍
```

### 误区二：好像没有分区时就没有取舍

没有网络分区时，CAP 的冲突确实不再那么尖锐。
但真实系统仍然经常要在一致性和延迟之间取舍。

这就是 PACELC 更想补充的视角：

```text
如果发生分区 P：在 A 和 C 之间取舍
否则 Else：在 L（低延迟）和 C 之间取舍
```

比如主从复制：

- 写主库后立刻返回，延迟低，但从库可能短暂落后
- 等多个副本确认后再返回，一致性更强，但写延迟更高

所以 CAP 解决的是分区时的理论边界，PACELC 提醒你：

**哪怕网络没断，分布式复制也常常要在“更一致”和“更快返回”之间付成本。**

### 误区三：好像系统会永久贴一个“我是 CP/我是 AP”的标签

真实系统往往不是纯粹的、全局性的 CAP 单标签。

同一个系统里不同功能可能选择不一样：

- 账户余额扣减更偏 CP
- 点赞数、浏览数统计更偏 AP
- 商品详情页可能允许旧一点的数据
- 支付结果查询可能宁可慢一点也要准

所以更成熟的回答方式应该是：

**CAP 决策通常落在具体链路、具体数据、具体场景上，而不是整个系统一刀切。**

## CAP 更适合讨论什么粒度？

还有一层经常被忽略：CAP 不是拿来给整个业务系统随手贴标签的。

它更适合讨论：

- 某个数据对象
- 某个副本组
- 某类读写操作
- 某条复制链路

比如一个电商系统里：

- 商品库存副本怎么复制，是一个 CAP 问题
- 注册中心分区时还能不能返回实例，是一个 CAP 问题
- 整个“电商系统是 CP 还是 AP”，这个问法就太粗

真实业务系统往往是很多组件拼起来的：

```text
业务代码
  -> 注册中心
  -> 数据库副本
  -> 缓存主从
  -> 消息队列
  -> 分布式锁
```

每个组件的分区行为不同，组合到一起才形成最终的业务表现。
所以面试里更稳的说法是：

**不要给整站粗暴贴 CP/AP 标签，要拆到数据对象、操作语义和底层组件行为上分析。**

## 用一个例子把 CAP 讲透

假设有两个机房节点：

```text
节点 A <----网络----> 节点 B
```

现在某个商品库存是 `10`。

### 正常情况下

- 用户在 A 扣减库存
- B 很快同步到新值
- 两边都读到 `9`

这时候看起来 C 和 A 都没问题。

### 分区发生后

假设 A 和 B 之间断网了。

这时如果又来了两个请求：

- 用户 1 打到 A，要扣库存
- 用户 2 打到 B，也要扣库存

系统就必须选：

#### 方案一：优先 C，牺牲一部分 A

为了避免双写冲突，B 暂停服务或拒绝部分写请求，等和 A 恢复同步后再处理。

结果：

- 一致性更强
- 但可用性下降

这更偏 CP。

#### 方案二：优先 A，接受后续修正

A、B 都先接请求，各自先处理，等网络恢复后再做冲突解决。

结果：

- 可用性更高
- 但短时间内两边数据可能不一致

这更偏 AP。

这个例子最重要的一句是：

**不是你平时想选 C 还是 A，而是分区发生时不能同时把两者都绝对守住。**

## CAP 里的 `C` 为什么这么“贵”？

因为你要保证线性一致，通常就意味着：

- 写操作不能随便先答成功
- 读操作也不能随便从旧副本返回
- 节点之间要有更严格的协调

这往往会带来：

- 更高延迟
- 更复杂的复制协议
- 分区时更容易拒绝请求

所以很多系统不是“不想一致”，而是：

**线性一致带来的代价太高，不是每条链路都值得付。**

## BASE 是什么？为什么它经常和 CAP 一起出现？

BASE 通常被看成是对分布式场景里强事务约束的一种放宽，也经常和 AP 风格系统一起出现。

常见拆法是：

- **Basically Available**：基本可用
- **Soft State**：软状态
- **Eventually Consistent**：最终一致

它的核心意思不是“不要一致性”，而是：

**允许系统在一段时间内不强求线性一致，换取更高可用和更好的扩展性，然后通过异步手段把状态慢慢收敛回来。**

如果拿它和 ACID 放在一起看，会更清楚：

- ACID 更强调单个事务边界内的原子性、一致性、隔离性和持久性
- BASE 更强调分布式链路里先保证基本可用，再通过异步补偿走向最终正确

所以 BASE 不是“反一致性”，而是：

**在强一致成本太高时，把一致性的完成时间从同步链路挪到后续收敛链路里。**

## BASE 不是 CAP 的数学替代，而是工程策略

这点最好主动讲出来。

CAP 更像：

- 一个分布式理论边界
- 说明分区场景下你没法把强一致和高可用同时拉满

BASE 更像：

- 工程系统在这个边界下的一套落地哲学
- 通过“先允许暂时不一致，再靠补偿和收敛恢复”的方式活下来

所以它们不是严格对立，而更像：

```text
CAP 告诉你天花板在哪
BASE 告诉你工程上怎么在这个天花板下活得更现实
```

## 什么叫“基本可用”？

不是说系统必须完全正常，而是：

**在压力、故障、分区等极端情况下，系统可以降级，但不能整体彻底不可用。**

比如：

- 搜索结果少一点
- 推荐列表旧一点
- 评论区延迟展示
- 某些非核心功能先降级关闭

这些都属于“基本可用”的典型工程表现。

## 什么叫“软状态”？

意思是：

**状态在一段时间内允许不是最终稳定真相。**

也就是你现在看到的值，可能只是阶段性状态，还会继续变化和收敛。

比如：

- 某个计数副本暂时落后主副本
- 某个缓存里还是旧值
- 某个订单视图页还没同步到最新状态

这类状态在 BASE 语境里是被允许的。

## 什么叫“最终一致”？

最容易答成“早晚会一致”，但最好再具体一点：

**如果没有新的更新继续进来，系统里的多个副本经过一段时间后会收敛到同一个结果。**

重点有两个：

1. 不是立刻一致
2. 不是永远乱着

这也是为什么很多互联网系统会接受：

- 秒级延迟
- 读修复
- 反熵同步
- 异步补偿
- 消息重试
- 定时对账

因为它们本质上都在为最终一致服务。

这里还要区分两种“最终一致”的语境：

| 语境         | 关注点                       | 常见手段                   |
| ------------ | ---------------------------- | -------------------------- |
| 副本数据收敛 | 多个副本最后看到同一个值     | 读修复、反熵同步、重放日志 |
| 业务事务收敛 | 中间态最后进入成功或失败终态 | 重试、补偿、对账、状态机   |

比如订单创建后库存扣减消息积压了，页面短时间看到“处理中”不是结束态。
后面要么扣减成功进入“已确认”，要么补偿失败进入“已取消”。
这才叫有工程闭环的最终一致。

如果只是说“迟早会好”，却没有重试、补偿、对账和人工兜底，那不是 BASE，而是把风险留给线上。

## 一个更贴近业务的对比

你可以这样理解：

| 场景               | 更偏向       |
| ------------------ | ------------ |
| 银行转账余额扣减   | CP           |
| 社交点赞数、阅读数 | AP / BASE    |
| 订单创建主流程     | 偏 CP        |
| 推荐流、排行榜刷新 | 偏 AP / BASE |

所以千万别答成：

> 互联网系统都选 AP，金融系统都选 CP。

这还是太粗。
更准确的是：

**同一个业务系统内部，不同链路按数据价值和容错成本做不同 CAP / BASE 取舍。**

## 换成组件视角会更像真实项目

面试追问项目经验时，不要只停在理论名词。
可以把分区发生时的行为说到组件级别：

| 组件或场景   | 分区时更该关注什么                           |
| ------------ | -------------------------------------------- |
| 注册中心     | 是拒绝不确定注册，还是继续返回可能过期的实例 |
| 分布式锁     | 是牺牲获取成功率，还是冒锁失效和并发写风险   |
| 限流计数器   | 是强同步计数，还是接受短时间限流不准         |
| 缓存主从     | 是读旧值、丢部分更新，还是等待复制确认       |
| 消息消费位点 | 是保证位点不倒退，还是接受重复消费后幂等     |

这样回答会比“某某组件是 CP/AP”更可靠。
因为真正影响线上故障的，通常是这些组件在分区、切主、复制延迟时的具体行为。

## 一个更稳的排障视角

如果线上遇到“数据怎么短时间不一致”，不要第一反应就说系统错了。

先问这几个问题：

```text
1. 这条链路设计目标是强一致还是最终一致？
2. 当前是不是在分区、重试、消息积压或副本延迟期间？
3. 这是允许的软状态，还是超出了设计窗口？
4. 有没有补偿、对账、重试机制在收敛？
```

很多“看起来不一致”的现象，其实是系统设计里本来允许存在的短暂状态。

## 容易踩的坑

### 把 CAP 的 `C` 说成数据库事务一致性

不对。
这里更接近线性一致性，不是 ACID 里那个泛化的一致性概念。

### 把 CAP 背成永远三选二

这太粗了。
更准确的是：

**当分区发生时，在 C 和 A 之间取舍。**

### 把 BASE 说成“不一致也无所谓”

也不对。
BASE 强调的是：

**阶段性允许不一致，但最终要收敛回来。**

## 小结

- CAP 真正讨论的是网络分区发生时，系统如何在强一致和高可用之间做取舍。
- CAP 里的 `C` 更接近线性一致性，`A` 是每个非故障节点都要及时给响应，`P` 是系统必须面对的网络分区现实。
- “CAP 是简单三选二”这个说法会误导人，更准确的是“分区发生时在 C 和 A 之间选”。
- BASE 不是 CAP 的数学替代，而是工程上接受基本可用、软状态和最终一致的一套落地思路。
- 真正的系统设计通常不是整站贴一个 CP/AP 标签，而是按链路、按数据价值、按业务容错成本细分选择。
