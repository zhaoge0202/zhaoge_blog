# 分布式 ID 怎么选：雪花、号段、数据库自增？

> 分布式 ID 不是“找个算法生成不重复数字”这么简单，它要同时回答：会不会重复、数据库写入是否友好、发号服务会不会成为瓶颈、故障后能不能继续发号。

## 为什么单库自增到了分布式环境就不够了？

单体系统里，MySQL 自增主键通常很好用：简单、递增、落库天然唯一。但一旦订单表按用户或时间拆到多个库，问题就来了：

```text
订单库 1：id = 1001
订单库 2：id = 1001
订单库 3：id = 1001
```

每个库内部都不重复，但放到全局订单系统里就冲突了。分布式 ID 解决的就是**跨节点、跨库、跨服务生成全局唯一标识**的问题。

一个合格方案至少要满足这些要求：

| 目标     | 解释                       | 常见影响                        |
| -------- | -------------------------- | ------------------------------- |
| 全局唯一 | 任意节点生成的 ID 不能冲突 | 主键冲突、幂等失效              |
| 高性能   | 发号不能拖慢核心链路       | 下单、支付、消息写入延迟        |
| 高可用   | 发号组件故障不能拖垮业务   | 单点、依赖数据库或 Redis        |
| 趋势递增 | 尽量让新 ID 落在索引尾部   | InnoDB 聚簇索引页分裂和缓存命中 |
| 可治理   | 能监控、扩容、排查和兜底   | 号段耗尽、时钟回拨、重复发号    |

这里要先纠正一个常见误区：**大多数业务不需要 ID 连续，只需要唯一**。为了追求连续 ID 去回收号段、补齐空洞，反而更容易引入重复发号风险。

这里可以把几个词先拆清楚：

| 概念         | 真正含义                       | 工程判断                       |
| ------------ | ------------------------------ | ------------------------------ |
| 唯一         | 任意节点生成的 ID 不冲突       | 底线，必须保证                 |
| 趋势递增     | 大体按时间向前增长             | 对数据库写入友好，常见需求     |
| 全局严格递增 | 任意两个 ID 都能按生成先后比较 | 往往需要中心协调，成本更高     |
| 连续         | 中间没有空洞                   | 大多数业务不需要，也不建议强求 |

面试里如果说“分布式 ID 要连续递增”，很容易被追问。
更稳的说法是：

**关键业务先保唯一和可治理，数据库主键尽量趋势递增，连续性通常不是目标。**

## 先分清内部主键和外部单号

分布式 ID 选型前，最好先问一句：

**这个 ID 是给数据库当主键，还是给用户 / 外部系统看的业务单号？**

两者要求不完全一样：

| 类型     | 更看重什么                           | 常见设计                        |
| -------- | ------------------------------------ | ------------------------------- |
| 内部主键 | 唯一、短、写入友好、关联查询高效     | 号段、Snowflake、数据库自增     |
| 外部单号 | 不容易猜、可脱敏、可携带少量业务信息 | 前缀 + 日期 + 随机后缀 + 校验位 |
| 幂等键   | 能识别同一次业务请求                 | 业务单号、请求号、唯一业务约束  |

不要直接把内部自增 ID 暴露成外部订单号。
否则别人很容易通过递增规律推算业务规模。

还有一个跨语言边界：如果内部 ID 是 Snowflake 或号段生成的 64 位整数，传给前端或弱类型系统时建议按字符串传输。
JavaScript 的安全整数范围有限，直接用 number 承接大整数可能丢精度，最后出现“页面展示的 ID 和后端不一致”。

更稳的做法是：

- 内部主键追求存储和查询效率。
- 外部单号单独设计可读性、不可猜性和风控规则。
- 幂等键要能和业务语义绑定，比如同一个支付单、同一个回调事件、同一次创建请求。

## 数据库自增适合什么场景？

先区分两种“数据库自增”：

| 类型                | 适合什么                      | 主要边界                     |
| ------------------- | ----------------------------- | ---------------------------- |
| 业务表自身自增主键  | 单库单表、本地唯一即可        | 分库分表后全局会冲突         |
| 独立发号表 / 序列表 | 中心化生成全局 ID，小流量可用 | 吞吐、单点、主从切换都要治理 |

最朴素的中心化做法是建一张发号表，用数据库自增或单行更新来生成 ID。

```sql
CREATE TABLE sequence_id (
  id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  stub CHAR(10) NOT NULL,
  PRIMARY KEY (id),
  UNIQUE KEY uk_stub (stub)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

它的优点很直接：

- 实现简单，依赖大家都熟悉的数据库事务。
- ID 严格递增，对数据库主键索引友好。
- 适合后台系统、小流量业务、早期单体系统。

缺点也很明显：

- 每生成一个 ID 都要访问数据库，吞吐上限低。
- 发号库会成为单点，需要主备、切换和监控。
- ID 递增会暴露业务规模，比如外部订单号容易被推算订单量。

生产里不建议为了“推进序列”滥用 `REPLACE INTO`。它的语义是冲突时先删再插，可能触发额外索引维护和触发器副作用。更稳的方式是使用单行 `UPDATE` 或直接进入号段模式。

## 号段模式为什么更适合高并发？

号段模式的核心思想是：**数据库不再每次发一个 ID，而是一次分配一段范围，应用在内存里慢慢用**。

假设发号表里有一行：

```text
biz_type = order
current_max_id = 10000
step = 1000
version = 7
```

应用抢号段时执行：

```sql
UPDATE sequence_id_generator
SET current_max_id = current_max_id + step,
    version = version + 1
WHERE biz_type = 'order'
  AND version = 7;
```

如果更新成功，本实例拿到的就是 `(10000, 11000]`，后续 `10001` 到 `11000` 都可以在本地内存直接发。

完整流程可以按这几步记：

```text
1. 读取 current_max_id、step、version
2. 用 version 做 CAS 更新
3. 检查 affected_rows 是否为 1
4. 成功后本地发放 (oldMax, oldMax + step]
5. 失败说明被别的实例抢走，重新读取后有限重试
```

这里不要设计所谓“降级 ID”。
发号系统短暂不可用可以降级业务入口、排队或熔断，但不能随便造一套不受统一约束的 ID，否则唯一性底线会被打穿。

```text
数据库分配号段
  │
  ├── 实例 A：10001 ~ 11000
  ├── 实例 B：11001 ~ 12000
  └── 实例 C：12001 ~ 13000
```

号段模式的收益是把“每次发号访问数据库”变成“每用完一段访问数据库”，吞吐和延迟都明显更稳。它适合订单、交易、流水、业务单据这类高并发但仍希望 ID 趋势递增的场景。

落地时要注意四个细节：

1. **CAS 更新必须检查影响行数**：更新失败说明号段被其他实例抢走，要重新读取后有限重试。
2. **号段会跳号**：实例重启后，内存里没用完的 ID 会浪费；这不影响唯一性，不要回收。
3. **提前加载下一段**：当前号段快用完时异步加载下一段，避免临界点请求等待数据库。
4. **数据库高可用要谨慎**：主从切换后不能回到旧的 `current_max_id`，否则会重复发号。

如果团队不想自己维护发号服务，可以优先评估成熟的号段模式方案或云厂商发号能力，但仍要压测 P99 延迟、数据库故障时的行为和号段耗尽告警。

号段模式还要特别注意“双号段”。

如果等当前号段完全用完才去数据库申请下一段，临界点请求会突然变慢。
更稳的做法是：

```text
当前号段消耗到阈值
 -> 异步预加载下一号段
 -> 当前号段用完后无缝切换
```

这样能把数据库访问带来的延迟毛刺摊平。
但双号段也带来更明显的跳号：实例重启时，两段里没用完的 ID 都可能浪费。
这不是 bug，只要不重复就可以接受。

故障恢复时也要坚持这个原则：

- 应用重启后，内存未用完的号段直接废弃，不回收
- 发号服务宕机后，新实例只能申请新号段，不复用旧内存段
- 数据库主从切换后，必须保证 `current_max_id` 不回退
- 多库发号时，各库号段范围、步长或业务分片必须互不重叠

号段模式最怕的不是浪费，而是“为了省号”把已经分配出去的范围再次发放。

## Snowflake 的本质是什么？

Snowflake 是一种 64 位趋势递增 ID 方案，常见结构如下：

```text
0 | 41 位时间戳 | 10 位机器/机房编号 | 12 位毫秒内序列号
```

它的好处是发号过程基本在本地完成，不必每次访问数据库或 Redis。只要不同节点的 Worker ID 不冲突，同一节点同一毫秒内的 sequence 不溢出，就能生成全局唯一 ID。

标准 Snowflake 的能力边界也很清楚：

- 41 位毫秒时间戳通常可支撑约 69 年，具体取决于自定义起始时间。
- 12 位序列号表示单个 Worker 每毫秒最多 4096 个 ID。
- 10 位 Worker 空间表示最多 1024 个节点，实际会按机房、机器、业务调整。

它最容易出问题的地方有两个。

第一是 **Worker ID 分配**。在 Kubernetes 里 Pod 会动态扩缩容，不能简单把机器号写死。常见做法是通过 ZooKeeper、etcd、数据库或 Redis 申领 Worker ID，并用租约避免冲突。

这里要把 Worker ID 当成一种租约资源，而不是普通配置项。
一套可用机制至少要回答：

- 谁负责分配 Worker ID
- 租约多久续一次
- 实例宕机后多久回收
- 旧实例网络恢复后怎么隔离
- 同一个 Worker ID 是否可能被两个实例同时持有

如果旧实例恢复后还在用原 Worker ID，新实例又已经拿到同一个 Worker ID，那么 Snowflake 的“本地发号”就会直接变成重复发号风险。

第二是 **时钟回拨**。NTP 同步、虚拟机迁移、人工改时间都可能让系统时间倒退。如果不处理，旧时间戳加相同 Worker ID 和 sequence 就可能生成重复 ID。

常见处理方式是：

| 处理方式              | 优点           | 代价                   |
| --------------------- | -------------- | ---------------------- |
| 小幅回拨等待          | 简单，保证唯一 | 回拨期间请求阻塞       |
| 超阈值拒绝发号        | 避免重复       | 业务要能承受短暂不可用 |
| 持久化 last timestamp | 重启后可校验   | 依赖本地或远端存储     |
| 动态备用 Worker ID    | 可用性更高     | 要处理租约和脑裂       |

所以 Snowflake 不是“无状态到什么都不用管”。它把数据库压力转移到了时间、Worker ID 和容量规划上。

Snowflake 还要算清容量。

标准 41 位时间戳、10 位 Worker、12 位序列的组合，大致意味着：

| 位段     | 影响                       |
| -------- | -------------------------- |
| 时间戳   | 决定算法生命周期           |
| Worker   | 决定最多可区分多少发号节点 |
| sequence | 决定单节点每毫秒发号上限   |

如果你把 Worker 位数调大，sequence 位数可能就变小。
如果 sequence 位数变小，单节点峰值发号能力就下降。
所以 Snowflake 改造不是随便挪几位，而是在“节点数、生命周期、峰值吞吐”之间做取舍。

单毫秒序列耗尽时，常见处理也有取舍：

| 方式           | 结果                        | 代价                        |
| -------------- | --------------------------- | --------------------------- |
| 等下一毫秒     | 保持时间语义简单            | 峰值时出现等待毛刺          |
| 增加 Worker 数 | 横向摊薄单节点压力          | Worker ID 治理更复杂        |
| 调整位数       | sequence 更大或 Worker 更多 | 生命周期、节点数要重新计算  |
| 借未来时间     | 用逻辑时间继续发号          | ID 解析出的时间可能短暂超前 |

一个常见回答是：

**Snowflake 不是中心化发号瓶颈，但它需要严肃治理时钟、Worker ID 和毫秒内序列耗尽。**

## Redis INCR 和 UUID 为什么不是万能答案？

Redis `INCR` 很快，也能保证单 key 在单主正常运行时递增：

```bash
INCR order:id
```

但它有几个工程边界：

- 单个计数 key 会形成热点。
- AOF/RDB 持久化存在延迟，极端崩溃后可能 ID 回退。
- 主从异步复制切换时，新主节点可能落后于旧主节点。

因此 Redis 更适合短周期流水号、业务可兜底去重的场景。核心订单主键仍建议用数据库唯一约束兜底，不能只相信 Redis 计数。

Redis 发号可以这么判断：

| 问题                      | 判断                           |
| ------------------------- | ------------------------------ |
| 是否能接受极端 ID 回退    | 不能接受就不要只依赖 Redis     |
| 是否已有 Redis 高可用能力 | 没有就别为了发号单独引入复杂度 |
| 是否会形成单 key 热点     | 高并发单业务发号要谨慎         |
| 是否有落库唯一约束兜底    | 关键主键必须有最终防线         |

所以 Redis 更适合临时流水、短周期序号、可重试去重的业务，不适合裸奔承接核心交易主键。

UUID 的优势是简单、无需中心组件。尤其 UUID v7 具备时间有序特性，比随机 UUID v4 更适合写入索引。但它通常是 128 位，比 64 位 Snowflake 更占空间；如果用字符串存储，还会进一步增加索引体积。

对于高写入 MySQL 主键，随机 UUID v4 不友好，因为写入位置分散，容易造成页分裂和缓存命中下降。若确实要用 UUID，优先使用 `BINARY(16)` 或数据库原生 UUID 类型，并评估 UUID v7 或时间重排方案。

UUID v7 可以作为一个值得知道的新选项：

| 维度         | UUID v4        | UUID v7                  |
| ------------ | -------------- | ------------------------ |
| 主要来源     | 随机数         | 毫秒时间戳 + 随机/计数位 |
| 写入局部性   | 差             | 更接近时间有序           |
| 运维复杂度   | 低             | 低                       |
| 存储空间     | 128 bit        | 128 bit                  |
| 适合做主键吗 | 高写入表不友好 | 可评估，但仍要压测       |

它的优势是不用分配 Worker ID，跨语言标准化程度更高。
它的代价是 ID 更长，且同一毫秒内的单调性取决于具体生成器实现。
如果是 MySQL 高写入主键，仍要压测索引大小、页分裂和缓存命中。

## 面试里怎么做选型？

可以按这条线回答：

1. **先确认业务要求**：只要求唯一，还是还要求趋势递增、可反查时间、外部不可猜、跨语言兼容？
2. **再看吞吐和可用性**：发号量低可以数据库自增，高并发更适合号段或 Snowflake。
3. **再看运维复杂度**：Snowflake 要管 Worker ID 和时钟；号段要管发号库和号段加载。
4. **最后做兜底**：数据库唯一索引、幂等表、发号监控、时钟监控、号段耗尽告警都不能省。

常见选型可以这样收口：

| 场景                         | 推荐方案               | 原因                         |
| ---------------------------- | ---------------------- | ---------------------------- |
| 后台系统、小流量单体         | 数据库自增             | 简单，维护成本低             |
| 订单、交易、流水高并发       | 号段模式               | 趋势递增，吞吐稳定           |
| 大规模多节点、本地高性能发号 | Snowflake 或成熟改良版 | 延迟低，不依赖每次远程调用   |
| 云原生中小系统、跨语言兼容   | UUID v7                | 接入简单，时间有序           |
| 临时流水号、可业务兜底去重   | Redis INCR             | 快，但要接受持久化和切换边界 |

一个实用建议是：**内部主键和外部展示号可以分开设计**。内部主键追求写入效率和关联查询；外部订单号可以加随机后缀、业务前缀或脱敏规则，避免泄露业务规模。

## 工业实现应该怎么评估

如果团队不想自己写发号器，可以评估成熟实现或云厂商能力。
不要只看“支持 Snowflake / 支持号段”，还要看这些问题：

| 评估项   | 要确认什么                             |
| -------- | -------------------------------------- |
| 部署模式 | 客户端本地发号，还是独立发号服务       |
| 依赖组件 | 是否依赖数据库、Redis、ZooKeeper 等    |
| 高可用   | 发号服务故障、数据库切主时会怎样       |
| 延迟指标 | 平均延迟之外，P99 / TP999 是否稳定     |
| 时钟处理 | 回拨时等待、拒绝、换 Worker 还是持久化 |
| 号段策略 | 是否支持双号段、预加载、耗尽告警       |
| 维护状态 | 项目是否活跃，依赖是否安全             |

如果是号段类实现，重点看数据库高可用、双号段和号段耗尽。
如果是 Snowflake 类实现，重点看 Worker ID 分配、时钟回拨和序列耗尽。
如果是可配置位段的实现，还要算清楚：

- 自定义起始时间能支撑到哪一年
- Worker 位数够不够扩容
- sequence 位数能不能覆盖峰值
- 生成器长期维护状态和依赖安全性是否可接受

## 发号系统要监控什么

分布式 ID 一旦出问题，往往是全站级事故。
生产上至少要监控：

- 发号成功率、失败率、超时率。
- P95、P99、TP999 延迟。
- 号段剩余量和号段加载失败次数。
- Worker ID 申领失败、租约过期、冲突次数。
- 时钟回拨次数和回拨幅度。
- Redis / 数据库 / 注册中心依赖状态。
- 落库唯一键冲突次数。

真正危险的不是“ID 跳号”，而是：

**重复发号、无法发号、或者发号延迟拖垮核心链路。**

## 容易踩的坑

### 把连续当成唯一

连续不是分布式 ID 的核心目标。
号段浪费、实例重启、序列预加载都会造成跳号。
只要业务不依赖连续性，就不应该为了补洞增加重复风险。

### 只看生成速度，不看数据库写入

随机 UUID 生成很快，但作为 MySQL 主键可能让 B+ 树写入更分散。
ID 方案要和落库方式一起看。

### 忽略 Worker ID 冲突

Snowflake 里两个节点拿到同一个 Worker ID，再遇到相同毫秒和相同 sequence，就可能重复。
容器化环境尤其要把 Worker ID 当成租约资源治理。

### 没有唯一约束兜底

任何发号算法都可能因为实现 bug、时钟、切主或配置错误出现重复。
关键业务落库时仍要靠唯一索引、幂等表或业务约束兜底。

## 小结

1. 分布式 ID 的核心不是连续，而是全局唯一、高可用、高性能和可治理。
2. 数据库自增简单但吞吐低；号段模式用“批量分配 + 本地发号”换取更高性能。
3. Snowflake 性能好，但必须处理 Worker ID 分配、时钟回拨和毫秒内序列耗尽。
4. Redis `INCR`、UUID v7、成熟发号组件都可以用，但要看持久化、热点、索引体积、维护状态和压测结果。
5. 内部主键、外部单号和幂等键最好分开设计，关键业务最终仍要靠唯一约束、幂等和监控告警兜底。
