# 如何用 AI 做代码审查但不放弃工程判断？

> AI 适合帮你扩大检查面，但不能替你承担最终判断。代码审查仍然要回到事实、测试和风险。

## AI Review 的定位是什么？

AI 做代码审查，最适合当“候选风险发现器”。它能快速扫 diff、对照规则清单、提醒你可能漏掉的边界，但它不能替代负责人做最终判断。

一次有效的 AI Review，目标不是让 AI 给出“能不能合并”的结论，而是让它输出这些问题：

- 这段改动可能引入哪些行为回归？
- 风险发生在哪个文件、哪个函数、哪段 diff？
- 它的判断依据是什么？
- 如果要确认这个风险，应该跑什么测试或补什么验证？

也就是说，AI 可以扩大检查面，但最终裁判仍然是工程证据：代码事实、业务规则、测试结果、运行日志和人工 Review。

## 审查前要给 AI 哪些材料？

AI 审查质量很依赖上下文。只丢一句“帮我看看这段代码”通常会得到泛泛建议。更稳的做法是把输入材料分层给清楚：

| 材料               | 作用                                 | 注意点                         |
| ------------------ | ------------------------------------ | ------------------------------ |
| PR diff / 当前改动 | 让 AI 聚焦本次变化                   | 不要让它泛扫全仓库后跑题       |
| 需求或 Spec        | 判断行为是否符合目标                 | 只给本次相关片段               |
| 架构和项目规则     | 判断是否破坏分层、命名、错误码等约定 | 规则要具体可验证               |
| 相关测试结果       | 区分已验证和未验证                   | 测试全绿不等于行为一定正确     |
| 错误日志或线上背景 | 帮助定位真实风险                     | 保留时间、参数、异常栈等证据   |
| 禁止修改范围       | 防止建议扩大改动面                   | 例如生成物、迁移脚本、生产配置 |

上下文不是越多越好。优先给本次审查必须用到的信息，长文档用链接或文件路径让 AI 按需读取。

## 一条可复用的审查流程

可以把 AI Review 固定成 6 步：

```text
明确目标 -> 收集输入材料 -> AI 分维度审查 -> 人工复核高风险项 -> 跑验证 -> 记录剩余风险
```

1. **明确目标**：本次是修 bug、加功能、重构，还是补测试？目标不同，风险重点不同。
2. **收集材料**：给 diff、需求、相关测试、项目规则和禁止范围。
3. **AI 审查**：要求按安全、数据一致性、异常、性能、测试缺口等维度输出候选问题。
4. **人工复核**：重点复核业务语义、兼容性、权限、事务和回滚成本。
5. **跑验证**：用测试、lint、构建、接口调用、日志或截图确认问题是否真实。
6. **记录风险**：采纳了什么建议、拒绝了什么建议、还有什么没覆盖，都要写清楚。

这条链路里，AI 只负责第 3 步的一部分。它不能跳过人工复核，也不能替代最终验收。

## 提示词怎么写？

提示词要避免“泛泛看看”。可以直接要求输入、维度和输出格式：

```text
你是一位负责 Java 后端代码审查的工程师。

任务：审查下面这次改动，只输出可能影响正确性、安全性、性能或可维护性的候选风险。

上下文：
- 需求目标：<本次需求或 bug 描述>
- 禁止范围：<不能改的目录/文件/行为>
- 已运行测试：<命令和结果>
- 相关业务规则：<权限、租户、事务、幂等等>

审查维度：
1. 行为回归和兼容性
2. 权限、租户、数据越权
3. 事务边界、并发一致性、幂等
4. 异常处理和失败语义
5. SQL、索引、循环调用、资源释放
6. 测试缺口和不可观测风险

输出格式：
- 严重级别：P0/P1/P2/P3
- 位置：文件路径 + 函数或行号
- 问题：一句话说明
- 证据：引用 diff 或上下文中的事实
- 影响：可能导致什么后果
- 建议：如何修改或验证
```

最关键的是“证据”和“验证方式”。没有证据的结论，只能算猜测；没有验证方式的建议，很难进入工程闭环。

## 输出结果怎么分级？

建议让 AI 先按严重级别排序，避免被风格建议淹没：

| 级别 | 含义                     | 例子                                 |
| ---- | ------------------------ | ------------------------------------ |
| P0   | 会导致严重事故或数据破坏 | 越权访问、资金错误、生产数据误删     |
| P1   | 高概率行为回归或安全问题 | 租户条件丢失、事务外写入、幂等失效   |
| P2   | 需要修复的质量问题       | 异常吞掉、资源未释放、缺少关键测试   |
| P3   | 建议优化                 | 命名不清晰、重复代码、局部可读性问题 |

AI 容易输出很多 P3 建议，但真正值得优先处理的是 P0/P1/P2。代码审查不是代码美化，优先级应该是 bug、安全、数据一致性、兼容性和测试缺口。

## 人工要兜住哪些判断？

人工 Review 的价值主要在 AI 不容易准确理解的地方：

- **业务语义**：这个状态能不能从 A 跳到 B？老用户是否还走旧链路？
- **权限和租户**：查询条件里有没有带租户、组织、数据范围？
- **事务边界**：外部调用是否放进事务？失败后会不会半成功？
- **兼容性**：接口字段、错误码、消息格式有没有破坏调用方？
- **并发一致性**：重复请求、重试、消息重复消费会不会导致脏数据？
- **回滚成本**：建议改动是否过大，是否会影响发布和灰度？

如果 AI 的建议和业务规则冲突，以业务规则和代码证据为准。不要因为它说得像专家，就把自然语言判断当事实。

## 一个订单导出接口怎么审？

假设本次 PR 新增“订单导出 CSV”接口。你可以给 AI 这些上下文：

```text
需求：支持按时间范围导出当前租户订单，最多 5000 条，时间跨度不超过 31 天。
风险边界：不能导出其他租户数据；不能全表扫描；失败要返回明确错误码。
已运行测试：OrderExportServiceTest 通过。
```

AI 可能输出：

| 候选风险                 | 人工复核方式                         |
| ------------------------ | ------------------------------------ |
| 查询条件缺少 `tenant_id` | 查 SQL 构造、补越权测试              |
| 时间范围没有上限         | 查参数校验、补超过 31 天用例         |
| 导出数量没有限制         | 查分页或 limit，压测大租户数据       |
| 测试只有成功路径         | 补无权限、无数据、越界时间、重复请求 |
| CSV 字段顺序没固定       | 对照产品约定或接口文档               |

这里 AI 的价值是把风险列出来；人的价值是判断这些风险是否真实，并用测试或代码证据确认。比如“缺少索引”不能只看 AI 说法，要用表结构、执行计划或实际慢查询证据判断。

## 审查不是验收

AI Review 通过，不代表代码可以合并。它只能说明“审查阶段没有发现更明显的问题”。最终还要回到验收链路：

```text
看 diff 范围 -> 跑测试 -> 跑 lint/构建 -> 验证关键路径 -> 记录未覆盖风险
```

如果 AI 提了问题，要么修复并补验证，要么明确拒绝理由。不要把“AI 没再提”当成“问题已经解决”。

## 小结

1. AI Code Review 是候选风险发现器，不是最终裁判。
2. 审查前要提供 diff、需求、项目规则、测试结果和风险边界，避免 AI 泛泛而谈。
3. 输出必须带严重级别、位置、证据、影响和验证方式。
4. 人工重点兜业务语义、权限租户、事务并发、兼容性和回滚成本。
5. AI 审查之后仍要用测试、lint、构建、日志或手工验证完成验收闭环。
