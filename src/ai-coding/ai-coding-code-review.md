# 如何用 AI 做代码审查但不放弃工程判断？

> AI 适合帮你扩大检查面，但不能替你承担最终判断。代码审查仍然要回到事实、测试和风险。

## AI 适合审什么？

AI 很适合发现：

- 空指针、边界条件、异常吞掉。
- 权限校验缺失。
- 并发共享状态。
- SQL 拼接和注入风险。
- 测试覆盖缺口。

但 AI 容易误判业务意图，所以不要把它的结论当最终结论。

## 怎么组织审查？

建议把提示词写成清单：

```text
请按以下维度审查：
1. 行为回归
2. 安全风险
3. 并发和事务边界
4. 性能与资源释放
5. 测试缺口
每条结论必须给文件位置和理由。
```

让 AI 输出“发现 + 证据 + 建议”，而不是只输出泛泛建议。

## 人工要兜什么？

人工重点看三件事：

1. AI 有没有误解业务规则。
2. 建议是否会引入更大改动。
3. 是否有测试或日志能验证结论。

如果没有证据，就不能把“看起来像问题”当成问题。

## 小结

1. AI Code Review 是扩展检查面，不是替代负责人。
2. 审查提示词要要求证据、位置和风险等级。
3. 人工要重点判断业务语义、改动收益和验证方式。
4. 没有测试或事实支撑的建议，不应该直接合入。
