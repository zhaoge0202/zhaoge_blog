# 如何给 AI 准备一个可复用的项目上下文？

> 好的上下文不是越长越好，而是让 AI 每次进项目都先知道最容易犯错的规则。

## 可复用上下文不是 README 压缩版

项目上下文的目标不是“让 AI 读完整个项目”，而是减少它猜错的概率。真正可复用的上下文，通常由三类信息组成：

```text
长期稳定规则 + 本次任务约束 + 按需检索证据
```

长期稳定规则告诉 AI 在这个仓库里怎么工作；本次任务约束告诉它这轮要做什么、不能碰什么、做到什么程度算完成；按需检索证据让它在需要时读取代码、日志、文档或测试，而不是一开始把所有资料塞进上下文。

所以项目上下文不应该写成第二份 README。README 给人看，项目上下文给 Agent 执行任务用，重点是“会影响 AI 行为的规则”。

## 上下文该分哪几层？

建议按作用范围分层：

| 层级      | 放什么                   | 加载方式         | 例子                                   |
| --------- | ------------------------ | ---------------- | -------------------------------------- |
| 全局规则  | 全仓库通用约束           | 每次任务常驻     | 包管理器、测试命令、提交规范、禁止目录 |
| 目录规则  | 局部模块特殊约定         | 处理该目录时加载 | `src/payment/` 的状态机规则            |
| 任务 Spec | 本次需求和验收标准       | 每次任务单独提供 | 目标、非目标、边界、完成标准           |
| 动态证据  | 代码片段、日志、接口文档 | 按需检索读取     | diff、错误栈、执行计划、截图           |
| 本地记忆  | 个人偏好和临时经验       | 不作为团队事实   | 本机启动 Redis 的特殊命令              |

全局规则要短，目录规则要具体，任务 Spec 要可验收，动态证据要可追溯。本地记忆不能替代团队共享规则，因为它没有版本化和审查。

## 项目上下文模板怎么写？

可以用下面这个模板起步：

```markdown
# Project Agent Context

## Stack

- Runtime: Node 20 / Java 17 / ...
- Package manager: npm / Maven / Gradle
- Frameworks: ...

## Commands

- Install: `npm ci`
- Test: `npm test`
- Lint: `npm run lint`
- Build: `npm run build`

## Architecture

- `src/api/`: HTTP 接口层，只做参数校验和响应转换。
- `src/service/`: 业务编排，事务边界在这里。
- `src/repository/`: 数据访问，不写业务规则。

## Must Follow

- 所有接口必须校验租户和权限。
- 外部 HTTP / LLM / 文件上传不能放在数据库事务内。
- 错误码使用统一枚举，不直接抛通用运行时异常。

## Never Touch

- 不修改 `dist/`、`node_modules/`、生成目录。
- 不提交 `.env`、密钥、生产配置。
- 不直接改数据库迁移脚本，除非任务明确要求。

## Verification

- 改 Markdown 后至少跑 `npm test`。
- 改接口后补单元测试和关键路径集成测试。
- 前端交互改动需要浏览器验证或截图证据。

## Deep References

- 架构说明：`docs/architecture.md`
- API 规范：`docs/api.md`
- 部署说明：`docs/deploy.md`
```

注意写法要具体可验证。“保持代码优雅”这种话基本没有约束力；“Controller 不写业务逻辑，事务只放 Service 层”才是 AI 能执行、你也能检查的规则。

## 怎么组织一次任务上下文？

每次给 AI 下任务时，建议固定提供四块：

```text
目标：这次要完成什么。
上下文：重点读哪些目录、文件、日志或文档。
约束：不能改什么，必须保留什么兼容行为。
完成标准：哪些命令、测试、截图或日志能证明完成。
```

比如：

```text
目标：修复订单导出接口没有限制时间范围的问题。
上下文：阅读 src/order/export、OrderExportServiceTest 和接口文档的导出章节。
约束：不改 CSV 字段顺序，不改数据库表结构，保持原错误码格式。
完成标准：补超过 31 天的失败测试，运行订单导出相关测试并汇报结果。
```

这种写法比“帮我修一下订单导出”稳定得多，因为它同时给了目标、范围、边界和验收证据。

## 运行时怎么加载上下文？

比较稳的流程是：

1. 先读项目根规则，确认命令、目录、禁止清单。
2. 用搜索定位相关文件，不确定位置时先语义检索或关键词检索。
3. 只读取和任务相关的代码、测试、文档和日志。
4. 如果任务变复杂，把阶段性结论写成短笔记，避免长对话丢失。
5. 改完后用测试、lint、构建和人工检查验证。
6. 如果 AI 反复犯同类错误，再把规则沉淀回项目上下文。

这就是渐进式上下文：先给最关键规则，再按任务逐步加载证据。它比一次性把 README、接口文档、历史需求全塞进去更稳定。

## 什么时候需要 RAG 或知识库？

不要一上来就把项目上下文做成向量库。规模和维护成本不到位时，手工索引反而更可控。

| 项目规模 | 推荐方式                     | 说明                               |
| -------- | ---------------------------- | ---------------------------------- |
| 小项目   | 一个根规则文件 + 少量链接    | 保持精简，AI 可以直接读代码        |
| 中等项目 | 根规则 + 目录规则 + 文档索引 | 用路径和标题帮 AI 找资料           |
| 大型项目 | 文档索引 + 检索系统 + 评测集 | 需要权限、版本、召回质量和更新机制 |

RAG 能解决“资料很多、不能全塞上下文”的问题，但它也会引入召回错误、权限过滤、版本更新和评测成本。没有稳定评测集之前，不要把“检索到了”当成“证据可靠”。

## 规则文件不是强约束

项目上下文是软约束。写了“不要改生产配置”，不代表 AI 一定不会改。高风险行为要靠工具级机制兜底：

- 敏感文件：用权限、Hook 或 CI 阻止修改。
- 密钥泄露：用 secret scan 和提交前检查。
- 数据库变更：用迁移审查和审批流程。
- 危险命令：用沙箱、审批和命令规则拦截。
- 格式规范：用 formatter、lint 和测试机械化执行。

能用工具强制的规则，不要只写成自然语言提醒。

## 怎么维护项目上下文？

上下文要像代码一样维护。

添加规则时，先问一句：这条规则删掉后，AI 是否大概率再次犯同类错误？如果答案是“会”，再写进去；如果只是一次偶发事故，不要急着变成永久规则。

删除规则时，也要果断。过时命令、废弃目录、已经由 lint/CI 强制的格式规则，都应该移出常驻上下文，避免稀释真正重要的约束。

长任务还要注意上下文变脏。连续做很多轮后，可以把已完成事项、关键决策、剩余风险压缩成结构化笔记，再清理无关历史。必要时把检索、审查、验证拆给不同子任务，主任务只保留高密度结论。

## 小结

1. 可复用项目上下文不是 README 压缩版，而是让 AI 少猜、少犯重复错误的工作规则。
2. 上下文要分成全局规则、目录规则、任务 Spec、动态证据和本地记忆，不同层级按需加载。
3. 规则要具体可验证，最好写清命令、目录职责、禁止清单、验收标准和深入资料入口。
4. 长上下文不能替代高信噪比，复杂任务要用渐进式加载、结构化笔记和必要的检索系统。
5. 规则文件只是软约束，高风险操作必须靠权限、Hook、lint、CI 和审计兜底。
