# 如何给 AI 准备一个可复用的项目上下文？

> 好的上下文不是越长越好，而是让 AI 每次进项目都先知道最容易犯错的规则。

## 上下文该写什么？

优先写代码里读不出来的信息：

- 技术栈版本和包管理器。
- 常用命令和测试要求。
- 目录职责。
- 禁止修改的生成物或资料源。
- 提交规范和分支规范。
- 业务术语和关键边界。

不要把 README、接口文档、历史背景全塞进去。上下文过长会挤占任务本身的信息。

## 怎么分层？

推荐三层：

```text
根规则：全仓库通用约束
目录规则：某个模块的特殊约定
任务提示：本次需求的目标和验收标准
```

这样既能复用，又不会让所有任务都背负无关信息。

## 怎么维护？

每次 AI 犯错后，不要只改这次输出，要反思是否需要把规则沉淀到上下文里。

判断标准很简单：这条规则删掉后，AI 是否大概率再次犯错？如果是，就保留；否则不要写。

## 小结

1. 可复用上下文要写项目约束，不写百科。
2. 规则要分层，避免所有任务加载所有背景。
3. 上下文要随着错误复盘持续更新。
4. 好上下文的目标是减少重复解释和重复犯错。
