# AI 生成代码后应该怎样验收？

> AI 生成代码不能用“看起来能跑”验收，必须用测试、检查和真实运行证据闭环。

## 验收到底在验什么？

AI 生成代码最容易出现的问题是“表面正确”。变量名像那么回事，测试也可能补了，但真正的业务边界、权限、异常、事务和兼容性不一定对。

所以验收不是问“AI 说完成了吗”，而是问：

- 改动范围是否符合任务？
- 行为是否满足需求和非目标？
- 风险路径有没有测试或运行证据？
- 代码是否符合项目约定？
- 未覆盖风险是否被明确记录？

验收结论必须来自证据，而不是来自 AI 的总结。

## 最小验收链路

```text
读 diff -> 跑测试 -> 静态检查 -> 关键路径手测 -> 复核日志和边界
```

如果是前端，还要截图或浏览器自动化；如果是后端，要覆盖接口、数据库状态和异常分支。

这条链路可以拆成五层：

| 层级      | 要看什么                           | 证据                                 |
| --------- | ---------------------------------- | ------------------------------------ |
| Diff 范围 | 是否只改了相关文件，有没有顺手重构 | `git diff`、文件清单                 |
| 行为测试  | 正常、边界、异常、回归是否覆盖     | 单测、集成测试、端到端测试           |
| 静态检查  | 格式、lint、类型、依赖、漏洞       | lint、build、安全扫描                |
| 运行验证  | 真实路径是否能跑通                 | 接口调用、页面截图、日志、数据库状态 |
| 风险复核  | 未覆盖风险是否明确                 | PR 描述、最终回复、Review 记录       |

测试全绿只是证据之一。它能证明“这些测试覆盖的路径没坏”，不能证明所有行为都正确。

## 不同风险要验收到什么程度？

验收强度要和风险匹配。低风险改动不需要走完整发布评审，高风险改动也不能只跑一个单测就合并。

| 风险等级 | 典型改动                                                           | 最低验收要求                                                   |
| -------- | ------------------------------------------------------------------ | -------------------------------------------------------------- |
| 低风险   | 文案、日志、样式微调、一次性脚本                                   | 看 diff、跑相关最小测试或格式检查                              |
| 中风险   | 普通业务接口、模块内流程、查询条件调整                             | 单测/集成测试、lint/build、关键路径手工验证                    |
| 高风险   | 权限、支付、订单、数据迁移、并发一致性、生产配置、密钥、外部写操作 | 先只读分析和人工确认方案，小步实现，必须有回滚、审计和灰度证据 |

风险等级不是看代码行数，而是看失败后果。改一行租户过滤条件，风险可能比重构几十行工具函数更高。

高风险任务尤其要防止 AI 自行扩 scope：它可能为了“顺手修好”改配置、改迁移脚本、改 CI、改鉴权入口。验收时要专门问一句：这些改动是否都在任务交付契约里？

## Diff 要先看什么？

先看 diff，能快速发现很多 AI 常见问题：

- 修改了任务无关文件；
- 改了生成物、锁文件、配置文件但没说明原因；
- 删除了兼容逻辑或灰度开关；
- 为了让测试通过改弱了断言；
- 引入新依赖但没有必要；
- 把小修变成大重构。

如果 diff 范围已经失控，先不要继续跑测试。应该收窄改动，确认每个文件变化都能对应到任务目标。

## 测试应该怎么补？

AI 很容易只补 happy path。验收时要按风险补测试：

| 场景      | 必查测试                                   |
| --------- | ------------------------------------------ |
| 权限接口  | 未登录、无权限、跨租户、越权资源           |
| 写数据库  | 成功写入、失败回滚、重复请求、并发冲突     |
| 查询接口  | 空结果、分页边界、排序、过滤条件、索引命中 |
| 外部调用  | 超时、失败、重试、降级、幂等               |
| 文件/导出 | 大文件、空数据、字段顺序、编码、资源释放   |

高风险 bug 修复最好先补一个失败测试，确认能复现问题，再改实现。这样可以避免 AI 写出“看起来修了，但其实没覆盖原问题”的代码。

## 静态检查不能省

静态检查适合抓机械化问题：

- 格式化是否一致；
- 类型和编译是否通过；
- lint 是否有新增错误；
- 是否有未使用代码、死代码、调试日志；
- 是否有依赖漏洞或许可证风险；
- 是否出现硬编码密钥、生产地址、敏感信息。

能机械化验证的，不要交给 AI 自然语言承诺。格式、类型、安全扫描、测试门禁都应该尽量由工具执行。

## 运行验证要看真实状态

后端改动不能只看接口返回成功，还要看副作用：

- 数据库是否写了正确记录；
- 事务失败时是否回滚；
- 消息是否只发送一次；
- 缓存是否正确更新或失效；
- 日志里是否有 trace、错误码和关键参数；
- 慢查询、线程池、连接池是否出现异常指标。

前端改动不能只看页面能打开，还要看：

- 交互状态是否正确；
- 错误态、空态、加载态是否可见；
- 移动端和桌面端是否布局正常；
- 表单校验和可访问性是否保留；
- 控制台是否有报错。

如果无法完整验证，要在最终说明里写清“没有覆盖哪些风险”，不要用“应该没问题”收尾。

## AI 生成代码的高频风险

验收时可以重点盯这些：

- **吞异常**：`catch` 后只打印日志或直接返回成功。
- **权限遗漏**：只校验登录，不校验租户、组织、资源归属。
- **事务扩大**：把外部 HTTP、文件上传、AI 调用放进数据库事务。
- **性能退化**：循环查库、无界队列、全表扫描、一次性加载大集合。
- **兼容破坏**：改响应字段、错误码、事件格式、消息 topic。
- **测试造假**：降低断言、过度 mock、只测内部实现不测行为。
- **过度抽象**：为了几行代码引入复杂层级或新依赖。

这些问题不一定每次都会出现，但很适合做固定验收清单。

## 验收证据怎么留？

最终回复或 PR 描述里要写清：

```text
变更范围：
- 修改了哪些文件，为什么改。

验证命令：
- npm test：通过
- npm run lint：通过

手工验证：
- 订单导出正常路径成功。
- 超过 31 天返回明确错误。
- 跨租户导出被拒绝。

未覆盖风险：
- 未做大租户 5000 条以上压测。
- 未验证生产导出链路的对象存储限流。
```

这比“已完成”更可信。好的验收记录应该能让后来的人知道：你验证了什么、没验证什么、为什么可以合并。

## 一个订单导出改动怎么验收？

假设 AI 生成了订单导出接口，验收可以这样走：

1. 看 diff：只改导出相关 service、controller、测试和文档，没有动登录、支付、全局配置。
2. 跑测试：覆盖正常导出、无权限、跨租户、时间超过 31 天、超过 5000 条。
3. 查 SQL：确认带 `tenant_id` 和时间范围，避免全表扫描。
4. 手工调接口：下载 CSV，确认字段顺序、编码、空数据返回。
5. 看日志：失败路径有错误码和 trace，不泄露敏感数据。
6. 写风险：没有做大数据量压测，就明确标记。

这套验收比“接口返回 200”可靠得多。

## 什么时候不能合并？

出现这些情况，不建议合并：

- AI 解释不清为什么改了某些文件；
- 关键路径没有测试或手工验证；
- 为了通过测试削弱断言；
- 发现高风险问题但没有修复或记录；
- 修改了数据结构、权限、生产配置却没有审批；
- 验证命令失败但被忽略；
- 未说明剩余风险。

AI 生成代码可以提高速度，但合并标准不能降低。

## 小结

1. AI 代码验收要看证据，不看“看起来能跑”或 AI 自述。
2. 最小链路是读 diff、跑测试、做静态检查、验证关键路径、记录剩余风险。
3. 权限、租户、事务、异常、性能、兼容性和测试断言是高频风险点。
4. 能机械化验证的规则交给测试、lint、build、Hook 和 CI，不要只靠自然语言提醒。
5. 没有覆盖的风险必须明确写出，不能用“应该没问题”代替验收。
