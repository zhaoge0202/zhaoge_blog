# 如何用 AI 辅助重写技术文章？

> AI 可以帮你组织表达，但资料核验、结论边界和最终口径必须由人负责。

## AI 写作的边界是什么？

用 AI 重写技术文章，不是让模型代笔。更准确的定位是：AI 负责提高表达效率，人负责事实、边界和最终口径。

AI 擅长：

- 把零散材料整理成提纲；
- 改写难懂句子；
- 补充类比、例子、表格；
- 检查结构是否跳跃；
- 按统一风格润色。

AI 不应该独自决定：

- 哪个技术结论是正确的；
- 资料之间冲突时采用哪个口径；
- 版本边界是否成立；
- 线上实践是否可以泛化；
- 引用和参考是否真实可靠。

一篇好文章不是把资料拼成摘要，而是把“问题、原理、例子、边界、验收”讲成一条清楚的主线。

## 一条完整的 AI 辅助写作流程

建议按这条流程做：

```text
定题 -> 收集资料 -> 交叉验证 -> 建证据表 -> AI 出提纲 -> 人工定主线 -> 分段改写 -> 事实验收 -> 链路同步
```

1. **定题**：一篇文章只回答一个追问点，不写大杂烩。
2. **收集资料**：优先官方文档、源码、权威规范，再看工程实践资料。
3. **交叉验证**：同一个结论至少找两处证据，冲突点单独标记。
4. **建证据表**：把“结论从哪里来、是否过时、最终采用什么口径”写清。
5. **AI 出提纲**：让 AI 帮你拆结构，不要直接生成全文。
6. **人工定主线**：调整成“问题 -> 原理 -> 例子 -> 边界”的讲解路径。
7. **分段改写**：每次只给 AI 一个小节的上下文和要求。
8. **事实验收**：逐条核验术语、版本、命令、代码和引用。
9. **链路同步**：补 frontmatter、prev/next、README、sidebar、测试清单。

这条流程的重点是先验证再写。没有证据的内容，宁可标成“待确认”，不要让 AI 编一个看起来合理的结论。

## 资料核验表怎么做？

写作前可以先建一个小表：

| 结论                    | 证据来源           | 版本或时间          | 是否冲突 | 最终口径                 |
| ----------------------- | ------------------ | ------------------- | -------- | ------------------------ |
| Java 17 是 LTS          | 官方版本说明、JEP  | 2021-09             | 无       | 可写成长期支持基线       |
| 文本块是 Java 17 新特性 | 版本资料           | 与 Java 15 转正冲突 | 有       | 改成 Java 17 基线可用    |
| 虚拟线程能提升吞吐      | 官方说明、压测经验 | 取决于阻塞 I/O      | 有边界   | 写成降低阻塞等待线程成本 |

表格不一定放进正文，但它能防止两个问题：

1. 把过时资料写成确定事实。
2. 把不同版本的特性边界混在一起。

技术文章最怕“看起来对”。尤其是 Java 版本、MySQL 行为、框架迁移、JVM 参数这类内容，必须核对版本。

## 给 AI 的上下文怎么组织？

不要把所有资料一次性丢给 AI。更稳的输入是：

| 上下文     | 用途                                |
| ---------- | ----------------------------------- |
| 目标读者   | 决定解释粒度，比如 3-5 年 Java 后端 |
| 文章问题   | 限定这篇只回答什么                  |
| 写作规则   | 禁用模板腔、必须有小结和参考        |
| 证据表     | 告诉 AI 哪些结论可用，哪些有边界    |
| 本小节提纲 | 控制当前输出范围                    |
| 反例和误区 | 防止生成绝对化说法                  |
| 输出格式   | 规定标题、表格、代码块、字数        |

一个可用提示词可以这样写：

```text
你要辅助重写一篇面试知识库文章。

目标读者：3-5 年 Java 后端工程师。
文章问题：<这篇要回答的问题>。
本小节目标：<只写当前小节，不扩展其他主题>。
证据和边界：
- <结论 1 + 来源类型 + 版本边界>
- <结论 2 + 冲突点 + 最终采用口径>

写作要求：
1. 用自己的话讲，不照抄资料。
2. 先解释为什么，再给例子。
3. 不使用废弃模板标题。
4. 不编造引用；没有证据的内容标记待验证。
5. 输出 Markdown。
```

这样做的好处是 AI 只处理当前小节，不会把整篇资料乱揉在一起。

## 分段改写怎么控制质量？

分段改写时，可以把每个小节拆成三步：

1. **让 AI 先列检查点**：这个小节要讲哪些事实、例子和边界。
2. **人工删改检查点**：去掉无关内容，补上必须强调的误区。
3. **再让 AI 写正文**：要求围绕检查点展开，不新增未经验证的结论。

例如写“Optional 能不能消灭 NPE”时，不要只让 AI 写“Optional 的优点”。应该明确：

- `Optional.of(null)` 会抛异常；
- `get()` 不判断也会出问题；
- `orElse` 会提前计算默认值；
- Optional 适合作为返回值，不适合作实体字段和参数滥用。

AI 负责把这些点讲顺，人负责确认这些点是否准确。

## 常见反例和正确做法

| 反例                         | 问题                       | 更好的做法                   |
| ---------------------------- | -------------------------- | ---------------------------- |
| 直接让 AI “根据资料生成全文” | 容易变成资料拼接           | 先做证据表和提纲，再分段改写 |
| 搬运资料标题                 | 风格割裂，还可能带入旧模板 | 用问句标题和讲解式结构重写   |
| 只写结论不写边界             | 面试追问容易翻车           | 写清版本、场景、例外和误区   |
| 引用不存在的资料             | 破坏可信度                 | 找不到依据就标待验证         |
| 把第三方经验写成通用规律     | 容易误导                   | 明确“在某类场景下”成立       |

如果正文里需要纠偏，写“更准确的说法是……”即可，不要把矛头指向某个资料来源或个人。

## 怎么验收文章？

最终验收不要只看“读起来通顺”。至少检查六类问题：

1. **事实**：版本、命令、参数、术语是否正确。
2. **证据**：关键结论是否能追溯到资料、源码、规范或实践。
3. **边界**：有没有把“通常”“适合”“不适合”讲清。
4. **表达**：是否像白板讲解，而不是模板问答。
5. **结构**：是否有例子、表格、图示或流程，能支撑理解。
6. **站点链路**：frontmatter、prev/next、README、sidebar、测试是否同步。

对本知识库来说，还要额外检查：不出现废弃模板标题，不出现外部个人名称，不照抄资料原文，每篇结尾有唯一 `## 小结` 和 `## 参考`。

## 小结

1. AI 辅助技术写作的核心不是代笔，而是让资料验证、提纲拆解、分段改写和事实验收更高效。
2. 写作前要建立证据表，区分官方事实、工程经验、版本边界和待验证结论。
3. 给 AI 的上下文要高信噪比，只给当前小节需要的资料、规则和输出要求。
4. 分段改写比一次生成全文更可控，人工必须负责最终技术口径。
5. 文章验收要同时检查事实、证据、边界、表达、结构和站点链路。
