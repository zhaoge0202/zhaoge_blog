# 接口幂等怎么设计？

> 幂等真正要解决的不是“用户会不会连点两次”，而是同一笔业务意图重复到达后，系统会不会把状态改乱。

幂等题几乎一定会被问到，但很多回答会停在一句：

> “前端防重复提交，后端加分布式锁。”

这太浅了。因为重复请求的来源远不止用户连点：

- 客户端超时后自动重试
- 网关或 SDK 自动重试
- 第三方回调重复发送
- MQ 至少一次投递
- 人工补偿脚本重复执行

所以幂等不是页面体验优化，而是系统状态保护。

## 先把两个概念分开：去重不等于幂等

这两个词经常混着用，但最好分开讲。

| 概念 | 重点                                           |
| ---- | ---------------------------------------------- |
| 去重 | 识别“这是不是同一笔请求”                       |
| 幂等 | 就算同一笔请求来多次，最终状态也只正确变化一次 |

去重更像“识别重复”，幂等更像“重复了也不能出事”。

所以很多稳妥方案其实是组合拳：

- 业务唯一键识别同一笔请求
- 数据库唯一约束或去重表兜底
- 状态机保证状态不会重复推进

## 幂等先按操作类型拆，不同类型做法不一样

### 查询类

查询天然更容易做幂等，因为不改状态。

同样查 10 次订单详情，结果应该一样，通常不需要额外设计。

### 创建类

创建类最怕重复插入。

典型例子：

- 重复创建订单
- 重复创建支付流水
- 重复发券

这类操作最常用的手段是：

- 业务唯一单号
- 请求幂等号
- 数据库唯一索引

### 更新类

更新类要分两种：

| 更新方式    | 幂等难度   |
| ----------- | ---------- |
| 赋值更新    | 更容易幂等 |
| 累加 / 扣减 | 更容易出事 |

比如：

- `status = PAID` 更容易做幂等
- `balance = balance - 100` 一旦重复执行就直接多扣一次

所以更新类通常要配状态机、版本号、业务校验一起看。

## 设计幂等时，最常用的是这几种手段

## 1. 业务唯一键

这是最优先考虑的方案。

比如：

- 订单号
- 支付流水号
- 请求号
- 消息 ID

如果系统能明确知道“这两次请求本来就是同一笔业务”，后续判断就有抓手。

一句很实用的话：

**能靠业务唯一键识别同一笔业务时，优先别上锁。**

## 2. 数据库唯一索引

这对创建类操作特别稳。

比如创建订单时，对 `request_id` 或 `order_no` 建唯一索引：

```sql
create unique index uk_order_request on t_order(request_id);
```

这样即使并发进来两次，数据库层也只会让一条成功。

它的价值在于：

- 简单
- 强约束
- 能兜住并发竞态

但要注意一点：

唯一索引只能兜住“不能插两条”，不等于完整幂等流程就结束了。
你还得决定：

- 第二次请求该返回什么
- 如何查到第一次执行的结果

## 3. 去重表

去重表适合“业务表不方便直接加唯一约束”或者“想把幂等记录单独管理”的场景。

常见思路是：

1. 用业务唯一号往去重表插一条记录
2. 插入成功，说明第一次处理，继续执行业务
3. 插入失败，说明重复请求，直接返回历史结果或忽略

这个方案的关键点不是表结构，而是：

**去重记录和业务操作最好放在同一个事务里。**

否则就会出现：

- 去重表写成功了
- 业务真正的状态却没改成功

后续请求再进来，就会被误判成“已经处理过”。

## 4. 状态机

只靠“有没有请求过”很多时候不够，还要看状态能不能重复推进。

比如订单支付：

```text
UNPAID -> PAID -> REFUNDED
```

更稳的写法是：

- 只有 `UNPAID -> PAID` 合法
- 已经是 `PAID` 的订单，再次支付请求只能返回历史结果

也就是说，状态机是在回答：

**这次重复请求就算又进来了，当前状态还允不允许它继续做事。**

这在资金、库存、发券、审核流里特别重要。

## 5. 乐观锁 / 版本号

它更适合更新类操作。

典型 SQL：

```sql
update t_order
set status = 'PAID', version = version + 1
where order_no = ? and status = 'UNPAID' and version = ?;
```

如果更新行数为 0，说明状态已经变了，或者版本不对，此时就应该回头查当前状态，而不是无脑再改一次。

乐观锁的重点不是“用了 version 字段就高级”，而是：

- 冲突后怎么处理
- 是返回冲突，还是查询最终结果

## 6. 分布式锁

分布式锁经常被当成幂等标准答案，但它其实只是辅助工具。

它能解决的是：

**同一资源在同一时刻只让一个请求进临界区。**

但它解决不了：

- 这笔业务之前到底有没有做过
- 历史状态是不是已经推进过

所以更稳的说法应该是：

**锁可以做串行化保护，但不能替代业务唯一键、状态机和持久化幂等记录。**

是连着的。

## 一个很实用的选择顺序

实际做题或做设计时，可以按这个顺序想：

1. 能不能给这笔业务一个稳定的唯一标识
2. 能不能直接用数据库唯一约束兜底
3. 状态是不是只能单向推进
4. 是否需要单独的去重表记录处理结果
5. 只有在并发竞争特别明显时，再补锁

这比上来就说“用 Redis 锁”更稳。

## 什么时候只靠前端防重复肯定不够

前端按钮置灰能减少重复点击，但挡不住这些情况：

- 用户刷新页面
- 客户端重放请求
- 网关自动重试
- 第三方系统重复通知
- 后端任务重复消费

所以前端优化可以有，但幂等边界必须在服务端。

## 容易踩的坑

### 把锁当成幂等本身

锁只能限制并发，不代表历史上没执行过。

### 第二次请求只返回“重复提交”，却拿不到第一次结果

对用户来说，这种体验并不一定好。很多场景更合理的是：

- 返回第一次的业务结果
- 或者返回“处理中”，再给查询接口

### 没有状态机，导致重复推进

比如订单已经 `PAID`，回调再来一次又重复发货，这本质上是状态校验没做好。

## 小结

- 幂等保护的是业务状态，不只是防用户重复点击。
- 去重负责识别“同一笔请求”，幂等负责保证“重复了也只正确执行一次”。
- 创建类优先考虑业务唯一键和唯一索引，更新类更要配状态机和版本控制。
- 去重表要和业务操作放在同一个事务里，不然很容易留下假成功记录。
- 分布式锁只能做串行化保护，不能替代业务幂等设计本身。
