# 接口幂等怎么设计？

> 幂等真正要解决的不是“用户会不会连点两次”，而是同一笔业务意图重复到达后，系统会不会把状态改乱。

幂等题几乎一定会被问到，但很多回答会停在一句：

> “前端防重复提交，后端加分布式锁。”

这太浅了。因为重复请求的来源远不止用户连点：

- 客户端超时后自动重试
- 网关或 SDK 自动重试
- 第三方回调重复发送
- MQ 至少一次投递
- 人工补偿脚本重复执行

所以幂等不是页面体验优化，而是系统状态保护。

## 先把两个概念分开：去重不等于幂等

这两个词经常混着用，但最好分开讲。

| 概念 | 重点                                           |
| ---- | ---------------------------------------------- |
| 去重 | 识别“这是不是同一笔请求”                       |
| 幂等 | 就算同一笔请求来多次，最终状态也只正确变化一次 |

去重更像“识别重复”，幂等更像“重复了也不能出事”。

所以很多稳妥方案其实是组合拳：

- 业务唯一键识别同一笔请求
- 数据库唯一约束或去重表兜底
- 状态机保证状态不会重复推进

## 先设计幂等键，而不是先选技术手段

幂等方案的第一步不是选 Redis、锁还是数据库，而是先回答：

**系统怎么识别“这是同一笔业务意图”？**

常见幂等键有这些来源：

| 来源             | 例子                          | 适合场景                     |
| ---------------- | ----------------------------- | ---------------------------- |
| 客户端生成       | `Idempotency-Key`、请求流水号 | 开放 API、创建订单、表单提交 |
| 服务端预生成     | 表单 token、下单 token        | 防重复提交、一次性操作       |
| 业务天然唯一标识 | 订单号、支付流水号、消息键    | 订单、支付、MQ 消费          |
| 外部系统流水     | 第三方交易号、回调流水号      | 支付回调、外部通知           |

幂等键设计时至少要把这几个问题说清楚：

1. **谁生成**：客户端生成还是服务端生成，能不能被伪造。
2. **作用范围**：按用户、商户、接口、订单还是全局唯一。
3. **保存多久**：保存太短挡不住晚到的重试，保存太久会占存储。
4. **保存什么**：只保存“处理过”，还是保存处理状态和返回结果。
5. **重复时怎么返回**：返回第一次结果、返回处理中，还是返回明确冲突。

比如创建订单接口，如果只拿用户 ID 当幂等键，就会把同一个用户的两次正常下单误判成重复。
更合理的是用“用户 ID + 客户端请求号”或服务端下发的一次性下单 token。

## 幂等先按操作类型拆，不同类型做法不一样

### 查询类

查询天然更容易做幂等，因为不改状态。

同样查 10 次订单详情，结果应该一样，通常不需要额外设计。

### 创建类

创建类最怕重复插入。

典型例子：

- 重复创建订单
- 重复创建支付流水
- 重复发券

这类操作最常用的手段是：

- 业务唯一单号
- 请求幂等号
- 数据库唯一索引

### 更新类

更新类要分两种：

| 更新方式    | 幂等难度   |
| ----------- | ---------- |
| 赋值更新    | 更容易幂等 |
| 累加 / 扣减 | 更容易出事 |

比如：

- `status = PAID` 更容易做幂等
- `balance = balance - 100` 一旦重复执行就直接多扣一次

所以更新类通常要配状态机、版本号、业务校验一起看。

### 删除类

删除类也要看业务语义。

如果是“把资源标记为已删除”，同一个删除请求重复来，最终状态仍然是已删除，通常可以视为幂等。
但如果删除动作还会触发退款、释放库存、发通知，就不能只看资源状态，还要保证这些副作用不会重复执行。

## 设计幂等时，最常用的是这几种手段

### 1. 业务唯一键

这是最优先考虑的方案。

比如：

- 订单号
- 支付流水号
- 请求号
- 消息 ID

如果系统能明确知道“这两次请求本来就是同一笔业务”，后续判断就有抓手。

一句很实用的话：

**能靠业务唯一键识别同一笔业务时，优先别上锁。**

但业务唯一键不是随便拼一个字符串。它必须满足：

- 稳定：重试时不会变化。
- 精确：不会把两笔正常业务误判成同一笔。
- 可追踪：日志、表字段、监控里能查到。
- 可校验：外部传入的幂等键不能越权复用别人的业务。

### 2. 数据库唯一索引

这对创建类操作特别稳。

比如创建订单时，对 `request_id` 或 `order_no` 建唯一索引：

```sql
create unique index uk_order_request on t_order(request_id);
```

这样即使并发进来两次，数据库层也只会让一条成功。

它的价值在于：

- 简单
- 强约束
- 能兜住并发竞态

但要注意一点：

唯一索引只能兜住“不能插两条”，不等于完整幂等流程就结束了。
你还得决定：

- 第二次请求该返回什么
- 如何查到第一次执行的结果

处理唯一索引冲突时，不能简单地 `catch duplicate key` 然后直接返回成功。
更稳的流程通常是：

```text
尝试插入业务记录
  ├─ 插入成功：继续执行业务并返回结果
  └─ 唯一冲突：按幂等键查询已有记录
       ├─ 已成功：返回第一次处理结果
       ├─ 处理中：返回处理中或让调用方稍后查询
       └─ 已失败：按业务规则决定是否允许重试
```

这也是为什么很多表会额外保存 `request_id`、`status`、`result_snapshot`、`created_at` 这类字段。
唯一索引负责兜住并发，状态和结果负责回答重复请求怎么返回。

### 3. 去重表

去重表适合“业务表不方便直接加唯一约束”或者“想把幂等记录单独管理”的场景。

常见思路是：

1. 用业务唯一号往去重表插一条记录
2. 插入成功，说明第一次处理，继续执行业务
3. 插入失败，说明重复请求，直接返回历史结果或忽略

这个方案的关键点不是表结构，而是：

**去重记录和业务操作最好放在同一个事务里。**

否则就会出现：

- 去重表写成功了
- 业务真正的状态却没改成功

后续请求再进来，就会被误判成“已经处理过”。

去重表不要只存一个 key，最好能表达处理生命周期：

| 字段              | 作用                                 |
| ----------------- | ------------------------------------ |
| `idempotency_key` | 标识同一笔业务意图                   |
| `biz_type`        | 区分下单、支付、发券、消息消费等场景 |
| `biz_id`          | 关联订单号、支付流水号、消息键       |
| `status`          | `PROCESSING`、`SUCCESS`、`FAILED`    |
| `result_snapshot` | 保存第一次成功后的关键返回结果       |
| `expire_at`       | 控制幂等记录保留时间                 |

这样重复请求进来时，不只是知道“来过”，还能知道它处于处理中、已成功还是已失败。

### 4. 状态机

只靠“有没有请求过”很多时候不够，还要看状态能不能重复推进。

比如订单支付：

```text
UNPAID -> PAID -> REFUNDED
```

更稳的写法是：

- 只有 `UNPAID -> PAID` 合法
- 已经是 `PAID` 的订单，再次支付请求只能返回历史结果

也就是说，状态机是在回答：

**这次重复请求就算又进来了，当前状态还允不允许它继续做事。**

这在资金、库存、发券、审核流里特别重要。

状态机最好写成数据库条件，而不是只在内存里判断：

```sql
update t_order
set status = 'PAID'
where order_no = ? and status = 'UNPAID';
```

如果更新行数为 1，说明这次状态推进成功。
如果更新行数为 0，再查询当前状态：已经是 `PAID` 就返回历史结果，其他状态则按业务规则处理。

### 5. 乐观锁 / 版本号

它更适合更新类操作。

典型 SQL：

```sql
update t_order
set status = 'PAID', version = version + 1
where order_no = ? and status = 'UNPAID' and version = ?;
```

如果更新行数为 0，说明状态已经变了，或者版本不对，此时就应该回头查当前状态，而不是无脑再改一次。

乐观锁的重点不是“用了 version 字段就高级”，而是：

- 冲突后怎么处理
- 是返回冲突，还是查询最终结果

### 6. 分布式锁

分布式锁经常被当成幂等标准答案，但它其实只是辅助工具。

它能解决的是：

**同一资源在同一时刻只让一个请求进临界区。**

但它解决不了：

- 这笔业务之前到底有没有做过
- 历史状态是不是已经推进过

所以更稳的说法应该是：

**锁可以做串行化保护，但不能替代业务唯一键、状态机和持久化幂等记录。**

这点和本站的
[`分布式锁有哪些实现？Redis 锁的边界在哪里？`](../distributed-system/distributed-lock-implementations.html)
是连着的。

拿到锁之后也不能马上执行业务，正确顺序更像这样：

1. 按业务唯一键加锁。
2. 进入临界区后查询业务状态或幂等记录。
3. 未处理才执行业务。
4. 已处理就返回历史结果。
5. 最后释放锁，并确保不会释放别人的锁。

如果少了第 2 步，锁只能防住“同时执行”，防不住“历史上已经执行过”。

### 7. Token 机制

Token 机制适合防表单重复提交、一次性确认这类场景。

常见流程是：

1. 页面或客户端先请求服务端生成 token。
2. 服务端把 token 写入 Redis 或幂等表，并设置过期时间。
3. 真正提交时携带 token。
4. 服务端原子消费 token，消费成功才继续执行业务。
5. 重复提交因为 token 已被消费，会被拒绝或引导重新获取。

这里最容易错的是先查 token 再删 token。两个请求并发进来时，都可能先查到 token 存在。
所以消费 token 要做成原子操作：支持原子取删时直接用原子取删，不支持时用 Lua 脚本或数据库唯一约束兜底。

Token 也不是万能方案：

- token 过期太短，慢请求重试可能拿不到原 token。
- token 过期太长，存储压力和重放窗口都会变大。
- token 已消费但业务事务失败时，要告诉用户重新发起，不能误判成功。
- 资金、库存这类高风险场景不能只靠 token，还要有业务唯一键、状态机和流水兜底。

## 重复请求到底该返回什么

幂等设计经常漏掉“重复请求怎么返回”。这会直接影响用户体验和上游重试策略。

| 当前状态       | 推荐返回                                  |
| -------------- | ----------------------------------------- |
| 第一次处理中   | 返回处理中，或者返回可查询的业务单号      |
| 第一次已成功   | 返回第一次成功结果，或返回当前业务状态    |
| 第一次已失败   | 看失败类型，允许重试或返回明确失败原因    |
| 幂等键冲突     | 返回冲突，提示同一个 key 不能用于不同请求 |
| 业务状态已推进 | 返回已有状态，不要重复执行副作用          |

对外部系统来说，“重复请求返回成功”通常比“重复请求返回重复提交错误”更友好，
前提是你能确认第一次确实已经成功处理。

## 一个很实用的选择顺序

实际做题或做设计时，可以按这个顺序想：

1. 能不能给这笔业务一个稳定的唯一标识
2. 能不能直接用数据库唯一约束兜底
3. 状态是不是只能单向推进
4. 是否需要单独的去重表记录处理结果
5. 只有在并发竞争特别明显时，再补锁

这比上来就说“用 Redis 锁”更稳。

## 事务边界是幂等能不能成立的关键

很多幂等事故不是没做去重，而是去重记录和业务状态不在同一个一致性边界里。

最稳的是同库同事务：

```text
begin
  插入幂等记录/消费记录
  更新业务状态
  写业务流水
commit
```

事务提交成功后，重复请求才能根据幂等记录返回历史结果。
如果事务回滚，幂等记录也要回滚，后续请求才能重新处理。

如果幂等记录在 Redis，业务状态在 MySQL，就要承认它们不是天然原子：

- Redis 写成功、MySQL 失败，会误判已经处理。
- MySQL 成功、Redis 失败，重复请求可能再次进入。
- Redis 过期太早，晚到重试可能穿透幂等保护。

这类场景要么把关键幂等记录落到业务库，要么设计补偿、对账和状态机兜底。
幂等不是单个中间件能力，而是一条完整的一致性链路。

## 什么时候只靠前端防重复肯定不够

前端按钮置灰能减少重复点击，但挡不住这些情况：

- 用户刷新页面
- 客户端重放请求
- 网关自动重试
- 第三方系统重复通知
- 后端任务重复消费

所以前端优化可以有，但幂等边界必须在服务端。

## 如何验证幂等真的生效

幂等不能只靠读代码判断，最好用重复请求和异常路径压一遍。

可以重点测这几类：

1. 同一个幂等键并发请求 50 次，只能有一次真正执行业务副作用。
2. 第一次请求处理很慢，客户端超时后重试，最终只能生成一笔业务。
3. 业务执行到一半抛异常，幂等记录和业务状态不能留下“假成功”。
4. 唯一索引冲突后，第二次请求能查到第一次结果或明确处理中。
5. 日志里能按幂等键、订单号、请求 ID 追到完整处理链路。

验证结束后还要查数据：订单表、流水表、库存表、去重表的数量和状态必须对得上。

## 容易踩的坑

### 把锁当成幂等本身

锁只能限制并发，不代表历史上没执行过。

### 第二次请求只返回“重复提交”，却拿不到第一次结果

对用户来说，这种体验并不一定好。很多场景更合理的是：

- 返回第一次的业务结果
- 或者返回“处理中”，再给查询接口

### 没有状态机，导致重复推进

比如订单已经 `PAID`，回调再来一次又重复发货，这本质上是状态校验没做好。

### Token 消费和业务处理割裂

Token 被消费了，但业务事务失败了，如果没有重新获取、状态查询或补偿路径，用户会卡在“不知道到底成没成功”的状态。

### 幂等记录过期太早

有些外部系统可能隔几分钟甚至更久才重试。幂等记录过早清理，重复请求就会被当成新请求处理。

## 小结

- 幂等保护的是业务状态，不只是防用户重复点击；先设计清楚幂等键的生成方、作用范围、保存时长和重复返回策略。
- 创建类优先考虑业务唯一键和唯一索引，更新类更要配状态机和版本控制。
- 去重表、唯一索引、业务状态和流水最好在同一个事务边界里闭环。
- 分布式锁只能做串行化保护，不能替代业务幂等设计本身。
- 幂等验证要覆盖并发重复、超时重试、事务回滚和历史结果返回。
