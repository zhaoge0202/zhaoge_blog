# 后端该埋哪些 RED/USE 指标？

> 监控埋点最容易踩的坑，不是“埋少了”，而是“埋了一堆但报警的时候没人知道该看哪个”。

很多人一说监控指标，第一反应是把 CPU、内存、QPS、延迟全部堆上看板，看着挺全，真出故障的时候反而不知道先看哪个。

其实后端指标可以先分成两大类：

- 请求驱动的服务，先看 `RED`
- 资源类组件，先看 `USE`

这两套指标解决的是“系统是不是健康”。但系统健康不代表业务是对的，所以还得再叠一层业务指标，比如下单成功率、支付成功率。

## RED 和 USE 到底在看什么

先给一个总览，别一上来就死记单词：

| 方法  | 全称                            | 更适合看什么                     | 回答的问题               |
| ----- | ------------------------------- | -------------------------------- | ------------------------ |
| `RED` | Rate、Error、Duration           | 请求驱动的服务，比如接口、RPC    | 这个服务对外表现好不好   |
| `USE` | Utilization、Saturation、Errors | 资源类组件，比如 CPU、连接池、MQ | 这个资源是不是快扛不住了 |

一句话记法：

- **RED 看的是“服务对外的结果”**，站在调用方视角。
- **USE 看的是“资源自己的状态”**，站在资源本身视角。

一个下单接口用 RED 来看，它背后依赖的数据库连接池、线程池、Redis、Kafka 消费者用 USE 来看，两套指标合起来才是一条链路的全貌。

## RED：请求驱动服务的三个指标

RED 对应三个字母：

| 字母       | 含义                   | 常见表达方式                       |
| ---------- | ---------------------- | ---------------------------------- |
| `Rate`     | 单位时间处理了多少请求 | QPS、TPS                           |
| `Error`    | 处理失败的比例         | 5xx 比例、业务错误码比例、超时比例 |
| `Duration` | 处理一个请求花了多久   | P50/P90/P99 延迟、平均耗时         |

拿下单接口举个例子：

| 指标       | 具体数值示例                 | 说明什么                       |
| ---------- | ---------------------------- | ------------------------------ |
| `Rate`     | 800 QPS                      | 当前流量水位                   |
| `Error`    | 5xx 占比 0.3%，超时占比 0.1% | 请求失败的比例和类型           |
| `Duration` | P50 = 40ms，P99 = 320ms      | 大多数请求快不快，长尾严不严重 |

RED 好用的地方在于：它天然对齐用户体感。用户不关心你 CPU 多高，只关心“请求快不快、成不成功”，RED 三个指标基本就是这两句话的量化版本。

埋点上通常按下面这几个维度切：

- 按接口 / 方法维度
- 按调用方维度（比如哪个上游服务调的）
- 按状态码 / 错误类型维度
- 按机房 / 集群维度

## USE：资源类组件的三个指标

USE 对应的三个字母换了一套视角：

| 字母          | 含义                  | 常见表达方式                              |
| ------------- | --------------------- | ----------------------------------------- |
| `Utilization` | 资源被占用的比例      | CPU 使用率、连接池占用率、磁盘使用率      |
| `Saturation`  | 资源排队 / 积压的程度 | 线程池队列长度、GC 停顿、消费者堆积消息数 |
| `Errors`      | 资源自身报错          | 连接失败次数、拒绝连接数、磁盘 IO 错误    |

拿数据库连接池举例：

| 指标          | 具体数值示例                 | 说明什么                   |
| ------------- | ---------------------------- | -------------------------- |
| `Utilization` | 连接池占用 85/100            | 快接近上限了               |
| `Saturation`  | 等待获取连接的请求排队 200ms | 已经开始有人在排队         |
| `Errors`      | 获取连接超时 12 次/分钟      | 已经有请求真的拿不到连接了 |

USE 的价值是它能在“用户还没感觉到明显变慢”之前，先看到资源已经吃紧。很多故障的顺序是：资源先 `Utilization` 抬高 → 再 `Saturation` 开始排队 → 最后 `Errors` 爆发 → 这时候 RED 里的 `Error` 和 `Duration` 才会明显恶化。

所以 USE 更像“早期预警”，RED 更像“最终结果”。

## RED 和 USE 怎么配合看一条链路

一条下单链路可以这样叠起来看：

```text
用户请求
  ↓
下单接口（RED：QPS / 错误率 / P99）
  ↓
数据库连接池（USE：占用率 / 排队 / 获取失败）
  ↓
库存服务调用（RED：QPS / 错误率 / P99）
  ↓
MQ 发送扣减消息（USE：生产者堆积 / Broker 磁盘水位 / 发送失败）
```

排障时一个很实用的顺序是：

1. 先看 RED，确认“用户到底受没受影响”，以及影响出现在哪个接口。
2. 再顺着调用链，看这个接口依赖的资源 USE 指标，找哪个资源先撑不住。
3. 资源指标定位到具体瓶颈后，再回头看这个资源背后的更细粒度 RED（比如它调下游的成功率和延迟）。

这也是为什么单独讲 RED 或单独讲 USE 都不完整：**RED 告诉你“坏没坏”，USE 告诉你“为什么坏”。**

## 光看系统健康不够，还要看业务对不对

RED 全绿、USE 也全绿，不代表业务是对的。

举个例子：下单接口 P99 是 40ms，错误率是 0，但因为一个逻辑 bug，所有请求其实都没真正扣减库存成功，只是接口返回了 200。RED 完全看不出这个问题。

所以还要再加一层业务指标：

| 业务指标   | 怎么算                              | 关注什么                 |
| ---------- | ----------------------------------- | ------------------------ |
| 下单成功率 | 下单成功数 / 下单请求总数           | 用户真的下成单了没有     |
| 支付成功率 | 支付成功数 / 发起支付总数           | 钱有没有真的走通         |
| 库存失败率 | 扣减失败数 / 扣减请求总数           | 库存链路是不是在稳定丢单 |
| 回调到账率 | 收到第三方回调并处理成功数 / 发起数 | 异步链路有没有卡住       |

业务指标和 RED/USE 的关系可以这样理解：

```text
USE：资源健不健康
  ↓
RED：接口对外表现好不好
  ↓
业务指标：这件事到底办成了没有
```

三层缺一不可。资源和接口都正常，业务结果不对，说明问题出在逻辑或状态机上；业务指标正常但 RED 已经在恶化，说明可能是靠重试硬扛出来的，迟早会崩。

## 告警怎么设：症状优先于原因

指标埋够了，接下来最容易埋坑的是告警。常见错误是给每一个指标都配一条阈值告警，结果半夜一个批量任务波动，几十条告警一起炸，没人知道先看哪个。

比较实用的原则是：**先对症状告警，再对原因告警**，让原因告警作为排查线索，而不是叫醒人的第一信号。

| 类型     | 例子                                                 | 适合用来做什么                         |
| -------- | ---------------------------------------------------- | -------------------------------------- |
| 症状告警 | 下单接口错误率 > 1%、P99 > 800ms、下单成功率跌破 99% | 直接反映用户和业务受没受影响，优先叫人 |
| 原因告警 | 连接池占用 > 90%、GC 停顿变长、MQ 堆积上涨           | 帮助排障定位，但不一定代表用户已受影响 |

拿前面下单链路举例：

- 数据库连接池占用率涨到 90%，但下单接口成功率和延迟还没明显恶化 → 可以只记录、观察，不一定要立刻报警叫人。
- 下单接口错误率已经涨到 2%，或者下单成功率跌破 99% → 这个必须立刻报警，因为用户已经在感知到问题了。

避免阈值噪声，还有几个常见做法：

- 同一个指标不要配多条重叠阈值，避免同一次异常触发好几条告警。
- 短暂抖动加持续时长条件，比如“连续 3 分钟错误率超过阈值”而不是“单次采样点超过阈值”。
- 把强相关的指标合并成一条告警，而不是每个维度单独一条。
- 原因类告警默认走静默通道或工单，只有症状类告警才直接电话 / IM 叫人。
- 定期清理没人处理、长期不痛不痒的告警规则，噪声告警多了大家会习惯性忽略，真正故障来了也不当回事。

一句话总结：**告警是给人看的，不是给仪表盘凑数的，先保证症状告警足够准，原因告警才有意义。**

## 埋点常见的坑

### 高基数标签

Prometheus 之类的时序库，每多一个标签维度，序列数量是乘法增长的。

比如给接口指标加上 `user_id` 这种标签，看起来能查得很细，但用户量一大，序列数会爆炸，直接把监控系统自己拖垮，甚至查询超时、写入丢数据。

常见处理办法：

| 场景             | 更稳妥的做法                                    |
| ---------------- | ----------------------------------------------- |
| 想按用户排查问题 | 不打进指标标签，走日志 / Trace 里带 `user_id`   |
| 想按商品统计热点 | 只对 Top N 热点商品单独打标签，其余归到 `other` |
| 想按接口区分     | 用接口名 / 方法名这种有限集合的维度打标签       |

一个基本判断标准：**这个标签的取值集合是不是有限、可枚举的**。用户 ID、订单号、商品 ID 这类无限增长的字段，不适合直接做指标标签。

### P99 被采样悄悄改了含义

延迟分位值很依赖采样方式，两个常见坑：

- 客户端做了采样上报，比如只上报 1% 的请求，这时候算出来的 P99 其实已经不是真实的 P99，长尾很容易被稀释掉。
- 多实例的 P99 不能直接平均。每个实例各自算的 P99，简单取平均没有意义，正确做法是用直方图（Histogram）做全局聚合后再算分位值。

所以看 P99 之前先确认两件事：这个 P99 是不是全量统计出来的，是不是跨实例正确聚合出来的。否则很容易出现“监控上 P99 一直很稳，用户却在抱怨慢”的割裂现象。

## 和 SLA、限流怎么串起来

这篇讲的是“埋什么指标、怎么告警”，但指标最终要服务于两件事：

- 指标撑起 [[SLA、几个 9、RTO、RPO 到底是什么意思？|SLA/SLO 目标]]。`SLI` 选的就是这里说的 RED、USE 和业务指标的一部分；`SLO` 定的阈值也要落到这些指标上才能被观测和验证。
- 指标反过来指导 [[限流算法怎么选：固定窗口、滑动窗口、漏桶、令牌桶？|限流阈值]] 的制定。限流阈值不能拍脑袋，正是要看 RED 里的 `Duration` 什么时候开始明显抬头、USE 里的连接池 / 线程池什么时候开始排队，再往前倒推留出安全余量。

可以理解成一条链：**埋点（RED/USE/业务指标）→ 目标（SLI/SLO/SLA）→ 保护手段（限流、熔断、降级）**，三篇内容其实在讲同一件事的三个环节。

## 一个比较实用的回答顺序

系统设计或排障题里被问到监控怎么建，可以按这个顺序说：

1. 先分请求驱动服务和资源类组件，分别套 RED 和 USE。
2. 再补一层业务指标，确认“系统健康”不等于“业务正确”。
3. 告警按症状优先、原因辅助的原则设计，避免每个指标都单独告警。
4. 埋点时提前规避高基数标签，延迟指标确认好统计口径和聚合方式。
5. 最后把这些指标和 SLO、限流阈值挂钩，让监控真正能驱动决策，而不是只是好看的仪表盘。

## 容易答错的点

### 把 RED 和 USE 用反对象

给数据库连接池套 RED，给接口套 USE，逻辑上讲得通但不是最优选择。请求驱动的东西看结果用 RED 更直接，资源类的东西看饱和度用 USE 更早发现问题，两者搭配比互换更高效。

### 只埋指标，不设告警优先级

指标全埋了，但告警不分症状和原因，出故障时一堆告警一起炸，反而拖慢定位速度。

### 延迟只看平均值

平均延迟很容易被大多数快请求拉低，掩盖掉长尾。P99、P999 才更接近真实的糟糕体验，尤其是线程池、连接池已经开始排队的阶段。

### 标签维度想怎么打就怎么打

高基数标签是监控系统最常见的自杀式用法，看着好用，规模上来后先把监控自己拖垮。

## 小结

1. `RED` 看请求驱动服务的对外表现，`USE` 看资源类组件的健康状态，两者搭配才能看清一条链路。
2. 系统健康不等于业务正确，还要再叠一层下单成功率、支付成功率这类业务指标。
3. 告警要症状优先于原因，原因类指标更适合做排查线索而不是直接叫醒人，并且要控制阈值噪声。
4. 埋点最容易踩的坑是高基数标签和 P99 采样口径，前者拖垮监控系统，后者让分位值失真。
5. 这些指标最终要落到 SLI/SLO 目标和限流阈值上，监控才不是只好看，而是真的能驱动决策。
