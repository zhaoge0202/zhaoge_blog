# 压测应该测什么？容量评估怎么做？

> 高可用设计如果不回到压测、容量和故障验证，很多时候只停留在“感觉应该能扛”。

很多系统设计题最后都会落到一句：

> “我们上线前会做压测。”

这句话本身没问题，但太空了。真正有价值的是继续往下说：

- 测哪些指标
- 用什么流量模型
- 怎么估算容量
- 怎么判断系统已经到拐点

因为压测不是为了跑一个好看的 `QPS` 数字，而是为了验证系统在目标流量下还能不能稳定工作。

这篇和前面的限流、熔断、容灾是连着的：限流阈值、线程池大小、连接池大小、容灾切流容量，
最后都要靠压测和演练给证据。

## 先区分 4 种常见测试，不然很容易测偏

| 类型       | 核心目标                           |
| ---------- | ---------------------------------- |
| 性能测试   | 在目标指标下验证系统是否达标       |
| 负载测试   | 逐步加压，找到稳定上限             |
| 压力测试   | 继续越线，观察系统失稳和自愈行为   |
| 稳定性测试 | 长时间运行，暴露泄漏和资源积累问题 |

这四种测试经常被统称成“压测”，但它们关注点不同。

比如：

- 你要验证大促前是否能扛 5000 QPS，更像性能测试 / 负载测试
- 你要看限流和熔断会不会生效，更像压力测试
- 你要看 12 小时后连接池会不会漏光，更像稳定性测试

还可以把它们放在一条水位线上理解：

```text
目标水位      饱和水位         失稳水位
  │             │                │
性能测试 ── 负载测试 ───── 压力测试
  │
稳定性测试：在目标或接近目标水位长时间运行
```

所以压测计划里要写清楚：这次到底是验收目标、找上限，还是故意越线验证保护动作。

## 压测最该盯的指标，不是只有 QPS

至少要同时看这几类：

| 指标                     | 关注点               |
| ------------------------ | -------------------- |
| RT / P99 / P999          | 用户体感和长尾延迟   |
| QPS / TPS                | 吞吐能力             |
| 成功率                   | 错误是否开始放大     |
| 并发数                   | 当前系统负载水平     |
| CPU / 内存 / GC          | 资源是不是被顶满     |
| 线程池 / 连接池 / 队列   | 有没有局部资源先耗尽 |
| 数据库 / Redis / MQ 指标 | 真正瓶颈是不是在下游 |

平均 RT 经常最会骗人。

一个系统可能平均只要 80ms，但 P99 已经 1.5s 了。
对用户来说，真正难受的往往就是这 1% 的长尾请求。

指标最好分三层看：

| 层次     | 关键指标                                         | 说明                       |
| -------- | ------------------------------------------------ | -------------------------- |
| 用户体验 | P95 / P99 / P999、成功率、超时率                 | 用户是否能正常完成操作     |
| 应用资源 | CPU、内存、GC、线程池、连接池、队列长度          | 应用是不是快到资源上限     |
| 依赖资源 | DB 慢 SQL、锁等待、Redis RT、MQ 积压、RPC 错误率 | 瓶颈是不是转移到下游依赖   |
| 保护动作 | 限流命中、熔断状态、降级命中、重试比例、拒绝数量 | 过载时是否按预期止血       |
| 业务结果 | 下单成功率、支付成功率、回调成功率、发货成功率   | 技术指标好看不等于业务成功 |

如果只看 QPS，很容易把“系统还能收请求”误判成“业务还能成功”。

## 为什么容量评估不能只靠“拍个日活系数”

很多粗略估算会这么做：

- `DAU × 某个经验系数 = 峰值 QPS`

这可以拿来做第一版粗估，但不能直接拿来当上线结论。

因为真正的峰值往往来自具体业务场景，比如：

- 整点开抢
- 秒杀开始
- 支付回调集中到达
- 大促某个会场流量爆发

所以容量评估更像一个分步骤过程。

## 一个更靠谱的容量评估顺序

### 第一步：先定业务场景

先别算，先定最重要的流量模型：

- 是读多写少，还是写多读多
- 峰值是不是出现在整点
- 哪条链路最关键
- 下游依赖有哪些

如果你测的是推荐接口，但真正的大促瓶颈在下单链路，那这个压测价值就很有限。

### 第二步：给出目标值

比如：

- 峰值 5000 QPS
- P99 小于 300ms
- 成功率不低于 99.95%
- 错误预算在活动窗口内不能被烧穿
- 核心下游 CPU、连接池、队列水位不超过安全阈值

没有目标值，压测就很容易变成“压到哪算哪”。

目标值最好分成“必须达标”和“观察项”：

| 类型     | 例子                                          |
| -------- | --------------------------------------------- |
| 必须达标 | 目标 QPS、P99、成功率、核心业务成功率         |
| 观察项   | CPU 水位、GC 次数、慢 SQL 数、MQ 积压、重试量 |
| 保护项   | 超过阈值后限流是否生效、是否返回明确错误码    |

这样压测结论才不是一句“压到了 5000 QPS”，而是“在 5000 QPS 下核心链路仍满足 SLO”。

### 第三步：用历史数据做初始估算

可以结合：

- 历史峰值
- 活动预期放大倍数
- 业务增长率
- 安全冗余

得到一个初始压测目标。

可以按这条线粗算：

```text
日均请求量
  -> 日均 QPS
  -> 峰值系数
  -> 活动放大倍数
  -> 安全冗余
  -> 压测目标 QPS
```

比如某接口日均 8640 万次请求：

```text
日均 QPS = 86400000 / 86400 = 1000
峰值系数 = 5
活动放大倍数 = 2
安全冗余 = 1.5
压测目标 ≈ 1000 × 5 × 2 × 1.5 = 15000 QPS
```

这个结果只是“压测目标起点”，不是生产容量结论。

### 第四步：再用 Little's Law 做量级校验

在系统还处在线性区间时，可以用：

```text
并发数 ≈ QPS × RT
```

比如目标 `QPS = 5000`，目标平均 `RT = 100ms = 0.1s`：

```text
并发数 ≈ 5000 × 0.1 = 500
```

这说明系统至少得能稳定承载 500 个在途请求量级。

但这只是校验，不是最终答案。
因为一旦系统接近瓶颈，`RT` 会非线性上升，这个公式就不再好用了。

还要注意这里的 `RT` 最好用目标延迟，而不是压测已经恶化后的延迟。
如果你拿瓶颈附近的高 RT 反推并发数，很容易把排队堆积也算进“正常容量”。

## 压测数据要尽量接近真实世界

很多压测结论偏乐观，不是工具问题，而是数据太干净。

压测数据至少要考虑这些维度：

| 维度     | 为什么重要                                   |
| -------- | -------------------------------------------- |
| 数据量   | 小表和大表的索引、分页、缓存命中完全不同     |
| 热点分布 | 真实流量往往集中在少量商品、商家、用户       |
| 读写比例 | 纯读接口和下单、扣库存、支付回调压力不同     |
| 用户会话 | 登录态、权限、购物车、优惠券会影响链路       |
| 参数分布 | 空参数、极端参数、大对象、深分页都可能变慢   |
| 下游依赖 | 第三方、DB、Redis、MQ 要么参与压测，要么模拟 |

例如秒杀压测如果每个请求都打不同商品，看起来缓存很均匀；真实活动里可能 90% 请求都打同一个商品，
热点 key、库存锁和数据库行锁才是真正瓶颈。

## 压测环境怎么选

压测环境通常有三种：

| 环境         | 优点                     | 风险                         |
| ------------ | ------------------------ | ---------------------------- |
| 测试环境     | 安全，不影响生产         | 配置、数据量、机器规格偏差大 |
| 预发环境     | 更接近生产               | 容量通常仍小于生产           |
| 生产影子压测 | 最接近真实链路和真实容量 | 必须做好隔离、限流和回滚     |

生产影子压测不是直接拿真实用户当实验品。至少要做到：

- 压测流量有明显标识，不能污染真实业务数据。
- 写操作走影子库、影子表或可回滚的隔离链路。
- 第三方支付、短信、物流等外部调用要 mock 或走沙箱。
- 压测期间有实时监控和一键停止开关。
- 明确压测窗口，提前通知值班和相关团队。

## 真正的拐点，通常看这几个信号

系统开始接近极限时，常见现象有：

- QPS 不再明显增长
- P99 / P999 突然陡升
- 线程池队列开始堆积
- 连接池等待时间明显升高
- 数据库慢查询变多
- Redis、MQ、RPC 下游延迟同步升高

这时候就不是“还能不能再多压一点”的问题，而是：

**系统已经离线性工作区不远了。**

更形象地看，系统通常会经历三段：

```text
线性区：并发增加，QPS 近似增加，RT 稳定
拐点区：QPS 增长变慢，P99 开始抬头
失稳区：QPS 不升反降，RT 和错误率快速上升
```

容量评估要拿的是“稳定上限”，不是“曾经打到过的最高峰值”。
如果某次压测瞬间冲到 8000 QPS，但 P99 已经飙升、错误率开始抬头，那它不能直接作为生产限流阈值。

更稳的做法是：

```text
生产限流阈值 = 稳定上限 × 安全系数
```

安全系数常见会按 `70% ~ 80%` 这类保守区间取，具体要结合业务风险、扩容速度和下游配额。

## 压力测试不只是看“什么时候挂”，还要看“挂成什么样”

真正有高可用意识的团队，不会只测崩溃点，还会看：

- 超过阈值后有没有返回 `429` / `503`
- 限流是否生效
- 熔断是否打开
- 队列是否有背压
- 流量回落后系统能不能自动恢复

也就是说，压力测试不仅看“会不会挂”，还看“挂的时候有没有优雅失败”。

这类测试要重点验证：

| 过载场景         | 期待结果                                 |
| ---------------- | ---------------------------------------- |
| 入口流量超过阈值 | 网关或服务限流，返回明确错误码           |
| 下游持续超时     | 调用方熔断，减少继续施压                 |
| 队列堆积         | 有队列长度上限和丢弃/降级策略            |
| MQ 消费落后      | 积压可观测，消费者扩容或限生产生效       |
| 流量回落         | 线程、连接、队列和错误率能恢复到正常水位 |

如果越线后系统假死、连接不释放、线程池长期回不来，即使峰值 QPS 好看，也不算高可用。

## 稳定性测试最常暴露的是积累型问题

这类问题短压 5 分钟经常看不出来，但长压几个小时会暴露：

- 内存泄漏
- 连接泄漏
- 线程数持续上涨
- Full GC 越来越频繁
- MQ 积压慢慢爬高

所以压测如果只看短时间吞吐，很容易漏掉真正的线上风险。

稳定性测试不要只盯开始和结束两个点，而要看趋势：

- 堆内存是否有涨有落，还是一路抬升；
- 老年代占用和 Full GC 是否越来越频繁；
- 线程数、文件句柄、连接数是否持续上涨；
- 连接池活跃数是否长期不释放；
- MQ lag 是否持续扩大；
- 日志量、磁盘写入、临时文件是否持续增长。

这类问题往往不是“瞬间打挂”，而是活动跑几个小时后慢慢把系统拖死。

## 一个很实用的压测清单

如果要把答案讲得更像工程实践，可以按这张清单说：

1. 明确业务场景和目标指标。
2. 用历史流量和活动预估确定压测目标。
3. 构造接近真实的数据量、热点分布和读写比例。
4. 逐步加压，观察 RT、成功率、线程池、连接池、数据库和缓存指标。
5. 超过阈值后继续压，看限流、熔断、降级和恢复是否符合预期。
6. 长时间跑稳定性测试，找泄漏、堆积和慢性资源问题。

这套说下来，面试官基本能感受到你知道压测不是“跑 JMeter 点开始”。

## 压测结束后要给出什么结论

压测报告不能只有截图和峰值数字，至少要回答这些问题：

1. 在目标 QPS 下，P99、成功率、业务成功率是否达标。
2. 稳定上限是多少，拐点在哪个水位出现。
3. 第一个瓶颈资源是什么：CPU、线程池、连接池、DB、Redis、MQ 还是下游。
4. 生产限流阈值、扩容阈值和告警阈值应该怎么设置。
5. 如果流量超过预期，系统会限流、降级、排队还是直接失败。
6. 是否需要扩容、改 SQL、拆热点、加缓存、调整线程池或改架构。

比如一个更有价值的结论是：

```text
目标 5000 QPS 下：
  P99 = 260ms，成功率 = 99.97%，核心业务成功率达标
稳定上限约 6200 QPS：
  超过后数据库连接池等待时间升高，P99 明显抬头
生产建议：
  入口限流设为 4500 ~ 5000 QPS
  DB 连接池等待时间 > 50ms 告警
  大促前读库扩容 2 个实例，并提前预热热点缓存
```

这才是压测对高可用设计真正有用的地方。

## 容易踩的坑

### 只测单接口，不测完整链路

单接口 1 万 QPS 很漂亮，但真实链路里还有：

- 数据库
- Redis
- MQ
- RPC 下游

真正扛不住的往往不是接口本身。

### 用空数据或小数据压

真实线上很多问题来自：

- 大表
- 热点 key
- 深分页
- 大对象序列化

如果压测数据太干净，结果会偏乐观。

### 只看平均值，不看长尾

平均 RT 好看，不能代表高峰期的真实体感。

### 只看吞吐，不看保护动作

高可用不是单纯追 QPS，还要验证限流、熔断、降级、自愈这些策略是不是按预期工作。

### 压测流量污染真实数据

下单、支付、发券、发消息这类链路必须有压测标识和隔离方案。
否则压测可能制造真实订单、真实扣款、真实短信，后果比不压测更糟。

### 把短时间峰值当长期容量

系统撑住 3 分钟和稳定跑 3 小时不是一回事。
连接泄漏、GC、MQ 积压、磁盘写满这类问题都需要更长时间才能暴露。

### 压测后没有更新限流和告警

如果压测发现稳定上限是 5000 QPS，但生产限流仍然设到 8000，压测结论就没有真正落地。

## 小结

- 压测不是为了跑一个漂亮数字，而是为了验证系统在目标流量下是否还能稳定工作。
- 性能测试、负载测试、压力测试、稳定性测试关注点不同，目标水位、饱和水位和失稳水位要分清。
- 容量评估先看业务场景、历史流量和峰值模型，再用 Little's Law 做量级校验，最后靠全链路压测确认。
- 到达瓶颈时最值得看的不是平均 RT，而是 P99、业务成功率、资源耗尽点和保护动作是否生效。
- 真正的高可用验证，不只看“能跑多快”，还看“过载时会不会优雅失败、回落后能不能恢复”。
