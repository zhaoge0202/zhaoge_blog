# 重试为什么可能放大故障？

> 重试本来是想提高成功率，但如果错误类型、次数和节奏没控好，它经常会从“补救动作”变成“二次打击”。

很多系统事故不是因为第一次请求失败，而是因为失败之后所有人都开始重试。

最典型的过程是：

1. 下游先慢了一点
2. 上游超时开始重试
3. 线程、连接、队列被更多请求占满
4. 故障从一个节点扩散到整条链路

所以重试不是越多越稳，它本质上是在拿更多资源换成功概率。

先把一句话压住：

**重试不是“失败了再请求一次”这么简单，而是对同一业务意图再次发起调用。**

这意味着它一定要同时回答四个问题：

1. 这个失败是不是短暂的？
2. 这个操作重复执行会不会改乱状态？
3. 这次重试会不会超过整条链路的时间预算？
4. 当前下游是否还有能力承接额外流量？

只回答“重试几次”，基本是不够的。

## 为什么重试会把流量放大

看一个简单例子。

如果一个服务在一次调用里允许“首试 + 2 次重试”，那单层最坏就是 3 次请求。

如果你有三层链路，而且每层都这么干：

```text
网关 -> 服务 A -> 服务 B -> 服务 C
```

最坏情况下，到最底层的调用数会变成：

```text
3 × 3 × 3 = 27 次
```

原本只是 1 次业务请求，现在可能变成 27 次打到最脆弱的下游。

这就是为什么很多线上经验都会强调：

**重试最好收敛在一层，而不是层层都开。**

更准确地说，重试要有“归口”。例如：

| 重试位置     | 适合做什么                             | 风险                                   |
| ------------ | -------------------------------------- | -------------------------------------- |
| 客户端 SDK   | 对网络抖动、连接重置做少量自动重试     | 客户端数量多，容易形成外部重试洪峰     |
| 网关         | 对幂等读请求、临时 5xx 做统一策略      | 可能看不到业务语义，容易误重试写操作   |
| 业务服务     | 能结合业务状态、幂等键和下游语义做判断 | 每层都写一套，容易层层放大             |
| MQ 消费者    | 对异步任务做退避重试、死信、人工补偿   | 积压可能扩大，重试节奏要和消费能力匹配 |
| 定时补偿任务 | 对最终一致链路做离线补偿               | 延迟更高，不适合用户同步等待           |

在线链路里通常更推荐把自动重试收敛在**最懂失败语义的一层**。如果 SDK、网关、服务框架和业务代码都默认重试，事故时很容易没人知道真实放大倍数是多少。

## 哪些情况下重试最容易失控

### 超时设置太激进

如果超时阈值比正常 P99 还小一点点，系统一抖就会误判为失败，然后立刻触发大量重试。

结果不是故障恢复，而是把本来还能勉强工作的下游彻底打趴下。

### 所有人用固定间隔一起重试

比如大家都设成：

- 失败后 1 秒重试
- 再失败再 1 秒重试

那下游一恢复，所有客户端会同时重新打过来。

这就是典型的“恢复瞬间二次洪峰”。

### 对不可重试错误也重试

比如：

- 参数错误
- 权限失败
- 余额不足
- 库存不足

这些不是短暂故障，再试十次也没用，只会白白消耗资源。

### 对非幂等写操作盲目重试

像创建订单、扣库存、发券、扣款这类操作，如果没有幂等保护，重试不是“提高成功率”，而是在给重复执行创造机会。

### 没有全链路时间预算

重试会增加尾部耗时。

假设一次读取超时是 `1.5s`，最多重试 2 次，退避间隔是 `200ms`、`500ms`：

```text
最坏耗时 = 1.5s × 3 + 0.2s + 0.5s = 5.2s
```

如果入口接口的 SLO 是 P99 小于 `800ms`，这套重试策略就已经不适合在线等待了。它可能让少量请求成功，但把用户体验、线程占用和上游等待时间都拖垮。

所以重试次数必须和 timeout budget 一起算，不要孤立配置。

## 重试真正适合解决什么问题

它更适合解决这类“瞬时失败”：

- 临时网络抖动
- 偶发连接超时
- 短暂 `502/503/504`
- 可重试的连接重置或临时限流

一句话：

**重试解决的是偶发性失败，不是持续性故障。**

如果下游已经持续变慢、错误率持续升高，就该优先想熔断，而不是继续加重试。

## 怎么判断一个错误能不能重试？

重试前至少按“错误类型 + 操作语义”一起判断。

| 场景                             | 是否适合重试 | 原因                               |
| -------------------------------- | ------------ | ---------------------------------- |
| 连接建立失败、连接重置           | 可以少量重试 | 很可能是瞬时网络或连接问题         |
| 短暂 `502/503/504`               | 可以少量重试 | 可能是临时过载或网关后端切换       |
| `429` 且带明确 `Retry-After`     | 可以按提示等 | 下游已经告诉你什么时候再来         |
| 读请求超时                       | 可以谨慎重试 | 一般副作用少，但仍要控制时间预算   |
| 创建订单、扣款、扣库存           | 谨慎         | 必须有幂等键、状态机或业务唯一约束 |
| 参数错误、鉴权失败、权限不足     | 不该重试     | 再试也不会变好                     |
| 余额不足、库存不足、业务规则失败 | 不该重试     | 这是业务结果，不是系统短暂失败     |
| 下游错误率持续升高               | 不该继续加码 | 应进入熔断、限流、降级或排障       |

一个实用判断是：

```text
能不能重试 = 是否短暂故障 × 操作是否幂等 × 是否还有时间预算 × 下游是否未熔断
```

四个条件缺一个，都要非常谨慎。

## 为什么指数退避和随机抖动很重要

如果必须重试，更稳的做法通常是：

- 次数少
- 间隔越来越长
- 每次间隔再加一点随机抖动

常见策略：

| 策略              | 问题                         |
| ----------------- | ---------------------------- |
| 固定间隔          | 最容易形成同步重试洪峰       |
| 线性退避          | 比固定好一点，但打散能力一般 |
| 指数退避          | 能明显降低重试密度           |
| 指数退避 + Jitter | 分布式场景里最稳妥           |

为什么要加 `Jitter`？

因为如果一万台客户端都按“1 秒、2 秒、4 秒”重试，不加随机数，它们还是会在同一时间点一起打过来。

可以把几种策略放到一条时间线上看：

```text
固定间隔：      1s ---- 1s ---- 1s ---- 1s
线性退避：      1s ------ 2s -------- 3s
指数退避：      1s -------- 2s ---------------- 4s
指数退避+抖动： 0.7s ------ 2.3s ---------- 3.6s   (每个客户端不同)
```

核心不是“指数”这个词，而是两件事：

- **越失败越慢**：给下游恢复时间；
- **不同客户端错开**：避免同时重试。

实际配置还要加上最大间隔和最大总耗时，否则指数退避可能把请求拖到用户早就不等了。

## 线上更实用的控制手段：重试预算

一个很重要但经常被忽略的概念是“重试预算”。

它的意思是：

**在一个时间窗口内，允许重试请求占原始请求的比例不能超过某个上限。**

比如规定：

- 1 分钟内最多允许重试量占原始请求量的 10%

一旦超过预算，后面的请求就不再重试，只允许首试或直接失败。

这样做的好处是：

- 故障时不会无限放大流量
- 能把下游保护在一个可控范围内

落地时可以按这些维度统计预算：

| 维度              | 适合解决什么                        |
| ----------------- | ----------------------------------- |
| 调用方 + 下游服务 | 防止某个调用方把某个下游打爆        |
| 接口维度          | 防止高成本接口被重试放大            |
| 租户 / 用户维度   | 防止单个租户或用户触发大量重试      |
| 错误类型维度      | 观察到底是超时、限流还是 5xx 在触发 |

重试预算不是为了“禁掉所有重试”，而是让重试有上限。预算耗尽后，可以快速失败、走降级、排队异步补偿，或者只允许首试。

## 重试和熔断、限流、隔离怎么配合？

这几个动作的关系可以这样排：

```text
入口限流：先控制进来的总量
  ↓
超时预算：控制每次请求最多等多久
  ↓
隔离：限制某个下游最多占用多少资源
  ↓
有限重试：只处理瞬时失败
  ↓
熔断：持续异常时停止继续请求
  ↓
降级 / 异步补偿：给用户或业务一个可接受结果
```

重试不能绕过熔断器，也不应该把隔离池打满。

比如库存服务已经进入熔断状态，此时继续在业务代码里偷偷重试，就是在破坏熔断策略。更稳的做法是先看熔断和限流状态，再决定是否还允许少量探测或直接失败。

## 在线请求、批处理和 MQ 消费策略不一样

不同链路的重试策略不能照抄。

| 场景       | 策略倾向                       | 关键约束                     |
| ---------- | ------------------------------ | ---------------------------- |
| 在线查询   | 少量、短超时、快速失败         | 用户正在等待，P99 很重要     |
| 在线写操作 | 极少重试或不自动重试           | 幂等、状态机、业务唯一键     |
| 批处理任务 | 可以更长退避、可恢复、可暂停   | 吞吐和完成时间，用户不在线等 |
| MQ 消费    | 退避重试 + 最大次数 + 死信队列 | 防重复消费、积压、顺序影响   |
| 定时补偿   | 慢一点没关系，重点是最终可追踪 | 幂等、审计、人工介入         |

MQ 消费经常可以比在线请求更有耐心，但也不能无限重试。否则同一批坏消息会反复占用消费者，导致正常消息也消费不了。常见做法是：有限次数重试，超过后进死信队列，再靠告警和人工/自动补偿处理。

## 一条常用的工程规则

很多场景里，可以直接记这一套：

1. 在线同步请求，重试尽量控制在 1～2 次。
2. 只对明确可重试的错误重试。
3. 采用指数退避并加随机抖动。
4. 不要每一层都做自动重试。
5. 对写操作先确认幂等，再谈重试。
6. 故障持续时优先熔断，而不是一直试。

这套规则虽然保守，但比“默认重试 3 次”更接近生产。

## 一个很容易说服人的真实场景

比如库存服务抖动：

- 正常 RT 50ms
- 突然抖到 800ms

上游如果把超时设成 200ms，再配 2 次重试，那一次库存查询很可能变成 3 次。

库存服务本来只是“变慢”，结果因为额外流量变成：

- 线程池打满
- Redis 连接占满
- 数据库压力升高

最后连原本能成功的请求也失败了。

这时候真正该做的往往是：

- 缩短排队
- 限制重试
- 熔断部分请求
- 对非核心链路直接降级

而不是“大家一起再试试”。

## 线上排障应该看哪些指标？

重试相关事故排查时，最好先把“原始流量”和“重试流量”拆开。

至少看这些指标：

| 指标                 | 看什么                                   |
| -------------------- | ---------------------------------------- |
| 原始请求量           | 真实用户或上游产生的业务流量             |
| 重试请求量           | 额外放大的流量                           |
| 重试比例             | 是否超过预算，例如重试量 / 原始请求量    |
| 重试成功率           | 重试到底有没有救回来                     |
| 重试后总耗时         | 成功率提升是否以巨大延迟为代价           |
| 触发重试的错误类型   | 是超时、连接失败、5xx、429，还是业务异常 |
| 下游 P99/P999        | 下游是否已经进入长尾延迟区               |
| 线程池 / 连接池队列  | 资源是否被重试占满                       |
| 熔断状态和限流拒绝量 | 保护动作是否已经触发                     |

如果看到“重试成功率很低，但重试比例很高”，通常说明重试已经从补救变成负担。此时优先收缩重试、打开熔断或降级，而不是继续拉高重试次数。

## 面试怎么答？

可以按这条线组织：

1. 先说重试的本质：用额外请求换瞬时故障下的成功率。
2. 再说风险：多层重试会指数级放大流量，固定间隔会形成重试洪峰。
3. 然后讲边界：只重试可重试错误，写操作必须先有幂等和状态机。
4. 接着落策略：少次数、总时间预算、指数退避、随机抖动、最大间隔、重试预算。
5. 最后讲治理：和限流、隔离、熔断联动，监控原始请求量、重试量、重试成功率和下游长尾。

## 容易踩的坑

### 只看成功率，不看成本

重试可能让成功率看起来变高了，但也可能让：

- 延迟更高
- 线程占用更久
- 下游更容易雪崩

所以不能只盯成功率。

### 重试指标没单独拆出来

至少要区分：

- 原始请求量
- 重试请求量
- 重试成功率
- 重试后总耗时
- 重试触发的错误类型

不拆开看，你很难知道到底是下游坏了，还是重试把它打坏了。

### MQ 消费和在线请求用同一套策略

异步消费可以更激进一点，因为它有死信队列、补偿和人工处理空间。

在线同步请求一般要保守得多，因为用户就卡在链路上等结果。

### 框架默认重试没梳理

很多系统里，HTTP 客户端、RPC 框架、SDK、网关和业务代码都可能有默认重试。

上线前要把它们列出来：

- 哪一层会重试；
- 重试几次；
- 是否带退避和抖动；
- 哪些错误会触发；
- 是否会重试写操作；
- 是否有统一监控。

否则你以为只重试 1 次，真实链路可能已经重试了好几层。

## 小结

1. 重试是在拿额外资源换成功率，故障时它也会放大流量。
2. 最危险的组合是：多层都重试、固定间隔重试、对不可重试错误也重试。
3. 重试只适合瞬时失败，不适合持续性故障；持续异常应该交给熔断、限流和降级。
4. 更稳的做法是少次数、总时间预算、指数退避、随机抖动、最大间隔，再加重试预算。
5. 写操作在没确认幂等之前，不要轻易开自动重试；MQ 消费也要有最大次数、死信和补偿。
