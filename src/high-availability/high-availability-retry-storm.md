# 重试为什么可能放大故障？

> 重试本来是想提高成功率，但如果错误类型、次数和节奏没控好，它经常会从“补救动作”变成“二次打击”。

很多系统事故不是因为第一次请求失败，而是因为失败之后所有人都开始重试。

最典型的过程是：

1. 下游先慢了一点
2. 上游超时开始重试
3. 线程、连接、队列被更多请求占满
4. 故障从一个节点扩散到整条链路

所以重试不是越多越稳，它本质上是在拿更多资源换成功概率。

## 为什么重试会把流量放大

看一个简单例子。

如果一个服务在一次调用里允许“首试 + 2 次重试”，那单层最坏就是 3 次请求。

如果你有三层链路，而且每层都这么干：

```text
网关 -> 服务 A -> 服务 B -> 服务 C
```

最坏情况下，到最底层的调用数会变成：

```text
3 × 3 × 3 = 27 次
```

原本只是 1 次业务请求，现在可能变成 27 次打到最脆弱的下游。

这就是为什么很多线上经验都会强调：

**重试最好收敛在一层，而不是层层都开。**

## 哪些情况下重试最容易失控

### 超时设置太激进

如果超时阈值比正常 P99 还小一点点，系统一抖就会误判为失败，然后立刻触发大量重试。

结果不是故障恢复，而是把本来还能勉强工作的下游彻底打趴下。

### 所有人用固定间隔一起重试

比如大家都设成：

- 失败后 1 秒重试
- 再失败再 1 秒重试

那下游一恢复，所有客户端会同时重新打过来。

这就是典型的“恢复瞬间二次洪峰”。

### 对不可重试错误也重试

比如：

- 参数错误
- 权限失败
- 余额不足
- 库存不足

这些不是短暂故障，再试十次也没用，只会白白消耗资源。

### 对非幂等写操作盲目重试

像创建订单、扣库存、发券、扣款这类操作，如果没有幂等保护，重试不是“提高成功率”，而是在给重复执行创造机会。

## 重试真正适合解决什么问题

它更适合解决这类“瞬时失败”：

- 临时网络抖动
- 偶发连接超时
- 短暂 `502/503/504`
- 可重试的连接重置或临时限流

一句话：

**重试解决的是偶发性失败，不是持续性故障。**

如果下游已经持续变慢、错误率持续升高，就该优先想熔断，而不是继续加重试。

## 为什么指数退避和随机抖动很重要

如果必须重试，更稳的做法通常是：

- 次数少
- 间隔越来越长
- 每次间隔再加一点随机抖动

常见策略：

| 策略              | 问题                         |
| ----------------- | ---------------------------- |
| 固定间隔          | 最容易形成同步重试洪峰       |
| 线性退避          | 比固定好一点，但打散能力一般 |
| 指数退避          | 能明显降低重试密度           |
| 指数退避 + Jitter | 分布式场景里最稳妥           |

为什么要加 `Jitter`？

因为如果一万台客户端都按“1 秒、2 秒、4 秒”重试，不加随机数，它们还是会在同一时间点一起打过来。

## 线上更实用的控制手段：重试预算

一个很重要但经常被忽略的概念是“重试预算”。

它的意思是：

**在一个时间窗口内，允许重试请求占原始请求的比例不能超过某个上限。**

比如规定：

- 1 分钟内最多允许重试量占原始请求量的 10%

一旦超过预算，后面的请求就不再重试，只允许首试或直接失败。

这样做的好处是：

- 故障时不会无限放大流量
- 能把下游保护在一个可控范围内

## 一条常用的工程规则

很多场景里，可以直接记这一套：

1. 在线同步请求，重试尽量控制在 1～2 次。
2. 只对明确可重试的错误重试。
3. 采用指数退避并加随机抖动。
4. 不要每一层都做自动重试。
5. 对写操作先确认幂等，再谈重试。
6. 故障持续时优先熔断，而不是一直试。

这套规则虽然保守，但比“默认重试 3 次”更接近生产。

## 一个很容易说服人的真实场景

比如库存服务抖动：

- 正常 RT 50ms
- 突然抖到 800ms

上游如果把超时设成 200ms，再配 2 次重试，那一次库存查询很可能变成 3 次。

库存服务本来只是“变慢”，结果因为额外流量变成：

- 线程池打满
- Redis 连接占满
- 数据库压力升高

最后连原本能成功的请求也失败了。

这时候真正该做的往往是：

- 缩短排队
- 限制重试
- 熔断部分请求
- 对非核心链路直接降级

而不是“大家一起再试试”。

## 容易踩的坑

### 只看成功率，不看成本

重试可能让成功率看起来变高了，但也可能让：

- 延迟更高
- 线程占用更久
- 下游更容易雪崩

所以不能只盯成功率。

### 重试指标没单独拆出来

至少要区分：

- 原始请求量
- 重试请求量
- 重试成功率
- 重试后总耗时
- 重试触发的错误类型

不拆开看，你很难知道到底是下游坏了，还是重试把它打坏了。

### MQ 消费和在线请求用同一套策略

异步消费可以更激进一点，因为它有死信队列、补偿和人工处理空间。

在线同步请求一般要保守得多，因为用户就卡在链路上等结果。

## 小结

- 重试是在拿额外资源换成功率，故障时它也会放大流量。
- 最危险的组合是：多层都重试、固定间隔重试、对不可重试错误也重试。
- 重试只适合瞬时失败，不适合持续性故障。
- 更稳的做法是少次数、指数退避、随机抖动，再加重试预算。
- 写操作在没确认幂等之前，不要轻易开自动重试。
