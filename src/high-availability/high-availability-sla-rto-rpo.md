# SLA、几个 9、RTO、RPO 到底是什么意思？

> 高可用题一上来先别急着讲限流和多活，先把“业务到底允许挂多久、丢多少数据”说清楚。

很多人答高可用，第一句话就是“上集群、做冗余、加限流”。这些都不算错，但它们回答的是“怎么做”，不是“为什么要做到这个程度”。

高可用里最先要定的是目标：

- 这条链路允许一年停多久
- 宕机之后多久必须恢复
- 恢复时最多允许丢多少数据

这三个问题没定，后面的容灾、多活、同步复制、异步复制都只是拍脑袋。

## 先分清 4 个词在说什么

最容易混的是下面这四个概念：

| 概念     | 它回答的问题                 | 更像哪一层               |
| -------- | ---------------------------- | ------------------------ |
| `SLA`    | 我对外承诺的服务水平是什么   | 商业承诺 / 对外契约      |
| 几个 9   | 这份承诺折算成一年能停多久   | 可用性目标               |
| `RTO`    | 出故障后最多多久必须恢复     | 容灾恢复时间约束         |
| `RPO`    | 出故障后最多允许丢多久的数据 | 容灾数据回退窗口约束     |
| `SLI`    | 用哪个指标衡量服务是否健康   | 监控指标                 |
| `SLO`    | 指标要达到什么目标           | 内部工程目标 / 团队目标  |
| 错误预算 | 在目标范围内还能消耗多少失败 | 发布、变更和稳定性的阀门 |

一句话概括：

- `SLA` 更像对外承诺。
- 几个 9 是可用性的量化方式。
- `RTO` 和 `RPO` 是容灾设计的内部约束。
- `SLI/SLO` 是工程团队真正每天盯的指标和目标。
- 错误预算决定你还能不能继续高速发布和冒险变更。

## 几个 9 到底在算什么

可用性最常见的表达方式是：

```text
可用性 = 成功服务时间 / 总服务时间
```

所以大家常说的“3 个 9、4 个 9、5 个 9”，本质上是在说允许的不可用时间。

| 可用性  | 年度停机预算 | 月度停机预算 | 更常见的业务感受       |
| ------- | ------------ | ------------ | ---------------------- |
| 99%     | 3.65 天      | 7.2 小时     | 内部系统还能忍         |
| 99.9%   | 8.76 小时    | 43.2 分钟    | 普通在线服务的起点     |
| 99.99%  | 52.6 分钟    | 4.32 分钟    | 核心交易链路常见目标   |
| 99.999% | 5.26 分钟    | 25.9 秒      | 极高要求，成本明显上升 |

面试里别把“几个 9”答成炫数字。重点是后面这句：

**几个 9 代表的是停机预算，不是技术栈标签。**

比如：

- 用户中心一年挂 40 分钟，也许业务还能接受。
- 支付链路一年挂 40 分钟，很多团队就接受不了。

所以先看业务损失，再决定要几个 9。

## 可用性到底按时间算，还是按请求算？

两种口径都常见，但含义不一样。

### 按时间算

```text
时间可用性 = 可服务时间 / 总统计时间
```

这种口径适合描述“系统整体是不是可访问”。比如一个服务 1 个月内有 30 分钟完全不可用，就可以按时间折算可用性。

但它有个问题：如果系统没有完全挂，只是部分接口大量失败，时间口径可能掩盖真实影响。

### 按请求算

```text
请求可用性 = 成功请求数 / 总请求数
```

这种口径更适合互联网服务。比如下单接口 100 万次请求里失败了 500 次，请求成功率就是 `99.95%`。

它也有边界：如果失败集中在大促最高峰，虽然全天成功率看起来还行，但业务损失可能非常大。

所以工程里通常会同时看：

| 口径       | 适合看什么                         | 容易漏什么                         |
| ---------- | ---------------------------------- | ---------------------------------- |
| 时间可用性 | 整体宕机、发布事故、机房故障       | 局部接口失败、灰度流量异常         |
| 请求成功率 | 接口级故障、用户真实请求结果       | 低流量时段故障被高流量时段稀释     |
| 延迟分位值 | P99/P999 长尾体验、线程池堆积风险  | 单纯看延迟不一定知道业务是否成功   |
| 业务成功率 | 下单成功、支付成功、回调成功等结果 | 需要业务埋点和口径统一，建设成本高 |

面试里可以主动补一句：**不要只说 99.99%，还要说这个 99.99% 是按哪个接口、哪个时间窗口、哪个成功判定口径算出来的。**

## SLA 不是一句“我们系统很稳”

`SLA` 全称是 `Service Level Agreement`，可以理解成服务提供方和使用方之间对可用性的正式约定。

它通常不会只写一个“99.9%”，还会顺手约束这些东西：

- 可用性目标
- 响应时间目标
- 故障响应时间
- 故障恢复时间
- 赔偿或降级条款

所以在工程里更实用的理解是：

**SLA 是业务承诺，几个 9 只是 SLA 里最容易被量化的一部分。**

## SLI、SLO 和错误预算怎么用？

很多团队内部不直接拿 `SLA` 管日常研发，而是把它拆成 `SLI`、`SLO` 和错误预算。

可以这么理解：

```text
SLI：选什么指标衡量
  ↓
SLO：这个指标要达到什么目标
  ↓
错误预算：允许偏离目标多少
  ↓
工程动作：发布节奏、降级策略、故障复盘、容量投入
```

举个订单服务的例子：

| 层次     | 例子                                               |
| -------- | -------------------------------------------------- |
| `SLI`    | 下单接口 5xx 比例、P99 延迟、下单业务成功率        |
| `SLO`    | 30 天内下单成功率 >= 99.95%，P99 <= 500ms          |
| 错误预算 | 30 天内最多允许 0.05% 的下单失败或超时             |
| 工程动作 | 错误预算快烧完时，暂停非必要发布，优先做稳定性治理 |

错误预算的价值是把“要稳定”和“要迭代”放到同一个尺子上。

- 错误预算还很充足：可以接受正常发布和小步试错。
- 错误预算消耗过快：说明稳定性在恶化，应该收紧发布、扩容、降级或修复核心问题。
- 错误预算已经耗尽：继续发布新功能就是在透支业务承诺。

这比一句“我们要高可用”更可执行。

## RTO 和 RPO 分别盯的是什么

这两个词是容灾题的核心。

### RTO：盯恢复时间

`RTO` 是 `Recovery Time Objective`，回答的是：

**出故障后，最长多久必须恢复服务。**

举例：

- 下单服务 `RTO = 10 分钟`：意味着事故发生后，10 分钟内至少要把主链路恢复。
- 报表系统 `RTO = 4 小时`：说明它不是最核心链路，可以接受恢复慢一点。

### RPO：盯数据回退窗口

`RPO` 是 `Recovery Point Objective`，回答的是：

**出故障后，最多允许丢失多久的数据。**

举个最容易理解的例子：

- 如果主库挂了，而从库只同步到了 30 秒前，那这 30 秒就是数据回退窗口。
- 如果业务要求 `RPO = 0`，就意味着恢复后不能丢这 30 秒的数据。

所以：

- `RTO` 看“恢复要多快”
- `RPO` 看“数据能丢多少”

可以用一条时间线理解：

```text
最后一次可恢复的数据点
        │<-------- RPO -------->│
        │                       │
        ├─────────────── 故障发生 ─────────────── 恢复服务
                                │<------ RTO ----->│
```

如果数据库主库在 10:00:00 故障，从库只同步到 09:59:45，最终 10:03:00 完成切换：

- `RPO` 实际结果约为 15 秒；
- `RTO` 实际结果约为 3 分钟。

注意这是“实际结果”，不是目标。目标要提前定，结果要靠演练和故障复盘验证。

## 为什么很多系统做不到既低 RTO 又低 RPO

因为它们通常要花钱，也要牺牲别的指标。

比如你希望：

- 切换几乎无感知
- 数据一条都不丢
- 跨机房部署
- 写入还不能变慢

这几个目标往往会互相打架。

常见取舍是：

| 方案                | 对 RTO 的帮助 | 对 RPO 的帮助        | 代价                               |
| ------------------- | ------------- | -------------------- | ---------------------------------- |
| 单机 + 定时备份     | 很弱          | 一般                 | 最便宜，但恢复慢、丢数多           |
| 主从复制 + 自动切换 | 较强          | 取决于复制延迟       | 可能切换快，但仍可能丢最后一段数据 |
| 同城双活 / 多活     | 很强          | 取决于复制和写入策略 | 架构复杂、冲突处理复杂             |
| 强同步跨站写入      | 对 RPO 很强   | 很强                 | 写延迟和可用性压力更大             |

这就是为什么面试里说“做多活”还不够，必须补一句：

**你准备拿什么换。**

## 依赖越多，可用性越容易被乘低

高可用还有一个很容易被忽略的事实：一条链路依赖越多，整体可用性越容易下降。

假设下单链路必须串行依赖 4 个服务：

```text
下单 API -> 库存服务 -> 优惠服务 -> 支付服务 -> 订单库
```

如果每个依赖都是 `99.9%`，并且任意一个失败都会导致下单失败，那么粗略看：

```text
整体可用性 ≈ 99.9% × 99.9% × 99.9% × 99.9%
          ≈ 99.6%
```

这不是精确建模，但能说明一个工程事实：

**下游依赖越多，核心链路越需要超时、降级、缓存、异步化、隔离和补偿。**

比如：

| 依赖       | 如果失败怎么办                         |
| ---------- | -------------------------------------- |
| 优惠服务   | 可以降级为不使用优惠，或后续补发优惠   |
| 推荐服务   | 可以直接跳过，不影响下单主链路         |
| 库存服务   | 通常不能随便跳过，需要强校验或预扣库存 |
| 支付服务   | 不能静默成功，要靠状态机和回调补偿     |
| 订单数据库 | 核心持久化依赖，要优先保障和演练切换   |

所以高可用不是把所有依赖都做到 5 个 9，而是先识别核心链路，再判断哪些依赖必须强依赖，哪些可以降级、异步或补偿。

## 不同业务应该怎么定目标？

不要整站一刀切。

可以按业务损失、用户体感和数据风险分级：

| 链路     | 可用性目标示例     | RTO 示例   | RPO 示例       | 设计重点                         |
| -------- | ------------------ | ---------- | -------------- | -------------------------------- |
| 登录     | 99.9% 或更高       | 分钟级     | 用户态可重建   | 限流、防刷、验证码、降级入口     |
| 下单     | 99.95% / 99.99%    | 分钟级     | 尽量接近 0     | 库存一致性、订单状态机、补偿     |
| 支付     | 99.99% 甚至更高    | 秒～分钟   | 不能丢交易状态 | 对账、回调幂等、状态机、人工兜底 |
| 推荐     | 99% 也可能够用     | 可较长     | 可丢一段缓存   | 缓存兜底、默认推荐、快速降级     |
| 离线报表 | 99% 或按任务成功率 | 小时级     | 可重跑         | 任务重试、断点续跑、数据校验     |
| 管理后台 | 99.9%              | 分钟～小时 | 不能越权       | 权限、安全审计、操作可追溯       |

这张表不是固定答案，重点是思路：**核心交易链路优先保护，辅助能力允许降级，离线链路优先保证可重跑和可校验。**

## 指标怎么落到监控和演练？

目标写在文档里没有用，必须能被看见、报警和验证。

一个比较实用的闭环是：

```text
业务目标
  ↓
SLI/SLO 指标
  ↓
监控看板和告警阈值
  ↓
压测 / 故障演练 / 切流演练
  ↓
复盘实际 RTO/RPO 和错误预算消耗
  ↓
修正架构、预案和目标
```

至少要落这些指标：

| 指标                   | 用来判断什么                           |
| ---------------------- | -------------------------------------- |
| 入口成功率 / 5xx 比例  | 用户请求有没有大面积失败               |
| P99 / P999 延迟        | 长尾是否已经影响用户或拖垮线程池       |
| 核心业务成功率         | 业务是否真的完成，比如下单、支付、回调 |
| 下游依赖错误率和超时率 | 故障是不是从依赖扩散过来的             |
| 复制延迟 / 消费积压    | RPO 风险和异步链路是否在累积           |
| 切换耗时               | 实际 RTO 是否满足目标                  |
| 数据校验差异           | 切换后是否出现数据丢失或不一致         |

故障演练时不要只看“服务能不能切过去”，还要记录：

- 从故障发生到告警触发用了多久；
- 从告警触发到定位用了多久；
- 从定位到切换完成用了多久；
- 切换后有没有数据回退、重复消费、缓存污染；
- 用户侧实际失败了多少请求。

这些数据才是下一次改进 `RTO/RPO` 的依据。

## 一个最实用的回答顺序

真到系统设计题里，可以按这个顺序说：

1. 先定业务等级：这是支付链路、订单链路，还是报表链路？
2. 再定指标：SLI 看什么，SLO 设多少，对外 SLA 怎么承诺？
3. 再定目标：一年允许停多久，最多允许丢多少数据？
4. 然后反推架构：单机备份、主从切换、同城灾备、异地多活分别够不够。
5. 最后补验证：压测、故障演练、切流演练有没有做过，实际 RTO/RPO 是否达标。

这样答，高可用就从“背名词”变成“先定目标再定方案”了。

## 容易答错的点

### `SLA` 不是系统真实结果

`SLA` 是目标和承诺，不等于你今天真的能做到。

真正能不能做到，要靠：

- 监控
- 演练
- 切换自动化
- 团队值班能力

### `RTO = 0` 和 `RPO = 0` 都很贵

这两个指标一旦逼近 0，背后通常意味着：

- 更严格的同步复制
- 更复杂的切换机制
- 更高的跨机房成本
- 更难处理网络分区

所以不要把它当成默认目标。

### `SLO` 和 `SLA` 混着说

`SLO` 是内部目标，通常要比对外 `SLA` 更严格一点，留出运营、赔付、统计口径和突发事故的缓冲。

如果对外承诺 99.9%，内部只按 99.9% 建设，一旦有一次发布事故或依赖故障，就很容易把承诺打穿。

### 只看平均响应时间

高可用指标不能只看平均 RT。

平均值会掩盖长尾问题：P99/P999 抬高时，线程池、连接池和上游重试可能已经在堆积，系统离雪崩不远了。

### 不是所有业务都要统一一个等级

订单、支付、报表、推荐、搜索的可用性要求通常根本不是一个等级。

更成熟的做法是分级，而不是整站一刀切地说“我们要 5 个 9”。

## 小结

1. `SLA` 是对外承诺，`SLO` 是内部目标，`SLI` 是衡量指标，几个 9 只是可用性的量化表达。
2. 可用性要说明统计口径：按时间、按请求、按业务成功率，结论可能完全不同。
3. `RTO` 约束恢复时间，`RPO` 约束数据回退窗口，二者都是目标，不是系统天然结果。
4. 链路依赖越多，整体可用性越容易被乘低，所以要识别核心依赖并设计降级、隔离和补偿。
5. 高可用方案不是先选技术，而是先定业务目标、错误预算和容灾指标，再用压测和演练验证。
