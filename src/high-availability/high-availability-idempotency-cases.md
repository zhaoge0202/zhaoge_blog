# 支付回调和消息重复消费怎么做幂等？

> 支付回调和 MQ 消费最容易把“幂等要做”说成一句正确废话，真正难的是落到状态、事务和返回策略。

这两个场景经常放在一起讲，因为它们有一个共同点：

**系统必须接受“同一笔业务可能重复送达”这件事。**

区别在于：

- 支付回调通常来自第三方，重复到达是外部现实
- MQ 重复消费通常来自至少一次投递，是中间件语义

所以这题真正要答的是：

重复到了以后，系统怎么做到“不多记账、不多发货、不多执行”。

把它们放到一条链路里看，会更直观：

```text
用户支付成功
  ↓
支付平台多次回调商户系统
  ↓
商户系统确认订单已支付
  ↓
发送“支付成功”消息
  ↓
发货、积分、优惠券、通知等消费者可能重复消费
```

这条链路里只要任意一步超时、响应丢失、消费者重启，就可能重复送达。
所以幂等不是某个接口的小技巧，而是贯穿“外部回调 -> 本地状态 -> 消息消费”的状态保护。

## 先抓住两个共同原则

不管是支付回调还是 MQ 消费，先记住这两句：

1. **要有稳定的业务唯一号**
2. **要把状态推进设计成只能成功一次**

如果没有这两条，后面全是补丁。

还要再补一句：

**幂等记录、业务状态和业务流水要尽量在同一个事务边界内完成。**

否则最容易出现“幂等记录说成功，业务其实没成功”的假成功。

## 场景一：支付回调为什么一定要做幂等

第三方支付平台为了确保商户最终能收到结果，通常会这样干：

- 你没按时返回成功，它会重试
- 网络抖动了，它也会重试
- 你返回丢了，它还是会重试

所以“同一笔支付成功回调收到多次”不是例外，而是默认前提。

### 支付回调最稳的主线

更稳的处理顺序通常是：

1. 校验签名和商户号
2. 取出第三方交易流水号
3. 查询本地订单状态
4. 只允许 `UNPAID -> PAID` 这一类合法推进
5. 写订单状态、支付流水、回调记录放进同一个事务
6. 重复回调直接返回成功

这里最关键的是最后一句：

**重复回调通常应该返回成功，而不是报错。**

因为如果你返回失败，第三方往往还会继续重试。

更完整一点，支付回调不要一上来就改订单，前置校验也必须做足：

| 步骤       | 目的                                   |
| ---------- | -------------------------------------- |
| 校验签名   | 确认回调确实来自可信平台，防止伪造通知 |
| 校验商户号 | 确认回调属于当前商户，防止串商户       |
| 校验金额   | 确认支付金额和本地订单金额一致         |
| 校验订单号 | 找到本地订单，确认不是未知订单         |
| 校验流水号 | 用第三方交易流水号识别同一笔支付       |
| 校验状态机 | 只允许未支付订单推进到已支付           |

这些校验里，签名、商户号、金额属于安全和业务正确性；流水号、状态机属于幂等和并发正确性。

## 支付回调常见落地组合

| 手段               | 它在支付回调里负责什么                     |
| ------------------ | ------------------------------------------ |
| 第三方交易流水号   | 标识同一笔回调                             |
| 回调记录表唯一索引 | 防止重复入库                               |
| 订单状态机         | 防止重复推进到已支付                       |
| 本地事务           | 保证订单、流水、回调记录一起成功或一起失败 |

这四样通常一起上，比单独依赖某一个稳很多。

一个更像生产系统的事务流程可以这样设计：

```text
begin
  插入回调记录，transaction_id 建唯一索引
  查询订单并校验金额、商户号、支付状态
  update 订单 set status = PAID where status = UNPAID
  写支付流水
  写 outbox 事件：PAYMENT_SUCCESS
commit

事务提交成功后：
  返回支付平台成功
  由 outbox / MQ 投递支付成功事件
```

这里把“写订单状态、写流水、写待发送事件”放在一个事务里，是为了避免订单已支付但后续发货消息丢失。
如果项目没有 outbox，也至少要保证订单状态和支付流水在同一个事务里，后续再通过可靠补偿把支付成功事件补出来。

## 一个典型误区：只靠“查订单状态”

很多人会说：

> “先查订单是不是已支付，已支付就直接返回。”

这思路没错，但不够。

因为在并发情况下，两个回调可能同时查到“还没支付”，然后一起往下执行。

所以更稳的做法是：

- 状态查询
- 唯一约束
- 事务更新

一起配。

还有一个容易忽略的点：**重复回调不是只有“成功回调重复”这一种。**

可能出现的情况包括：

- 同一笔成功回调并发到达。
- 成功回调先到，稍后又来了同一流水的重复成功回调。
- 本地已经处理成功，但返回给支付平台的响应丢了。
- 回调记录写成功，但业务事务失败后回滚。

所以支付回调的返回策略可以按状态拆：

| 本地状态                 | 处理方式                                     |
| ------------------------ | -------------------------------------------- |
| 订单未支付且校验通过     | 推进到已支付，写流水，返回成功               |
| 订单已支付且流水一致     | 直接返回成功，避免外部继续重试               |
| 订单已关闭或已退款       | 记录异常，按业务规则返回失败或人工介入       |
| 金额、商户号、签名不匹配 | 拒绝处理，记录安全审计                       |
| 本地处理中               | 可以返回失败让外部稍后重试，或返回处理中状态 |

资金链路里不要为了“少重试”而乱返回成功。只有确认本地已经正确处理过，重复回调才应该直接成功。

## 场景二：消息为什么天然要接受重复消费

绝大多数 MQ 在工程语义上更接近：

**至少一次投递（at least once）**

这意味着：

- 消费成功但 ack 丢了，消息可能再投一次
- 消费者重启、超时、rebalance，也可能重复投
- 生产者重试时，同一业务事件可能被重新发送
- 消费者处理太慢，消息可见性超时后可能再次被投递

所以消息消费者正确的心态不是“怎么保证绝不重复”，而是：

**即使重复消费，也不能把业务状态做坏。**

这里要区分两个重复：

| 类型       | 例子                                         | 幂等抓手                        |
| ---------- | -------------------------------------------- | ------------------------------- |
| 消息级重复 | 同一个消息 ID 被投递两次                     | 消费记录、消息 ID、投递状态     |
| 业务级重复 | 同一笔订单支付成功事件被补偿脚本重新发了一次 | 订单号、支付流水号、业务事件 ID |

更稳的是优先用业务唯一键，因为业务补偿、重建消息、跨系统转发时，消息 ID 可能变了，
但业务事件还是同一笔。

## MQ 重复消费最常见的做法

### 1. 用消息 ID 或业务流水号做幂等键

只要消息里有稳定标识，就能知道这是不是同一笔业务。

常见标识包括：

- 消息 ID
- 订单号
- 支付流水号
- 业务请求号

实际更推荐把“业务类型 + 业务唯一号 + 事件类型”组合成幂等键。

比如：

```text
PAYMENT_SUCCESS:pay_202606300001
ORDER_DELIVER:order_202606300001
COUPON_ISSUE:req_202606300001
```

这样可以避免不同业务误用同一个 key，也能在日志里快速定位是哪类事件重复。

### 2. 先落幂等记录，再做业务动作

常见流程是：

1. 尝试往消费记录表插入一条唯一记录
2. 插入成功，说明第一次消费，继续执行业务
3. 插入失败，说明重复消费，直接返回成功

这个流程的关键不是“有表就行”，而是：

**消费记录和业务处理最好在同一个事务里。**

不然也会出现：

- 消费记录写进去了
- 真正的业务没做成

下一次重投时，你又会把它误判成“已经消费过”。

消费记录也最好有状态，而不是只存“已消费”：

| 状态         | 含义                         | 重复消息到达时怎么处理             |
| ------------ | ---------------------------- | ---------------------------------- |
| `PROCESSING` | 已开始处理但还没提交业务结果 | 返回稍后重试，或由超时扫描任务接管 |
| `SUCCESS`    | 已处理成功                   | 直接确认成功，不再执行业务动作     |
| `FAILED`     | 已失败                       | 看失败类型，允许重试或进入人工补偿 |

如果消费记录和业务表在同一个库，优先放同一个事务里。
如果不在同一个库，就要用状态机、补偿任务、对账任务兜底，不能假装它们天然一致。

### 3. 状态流转要能挡住重复执行

比如发货消息：

```text
PAID -> DELIVERING -> DELIVERED
```

如果订单已经是 `DELIVERED`，同一条消息再来，应该直接跳过，而不是再发一次货。

所以消息幂等本质上还是离不开状态机。

状态推进最好写成条件更新：

```sql
update t_order
set status = 'DELIVERING'
where order_no = ? and status = 'PAID';
```

更新行数为 1，说明这次消息真正推进了状态。
更新行数为 0，就要查当前状态：如果已经是 `DELIVERING` 或 `DELIVERED`，说明重复消息可以直接确认；
如果是 `CLOSED`、`REFUNDED` 这类状态，就不能继续发货，要记录异常。

### 4. ack 要在业务提交之后

消息消费里最常见的顺序坑是 ack 太早。

```text
错误顺序：
  先 ack
  再执行业务
  业务失败
  消息不会再投，业务丢了

更稳顺序：
  执行业务事务
  提交成功
  再 ack / commit offset
```

如果业务提交成功，但 ack 或 offset commit 失败，消息可能还会再投一次。
这正是为什么消费侧必须幂等：它用“允许重复”换“业务不丢”。

## 支付回调和 MQ 重复消费有什么不同

| 维度     | 支付回调                  | MQ 重复消费                      |
| -------- | ------------------------- | -------------------------------- |
| 请求来源 | 第三方平台                | 消息中间件                       |
| 重复原因 | 超时、响应丢失、平台重试  | 至少一次投递、重平衡、消费者故障 |
| 返回策略 | 重复回调通常直接返回成功  | 重复消息通常直接确认成功         |
| 核心抓手 | 第三方流水号 + 订单状态机 | 消息键 + 消费记录 + 业务状态机   |
| 风险重点 | 资金入账、订单状态一致性  | 重复发货、重复发券、消息积压     |

所以它们的核心思想一样，但返回策略和落点不完全一样。

## 最容易被追问的一点：失败了该怎么返回

### 支付回调

如果你已经确认这笔回调此前成功处理过，通常应该返回成功，告诉第三方别再重试。

### MQ 消费

如果你确定这条消息已经处理过，也通常应该当作成功处理，而不是再抛异常让它回队列。

否则重复消息就会在队列里来回打转。

但如果这次消费执行到一半失败了，就不要把它当成功吞掉。
更合理的是按错误类型拆：

| 错误类型               | 处理方式                                 |
| ---------------------- | ---------------------------------------- |
| 临时网络错误、下游超时 | 抛出异常或返回失败，让消息按退避策略重试 |
| 参数缺失、业务状态非法 | 记录异常，必要时进入死信或人工处理       |
| 已处理成功的重复消息   | 直接确认成功                             |
| 处理中状态超时         | 由补偿任务检查业务状态后重置或补偿       |

不要把所有异常都无限重试。无限重试会拖慢整个消费组，最后演变成消息积压。

## 什么时候需要额外上锁

如果同一业务资源并发竞争非常激烈，比如：

- 同一订单多个异步事件同时到达
- 同一支付流水在多个消费者实例同时撞上

可以考虑补一层锁或单线程分区消费。

但还是那句话：

**锁是辅助串行化，不是幂等本身。**

幂等的底座仍然是：

- 业务唯一标识
- 唯一约束
- 状态机
- 事务边界

## “精确一次”不要答成没有重复

有些消息系统或流处理框架会提供特定边界内的“精确一次”能力。
面试里要小心，不要把它理解成端到端业务永远不会重复。

更稳的说法是：

- 消息系统可以减少重复、约束生产和消费边界。
- 但网络、数据库提交、外部接口、消费者重启仍然会带来不确定性。
- 真正的业务正确性仍要靠业务唯一键、状态机、事务和补偿兜底。

所以回答 MQ 幂等时，不要把“中间件支持精确一次”当作最终答案。

## 上线后怎么观测

支付回调和 MQ 幂等都要能从日志和指标里查清楚。

| 维度       | 关键指标或日志字段                              |
| ---------- | ----------------------------------------------- |
| 支付回调   | 第三方流水号、订单号、回调次数、验签失败次数    |
| 状态推进   | `UNPAID -> PAID` 成功次数、重复回调命中次数     |
| 消费幂等   | 消息键、业务事件 ID、消费记录状态、重复命中次数 |
| 消费失败   | 重试次数、死信数量、失败原因、积压量            |
| 数据一致性 | 订单状态、支付流水、发货记录、消费记录对账结果  |

没有这些字段，线上事故发生后很难判断：这次请求是第一次、重复、处理中，还是已经成功但响应丢了。

## 怎么验证这套设计

可以专门做几类测试：

1. 同一笔支付回调并发发送 50 次，订单只能从 `UNPAID` 变为 `PAID` 一次。
2. 支付回调处理成功后模拟响应丢失，再次回调应该直接返回成功。
3. 支付流水写入失败时，订单状态和回调记录必须一起回滚。
4. 同一条发货消息重复投递，最终只能生成一次发货记录。
5. 消费业务成功但 ack 失败后，消息重投不能重复发货或重复发券。
6. 消费记录处于 `PROCESSING` 太久时，补偿任务能判断真实业务状态。

## 一个更像生产系统的回答方式

如果面试被问到这题，可以直接按这条线答：

1. 支付回调和 MQ 都默认会重复到达，所以不能假设“只会来一次”。
2. 先给每笔业务一个稳定唯一标识，比如订单号、交易流水号、消息键。
3. 用唯一约束或去重表识别是不是已经处理过。
4. 用状态机保证业务状态只能从合法状态推进一次。
5. 把幂等记录和业务状态更新放在同一事务里。
6. 支付回调要验签、验商户、验金额，MQ 消费要在业务提交后再 ack。
7. 对已处理过的请求或消息，直接按成功返回，避免外部系统继续重试。

## 小结

- 支付回调和 MQ 重复消费都要接受“重复到达是正常现象”这个前提。
- 支付回调要先验签、验商户、验金额，再用第三方流水号和订单状态机做幂等。
- MQ 消费优先用业务唯一键，消费记录、业务状态和 ack 顺序要配合好。
- 只查状态不够，最好再配唯一约束和同库事务，防并发竞态和假成功。
- 对已经处理过的回调或消息，通常应该按成功返回，别让外部系统继续重试。
