# 支付回调和消息重复消费怎么做幂等？

> 支付回调和 MQ 消费最容易把“幂等要做”说成一句正确废话，真正难的是落到状态、事务和返回策略。

这两个场景经常放在一起讲，因为它们有一个共同点：

**系统必须接受“同一笔业务可能重复送达”这件事。**

区别在于：

- 支付回调通常来自第三方，重复到达是外部现实
- MQ 重复消费通常来自至少一次投递，是中间件语义

所以这题真正要答的是：

重复到了以后，系统怎么做到“不多记账、不多发货、不多执行”。

## 先抓住两个共同原则

不管是支付回调还是 MQ 消费，先记住这两句：

1. **要有稳定的业务唯一号**
2. **要把状态推进设计成只能成功一次**

如果没有这两条，后面全是补丁。

## 场景一：支付回调为什么一定要做幂等

第三方支付平台为了确保商户最终能收到结果，通常会这样干：

- 你没按时返回成功，它会重试
- 网络抖动了，它也会重试
- 你返回丢了，它还是会重试

所以“同一笔支付成功回调收到多次”不是例外，而是默认前提。

### 支付回调最稳的主线

更稳的处理顺序通常是：

1. 校验签名和商户号
2. 取出第三方交易流水号
3. 查询本地订单状态
4. 只允许 `UNPAID -> PAID` 这一类合法推进
5. 写订单状态、支付流水、回调记录放进同一个事务
6. 重复回调直接返回成功

这里最关键的是最后一句：

**重复回调通常应该返回成功，而不是报错。**

因为如果你返回失败，第三方往往还会继续重试。

## 支付回调常见落地组合

| 手段               | 它在支付回调里负责什么                     |
| ------------------ | ------------------------------------------ |
| 第三方交易流水号   | 标识同一笔回调                             |
| 回调记录表唯一索引 | 防止重复入库                               |
| 订单状态机         | 防止重复推进到已支付                       |
| 本地事务           | 保证订单、流水、回调记录一起成功或一起失败 |

这四样通常一起上，比单独依赖某一个稳很多。

## 一个典型误区：只靠“查订单状态”

很多人会说：

> “先查订单是不是已支付，已支付就直接返回。”

这思路没错，但不够。

因为在并发情况下，两个回调可能同时查到“还没支付”，然后一起往下执行。

所以更稳的做法是：

- 状态查询
- 唯一约束
- 事务更新

一起配。

## 场景二：消息为什么天然要接受重复消费

绝大多数 MQ 在工程语义上更接近：

**至少一次投递（at least once）**

这意味着：

- 消费成功但 ack 丢了，消息可能再投一次
- 消费者重启、超时、 rebalance，也可能重复投

所以消息消费者正确的心态不是“怎么保证绝不重复”，而是：

**即使重复消费，也不能把业务状态做坏。**

## MQ 重复消费最常见的做法

### 1. 用消息 ID 或业务流水号做幂等键

只要消息里有稳定标识，就能知道这是不是同一笔业务。

常见标识包括：

- 消息 ID
- 订单号
- 支付流水号
- 业务请求号

### 2. 先落幂等记录，再做业务动作

常见流程是：

1. 尝试往消费记录表插入一条唯一记录
2. 插入成功，说明第一次消费，继续执行业务
3. 插入失败，说明重复消费，直接返回成功

这个流程的关键不是“有表就行”，而是：

**消费记录和业务处理最好在同一个事务里。**

不然也会出现：

- 消费记录写进去了
- 真正的业务没做成

下一次重投时，你又会把它误判成“已经消费过”。

### 3. 状态流转要能挡住重复执行

比如发货消息：

- 订单从 `PAID -> DELIVERING -> DELIVERED`

如果订单已经是 `DELIVERED`，同一条消息再来，应该直接跳过，而不是再发一次货。

所以消息幂等本质上还是离不开状态机。

## 支付回调和 MQ 重复消费有什么不同

| 维度     | 支付回调                  | MQ 重复消费                      |
| -------- | ------------------------- | -------------------------------- |
| 请求来源 | 第三方平台                | 消息中间件                       |
| 重复原因 | 超时、响应丢失、平台重试  | 至少一次投递、重平衡、消费者故障 |
| 返回策略 | 重复回调通常直接返回成功  | 重复消息通常直接确认成功         |
| 核心抓手 | 第三方流水号 + 订单状态机 | 消息键 + 消费记录 + 业务状态机   |

所以它们的核心思想一样，但返回策略和落点不完全一样。

## 最容易被追问的一点：失败了该怎么返回

### 支付回调

如果你已经确认这笔回调此前成功处理过，通常应该返回成功，告诉第三方别再重试。

### MQ 消费

如果你确定这条消息已经处理过，也通常应该当作成功处理，而不是再抛异常让它回队列。

否则重复消息就会在队列里来回打转。

## 什么时候需要额外上锁

如果同一业务资源并发竞争非常激烈，比如：

- 同一订单多个异步事件同时到达
- 同一支付流水在多个消费者实例同时撞上

可以考虑补一层锁或单线程分区消费。

但还是那句话：

**锁是辅助串行化，不是幂等本身。**

幂等的底座仍然是：

- 业务唯一标识
- 唯一约束
- 状态机
- 事务边界

## 一个更像生产系统的回答方式

如果面试被问到这题，可以直接按这条线答：

1. 支付回调和 MQ 都默认会重复到达，所以不能假设“只会来一次”。
2. 先给每笔业务一个稳定唯一标识，比如订单号、交易流水号、消息键。
3. 用唯一约束或去重表识别是不是已经处理过。
4. 用状态机保证业务状态只能从合法状态推进一次。
5. 把幂等记录和业务状态更新放在同一事务里。
6. 对已处理过的请求或消息，直接按成功返回，避免外部系统继续重试。

## 小结

- 支付回调和 MQ 重复消费都要接受“重复到达是正常现象”这个前提。
- 支付回调用第三方流水号和订单状态机，MQ 用消息键和消费记录表，底层思路是一致的。
- 只查状态不够，最好再配唯一约束和事务，防并发竞态。
- 对已经处理过的回调或消息，通常应该按成功返回，别让外部系统继续重试。
- 锁只能做并发辅助，不能替代幂等的持久化设计。
