# 容灾、多活、故障转移分别解决什么问题？

> “多部署几台机器”只能算冗余，离完整容灾还差故障检测、切流、数据复制和恢复验证。

高可用题答到后面，经常会被追问：

- 同城灾备和多活有什么区别
- 异地多活为什么难
- 故障转移是自动还是人工

这题真正要答清的是三层东西：

1. 你是在做冗余，还是在做容灾
2. 你要的是主备切换，还是多活承载
3. 故障发生后，流量和数据怎么恢复

这篇和前面的
[`SLA、几个 9、RTO、RPO 到底是什么意思？`](./high-availability-sla-rto-rpo.html)
是连着的：前一篇定目标，这一篇讲方案怎么落。

## 先分清几个经常混在一起的词

| 概念     | 它主要解决什么                         |
| -------- | -------------------------------------- |
| 冗余     | 某个资源挂了，还有备份资源顶上         |
| 容灾     | 机房级、地域级事故时，系统还能恢复     |
| 多活     | 不止备着，而是多个站点同时承接生产流量 |
| 故障转移 | 发现故障后，把流量切到健康节点         |

一句话概括：

- 冗余回答“有没有备用”
- 容灾回答“出大故障能不能恢复”
- 多活回答“备用资源平时是不是也在工作”
- 故障转移回答“真的挂了之后怎么切”

## 从简单到复杂，常见方案大概是这几层

| 方案        | 解决的问题                   | 成本和复杂度 |
| ----------- | ---------------------------- | ------------ |
| 单机 + 备份 | 机器坏了还能恢复数据         | 最低         |
| 主从 / 主备 | 节点挂了能切到备用节点       | 较低         |
| 同城灾备    | 单个机房事故时还能恢复       | 中等         |
| 异地灾备    | 城市 / 区域级事故时还能恢复  | 更高         |
| 同城多活    | 多机房同时承载流量，切换更快 | 高           |
| 异地多活    | 跨地域同时承载流量，容灾最强 | 很高         |

所以别把“多活”当默认答案。很多业务做到同城主备，就已经够了。

如果从资源状态看，还可以把灾备分成冷备、温备、热备和多活：

| 层级 | 备用资源状态                   | 常见恢复速度 | 适合场景                 |
| ---- | ------------------------------ | ------------ | ------------------------ |
| 冷备 | 资源和服务平时不运行           | 小时级以上   | 非核心系统、低成本备份   |
| 温备 | 关键资源已准备，接管前还要扩容 | 分钟级       | 一般业务系统             |
| 热备 | 资源常驻运行，数据持续同步     | 秒级到分钟级 | 核心链路、重要存储       |
| 多活 | 多个站点平时都承载生产流量     | 接近实时     | 极高可用要求、强运维能力 |

这张表的重点不是背名词，而是看它能不能满足目标 `RTO/RPO`。

## 用 RTO/RPO 反推方案

容灾方案不能从“我要做多活”开始，而要从业务约束反推。

| 业务目标                         | 可能的方案                      | 需要接受的代价                       |
| -------------------------------- | ------------------------------- | ------------------------------------ |
| 小时级恢复、可丢少量离线数据     | 备份恢复、冷备                  | 恢复慢，人工步骤多                   |
| 分钟级恢复、允许秒级到分钟级丢数 | 主从 / 主备、同城灾备、异步复制 | 切换时可能有复制延迟                 |
| 秒级恢复、尽量少丢数据           | 热备、同城双机房、半自动切换    | 成本高，切换流程更复杂               |
| 接近无感恢复、尽量不丢核心数据   | 同城多活、单元化、多副本强约束  | 写延迟、冲突处理、运维复杂度明显上升 |
| 跨区域灾难仍要服务               | 异地灾备 / 异地多活             | 数据一致性、流量调度、合规成本更高   |

比如报表系统可以接受 `RTO = 4 小时`、`RPO = 1 小时`，那冷备和定时备份可能就够了。
但支付链路如果要求分钟级恢复、不能丢交易状态，就必须考虑交易流水、回调幂等、对账补偿和更严格的数据复制。

## 主备、灾备、多活最大的区别是什么

### 主备

最常见的是：

- 主节点提供服务
- 备节点平时待命
- 主节点故障时切到备节点

它解决的是单点故障问题，优点是简单，缺点是备用资源利用率低。

### 灾备

灾备更强调“故障域隔离”。

比如主备不放在同一机房，而是放在：

- 同城不同机房
- 异地不同机房

这样即使整机房停电、网络断了，也还有恢复机会。

### 多活

多活不是“有备份”，而是：

**多个站点平时就一起对外服务。**

这样好处是：

- 资源利用率更高
- 一个站点故障时，另一个站点已经在跑生产流量

但代价也最明显：

- 数据同步难
- 流量调度难
- 冲突处理难

## 故障域要先画清楚

容灾不是简单把机器数量翻倍，而是要让副本处在不同故障域里。

常见故障域可以这样分：

```text
进程 -> 主机 -> 机架 -> 可用区 / 机房 -> 城市 -> 区域
```

如果两个副本在同一台物理机上，进程挂了能救，机器断电救不了。
如果两个机房共用同一条网络出口或同一个供电故障点，看起来是双机房，实际故障域并不独立。

所以设计容灾时要问：

- 副本是不是跨主机、跨机架、跨机房？
- 网络链路、供电、负载均衡、DNS、证书、配置中心有没有单点？
- 数据副本之间是同步、半同步还是异步？
- 备用站点平时有没有真实流量和健康检查？

## 同城灾备和同城多活，不是一回事

这两个特别容易混。

### 同城灾备

更像：

- 主站点承担生产流量
- 备站点平时不承担或只少量承担
- 出故障时接管

它重点解决的是“机房挂了怎么办”。

### 同城多活

更像：

- 多个机房同时承载流量
- 平时就有调度和分流
- 故障时把剩余流量压到健康站点

它重点不只是恢复，还包括平时就让多机房一起工作。

同城灾备更像“平时主用，故障时接管”；同城多活更像“平时就分担生产流量”。
两者对团队的要求差别很大：

| 维度     | 同城灾备                       | 同城多活                         |
| -------- | ------------------------------ | -------------------------------- |
| 平时流量 | 主要在主站点                   | 多站点同时承载                   |
| 切换压力 | 故障时压力集中爆发             | 平时就能验证调度和容量           |
| 数据写入 | 通常单主更容易控制             | 要处理跨站点写入和冲突           |
| 运维难度 | 中等                           | 高                               |
| 典型风险 | 备站长期没流量，切时才暴露问题 | 多站点状态不一致、路由和容量误判 |

## 异地多活为什么总被说“很难”

因为它不是多放几份服务就结束了。

真正难点通常在这里：

### 数据一致性

跨地域写入的延迟和分区不可避免，多个站点同时写时，冲突怎么处理是核心难点。

### 流量调度

用户该路由到哪个站点：

- 按地域
- 按用户
- 按租户
- 按业务单元

都要提前想清楚。

### 状态共享

登录态、购物车、会话、库存、订单这些状态怎么在多个站点之间协作，复杂度很高。

### 演练和运维

异地多活最怕“纸面上很强，真切流时没人敢动”。

所以很多团队不是做不了多活，而是：

**做了以后没有足够强的演练、监控和治理能力去托住它。**

异地多活常见的落地思路不是“所有数据随便跨地域同时写”，而是尽量把写入收敛到一个业务单元：

- 按用户、租户、商户或地域做单元化路由。
- 同一用户或同一商户的核心写入固定到一个站点。
- 跨单元操作用异步事件、TCC / Saga、对账补偿处理。
- 冲突必须有业务规则，比如按版本、按时间、按状态机或人工仲裁。

如果业务模型无法切分、写冲突很多、又没有补偿和对账能力，异地多活反而会把系统复杂度推到不可控。

## 故障转移真正包含哪几步

很多回答只说“节点挂了就切换”，其实至少要有这几步：

1. 故障检测
2. 故障确认
3. 流量切换
4. 告警通知
5. 节点恢复和重新纳管

这里最容易漏掉的是第二步：故障确认。

因为如果误判了，可能会造成双主、脑裂，反而比原故障更糟。

更完整的流程可以拆得更细：

```text
监控发现异常
  ↓
多维度确认故障：心跳、业务探测、复制延迟、错误率
  ↓
冻结或隔离旧主，防止继续写入
  ↓
确认备节点数据位置和复制延迟
  ↓
切流：DNS / GSLB / 网关 / 注册中心 / 客户端配置
  ↓
验证核心读写、业务成功率和数据一致性
  ↓
故障节点修复后重新同步，再决定是否回切
```

无状态服务可以更激进地自动切换；数据库、缓存主节点、协调服务、资金链路则要谨慎得多。
自动切换不是越快越好，误切和双写可能比短暂停机更难收拾。

## 为什么有状态系统比无状态系统更难切

无状态服务切流相对简单，重新拉起实例、把流量拨过去就行。

有状态系统难在：

- 数据复制有没有追平
- 旧主是不是还可能继续写
- 切换后客户端会不会读到旧数据

所以数据库、缓存主节点、协调服务这类系统，故障转移往往比无状态应用谨慎得多。

这里会牵出一个关键词：**fencing**。

fencing 的目标是：切到新主以后，旧主不能再继续写。

常见做法包括：

- 通过租约或写入令牌控制谁有写权限。
- 用仲裁节点确认只有一个主。
- 旧主恢复后先隔离，只允许同步数据，不能立刻提供写服务。
- 对共享存储或关键写路径做强制隔离，避免双主写。

没有 fencing 的自动切换很危险：网络分区时，旧主以为自己还活着，新主也开始写，
最终就会出现双主和数据冲突。

## 回切不是把流量切回去这么简单

很多预案只写“主站恢复后回切”，但回切本身也是一次高风险变更。

回切前至少要确认：

- 故障原因已经修复，不会切回去又复发。
- 新旧站点数据已经追平，或者差异已经有补偿方案。
- 客户端、缓存、连接池、注册中心里的旧地址已经清理。
- 回切窗口、灰度比例、回滚方案都已经明确。
- 回切后继续观察业务成功率、复制延迟和错误率。

更稳的做法是先小流量验证，再逐步放量。不要因为“原主恢复了”就直接把全量流量打回去。

## 一个更稳的回答顺序

如果被问“容灾和多活怎么理解”，可以直接按这条线答：

1. 先定业务的 `RTO` 和 `RPO` 目标。
2. 如果只是防单机故障，主备或集群就够了。
3. 如果要防机房级故障，就上同城灾备。
4. 如果要防区域级故障，再考虑异地灾备。
5. 如果还要求切换快、资源利用高，才进一步考虑多活。
6. 不管是哪种方案，最后都要落到故障检测、切流、数据复制和演练验证。

这样就不是“背名词”，而是在用目标反推架构层级。

## 容灾演练要验证什么

容灾能力不是架构图证明的，而是演练数据证明的。

一次有效演练至少要记录：

| 指标           | 说明                               |
| -------------- | ---------------------------------- |
| 故障发现时间   | 故障发生到告警触发用了多久         |
| 决策时间       | 告警到确认切换用了多久             |
| 切流时间       | 流量真正切到备用站点用了多久       |
| 实际 RTO       | 从故障到核心服务恢复用了多久       |
| 实际 RPO       | 切换后数据最多回退了多久           |
| 复制延迟       | 主备或多站点之间是否存在积压       |
| 业务成功率     | 下单、支付、登录等核心链路是否恢复 |
| 数据一致性校验 | 订单、流水、库存、消息是否对得上   |

演练还要覆盖回切，而不只是切过去。很多系统能切到备站，但回切时才暴露数据差异、缓存污染和配置漂移。

## 容易踩的坑

### 把备份当成容灾

备份只能帮你恢复数据，不代表服务能快速恢复。

### 把主从复制当成 RPO 一定为 0

如果复制是异步的，切换时就可能丢最后那一段还没同步的数据。

### 把多活想得太轻

多活最难的不是部署，而是：

- 路由
- 一致性
- 冲突解决
- 演练

### 没有切流演练

再漂亮的容灾架构，没有演练过，线上切的时候还是会慌。

### 以为 DNS 切换是瞬时的

DNS 受缓存、TTL、客户端解析策略影响，切换不会保证所有用户马上生效。
核心链路更常见的是配合 GSLB、网关、注册中心或客户端重试一起做。

### 旧主恢复后直接纳管

旧主故障恢复后如果没有重新同步和 fencing 校验，直接重新加入写路径，可能造成数据回退或双写。

## 小结

- 冗余解决“有没有备用”，容灾解决“大故障后怎么恢复”，多活解决“多个站点能不能同时承载流量”。
- 主备、同城灾备、异地灾备、同城多活、异地多活是不同层级，不是同义词。
- 多活不是默认答案，只有业务目标和团队能力都撑得住时才值得做。
- 故障转移不是一句“自动切换”，有状态系统还要处理复制延迟、fencing、回切和数据校验。
- 容灾题答得稳不稳，关键在于能不能从 `RTO/RPO` 反推方案，而不是直接堆架构名词。
