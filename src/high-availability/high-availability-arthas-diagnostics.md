# Arthas 怎么做线上诊断？

> 线上机器不让你重启、不让你加日志重新发版，你只有一个正在跑的 Java 进程——这时候 Arthas 干的事，就是在不停机的前提下，把这个进程"扒开"给你看。

## 先搞清楚：为什么不是 jstack/jmap 一把梭

`jstack`、`jmap`、`jinfo` 这些 JDK 自带工具能做的事情，本质上都是**给你一个静态快照**：线程此刻在哪、堆里有什么对象、参数是什么值。它们够用的场景是死锁、内存暴涨这种"状态问题"。

但线上更常见的是这类问题：

- 某个接口偶尔慢，但慢的时候你根本不知道是哪一行代码卡住了；
- 某个方法的入参明明看起来没问题，出参却是错的，但你没法当场加日志再发版；
- 怀疑某个类被人重复加载了，或者线上跑的代码根本不是你以为的那个版本；
- CPU 打满了，但 `top` 只能告诉你哪个线程占用高，不知道它在执行什么业务逻辑。

这些问题的共同点是：**你需要在方法执行的那一刻，动态地看到入参、出参、调用链和耗时**，而且不能重启、不能重新打包发布。这正是 Arthas 要解决的场景——它通过 Java Agent + 字节码增强，在运行时给目标类的方法插入探针，采集完再把字节码还原，进程全程不停。

一句话概括分工：

| 工具                        | 解决什么                                                  | 代价                                      |
| --------------------------- | --------------------------------------------------------- | ----------------------------------------- |
| `jstack`                    | 某一时刻全部线程在哪                                      | 快照，看不到"接下来会怎样"                |
| `jmap`                      | 堆里对象分布、生成 dump                                   | dump 大堆时会 STW，谨慎用                 |
| JFR（Java Flight Recorder） | 低开销、长时间的事件采集，事后分析                        | 需要提前开启或支持动态 attach，分析要离线 |
| Arthas                      | 运行时动态观测方法级的入参/出参/耗时/调用链，还能临时改值 | 现场增强有性能开销，改配置有风险          |

它们不是替代关系，是分工：**jstack/jmap 拍照，JFR 录像，Arthas 是带着示波器现场扎进去看**。排障顺序一般是：先用 `top`/`jstack`/监控大盘定位大方向（是线程问题还是内存问题还是某个接口慢），再用 Arthas 精确到具体方法。

## 怎么开始：attach 到进程

```bash
# 下载后直接运行，会列出当前机器所有 Java 进程供你选择
java -jar arthas-boot.jar
```

选中进程号后就进入了 Arthas 的交互式命令行，后面所有命令都在这个会话里敲。退出用 `quit`（只退出会话，被观测的进程不受影响），`stop` 才是彻底卸载 Arthas。

## 第一步永远是 dashboard：看全局

进程刚接上，第一个命令基本都是：

```bash
dashboard
```

它会持续刷新展示：线程列表（按 CPU 使用率排序）、内存分区使用率、GC 次数与耗时、加载的类和线程总数。它其实就是把 `jstat` + `jstack` + `top -H` 揉在一起显示。

看 `dashboard` 主要是回答两个问题：CPU 是不是被某几个线程占满了？GC 是不是过于频繁（这一列会显示 YGC/FGC 次数和累计耗时，异常高就要怀疑内存问题）。

## thread：揪出那个占 CPU 的线程

`dashboard` 里如果发现某个线程 CPU 占用异常高，直接用它的 ID 深挖：

```bash
thread 1                # 查看具体某个线程的栈
thread -n 3             # 按 CPU 使用率列出最忙的 3 个线程
thread -b               # 找出当前阻塞其他线程的那个线程（排查锁竞争很好用）
```

`thread -n 3` 是 CPU 打满时最常用的一条：它会在一段采样窗口内统计各线程的 CPU 占用，直接告诉你"这几个线程最忙"，比自己拿 `jstack` 输出去对 `top -H` 的线程 ID 快得多。

## jvm：一眼看完虚拟机参数和运行时信息

```bash
jvm
```

把 `-XX` 参数、类加载信息、GC 信息、线程数、系统属性汇总展示，排查"这个参数到底有没有生效"时比翻启动脚本靠谱——启动脚本上写的参数不代表实际生效的参数（比如被后加的参数覆盖过），`jvm` 命令看到的是运行时真实值。

## sc / sm：类到底加载了没有、是哪个版本

`sc`（Search Class）和 `sm`（Search Method）用来确认"线上跑的到底是不是我以为的那份代码"。

```bash
sc -d com.demo.UserService          # 查看类的详细信息，包括类加载器、代码源路径
sc *UserService*                    # 模糊匹配类名
sm com.demo.UserService *           # 查看这个类的所有方法签名
sm -d com.demo.UserService getUser  # 查看某个方法的详细字节码信息（含行号表）
```

排查过一种典型场景：某次发版后线上行为没变，怀疑代码没生效——`sc -d` 里的 `code source` 字段能告诉你这个类是从哪个 jar 包加载的，跟本次发布的包路径、修改时间一对，立刻知道是不是发漏了或者被别的类加载器加载了旧版本。

## watch：看方法的入参、出参、异常

这是 Arthas 里最常被面试问到、也最常用的命令。语法：

```bash
watch 类名 方法名 表达式 [事件类型] [条件表达式]
```

举个例子，怀疑 `getUser` 返回了不对的数据：

```bash
watch com.demo.UserService getUser "{params, returnObj}" -x 2
```

- `{params, returnObj}` 是 OGNL 表达式，`params` 是入参数组，`returnObj` 是返回值；
- `-x 2` 控制对象展开的深度，默认是 1，太浅经常看不到关键字段，但也别调太大，容易刷屏还耗资源；
- 不加事件类型默认在方法返回后触发，想看异常就用 `-e`（抛异常时触发）或者 `-b`（进入方法时，此时还没有 `returnObj`）。

再加条件表达式，只看你关心的那次调用，减少噪音：

```bash
watch com.demo.UserService getUser "{params[0], returnObj}" "params[0]==123" -x 2
```

只在 `userId==123` 这次调用触发，不然生产环境这个方法一秒几百次调用，日志能把你淹死。

`watch` 的本质是给方法织入一段观测代码，**它是有性能开销的**，而且开的时间越长、匹配越宽泛（比如不加条件表达式、类名方法名用了通配符匹配一大批方法），开销越大。所以正确用法是：**收窄范围、限制次数、看完就关**。

```bash
watch com.demo.UserService getUser "{params, returnObj}" -n 5   # 只抓 5 次就自动停止
```

## trace / stack：定位调用链和耗时瓶颈

`watch` 看的是一个方法的输入输出，`trace` 和 `stack` 看的是调用链路。

```bash
trace com.demo.OrderService createOrder
```

会输出这次调用内部经过的每一层方法调用和各自耗时，类似把一次调用的火焰图用文本树的形式打出来，非常适合定位"这个接口 200ms，到底卡在哪一层"：

```text
`---ts=2026-07-14 10:00:01;thread_name=http-nio-8080-exec-1;cost=182.4ms
    `---[182.4ms] OrderService:createOrder()
        +---[5.2ms] InventoryService:checkStock()
        +---[168.9ms] PaymentService:pay()      # 明显是这里慢
        `---[8.1ms] OrderMapper:insert()
```

`trace` 默认只显示耗时超过一定阈值（`--skip-jdk-method` 之类的选项）的调用，也支持类似 `watch` 的条件过滤，避免输出爆炸。

`stack` 则是反过来：不知道这个方法是被谁调用的时候用，回答"这次调用是从哪条链路进来的"：

```bash
stack com.demo.OrderService createOrder
```

三者配合起来是一套组合拳：**先 `trace` 定位到耗时的那一层方法，再用 `watch` 看那一层方法的具体入参出参有没有问题，需要的话用 `stack` 反查调用来源。**

## profiler：CPU 火焰图

`trace` 适合"我知道大概是哪条链路慢，帮我细化"；但如果连大概方向都没有，或者怀疑是大量方法叠加导致的整体慢（而不是某一个方法特别慢），就该上火焰图。

```bash
profiler start                         # 开始采样，默认采 CPU
# ... 让问题复现一段时间 ...
profiler stop --format html            # 停止并生成 html 火焰图
```

生成的 svg/html 打开后，横轴是采样次数占比（越宽代表这个方法及其子调用占用 CPU 时间越多），纵轴是调用栈深度。**看火焰图的核心技巧是找"平顶山"**——顶部很宽但没有再往上分叉的方块，说明 CPU 时间花在这个方法自己身上，而不是它调的子方法，这类方法是重点排查对象。

`profiler` 底层用的是 async-profiler，采样式而非插桩式，所以开销比 `watch`/`trace` 小很多，适合更长时间地开着采集，但也不是零开销，长期常驻建议用独立的 APM/JFR 方案，Arthas 的 profiler 更适合应急定位。

## OGNL 改配置：能救命，但也能捅娄子

Arthas 支持通过 OGNL 表达式直接读写目标 JVM 里的静态字段、甚至调用方法，典型用法是应急改一个开关：

```bash
# 查看某个类里的静态字段
ognl '@com.demo.FeatureSwitch@ENABLE_NEW_LOGIC'

# 直接改掉它的值——不重启就能让开关生效
ognl '@com.demo.FeatureSwitch@ENABLE_NEW_LOGIC=false'
```

这在"线上某个开关配错了，业务方等着恢复"的场景下确实能救命，比走完整的配置中心发布流程快得多。但必须清楚它的边界：

- **改的是内存里的值，不是配置源**。进程重启、发新版本，这次修改就没了，事后一定要把真正的配置改对，否则这只是临时止血；
- **只能改 Arthas 能"够得到"的静态字段和可访问方法**，业务逻辑深层的私有状态、局部变量改不了；
- **没有变更审计**，谁在什么时候通过 OGNL 改了什么值，默认只在 Arthas 的会话日志里，不会像走配置中心那样留下规范的变更记录，事后排查容易两眼一抹黑；
- **表达式本身可以调用任意方法**，如果这台机器的 Arthas 权限没管住，相当于给了一个"在线执行任意代码"的入口。

所以团队里但凡用 OGNL 做过线上改值，都应该有一条铁律：**改完立刻在群里/工单里报备，并在配置源头同步修正**，不能让内存态和配置源长期不一致。

## 权限和安全：Arthas 不是随便谁都能连

Arthas attach 到进程后，能做的事情几乎等同于拥有这个进程的完全控制权——看所有内存数据、改静态变量、甚至动态替换类的实现（`redefine`/`retransform`）。这意味着几件事必须管住：

| 风险点      | 说明                                     | 建议                                                                                 |
| ----------- | ---------------------------------------- | ------------------------------------------------------------------------------------ |
| attach 权限 | 谁能连上生产进程                         | 生产环境限制能执行 `arthas-boot.jar` 的账号，走跳板机+审批                           |
| 命令审计    | 谁在什么时候执行了什么命令               | Arthas 支持 telnet/http 两种连接方式并有操作日志，生产建议开启并集中采集             |
| 数据脱敏    | `watch`/`trace` 会把入参出参明文打印出来 | 涉及密码、身份证、卡号等敏感字段的方法，观测前想清楚会不会把敏感数据打到日志或终端里 |
| 性能影响    | 增强字节码、长时间开 `watch`/`trace`     | 收窄条件、限制次数、用完立刻关闭                                                     |

**关于 `redefine`：这是风险最高的一个能力，慎用。** 它能在运行时把某个类的字节码整体替换掉，理论上可以做到"不重启修复一个 bug"。但它有几个硬限制和风险，决定了它基本只该在万不得已、且你完全清楚后果的时候用：

- 只能改方法体，**不能新增/删除字段和方法**，也不能改类的继承关系，改动稍微大一点就会失败；
- 替换后不会体现在源码仓库里，**代码和线上运行的字节码从这一刻起就不一致**，如果没有人记录，下一个接手的人会被坑得莫名其妙；
- 一旦这个类之前被其他方式增强过（比如某些 APM 探针也在动它的字节码），`redefine` 可能冲突失败，报错也不总是直观；
- 这本质是在生产环境做"热修复"，跳过了代码评审、测试、灰度发布的全部流程，出问题的排查成本极高。

更负责任的做法是：`redefine` 只用来验证"改这几行是不是真能解决问题"，验证完立刻走正常发布流程把改动落到代码里，而不是让它成为长期存在于内存里的"隐藏补丁"。

## replay 回放要不要谨慎？

`watch`/`trace` 一类命令本身不会重新调用业务方法（不是 mock 一次调用去跑，而是在真实调用发生时"顺手"采集数据），所以它不会对业务产生"多跑了一次"的副作用，这点比很多人想象的要安全。但真正需要小心的是：

- 观测条件写错，导致匹配到了大范围的方法（比如类名或方法名用了过宽的通配符），会让大量线上调用都被插桩，性能开销叠加起来可能拖慢整个服务；
- 表达式里如果调用了有副作用的方法（比如在 `watch` 的表达式里手滑调用了一个会修改状态的 getter），会实实在在地影响业务数据，OGNL 表达式的"读"和"写"边界一定要分清楚；
- 长时间不清理的增强点会持续消耗资源，即便当前没人在看输出。

## 一条实用的排障顺序

线上接口变慢或者报错，可以按这个顺序走：

1. 先看监控大盘和错误日志，判断问题范围（单机还是全局、某个接口还是全站）；
2. `dashboard` 看整体 CPU、内存、GC 是否异常；
3. CPU 高就 `thread -n 3` 找忙线程，`thread -b` 找锁竞争；
4. 怀疑某个接口慢，用 `trace` 定位到具体哪一层方法耗时高；
5. 定位到方法后用 `watch` 看这次调用的入参出参是否符合预期；
6. 怀疑是代码没生效或者加载了旧版本，用 `sc -d` 核实类的来源；
7. 需要长时间观察大范围性能问题，上 `profiler` 生成火焰图；
8. 确认是配置开关问题、且必须立刻止血，才考虑 `ognl` 改值，改完同步修正配置源；
9. `redefine` 留到最后一步，且事后必须走正式发布把改动落地到代码。

## 小结

1. Arthas 解决的是"不能重启、要看方法级入参出参和调用耗时"这类动态诊断问题，和 jstack（快照）、JFR（长时间低开销采集）是分工关系，不是互相替代。
2. 排障从 `dashboard` 摸全局开始，CPU 问题找 `thread -n`，慢请求先用 `trace` 定位耗时层，再用 `watch` 看具体入参出参，类版本问题用 `sc -d` 核实。
3. `watch`/`trace` 有真实性能开销，务必用条件表达式收窄范围、限制触发次数，看完就关。
4. `profiler` 火焰图适合定位没有明确方向的整体性能问题，看"平顶山"找自身耗时高的方法。
5. `ognl` 改配置和 `redefine` 改代码都只是应急止血手段，改完必须同步修正配置源/代码仓库，并留好操作审计，不能让内存态长期偏离真实状态。
