# 限流算法怎么选：固定窗口、滑动窗口、漏桶、令牌桶？

> 限流不是背 4 个算法的定义，而是先说清楚你要保护的是入口、下游，还是某个稀缺资源。

很多人一提限流就开始背：

- 固定窗口
- 滑动窗口
- 漏桶
- 令牌桶

但真实系统里先要回答的是这两个问题：

1. 你到底在限制什么
2. 你允许不允许短时间突发

这两个问题定了，算法基本就能选出来。

## 先定限流目标，不然算法选不准

同样是“限流”，目标可能完全不一样：

| 场景                       | 真正要保护的东西                 |
| -------------------------- | -------------------------------- |
| 网关入口                   | 不让突发流量把应用线程池打满     |
| 登录 / 发短信              | 控制用户或 IP 的调用频率         |
| 调第三方支付 / 短信 / 风控 | 严格遵守对方给你的 QPS 配额      |
| 库存扣减 / 秒杀接口        | 保护数据库、Redis、MQ 的承压能力 |

所以限流不仅要选算法，还要选对象：

- 按 IP 限
- 按用户限
- 按接口限
- 按租户限
- 按热点参数限

## 四种常见算法到底差在哪

先给一张总表：

| 算法     | 核心特点     | 适合场景               | 主要问题             |
| -------- | ------------ | ---------------------- | -------------------- |
| 固定窗口 | 实现最简单   | 粗粒度保护、低成本实现 | 窗口边界突刺明显     |
| 滑动窗口 | 统计更平滑   | 网关入口、接口限频     | 实现和存储更复杂     |
| 漏桶     | 出流速率固定 | 严格整形、对接外部配额 | 不擅长应对突发       |
| 令牌桶   | 允许一定突发 | 大多数业务入口限流     | 桶容量和发牌速率要调 |

## 固定窗口：最简单，但边界最容易刺穿

固定窗口的思路非常直接：

- 每 1 秒或每 1 分钟做一个窗口
- 窗口内计数
- 超过阈值就拒绝

比如“1 秒最多 100 个请求”，实现起来最省事。

### 它适合什么场景

- 对精度要求没那么高
- 只想快速做一个第一道防线
- 请求量不算特别大

### 它最大的坑是什么

窗口边界突刺。

比如限制“每秒 100 个请求”：

- `10:00:00.999` 来了 100 个
- `10:00:01.001` 又来了 100 个

从窗口视角看都没超，但真实两毫秒里放过了 200 个。

所以固定窗口适合“简单挡一下”，不适合很精细的入口治理。

## 滑动窗口：更平滑，适合做网关级限流

滑动窗口本质上是把大窗口拆成很多小格子，再随着时间滚动统计。

这样好处是：

- 不容易被窗口边界刺穿
- 可以更接近真实的瞬时流量

### 什么时候优先考虑它

- 入口流量大，而且波峰波谷明显
- 你更在意“统计平滑”
- 想按秒级甚至更细粒度做控制

### 代价是什么

- 计数更复杂
- Redis / 内存占用会更高
- 多实例下需要更仔细地处理时间窗口和共享状态

如果你是在网关层做全局限流，滑动窗口通常比固定窗口更稳。

## 漏桶：适合“必须匀速放行”的场景

漏桶可以理解成：

- 请求先进桶
- 桶按固定速率往外漏

这意味着它更像“整形”，而不是“允许突发后再均摊”。

### 什么时候它很合适

最典型的是对外部系统配额非常严格的时候，比如：

- 第三方接口明确规定每秒最多多少次
- 发短信 / 发邮件通道要求匀速发
- 你要把上游突发流量削成稳定流量

### 它的问题是什么

它对突发不友好。

就算系统现在其实还有余量，漏桶也不会因为你突然多来一点流量就多放行，它只认固定出流速率。

所以如果业务允许“平时 100 QPS，但偶尔 300 QPS 突发也能接住”，漏桶往往不如令牌桶。

## 令牌桶：大多数业务限流的默认选项

令牌桶的思路是：

- 系统按固定速率往桶里放令牌
- 请求拿到令牌才能通过
- 桶里有积攒的令牌时，可以扛住一小段突发

### 它为什么最常用

因为它兼顾两件事：

1. 限制长期平均速率
2. 允许一定程度的短时突发

这很符合大多数业务入口流量的特点。

比如平时 100 QPS，但活动页突然冲一波 200 QPS，如果桶里还有积累的令牌，就可以让这一小波过去，而不是全打死。

### 它最需要调什么

两个参数：

- 发牌速率
- 桶容量

发牌速率决定长期平均吞吐，桶容量决定你能容忍多大的短时突发。

## 选算法时，别脱离“允许不允许突发”

这是最实用的一条判断线：

| 诉求                           | 更常见的选择 |
| ------------------------------ | ------------ |
| 只想低成本挡一下               | 固定窗口     |
| 想让统计更平滑                 | 滑动窗口     |
| 必须严格匀速对外放行           | 漏桶         |
| 想限制平均速率，但允许短时突发 | 令牌桶       |

如果你实在要一句话回答：

**入口限流大多数优先令牌桶或滑动窗口；严格对外配额控制更偏漏桶。**

## 单机限流和分布式限流不是一回事

### 单机限流

如果你的限额本来就是“每个实例自己控制”，那单机限流就够了。

典型做法：

- 本地计数器
- `Guava RateLimiter`
- `Bucket4j`

优点是性能高、实现简单。

### 分布式限流

如果你的限额是全局配额，比如：

- 整个集群一共只能放 2000 QPS
- 同一个用户在所有实例加起来每分钟只能调用 5 次

那就得共享限流状态。

最常见的办法是：

- Redis + Lua
- Sentinel
- 网关限流器

这里要注意一个细节：

**不是上了微服务就一定要分布式限流。**

如果总配额可以按实例数均分，单机限流反而更简单，也更便宜。

## 面试里最好顺手补一句“限流之后怎么办”

限流不是只有“拒绝请求”这一种处理方式。

常见处理策略包括：

- 直接返回 `429`
- 排队等待
- 返回静态兜底结果
- 只保核心功能，非核心请求丢掉

所以限流题答完整，通常会变成：

**限什么、怎么限、限住后怎么处理。**

## 容易踩的坑

### 只讲算法，不讲限流对象

按 IP、按用户、按接口、按租户，结果差别非常大。

### 只会限制入口，不会保护下游

如果第三方配额是 200 QPS，而你网关给放到了 500 QPS，系统内部没挂，下游先被你打死了。

### 限流没有配监控

至少要盯这些指标：

- 被限流请求数
- 放行请求数
- 限流命中比例
- 限流后用户报错率
- 下游资源指标是否真的缓下来

限流规则不是写完就结束，得看效果。

## 小结

- 限流先看目标：你是保护入口、保护下游，还是控制某个稀缺资源。
- 固定窗口简单但边界突刺明显，滑动窗口更平滑。
- 漏桶适合严格匀速出流，令牌桶更适合大多数允许突发的业务场景。
- 单机限流和分布式限流的选择，取决于你的配额是不是全局共享。
- 限流题真正要答完整，得把“限什么、怎么限、限住后怎么处理”一起说出来。
