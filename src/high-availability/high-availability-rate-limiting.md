# 限流算法怎么选：固定窗口、滑动窗口、漏桶、令牌桶？

> 限流不是背 4 个算法的定义，而是先说清楚你要保护的是入口、下游，还是某个稀缺资源。

很多人一提限流就开始背：

- 固定窗口
- 滑动窗口
- 漏桶
- 令牌桶

但真实系统里先要回答的是这两个问题：

1. 你到底在限制什么
2. 你允许不允许短时间突发

这两个问题定了，算法基本就能选出来。

## 先定限流目标，不然算法选不准

同样是“限流”，目标可能完全不一样：

| 场景                       | 真正要保护的东西                 |
| -------------------------- | -------------------------------- |
| 网关入口                   | 不让突发流量把应用线程池打满     |
| 登录 / 发短信              | 控制用户或 IP 的调用频率         |
| 调第三方支付 / 短信 / 风控 | 严格遵守对方给你的 QPS 配额      |
| 库存扣减 / 秒杀接口        | 保护数据库、Redis、MQ 的承压能力 |

所以限流不仅要选算法，还要选对象：

- 按 IP 限
- 按用户限
- 按接口限
- 按租户限
- 按热点参数限

限流最好先画出保护对象：

```text
用户请求
  ↓
网关入口：保护整体入口和应用实例
  ↓
业务接口：保护高成本接口、热点参数、租户配额
  ↓
下游调用：保护数据库、缓存、MQ、第三方 API
  ↓
核心资源：库存、优惠券、短信通道、支付通道
```

不同层限流不是互相替代，而是分工：

| 层次       | 更适合限制什么                        | 常见做法                          |
| ---------- | ------------------------------------- | --------------------------------- |
| CDN / WAF  | 恶意流量、爬虫、DDoS 前置清洗         | IP 规则、黑白名单、挑战验证       |
| 网关       | 全站入口、接口、租户、调用方配额      | 令牌桶、滑动窗口、网关限流插件    |
| 应用服务   | 业务语义更强的资源保护                | 用户 / 订单 / 商品 / 热点参数限流 |
| 下游客户端 | 第三方接口、数据库、缓存、MQ 调用保护 | 漏桶、并发限制、连接池上限        |
| 单机实例   | 防止本实例线程池、连接池被打满        | 本地计数器、信号量、RateLimiter   |

所以面试里不要只说“在网关做限流”。网关适合挡入口流量，但它未必知道“某个商品是热点”“某个租户买了 100 QPS 配额”“某个第三方短信通道只能匀速 50 QPS”。这些需要业务层或调用方限流一起补。

## 四种常见算法到底差在哪

先给一张总表：

| 算法     | 核心特点     | 适合场景               | 主要问题             |
| -------- | ------------ | ---------------------- | -------------------- |
| 固定窗口 | 实现最简单   | 粗粒度保护、低成本实现 | 窗口边界突刺明显     |
| 滑动窗口 | 统计更平滑   | 网关入口、接口限频     | 实现和存储更复杂     |
| 漏桶     | 出流速率固定 | 严格整形、对接外部配额 | 不擅长应对突发       |
| 令牌桶   | 允许一定突发 | 大多数业务入口限流     | 桶容量和发牌速率要调 |

## 固定窗口：最简单，但边界最容易刺穿

固定窗口的思路非常直接：

- 每 1 秒或每 1 分钟做一个窗口
- 窗口内计数
- 超过阈值就拒绝

比如“1 秒最多 100 个请求”，实现起来最省事。

### 它适合什么场景

- 对精度要求没那么高
- 只想快速做一个第一道防线
- 请求量不算特别大

### 它最大的坑是什么

窗口边界突刺。

比如限制“每秒 100 个请求”：

- `10:00:00.999` 来了 100 个
- `10:00:01.001` 又来了 100 个

从窗口视角看都没超，但真实两毫秒里放过了 200 个。

所以固定窗口适合“简单挡一下”，不适合很精细的入口治理。

## 滑动窗口：更平滑，适合做网关级限流

滑动窗口本质上是把大窗口拆成很多小格子，再随着时间滚动统计。

这样好处是：

- 不容易被窗口边界刺穿
- 可以更接近真实的瞬时流量

### 什么时候优先考虑它

- 入口流量大，而且波峰波谷明显
- 你更在意“统计平滑”
- 想按秒级甚至更细粒度做控制

### 代价是什么

- 计数更复杂
- Redis / 内存占用会更高
- 多实例下需要更仔细地处理时间窗口和共享状态

如果你是在网关层做全局限流，滑动窗口通常比固定窗口更稳。

## 漏桶：适合“必须匀速放行”的场景

漏桶可以理解成：

- 请求先进桶
- 桶按固定速率往外漏

这意味着它更像“整形”，而不是“允许突发后再均摊”。

### 什么时候它很合适

最典型的是对外部系统配额非常严格的时候，比如：

- 第三方接口明确规定每秒最多多少次
- 发短信 / 发邮件通道要求匀速发
- 你要把上游突发流量削成稳定流量

### 它的问题是什么

它对突发不友好。

就算系统现在其实还有余量，漏桶也不会因为你突然多来一点流量就多放行，它只认固定出流速率。

所以如果业务允许“平时 100 QPS，但偶尔 300 QPS 突发也能接住”，漏桶往往不如令牌桶。

## 令牌桶：大多数业务限流的默认选项

令牌桶的思路是：

- 系统按固定速率往桶里放令牌
- 请求拿到令牌才能通过
- 桶里有积攒的令牌时，可以扛住一小段突发

### 它为什么最常用

因为它兼顾两件事：

1. 限制长期平均速率
2. 允许一定程度的短时突发

这很符合大多数业务入口流量的特点。

比如平时 100 QPS，但活动页突然冲一波 200 QPS，如果桶里还有积累的令牌，就可以让这一小波过去，而不是全打死。

### 它最需要调什么

两个参数：

- 发牌速率
- 桶容量

发牌速率决定长期平均吞吐，桶容量决定你能容忍多大的短时突发。

举个例子，如果接口长期能力是 `100 QPS`，但允许 2 秒的短时突发，可以粗略设：

```text
发牌速率 = 100 token/s
桶容量   = 200 token
```

这意味着系统长期不会超过 100 QPS，但如果桶里提前攒满了令牌，可以短时间放过最多约 200 个请求。

桶容量不是越大越好。桶太大，突发放行太多，后端还是会被打穿；桶太小，又会把本来可以承接的自然波峰打掉。

## 选算法时，别脱离“允许不允许突发”

这是最实用的一条判断线：

| 诉求                           | 更常见的选择 |
| ------------------------------ | ------------ |
| 只想低成本挡一下               | 固定窗口     |
| 想让统计更平滑                 | 滑动窗口     |
| 必须严格匀速对外放行           | 漏桶         |
| 想限制平均速率，但允许短时突发 | 令牌桶       |

如果你实在要一句话回答：

**入口限流大多数优先令牌桶或滑动窗口；严格对外配额控制更偏漏桶。**

## 限流阈值怎么定？

限流阈值不能只靠拍脑袋。

比较稳的顺序是：

1. 先看业务 SLO：接口目标成功率、P99 延迟和错误预算是多少；
2. 再看压测结果：系统在线性区间内稳定承载多少 QPS；
3. 再看瓶颈资源：数据库连接池、线程池、Redis、MQ、第三方配额谁最先到上限；
4. 最后留安全余量：不要把阈值贴着压测极限设置。

例如压测发现下单链路在 `5000 QPS` 时 P99 开始明显抬头，数据库连接池等待时间也开始变长，那入口限流阈值就不应该直接设到 5000。更常见的是按核心链路目标和安全系数向下收：

```text
稳定上限：5000 QPS
安全系数：70% ~ 80%
入口限流：3500 ~ 4000 QPS
```

如果还要按租户、用户或接口拆配额，就不能只看总 QPS：

| 维度      | 主要解决什么                         |
| --------- | ------------------------------------ |
| 全局 QPS  | 防止整体入口被打穿                   |
| 接口 QPS  | 防止高成本接口拖垮服务               |
| 用户 / IP | 防刷、限频、防止单个主体占满资源     |
| 租户      | SaaS 场景的资源公平和套餐配额        |
| 热点参数  | 防止单个商品、活动、优惠券被瞬间打爆 |
| 下游依赖  | 严格遵守第三方、数据库、缓存承载能力 |

限流阈值还要能动态调整。大促、灰度、故障、下游降级时，固定阈值往往不够用，需要配置中心、网关规则或流量治理平台支持灰度生效和快速回滚。

## 单机限流和分布式限流不是一回事

### 单机限流

如果你的限额本来就是“每个实例自己控制”，那单机限流就够了。

典型做法：

- 本地计数器
- `Guava RateLimiter`
- `Bucket4j`

优点是性能高、实现简单。

### 分布式限流

如果你的限额是全局配额，比如：

- 整个集群一共只能放 2000 QPS
- 同一个用户在所有实例加起来每分钟只能调用 5 次

那就得共享限流状态。

最常见的办法是：

- Redis + Lua
- Sentinel
- 网关限流器

这里要注意一个细节：

**不是上了微服务就一定要分布式限流。**

如果总配额可以按实例数均分，单机限流反而更简单，也更便宜。

### 分布式限流的几个坑

分布式限流的难点不是“共享计数”四个字，而是共享状态本身也会变成依赖。

常见风险如下：

| 风险                 | 说明                                         | 应对思路                                   |
| -------------------- | -------------------------------------------- | ------------------------------------------ |
| Redis / 中间件不可用 | 限流器本身挂了，业务不知道该放行还是拒绝     | 提前定义 fail-open / fail-closed 策略      |
| 热点限流 key         | 所有请求打到同一个 key，Redis 自己成瓶颈     | 分片 key、分桶统计、局部限流 + 全局校准    |
| 时间不一致           | 多实例时间偏差影响窗口统计                   | 统一使用服务端时间或降低对客户端时间的依赖 |
| Lua 脚本过重         | Redis 单线程执行脚本，复杂脚本会阻塞其他命令 | 脚本保持短小，避免大范围遍历               |
| Redis Cluster 跨槽   | 多 key 脚本在集群下受 slot 限制              | key 设计使用 hash tag 或避免跨 key 原子性  |
| 规则变更不一致       | 部分实例已更新限流规则，部分还没更新         | 规则版本号、灰度发布、配置变更审计         |

这里最容易被追问的是：限流系统故障时怎么办？

可以按业务风险选：

| 策略             | 含义                           | 适合场景                         |
| ---------------- | ------------------------------ | -------------------------------- |
| fail-open        | 限流器异常时先放行             | 用户体验优先，后端仍有兜底保护   |
| fail-closed      | 限流器异常时拒绝或强降级       | 防刷、防资损、第三方配额非常严格 |
| 本地兜底限流     | 分布式限流异常时退化成本地限流 | 大多数在线服务更稳妥             |
| 核心接口人工开关 | 关键链路保留手动降级和限流开关 | 大促、支付、库存、发券等场景     |

不要把“Redis 高可用”直接等同于“限流永远可用”。主从切换、网络抖动、热点 key、脚本阻塞都会影响限流判定。

## 面试里最好顺手补一句“限流之后怎么办”

限流不是只有“拒绝请求”这一种处理方式。

常见处理策略包括：

- 直接返回 `429`
- 排队等待
- 返回静态兜底结果
- 只保核心功能，非核心请求丢掉

可以再细一点：

| 处理方式       | 适合场景                           | 注意点                         |
| -------------- | ---------------------------------- | ------------------------------ |
| 直接拒绝 `429` | API、开放平台、明确可重试的客户端  | 最好带 `Retry-After` 或错误码  |
| 快速失败 `503` | 系统过载、下游不可用               | 避免客户端立刻无脑重试         |
| 短队列等待     | 流量尖峰很短，用户可接受少量等待   | 队列必须有长度和等待时间上限   |
| 返回兜底数据   | 推荐、评价、活动挂件等非核心读接口 | 兜底数据要明确过期和一致性边界 |
| 只保核心功能   | 大促、故障、资源紧张               | 业务要提前定义哪些功能可以牺牲 |
| 异步受理       | 非实时任务、导出、通知、补偿       | 要有状态查询、幂等和失败补偿   |

所以限流题答完整，通常会变成：

**限什么、怎么限、限住后怎么处理。**

## 限流和熔断有什么区别？

这两个词经常混在一起，但方向不一样：

| 机制 | 触发依据                       | 主要保护谁                 |
| ---- | ------------------------------ | -------------------------- |
| 限流 | 请求量、并发数、配额、热点参数 | 保护入口、下游和稀缺资源   |
| 熔断 | 错误率、慢调用比例、连续失败   | 保护调用方和正在故障的下游 |
| 降级 | 业务优先级、故障状态、资源紧张 | 保核心链路和用户可接受结果 |
| 隔离 | 资源池、线程池、连接池、信号量 | 防止局部依赖拖垮全局       |

限流通常发生在请求进入之前或调用下游之前；熔断通常发生在发现某个依赖持续异常之后。

例如：

- 秒杀入口流量太大：先限流；
- 营销服务错误率持续升高：熔断它；
- 营销服务熔断后：下单链路走降级，不再计算复杂优惠。

这几个动作配合起来，才是完整的稳定性治理。

## 项目里怎么落地更稳？

一个比较稳的限流方案通常分三层：

```text
第一层：入口粗限流
  - 全局 QPS、接口 QPS、租户 / IP / 用户限频

第二层：业务细限流
  - 商品、活动、优惠券、短信、支付通道等热点资源

第三层：下游保护
  - 数据库连接池、Redis、MQ、第三方接口配额
```

落地步骤可以这样拆：

1. 先按业务链路分级，确定哪些接口必须保、哪些可以降级。
2. 根据压测和下游配额定初始阈值。
3. 对入口、接口、租户、热点参数分别设计限流 key。
4. 单机限流优先保护本实例，分布式限流只用于全局共享配额。
5. 限流命中后返回明确错误码、重试建议或兜底结果。
6. 配置规则要能灰度、生效可观测、变更可审计。
7. 监控限流命中率、拒绝量、等待时间、下游资源是否恢复。

限流不是越严越好。限得太松，保护不了系统；限得太严，会把正常流量打掉，业务成功率和用户体验都会受影响。

## 容易踩的坑

### 只讲算法，不讲限流对象

按 IP、按用户、按接口、按租户，结果差别非常大。

### 只会限制入口，不会保护下游

如果第三方配额是 200 QPS，而你网关给放到了 500 QPS，系统内部没挂，下游先被你打死了。

### 限流没有配监控

至少要盯这些指标：

- 被限流请求数
- 放行请求数
- 限流命中比例
- 限流后用户报错率
- 下游资源指标是否真的缓下来
- 限流器自身延迟和错误率
- Redis / 中间件限流依赖是否异常
- 规则变更前后的成功率和 P99

限流规则不是写完就结束，得看效果。

### 所有接口用同一个阈值

查询、下单、支付、导出、短信发送的成本完全不同。

统一阈值很容易出现两个问题：

- 低成本接口被过度限制；
- 高成本接口仍然能把下游打穿。

### 只做全局限流，不做热点参数限流

总 QPS 没超，不代表某个热点商品、活动或优惠券没被打爆。

秒杀、抢券、热点商品详情这类场景，往往要按资源 ID 做更细粒度限制。

### 限流器异常时没有预案

分布式限流依赖 Redis、配置中心或网关组件。它们异常时，是放行、拒绝、退化成本地限流，必须提前定好。

## 小结

1. 限流先看目标：你是保护入口、保护下游，还是控制某个稀缺资源。
2. 固定窗口简单但边界突刺明显，滑动窗口更平滑；漏桶适合严格匀速，令牌桶适合允许短时突发。
3. 阈值要从 SLO、压测结果、下游配额和安全余量反推，不要拍脑袋写死。
4. 单机限流和分布式限流的选择，取决于配额是不是全局共享；分布式限流还要考虑 Redis、中间件和规则变更故障。
5. 限流题真正要答完整，得把“限什么、怎么限、限住后怎么处理、怎么观测效果”一起说出来。
