# 慢 SQL、深分页、索引优化如何串成排查链路？

> 慢 SQL、深分页、索引优化不是三道分开的面试题，它们在真实排查里通常就是同一条链。

很多人答数据库性能题时会切成三块：

- 慢 SQL 怎么看
- 深分页怎么优化
- 索引怎么设计

但线上排查时，这三件事往往连在一起。

比如一个订单列表接口慢，最后很可能会落到：

1. 先发现慢 SQL
2. 再发现它是 `LIMIT offset, size` 深分页
3. 再发现排序和过滤没有命中合适索引

所以更稳的回答方式，不是背散点，而是把它们串成一条证据链。

## 第一步：先确认慢的是哪条 SQL

不要上来就讲索引。

先用证据把慢 SQL 找出来：

- 慢查询日志
- APM / 链路追踪
- 数据库监控
- 应用侧接口埋点

重点先确认三件事：

- 是高频 SQL 慢，还是偶发 SQL 慢
- 是数据库执行慢，还是连接等待 / 网络传输慢
- 是稳定慢，还是某些页码、某些条件下才慢

如果应用显示 3 秒，但数据库执行只有 20ms，那就别把锅先甩给 SQL。

慢查询日志里有几类指标特别值得看：

| 指标            | 说明                               |
| --------------- | ---------------------------------- |
| `Query_time`    | SQL 总耗时，先判断是不是慢查询本身 |
| `Lock_time`     | 等锁时间，判断慢是不是锁等待导致   |
| `Rows_examined` | 扫描行数，判断有没有大量无效扫描   |
| `Rows_sent`     | 返回行数，和扫描行数对比看过滤效率 |
| 执行频次        | 单次不慢但高频，也可能拖垮数据库   |

一个 SQL 值不值得优先治理，不只看单次耗时。
更常见的排序是：

**总耗时 = 单次耗时 × 执行频次。**

低频 5 秒的后台 SQL 当然要管，但高频 80ms 的核心链路 SQL 也可能是数据库压力大头。

## 第二步：先看执行计划，再决定是改 SQL 还是改索引

定位到 SQL 后，第一步通常是：

```sql
EXPLAIN SELECT ...;
EXPLAIN FORMAT=JSON SELECT ...;
```

重点看：

| 字段       | 要看什么                                             |
| ---------- | ---------------------------------------------------- |
| `type`     | 是不是 `ALL`、`index` 这种大范围扫描                 |
| `key`      | 实际用了哪个索引                                     |
| `rows`     | 预估扫描多少行                                       |
| `filtered` | 回来之后还要过滤多少                                 |
| `Extra`    | 是否有 `Using temporary`、`Using filesort`、大量回表 |

这里和本站
[`执行计划 EXPLAIN 怎么看？`](/database/mysql/mysql-explain.html)
是直接连着的。

如果是 MySQL 8.0.18 及之后的版本，还可以用 `EXPLAIN ANALYZE` 看真实执行耗时和实际行数。

```sql
EXPLAIN ANALYZE
SELECT id, user_id, status, create_time
FROM t_order
WHERE user_id = 10001
ORDER BY create_time DESC
LIMIT 20;
```

它和普通 `EXPLAIN` 的区别是：

| 工具              | 主要价值                         |
| ----------------- | -------------------------------- |
| `EXPLAIN`         | 预估执行计划，不真正执行查询     |
| `EXPLAIN ANALYZE` | 真正执行并输出实际耗时与实际行数 |

如果版本较老，也可以结合慢查询日志、Performance Schema 或 APM 观察耗时。
`SHOW PROFILE` 在老版本里能看资源消耗，但已经不适合作为新系统的首选方案。

## 第三步：别只问“有没有索引”，要问“索引为什么没帮上忙”

很多 SQL 其实不是完全没索引，而是：

- 联合索引顺序不对
- 过滤条件和排序字段断开了
- 范围条件卡在联合索引中间
- 对字段做了函数、计算或隐式类型转换
- `LIKE '%xxx'` 让 B+ 树没法定位

所以索引优化真正要看的不是“有无”，而是：

**访问路径是不是和查询模式对齐。**

一个很常见的误区是只看到 `key` 不为空，就以为“已经走索引了”。
其实：

- 走错索引也可能很慢
- 全索引扫描也可能很慢
- 走了索引但回表过多，一样可能很慢

举个简化例子。

订单表常见查询是：

```sql
SELECT id, status, amount, create_time
FROM t_order
WHERE user_id = ?
  AND status = ?
ORDER BY create_time DESC
LIMIT 20;
```

如果只有单列索引 `idx_user_id(user_id)`，数据库能先按用户过滤，但还可能要继续筛状态、排序并回表。

更贴合访问路径的索引通常是：

```sql
CREATE INDEX idx_order_user_status_time
ON t_order(user_id, status, create_time);
```

这个优化背后的思路不是“多加一个索引”，而是让索引顺序对齐：

1. 先等值过滤 `user_id`。
2. 再等值过滤 `status`。
3. 最后按 `create_time` 顺序取最新记录。

如果查询字段固定，还可以把列表展示字段做成覆盖索引的一部分，但要权衡写入成本。

## 第四步：深分页为什么总和慢 SQL 一起出现

深分页慢，根本原因不是“页码大了不吉利”，而是数据库得先扫过前面的海量记录。

典型 SQL：

```sql
SELECT * FROM t_order
WHERE user_id = ?
ORDER BY id DESC
LIMIT 1000000, 20;
```

这种写法的问题通常有两层：

1. `offset` 很大，前面 100 万行都要先跳过去
2. 如果还不能覆盖索引，就会出现大量回表

所以深分页问题通常不是单独一题，而是慢 SQL 的一种高频形态。

## 第五步：深分页优化，本质是减少“无效扫描”和“无效回表”

常见优化思路有 4 类：

| 思路            | 适合场景               |
| --------------- | ---------------------- |
| 游标 / 范围查询 | 只需要上一页 / 下一页  |
| 子查询定位起点  | 仍要兼容传统页码       |
| 延迟关联        | 大偏移量但仍要拿完整行 |
| 覆盖索引        | 查询字段固定且较少     |

如果业务允许“加载更多”，游标分页通常最稳。

如果必须支持后台跳页，延迟关联往往比直接 `LIMIT offset, size` 更靠谱。

### 方案一：游标分页

适合信息流、评论、消息列表这类只需要“上一页 / 下一页”的场景。

```sql
SELECT id, user_id, status, create_time
FROM t_order
WHERE user_id = ?
  AND id < ?
ORDER BY id DESC
LIMIT 20;
```

它的关键不是 `id` 必须连续，而是数据库能从上一次的游标位置继续往后扫。
如果按 `create_time` 排序，最好用 `(create_time, id)` 这种联合游标，避免同一时间戳下顺序不稳定。

### 方案二：延迟关联

适合还要兼容传统页码，但偏移量已经很大的后台查询。

```sql
SELECT o.*
FROM t_order o
INNER JOIN (
  SELECT id
  FROM t_order
  WHERE user_id = ?
  ORDER BY create_time DESC
  LIMIT 100000, 20
) page ON page.id = o.id
ORDER BY o.create_time DESC;
```

这类写法的核心是：

**先让子查询在更窄的索引上定位目标 ID，再回表拿完整行。**

它不能让大偏移量完全免费，但能减少大量无意义回表。

### 方案三：覆盖索引

如果列表页只展示少量字段，可以让索引覆盖查询字段。

```sql
CREATE INDEX idx_order_user_time_status
ON t_order(user_id, create_time, status, id);

SELECT id, status, create_time
FROM t_order
WHERE user_id = ?
ORDER BY create_time DESC
LIMIT 100000, 20;
```

覆盖索引的代价是索引更大，写入和维护成本更高。
所以它适合字段稳定、查询高频、收益明确的列表，不适合把大字段都塞进索引。

[`慢 SQL 应该按什么顺序排查？`](/database/mysql/mysql-slow-query-troubleshooting.html)
以及后续 MySQL 分页、索引文章是同一条线。

## 第六步：索引优化不是默认终点

有些查询即使加完索引，还是不值得继续在 MySQL 上硬扛。

比如：

- 超深分页
- 多维复杂筛选
- 大范围模糊搜索
- 报表聚合和离线统计

这时更合理的动作可能是：

- 限制最大翻页深度
- 改成游标分页
- 用 ES / OLAP 做检索
- 做缓存或预计算

也就是说：

**索引优化是数据库优化的重要手段，但不是所有查询的最后归宿。**

## 优化后怎么验证，别只看“跑快了”

SQL 优化最容易出现一个假象：本地跑快了，线上还是慢。

验证时至少要看：

| 维度     | 为什么要看                                  |
| -------- | ------------------------------------------- |
| 数据量   | 小表上有效的索引，大表上可能选择完全不同    |
| 数据分布 | 热门状态、热门用户、时间范围会影响执行计划  |
| 扫描行数 | RT 下降但扫描行数仍高，说明风险还在         |
| 回表次数 | 大量回表在高并发下会放大随机 I/O            |
| 排序方式 | `Using filesort` 和临时表可能成为高峰期瓶颈 |
| 并发压测 | 单次查询快，不代表连接池和磁盘 I/O 扛得住   |

一个比较稳的验证闭环是：

1. 用生产级数据量复现慢查询。
2. 记录优化前的执行计划、扫描行数、耗时和返回行数。
3. 调整索引或 SQL 后重新跑同样条件。
4. 对比 `rows`、`filtered`、临时表、文件排序和回表情况。
5. 在并发下看数据库 CPU、I/O、连接数和接口 P99。

如果只看单次耗时，很容易把缓存命中、数据量差异或测试环境偶然性当成优化效果。

## 上线索引也要考虑成本

SQL 优化经常最后落到“加索引”，但线上加索引本身也有风险。

要提前确认：

- 表有多大，DDL 会不会长时间占资源。
- 当前数据库版本和存储引擎是否支持在线 DDL。
- 新索引会不会影响写入吞吐和磁盘空间。
- 新索引是否和已有索引高度重复。
- 回滚时是否能安全删除索引。

如果是大表，可以考虑低峰执行、灰度验证、先建不可见索引观察，或者借助成熟的在线变更工具。
核心点是：

**索引优化也要走变更流程，不是随手执行一条 DDL。**

## 一条更像工程实战的排查链

如果面试官问你“慢 SQL、深分页、索引怎么联系起来”，可以直接按这个顺序答：

1. 先通过慢查询日志和接口监控找出最慢、最频繁的 SQL。
2. 用 EXPLAIN 看扫描行数、排序、临时表和回表情况。
3. 判断慢点是在过滤、排序，还是偏移量过大。
4. 如果是访问路径不对，就补联合索引或改写查询。
5. 如果是深分页，就优先考虑游标、延迟关联、覆盖索引。
6. 如果业务天然不适合继续压 MySQL，就把检索或报表能力迁出去。

这条链最大的价值是：

**先定根因，再选优化层次。**

## 容易踩的坑

### 一慢就加索引

如果真正问题是深分页或返回数据量太大，单纯加索引未必解决。

### 只看 `EXPLAIN`，不看真实数据分布

同一条 SQL 在测试库和生产库上完全可能是两回事，尤其是数据量、热点和统计信息不同的时候。

### 只会优化 SQL，不会反推产品形态

后台查询如果允许用户随便翻到第 50000 页，本身就是产品和数据访问模型没设计好。

### 盲目相信 `FORCE INDEX`

强制索引可以作为临时验证手段，但不能默认当成最终方案。

因为优化器的选择和数据分布、统计信息、版本都有关系。
如果强制了一个在当前样本上看起来不错的索引，后续数据分布变化时反而可能更慢。

### 忽略写入成本

索引不是越多越好。

每多一个索引，写入、更新、删除和页分裂的成本都会增加。
所以高频写表上的索引设计，要看读写收益是否真的划算。

## 小结

- 慢 SQL、深分页、索引优化通常属于同一条数据库性能排查链，不该拆开孤立回答。
- 先定位慢 SQL，再看执行计划，最后才决定是改索引、改 SQL 还是改访问模型。
- 深分页的核心问题是大量无效扫描和回表，它经常是慢 SQL 的具体形态。
- 索引优化真正要看的是访问路径有没有对齐查询模式，不是只看“有没有索引”。
- 优化后要用生产级数据量、执行计划、扫描行数和并发指标验证，不要只看单次耗时。
