# Kafka、RocketMQ、RabbitMQ 怎么选？

> MQ 选型题最怕背参数表。真正有价值的是先说业务诉求，再解释为什么这个中间件更匹配。

“Kafka、RocketMQ、RabbitMQ 怎么选”是典型的技术选型题。

很多回答会直接背：

- Kafka 吞吐高
- RabbitMQ 功能全
- RocketMQ 比较均衡

这三个结论不算错，但还是太平。
更像工程判断的答法，应该先问：

1. 你最看重的是吞吐、低延迟，还是功能完整度
2. 你要不要事务消息、顺序消息、延时消息
3. 团队的运维和治理能力跟不跟得上

## 先给一张总表

| 维度     | Kafka                      | RocketMQ                             | RabbitMQ                         |
| -------- | -------------------------- | ------------------------------------ | -------------------------------- |
| 典型强项 | 高吞吐、流式处理、日志链路 | 业务消息、事务 / 顺序 / 延时能力均衡 | 路由灵活、协议成熟、功能细       |
| 常见场景 | 日志采集、埋点、流式数据   | 订单、支付、通知、业务事件           | 中小规模业务消息、复杂路由       |
| 运维感受 | 集群和存储模型更偏数据平台 | 业务中台常见，能力较均衡             | 使用门槛低，但高吞吐下压测要谨慎 |

真正选型时，最好不要把它们讲成“谁绝对更强”，而是讲“谁更匹配当前诉求”。

## Kafka 更像高吞吐事件流平台

Kafka 最突出的特点通常是：

- 吞吐高
- Partition 模型清晰
- 保留日志能力强
- 和流式处理生态结合紧

所以它很适合：

- 日志采集
- 埋点与行为流
- 实时数仓
- 大吞吐异步任务链路

但它的使用心智更偏：

**日志流 / 事件流**

而不是“每一种业务消息都默认最适合 Kafka”。

## RocketMQ 更像业务消息的均衡型选手

RocketMQ 常被很多后端团队偏爱，一个重要原因是它在业务消息里比较均衡：

- 顺序消息
- 延时 / 定时消息
- 事务消息
- 较完整的业务投递语义

所以它很适合：

- 订单事件
- 支付流程
- 库存流转
- 业务通知

也就是：

**需要把业务语义讲得比较细的消息场景。**

## RabbitMQ 更像路由能力细、接入友好的消息中间件

RabbitMQ 一个很典型的强项是 Exchange 路由模型灵活：

- direct
- topic
- fanout
- headers

所以它在这些场景里常见：

- 中小规模业务系统
- 复杂订阅路由
- 多种消费者模式并存
- 对 AMQP 生态兼容有要求

它的使用门槛通常比较友好，但如果场景是超大吞吐日志流，很多团队会更偏 Kafka。

## 选型不要只看吞吐，还要看“业务语义支持”

很多选型题最大误区是只比 QPS。

但真实系统经常更关心的是：

- 要不要顺序
- 要不要事务消息
- 要不要延时消息
- 消费失败后怎么治理
- 回溯、补偿、治理工具链如何

可以把这几个维度单独拎出来答。

### 1. 如果更看重吞吐和流式处理

更常见偏 Kafka。

### 2. 如果更看重业务消息能力完整度

更常见偏 RocketMQ。

### 3. 如果更看重灵活路由和成熟协议模型

更常见偏 RabbitMQ。

## 顺序、事务、延时，这三个点最容易被追问

### 顺序

- Kafka 可以在单 Partition 维度保证顺序
- RocketMQ 也支持顺序消息
- RabbitMQ 也能做，但要看具体消费模型和并发配置

所以顺序不是“谁有没有”的问题，而是：

**你的顺序粒度有多细，吞吐愿意牺牲多少。**

### 事务

如果业务链路很强调“本地事务和消息发送协同”，RocketMQ 的事务消息会更常被提到。

但也别把它说成万能解法，因为最终还是要靠业务状态机和回查逻辑托底。

### 延时

延时 / 定时能力在订单超时关单、定时触发、补偿任务这类场景里很常见。

RocketMQ、RabbitMQ 都常被拿来做这类能力，但具体实现模型和限制不同。

所以答题时更稳的方式是：

**别只说支持，顺手讲一下适用场景和限制。**

## 运维复杂度和团队习惯也要进选型

选型不是纸面功能大战。

你还得看：

- 团队现在更熟哪个
- 监控、告警、排障体系是否成熟
- 是否已经有现成运维平台
- 和现有技术栈、数据平台是否顺手

一个功能更强的 MQ，如果团队根本驾驭不住，落地效果未必更好。

## 一个比较稳的回答模板

如果面试官问你怎么选，可以直接按这个模板说：

1. 先看场景是业务消息、日志流，还是复杂路由消息。
2. 如果更看重大吞吐和流式处理，优先考虑 Kafka。
3. 如果更看重顺序、事务、延时这些业务语义，RocketMQ 往往更合适。
4. 如果更看重路由灵活性、协议成熟度和接入友好性，RabbitMQ 很常见。
5. 最终还要结合团队运维能力、现有生态和监控治理体系做决定。

这套回答会比单纯背对比表更像真实选型。

## 容易踩的坑

### 把“吞吐最高”当成唯一标准

很多业务消息场景，真正痛点不是吞吐，而是事务、幂等、延时和治理。

### 不说使用边界

说一个 MQ 适合某场景时，最好顺手补一句它的代价，不然答案会显得太绝对。

### 忽略团队能力

技术选型从来不只是看产品特性，还要看团队能不能长期稳定运维。

## 小结

- Kafka、RocketMQ、RabbitMQ 没有绝对高下，核心是业务诉求是否匹配。
- Kafka 更偏高吞吐事件流，RocketMQ 更偏业务消息均衡能力，RabbitMQ 更偏灵活路由和成熟协议模型。
- 顺序、事务、延时和可靠性治理，往往比单纯吞吐更能决定业务选型。
- 真正成熟的选型答案，应该同时讲功能匹配度、性能边界和团队运维能力。
- 这题最稳的答法不是背参数，而是“先定场景，再解释为什么这个更合适”。
