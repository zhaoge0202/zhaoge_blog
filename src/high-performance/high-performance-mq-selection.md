# Kafka、RocketMQ、RabbitMQ 怎么选？

> MQ 选型题最怕背参数表。真正有价值的是先说业务诉求，再解释为什么这个中间件更匹配。

“Kafka、RocketMQ、RabbitMQ 怎么选”是典型的技术选型题。

很多回答会直接背：

- Kafka 吞吐高
- RabbitMQ 功能全
- RocketMQ 比较均衡

这三个结论不算错，但还是太平。
更像工程判断的答法，应该先问：

1. 你最看重的是吞吐、低延迟，还是功能完整度
2. 你要不要事务消息、顺序消息、延时消息
3. 团队的运维和治理能力跟不跟得上
4. 这条链路失败后怎么补偿、重放和排障

也就是说，MQ 选型不是“哪个产品更强”，而是：

**当前业务的消息模型、可靠性要求、吞吐规模、团队能力，和哪个 MQ 更匹配。**

## 先给一张总表：不要把它们排成绝对名次

| 维度     | Kafka                      | RocketMQ                             | RabbitMQ                         |
| -------- | -------------------------- | ------------------------------------ | -------------------------------- |
| 典型强项 | 高吞吐、流式处理、日志链路 | 业务消息、事务 / 顺序 / 延时能力均衡 | 路由灵活、协议成熟、功能细       |
| 常见场景 | 日志采集、埋点、流式数据   | 订单、支付、通知、业务事件           | 中小规模业务消息、复杂路由       |
| 运维感受 | 集群和存储模型更偏数据平台 | 业务中台常见，能力较均衡             | 使用门槛低，但高吞吐下压测要谨慎 |
| 主要代价 | 业务语义要自己治理得更多   | 要理解消息类型和 Broker 运维         | 高吞吐和大积压场景要谨慎压测     |

真正选型时，最好不要把它们讲成“谁绝对更强”，而是讲“谁更匹配当前诉求”。

## 选型前先问五个问题

在比较产品之前，先把业务问题问清楚：

| 问题                         | 为什么重要                           |
| ---------------------------- | ------------------------------------ |
| 消息是日志流还是业务事件？   | 决定更偏事件流平台还是业务消息中间件 |
| 峰值吞吐和消息大小多大？     | 决定分区、队列、磁盘和网络设计       |
| 是否要求顺序、事务、延时？   | 决定是否需要更强业务语义支持         |
| 失败后如何重试、死信、补偿？ | 决定治理工具链是否够用               |
| 团队现有运维能力在哪？       | 决定长期稳定性，而不是只看功能表     |

如果这些问题答不上来，直接说“上 Kafka”或“上 RocketMQ”都不稳。

## Kafka 更像高吞吐事件流平台

Kafka 最突出的特点通常是：

- 吞吐高
- Partition 模型清晰
- 保留日志能力强
- 和流式处理生态结合紧

所以它很适合：

- 日志采集
- 埋点与行为流
- 实时数仓
- 大吞吐异步任务链路

但它的使用心智更偏：

**日志流 / 事件流**

而不是“每一种业务消息都默认最适合 Kafka”。

### Kafka 适合什么

更适合这些场景：

- 日志采集、用户行为埋点。
- 实时数仓、流式计算、数据管道。
- 大吞吐事件流分发。
- 需要按时间保留消息并支持回放的链路。
- 多个下游独立订阅同一类事件流。

### Kafka 要注意什么

如果用 Kafka 承接强业务消息，要额外把这些事讲清楚：

- 同一业务 key 是否落到同一 Partition。
- 消费端 offset 提交时机。
- 生产者 ack、副本、ISR 配置。
- 消费端幂等和状态机。
- Rebalance 对消费暂停和重复消费的影响。

所以 Kafka 不是“不可靠”，这个说法已经过时。
更准确的说法是：Kafka 的优势在高吞吐事件流，业务语义要靠应用侧和治理体系一起补齐。

## RocketMQ 更像业务消息的均衡型选手

RocketMQ 常被很多后端团队偏爱，一个重要原因是它在业务消息里比较均衡：

- 顺序消息
- 延时 / 定时消息
- 事务消息
- 较完整的业务投递语义

所以它很适合：

- 订单事件
- 支付流程
- 库存流转
- 业务通知

也就是：

**需要把业务语义讲得比较细的消息场景。**

### RocketMQ 适合什么

更适合这些场景：

- 订单、支付、库存、履约等业务事件。
- 需要定时 / 延时触发的任务。
- 需要事务消息协调本地事务和消息发送。
- 局部顺序消息，比如同一订单的状态流转。
- 希望业务消息能力相对完整，少一些自己拼装。

### RocketMQ 要注意什么

RocketMQ 的“业务能力更完整”不等于可以省掉治理。

还要关注：

- 事务消息只解决生产端一致，不保证下游消费一定成功。
- 顺序消息会降低并行度，失败重试时可能阻塞同一组后续消息。
- 延时 / 定时能力要看版本、精度、最大延迟和实现限制。
- 队列数、消费组、Broker 部署方式会影响积压和扩容。

所以更稳的说法是：

**RocketMQ 更适合业务消息，但仍然要配幂等、补偿、死信和监控。**

## RabbitMQ 更像路由能力细、接入友好的消息中间件

RabbitMQ 一个很典型的强项是 Exchange 路由模型灵活：

- direct
- topic
- fanout
- headers

所以它在这些场景里常见：

- 中小规模业务系统
- 复杂订阅路由
- 多种消费者模式并存
- 对 AMQP 生态兼容有要求

它的使用门槛通常比较友好，但如果场景是超大吞吐日志流，很多团队会更偏 Kafka。

### RabbitMQ 适合什么

更适合这些场景：

- 中小规模业务消息。
- 复杂路由规则。
- 多协议或多语言系统接入。
- 需要灵活 Exchange 路由模型的业务。
- 对管理界面、插件生态和接入友好性有要求。

### RabbitMQ 要注意什么

RabbitMQ 的高可用和队列类型要按版本看。
新项目如果需要复制和高可靠队列，应优先评估 Quorum Queue 或 Streams，而不是沿用早期镜像队列思路。

还要关注：

- Quorum Queue 可靠性更强，但对磁盘 IO 和吞吐有代价。
- Classic Queue 更轻，但可靠性和高可用语义不同。
- 大规模积压、超高吞吐、海量 fanout 场景要压测。
- 路由灵活不代表消费端就不用做幂等和死信治理。

## 选型不要只看吞吐，还要看“业务语义支持”

很多选型题最大误区是只比 QPS。

但真实系统经常更关心的是：

- 要不要顺序
- 要不要事务消息
- 要不要延时消息
- 消费失败后怎么治理
- 回溯、补偿、治理工具链如何

可以把这几个维度单独拎出来答。

### 如果更看重吞吐和流式处理

更常见偏 Kafka。

但要补一句：业务端要处理好幂等、offset、分区 key、重平衡和补偿。

### 如果更看重业务消息能力完整度

更常见偏 RocketMQ。

但要补一句：事务、顺序、延时都不是免费能力，要结合版本、吞吐和运维成本看。

### 如果更看重灵活路由和成熟协议模型

更常见偏 RabbitMQ。

但要补一句：高可靠队列和超大吞吐场景要单独压测，不能默认所有队列类型都适合。

## 顺序、事务、延时，这三个点最容易被追问

### 顺序

- Kafka 可以在单 Partition 维度保证顺序
- RocketMQ 也支持顺序消息
- RabbitMQ 也能做，但要看具体消费模型和并发配置

所以顺序不是“谁有没有”的问题，而是：

**你的顺序粒度有多细，吞吐愿意牺牲多少。**

更完整的回答是：

- 全局有序通常代价很高，业务里更常见的是局部有序。
- Kafka 常按 key 进同一 Partition。
- RocketMQ 可以按消息组 / 队列粒度保证局部顺序。
- RabbitMQ 要看队列、消费者并发、ack 和重入队行为。
- 消费失败和重试可能打乱业务效果，最终仍要靠状态机兜底。

### 事务

如果业务链路很强调“本地事务和消息发送协同”，RocketMQ 的事务消息会更常被提到。

但也别把它说成万能解法，因为最终还是要靠业务状态机和回查逻辑托底。

更稳的边界是：

**事务消息解决的是生产端本地事务和消息发送的一致性，不保证消费者一定处理成功。**

所以即使选了事务消息，消费端仍然要设计幂等、重试、死信和补偿。

### 延时

延时 / 定时能力在订单超时关单、定时触发、补偿任务这类场景里很常见。

RocketMQ、RabbitMQ 都常被拿来做这类能力，但具体实现模型和限制不同。

所以答题时更稳的方式是：

**别只说支持，顺手讲一下适用场景和限制。**

如果延时规模很大、精度要求很高、任务生命周期复杂，也要评估专门的调度系统，而不是默认所有延时任务都塞进 MQ。

## 可靠性、积压和回放也要一起看

选型时还要把前几篇串起来：

- [`MQ 如何保证消息不丢？`](/high-performance/high-performance-message-reliability.html)
- [`MQ 如何处理重复消费和幂等？`](/high-performance/high-performance-message-idempotency.html)
- [`MQ 消息积压怎么排查？`](/high-performance/high-performance-message-backlog.html)

因为真实系统里，选型之后最常见的问题不是“这个 MQ 能不能发消息”，而是：

- 丢消息怎么兜？
- 重复消费怎么处理？
- 积压怎么排查？
- 死信怎么处理？
- 历史消息能不能回放？
- 监控告警和运维工具是否成熟？

可以用这张表补充回答：

| 关注点     | Kafka                        | RocketMQ                     | RabbitMQ                       |
| ---------- | ---------------------------- | ---------------------------- | ------------------------------ |
| 可靠性抓手 | 副本、acks、ISR、offset      | 刷盘、复制、事务消息、重试   | Confirm、Return、Ack、队列类型 |
| 幂等重点   | offset 与业务提交边界        | 消费结果、重试、业务唯一键   | 手动 ack、重入队、死信         |
| 积压治理   | Partition、Consumer Lag      | Queue、消费组、队列数        | Queue 深度、Unacked、Prefetch  |
| 回放能力   | 日志保留和 offset 回退较常见 | 支持按保留机制回溯           | 视队列类型和保留策略而定       |
| 运维关注   | 分区、副本、磁盘、重平衡     | Broker、NameServer、队列规划 | 内存水位、磁盘、连接、队列类型 |

这张表不是要背，而是提醒你：选型最终会落到治理能力。

## 运维复杂度和团队习惯也要进选型

选型不是纸面功能大战。

你还得看：

- 团队现在更熟哪个
- 监控、告警、排障体系是否成熟
- 是否已经有现成运维平台
- 和现有技术栈、数据平台是否顺手

一个功能更强的 MQ，如果团队根本驾驭不住，落地效果未必更好。

可以把团队能力拆成几个问题：

1. 有没有人能定位 Broker、队列、消费组、分区层面的问题？
2. 有没有现成监控和告警？
3. 有没有死信处理和重放工具？
4. 有没有容量评估和压测数据？
5. 是否能接受引入新中间件的学习和运维成本？

如果答案都是否定的，选一个纸面更强但团队不熟的 MQ，风险反而更高。

## 一个更实用的选择流程

可以按下面顺序做判断：

```text
先定场景
  ├─ 日志 / 埋点 / 流式数据：优先看 Kafka
  ├─ 订单 / 支付 / 库存等业务消息：优先看 RocketMQ
  ├─ 复杂路由 / 多协议 / 中小规模业务：优先看 RabbitMQ
  └─ 已有成熟平台：优先复用，除非现有方案明显不满足

再定约束
  ├─ 吞吐和延迟
  ├─ 顺序、事务、延时
  ├─ 可靠性、幂等、补偿
  └─ 运维、监控、成本
```

这个流程比直接背对比表更像真实选型。

## 一个比较稳的回答模板

如果面试官问你怎么选，可以直接按这个模板说：

1. 先看场景是业务消息、日志流，还是复杂路由消息。
2. 如果更看重大吞吐和流式处理，优先考虑 Kafka。
3. 如果更看重顺序、事务、延时这些业务语义，RocketMQ 往往更合适。
4. 如果更看重路由灵活性、协议成熟度和接入友好性，RabbitMQ 很常见。
5. 然后补可靠性、幂等、积压、死信、回放和监控治理方案。
6. 最终还要结合团队运维能力、现有生态和迁移成本做决定。

这套回答会比单纯背对比表更像真实选型。

## 容易踩的坑

### 把“吞吐最高”当成唯一标准

很多业务消息场景，真正痛点不是吞吐，而是事务、幂等、延时和治理。

### 不说使用边界

说一个 MQ 适合某场景时，最好顺手补一句它的代价，不然答案会显得太绝对。

### 忽略团队能力

技术选型从来不只是看产品特性，还要看团队能不能长期稳定运维。

### 把版本差异忽略掉

比如队列类型、延时消息、事务消息、高可用方案都可能和版本有关。
选型时要按当前实际版本看，不要拿旧经验直接套。

### 忽略迁移成本

如果公司已有成熟 MQ 平台，新业务只是中等规模，复用现有平台通常比引入全新中间件更稳。
除非现有平台在吞吐、可靠性、业务语义或运维隔离上已经明显不满足。

### 不做压测就拍板

选型结论最终要落到数据。
至少要压测：

- 单实例生产和消费吞吐。
- 消息大小变化后的性能。
- 积压后的恢复速度。
- Broker 故障或消费者重启时的表现。
- 下游依赖是否会被消费扩容打穿。

## 小结

- Kafka、RocketMQ、RabbitMQ 没有绝对高下，核心是业务诉求是否匹配。
- Kafka 更偏高吞吐事件流，RocketMQ 更偏业务消息均衡能力，RabbitMQ 更偏灵活路由和成熟协议模型。
- 顺序、事务、延时、可靠性、幂等、积压和回放能力，往往比单纯吞吐更能决定业务选型。
- 真正成熟的选型答案，应该同时讲功能匹配度、性能边界、版本限制、治理体系和团队运维能力。
- 这题最稳的答法不是背参数，而是“先定场景，再解释为什么这个更合适，以及代价是什么”。
