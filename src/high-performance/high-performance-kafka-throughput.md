# Kafka 为什么吞吐高？顺序写、零拷贝和批量做了什么？

> Kafka 单机能跑到每秒几十万甚至上百万条消息，靠的不是某个黑科技，而是顺序写磁盘、页缓存、零拷贝、批量发送、分区并行这几件事叠在一起的结果。拆开看，每一项都不神秘，甚至操作系统层面的两个基础机制（[Page Cache](/cs-basics/operating-system/os-page-cache.html)、[零拷贝](/cs-basics/operating-system/os-zero-copy.html)）Kafka 只是顺手借用了。

面试里问“Kafka 为什么快”，最容易踩的坑是背答案背成一句“零拷贝 + 顺序写”，然后被追问一句“零拷贝拷了几次”就答不上来。这篇把每一层拆开讲清楚，顺便说说吞吐高不代表的东西。

## 先明确一件事：磁盘不慢，随机写才慢

很多人对“Kafka 靠磁盘做消息队列还能快”这件事本能地不信，因为脑子里默认磁盘慢。这个直觉只对了一半。

机械磁盘慢在**寻道**，也就是磁头移动到目标磁道再等盘片转到目标扇区的过程，这个物理动作按毫秒计。但如果不用寻道、只是在当前磁头位置往后接着写，磁盘的顺序吞吐可以做到接近甚至超过普通内存随机访问的量级——这也是为什么业界经常说“顺序写磁盘比随机写内存还快”，前提是都跟各自的随机模式对比。

| 访问模式   | 典型行为                                                | 瓶颈            |
| ---------- | ------------------------------------------------------- | --------------- |
| 随机写磁盘 | 每次写之前先寻道、等盘片转到位                          | 寻道 + 旋转延迟 |
| 顺序写磁盘 | 磁头基本不用大范围移动，往后接着写                      | 磁盘持续吞吐    |
| SSD 场景   | 寻道成本几乎消失，但顺序写依然更省 IOPS、更利于垃圾回收 | 写放大          |

Kafka 的存储结构天然是顺序写的：每个 Partition 对应磁盘上的一段日志文件，生产者发消息只做一件事——**在文件末尾追加（append）**，不修改历史数据，也不需要按 key 定位到文件中间去更新。这跟数据库那种“按主键随机更新记录”的写模式完全不同。

回到前面 [Kafka 常见问题总结](./kafka-questions-01.md) 里提到的一个细节：Partition 内部靠 offset 保证顺序，而这个 offset 恰好就是顺序追加写留下的自然产物——写到哪个位置，offset 就是多少，读的时候按 offset 定位也是顺序读。写是顺序的，读（消费者正常追尾消费）大多数情况下也是顺序的，这两头顺序叠在一起才是 Kafka 存储层快的根本原因。

打个比方：随机写磁盘像是让你在一本厚字典里不断翻页去改某一页的内容，翻页（寻道）本身就占了大部分时间；顺序写像是拿一个新本子，从第一页开始一直往后写，从不回头翻页。Kafka 的日志文件（`.log`）就是这个“新本子”，Producer 写消息就是在本子末尾续写一行，Consumer 消费就是从上次读到的位置继续往后读，两边动作都不涉及“翻回前面某页”。

## Page Cache：写磁盘其实没有马上写磁盘

顺序写只解决了磁盘 IO 模式的问题，Kafka 还有一招更狠：**能不落盘就先不落盘**。

Kafka 的 Producer 把消息写进 Partition 时，走的是标准的 `write()` 系统调用，数据先进内核的 [Page Cache](/cs-basics/operating-system/os-page-cache.html)，`write` 就返回了，并不等真正刷到磁盘扇区。后台由内核的回写线程根据脏页比例、时间间隔等策略异步刷盘。

这带来两个直接好处：

1. **写延迟低**：Producer 感知到的写入耗时只是内存拷贝的耗时，不是磁盘 IO 耗时。
2. **读也可能不用碰磁盘**：Consumer 追着最新消息消费时，这些数据大概率还在 Page Cache 里没被换出，读操作直接命中内存缓存，不用真的发起磁盘 IO。

这也是为什么 Kafka 官方一直强调“别给 Broker 的 JVM 堆分配太大内存”——JVM 堆占得越多，留给操作系统做 Page Cache 的内存就越少，而 Kafka 真正依赖的高速缓存其实是这部分操作系统管理的页缓存，不是 JVM 堆内缓存。堆内缓存还要背 GC 成本，Page Cache 完全不占这个成本，进程重启也不会丢（只要机器没重启）。

需要注意的是，`write` 返回不等于数据已经安全落盘，这一点在讲可靠性时是个关键坑，后面“容易踩的坑”那节会展开。

## 零拷贝：消费者读消息时少搬了两次数据

写路径靠 Page Cache 省了磁盘 IO，读路径（也就是 Consumer 拉消息、Broker 把数据发给 Consumer）靠的是**零拷贝**。

不用零拷贝时，Broker 把磁盘文件里的消息发给 Consumer 的常规做法是：

```text
磁盘 -> 内核 Page Cache -> 用户态缓冲区 -> Socket 缓冲区 -> 网卡
```

这条路径里有两次没必要的 CPU 拷贝：数据从内核 Page Cache 拷到用户态缓冲区（Broker 进程内存），处理完之后又原样拷回内核的 Socket 缓冲区——数据本身根本没被应用层改动过，纯粹是在内核态和用户态之间来回搬了一趟，还搭上两次上下文切换。

Kafka 用 Java NIO 的 `FileChannel.transferTo()`，底层对应 Linux 的 `sendfile()` 系统调用，让数据**不经过用户态**，直接从内核 Page Cache 拷到 Socket 缓冲区（如果网卡支持 SG-DMA，甚至可以再省一次 CPU 拷贝，只留 DMA 搬运）。完整的拷贝次数、上下文切换次数拆解可以看 [零拷贝到底少拷了什么？](/cs-basics/operating-system/os-zero-copy.html)，这里不重复画图，只强调结论：**零拷贝省的是 CPU 参与的数据搬运，不是磁盘 IO 本身**。

顺序写决定了这批数据大概率就在 Page Cache 里，零拷贝决定了把这批数据发给消费者时不用绕用户态一圈——这两个机制是配合关系，不是各自独立生效。如果消息早就被换出 Page Cache（比如消费者在追很久以前的历史数据），零拷贝一样要触发磁盘 IO，只是省下的还是那两次 CPU 拷贝。

## 批量发送和压缩：把很多小事凑成一件大事

顺序写和零拷贝解决的是“单次 IO 怎么更快”，批量解决的是“怎么让 IO 次数更少”。

Kafka Producer 默认不是来一条消息就发一次网络请求，而是攒一批再发，核心靠两个参数配合：

| 参数         | 作用                                             |
| ------------ | ------------------------------------------------ |
| `batch.size` | 一个批次最多攒多少字节就触发发送                 |
| `linger.ms`  | 就算批次没攒满，等待这么久也要发出去，避免无限等 |

两者是“谁先满足谁触发”的关系：消息量大的时候基本被 `batch.size` 触发，凑够一批就发；消息量小、稀疏的时候靠 `linger.ms` 兜底，避免消息在客户端攒到天荒地老。

批量还带来一个连带收益：**压缩更划算**。Kafka 支持对整个批次做压缩（`compression.type`，常见 gzip、snappy、lz4、zstd），批次越大，压缩比通常越好，同样的网络带宽能塞进更多有效消息，Broker 磁盘也是按压缩后的批次整体落盘、整体读出，进一步减少实际 IO 量。

一个直观对比：

| 发送方式         | 网络请求次数 | 每次请求开销占比                       | 压缩效果                           |
| ---------------- | ------------ | -------------------------------------- | ---------------------------------- |
| 逐条发送，不攒批 | 高           | 高（协议头、TCP/TLS 握手成本相对显著） | 几乎没有，单条消息压缩收益有限     |
| 批量发送 + 压缩  | 低           | 低（分摊到一批消息上）                 | 好，尤其是结构相似的消息（如日志） |

Broker 收到批次后，很多场景下可以保持压缩状态直接顺序写入磁盘，不用解压再压缩，进一步省 CPU。

## 分区并行：把一条队伍拆成多条队伍

前面三项都是在优化“一条流水线”的效率，分区（Partition）解决的是并行度问题。

一个 Topic 可以有多个 Partition，各 Partition 可以分布在不同 Broker 上，Producer 写不同 Partition互不阻塞，Consumer Group 内不同 Consumer 也可以并行消费不同 Partition。单机、单磁盘、单线程的顺序写吞吐再高也有上限，Kafka 靠“横向拆分成多条独立的顺序写日志”把总吞吐堆上去——这跟数据库分库分表、Web 服务器多进程/多线程处理并发请求是同一个思路，不是 Kafka 独有的魔法。

但分区数不是越多越好，这里容易被面试追问：

- Partition 越多，Broker 需要维护的文件句柄、内存索引结构越多，Controller 做元数据管理和 Leader 选举时开销也越大。
- 单个 Partition 内部依然是严格顺序的，[顺序消费还要注意分区粒度这件事](./kafka-questions-01.md)——分区数直接决定了你的顺序保证粒度和消费并行度的上限，这也是为什么“按 key 分区、单分区内有序”是最常见的顺序性方案。

## 别当成“Kafka 魔法”，这是一套工程组合拳

到这里应该能看出，Kafka 的高吞吐没有哪一项是它独创的黑科技：

- 顺序写磁盘：数据库的 WAL（预写日志）、RocketMQ 的 CommitLog 都在用同一个思路。
- Page Cache：操作系统级机制，Kafka 只是设计上刻意贴合它、不重复造一套用户态缓存。
- 零拷贝：`sendfile` 是 Linux 内核提供的通用系统调用，Nginx 传静态文件、RocketMQ 读 CommitLog 也在用。
- 批量 + 压缩：几乎所有高吞吐系统的通用套路，数据库批量插入、RPC 框架批量调用同理。
- 分区并行：分而治之的老思路，跟分库分表、多线程处理是一回事。

Kafka 的价值在于把这几件事**在同一套存储和网络路径上拼接得很顺**：Partition 的顺序写天然贴合 Page Cache 的顺序预读，零拷贝天然贴合“消费者大概率读的是刚写进 Page Cache 的新消息”这个假设，批量发送天然贴合顺序写“越大块越划算”的特性。理解了这套组合关系，比背一句“Kafka 用了零拷贝所以快”有用得多。

拿一个更直观的对比来看这套组合拳省在哪：

| 场景                  | 每条消息单独处理               | Kafka 的组合方式                  |
| --------------------- | ------------------------------ | --------------------------------- |
| 磁盘写入              | 每条各自找位置写，可能触发寻道 | 顺序追加，全走一条日志尾部        |
| 内核态/用户态数据搬运 | 每次读写都可能来回拷贝         | 零拷贝，能不出内核态就不出        |
| 网络请求              | 一条消息一次请求，协议头开销高 | 攒批发送，协议头开销摊薄          |
| 单机处理能力上限      | 单线程/单文件顶天              | 多 Partition 横向扩展，总吞吐叠加 |

单独看任何一行，节省的都只是某个环节的固定开销；四行叠在一起，才是 Kafka 官方宣传里“单机可达每秒几十万到百万级消息”这种数量级的来源。这也是为什么面试时只答“零拷贝”或者只答“顺序写”都不算完整答案——考察的其实是能不能把这几个环节串成一条线讲清楚。

## 吞吐高不等于业务延迟低，acks=all 是拿吞吐换可靠性

这是最容易被面试官抓住的认知误区：**吞吐（throughput）和延迟（latency）是两个维度，不是同一件事的两种说法**。

- 吞吐高，说的是单位时间能处理的消息总量大。
- 延迟低，说的是单条消息从发出到被确认/被消费的时间短。

`linger.ms` 调大能提升吞吐（攒的批次更大，单位开销更低），但对单条消息来说，它明摆着是在**故意多等一会儿**，反而拉高了这条消息的端到端延迟。这就是典型的吞吐和延迟互相拉扯的例子。

可靠性也是同一个道理。[前面提到](./kafka-questions-01.md) 的 `acks` 参数：

| acks 取值 | 含义                            | 对吞吐/延迟的影响  | 可靠性                          |
| --------- | ------------------------------- | ------------------ | ------------------------------- |
| `0`       | 发出去就算成功，不等任何确认    | 最快               | 最低，网络抖动直接丢消息        |
| `1`       | 只等 leader 副本写入成功        | 较快               | 中，leader 挂了未同步的数据会丢 |
| `all`     | 等 ISR 列表里所有副本都确认收到 | 最慢，延迟明显升高 | 最高                            |

`acks=all` 本质是用等待更多副本确认的时间，换取“leader 挂了也不丢数据”的保证。想要高吞吐可以放宽 `acks`，想要强可靠性就要接受延迟升高——这是工程取舍，不是配置越高越好这么简单。真被问到“Kafka 怎么保证可靠性”，答案从来不是单一参数，而是 Topic 副本数、ISR、Producer ACK、Consumer offset 提交时机这几件事一起决定的。

## 容易踩的坑

### 坑一：消息太小、太稀疏，批量根本攒不起来

如果业务场景就是消息体积很小、发送频率很低（比如几秒钟才发一条），`batch.size` 永远攒不满，每次都是 `linger.ms` 超时后发一条小包，批量和压缩的收益基本用不上，这时候吞吐优化的效果会远低于预期。这种场景下，与其死磕批量参数，不如先看看能不能在业务侧合并消息，或者接受这本来就是低吞吐场景，别用高吞吐场景的调优思路硬套。

### 坑二：跨机房部署，网络延迟直接吃掉批量和零拷贝的收益

零拷贝省的是本机 CPU 拷贝的开销，批量省的是网络请求次数的开销，但两者都改变不了物理链路的传播延迟。如果 Producer 和 Broker、或者 Broker 之间的副本同步跨机房甚至跨地域，几十毫秒的网络延迟会直接叠加到每次交互上，尤其是 `acks=all` 需要等远端副本确认时，延迟会被放大得更明显。这种情况下盲目调大 `batch.size`、指望“零拷贝很快”来抵消网络延迟，方向是错的，得先看部署拓扑。

### 坑三：把“Page Cache 写入快”误解成“同步刷盘、绝对安全”

前面讲过，`write` 返回只代表数据进了 Page Cache，不代表已经落到磁盘扇区。Kafka 默认依赖操作系统的后台刷盘策略，而不是每条消息强制 `fsync`。这意味着：如果 Broker 所在机器在数据刷盘前发生断电或内核崩溃，Page Cache 里还没落盘的这部分数据可能丢失。

Kafka 用多副本机制而不是同步刷盘来兜底这个风险——`replication.factor >= 3` 加上合理的 `min.insync.replicas`，让单机的 Page Cache 未刷盘风险被“至少一个副本已经持久化”这件事覆盖掉，而不是指望单机刷盘策略。把“Kafka 顺序写快”和“Kafka 消息绝对不丢”划等号，是这里最常见的理解偏差。

## 面试常见追问

### 追问一：Kafka 和 RocketMQ 都靠零拷贝，实现方式一样吗？

不完全一样。Kafka 用的是 `sendfile` 这条路径（Java 里对应 `FileChannel.transferTo()`），数据不进用户态，直接从 Page Cache 拷到 Socket 缓冲区，优点是拷贝次数少，缺点是应用拿不到、也改不了这份数据内容，只能原样转发。RocketMQ 的 CommitLog 读写更多依赖 `mmap`，把内核 Page Cache 映射到用户地址空间，应用可以直接读写这块映射内存，代价是比 `sendfile` 多一次用户态到 Socket 缓冲区的 CPU 拷贝。两者的取舍点在于：要不要在发送前修改数据内容。Kafka 场景下消息原样转发，`sendfile` 更合适；RocketMQ 有索引、过滤、消息体重组等更复杂的读取逻辑，`mmap` 更灵活。

### 追问二：既然顺序写这么快，为什么不干脆一个 Topic 只用一个 Partition？

单 Partition 确实能保证全局顺序，也不影响顺序写本身的效率，但它把并行度锁死了：Producer 写这个 Partition 只能落在一个 Broker 上，Consumer Group 里再多消费者，同一时刻也只有一个能真正消费这个 Partition（同组内一个 Partition 只能分给一个 Consumer）。顺序写解决的是“单条流水线快不快”，Partition 数解决的是“有几条流水线”，这是两个独立的维度，不能互相替代。

### 追问三：`linger.ms` 设多大合适？

没有放之四海而皆准的数值，取决于业务对延迟的容忍度。日志采集、埋点上报这类对单条延迟不敏感、追求整体吞吐的场景，`linger.ms` 可以设到几十毫秒甚至更高；下单支付这类要求消息尽快被消费的链路，`linger.ms` 要压得很低甚至保持默认（默认是 0，来消息就尽量发，只是仍会受 `batch.size` 影响攒批）。判断标准是先想清楚这条链路是吞吐优先还是延迟优先，再回头调参数，不要反过来先定参数再找理由。

## 小结

1. Kafka 高吞吐是**工程组合**：顺序写、Page Cache、零拷贝、批量压缩、分区并行，不是单一魔法。
2. 零拷贝省的是 **CPU 拷贝路径**，不是磁盘 IO 本身；跨机房延迟也抵消不了。
3. 批量和 `linger.ms` 用吞吐换延迟；小消息低频场景收益有限。
4. `acks=all` 会明显拉高写入延迟，吞吐和可靠性要一起谈。
5. Page Cache 写入快 ≠ 已落盘；可靠性更多靠多副本，而不是单机同步刷盘幻想。
