# 系统性能瓶颈应该怎么定位？

> 高性能题最怕一上来就堆方案。先把瓶颈定位到哪一层，再谈缓存、扩容、分库分表，答案才站得住。

“系统变慢了怎么办”是高性能场景题的总入口。

很多回答会直接跳到方案：

- 上缓存
- 扩容
- 异步化
- 分库分表

但真实线上排查不是这么做的。
先问一句：

**到底慢在入口、应用、缓存、数据库、消息链路，还是下游依赖？**

这一步不做，后面的优化很容易做反。

## 先建立一张分层地图

从用户请求进来，到结果返回，常见链路可以拆成：

```text
客户端
 -> 网关 / 负载均衡
 -> 应用服务
 -> Redis / 本地缓存
 -> MySQL / ES / MongoDB
 -> MQ / 异步任务
 -> 第三方或其他内部下游
```

所以“慢”也要分层看：

| 层次     | 典型问题                             |
| -------- | ------------------------------------ |
| 入口层   | 流量突增、连接数打满、网关限流不当   |
| 应用层   | 线程池排队、锁竞争、GC、对象创建过多 |
| 缓存层   | 命中率下降、热点 key、Redis 卡顿     |
| 数据层   | 慢 SQL、索引失效、深分页、主从延迟   |
| 异步层   | MQ 积压、消费变慢、重试风暴          |
| 下游依赖 | 第三方超时、RPC 抖动、连接池耗尽     |

真正稳定的排查，是沿着这张图一层层缩。

入口层不要只看“有没有流量进来”，还要看流量有没有被错误地分配。

常见入口问题包括：

- 负载均衡把流量持续打到少数慢节点。
- 健康检查太宽松，半死不活的节点还在接请求。
- 新节点刚上线，缓存没热、连接没建好，就被打满流量。
- 灰度发布后，少量实例承担了不成比例的请求。
- 网关限流规则误伤，把正常请求也排队或拒绝。

所以入口层排查要顺手看：

| 入口观察点 | 要判断什么                         |
| ---------- | ---------------------------------- |
| 单实例 QPS | 流量是否均匀，是否有热点实例       |
| 健康检查   | 慢节点是否还被认为“健康”           |
| 摘流量记录 | 故障节点有没有及时下线             |
| 灰度比例   | 新版本是否承担了超过预期的流量     |
| 预热状态   | 缓存、连接池、JIT 是否已经稳定下来 |

## 第一步：先确认“慢”是什么慢

很多人把所有性能问题都混成一句“接口超时”。

其实最先要拆的是：

- 平均 RT 变慢，还是 `P99/P999` 变慢
- 是所有接口都慢，还是某条链路慢
- 是稳定慢，还是偶发尖刺
- 是成功率下降，还是只是耗时上升

这几个判断很重要，因为它们对应的方向完全不同。

比如：

- 平均 RT 一直高，常见是容量不足或 SQL 本身慢
- `P99` 突然高，常见是锁等待、GC、热点流量、下游抖动
- 成功率也掉了，往往已经不仅是性能问题，而是高可用问题

## 第二步：先看监控，不要先改代码

性能定位最忌讳“凭直觉改”。

建议先拉几类监控一起看：

| 维度       | 重点看什么                         |
| ---------- | ---------------------------------- |
| 接口指标   | QPS、RT、P99、错误率               |
| 应用资源   | CPU、内存、GC、线程池、连接池      |
| 缓存指标   | 命中率、慢查询、热点 key、网络延迟 |
| 数据库指标 | 慢 SQL、活跃连接、锁等待、磁盘 I/O |
| MQ 指标    | 生产速率、消费速率、积压量、重试量 |
| 下游依赖   | RPC RT、超时率、错误码分布         |

如果接口 RT 高，但数据库和 Redis 都很稳，瓶颈可能就在应用层。

如果应用 CPU 很低，但数据库磁盘 I/O 高、慢 SQL 飙升，那就别先怪 Java 线程池。

这里有一个很实用的判断方式：**先对比正常时段，再看异常时段。**

不要只拿故障时刻的一张监控截图下结论，最好拉三段数据：

| 时间窗口 | 要回答的问题                         |
| -------- | ------------------------------------ |
| 正常基线 | 平时 QPS、RT、CPU、连接数大概是多少  |
| 异常窗口 | 哪个指标最先偏离基线                 |
| 变更窗口 | 最近有没有发布、活动、配置或数据变化 |

如果发现数据库 CPU 是最后才升上去的，而缓存命中率先掉下去，那数据库很可能只是被拖慢。
如果发现接口 P99 先抖，随后线程池队列变长，再出现超时，那更像下游或锁竞争把线程拖住了。

## 第三步：先看是不是流量模型变了

性能问题不一定是代码变差，也可能是流量变了。

常见变化包括：

- 活动流量突然上来
- 某个接口被爬虫打爆
- 热点商家 / 热点商品集中访问
- 消息重试导致请求量翻倍

所以定位时最好先问：

**这波慢，是代码慢了，还是请求模式变了？**

很多“系统扛不住”的根因，其实不是平均流量变大，而是热点更集中了。

## 第四步：按“最贵资源”去找瓶颈

排查性能，本质是在找哪个资源最先被打满。

最常见的几个瓶颈点：

### 1. 线程池 / 连接池

现象通常是：

- 线程池队列变长
- 获取连接等待时间增加
- 接口 RT 上升但 CPU 不高

这说明请求不是在算，而是在等。

排查时可以继续问：

- 业务线程池是不是所有接口共用一个池。
- HTTP / RPC / 数据库连接池是否到达上限。
- 连接池等待时间是不是明显高于真正执行时间。
- 慢请求堆积后，有没有继续放大重试。

这类问题的修复不一定是“加线程”。
如果下游已经慢了，盲目加线程只会让更多请求同时压过去，把排队从应用层挪到下游。

### 2. 数据库

现象通常是：

- 慢查询增多
- 活跃连接高
- 锁等待升高
- 主从延迟变大

这时候就该往 SQL、索引、分页和事务上钻，而不是先上分库分表。

数据库层要特别注意区分三类慢：

| 类型       | 典型信号                             | 常见动作                     |
| ---------- | ------------------------------------ | ---------------------------- |
| SQL 本身慢 | `rows_examined` 高、执行计划差       | 改 SQL、补索引、改访问路径   |
| 锁等待慢   | 活跃事务多、等待锁、死锁日志出现     | 缩短事务、降低锁范围、拆批次 |
| 资源耗尽慢 | 磁盘 I/O 高、连接打满、Buffer 命中低 | 扩容、限流、读写分离或归档   |

这三类慢的解法不同，混在一起答就容易把方案说偏。

### 3. Redis / 缓存

现象通常是：

- 命中率下降
- Redis RT 变高
- 大 key / 热 key 增多
- 持久化或复制影响延迟

缓存层慢了，数据库通常会被连带拖高。

### 4. MQ

现象通常是：

- 生产速率明显高于消费速率
- backlog 持续变大
- 消费重试增多
- 下游任务堆积

这类问题表面看是“异步链路慢”，本质还是在问消费侧为什么跟不上。

MQ 层还要看“积压是新产生的，还是旧积压恢复不动”：

- 如果生产速率突然升高，优先看流量、重试和上游发布。
- 如果生产速率正常但消费速率下降，优先看消费者线程、下游依赖和单条消息耗时。
- 如果消费速率正常但积压恢复很慢，说明并行度、分区 / 队列数或下游容量可能限制了追赶速度。

所以 MQ 慢不能只说“加消费者”，还要确认瓶颈是不是消费者本身。

## 第五步：别把“局部症状”错认成“最终瓶颈”

一个非常常见的误判是：

- 看到数据库 CPU 高，就认定数据库是根因
- 看到应用超时多，就认定应用有 bug

但高负载常常是连锁反应。

例如：

1. Redis 命中率下降
2. 更多请求打到 MySQL
3. MySQL RT 上升
4. 应用线程池排队
5. 接口超时

这时候真正的首发点可能是缓存问题，不是数据库本身。

所以排查性能问题时，最好区分：

- **首发瓶颈**
- **被拖慢的后续层**

一个更工程化的做法是按时间线复盘：

```text
指标最先异常的时间点
 -> 请求量 / 发布 / 配置 / 数据变化
 -> 第一层资源打满
 -> 后续层级被拖慢
 -> 用户侧错误或超时
```

时间线能帮你避免“谁现在最红，就怪谁”的误判。
很多事故里，最后报警最多的组件只是承受了连锁反应。

## 一条很实用的定位顺序

如果你想给面试官一个更工程化的回答，可以直接按这条线说：

1. 先确认慢的是平均 RT、P99，还是成功率。
2. 再对齐最近的流量、发布、活动和配置变化。
3. 然后按入口、应用、缓存、数据库、MQ、下游这几层拉监控。
4. 找出最先被打满的资源，是线程、连接、CPU、I/O 还是消费能力。
5. 最后再决定动作：是改 SQL、补缓存、扩容、异步化，还是限流降级。

这条线的重点是：

**先定位，再优化。**

## 常用观察点可以怎么落地

面试里不一定要背命令，但能说出观察点会更像做过线上排查。

| 层次     | 常用观察点                                       |
| -------- | ------------------------------------------------ |
| Linux    | CPU 使用率、load、上下文切换、磁盘 I/O、网卡流量 |
| JVM      | GC 次数和耗时、堆使用、线程状态、阻塞栈          |
| 应用     | 线程池队列、连接池等待、接口 P99、错误码         |
| MySQL    | 慢查询、执行计划、活跃连接、锁等待、主从延迟     |
| Redis    | 命中率、慢查询、大 key、热 key、网络延迟         |
| MQ       | 生产速率、消费速率、积压、重试、死信             |
| 下游依赖 | RPC RT、超时率、限流数、连接池水位               |

可以再把观察点翻译成“看到什么就查什么”：

| 现象                         | 初步判断                   | 下一步动作                        |
| ---------------------------- | -------------------------- | --------------------------------- |
| CPU 不高但线程池队列变长     | 等待外部资源或锁           | 看线程栈、连接池等待、下游 RT     |
| CPU 高且 GC 时间上升         | 对象创建、内存压力或大查询 | 看 GC 日志、堆对象、接口分布      |
| MySQL `rows_examined` 飙升   | SQL 扫描行数失控           | 看慢查询、执行计划、索引和分页    |
| Redis 命中率下降             | 缓存穿透、击穿或热点变化   | 看 key 分布、过期策略、回源数据库 |
| MQ 消费速率低于生产速率      | 消费能力不足或下游慢       | 看单条消费耗时、重试、分区并行度  |
| 下游超时率上升且上游线程堆积 | 依赖抖动拖垮调用方         | 缩短超时、限流隔离、熔断或降级    |

如果只能说“看监控”，还不够。
更稳的表达是：

1. 先看用户侧 RT 和错误率确认影响面。
2. 再看应用线程、连接、GC 判断是不是本服务卡住。
3. 然后看数据库、缓存、MQ、下游哪个指标先偏离基线。
4. 最后用日志、链路追踪或压测复现验证猜测。

定位结论最好能落到一句话：

**瓶颈是哪个资源先打满，以及它为什么会在这个时间点打满。**

## 一个订单接口 P99 飙升的排查例子

假设订单详情接口平时 P99 是 120ms，活动开始后升到 2s，但平均 RT 只有轻微上涨。

这时候不要直接说“加机器”，可以按时间线查：

```text
活动开始
 -> 某个热门商品订单查询集中
 -> Redis 热 key 命中率波动
 -> 部分请求回源 MySQL
 -> MySQL 慢查询和连接等待升高
 -> 应用线程池队列变长
 -> P99 被少量慢请求拉高
```

对应动作也应该分层：

1. 入口层先确认是不是某几个实例流量过高，必要时摘慢节点或调整权重。
2. 应用层看线程池队列和慢请求日志，确认请求卡在缓存、数据库还是下游。
3. 缓存层看热 key、命中率、Redis 慢日志和回源量。
4. 数据库层看慢 SQL、扫描行数、连接等待和锁等待。
5. 修复后用同一类流量复测 P99，而不是只看平均 RT 回落。

这个例子的重点是：

**P99 问题经常不是全链路都慢，而是少数请求被某个资源拖住。**

## 修复后要有观察窗口和回滚策略

性能优化不是改完就结束。

上线后至少要观察：

- P50、P95、P99 是否一起改善。
- 错误率和超时率有没有反弹。
- 数据库、缓存、MQ、下游有没有被转移压力。
- 线程池和连接池水位是否回到正常区间。
- 新策略是否只对目标流量生效，没有误伤其他链路。

如果优化动作比较重，比如切流量、改索引、扩容消费者、调整限流阈值，就要提前准备回滚：

| 优化动作     | 回滚关注点                       |
| ------------ | -------------------------------- |
| 网关规则调整 | 能否快速恢复旧规则               |
| 缓存策略调整 | 是否会造成缓存击穿或脏数据       |
| SQL / 索引   | 是否有灰度开关，是否影响写入     |
| MQ 扩容      | 下游是否能承受更高并发           |
| 线程池调整   | 是否把排队压力转移到数据库或下游 |

## 容易踩的坑

### 一慢就分库分表

分库分表是容量扩展动作，不是默认的性能优化第一步。

很多系统在上分库分表之前，慢 SQL、索引、深分页、缓存、读写分离都还没做完。

### 只盯平均值

平均 RT 好看，不代表高峰期用户体验好。

高性能题里，`P99/P999` 往往比平均值更有诊断价值。

### 只看应用，不看依赖

一个下游超时，完全可以把上游线程池拖满，表面看像应用慢，实际根因在外部依赖。

### 只做压测，不对齐生产流量

压测平均 QPS 很高，不代表生产就稳。

真实生产里可能有：

- 热点 key
- 热点商家 / 商品
- 大消息
- 慢下游
- 突发重试
- 冷缓存启动

所以压测数据要尽量贴近真实流量模型，否则结论只能说明“压测脚本没问题”。

## 小结

- 性能定位先回答“慢在哪一层”，再回答“用什么方案”。
- 接口 RT、应用资源、缓存、数据库、MQ、下游依赖要一起看，别只盯一层。
- 很多性能问题是流量模型变化或热点集中，不一定是代码本身变差。
- 真正要找的是最先被打满的资源，而不是最后表现最明显的症状。
- 结论要能落到具体资源：线程、连接、CPU、I/O、锁、缓存命中率或消费能力。
