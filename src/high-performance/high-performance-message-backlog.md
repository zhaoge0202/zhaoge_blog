# MQ 消息积压怎么排查？

> 消息积压最怕一句“加消费者就行”。先搞清楚是生产太猛、消费太慢，还是并行度根本起不来。

MQ 积压本质上是一个速率问题：

```text
生产速率 > 消费速率
```

但这只是表象。
真正往下排查时，至少要区分三类根因：

1. 上游生产突然变猛了
2. 下游消费变慢了
3. 队列 / 分区 / 路由设计让并行度起不来

不分清这三类，“加机器”很容易白加。

## 先看积压是不是还在持续扩大

第一步先确认 backlog 是：

- 短时尖峰后在回落
- 还是一直单边增长

如果只是大促瞬间冲高，但十几分钟后能逐步消化，那它更像容量波峰。

如果积压持续增长，就说明消费能力长期小于生产能力，必须找根因。

## 第一条线：是不是生产侧突然放大了

常见场景包括：

- 活动流量冲高
- 重试风暴
- 定时任务误触发
- 上游 bug 导致同类消息暴涨

这个时候要先看：

- 入队速率是否明显高于历史
- 是单个 Topic / 队列暴涨，还是全局都涨
- 是否和发布、活动、配置变更同时发生

如果源头是流量异常，不先止血，只扩消费者通常追不上。

## 第二条线：消费为什么变慢了

积压排查里最常见的根因其实在消费端。

### 1. 消费逻辑本身变慢

典型原因：

- 慢 SQL
- 外部接口变慢
- 锁竞争
- 大对象序列化 / 反序列化
- 批处理过小或过大

这时常见现象是：

- 单条消息处理耗时升高
- 消费线程活着，但吞吐下降
- 下游数据库 / Redis / RPC 也同步变慢

### 2. 消费端重试过多

如果大量消息都在失败重试，表面看是“消费者在工作”，实际有效吞吐很低。

这时要看：

- 重试量
- 死信量
- 同类异常是否集中爆发

很多时候真正拖慢消费的不是新消息，而是失败消息来回打转。

### 3. 消费端资源被打满

比如：

- 线程池满了
- 连接池获取慢
- JVM GC 抖动
- 容器 CPU 被限

这类问题不能只看 MQ 指标，要把应用资源一起拉进来。

## 第三条线：并行度是不是根本没起来

这是很容易被忽略的一类问题。

你明明加了很多消费者实例，但吞吐还是上不去，常见原因是：

- 队列数量太少
- 分区数量太少
- 单分区顺序消费限制太强
- 某类消息被某个 key 严重倾斜

比如 Topic 只有 4 个分区，你加到 20 个消费者，也不会凭空变成 20 路并发。

所以积压排查时必须问一句：

**当前并行度上限到底受什么约束？**

## 一个很实用的排查顺序

如果线上已经出现积压，可以按这条线走：

1. 先看 backlog 是不是持续增长。
2. 对比生产速率和消费速率，看问题在入口还是在消费侧。
3. 如果是生产暴涨，先止血限流或降级。
4. 如果是消费变慢，继续查 SQL、RPC、Redis、线程池、GC。
5. 如果消费逻辑本身没明显变慢，再查分区数、队列数、顺序约束和热点倾斜。

这条顺序能避免一上来盲目扩容。

## 应急动作和根因修复要分开

线上真积压时，通常要分两层：

### 先止血

- 限流上游
- 临时扩消费者
- 增加分区 / 队列（前提是框架和业务允许）
- 非核心消息先降级或延后

### 再修根因

- 慢 SQL 优化
- 下游依赖恢复
- 重试策略调整
- 热点 key / 热点分区治理
- 消费逻辑拆批或并行化

只做止血，不修根因，积压还会回来。

## 什么时候“加消费者”真的有效

只有在这几种前提下才更有效：

- 分区 / 队列还有并行空间
- 消费逻辑本身不是瓶颈
- 下游依赖还有余量
- 当前没有顺序消费约束卡住

否则你只是多部署几台“同样吃不动”的消费者。

## 容易踩的坑

### 只看 MQ，不看消费业务

很多积压的根因根本不在 Broker，而在消费者里的一条慢 SQL 或一个下游超时。

### 一扩容就把下游也打穿

如果消费端的瓶颈其实在数据库，盲目加消费者可能把数据库先打挂。

### 不区分“峰值积压”和“永久积压”

短时积压可以靠缓冲消化，长期积压说明系统容量设计已经不匹配。

## 小结

- 消息积压本质是生产速率和消费速率失衡，但根因通常分成生产暴涨、消费变慢、并行度不足三类。
- 排查时先看 backlog 是否持续增长，再比较生产 / 消费速率。
- 很多积压问题真正卡在消费逻辑、下游依赖或分区并行度，而不是 Broker 本身。
- 应急动作和根因修复要分开，先止血，再修慢点。
- “加消费者”只有在并行度和下游容量都还有空间时才真有用。
