# MQ 消息积压怎么排查？

> 消息积压最怕一句“加消费者就行”。先搞清楚是生产太猛、消费太慢，还是并行度根本起不来。

MQ 积压本质上是一个速率问题：

```text
生产速率 > 消费速率
```

但这只是表象。
真正往下排查时，至少要区分四类根因：

1. 上游生产突然变猛了
2. 下游消费变慢了
3. 队列 / 分区 / 路由设计让并行度起不来
4. Broker 自身资源或配置出了问题

不分清这三类，“加机器”很容易白加。

## 第一步：先判断积压趋势和影响范围

看到 backlog 上升，不要立刻扩容。
先把两个问题问清楚：

1. 积压是不是还在持续扩大？
2. 积压影响的是单个业务，还是整个 MQ 集群？

### 先看趋势

如果只是大促瞬间冲高，但十几分钟后能逐步消化，那它更像容量波峰。

如果积压持续增长，就说明消费能力长期小于生产能力，必须找根因。

可以按这张表快速判断：

| 现象                       | 初步判断                 | 处理重点                         |
| -------------------------- | ------------------------ | -------------------------------- |
| backlog 突然升高后缓慢下降 | 短时峰值                 | 观察消化速度，确认是否超过 SLA   |
| backlog 持续单边增长       | 生产长期大于消费         | 立刻定位入口、消费端或并行度瓶颈 |
| backlog 锯齿状波动         | 批任务、重试或周期性流量 | 对齐定时任务、发布、活动时间     |
| backlog 不高但延迟很高     | 单条处理慢或顺序卡住     | 查消费耗时、热点 key、阻塞消息   |

### 再看范围

范围决定排查方向：

| 范围                    | 更可能的问题                           |
| ----------------------- | -------------------------------------- |
| 单个 Topic / Queue 积压 | 某个业务生产暴涨、消费逻辑变慢         |
| 单个分区 / 队列积压     | 热点 key、顺序消费、分区倾斜           |
| 一个消费组积压          | 消费组代码、实例、线程池或下游依赖     |
| 多个业务同时积压        | Broker、网络、磁盘、集群资源或公共依赖 |

先定范围，能避免从错误层开始排查。

## 先把指标拉齐：不要只看 backlog

排查积压时，至少要把这些指标放在一起看：

| 指标            | 说明                           | 典型用途                       |
| --------------- | ------------------------------ | ------------------------------ |
| 生产速率        | 每秒写入多少消息               | 判断是不是入口流量暴涨         |
| 消费速率        | 每秒成功消费多少消息           | 判断消费能力是否下降           |
| 消费延迟        | 消息从入队到处理完成的时间     | 判断业务是否已经受影响         |
| backlog / lag   | 未消费消息量或 offset 差距     | 判断积压规模                   |
| 单条处理耗时    | 消费者处理一条消息要多久       | 定位慢 SQL、RPC、锁竞争        |
| 重试量 / 死信量 | 失败消息有多少                 | 判断是不是失败消息拖慢有效吞吐 |
| 分区 / 队列倾斜 | 各分区 backlog 是否均衡        | 定位热点 key 或分区设计问题    |
| Broker 资源     | 磁盘、IO、网络、内存、句柄水位 | 判断 MQ 集群自身是否成为瓶颈   |

只看 backlog 容易误判。
比如 backlog 在涨，可能是生产暴涨；也可能是消费者线程池满了；还可能是一个分区被热点 key 打穿。

## 第一条线：是不是生产侧突然放大了

常见场景包括：

- 活动流量冲高
- 重试风暴
- 定时任务误触发
- 上游 bug 导致同类消息暴涨
- 补偿脚本批量重放历史消息
- 新版本把原本同步调用改成了批量发消息

这个时候要先看：

- 入队速率是否明显高于历史
- 是单个 Topic / 队列暴涨，还是全局都涨
- 是否和发布、活动、配置变更同时发生
- 消息的业务类型、来源应用、用户或商户是否集中
- 是否有重复消息、异常补偿消息或错误路由消息

如果源头是流量异常，不先止血，只扩消费者通常追不上。

### 生产侧止血怎么做

如果确认是生产暴涨，优先目标不是“马上清空积压”，而是先让积压别继续扩大。

常见动作：

- 对上游入口限流。
- 暂停非核心生产者。
- 降级低优先级消息。
- 暂停异常补偿脚本或定时任务。
- 把大批量任务改成分批投递。
- 对明显重复或无效的消息在入口侧拦截。

注意：如果消息是核心业务事件，不能简单丢。
更稳的是降速、分批、延后，而不是把数据直接抛掉。

## 第二条线：消费为什么变慢了

积压排查里最常见的根因其实在消费端。

### 1. 消费逻辑本身变慢

典型原因：

- 慢 SQL
- 外部接口变慢
- 锁竞争
- 大对象序列化 / 反序列化
- 批处理过小或过大
- 消费逻辑里加了新校验或新下游调用
- 单条消息携带数据过大

这时常见现象是：

- 单条消息处理耗时升高
- 消费线程活着，但吞吐下降
- 下游数据库 / Redis / RPC 也同步变慢
- 应用 CPU、连接池、线程池或 GC 指标异常

### 2. 消费端重试过多

如果大量消息都在失败重试，表面看是“消费者在工作”，实际有效吞吐很低。

这时要看：

- 重试量
- 死信量
- 同类异常是否集中爆发

很多时候真正拖慢消费的不是新消息，而是失败消息来回打转。

这类问题要特别小心。
如果消费者一直拿到同一批失败消息，正常新消息就会被挤在后面。
处理方式通常是：

- 对临时异常做退避重试，不要立即无限重试。
- 对参数错误、状态非法这类不可恢复异常，尽快进死信或异常表。
- 对死信建立处理入口，修复后再安全重放。
- 失败消息重放前必须确认消费幂等。

### 3. 消费端资源被打满

比如：

- 线程池满了
- 连接池获取慢
- JVM GC 抖动
- 容器 CPU 被限
- Pod / 机器频繁重启
- 消费者心跳异常导致频繁重平衡

这类问题不能只看 MQ 指标，要把应用资源一起拉进来。

### 4. 消费组发布或重平衡造成抖动

一些积压不是业务逻辑慢，而是消费组一直在抖。

典型情况：

- 消费者频繁上下线。
- 发布时一次性重启全部实例。
- 单批消息处理太久，消费者被认为失联。
- 分区重新分配期间，消费短暂停顿。

这时会看到：

- 消费速率周期性归零或下降。
- 重复消费增加。
- 部分分区 lag 突然跳高。

治理方式包括：

- 滚动发布，避免全量重启。
- 控制单批拉取数量。
- 把耗时任务拆小，避免单次处理超过心跳或 poll 间隔。
- 消费端保持幂等，接受重平衡带来的重复投递。

## 第三条线：并行度是不是根本没起来

这是很容易被忽略的一类问题。

你明明加了很多消费者实例，但吞吐还是上不去，常见原因是：

- 队列数量太少
- 分区数量太少
- 单分区顺序消费限制太强
- 某类消息被某个 key 严重倾斜

比如 Topic 只有 4 个分区，你加到 20 个消费者，也不会凭空变成 20 路并发。

所以积压排查时必须问一句：

**当前并行度上限到底受什么约束？**

### 分区 / 队列数决定了消费并行度上限

不同 MQ 的实现细节不同，但常见规律类似：

- 同一消费组里，一个分区或队列通常不能被无限多个消费者同时有效消费。
- 分区 / 队列数量太少，消费者实例再多也可能空闲。
- 如果要求单 key 顺序消费，同一个 key 的消息必须串行处理。

所以扩容消费者前先看：

| 问题                          | 如果答案是否定的，扩容效果 |
| ----------------------------- | -------------------------- |
| 分区 / 队列数量还有空余吗？   | 很可能无效                 |
| 下游数据库或 RPC 还有余量吗？ | 可能把下游打穿             |
| 消费逻辑能并行处理吗？        | 可能被锁或顺序约束卡住     |
| 热点 key 是否分散？           | 可能只有单分区在忙         |

### 热点 key 会让“总分区数够”也没用

比如一个订单系统按 `shop_id` 做分区。
如果某个大商家在活动期间产生了 80% 的消息，它所在分区就会堆积，而其他分区很空。

这时加消费者未必有效，因为热点分区还是只有有限消费者能处理。

更合理的治理方式可能是：

- 调整分区 key，让热点打散。
- 对大商家或大用户做单独 Topic。
- 对热点业务单独扩队列、拆 Topic 或拆消费逻辑。
- 如果必须按 key 有序，就接受吞吐上限，并提前做容量评估。

## 第四条线：Broker 自己是不是扛不住了

虽然很多积压根因在消费端，但 Broker 自身也可能出问题。

常见信号：

- 磁盘使用率高。
- 磁盘 IO 打满。
- 网络带宽打满。
- 内存水位过高，开始限流生产者。
- 文件句柄或连接数耗尽。
- 副本同步延迟变大。
- Controller / NameServer / 元数据组件异常。

如果多个业务同时积压，或者生产、消费都变慢，就要把 Broker 集群拉进来查。

重点看：

- Broker 是否有频繁重启。
- 磁盘剩余空间和写入延迟。
- Topic / Queue 是否达到保留时间或容量上限。
- 副本同步是否落后。
- 是否存在大量未确认消息。
- 是否有全局背压或限流。

Broker 问题不能靠加消费者解决。
先恢复 MQ 集群自身能力，再谈消费扩容。

## 一个很实用的排查顺序

如果线上已经出现积压，可以按这条线走：

1. 先看 backlog 是不是持续增长。
2. 看积压范围：单 Topic、单分区、单消费组，还是全局。
3. 对比生产速率和消费速率，看问题在入口还是在消费侧。
4. 如果是生产暴涨，先止血限流或降级。
5. 如果是消费变慢，继续查 SQL、RPC、Redis、线程池、GC、重试和死信。
6. 如果消费逻辑本身没明显变慢，再查分区数、队列数、顺序约束和热点倾斜。
7. 如果多个业务同时积压，再查 Broker 磁盘、网络、内存、副本和集群状态。

这条顺序能避免一上来盲目扩容。

## 应急动作和根因修复要分开

线上真积压时，通常要分两层：

### 先止血

- 限流上游
- 临时扩消费者
- 增加分区 / 队列（前提是框架和业务语义允许）
- 非核心消息先降级或延后
- 暂停异常重试或补偿脚本
- 把明显不可恢复的失败消息转入死信

### 再修根因

- 慢 SQL 优化
- 下游依赖恢复
- 重试策略调整
- 热点 key / 热点分区治理
- 消费逻辑拆批或并行化

只做止血，不修根因，积压还会回来。

## 历史积压怎么处理

积压已经形成后，除了恢复实时消费，还要考虑历史消息怎么追。

常见方案有三种：

### 1. 原消费组继续追

适合积压不大、业务还能接受延迟的场景。

优点是简单，不容易绕开原来的幂等和状态机。
缺点是恢复时间可能较长。

### 2. 临时扩容或临时消费组批处理

适合历史积压很大，但消息可以并行处理的场景。

要注意：

- 不要绕过消费幂等。
- 不要绕过业务限流。
- 不要把数据库或下游服务打穿。
- 处理完要下线临时任务，避免重复消费。

### 3. 从业务库重建并重新投递

如果消息已经过期、被清理，或者死信太多，可以从业务库按状态重建事件。

比如：

- 查询已支付但未发货的订单。
- 查询已完成但未发券的活动记录。
- 查询消费记录长时间 `PROCESSING` 的业务单。

重新投递前必须保证幂等，否则补偿会变成重复发货、重复发券。

## 什么时候“丢弃”是可以接受的

很多人听到积压就默认所有消息都不能丢。
但不同消息的业务价值不同：

| 消息类型     | 是否能丢弃                 |
| ------------ | -------------------------- |
| 支付、订单   | 通常不能丢，必须补偿和对账 |
| 库存、履约   | 通常不能丢，但可以延后处理 |
| 通知、短信   | 可按业务等级决定是否补发   |
| 日志、埋点   | 可以采样、丢弃或低峰补导   |
| 监控诊断数据 | 可按保留策略和成本取舍     |

所以处理积压前要先给消息分级。
核心业务事件以补偿和对账为主，低价值消息可以降级或丢弃，避免拖垮主链路。

## 什么时候“加消费者”真的有效

只有在这几种前提下才更有效：

- 分区 / 队列还有并行空间
- 消费逻辑本身不是瓶颈
- 下游依赖还有余量
- 当前没有顺序消费约束卡住

否则你只是多部署几台“同样吃不动”的消费者。

可以用一个简单判断：

```text
有效吞吐 = min(消费者处理能力, 分区并行度, 下游容量, Broker 能力)
```

加消费者只是在提高第一项。
如果瓶颈在后三项，加消费者不会解决问题。

## 平时怎么预防积压

积压治理最好不要等事故时才做。

上线前要提前准备：

- 给核心 Topic 配生产速率、消费速率、lag、消费延迟告警。
- 压测消费者单实例吞吐和下游承载能力。
- 给重试和死信设置上限，避免失败消息无限打转。
- 设计消息分级，明确哪些必须补偿，哪些可以降级。
- 评估分区 / 队列数是否匹配峰值并行度。
- 发布消费者时滚动发布，避免消费组大面积停顿。
- 保留重放和补偿工具，但必须带幂等保护。

这部分和高可用专题里的压测、限流、幂等是连着的。
MQ 不是只要能收发消息就算上线完成，核心链路还要能观测、能限流、能补偿。

## 容易踩的坑

### 只看 MQ，不看消费业务

很多积压的根因根本不在 Broker，而在消费者里的一条慢 SQL 或一个下游超时。

### 一扩容就把下游也打穿

如果消费端的瓶颈其实在数据库，盲目加消费者可能把数据库先打挂。

### 不区分“峰值积压”和“永久积压”

短时积压可以靠缓冲消化，长期积压说明系统容量设计已经不匹配。

### 不看分区倾斜

总 backlog 不高，不代表没问题。
单分区或单队列积压也可能让某一类订单长时间不处理。

### 无限重试毒性消息

不可恢复的失败消息一直回队列，会拖慢整个消费组。
应该进入死信或异常表，修复后再重放。

### 历史积压追赶时绕过幂等

临时脚本最容易漏掉幂等和限流。
一旦重放重复，就可能重复发货、重复发券或重复写流水。

## 面试里可以这样回答

如果被问“MQ 消息积压怎么排查”，可以按这条线说：

1. 先看 backlog 趋势，是短时峰值还是持续增长。
2. 再看影响范围，是单 Topic、单分区、单消费组，还是全局积压。
3. 对比生产速率和消费速率，区分生产突增还是消费下降。
4. 生产突增先限流、降级、暂停异常任务；消费变慢就查慢 SQL、下游、线程池、GC、重试和死信。
5. 如果消费者扩了还没用，继续查分区 / 队列数、顺序消费和热点 key。
6. 如果多个业务一起积压，要查 Broker 磁盘、网络、内存、副本同步和集群状态。
7. 应急止血和根因修复要分开，历史积压要按业务等级选择追赶、重放或降级。

## 小结

- 消息积压本质是生产速率和消费速率失衡，但根因可能在生产、消费、并行度或 Broker 资源。
- 排查时先看趋势和范围，再比较生产 / 消费速率，避免一上来盲目扩容。
- 很多积压问题真正卡在消费逻辑、下游依赖、重试死信、分区并行度或热点 key，而不是 Broker 本身。
- 应急止血和根因修复要分开，历史积压追赶必须配合幂等、限流和业务分级。
- “加消费者”只有在分区并行度、消费逻辑、下游容量和 Broker 能力都有余量时才真有用。
