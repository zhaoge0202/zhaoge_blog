# 分库分表之后查询、事务、扩容会变复杂在哪里？

> 分库分表不是“数据大了就拆”，它更像一笔交易：换来容量和吞吐，也把查询、事务、迁移复杂度一起买回来。

高性能题里，分库分表特别容易被答成一句口号：

> “单表太大就分库分表。”

这句话最大的问题是，它只说了收益，没说代价。

真实系统里，分库分表从来不是优化第一步，而是常规优化扛不住之后的容量扩展方案。

## 先说清：分库分表到底在解决什么

它主要解决两类问题：

1. 单库 / 单表数据量太大
2. 单机读写吞吐已经接近瓶颈

典型信号包括：

- 单表数据量巨大，索引维护和备份越来越慢
- 单库磁盘、连接数、写入 QPS 接近极限
- 常规索引、缓存、读写分离之后仍然扛不住

所以它解决的是：

**容量和吞吐扩展问题。**

不是所有慢 SQL 都应该靠它解决。

## 在动手拆之前，先问 4 个问题

1. 慢 SQL、索引、分页、缓存、读写分离都做过了吗？
2. 核心查询路径能不能被一个稳定分片键覆盖？
3. 业务能不能接受跨分片查询和最终一致事务？
4. 团队是否有迁移、扩容、路由、运维能力托住它？

如果这四个问题里前两项都答不稳，分库分表大概率会带来更多麻烦。

## 分库和分表，别混着讲

### 垂直拆分

按业务域拆：

- 用户库
- 订单库
- 商品库

它解决的是：

- 不同业务互相拖累
- 单库职责太重

### 水平拆分

按数据分片规则拆：

- 订单 1~N 打到不同库表
- 用户 1~N 打到不同库表

它解决的是：

- 单表太大
- 单节点吞吐不够

面试里一般真正往下追的，更多是水平分库分表。

## 分片键选得对不对，基本决定后面痛不痛苦

分片键理想状态下要同时满足：

- 高频查询能命中
- 分布尽量均匀
- 不容易变
- 扩容时别太难迁

但现实里通常做不到全满足，只能取舍。

比如订单系统：

- 按用户 ID 分，用户查订单方便，但商家维度可能麻烦
- 按商家 ID 分，商家后台方便，但用户视角可能变复杂
- 按订单号分布更均匀，但很多业务查询不好命中

所以分片键选择本质上是在回答：

**你最核心的查询路径是谁。**

## 分库分表之后，最先变复杂的通常是查询

### 1. 跨分片查询

以前一条 SQL 就够：

```sql
select * from t_order where user_id = ?;
```

拆完之后，如果路由键不命中，可能变成：

- 广播到多个分片
- 各分片分别查
- 应用层聚合结果

这样会带来：

- 查询更慢
- 排序和分页更难
- 带宽和连接消耗更高

### 2. JOIN 变麻烦

以前同库 JOIN 很自然，拆完后：

- 订单在库 A
- 商品在库 B

数据库层 JOIN 就很难做了。

常见替代方案是：

- 应用层多次查询后组装
- 冗余部分字段
- 宽表 / 搜索索引 / 报表侧承担复杂查询

所以分库分表往往会倒逼你重新设计查询模型。

## 第二个复杂点：事务不再是单库事务

以前一个本地事务就能完成：

- 写订单
- 扣库存
- 改账户

拆完后如果落在不同分片，问题就来了。

这时候你通常要在这些方案里选：

- 柔性事务 / 最终一致
- TCC / Saga
- 本地消息表 / 事务消息
- 极少场景用强一致分布式事务

也就是说：

**分库分表经常不是把事务问题解决了，而是把事务问题显式暴露出来。**

## 第三个复杂点：全局 ID 和路由

单库自增主键在分片场景里通常不够用了。

你需要解决：

- 多分片下全局唯一
- 趋势递增或可排序需求
- 路由时能不能反推出分片位置

所以常见会引入：

- 雪花 ID
- 号段模式
- 业务编码型 ID

全局 ID 不只是“保证唯一”，很多时候还会和分片路由绑定。

## 第四个复杂点：扩容和数据迁移

这往往是最容易被忽略、但线上最疼的一块。

因为分片规则一旦定了，不是想改就能改。

常见难点包括：

- 新增分片后老数据怎么迁
- 路由规则如何灰度切换
- 迁移期间怎么双写或校验
- 如何避免长时间停机

特别是哈希分片，扩容时历史数据重分布会很麻烦。

所以扩容从来不只是“多加几个库”，而是：

**路由变更 + 数据迁移 + 校验对账 + 回滚方案**

一起上。

## 分库分表之后，分页也会变复杂

这一点经常被漏答。

以前：

```sql
order by create_time desc limit 0, 20
```

拆完后如果数据散在多个分片，你可能要：

1. 每个分片先取一部分
2. 应用层再合并排序
3. 再截断分页结果

这样做不仅贵，而且深分页会更麻烦。

所以很多分片后的复杂列表页，最终会转到：

- 搜索引擎
- 宽表
- 离线明细表

## 一条更稳的面试回答方式

如果被问“分库分表会带来什么问题”，可以直接按这条线答：

1. 它先解决的是容量和吞吐扩展，不是第一步性能优化手段。
2. 拆完之后最先复杂的是查询：跨分片、JOIN、排序和分页。
3. 然后是事务：本地事务变成分布式事务，通常要接受最终一致。
4. 再然后是全局 ID、路由规则和分片键设计。
5. 最后是最麻烦的扩容和迁移，尤其是历史数据重分布和灰度切换。

这样答，不会停留在“知道有这回事”的层面。

## 容易踩的坑

### 为了“提前设计”过早分库分表

系统还没到瓶颈，先把复杂度背上，往往得不偿失。

### 分片键只看均匀，不看查询路径

分得再均匀，如果核心查询老是命不中，最后查询体验会很差。

### 只想着拆，不想着迁

分片规则改一次的成本通常比最初设计时想象得大得多。

## 小结

- 分库分表解决的是容量和吞吐扩展，不是数据库优化第一步。
- 它最大的代价在查询、事务、路由、全局 ID 和扩容迁移复杂度。
- 分片键选择本质上是在核心查询路径、分布均匀性和扩容成本之间做取舍。
- 跨分片查询、JOIN、排序和分页往往会倒逼你重构访问模型。
- 面试里答这题，重点不是“会拆”，而是知道拆完之后系统会在哪些地方变复杂。
