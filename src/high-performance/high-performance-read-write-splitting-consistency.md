# 读写分离会带来哪些一致性问题？

> 读写分离的价值是分散读压力，但它最容易被追问的点，从来不是“怎么路由”，而是“为什么刚写完却读不到”。

很多人讲读写分离时只会说：

- 一主多从
- 主库写，从库读
- 提升读性能

这些都对，但面试真正的追问基本都会落到一句：

**写完主库之后，马上去从库读，为什么可能还是旧数据？**

这就是读写分离最核心的一类一致性问题。

## 先把问题说清：读写分离优化了吞吐，但引入了复制延迟

典型架构是：

```text
应用
 -> 写请求到主库
 -> 读请求到从库
```

主库写完之后，binlog 还要同步到从库。
这个同步不是零成本，也不是绝对实时。

所以写完立刻读时，可能出现：

- 主库里已经是新值
- 从库还没追上

这就是最典型的“写后读不一致”。

## 读请求到底怎么被路由到从库

读写分离通常有两种落地方式：

| 方式         | 优点                       | 代价                         |
| ------------ | -------------------------- | ---------------------------- |
| 代理层路由   | 对业务代码侵入小，统一治理 | 代理本身要高可用，排障多一层 |
| 应用组件路由 | 应用能感知读主 / 读从语义  | 业务要管理路由上下文和规范   |

无论哪种方式，关键链路都要能表达“这次必须读主库”。

比如写请求成功后，可以在请求上下文里打一个短时标记：

```text
order:123 写入成功
 -> 3 秒内 order:123 相关读请求强制读主
 -> 过期后恢复普通读从策略
```

这比“所有读都走主库”更可控，也比完全透明路由更容易解释一致性边界。

## 主从复制可以拆成三个时间点看

读写分离的延迟，本质上来自复制链路。

可以先把主从复制简化成这条线：

```text
主库提交事务
 -> 写 binlog
 -> 从库 I/O 线程拉取并写 relay log
 -> 从库 SQL 线程回放 relay log
 -> 从库可以读到新数据
```

再抽象成三个时间点：

| 时间点 | 含义                               |
| ------ | ---------------------------------- |
| T1     | 主库事务提交并写入 binlog          |
| T2     | 从库接收到 binlog 并写入 relay log |
| T3     | 从库回放完成，数据真正可见         |

`T2 - T1` 更偏网络和主从传输问题。
`T3 - T2` 更偏从库回放能力问题。

所以说“主从延迟”时，不要只说一个抽象词。
最好能继续说明：到底是日志传不过来，还是从库回放不过来。

## 最常见的一致性问题有哪些

### 1. 写后立刻读，读到旧数据

这是最经典的场景。

比如：

1. 用户刚下单成功
2. 立刻查询订单详情
3. 查询请求被路由到从库
4. 从库还没同步到这条订单

结果就是用户看到“订单不存在”。

### 2. 同一个用户连续两次读，结果越读越旧

如果一次读落在主库，一次读落在从库，或者两次读落在不同延迟程度的从库，就可能出现：

- 第一次读到新数据
- 第二次反而读到旧数据

这类问题常被叫作“读不单调”，对用户体验也很差。

### 3. 不同业务链路对同一份数据看到的版本不一样

例如：

- 账户服务刚更新余额
- 查询服务从从库读
- 风控服务从主库读

结果多个服务对同一时刻的数据理解不一致。

如果链路里又带缓存，这个问题会更绕。

## 主从延迟为什么会发生

根因通常不只一个。

常见原因包括：

- 从库机器规格比主库差
- 从库读流量太大
- 大事务或慢 SQL 导致回放变慢
- 从库数量太多，主库同步压力大
- 网络抖动
- 复制线程并行能力不足

所以“有主从延迟”不是异常，而是默认要面对的现实。

## 主从延迟应该怎么观测

线上不能只靠用户反馈“刚写完看不到”。
至少要有复制延迟和读路由两类观测。

| 观察点       | 说明                                       |
| ------------ | ------------------------------------------ |
| 复制延迟     | 从库落后主库多久，是否持续扩大             |
| relay log    | 日志是否已经到从库，但还没回放             |
| SQL 线程状态 | 从库回放是否被大事务、慢 SQL 或锁等待卡住  |
| 从库负载     | 从库 CPU、I/O、连接数是否被读请求打满      |
| 路由命中     | 写后读请求到底去了主库、从库，还是不同从库 |

MySQL 8 里常见命令是：

```sql
SHOW REPLICA STATUS\G
```

老版本或旧语境里也会看到：

```sql
SHOW SLAVE STATUS\G
```

面试里不需要背所有字段，但要知道它们解决的是“从库落后多少、卡在哪一步”。
如果系统用了数据库中间件或读写分离组件，还要看应用侧路由日志，否则很难判断请求到底读到了哪台库。

延迟治理要把原因和动作对应起来：

| 延迟原因       | 治理动作                                   |
| -------------- | ------------------------------------------ |
| 从库规格太低   | 提升规格，避免从库承担过多分析查询         |
| 从库读流量过高 | 加缓存、拆读流量、摘除热点从库             |
| 大事务回放慢   | 拆小批次，避免长事务和大批量更新           |
| 慢 SQL 卡回放  | 优化从库执行路径，避免回放被锁或慢查询拖住 |
| 网络抖动       | 看跨机房链路、带宽、丢包和复制通道稳定性   |
| 并行复制不足   | 结合版本能力评估并行复制参数和业务写入模型 |

注意，这些动作是“尽量降低延迟”，不是承诺读写分离完全没有延迟。

## 先别幻想绝对没有延迟，先看业务能不能接受

这里最关键的是业务分层。

不是所有读都必须强一致。

可以先这样分：

| 场景                           | 一致性要求         |
| ------------------------------ | ------------------ |
| 支付后查余额、提交后查订单状态 | 强一些             |
| 推荐列表、浏览数、普通详情页   | 可接受短暂最终一致 |
| 报表、统计、搜索索引           | 更能接受延迟       |

所以读写分离设计不是一句“全量读从库”，而是：

**哪些读必须主库，哪些读可以容忍从库延迟。**

## 常见的处理办法，不是只有“全都查主库”

### 1. 核心链路强制读主

对极少数必须马上看到最新结果的请求，直接路由主库。

典型场景：

- 支付成功后查支付状态
- 下单成功后查订单详情
- 用户刚改完密码就查账户安全信息

这个办法最直接，但不能滥用。

因为如果大量读请求都回退主库，读写分离的价值就没了。

### 2. 业务流程上避免“写后立刻查”

有些链路不是非得写完立刻读。

比如：

- 提交成功后先返回结果页
- 再异步刷新状态
- 或者让用户稍后进入详情页

这本质上是在用业务交互吸收一部分复制延迟。

### 3. 对读一致性要求高的场景做会话级策略

常见做法是：

- 某次写之后一段时间内，同一用户相关读都走主库
- 或者记录最近写时间，在短窗口内强制主读

这能降低“自己刚写完自己看不到”的概率。

更工程化一点，可以把它做成“短时读主令牌”：

1. 写请求成功后，在用户维度、订单维度或业务 key 维度记录一个短 TTL 标记。
2. 这个标记存在期间，相关读请求强制走主库。
3. TTL 过期后恢复读从库。
4. 如果监控发现从库延迟超过阈值，也可以临时扩大读主窗口或熔断部分从库。

这样比“所有读都走主库”更可控。
关键是标记粒度要小，时间窗口要短，否则主库会被读流量打回去。

### 4. 接受最终一致，并把界面文案讲明白

对非核心链路，可以直接承认它是最终一致。

比如：

- “数据同步中”
- “请稍后刷新”

这比强行把所有查询都打回主库更现实。

### 5. 对从库延迟做降级和摘除

如果某台从库明显落后，就不要继续把强依赖新数据的请求路由过去。

常见策略包括：

- 延迟超过阈值的从库临时摘除。
- 关键读请求在延迟过高时走主库。
- 非关键读请求展示“数据同步中”。
- 长时间延迟时触发告警，排查大事务、慢 SQL、网络和从库负载。

这里的边界也要讲清楚：

**读主兜底能保护一致性，但会牺牲主库读压力。**

所以它必须配合限流、降级和监控，而不是无脑切主。

## 一个很重要的误区：半同步复制不等于完全没延迟

很多人会觉得：

> 开了半同步复制，不就一致了吗？

这不对。

半同步的意义是提高数据安全性，让主库在某些阶段等待至少一个从库确认收到日志。

但它不等于：

- 所有从库都已经追平
- 查询一定不会读到旧数据

所以半同步能降低部分风险，但不能把所有读一致性问题一把抹掉。

还有一个容易混淆的点：

**数据安全性和读一致性不是同一个目标。**

半同步更偏“主库返回成功前，至少有从库收到日志”，它减少的是主库宕机后数据丢失风险。
但应用读某台从库时，这台从库是否已经回放完成，仍然要看复制进度和读路由。

可以用一张表记住三种复制取舍：

| 模式   | 大致特点                       | 主要代价                     |
| ------ | ------------------------------ | ---------------------------- |
| 异步   | 写入吞吐好，主库提交不等从库   | 主库故障时可能丢最近一段数据 |
| 半同步 | 至少等待部分从库收到日志再返回 | 写入延迟上升，仍可能读到旧值 |
| 全同步 | 多副本确认后才提交，安全性更强 | 写入性能和可用性代价更高     |

面试里最容易答错的是把“更安全”说成“读一定最新”。
复制模式改善的是数据丢失窗口，不自动解决所有读路由问题。

## 读写分离题更像“性能和一致性的交换”

它的本质不是纯技术动作，而是取舍：

| 你想要的     | 你可能付出的代价     |
| ------------ | -------------------- |
| 更多读吞吐   | 主从延迟、一致性变弱 |
| 更强读一致性 | 更多主库压力         |
| 更少主库压力 | 业务更要接受最终一致 |

所以回答这题时，别只讲“怎么实现”，最好补一句：

**读写分离本质是在读性能和读一致性之间做分层取舍。**

## 一条更稳的回答顺序

如果被问“读写分离会带来什么问题”，可以直接这样答：

1. 读写分离通过主写从读提升吞吐，但一定会引入主从复制延迟。
2. 最典型的问题是写后读不一致、连续读不单调、不同链路看到不同数据版本。
3. 处理时不能简单粗暴地所有查询都回主库，而要按业务一致性要求分层。
4. 对核心链路强制读主，对非核心链路接受最终一致，必要时配合会话级短时主读。
5. 半同步复制只能降低部分风险，不能等同于完全没有延迟。

如果想再往工程落地上补一句，可以说：

6. 线上还要监控复制延迟、从库负载和读路由命中；延迟过高时要能摘除从库或短时读主。

## 容易踩的坑

### 把读写分离当成透明优化

如果业务代码完全不知道读请求去了哪里，就很难处理写后读一致性。
读写分离可以封装，但关键链路必须能显式表达“这次要读主库”。

### 在事务里读错库

一个请求里先写后读，如果读被路由到从库，就可能在同一业务流程里看到旧数据。
这类链路应该在事务边界或请求上下文里明确读主。

### 从库越多越好

从库能分摊读压力，但从库太多也会增加复制、运维和延迟治理复杂度。
更稳的方式是按业务读模型扩展，而不是为了数量好看盲目加从库。

## 小结

- 读写分离最核心的代价不是实现复杂，而是主从延迟带来的读一致性问题。
- 最常见的问题是写后读旧数据、连续读不单调，以及不同链路看到不同版本。
- 主从延迟要拆成日志传输和从库回放两段看，线上要监控复制延迟和读路由命中。
- 核心链路可以短时读主，非核心链路更适合接受最终一致，并配合从库延迟摘除。
- 读写分离本质上是在读吞吐和读一致性之间做取舍，不是白拿性能。
