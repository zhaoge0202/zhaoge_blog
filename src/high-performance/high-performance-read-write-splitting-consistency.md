# 读写分离会带来哪些一致性问题？

> 读写分离的价值是分散读压力，但它最容易被追问的点，从来不是“怎么路由”，而是“为什么刚写完却读不到”。

很多人讲读写分离时只会说：

- 一主多从
- 主库写，从库读
- 提升读性能

这些都对，但面试真正的追问基本都会落到一句：

**写完主库之后，马上去从库读，为什么可能还是旧数据？**

这就是读写分离最核心的一类一致性问题。

## 先把问题说清：读写分离优化了吞吐，但引入了复制延迟

典型架构是：

```text
应用
 -> 写请求到主库
 -> 读请求到从库
```

主库写完之后，binlog 还要同步到从库。
这个同步不是零成本，也不是绝对实时。

所以写完立刻读时，可能出现：

- 主库里已经是新值
- 从库还没追上

这就是最典型的“写后读不一致”。

## 最常见的一致性问题有哪些

### 1. 写后立刻读，读到旧数据

这是最经典的场景。

比如：

1. 用户刚下单成功
2. 立刻查询订单详情
3. 查询请求被路由到从库
4. 从库还没同步到这条订单

结果就是用户看到“订单不存在”。

### 2. 同一个用户连续两次读，结果越读越旧

如果一次读落在主库，一次读落在从库，或者两次读落在不同延迟程度的从库，就可能出现：

- 第一次读到新数据
- 第二次反而读到旧数据

这类问题常被叫作“读不单调”，对用户体验也很差。

### 3. 不同业务链路对同一份数据看到的版本不一样

例如：

- 账户服务刚更新余额
- 查询服务从从库读
- 风控服务从主库读

结果多个服务对同一时刻的数据理解不一致。

如果链路里又带缓存，这个问题会更绕。

## 主从延迟为什么会发生

根因通常不只一个。

常见原因包括：

- 从库机器规格比主库差
- 从库读流量太大
- 大事务或慢 SQL 导致回放变慢
- 从库数量太多，主库同步压力大
- 网络抖动
- 复制线程并行能力不足

所以“有主从延迟”不是异常，而是默认要面对的现实。

## 先别幻想绝对没有延迟，先看业务能不能接受

这里最关键的是业务分层。

不是所有读都必须强一致。

可以先这样分：

| 场景                           | 一致性要求         |
| ------------------------------ | ------------------ |
| 支付后查余额、提交后查订单状态 | 强一些             |
| 推荐列表、浏览数、普通详情页   | 可接受短暂最终一致 |
| 报表、统计、搜索索引           | 更能接受延迟       |

所以读写分离设计不是一句“全量读从库”，而是：

**哪些读必须主库，哪些读可以容忍从库延迟。**

## 常见的处理办法，不是只有“全都查主库”

### 1. 核心链路强制读主

对极少数必须马上看到最新结果的请求，直接路由主库。

典型场景：

- 支付成功后查支付状态
- 下单成功后查订单详情
- 用户刚改完密码就查账户安全信息

这个办法最直接，但不能滥用。

因为如果大量读请求都回退主库，读写分离的价值就没了。

### 2. 业务流程上避免“写后立刻查”

有些链路不是非得写完立刻读。

比如：

- 提交成功后先返回结果页
- 再异步刷新状态
- 或者让用户稍后进入详情页

这本质上是在用业务交互吸收一部分复制延迟。

### 3. 对读一致性要求高的场景做会话级策略

常见做法是：

- 某次写之后一段时间内，同一用户相关读都走主库
- 或者记录最近写时间，在短窗口内强制主读

这能降低“自己刚写完自己看不到”的概率。

### 4. 接受最终一致，并把界面文案讲明白

对非核心链路，可以直接承认它是最终一致。

比如：

- “数据同步中”
- “请稍后刷新”

这比强行把所有查询都打回主库更现实。

## 一个很重要的误区：半同步复制不等于完全没延迟

很多人会觉得：

> 开了半同步复制，不就一致了吗？

这不对。

半同步的意义是提高数据安全性，让主库在某些阶段等待至少一个从库确认收到日志。

但它不等于：

- 所有从库都已经追平
- 查询一定不会读到旧数据

所以半同步能降低部分风险，但不能把所有读一致性问题一把抹掉。

## 读写分离题更像“性能和一致性的交换”

它的本质不是纯技术动作，而是取舍：

| 你想要的     | 你可能付出的代价     |
| ------------ | -------------------- |
| 更多读吞吐   | 主从延迟、一致性变弱 |
| 更强读一致性 | 更多主库压力         |
| 更少主库压力 | 业务更要接受最终一致 |

所以回答这题时，别只讲“怎么实现”，最好补一句：

**读写分离本质是在读性能和读一致性之间做分层取舍。**

## 一条更稳的回答顺序

如果被问“读写分离会带来什么问题”，可以直接这样答：

1. 读写分离通过主写从读提升吞吐，但一定会引入主从复制延迟。
2. 最典型的问题是写后读不一致、连续读不单调、不同链路看到不同数据版本。
3. 处理时不能简单粗暴地所有查询都回主库，而要按业务一致性要求分层。
4. 对核心链路强制读主，对非核心链路接受最终一致，必要时配合会话级短时主读。
5. 半同步复制只能降低部分风险，不能等同于完全没有延迟。

## 小结

- 读写分离最核心的代价不是实现复杂，而是主从延迟带来的读一致性问题。
- 最常见的问题是写后读旧数据、连续读不单调，以及不同链路看到不同版本。
- 解决关键不在于“全部读主库”，而在于按业务场景分层处理一致性要求。
- 核心链路可以强制读主，非核心链路更适合接受最终一致。
- 读写分离本质上是在读吞吐和读一致性之间做取舍，不是白拿性能。
