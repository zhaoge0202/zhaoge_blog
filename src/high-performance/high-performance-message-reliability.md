# MQ 如何保证消息不丢？

> “开启持久化”不是消息可靠性的完整答案。消息可能在生产者、Broker、消费者三段里分别丢。

“MQ 如何保证消息不丢”是高频经典题，但最容易答成一句：

> 持久化、ack、重试。

这些词没错，但还不够系统。
更稳的讲法应该先分三段：

1. 生产者发的时候会不会丢
2. Broker 存的时候会不会丢
3. 消费者处理的时候会不会丢

只有这样拆，排查和设计才不会漏。

## 第一段：生产者到 Broker，消息怎么不丢

这个阶段最常见的问题是：

- 业务本地事务已经成功
- 但消息还没真正发到 MQ

比如订单已经写库成功，但“订单创建事件”没发出去，后面库存、积分、通知都没收到。

### 生产者侧常见手段

#### 1. 发送确认

生产者不能只“调用了 send”，要知道 Broker 到底有没有收下。

不同 MQ 叫法不一样，但核心都类似：

- 发送成功才算真正交付给 Broker
- 发送失败要有明确异常或回调

#### 2. 失败重试

如果是临时网络抖动或 Broker 短暂不可达，生产者可以做有限重试。

但注意：

**重试解决的是“没送到”，不是“送到后业务就一定完成”。**

#### 3. 本地消息表 / Outbox

这是更稳的一层。

思路是：

1. 先把业务数据和待发送消息一起写到本地库
2. 由后台任务把消息表中的记录投递到 MQ
3. 发送成功后再标记完成

它的价值在于：

**避免“本地事务成功，但消息没发出去”这个最典型的不一致问题。**

#### 4. 事务消息

像 RocketMQ 这类 MQ 提供了事务消息能力，本质上也是想解决：

本地业务和消息发送如何协调成功 / 失败。

但它不是银弹，因为事务消息本身也会带来：

- 回查逻辑
- 状态机设计
- 额外运维成本

## 第二段：Broker 收到之后，怎么避免存储阶段丢失

消息到 Broker 了，也不代表万事大吉。

还要继续问：

- 是只进内存，还是已经落盘
- 是单副本，还是多副本
- 宕机切换时有没有数据窗口

### 常见手段

#### 1. 持久化

Broker 至少要把消息写到可靠存储，而不是只放在内存里。

但只说“持久化”还不够，还要继续看：

- 同步刷盘还是异步刷盘
- 刷盘点位是什么
- 落盘前崩溃会怎样

这本质上是在性能和数据安全之间做取舍。

#### 2. 多副本

单副本 Broker 宕了，磁盘坏了，消息还是可能丢。

所以核心链路一般还会看：

- 副本数
- 副本同步方式
- leader/follower 切换时的数据窗口

副本越可靠，通常写入成本越高。

#### 3. 高可用集群

Broker 不能只是一台机器，尤其是核心 Topic。

高可用集群要解决的是：

- 节点挂了以后还能不能继续收发
- 切换时会不会丢最后一小段数据

所以这里和高可用专题是连着的，不是纯 MQ 内部细节。

## 第三段：消费者处理时，怎么避免“看似消费了，实际没处理成”

这是消息链路里最容易被漏掉的一段。

因为很多人只盯着 Broker，不盯消费端。

### 最常见的错误顺序

1. 先 ack / 提交 offset
2. 再执行业务
3. 业务失败

这时消息对 MQ 来说已经“消费成功”，但业务实际没成功，这类丢失最隐蔽。

### 更稳的原则

**业务处理成功后，再确认消费成功。**

也就是：

- 先执行业务
- 成功后再 ack / commit

如果处理失败，则：

- 重试
- 回队列
- 进死信
- 进补偿流程

## 真正完整的可靠性思路，是“确认 + 持久化 + 重试 + 补偿”

如果压缩成一句高频面试答案，可以这样说：

1. 生产者侧要有发送确认，必要时用本地消息表或事务消息兜住发送前后不一致。
2. Broker 侧要有持久化、多副本和高可用机制，防节点故障和刷盘窗口丢失。
3. 消费者侧要在业务成功后再确认消费，失败时走重试、死信和补偿。
4. 最后还要靠对账任务、告警和人工介入兜住极端异常。

这四层一起，才更接近“可靠性设计”。

## 可靠性和性能本来就是交换关系

很多人答这题会有一个误区：

> 只要把所有可靠性配置都拉满，不就最稳了吗？

现实里不是这样。

比如：

- 同步刷盘更稳，但写入更慢
- 等待更多副本确认更稳，但吞吐会下降
- 重试更多次更稳，但链路更长、重复风险更高

所以消息可靠性从来不是孤立目标，而是：

**和吞吐、延迟、成本一起权衡。**

## 容易踩的坑

### 只说“开启持久化”

如果不讲生产者确认和消费者确认，答案是不完整的。

### 把“消息进 Broker”当成“业务完成”

消息只是进入队列，不代表消费者已经正确执行业务。

### 没有补偿和对账

再可靠的链路也会有极端异常，没有补偿的话，理论可靠性很难变成业务可靠性。

## 小结

- 消息不丢要分三段看：生产者发送、Broker 存储、消费者处理。
- 生产者侧重点是发送确认、本地消息表或事务消息。
- Broker 侧重点是持久化、多副本和高可用切换边界。
- 消费者侧重点是“业务成功后再确认”，以及失败后的重试、死信和补偿。
- 消息可靠性本质上是在可靠性、吞吐和延迟之间做取舍，不是简单把所有开关拉满。
