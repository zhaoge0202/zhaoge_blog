# 面试里怎么把源码阅读讲成项目能力？

> 源码阅读不能只说“我看过某框架源码”，要讲清你为了解决什么问题、看到了什么证据、最后怎么影响项目决策。

## 好表达不是“我通读过源码”

“我通读过 Spring、MyBatis、Netty 源码”这类说法听起来很大，但可信度不高。面试官真正关心的是：你能不能从项目问题出发，找到框架关键路径，用源码解释现象，再把结论转成配置、代码、排障或选型上的改进。

更可靠的结构是：

```text
项目问题
  -> 现象证据
  -> 为什么需要读源码
  -> 源码关键路径
  -> 我验证了什么
  -> 我怎么改
  -> 结果和边界
```

可以直接套这个句式：

```text
我们项目当时遇到 xxx 问题，表现是 xxx。
单看文档或经验只能知道 xxx，但无法解释 xxx。
所以我顺着 xxx 入口读到 xxx 关键路径，确认 xxx 行为。
最后我把 xxx 改成 xxx，效果是 xxx。
这个结论的边界是 xxx，不能泛化到 xxx 场景。
```

这套表达的重点不是“我看过多少类”，而是“我用源码解决过什么判断问题”。

## 源码证据链怎么组织？

源码阅读不能只报类名，要讲清证据链。建议按五个维度整理：

| 维度     | 要回答的问题                   | 表达示例                                                        |
| -------- | ------------------------------ | --------------------------------------------------------------- |
| 入口方法 | 业务请求从哪里进入框架         | `MapperProxy#invoke`、`refresh()`、`ThreadPoolExecutor#execute` |
| 关键对象 | 谁保存状态，谁做调度，谁做扩展 | `MappedStatement`、`Executor`、`EventLoop`                      |
| 核心分支 | 什么条件下走 A，什么条件下走 B | 队列满才扩非核心线程、条件装配命中才注册 Bean                   |
| 失败路径 | 异常、拒绝、超时、回滚怎么处理 | 拒绝策略、事务回滚、连接超时                                    |
| 验证方式 | 你怎么确认不是猜的             | 单测、断点、日志、压测、线上指标                                |

只说“我看了 `Executor`”是不够的。更好的说法是：“我从 Mapper 调用入口进入，确认 `MappedStatement` 如何定位 SQL，再顺着 `Executor` 看缓存和 Statement 创建，最后通过 SQL 日志验证分页插件改写是否生效。”

## 三个项目化例子

**线程池拒绝问题**

不要只背核心线程、最大线程、队列和拒绝策略。项目表达可以这样讲：

```text
异步任务高峰期偶尔出现拒绝。
一开始只看线程池参数，容易误以为最大线程数会优先扩容。
后来顺着 ThreadPoolExecutor#execute 看执行顺序，
确认任务会先填核心线程，再进队列，队列满了才创建非核心线程。
所以我没有盲目加最大线程数，而是重新评估队列容量、拒绝策略和任务隔离。
```

落地动作可以是：核心链路和非核心任务拆线程池、调整队列容量、补拒绝次数监控、按业务场景选择降级或失败策略。

**MyBatis 分页和慢 SQL**

可以这样讲：

```text
列表接口在大页码时变慢。
我顺着 MapperProxy、MapperMethod、MappedStatement 找到最终 SQL，
再沿 Executor、StatementHandler 看分页插件改写点。
最后确认原 SQL 排序字段没有合适索引，分页插件只是加了 limit，不能解决扫描成本。
```

落地动作可以是：改索引、限制深分页、改成游标翻页、补 SQL 日志和慢 SQL 告警。

**Spring 事务和启动慢**

可以这样讲：

```text
某个事务方法偶尔不回滚，应用启动也偏慢。
我先从 refresh() 看启动主线，定位慢点在非懒加载单例初始化；
再看 AOP 代理和事务拦截器，发现事务方法被同类自调用绕过代理。
最后把耗时初始化移出 Bean 创建链路，并把事务方法拆到独立服务中。
```

落地动作可以是：优化初始化逻辑、调整 Bean 依赖、拆分服务边界、补启动耗时日志和事务用例。

## 怎么把结果说得可信？

源码阅读最后一定要落到“我解决了什么问题”。可以从四类结果里选：

- 性能结果：响应时间、排队时间、启动耗时、吞吐量变化。
- 稳定性结果：拒绝次数、超时次数、错误率、重试量变化。
- 排障结果：日志、指标、线程栈、SQL 追踪是否更容易定位。
- 设计结果：配置边界、调用方式、线程池隔离、事务边界是否更清晰。

没有量化数据时，也要讲清验证方式。例如“压测下拒绝次数下降”“线程栈不再卡在 IO 线程的业务方法”“SQL 日志能看到分页改写后的语句”。比起一句“优化了性能”，这些证据更可信。

## 避免什么？

- 不要说“通读过某框架源码”，范围太大，也很难追问下去。
- 不要背类名清单，却讲不出入口、调用关系、核心分支和失败路径。
- 不要把看过的结论说成自己验证过的源码行为。
- 不要只讲源码细节，不讲项目问题、改动和收益。
- 不要把源码行为绝对化，要说明版本、配置和业务场景边界。
- 不要把开源项目当简历装饰，必须讲出自己跑通、改造、验证过什么。

## 小结

1. 源码阅读要问题驱动，不要为了读而读。
2. 好表达要包含项目问题、源码证据、验证方式、落地改进和边界。
3. 只讲类名和流程图不够，要讲入口、状态、分支、失败路径和结果。
4. 最有价值的是把源码行为转成排障、调优、选型或设计能力。
5. 面试里宁可讲小而完整的问题，也不要泛泛说通读过大框架。
