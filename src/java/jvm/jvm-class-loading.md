# 类加载过程和双亲委派模型怎么理解？

> `.class` 文件不能直接运行，JVM 要先把字节流变成可用的类元数据，再通过类加载器层级决定“由谁加载”。

## 类从哪里开始？

Java 源码编译后得到 Class 文件。Class 文件里有魔数、版本号、常量池、字段表、方法表和属性表。

其中常量池很关键：它保存类、字段、方法的符号引用。后续解析阶段会把这些符号引用替换成 JVM 可以直接使用的引用。

可以用命令观察：

```bash
javap -v Demo.class
```

这能看到版本号、常量池、方法字节码等信息。排查“编译版本过高导致低版本 JDK 跑不起来”时，版本号就很有用。

## 类加载过程到底是哪几步？

完整生命周期可以记成：

```text
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
```

其中验证、准备、解析合称连接。

```mermaid
flowchart LR
  A[加载] --> B[验证]
  B --> C[准备]
  C --> D[解析]
  D --> E[初始化]
  E --> F[使用]
  F --> G[卸载]
```

常说的“类加载过程”通常重点讲前 5 步。

## 加载：把字节流拿进 JVM

加载阶段做三件事：

1. 通过全类名获取二进制字节流。
2. 把字节流代表的静态结构转成方法区里的运行时数据结构。
3. 在堆里生成一个 `java.lang.Class` 对象，作为访问入口。

字节流不一定来自磁盘 Class 文件，也可以来自 Jar、网络、动态代理生成、JSP 编译结果、加密文件解密结果等。

## 连接：验证、准备、解析

验证是安全检查，确保字节流符合 JVM 规范，不会破坏虚拟机安全。

准备是给类变量分配内存并设置默认值：

```java
static int count = 10;
```

准备阶段 `count` 通常是 0，真正赋值为 10 是初始化阶段。

但编译期常量不同：

```java
static final int LIMIT = 100;
```

这类常量可能在准备阶段就通过 ConstantValue 属性拿到 100。

解析是把常量池里的符号引用替换为直接引用。例如把“某个类的某个方法”这种符号描述，变成能直接定位目标方法的引用。

## 初始化：执行 `<clinit>`

初始化阶段才真正执行类里的静态变量赋值和静态代码块。编译器会把它们合并成 `<clinit>` 方法。

但加载完成不代表一定初始化。只有主动使用才会触发，比如：

- `new` 一个对象。
- 读写非编译期常量的静态字段。
- 调用静态方法。
- 反射调用。
- JVM 启动主类。
- 某些 MethodHandle/VarHandle 场景。
- 初始化实现类时，接口里有 default 方法也可能触发接口初始化。

排查类初始化阻塞时，可以看线程栈：

```bash
jstack <pid>
```

如果多个线程卡在同一个类的 `<clinit>`，往往是静态初始化里做了慢操作、锁等待或远程调用。

## 双亲委派模型解决什么问题？

类加载器大致分层：

| JDK 8 口径            | 作用                                   |
| --------------------- | -------------------------------------- |
| Bootstrap ClassLoader | 加载核心类库，Java 里通常显示为 `null` |
| Extension ClassLoader | 加载扩展目录                           |
| AppClassLoader        | 加载应用 classpath                     |

JDK 9+ 模块化后，Extension 口径变成 Platform ClassLoader，`rt.jar` 这类说法也要带版本边界。

双亲委派流程：

```text
先查自己是否加载过
  ↓
没加载就委派父加载器
  ↓
父加载器找不到
  ↓
自己再尝试 findClass
```

好处有两个：

1. 避免核心类重复加载。
2. 防止应用随便伪造 `java.lang.Object` 这类核心 API。

注意，“双亲”不是两个父加载器，它表达的是父级委派关系。

## 小结

- 类加载过程通常指加载、验证、准备、解析、初始化。
- 准备阶段给静态变量默认值，初始化阶段才执行业务赋值和静态代码块。
- 初始化不是加载后必然发生，只有主动使用才触发。
- 双亲委派先让父加载器查找，父加载器找不到才由子加载器加载。
- JVM 判断类身份不只看类名，还看定义它的 ClassLoader。
