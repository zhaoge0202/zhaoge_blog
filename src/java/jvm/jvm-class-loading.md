# 类加载过程和双亲委派模型怎么理解？

> `.class` 文件不能直接运行，JVM 要先把字节流变成可用的类元数据，再通过类加载器层级决定“由谁加载”。

## 类从哪里开始？

Java 源码编译后得到 Class 文件。Class 文件里有魔数、版本号、常量池、字段表、方法表和属性表。

其中常量池很关键：它保存类、字段、方法的符号引用。后续解析阶段会把这些符号引用替换成 JVM 可以直接使用的引用。

可以用命令观察：

```bash
javap -v Demo.class
```

这能看到版本号、常量池、方法字节码等信息。排查“编译版本过高导致低版本 JDK 跑不起来”时，版本号就很有用。

Class 文件可以先抓几个关键字段：

| 结构          | 作用                                  | 排查价值                       |
| ------------- | ------------------------------------- | ------------------------------ |
| 魔数          | 判断是不是 JVM 可识别的 Class 文件    | 非法字节码会被拒绝加载         |
| 主/次版本号   | 标记编译目标版本                      | 定位高版本编译、低版本运行问题 |
| 常量池        | 保存字面量和类/字段/方法符号引用      | 理解解析阶段                   |
| 字段表/方法表 | 描述字段、方法签名和访问标志          | 理解反射、字节码增强           |
| 属性表        | 保存 Code、LineNumberTable 等附加信息 | 调试、异常栈行号、泛型信息     |

比如低版本 JDK 运行高版本字节码时，常见不是业务逻辑错，而是 Class 文件版本不被当前 JVM 支持。

## 类加载过程到底是哪几步？

完整生命周期可以记成：

```text
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
```

其中验证、准备、解析合称连接。

```mermaid
flowchart LR
  A[加载] --> B[验证]
  B --> C[准备]
  C --> D[解析]
  D --> E[初始化]
  E --> F[使用]
  F --> G[卸载]
```

常说的“类加载过程”通常重点讲前 5 步。

## 加载：把字节流拿进 JVM

加载阶段做三件事：

1. 通过全类名获取二进制字节流。
2. 把字节流代表的静态结构转成方法区里的运行时数据结构。
3. 在堆里生成一个 `java.lang.Class` 对象，作为访问入口。

字节流不一定来自磁盘 Class 文件，也可以来自 Jar、网络、动态代理生成、JSP 编译结果、加密文件解密结果等。

数组类有个特殊点：数组类不是由普通 ClassLoader 直接读取二进制字节流创建，而是 JVM 在需要时自动创建。数组类的加载器通常和元素类型一致；如果元素是基本类型，则数组类也没有普通意义上的类加载器。

## 连接：验证、准备、解析

验证是安全检查，确保字节流符合 JVM 规范，不会破坏虚拟机安全。

验证阶段会检查文件格式、元数据、字节码语义和符号引用等。它有成本，但安全收益更重要。历史上可以通过参数关闭大部分验证来缩短加载时间，不过这个方向已经逐渐不推荐，生产上不要为了微小启动收益轻易牺牲字节码校验。

准备是给类变量分配内存并设置默认值：

```java
static int count = 10;
```

准备阶段 `count` 通常是 0，真正赋值为 10 是初始化阶段。

但编译期常量不同：

```java
static final int LIMIT = 100;
```

这类常量可能在准备阶段就通过 ConstantValue 属性拿到 100。

但不是所有 `static final` 都是编译期常量：

```java
static final int RUNTIME = new Random().nextInt();
```

这种值要等初始化阶段执行代码后才确定，准备阶段仍然只能拿到默认零值。

解析是把常量池里的符号引用替换为直接引用。例如把“某个类的某个方法”这种符号描述，变成能直接定位目标方法的引用。

## 初始化：执行 `<clinit>`

初始化阶段才真正执行类里的静态变量赋值和静态代码块。编译器会把它们合并成 `<clinit>` 方法。

但加载完成不代表一定初始化。只有主动使用才会触发，比如：

- `new` 一个对象。
- 读写非编译期常量的静态字段。
- 调用静态方法。
- 反射调用。
- JVM 启动主类。
- 某些 MethodHandle/VarHandle 场景。
- 初始化实现类时，接口里有 default 方法也可能触发接口初始化。

也有一些“看起来用到了类，但不触发初始化”的被动引用：

```java
System.out.println(Child.parentValue); // 字段定义在 Parent，通常只初始化 Parent
Parent[] array = new Parent[10];       // 创建数组不会初始化 Parent
System.out.println(ConstHolder.LIMIT); // 编译期常量可能已进调用方常量池
```

这类题的重点不是背例子，而是分清“加载/解析到这个类型”和“真正执行这个类型的 `<clinit>`”不是同一件事。

排查类初始化阻塞时，可以看线程栈：

```bash
jstack <pid>
```

如果多个线程卡在同一个类的 `<clinit>`，往往是静态初始化里做了慢操作、锁等待或远程调用。

`<clinit>` 由 JVM 保证线程安全，同一时刻只有一个线程执行某个类的初始化，其他线程会等待。这保证了静态初始化的一致性，但也意味着不要在静态代码块里做慢 SQL、远程调用、复杂锁等待，否则可能把一批业务线程一起卡住。

## 双亲委派模型解决什么问题？

类加载器大致分层：

| JDK 8 口径            | 作用                                   |
| --------------------- | -------------------------------------- |
| Bootstrap ClassLoader | 加载核心类库，Java 里通常显示为 `null` |
| Extension ClassLoader | 加载扩展目录                           |
| AppClassLoader        | 加载应用 classpath                     |

JDK 9+ 模块化后，Extension 口径变成 Platform ClassLoader，`rt.jar` 这类说法也要带版本边界。

双亲委派流程：

```text
先查自己是否加载过
  ↓
没加载就委派父加载器
  ↓
父加载器找不到
  ↓
自己再尝试 findClass
```

好处有两个：

1. 避免核心类重复加载。
2. 防止应用随便伪造 `java.lang.Object` 这类核心 API。

注意，“双亲”不是两个父加载器，它表达的是父级委派关系。

JDK 8 和 JDK 9+ 的加载器口径也要分清：

| 版本口径 | 典型层级                    | 说明                       |
| -------- | --------------------------- | -------------------------- |
| JDK 8    | Bootstrap → Extension → App | 常见于老资料和存量系统     |
| JDK 9+   | Bootstrap → Platform → App  | 模块化后扩展加载器口径变化 |

所以不要在 JDK 17 项目里继续机械回答 `ExtClassLoader` 和 `rt.jar`。它们更适合描述 JDK 8 及以前的类路径模型。

## 类什么时候会卸载？

类卸载不是“对象没了类就没了”。通常要同时满足：

1. 这个类的实例都已经不可达。
2. 这个类对应的 `Class` 对象不可达。
3. 定义这个类的 ClassLoader 也不可达。

这就是为什么自定义加载器场景更容易讨论类卸载：插件、热部署、脚本引擎用新的加载器加载类，旧加载器如果断开引用，旧类才有机会卸载。

观察加载和卸载：

```bash
# JDK 9+
java -Xlog:class+load=info,class+unload=info -jar app.jar

# 运行中看加载器统计
jcmd <pid> VM.classloader_stats
```

如果类数量和 ClassLoader 数量持续增长，就要怀疑动态代理、脚本、热部署或插件卸载不干净。

## 容易踩的坑

1. 类“加载了”不等于“初始化了”，主动使用才会触发 `<clinit>`。
2. 准备阶段通常是默认零值，业务赋值在初始化阶段；编译期常量是例外。
3. `<clinit>` 是线程安全的，但静态初始化里做慢操作会阻塞其他线程。
4. 双亲委派不是继承关系，加载器父子通常是组合关系。
5. JDK 9+ 的 Platform ClassLoader 让 JDK 8 的 Extension 口径不再完全适用。

## 小结

- 类加载过程通常指加载、验证、准备、解析、初始化。
- 准备阶段给静态变量默认值，初始化阶段才执行业务赋值和静态代码块。
- 初始化不是加载后必然发生，只有主动使用才触发。
- 双亲委派先让父加载器查找，父加载器找不到才由子加载器加载。
- JVM 判断类身份不只看类名，还看定义它的 ClassLoader。
