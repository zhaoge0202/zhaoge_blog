# 常见垃圾收集器怎么选？

> GC 选择不是背“谁更先进”，而是在吞吐、延迟、内存占用、CPU 成本和 JDK 版本之间做取舍。

## 先看目标：吞吐还是低延迟？

垃圾收集器大致在几个目标之间取舍：

| 目标     | 含义                      | 典型关注             |
| -------- | ------------------------- | -------------------- |
| 吞吐量   | 用户代码运行时间占比高    | 批处理、离线任务     |
| 低延迟   | 单次停顿尽量短            | 交易、网关、实时接口 |
| 内存占用 | 额外元数据和复制空间少    | 小规格机器           |
| CPU 成本 | GC 并发线程不要抢太多 CPU | CPU 紧张服务         |

所以不存在“最强 GC”。只有“这个服务在这个 JDK、这个堆大小、这个延迟目标下更合适”。

GC 选型还建立在分代假说上：

1. 大多数对象很快变成垃圾。
2. 少数对象会存活很久。

所以年轻代通常用复制思路，快速清掉大量短命对象；老年代存活对象多，更关注碎片、整理成本和长停顿。线上看到老年代持续增长时，不能只说“GC 参数小了”，还要问对象为什么从年轻代活到了老年代。

## 先把 GC 名词边界说清

讨论收集器前，先统一几个日志里经常出现的名词：

| 名词                | 常见含义                           | 注意点                                   |
| ------------------- | ---------------------------------- | ---------------------------------------- |
| Minor GC / Young GC | 主要回收新生代                     | 频繁不一定坏，要结合耗时和晋升量看       |
| Major GC / Old GC   | 主要回收老年代                     | 不同资料和日志里口径不完全一致           |
| Mixed GC            | 回收整个年轻代和部分老年代         | G1 里很常见，不等同于 Full GC            |
| Full GC             | 回收整个 Java 堆，通常还涉及元数据 | 停顿重，排查时要看触发原因和前后占用变化 |

所以不要简单说“Major GC 就是 Full GC”。不同 JDK、收集器和日志格式里，Major/Old/Full 的表达可能不同，排障时要回到 GC 日志的触发原因和回收范围。

对象晋升也不要死背“默认 15 次 Minor GC”。年龄阈值会受 `MaxTenuringThreshold`、Survivor 空间占用、动态年龄判定和具体收集器默认值影响。排查晋升过快时，要看对象年龄分布和 Survivor 是否放得下，而不是只改一个年龄参数。

## 常见收集器怎么定位？

| 收集器     | 典型定位                   | 版本边界                          |
| ---------- | -------------------------- | --------------------------------- |
| Serial     | 单线程、小堆、工具类应用   | 现代服务端少用                    |
| Parallel   | 吞吐优先，批处理友好       | JDK 8 常见默认                    |
| CMS        | 历史上的低停顿方案         | JDK 9 废弃，JDK 14 移除           |
| G1         | 通用服务端，兼顾停顿和吞吐 | JDK 9+ 默认                       |
| ZGC        | 大堆、低延迟               | JDK 15 生产可用，JDK 21+ 分代演进 |
| Shenandoah | 低延迟，和应用并发整理     | JDK 15 生产可用                   |

如果是新项目，不要再推荐 CMS。它适合当历史系统知识和排障背景，而不是现代 JDK 的新选择。

## 按服务画像怎么选？

可以先用一张简化表做初筛：

| 服务画像                 | 优先考虑         | 说明                           |
| ------------------------ | ---------------- | ------------------------------ |
| JDK 8 离线任务、批处理   | Parallel         | 吞吐优先，能接受较长停顿       |
| JDK 8 老系统、低停顿诉求 | CMS              | 只作为存量系统理解，不建议新增 |
| JDK 11/17 常规后端服务   | G1               | 默认通用选择，先用日志验证     |
| 大堆、低延迟接口         | ZGC / Shenandoah | 需要较新 JDK 和压测验证        |
| 小工具、小堆、单核环境   | Serial           | 简单，线程交互成本低           |

真实项目里，初筛之后还要压测三件事：P99/P999 延迟、吞吐损失、CPU 占用。只看平均停顿或只看一次 GC 日志，都不足以证明选型正确。

## Parallel GC 适合什么？

Parallel 关注吞吐量。比如离线报表、批量计算、后台任务，停顿几百毫秒不敏感，但希望总体跑得快。

JDK 8 常见默认是 Parallel Scavenge + Parallel Old。可以用：

```bash
java -XX:+PrintCommandLineFlags -version
```

确认当前 JVM 实际启用的收集器。

## G1 为什么成为通用选择？

G1 把堆切成很多 Region，不再只按传统连续新生代/老年代大块管理。它会估算每个 Region 的回收收益和成本，在停顿目标内优先回收收益高的 Region。

它适合多数中大型 Java 服务：

- 堆比较大。
- 希望停顿可控。
- 不想维护 CMS 的碎片和并发失败问题。
- 使用 JDK 9+ 默认配置。

但 G1 也不是永远更快。停顿目标设得过低、堆太小、对象分配过猛，都可能让它付出更高 CPU 成本。

G1 调优时最容易误用的是：

```bash
-XX:MaxGCPauseMillis=50
```

这个值不是越低越好。目标设得过低，G1 会倾向于每次少回收一点，导致 GC 更频繁、后台并发线程更忙，最终吞吐下降。通常先保守使用默认或接近业务 SLA 的目标，再用压测和日志逐步收敛。

## ZGC 和 Shenandoah 适合什么？

ZGC 和 Shenandoah 都属于低延迟方向，目标是把大部分 GC 工作与应用线程并发执行，减少 STW 时间。

适合：

- 大堆。
- 延迟敏感接口。
- 不能接受长时间 Full GC 的服务。

代价是：

- 可能牺牲一部分吞吐。
- 需要较新 JDK。
- 观测指标和调优方式与传统分代 GC 不完全一样。

版本边界要说清：

- ZGC：JDK 11 实验，JDK 15 生产可用。
- 分代 ZGC：JDK 21 引入，后续版本持续演进。
- Shenandoah：JDK 12 实验，JDK 15 生产可用。

## 怎么用数据判断选型？

先开启 GC 日志，再谈调优。

JDK 8：

```bash
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/var/log/app/gc-%t.log
-XX:+UseGCLogFileRotation
-XX:NumberOfGCLogFiles=14
-XX:GCLogFileSize=50M
```

JDK 11+：

```bash
-Xlog:gc*,safepoint:file=/var/log/app/gc-%t.log:time,uptime,level,tags:filecount=14,filesize=50M
```

运行时先看：

```bash
jstat -gcutil <pid> 1000 10
```

重点观察 Young GC、Full GC 次数和耗时，老年代是否持续上升，GC 后是否能明显回落。

还要把 GC 指标和业务指标对齐：

```text
GC 日志时间点
├── 是否对应接口 P99 抖动？
├── 是否对应 CPU 飙高？
├── 是否对应分配速率突增？
└── GC 后老年代/堆占用是否回落？
```

如果 GC 很频繁但停顿很短、业务延迟稳定，未必需要立刻换收集器；如果 GC 次数不多但单次停顿打穿 SLA，就要优先关注停顿来源。

## 容易踩的坑

1. 不要把“JDK 版本默认 GC”当成所有服务的最优解，默认只是起点。
2. 不要把 Major GC、Old GC、Full GC 混为一个概念，要看日志语义。
3. CMS 已经退出主线，新项目不要拿它当推荐方案。
4. 低延迟 GC 通常会付出吞吐、CPU 或版本升级成本。
5. GC 选型必须配合压测和 GC 日志，不能只凭收集器名字判断。

## 小结

- GC 选择本质是在吞吐、延迟、CPU、内存和版本之间做取舍。
- JDK 8 常见默认是 Parallel，JDK 9+ 默认是 G1。
- CMS 已废弃并移除，新项目不要推荐 CMS。
- G1 是通用服务端常见选择，但不是所有场景都更快。
- ZGC 和 Shenandoah 面向低延迟、大堆场景，要结合 JDK 版本、压测和观测数据选择。
