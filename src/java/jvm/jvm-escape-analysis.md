# 逃逸分析、标量替换、锁消除到底优化了什么？

> 逃逸分析不是一个单独的“提速开关”，而是 JIT 判断对象能不能被外部看见的分析过程。分析结果可以触发标量替换、分配消除、锁消除等优化，真正减少对象分配、GC 压力和无意义的同步开销。

## 先看一个临时对象

```java
record Point(int x, int y) {}

public int distance(int x, int y) {
    Point point = new Point(x, y);
    return point.x() * point.x() + point.y() * point.y();
}
```

从 Java 语义看，`new Point(x, y)` 创建了一个对象。可是站在 JIT 的角度看，这个对象只在方法内部使用：

- 没有返回出去。
- 没有写到成员变量、静态变量、数组或集合里。
- 没有传给 JIT 看不透的方法。
- 没有被其他线程拿到。

如果 `distance()` 变成热点方法，并且相关调用都被 JIT 看清楚，JIT 可能不再真的分配 `Point` 对象，而是把它拆成 `x`、`y` 两个普通值继续优化。

这就是逃逸分析背后的直觉：**对象如果不会逃出当前优化范围，JIT 就有机会把“对象”这层抽象拿掉。**

## 逃逸分析到底判断什么？

逃逸分析（Escape Analysis）回答的是：某个对象引用会不会被当前方法、当前线程以外的代码看见。

可以粗略分成三类：

| 逃逸级别 | 例子                                       | 优化空间                         |
| -------- | ------------------------------------------ | -------------------------------- |
| 不逃逸   | 纯方法内部临时对象，只读字段后就结束       | 可能标量替换、分配消除、锁消除   |
| 参数逃逸 | 传给某个可分析的方法，但没有全局逃逸       | 取决于方法是否内联、JIT 是否看透 |
| 全局逃逸 | 返回对象、写入字段、放入集合、交给其他线程 | 通常必须按对象语义保守处理       |

几个典型逃逸写法：

```java
private Object field;

public Object returnEscape() {
    return new Object(); // 返回给调用方，全局逃逸
}

public void fieldEscape() {
    field = new Object(); // 写入对象字段，全局逃逸
}

public void threadEscape() {
    Object lock = new Object();
    new Thread(() -> System.out.println(lock)).start(); // 被其他线程捕获
}
```

逃逸分析不是源码层面的简单规则。JIT 会结合方法内联后的图来判断：一个对象虽然传给了另一个方法，但如果那个方法被内联进来，JIT 仍然可能看出它没有逃逸；反过来，如果方法边界太复杂、没法内联，就可能保守地认为对象逃逸。

## 标量替换：对象被拆成普通变量

标量替换（Scalar Replacement）是逃逸分析后最关键的优化之一。

“标量”可以理解成不能再拆的小值，比如 `int`、`long`、对象引用；“聚合量”则是由多个字段组成的对象。标量替换就是把一个对象的字段拆成一组局部值。

```java
class Money {
    final long cents;
    final String currency;

    Money(long cents, String currency) {
        this.cents = cents;
        this.currency = currency;
    }
}

public long centsOf(long value) {
    Money money = new Money(value, "CNY");
    return money.cents;
}
```

优化前可以理解为：

```text
分配 Money 对象 -> 写 cents/currency 字段 -> 读取 cents -> 返回
```

优化后可能变成：

```text
long cents = value -> 返回 cents
```

中间的对象分配、字段写入、字段读取都可能被消掉。这样带来的收益不只是少一次 `new`：

- 年轻代分配减少。
- Minor GC 压力降低。
- 字段访问可能变成寄存器或栈上的局部值访问。
- 后续常量折叠、死代码消除更容易发生。

## “栈上分配”这句话为什么要谨慎？

很多资料会把逃逸分析的结果说成“对象没有逃逸，所以可以栈上分配”。这句话用于粗略理解可以，但面试和写文章时最好补上边界。

更准确的说法是：

1. Java 语义上，对象通常按“在堆上创建、由 GC 管理”理解。
2. HotSpot JIT 通过逃逸分析发现对象不逃逸时，常见优化是**标量替换和分配消除**。
3. 对程序可观察行为来说，好像这个对象没有进入堆；但这不等于 JVM 一定真的在栈上按对象布局放了一个完整对象。

OpenJDK/Oracle 的资料也强调过这个边界：HotSpot 的逃逸分析能消除可标量替换对象的分配和相关锁，但不要把它简单等同为“所有不逃逸对象都栈上分配”。

面试里建议这样说：

> 逃逸分析可能让不逃逸对象不再真实分配，典型结果是标量替换。很多人口头说“栈上分配”，但 HotSpot 里更常见、更严谨的表达是分配消除，而不是保证对象按完整对象形态放到栈上。

## 锁消除：对象没人共享，锁就没意义

逃逸分析还能帮助 JIT 判断锁有没有竞争可能。

```java
public String concat(String a, String b) {
    StringBuffer sb = new StringBuffer();
    sb.append(a);
    sb.append(b);
    return sb.toString();
}
```

`StringBuffer.append()` 是同步方法，理论上会进入 `synchronized`。但这里 `sb` 是方法内部新建的局部对象，没有发布给其他线程。既然其他线程不可能拿到这把锁，JIT 就可能把这些锁操作消掉。

这就是锁消除（Lock Elimination）：

```text
逃逸分析确认对象不会被其他线程访问
  -> 锁不可能发生真实竞争
  -> 删除 monitorenter / monitorexit 相关开销
```

它和“不要写 synchronized”不是一回事。你仍然要按正确并发语义写代码；JIT 只是在线程安全语义不变的前提下，把确定没必要的锁优化掉。

## 锁粗化：连续加锁不如合并一次

锁粗化（Lock Coarsening）解决的是另一个问题：频繁对同一个对象加锁、解锁，开销可能比临界区本身还大。

```java
for (int i = 0; i < 1000; i++) {
    synchronized (lock) {
        update(i);
    }
}
```

如果 JIT 判断把锁范围扩大后不会改变语义，可能优化成：

```java
synchronized (lock) {
    for (int i = 0; i < 1000; i++) {
        update(i);
    }
}
```

锁粗化不一定依赖对象不逃逸，它更关注“同一把锁连续反复获取释放是否可以合并”。但它和锁消除一样，都属于 JIT 在同步语义上的运行时优化。

## 哪些写法容易让优化失效？

逃逸分析和标量替换不是承诺。下面这些写法都会让 JIT 更难优化：

| 写法                         | 为什么影响优化                         |
| ---------------------------- | -------------------------------------- |
| 返回新对象                   | 调用方能看到对象，通常全局逃逸         |
| 写入成员变量、静态变量、集合 | 对象生命周期超出当前方法               |
| 传给未内联的大方法           | JIT 看不透被调用方法内部会不会保存引用 |
| 捕获到 lambda / 线程任务里   | 对象可能跨线程或跨作用域使用           |
| 复杂分支中部分路径逃逸       | HotSpot C2 可能保守认为整体逃逸        |
| 关闭 JIT 或未达到热点        | 解释执行阶段不会触发这些编译优化       |

所以不要为了“让逃逸分析生效”刻意写奇怪代码。正常写清晰、局部性好的代码就行；是否优化，交给 JIT 和压测结果判断。

## 和 GC、性能排查有什么关系？

逃逸分析能减少分配，但它不是解决 GC 的万能药。

它适合优化的是热点路径上的短命临时对象。如果一个服务分配速率很高，可以先用 JFR 或 async-profiler 的 allocation profiling 看对象从哪里来，再判断：

- 是不是业务确实在构造大量返回对象？
- 是不是集合、JSON、日志、字符串拼接带来的临时对象？
- 是不是某些热点对象理论上可以被 JIT 消掉，但因为方法过大、未内联、反射/代理边界太多而没有优化？

可以用 JMH 做对比实验，比如分别打开/关闭逃逸分析观察分配量：

```bash
# 仅限实验，不建议生产这么关
-XX:-DoEscapeAnalysis
```

如果关闭逃逸分析后分配量明显上升，说明 JIT 原本确实帮你消掉了一些对象。但线上优化不要靠猜开关，应该先有采样证据，再做代码结构或参数调整。

## 面试怎么讲更稳？

可以按这条线回答：

```text
逃逸分析是 JIT 的一种分析能力，用来判断对象引用会不会逃出当前方法或当前线程。
如果对象不逃逸，JIT 可能做标量替换，把对象字段拆成局部变量，从而消除对象分配，减少 GC 压力。
如果锁对象不可能被其他线程访问，JIT 还可能做锁消除；连续对同一对象加锁，也可能做锁粗化。
“栈上分配”是常见但容易过度简化的说法，HotSpot 更常见的结果是标量替换和分配消除，不保证所有未逃逸对象都真的按对象形态放到栈上。
这些优化依赖热点编译、方法内联和运行时信息，不是源码一写就必然发生。
```

## 容易踩的坑

| 误区                             | 更准确的说法                                       |
| -------------------------------- | -------------------------------------------------- |
| 所有对象都在堆上                 | 通常在堆上，但 JIT 可能通过标量替换消除部分分配    |
| 逃逸分析就是栈上分配             | 逃逸分析是分析过程，结果可能是标量替换、分配消除等 |
| 没逃逸就一定优化                 | 还要看是否热点、是否内联、JIT 阈值和编译器判断     |
| 锁消除表示 synchronized 可以乱写 | 锁消除只在语义安全时发生，正确同步仍要靠代码表达   |
| GC 压力大就指望逃逸分析自动解决  | 先用分配火焰图找证据，再优化对象创建和数据结构     |

## 小结

- 逃逸分析判断对象是否会被当前方法或线程以外的代码看见，是 JIT 后续优化的前提。
- 标量替换会把对象字段拆成局部值，进一步消除对象分配、字段访问和部分 GC 压力。
- “栈上分配”不要说绝对，HotSpot 更常见的表述是标量替换和分配消除。
- 锁消除依赖逃逸分析确认锁对象不会被共享；锁粗化则减少连续加锁解锁的开销。
- 这些优化依赖热点编译、方法内联和运行时信息，是否生效要用 JMH、JFR、async-profiler 等证据验证。
