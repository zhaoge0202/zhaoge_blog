# Java 21 虚拟线程对传统线程池有什么影响？

> 虚拟线程降低的是阻塞等待的线程成本，不是让数据库、网络和 CPU 资源无限扩容。

## 传统线程池解决什么？

平台线程是操作系统线程，创建和切换成本高，所以传统服务会用线程池复用线程，并用队列控制并发。

传统线程池有两个作用：

1. 复用昂贵的平台线程，避免每个任务都创建 OS 线程。
2. 通过线程数和队列限制并发，避免请求无限进入系统。

虚拟线程改变的是第一个作用。它由 JDK 管理，数量可以远大于平台线程。阻塞 I/O 时，虚拟线程通常可以从载体线程上卸载，让载体线程继续执行其他虚拟线程。

这会让“一个请求一个线程”的编程模型重新变得可行：

```java
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (Request request : requests) {
        executor.submit(() -> handle(request));
    }
}
```

这里的 executor 不是传统意义上的固定线程池。它的语义是“每个任务创建一个虚拟线程”，而不是复用少量虚拟线程。

## 平台线程、虚拟线程、载体线程是什么关系？

可以把三者这样理解：

```text
业务任务
  ↓
虚拟线程：承载 Java 代码执行语义，数量可以很多
  ↓ 挂载 / 卸载
载体线程：真正执行虚拟线程的平台线程
  ↓
操作系统线程：由 OS 调度
```

虚拟线程不是凭空执行代码。它运行时仍然要挂载到某个平台线程上。优势在于：当虚拟线程在可卸载的阻塞点等待 I/O 时，它可以让出载体线程，避免平台线程跟着一起闲等。

这正是它对阻塞式服务有价值的地方：代码仍然可以按同步写法组织，但等待 I/O 的线程成本大幅下降。

## 虚拟线程适合什么？

适合大量阻塞 I/O 场景：

- HTTP/RPC 调用。
- JDBC 查询。
- 文件或网络等待。
- 调用多个下游服务后汇总结果。
- Spring MVC 这类同步请求处理模型的阻塞等待场景。

不适合指望它解决 CPU 密集任务。CPU 还是那些核心数，计算任务多了仍然要限流。

如果任务主要是在做加密、压缩、图片处理、大量 JSON 序列化、复杂规则计算，虚拟线程不会把 CPU 变多。它能降低等待成本，不能提高 CPU 算力，也不能扩大数据库连接数。

## 线程池会消失吗？

不会。变化的是“池化线程”这件事不再适合虚拟线程。虚拟线程通常按任务创建，用完结束。

但仍然需要：

- 数据库连接池。
- RPC 并发限流。
- 下游隔离。
- 任务队列和背压。
- 超时、熔断和重试预算。

换句话说，线程池的复用价值下降了，资源治理价值还在。

一个常见误区是：以前 Tomcat 线程池 200 个，请求容易被慢接口占满；换虚拟线程后，可以同时挂起更多请求，于是下游数据库或 RPC 也被打出更高并发。

所以并发预算要从“线程数”迁移到“资源数”：

| 资源     | 仍然要限制什么                   |
| -------- | -------------------------------- |
| 数据库   | 连接池大小、慢 SQL、事务持有时间 |
| RPC 下游 | 最大并发、超时、重试、隔离仓     |
| 消息发送 | 生产速率、broker 配额、失败补偿  |
| 本机 CPU | 计算任务并发和队列长度           |

如果需要限制某个下游的并发，不要靠池化虚拟线程，可以用信号量、限流器、隔离组件或连接池本身表达。

```java
Semaphore permits = new Semaphore(100);

try {
    permits.acquire();
    return remoteClient.call(request);
} finally {
    permits.release();
}
```

## ThreadLocal 还能用吗？

能用，但要更谨慎。

传统线程池里线程数量有限，`ThreadLocal` 的主要风险是线程复用导致脏数据和内存泄漏。虚拟线程通常用完就销毁，脏数据残留问题会缓和，但数量可能非常大，新的风险是把大对象、连接、缓存、用户上下文无限塞进每个虚拟线程。

更稳的做法是：

- 只放小而明确的请求上下文；
- 及时清理框架兼容层里的 `ThreadLocal`；
- 不把数据库连接、客户端对象、大集合放进 `ThreadLocal`；
- 关注作用域值等配套机制，但要核对当前版本状态。

## pinning 为什么会影响效果？

虚拟线程阻塞时能否卸载，决定了载体线程是否能被释放。某些场景会发生 pinning，也就是虚拟线程阻塞时仍然“钉”在载体线程上。

常见风险包括：

- 在 `synchronized` 临界区里执行长时间阻塞 I/O；
- 调用本地方法或外部函数时发生阻塞；
- 某些旧库在关键路径上持有监视器锁并阻塞。

短小的 `synchronized` 不是问题，问题是锁里做慢操作：

```java
synchronized (lock) {
    remoteClient.call(request); // 不推荐：锁内长时间阻塞
}
```

迁移虚拟线程时，如果发现载体线程被长时间占住，要检查线程栈、JFR 事件、锁竞争和慢 I/O 位置。必要时把长时间阻塞移出 `synchronized`，或改用更细粒度的锁和并发工具。

## Spring 项目怎么迁移更稳？

虚拟线程不是“打开开关就吞吐翻倍”。更稳的迁移顺序是：

1. 先挑阻塞 I/O 明显、业务逻辑相对简单的接口试点。
2. 压测前后对比吞吐、p95、p99、CPU、内存、线程数。
3. 重点看数据库连接池等待、RPC 下游 QPS、超时和重试量。
4. 检查日志 MDC、链路追踪、事务上下文、权限上下文是否依赖 `ThreadLocal`。
5. 观察 pinning、锁竞争和慢调用，不要只看请求成功率。

如果接口瓶颈在数据库慢查询，虚拟线程只会让更多请求同时等数据库；如果瓶颈在 CPU，虚拟线程也不会提升核心数。它最适合把“大量线程卡在等待 I/O 上”的成本降下来。

## 小结

1. 虚拟线程降低的是阻塞 I/O 等待期间的平台线程占用成本，不是提升 CPU 或下游容量。
2. 平台线程仍然负责真实执行，虚拟线程在可卸载阻塞点让出载体线程，才能发挥效果。
3. 不要池化虚拟线程，通常应该按任务创建；并发治理要转向连接池、限流、隔离和背压。
4. `ThreadLocal`、锁内阻塞、pinning、旧依赖兼容性是虚拟线程迁移的重点风险。
5. Spring 项目迁移要用压测和监控验证 p95/p99、连接池等待、下游 QPS、线程数和锁竞争。
