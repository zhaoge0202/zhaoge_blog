# Java IO 流体系是怎么组织的？字节流和字符流有什么区别？

> Java IO 看着类特别多，其实抓住两条轴线就够了：数据是流入还是流出，处理的是原始字节还是带编码语义的文本字符。

## 先别背类名，先看 Java IO 在解决什么问题

应用程序读文件、写文件、收网络报文、吐网络响应，本质上都在做 I/O。
问题是，底层世界真正流动的最小单位始终是**字节**，但业务代码很多时候关心的是**文本**、**对象**、**缓冲**、**编码**这些更高一层的东西。

Java IO 的设计思路，就是先给你一套最底层的“字节搬运”接口，再在上面一层层叠出：

- 输入还是输出；
- 字节还是字符；
- 要不要缓冲；
- 要不要按基本类型读写；
- 要不要在字节和字符之间做编码转换。

所以 IO 类多，不代表体系乱。恰恰相反，它是一个很典型的“基类 + 包装 + 桥接”的结构。

## 两条主线：输入/输出 + 字节/字符

Java IO 可以先按两个维度拆开。

第一条轴线是**方向**：

- 输入：把外部数据读进程序；
- 输出：把程序里的数据写到外部。

第二条轴线是**处理单位**：

- 字节：面向原始二进制数据；
- 字符：面向文本语义。

于是就有四个最核心的抽象基类：

| 维度 | 输入          | 输出           |
| ---- | ------------- | -------------- |
| 字节 | `InputStream` | `OutputStream` |
| 字符 | `Reader`      | `Writer`       |

很多同学学 IO 时一上来就陷进 `FileInputStream`、`BufferedReader`、`ObjectOutputStream` 这些具体类里，结果越看越乱。更好的顺序是先记住一句话：

**所有 IO 类，基本都可以看成是在这四个抽象基类之下继续细分。**

## 为什么既要字节流，又要字符流

这个问题最容易被答成“字符流更高级，所以读文本用字符流”。这个说法不够准。

先把结论说在前面：

- **字节流处理的是原始数据**，不关心编码；
- **字符流处理的是已经按某种字符集解码后的文本**。

比如磁盘上的 `hello.txt`，底层存的并不是“字符”，而是一串字节。
如果文件用 UTF-8 编码，那么中文“你”通常会占 3 个字节；如果是 UTF-16，字节布局又不一样。

这意味着：

- 当你操作图片、音频、压缩包、网络协议报文时，根本不该谈“字符”，因为它们本来就不是文本；
- 当你操作文本时，真正麻烦的不是“读几个字节”，而是“这些字节该按什么编码解释”。

这就是字符流存在的意义：**把“按某种字符集把字节解释成文本”这件事封装起来。**

## 字节流适合哪些场景

字节流是 IO 的底座。只要你处理的是二进制内容，或者你不想太早掺进编码语义，就该先站在字节流这一层。

典型场景包括：

- 图片、视频、音频、压缩包；
- 网络 socket 收发的原始报文；
- 文件复制；
- 对象序列化前后的字节传输；
- 你明确要自己控制编码解码。

常见字节输入/输出类可以先这么分：

| 类型   | 例子                                           | 解决什么问题                       |
| ------ | ---------------------------------------------- | ---------------------------------- |
| 文件流 | `FileInputStream` / `FileOutputStream`         | 直接和文件打交道                   |
| 缓冲流 | `BufferedInputStream` / `BufferedOutputStream` | 减少系统调用次数                   |
| 数据流 | `DataInputStream` / `DataOutputStream`         | 按 `int`、`long`、`UTF` 等类型读写 |
| 对象流 | `ObjectInputStream` / `ObjectOutputStream`     | Java 对象序列化和反序列化          |

这里顺手纠正一个很常见的误区：
`DataInputStream` / `DataOutputStream` 不是“更高级的文件流”，它们只是给已有字节流补上“按类型读写”的能力，本质仍然是字节流。

## 字符流适合哪些场景

字符流适合**明确在处理文本**的场景，比如：

- 读取配置文件、日志文件、SQL 文件；
- 写 CSV、JSON、HTML、脚本；
- 按“字符”而不是“字节”来做读取、拼接和输出。

字符流这一层最关键的不是 `FileReader`、`BufferedReader` 这些类名，而是要理解：

**字符流并不是绕过字节，而是在字节之上多了一步编码/解码。**

这也是为什么它比字节流更适合文本，但不适合图片、压缩包这类二进制内容。因为一旦你把纯二进制错误地当字符解释，数据就已经变味了。

## 真正的桥：`InputStreamReader` 和 `OutputStreamWriter`

IO 体系里最关键的一对桥接类其实是：

- `InputStreamReader`：字节输入流 -> 字符输入流
- `OutputStreamWriter`：字符输出流 -> 字节输出流

它们的重要性远高于 `FileReader` / `FileWriter`，因为这两个类才真正承担了**编码转换**。

看一个最典型的读文本组合：

```java
BufferedReader reader = new BufferedReader(
    new InputStreamReader(
        new FileInputStream("app.log"),
        StandardCharsets.UTF_8
    )
);
```

这段代码分工其实非常清楚：

- `FileInputStream`：把文件里的原始字节读出来；
- `InputStreamReader`：按 UTF-8 把字节解码成字符；
- `BufferedReader`：再给字符读取加缓冲和按行读取能力。

所以别把 `Reader` 和 `InputStream` 理解成两个平行世界。更准确地说，**字符流常常是“字节流 + 编码桥接”之后的产物**。

## `FileReader` / `FileWriter` 为什么经常不建议直接用

很多入门资料会把 `FileReader` / `FileWriter` 当作读写文本的首选类，但工程里通常会更谨慎一点。

原因不复杂：它们是方便，但**方便意味着你容易忽略字符集这件事**。

如果你直接这样写：

```java
FileReader reader = new FileReader("app.log");
```

那读出来用什么字符集，取决于你走的是哪个构造方法，以及运行环境默认字符集是什么。
只要文件编码和运行环境默认字符集不一致，就可能出现乱码。

更稳的方式通常是显式指定字符集，比如：

```java
BufferedReader reader = new BufferedReader(
    new InputStreamReader(
        new FileInputStream("app.log"),
        StandardCharsets.UTF_8
    )
);
```

或者直接用 NIO 提供的更现代写法：

```java
BufferedReader reader = Files.newBufferedReader(path, StandardCharsets.UTF_8);
```

一句话概括：**`FileReader` / `FileWriter` 不是不能用，而是显式字符集通常更安全。**

## 缓冲流解决的不是“类型”，而是“效率”

很多人第一次看 `BufferedInputStream`、`BufferedReader` 时，会误以为它们是另一种独立流类型。其实它们解决的是另一个问题：**性能**。

如果你每读一个字节、每写一个字符，就直接触发一次底层系统调用，成本会很高。缓冲流的思路是先在内存里准备一块缓冲区，批量读、批量写，减少频繁和内核打交道的次数。

所以要把这两类能力分开看：

- 字节流 / 字符流：决定你处理的是哪种语义单位；
- 缓冲流：决定你用什么方式更高效地搬运数据。

这也是为什么你经常会看到它们套娃出现：

```java
BufferedInputStream bis = new BufferedInputStream(
    new FileInputStream("a.zip")
);

BufferedReader br = new BufferedReader(
    new InputStreamReader(
        new FileInputStream("a.txt"),
        StandardCharsets.UTF_8
    )
);
```

前者是“字节 + 缓冲”，后者是“字节 + 解码 + 字符缓冲”。

## 用什么流，先看数据类型，再看操作目标

真正写业务代码时，不要从“我记住了哪个类”出发，而要从“我现在操作的是什么数据”出发。

| 场景                   | 更合适的选择                 | 原因                           |
| ---------------------- | ---------------------------- | ------------------------------ |
| 复制图片、压缩包、PDF  | 字节流                       | 这些都是二进制，不能按字符解释 |
| 读取 UTF-8 配置文件    | 字符流 + 显式字符集          | 业务关心文本，不只是原始字节   |
| 网络协议编解码         | 先字节流 / ByteBuffer        | 协议边界通常按字节定义         |
| 按行读日志             | `BufferedReader`             | 有字符语义，还要按行处理       |
| 输出 JSON / SQL / HTML | `Writer` 或 `BufferedWriter` | 生成的是文本                   |

一个实用判断是：

- **先问自己“这是文本吗？”**
- 如果不是，别碰字符流；
- 如果是，再问“编码我能不能明确写出来？”能的话就别偷懒交给默认值。

## 容易踩的坑

### 1. 以为字符流比字节流“高级”

不是。字符流只是更适合文本。
图片、视频、压缩包、网络二进制协议这类内容，字符流不但不高级，反而是错工具。

### 2. 以为字符流天然不会乱码

也不对。字符流只是帮你做了解码，但**解码也得基于正确字符集**。
字符集选错，一样乱码。

### 3. 把 `available()` 当成“文件剩余总长度”

`available()` 表示的是**当前不阻塞就能读到多少字节**，不是所有场景下都等于总大小。
对文件流你可能感觉它“差不多能用”，但对网络流这么理解经常出错。

### 4. 以为 `close()` 只关最外层那一层

IO 套娃里，通常只需要关闭最外层流，外层会把内层一起关掉。
反过来，如果你提前把内层关了，外层也就没法继续工作了。

### 5. 把“缓冲”和“编码”混在一起

`BufferedReader` 的核心价值是缓冲和按行读取，`InputStreamReader` 的核心价值是解码。
它们经常一起出现，但解决的不是同一个问题。

## 小结

- Java IO 先按两条轴线理解：输入/输出、字节/字符；四个核心抽象基类分别是 `InputStream`、`OutputStream`、`Reader`、`Writer`。
- 字节流处理原始二进制数据，字符流处理按某种字符集解释后的文本；字符流不是更高级，只是更适合文本。
- `InputStreamReader` 和 `OutputStreamWriter` 是字节流与字符流之间真正的桥，编码转换主要发生在这里。
- 缓冲流解决的是性能问题，不是数据语义问题；它常和字节流、字符流叠加使用。
- 工程里选流的关键不是背类名，而是先判断“是不是文本”和“字符集要不要显式指定”。
