# BIO、NIO、AIO 有什么区别？分别适合什么场景？

> 三个名字看着像三代技术升级，其实它们对应的是操作系统里几种截然不同的 IO 处理思路。想讲明白区别，得先把「同步/异步」和「阻塞/非阻塞」这两把尺子分开。

## 先分清两组正交的概念

面试里被问 BIO/NIO/AIO 的区别，十有八九会先卡在概念上。很多人把「同步阻塞」当成一个词、「异步非阻塞」当成另一个词，好像只有这两种搭配。其实这是**两组彼此独立的维度**，理论上能排列组合出四种。

先说清楚一次网络 IO（比如 `read` 一个 socket）在内核眼里分成两个阶段：

1. **等数据就绪**：数据还在网卡缓冲区、还没到内核，内核得等它到齐。
2. **拷数据**：数据到了内核缓冲区，再从内核空间拷贝到你的用户空间缓冲区。

这两个阶段是理解所有 IO 模型的钥匙。很多人一上来就背「BIO 阻塞、NIO 非阻塞、AIO 异步」，却说不清阻塞的到底是哪一步、异步又异步在哪里，根子就在于没把这两个阶段拆开。下面两把尺子，量的就是这两个阶段。

**阻塞 vs 非阻塞**，说的是「等数据就绪」这个阶段，调用方线程会不会被挂起：

- 阻塞：调了 `read`，数据没来，线程就在那睡着，啥也干不了，直到有数据。
- 非阻塞：调了 `read`，数据没来立刻返回一个「还没好」（Linux 上是 `EWOULDBLOCK`），线程可以先去干别的，过会儿再来问。

**同步 vs 异步**，说的是「拷数据」这个动作到底谁来做、谁在等：

- 同步：不管前面等的时候阻不阻塞，真正把数据从内核拷到用户空间这一步，是你自己的线程发起并等着完成的。
- 异步：你把活儿全交给内核，内核把两个阶段都干完、数据都放进你的缓冲区了，再回调通知你一声。整个过程你的线程一秒都不用等 IO。

对照一下就清楚了：

| 维度          | 关注的问题                 | 关键区别                           |
| ------------- | -------------------------- | ---------------------------------- |
| 阻塞 / 非阻塞 | 等数据就绪时线程要不要挂起 | 挂起 = 阻塞；立刻返回 = 非阻塞     |
| 同步 / 异步   | 拷数据这一步谁在等         | 自己等 = 同步；内核干完回调 = 异步 |

一句话记牢：**只要拷数据那一步还得自己等，就都是「同步」**。BIO、NIO 全是同步的，只有 AIO 是异步。这也是为什么 NIO 虽然能扛高并发，却仍归在同步阵营——它省的是「等就绪」的开销，拷数据那一步依然要自己来。

把三种模型在这两个阶段上线程做了什么摊开看，区别一目了然：

```
                等数据就绪          拷数据（内核 -> 用户）
BIO   线程 |------ 阻塞等 ------|------ 阻塞等 ------| 拿到数据
NIO   线程 |  select 就绪通知   |------ 阻塞等 ------| 拿到数据   ← 就绪前可管别的连接
AIO   线程 | 发起后立刻返回，去干别的事……            | 内核回调：数据已就位
```

BIO 两段全阻塞；NIO 靠 `select` 省掉了在单个连接上死等就绪的开销，但拷数据仍要自己等；AIO 两段都交给内核，线程全程不为这次 IO 停留。

UNIX 下完整的 IO 模型有五种：同步阻塞、同步非阻塞、IO 多路复用、信号驱动、异步 IO。Java 只用到了其中三条路子，也就是下面要讲的 BIO、NIO、AIO。信号驱动 IO 在 Java 里没有对应封装，平时基本不会考到，了解即可。

## BIO：一个连接一个线程

BIO（Blocking I/O）是 **同步阻塞** 模型，也是 JDK 1.0 就有的传统 IO，对应 `java.io` 包里那些 `InputStream`/`OutputStream` 和 `Socket`/`ServerSocket`。

它的工作方式非常直白：线程调用 `accept()` 等连接，没连接来就阻塞；连接来了拿到 socket，再调 `read()` 读数据，没数据来又阻塞。等数据、拷数据两个阶段线程全程被挂起。

问题在于，一个线程同一时刻只能盯着一个连接。要同时服务多个客户端，最朴素的写法就是「一个连接开一个线程」：

```java
ServerSocket server = new ServerSocket(8080);
while (true) {
    Socket socket = server.accept();      // 没连接就阻塞
    new Thread(() -> {
        InputStream in = socket.getInputStream();
        byte[] buf = new byte[1024];
        int len = in.read(buf);           // 没数据就阻塞
        // 处理请求...
    }).start();
}
```

连接数少的时候这套跑得很稳，代码也好懂。可以做个粗略的估算：假设一台机器有 4 GB 可用内存、每个线程栈占 1 MB，那么单纯线程栈就把内存吃干时，也就撑起几千个线程，离十万连接差着两个数量级。而且这还没算线程调度的开销。连接一多就撑不住，原因有两个：

- **线程数随连接数线性暴涨**。一万个连接就得一万个线程，每个线程栈默认要占几百 KB 到 1 MB 内存，光内存就顶不住；线程一多，CPU 花在上下文切换上的时间也急剧上升。
- **大量线程阻塞着空耗资源**。真实场景里,绝大多数连接在大多数时间都是没数据的（比如长连接的心跳间隙）。这些线程全都阻塞在 `read()` 上睡大觉，占着内存却啥也不干，纯属浪费。

所以 BIO 的适用场景很明确：**连接数不多、但每个连接都比较活跃**的场合，比如内部服务之间少量的长连接、或者并发压力本来就不大的应用。这类场景下 BIO 编程模型简单、调试直观，反而是首选。真正需要扛十万、百万连接时，就得换思路了。

## NIO：一个线程管一堆连接

NIO 是 Java 1.4 引入的，对应 `java.nio` 包，提供了 `Channel`（通道）、`Buffer`（缓冲区）、`Selector`（选择器）三大抽象。名字里的 N，与其说是 New，不如理解成 Non-blocking 更贴切。

这里要专门澄清一个常见的说法。不少资料把 Java NIO 简单归成「同步非阻塞 IO 模型」，这不算错，但**不够准确，容易误导**。得把「非阻塞」和「多路复用」分两层看。

先看纯粹的**同步非阻塞**是怎么回事。相比 BIO，非阻塞的改进在于 `read` 不再挂起线程：数据没就绪就立刻返回「还没好」，线程可以先做点别的、过会儿再来问。这确实比一直阻塞强，但它的问题是——**得靠线程自己一遍遍轮询**。成千上万个连接，你得挨个去问「你好了没」，绝大多数时候得到的都是「没好」。这种空转的系统调用非常费 CPU，实际上没人会这么裸用。

真正让 NIO 能扛高并发的，是 **IO 多路复用**。它的思路是：与其自己一个个轮询，不如把「我关心哪些连接」一次性交给内核，内核帮你盯着，哪个连接的数据就绪了就告诉你，你只处理就绪的那几个。Java 里干这活儿的就是 `Selector`：

```java
Selector selector = Selector.open();
channel.configureBlocking(false);                    // 通道设为非阻塞
channel.register(selector, SelectionKey.OP_READ);    // 注册关心的事件
while (true) {
    selector.select();                               // 阻塞，直到有通道就绪
    Set<SelectionKey> keys = selector.selectedKeys();
    for (SelectionKey key : keys) {
        if (key.isReadable()) {
            // 只处理真正有数据的连接
        }
    }
    keys.clear();
}
```

一个线程 + 一个 `Selector`，就能管理成千上万个 `Channel`，只在连接真正有数据时才为它服务。空闲连接不占线程、不空转，这才是它省资源的根子。

那 `Selector` 底层靠什么？靠的正是操作系统提供的多路复用系统调用——Linux 上是 `epoll`、老一点的是 `select`/`poll`。所以 Java NIO 的高并发能力，**本质是操作系统 IO 多路复用能力的封装**。至于 `select`/`poll`/`epoll` 各自的机制、为什么 `epoll` 更高效、以及 Reactor 事件驱动模型这些操作系统层面的细节，在计算机基础/操作系统专题有专篇展开，可以看 [IO 多路复用](/cs-basics/operating-system/os-io-multiplexing.html)，这里就不铺开了，本篇聚焦 Java 视角。

回到那个「同步非阻塞」的说法：Java NIO 里的 `Channel` 确实是配置成非阻塞的，但真正扛并发的机制是 `Selector` 背后的多路复用，而不是靠线程空转轮询。所以更准确的定位是——**NIO = 非阻塞 Channel + IO 多路复用**。另外它仍然是同步的：`select()` 告诉你数据就绪后，把数据从内核拷到 `Buffer` 这一步，还是当前线程自己发起 `read` 并等着完成的。

`Channel`、`Buffer`、`Selector` 三者具体怎么配合，是 NIO 编程的重头戏，单独用下一篇讲。这里只需要记住 NIO 解决了 BIO「一连接一线程」的死穴。

## AIO：把活儿全丢给内核

AIO（Asynchronous I/O）也叫 NIO 2，是 Java 7 引入的，对应 `java.nio.channels` 里的 `AsynchronousSocketChannel` 这类 API。它是三者里唯一真正的 **异步** 模型。

前面说过，NIO 虽然不阻塞在等待上，但拷数据那一步还得自己等。AIO 更进一步：应用发起调用后立刻返回，**等数据就绪和拷数据两个阶段全交给内核**，内核把数据都放进你的缓冲区了，再通过回调（`CompletionHandler`）或 `Future` 通知你去处理结果。整个 IO 过程，你的线程一秒都不用为它等待。

```java
AsynchronousSocketChannel channel = AsynchronousSocketChannel.open();
channel.read(buffer, attachment, new CompletionHandler<>() {
    public void completed(Integer result, Object att) {
        // 内核已经把数据读进 buffer 了，这里直接处理
    }
    public void failed(Throwable exc, Object att) { /* 处理异常 */ }
});
// read 立刻返回，线程继续干别的
```

听起来很美，但 AIO 在 Java 服务端用得并不广，原因主要在操作系统层面：

- **Linux 上的异步 IO 实现一直不够成熟**。Windows 有 IOCP 这套完整的异步机制，而 Linux 早期的 `AIO` 支持有限、坑多，JDK 在 Linux 上的 AIO 实际是用 `epoll` 模拟出来的，并不是真正的内核级异步。结果就是——**性能相比基于 `epoll` 的 NIO 并没有明显提升**，`epoll` 已经够用了，多包一层反而不划算。
- **生态和心智成本**。Netty 早期尝试过 AIO，测下来在 Linux 上没占到便宜，后来又退回了 NIO。业界主流网络框架几乎都建立在 NIO 之上，AIO 缺少成熟的生态支撑。

所以现实是：需要异步、高并发的服务端，大家用的是「NIO + Reactor + 线程池」这套（Netty 就是代表），而不是 AIO。AIO 更多出现在特定的文件异步读写等场景。

这里也提醒一句：AIO 的「异步」和我们平时说的「异步编程」（回调、`Future`、`CompletableFuture`）不是一个层面的事。AIO 的异步是**内核帮你把整个 IO 干完**这种系统层面的异步；而应用层拿线程池模拟的「异步」，底层那次 IO 往往还是同步阻塞的，只是换了个线程去等而已。答题时别把两者混为一谈。

## 三者放一起对比

| 模型 | 同步/异步 | 阻塞/非阻塞          | 线程模型               | 适用场景                      | 代表                          |
| ---- | --------- | -------------------- | ---------------------- | ----------------------------- | ----------------------------- |
| BIO  | 同步      | 阻塞                 | 一连接一线程           | 连接数少、连接活跃、追求简单  | `java.io`、传统 `Socket`      |
| NIO  | 同步      | 非阻塞（+ 多路复用） | 一线程管多连接         | 高并发、连接多但多数不活跃    | `java.nio`、`Selector`、Netty |
| AIO  | 异步      | 非阻塞               | 内核回调，不占线程等待 | 理论上高并发，实际 Linux 少用 | NIO 2、`AsynchronousChannel`  |

看这张表容易记混，抓两个锚点：**只有 AIO 是异步**（拷数据不用自己等）；**BIO 和 NIO 都是同步**，区别在等就绪时阻不阻塞、以及一个线程能不能管多个连接。

顺带说一句，实际项目里我们几乎不直接裸写 NIO——它的 API 繁琐、还有臭名昭著的 `epoll` 空轮询 bug 之类的坑。真正被广泛使用的是 **Netty**，它在 NIO 之上做了封装，屏蔽了这些复杂度。所以「Netty 基于什么 IO 模型」的答案就是 NIO，这也是下一条主线要展开的内容。

## 容易踩的坑

- **把「同步/异步」和「阻塞/非阻塞」当成一回事**。它俩是两个维度：前者看拷数据谁在等，后者看等就绪时线程挂不挂起。搞混就会答不清 NIO 为什么是「同步非阻塞」。
- **说「NIO 就是非阻塞 IO」**。不够准确。纯非阻塞轮询很费 CPU、没人裸用；NIO 真正靠的是 `Selector` 背后的 IO 多路复用。答题时点出「非阻塞 + 多路复用」才到位。
- **以为 NIO 全程不阻塞、是异步的**。`select()` 本身会阻塞等就绪，拷数据也要自己等，它仍是同步模型。真正异步的只有 AIO。
- **鼓吹 AIO 性能最强所以最该用**。理论上层级最高，但 Linux 下实现不成熟、相比 `epoll` 没明显优势，生态也弱，服务端主流仍是 NIO。
- **N 只理解成 New**。理解成 Non-blocking 更能抓住它区别于 BIO 的本质。

## 小结

- IO 分「等数据就绪」和「拷数据」两阶段；阻塞/非阻塞看第一阶段线程挂不挂起，同步/异步看第二阶段谁在等。
- BIO 同步阻塞，一连接一线程，简单直观，但线程随连接暴涨、大量阻塞空耗资源，只适合连接少的场景。
- NIO 同步非阻塞 + IO 多路复用，一个 `Selector` 管一堆 `Channel`，只处理就绪连接，靠操作系统的 `epoll` 等能力扛高并发；拷数据仍要自己等，所以还是同步。
- AIO 是唯一的异步模型，内核干完两阶段再回调；但 Linux 实现不成熟、相比 NIO 无明显收益，Netty 也弃用了，实际用得少。
- 记忆锚点：只有 AIO 异步；高并发服务端主流是 NIO（Netty），不是 AIO。
