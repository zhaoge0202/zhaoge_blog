# Java 是编译型语言还是解释型语言？

> 标准答案是“编译与解释并存”。但真正要讲清的是：一段 `.java` 从源码到跑起来，中间经过了谁的手、为什么这么设计。

## 一段代码从写完到跑起来经历了什么？

写一个 `Hello.java`，到它在机器上打印出结果，中间有两道关卡：

```text
Hello.java  --javac 编译-->  Hello.class（字节码）  --JVM 执行-->  机器码 -> CPU 运行
```

第一道关卡是 `javac`，它把源码编译成 `.class` 文件，里面装的不是机器码，而是**字节码**（bytecode）。第二道关卡是 JVM，它在运行时把字节码变成当前这台机器的 CPU 能认的机器码再执行。

关键在于：`javac` 编译出来的字节码**不针对任何具体 CPU**，它只面向 JVM 这个“虚拟的机器”。这就是 Java 那句老口号 “Write Once, Run Anywhere” 的技术底座——同一份 `.class`，在 Windows、Linux、macOS 上都能跑，因为每个平台都有自己的 JVM 实现负责把字节码翻译成本地机器码。

## 字节码到底是什么？为什么要多这一层？

如果直接把 Java 编译成机器码（像 C 那样），那 Windows 编出来的程序拿到 Linux 就跑不了，因为机器码是和操作系统、CPU 指令集强绑定的。Java 的办法是中间插一层字节码：

- **对上**：`javac` 只需要生成一种字节码，不用关心最终跑在什么机器上。
- **对下**：字节码由各平台的 JVM 负责翻译，跨平台的脏活累活都甩给 JVM。

这层抽象还带来一个额外好处：任何语言只要能编译成合法的字节码，就能跑在 JVM 上。Kotlin、Scala、Groovy 就是这么和 Java 共享同一套运行时的。

## 为什么说“编译与解释并存”？

先分清两类语言：

- **编译型**：源码被一次性翻译成机器码再执行，比如 C、C++、Go、Rust。执行快，但改一行就要重新编译，且不跨平台。
- **解释型**：由解释器一行行边翻译边执行，比如 Python、JavaScript。灵活、跨平台，但执行慢。

Java 两头都沾：

1. **编译**这一步是 `javac` 把 `.java` 编成 `.class`，这是真的编译。
2. **解释**这一步是 JVM 启动后，用**解释器**逐条把字节码解释成机器码执行。

所以 Java 既不是纯编译，也不是纯解释，而是“先编译成字节码，再解释执行字节码”。这就是“编译与解释并存”的由来。

但只到解释这步还不够快。真正让 Java 性能拉起来的是下面的 JIT。

## JIT：把热点代码直接编译成机器码

纯解释执行的问题是：同一段循环体被跑一百万次，解释器就要把它翻译一百万次，纯属浪费。

JVM（以最常见的 HotSpot 为例）的对策是 **JIT（Just-In-Time，即时编译）**：

- 运行时统计每段代码被执行的次数，把频繁执行的那部分认定为**热点代码**。
- 对热点代码，JIT 直接编译成机器码并**缓存起来**，之后再执行就跳过解释、直接跑机器码。

这就是为什么一个 Java 服务刚启动时慢、跑一会儿“热”起来之后就快了——热点代码被 JIT 编译并缓存了，这段“热身”过程通常叫**预热（warm up）**。

所以完整的执行链路是：**解释器保证启动就能跑，JIT 负责把跑得多的部分优化成机器码**，两者配合。

## AOT：干脆在运行前就编译好

JIT 有个天生的短板：它是运行时才编译的，服务启动初期还在解释执行 + 收集热点信息，这段预热期性能低、启动慢。对那种“启动完可能没跑几下就被销毁”的场景（Serverless、CLI 工具、云原生里频繁扩缩容的实例），预热成本就很不划算。

JDK 9 引入的 **AOT（Ahead-Of-Time，提前编译）** 就是冲这个来的：在程序运行**之前**就把字节码编译成机器码，跟 C/C++ 的静态编译一个路子。启动即巅峰，不用预热，内存占用也更低。GraalVM 是这条路线上最有代表性的实现。

JIT 和 AOT 不是谁取代谁，而是各有取舍：

| 对比维度 | JIT（即时编译）                | AOT（提前编译）                    |
| -------- | ------------------------------ | ---------------------------------- |
| 编译时机 | 运行时                         | 运行前                             |
| 启动速度 | 慢（要预热）                   | 快（无需预热）                     |
| 峰值性能 | 更高（能拿到运行时信息做优化） | 略低（缺运行时信息）               |
| 内存占用 | 较高                           | 较低                               |
| 包体积   | 较小                           | 较大（含机器码）                   |
| 动态特性 | 完整支持                       | 受限（反射、动态代理等要额外适配） |
| 适用场景 | 长期运行的服务                 | Serverless、CLI、云原生            |

**为什么不干脆全用 AOT？** 因为 AOT 是提前静态编译的，它没法支持 Java 的很多动态特性——反射、动态代理、运行时生成字节码（比如 CGLIB 用 ASM 在内存里现造 `.class`）。而 Spring、MyBatis 这些主流框架大量依赖这些能力。全 AOT 就等于放弃了这套生态，所以 JIT 仍是服务端的主流，AOT 是特定场景的补充。

## JVM、JDK、JRE 到底谁包含谁？

这三个概念经常被搅混，其实是一层套一层的关系：

```text
JDK（开发工具包）
├── JRE（运行环境）
│   ├── JVM（虚拟机，执行字节码）
│   └── 核心类库（java.lang、java.util、IO、网络……）
└── 开发工具（javac 编译器、javadoc、jdb 调试器、javap 反编译……）
```

- **JVM**：执行字节码的虚拟机。它只是一套**规范**，HotSpot 只是最常见的一种实现，此外还有 J9、Zing 等。
- **JRE**：运行一个已经编译好的 Java 程序所需的最小环境 = JVM + 核心类库。只想“跑”程序，有 JRE 就够。
- **JDK**：在 JRE 基础上加了开发工具（`javac`、`javap` 等）。要“写、编译、调试”，必须装 JDK。

一句话记：**JDK 用来开发，JRE 用来运行，JVM 负责执行字节码。**

> 有个容易过时的说法值得点一下：早年安装时确实要区分“装 JDK 还是装 JRE”。但从 **JDK 9 起** JDK 被模块化重组，配合 `jlink` 可以按需裁剪出自定义运行时镜像；从 **JDK 11 起 Oracle 不再单独提供 JRE 下载**。所以在新版本里，“单独下一个 JRE”这件事本身已经淡出了，别再按十年前的经验答。

## Oracle JDK 和 OpenJDK 怎么选？

- **OpenJDK**：完全开源（GPL v2），商业免费。
- **Oracle JDK**：基于 OpenJDK 构建，不完全开源，部分版本商用有时间/付费限制。

从 Java 11 之后，两者功能基本一致，Oracle JDK 曾经的私有组件大多已捐给开源社区。实践中直接选 OpenJDK 或它的发行版更省心：AWS Corretto、阿里 Dragonwell、Azul Zulu 等都是长期支持、生产可用的选择。

## 小结

- Java 源码先由 `javac` 编译成**字节码**（`.class`），再由 JVM 在运行时翻译成机器码执行，所以叫“编译与解释并存”。
- 字节码这层抽象让 Java 跨平台，也让 Kotlin、Scala 等语言能共享 JVM。
- JVM 启动靠**解释器**保证能跑，再用 **JIT** 把热点代码编译成机器码缓存，这就是服务需要“预热”的原因。
- **AOT** 在运行前就编译好，启动快、省内存，适合云原生；但受限于动态特性，无法完全取代 JIT。
- 关系上 JDK ⊃ JRE ⊃ JVM：开发用 JDK，运行用 JRE，执行字节码靠 JVM。JDK 9 后 JRE 已不再单独分发。
