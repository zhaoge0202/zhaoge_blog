# 为什么说 Java 只有值传递？

> 一句话点题：不管你传的是基本类型还是对象，方法拿到的永远是实参的一份拷贝；判断是不是引用传递，只看一件事——能不能在方法里让实参本身指向另一个对象。

这道题几乎是 Java 面试的必考题，坑也集中在一个地方：很多人看到「方法里改了对象的字段，外面也变了」，就一口咬定这是引用传递。这个结论是错的。下面我们把概念先掰开，再用三个递进的例子把它证死。

## 先分清两组容易混的概念

第一组是**形参和实参**。定义方法时写在括号里的变量叫形参（parameter），它只是个占位的名字；真正调用方法时塞进去的那个具体的值叫实参（argument）。

```java
String hello = "Hello!";
sayHello(hello);              // hello 是实参

void sayHello(String str) {   // str 是形参
    System.out.println(str);
}
```

第二组才是重点，**值传递和引用传递**，它俩描述的是「实参怎么交给形参」：

|                                   | 形参拿到的是什么           | 在方法里给形参重新赋值                 |
| --------------------------------- | -------------------------- | -------------------------------------- |
| **值传递（pass by value）**       | 实参的一份**拷贝**（副本） | 只动副本，实参纹丝不动                 |
| **引用传递（pass by reference）** | 实参本身的**别名**         | 形参就是实参，重新赋值会同步反映到实参 |

关键差别就在最后一列：引用传递里形参和实参是**同一个变量的两个名字**，你把形参重新指向别的东西，实参也跟着变；值传递里形参是**另一个独立的变量**，只是初始值抄自实参，怎么折腾形参都传不回去。

C++、Pascal 这些语言两种方式都提供，而 **Java 只有值传递**。下面证明它。

## 案例 1：传基本类型，改了个寂寞

```java
public static void main(String[] args) {
    int num1 = 10, num2 = 20;
    swap(num1, num2);
    System.out.println("num1 = " + num1);   // 10
    System.out.println("num2 = " + num2);   // 20
}

public static void swap(int a, int b) {
    int temp = a; a = b; b = temp;
    System.out.println("a = " + a);          // 20
    System.out.println("b = " + b);          // 10
}
```

方法里 `a`、`b` 确实换过来了，但 `num1`、`num2` 完全没变。因为 `a`、`b` 是 `num1`、`num2` 的副本，改副本影响不到原件。

```
栈:  num1 = 10      调用 swap 时拷贝一份 →   a = 10  →交换→  a = 20
     num2 = 20                              b = 20          b = 10
     （main 里的 num1/num2 始终是 10 和 20）
```

这一步大家都没异议，分歧从对象开始。

## 案例 2：传对象引用，对象的字段真的被改了

```java
public static void main(String[] args) {
    int[] arr = {1, 2, 3, 4, 5};
    System.out.println(arr[0]);   // 1
    change(arr);
    System.out.println(arr[0]);   // 0
}

public static void change(int[] array) {
    array[0] = 0;                 // 改的是数组内容
}
```

这回外面的 `arr[0]` 真变成 0 了。**这就是最容易让人误判成引用传递的地方**，但它恰恰仍是值传递。

诀窍在于想清楚「拷贝的到底是什么」。`arr` 这个变量本身存的不是数组，而是指向堆里那个数组对象的**引用值**。调用 `change` 时，Java 拷贝的是这个引用值，于是 `array` 和 `arr` 是两个不同的变量，但它们里面装的引用值相同，**指向堆里同一个数组**。所以顺着 `array` 去改数组内容，从 `arr` 看过去当然也变了——它俩看的是同一个对象。

```
        栈                          堆
  ┌──────────────┐
  │ arr    ●─────┼──┐
  └──────────────┘  │      ┌───────────────────┐
  ┌──────────────┐  ├────► │ int[]: [0,2,3,4,5] │
  │ array  ●─────┼──┘      └───────────────────┘
  └──────────────┘
   两个变量各存一份引用，但指向同一个数组
```

注意：这里被改的是**堆里的对象**，不是实参变量 `arr`。`arr` 存的引用值从头到尾没动过。要证明这一点，看案例 3。

## 案例 3：在方法里 new 一个新对象，实参岿然不动

```java
public static void main(String[] args) {
    int[] arr = {1, 2, 3};
    reset(arr);
    System.out.println(arr[0]);   // 还是 1，不是 100
}

public static void reset(int[] array) {
    array = new int[]{100, 200};  // 让形参改指向一个全新数组
    System.out.println(array[0]); // 100
}
```

`reset` 里我们没去改原数组的内容，而是让形参 `array` **重新指向**一个 `new` 出来的新数组。方法内部打印 `100`，可 `main` 里的 `arr[0]` 依然是 `1`。

```
                        栈                     堆
调用后:   arr    ●───────────────────────►  [1, 2, 3]   （原数组，没人动它）

reset 内: array  ●───────────────────────►  [100, 200]  （new 出来的新数组）
          array 重新指向新对象，跟 arr 再无关系
```

这就是决定性证据。如果 Java 是引用传递，`array` 就是 `arr` 的别名，`array = new int[]{...}` 会让 `arr` 也指向新数组，外面就该读到 `100`。但事实是 `arr` 纹丝不动——**说明 `array` 只是 `arr` 引用值的一份拷贝，重新赋值只改了这份拷贝的指向，传不回去。**

对比案例 2 和案例 3：**改对象内部（`array[0]=0`）会影响外面，改形参指向（`array=new`）不会影响外面。** 前者动的是共享的堆对象，后者动的是各自独立的引用变量。这一组对照把「值传递」讲得明明白白。

## 经典反例：为什么 Java 写不出真正交换引用的 swap

顺着案例 3 的逻辑，就能理解一个经典结论：**Java 里没法写出一个能真正交换两个实参对象引用的 swap 方法。**

```java
public static void swap(Person p1, Person p2) {
    Person temp = p1;
    p1 = p2;
    p2 = temp;   // 换的只是两个局部形参的指向
}

Person zhang = new Person("小张");
Person li    = new Person("小李");
swap(zhang, li);
// zhang 还是小张，li 还是小李，没换成
```

`p1`、`p2` 只是 `zhang`、`li` 引用值的拷贝。方法里把 `p1`、`p2` 换来换去，动的全是这两个局部副本的指向，方法一返回它们就销毁了，`zhang`、`li` 的指向从没被触碰。

想在 Java 里实现交换效果，只能绕道：要么交换两个对象**内部的字段值**（借用案例 2 的原理，改共享对象的内容），要么把两个对象装进一个数组/容器里传进去，去交换容器里的元素。你永远无法直接交换调用方那两个变量本身的指向——因为语言根本没给你引用传递这个能力。

## 顺带看看真正的引用传递长什么样（C++）

C++ 用 `&` 声明引用形参，那才是货真价实的引用传递：

```cpp
void incr(int& num) {   // 注意是 int&，不是 int
    num++;
}
int main() {
    int age = 10;
    incr(age);
    std::cout << age;   // 输出 11，实参被改了
}
```

这里 `num` 就是 `age` 的别名，`num++` 直接作用在 `age` 上。如果去掉 `&` 写成 `int num`，它立刻退化成值传递，`age` 又不变了。对照之下 Java 没有任何语法能达到这个效果，这也是「Java 只有值传递」的直观佐证。

## 一个要澄清的措辞：引用值 ≠ 内存地址

很多资料（包括常见的讲法）会说「传的是实参的地址」。这个说法帮助理解没问题，但严格讲不够准确，面试时最好心里有数：

- Java 里变量存的是**引用值（reference）**，它是 JVM 层面对对象的一个句柄，并**不保证等于对象在内存里的物理地址**。
- HotSpot 的垃圾回收器（如标记-整理、复制算法）会在 GC 时**移动对象、压缩堆空间**，对象的物理地址会变，但你手里的引用依旧有效。这说明引用值和裸指针（C 里的内存地址）不是一回事。

所以更严谨的表述是：**传对象时，拷贝的是实参的「引用值」**，而不是「内存地址」。结论不变（依然是值传递），但用词更经得起追问。

## 为什么 Java 要这么设计

这属于设计取舍，官方没有权威定论，通常认为有两点考量：

- **安全和封装**。方法对外是个「黑盒接口」，调用方不该被方法内部随手重新指向实参的行为搞出意外。打个比方，你拿卡去取 100 块，结果卡被换成了别人的卡，这种「实参被偷偷换掉」的风险，值传递从语言层面就杜绝了。
- **简单**。Java 在设计之初就刻意砍掉了 C/C++ 里容易踩坑的特性（指针运算、引用传递等），语言里少一个概念，开发者就少一类 bug。用「统一的值传递 + 引用类型」这一套，既能共享对象，又不会让人写出难以预料的代码。

## 容易踩的坑

- **「对象被改了 = 引用传递」是错的。** 那只是拷贝的引用和原引用指向同一个堆对象而已。判断标准永远是：能不能在方法里让**实参本身**改指向另一个对象——Java 做不到。
- **String 参数的迷惑性。** 在方法里给 `String` 形参重新赋值（`str = "new"`）外面不变，很多人误以为是「String 特殊」。其实和案例 3 一模一样，只是重新赋值了形参这份引用拷贝；再叠加 String 不可变，就更没法从内部改动原对象了。
- **别背「传地址」当标准答案。** 更准确是「传引用值的拷贝」，引用值不等于物理内存地址（GC 会移动对象）。
- **包装类型、不可变对象同理。** `Integer`、`Long`、`String` 这类不可变对象传进方法，内部无论怎么「改」，本质都是让形参指向新对象，实参不受影响。

## 小结

1. Java 只有值传递，没有引用传递——形参永远是实参的一份拷贝。
2. 基本类型拷贝的是字面量值；引用类型拷贝的是**引用值**（不是对象本身，也不严格等于内存地址）。
3. 传对象时能改到对象的字段，是因为拷贝的引用和原引用**指向同一个堆对象**，不是引用传递。
4. 判断是否引用传递的唯一标准：能否在方法内让**实参本身**指向新对象。Java 里做不到（案例 3、swap 反例都能证明）。
5. C++ 用 `int&` 才是真引用传递；Java 出于安全与简单的取舍没有引入这个能力。
