# 如何保证缓存和数据库一致性？

> 缓存一致性是 Redis 面试最容易接项目场景的题。

## 30 秒回答

常见方案是 Cache Aside：更新数据库后删除缓存，并通过重试、消息队列或 binlog 订阅提高删除成功率。

## 2 分钟回答

回答时不要承诺强一致。要先说明缓存和数据库通常追求最终一致，再讲先更新数据库后删缓存的原因、删除失败补偿、并发读写下的短暂不一致窗口。

## 深度解释

进阶点在于能讲清业务容忍度。对一致性要求极高的场景，应减少缓存或改用更强约束；对读多写少场景，可以接受短暂不一致并用补偿机制兜底。

## 回答策略

先定义一致性目标，再讲 Cache Aside，再讲失败补偿和项目取舍。

## 追问链路

- **为什么一般删除缓存而不是更新缓存？**：删除让下一次读取回源重建，避免复杂写入路径下更新缓存带来更多不一致。

## 常见误区

### 先删缓存再更新数据库一定没问题。

- 为什么错：并发读可能在数据库更新前回源旧值并写回缓存。
- 正确说法：常用策略是先更新数据库再删除缓存，并准备删除失败补偿。

## 纠偏记录

### 缓存一致性不是强一致承诺

- 问题：很多答案把延迟双删当成万能方案。
- 修正：先定义一致性目标，再选择 Cache Aside、补偿或更强约束。
- 证据：缓存架构实践和业务一致性要求。
- 来源：PERSONAL_REVIEW

## 项目映射

### 商品详情、配置、用户会话等读多写少场景。

- 项目表达：说明 Cache Aside、删除失败重试和过期兜底。
- 风险点：缓存删除失败会产生长时间脏数据。
- 面试回答：我会先更新数据库，再删除缓存；删除失败进入重试或消息补偿，并设置合理过期时间。
