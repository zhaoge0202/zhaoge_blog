# 复杂冲突和多人协作怎么处理？

> 冲突不是 Git 的病，是并行修改同一区域的结果。流程能显著减少冲突，但不能消灭；消灭的是“瞎选一边”的坏习惯。

## 冲突到底是什么

Git 在合并两个历史时，若同一文件的同一区域被双方以不同方式改过，且无法自动判定，就会停下让人决策：

```text
<<<<<<< HEAD
return discount.rate();   // 你的分支
=======
return discount.rate(ctx); // 对方合入的
>>>>>>> main
```

标记里的两边都只是文本快照。**正确结果可能是某一边，也可能是两边语义的合并**，甚至要引入第三个函数。工具只能提示冲突位置，不能替你理解业务。

二进制文件、生成物、锁文件（某些场景）冲突更麻烦：没有有意义的行级合并，只能选定版本或重新生成。

## 预防：比解决冲突便宜一个数量级

### 1. 短分支、小步提交

分支活得越久，和主干漂移越大，冲突面积越大。经验上：

- 功能分支尽量按天级合入，而不是按月
- 提交按“可理解的意图”切开，而不是一天结束一锅端
- 中间用 rebase/merge 频繁同步主干，把冲突拆成多次小的

```bash
git fetch origin
git rebase origin/main    # 或 merge，看团队规范
```

### 2. 模块边界清晰

冲突高发区往往是：

- 巨型 `Utils`、全局配置、巨大的 `routes`/`beans` 清单
- 所有人改同一张“中心表”的代码入口
- 自动格式化未统一导致的整文件重排

对策是物理拆分与所有权：公共库改动走 RFC/小 PR；生成代码与手写代码目录分离；格式化规则进仓库并用 CI 卡死。

### 3. 格式与导入顺序工具化

人为争论 tab/空格、import 排序，只会制造噪声 diff。统一：

```bash
# 例：Java
./mvnw spotless:apply
# 或项目约定的 format 任务
```

编辑器保存时格式化与 CI 校验同一套规则，冲突标记里就不会再夹着“只动了空格”的假冲突。

### 4. 约定谁先动公共文件

改 `schema.sql`、网关路由总表、权限枚举前，在频道里知会一声，或拆成可并行的附加点。协调五分钟，往往省半小时解冲突。

## 处理：同步 → 理解 → 合并语义 → 验证

推荐操作序（以更新个人分支为例）：

```bash
git status                 # 确认工作区干净
git fetch origin
git rebase origin/main     # 或 git merge origin/main
```

出现冲突后：

```bash
git status                 # 看哪些 unmerged
# 打开文件，去掉 <<<<<<< ======= >>>>>>> 后留下正确语义
git add path/to/file
git rebase --continue      # merge 流程则 git commit
```

原则：

1. **先读双方改动意图**，用 `git log` / PR 描述 / 对方提交说明，不要只看当前标记块。
2. **需要时把两边 diff 拉全**：

   ```bash
   git show :2:path/to/file   # 当前分支阶段（rebase 中语义需对照文档）
   git show :3:path/to/file
   # 或用 IDE 的三方合并视图
   ```

3. **语义合并优先于“选左边/右边”**。
4. **跑测试**，至少跑冲突相关模块。
5. 吃不准就找改另一边的人确认，比上线后吵责任便宜。

想放弃本次同步：

```bash
git rebase --abort
# 或 merge 场景
git merge --abort
```

## 复杂冲突的几类典型场景

### 双方改同一业务逻辑

例如折扣计算，一人加会员价，一人加优惠券。正确结果通常是两者都保留并规定优先级，而不是留其中一版。解完后补单测锁住组合行为。

### 重构遇到功能提交

一人重命名/搬迁文件，一人在旧路径改逻辑。Git 有时能检测 rename，有时不能。处理方式：

- 接受重构后的路径
- 把功能改动手工移植到新位置
- 必要时先合重构 PR，再 rebase 功能分支，避免“功能 + 重构”绞在一次冲突里

### 锁文件 / 生成代码

`package-lock.json`、`go.sum`、部分 protobuf 生成物：选定策略后重新生成往往比手改更稳。

```bash
# 例：选一边后重装生成 lock
git checkout --ours package-lock.json
npm install
git add package-lock.json
```

团队应写明：lock 冲突时谁的流程为准（通常是重算，不是手搓 diff）。

### 错误 resolve 后的补救

已经 `continue` 完才发现合错：

```bash
git reflog
# 找到冲突解决前的点，开分支对比
git diff <good-sha> HEAD
```

小范围可用新提交修正；大范围且尚未 push，可回到原点重做同步。已 push 的错误合并，用正向修复提交，避免再改写公共历史。

## 协作与 Code Review

冲突治理有一半发生在 PR 流程，而不在命令行：

| 实践                 | 作用                               |
| -------------------- | ---------------------------------- |
| PR 小而可审          | 审得动，冲突面也小                 |
| 描述动机、范围、风险 | 解决冲突时别人知道你想干什么       |
| 危险区双人看         | 支付、权限、迁移、合并冲突密集文件 |
| CI 必绿再合          | 防止“冲突解决了但编译不过”         |
| 禁止直推主干         | 所有改动过评审与流水线             |

PR 描述里写清“依赖尚未合入的某某 PR”或“改了公共配置”，能减少并行开发踩脚。

评审者看到巨大 diff 且夹杂格式化，应要求拆分，而不是硬审完——那是在为下一次冲突制造条件。

## 和 merge / rebase 策略的关系

- 用 **merge** 同步主干：冲突一次处理，历史留合并节点
- 用 **rebase** 同步主干：可能按提交多次冲突，线性更干净

无论哪种，**解决冲突的验收标准都是测试与业务语义**，不是“标记消失了”。多人长期 feature 分支上反复 rebase，会把冲突成本转嫁给所有协作者；这类分支更适合 merge 同步，或先拆短分支。

## 工具可以帮什么，不能帮什么

有帮助：

- IDE 三方合并视图
- `git mergetool`
- 冲突相关的单测 / 契约测试
- `git rerere`（重复相同冲突时复用上次解决，适合长期维护的分支）

帮不了：

- 替你决定业务优先级
- 发现“两边都留下但逻辑互斥”
- 保证二进制资源选对版本

把希望寄托在“合并工具够聪明”上，是冲突事故的常见前言。

## 小结

1. 短分支、小 PR、清晰模块边界，是成本最低的冲突策略。
2. 解决冲突是合并语义，不是机械选择 HEAD 或 incoming。
3. 格式化与 lock 文件策略要用工具和文档固化，去掉噪声冲突。
4. 重构与功能并行时，优先拆 PR 顺序，避免一次硬刚。
5. 每次冲突解决后以测试和必要沟通为验收；公共历史的错误用正向提交修，不靠改写掩盖。
