# 线上回滚和 Git 回退怎么配合？

> 线上回滚的第一目标是恢复服务，不是保持 Git 历史好看。制品先止血，仓库后补审计。

## 先分清三件不同的事

事故现场常把三个动作搅在一起：

| 动作         | 作用对象                 | 目的                             |
| ------------ | ------------------------ | -------------------------------- |
| 发布系统回滚 | 运行中的制品/镜像/包     | 立刻把流量打回上一好版本         |
| `git revert` | 公共分支历史             | 用新提交抵消某次坏提交，留下记录 |
| `git reset`  | 分支指针（常配合工作区） | 移动 HEAD，可能丢掉或改写提交    |

它们不是互斥，而是**时间顺序不同**。正确默认顺序几乎总是：

1. 确认坏版本与影响面
2. 发布平台一键回滚/切流量到上一制品
3. 主干用 `revert` 固化“这次线上认定有问题”
4. 修 bug 后正向发新版本
5. 复盘漏测与发布门槛

先在 Git 里 `reset --hard` 再纠结怎么发版，是把止血和考古顺序反了。

## 发布回滚：最快的止血

现代发布链路通常有不可变制品：

```text
Git tag / commit → CI 构建 → 镜像 digest / jar 版本 → 部署到环境
```

回滚理想状态是：**不重新构建**，直接把编排里的镜像 tag/digest 指回上一已知良好版本。

```bash
# 例：Kubernetes 把部署镜像拨回
kubectl set image deploy/order-api order-api=registry.example.com/order-api:1.8.3
# 或 rollout undo
kubectl rollout undo deploy/order-api
```

要点：

- 生产跑的是**制品坐标**，不是开发机上的某次 `git checkout`
- 上一版本必须仍在仓库/制品库里，且配置、DB 兼容
- 回滚后立刻看错误率、延迟、关键业务探针，而不是只看 Pod Ready

如果上一版本依赖的配置项、feature flag、数据库结构已经在坏版本里“向前不兼容”地改过，单纯拨镜像可能不够，这时要准备**配置回滚**或**兼容性修复**，不能假设 Git 回退等于世界回退。

## `git revert`：公共分支的安全反向

`revert` 生成一个新提交，内容是指定提交的逆补丁：

```bash
# 撤销某一个提交
git revert <bad-commit-sha>

# 撤销一段（注意区间语义）
git revert <oldest>^..<newest>

# 合并提交要指明主线父母
git revert -m 1 <merge-commit-sha>
```

优点：

- 不改写已有历史，同事 pull 不会炸
- CI、Code Review、审计日志都看得到“谁在何时撤销了什么”
- 和“先回滚制品”不冲突：线上已经旧版本了，Git 上补一刀防止坏提交继续被部署

合并提交的 `-m 1` 表示保留第一父（通常是主干方向）为基准。选错父母会把整次合入的另一侧逻辑弄反，操作前用 `git show <merge>` 看清父母顺序。

## `git reset`：什么时候能动

```bash
git reset --soft  <commit>   # 只移动 HEAD，改动留在暂存区
git reset --mixed <commit>   # 默认：HEAD+暂存区回退，工作区保留
git reset --hard  <commit>   # HEAD+暂存+工作区全对齐，未提交改动没了
```

适用：

- 本地还没 push 的乱提交整理
- 个人分支推错了，且确定没有别人基于它开发，再用 force-with-lease 纠正

**不适合**：已经在 `main` 上、已经被多人拉取、已经打过发布 tag 的提交。对公共历史 `reset --hard` 再 force push，等于要求所有人手工对齐，还可能让已发制品对应的 commit 从默认分支“消失”（对象或许还在，但协作成本爆炸）。

若确实要在本地找回 reset 前的位置：

```bash
git reflog
git checkout -b recover HEAD@{2}
```

`reflog` 是安全网，不是日常回滚流程的一部分。

## 推荐事故流程（可照抄演练）

```text
告警/客诉
  → 锁定坏版本号（镜像 tag / 发布单）
  → 发布平台回滚到 last-known-good
  → 验证核心链路与数据面
  → 在主干 git revert 坏提交（或禁用对应 feature flag）
  → 写事故简报：影响面、时间线、临时措施
  → 修复 → 新版本灰度 → 全量
  → 复盘：测试缺口、发布开关、监控是否滞后
```

命令层面的主干清理示例：

```bash
git checkout main
git pull
git revert <bad-sha>        # 解决冲突后提交
git push
# 触发 CI 出“已撤销”的新制品；线上若已靠发布系统回滚，可不必立刻再发
```

注意：线上已经回滚制品后，主干 `revert` 的意义是**堵住下一轮误部署**，并留下可追溯记录，不是重复止血。

## 数据兼容：回滚最容易翻车的地方

代码能拨回去，数据不一定。

| 变更类型             | 回滚风险 | 应对                 |
| -------------------- | -------- | -------------------- |
| 纯代码逻辑           | 低       | 直接拨制品           |
| 新增可空列/新表      | 中       | 旧代码一般可读       |
| 删列、改类型、改语义 | 高       | 需要扩展-收缩发布    |
| 已写出的错误业务数据 | 高       | 补偿任务，不只靠 Git |

原则：**向前兼容的迁移才能支持快速回滚**。先加列后写代码、先双写再切读、删字段放到确认无回滚需求之后——这些是发布设计，不是 Git 技巧。

回滚验证清单建议固定下来：

1. 应用健康检查与关键接口
2. 消息消费是否位点/幂等正常
3. 定时任务是否会用旧逻辑打坏新数据
4. 缓存 key 结构是否与旧代码一致
5. 外部回调/支付异步通知是否仍能对上版本

## 标签、制品与可追溯

出事后最怕“不知道线上到底跑的哪次提交”。最低要求：

```bash
# 构建时写入
git rev-parse HEAD
# 镜像打不可变 tag
order-api:1.8.4          # 语义版本
order-api@sha256:abcd…   # digest 更稳
```

发布单里同时记：Git SHA、镜像 digest、配置版本、迁移版本。回滚时按发布单拨，而不是在群里问“谁记得上一个包叫什么”。

`latest` 或可变 tag 不适合生产回滚依据——你以为拨回 latest，可能已经又被覆盖。

## 容易踩的坑

1. **公共分支 reset + force push 当回滚**  
   线上暂时“对应”了旧代码，协作历史与审计被破坏，二次事故率更高。

2. **只 revert 代码，不处理迁移/缓存**  
   服务起来了，读写却对不上，表现为偶发 5xx 或脏数据。

3. **revert 合并提交忘了 `-m`**  
   命令直接失败或撤错方向。

4. **回滚后不做验证就宣布恢复**  
   Ready 不等于业务正确。用黄金指标和一条核心用户路径冒烟。

5. **热修直接在 main 上 reset 出“干净历史”**  
   用正向前向修复 + revert，比改写历史更可运营。

## 小结

1. 事故第一动作是回滚制品恢复服务，不是先改 Git 历史。
2. 公共分支用 `revert` 留下反向提交；`reset --hard` 仅限本地或未共享分支。
3. 回滚要验证配置、数据库、缓存、消息与外部依赖的兼容，不只看进程起来。
4. 发布链路应保证制品不可变、版本可追溯，避免依赖可变 tag。
5. 修完后正向发版并复盘，把“能快速回滚”设计进迁移与开关，而不是依赖英雄操作。
