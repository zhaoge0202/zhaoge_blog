# 镜像、容器、仓库是什么关系？

> 镜像是模板，容器是进程级实例，仓库是分发渠道。把这三者关系说清，后面的构建、网络、编排才有落脚点。

## 先把对象模型钉死

很多人一上来背命令，却答不清「镜像和容器到底差在哪」。可以先记这条链路：

```text
Dockerfile  --build-->  镜像(Image)  --run-->  容器(Container)
                              ^
                              |
                         仓库(Registry / Repository)
                              |
                         pull / push
```

- **镜像**：只读的分层文件系统 + 启动元数据（入口命令、环境变量、暴露端口等）。构建完成后内容不变。
- **容器**：基于镜像创建的运行实例。镜像层之上再加一层可写层，进程跑在自己的命名空间里。
- **仓库**：集中存放和分发镜像的地方。本地能跑的镜像，换机器通常要先 pull。

类比：镜像像类，容器像对象实例；仓库像制品库。一个镜像可以同时起多个容器，彼此状态互不影响。

## 镜像为什么要分层？

镜像不是一整块 tar 包，而是多层文件系统叠在一起（UnionFS / overlay 一类机制）。构建时每条会改动文件系统的指令，大致对应一层：

```text
┌─────────────────────┐
│  业务代码层           │  经常变
├─────────────────────┤
│  依赖安装层           │  偶尔变
├─────────────────────┤
│  基础运行时层         │  很少变
├─────────────────────┤
│  OS 基础层            │  基本不动
└─────────────────────┘
```

分层带来两个直接收益：

1. **复用**：多个镜像共享同一基础层，磁盘上只存一份，拉取时也只下缺失的层。
2. **增量构建**：只改业务代码时，下面依赖层可以走缓存，不必每次重装。

要注意一个反直觉点：在上层「删除」下层文件，并不是真从下层抹掉，而是在本层打删除标记。最终运行时看不到那个文件，但层数据仍在镜像里。所以构建时临时文件、缓存包，最好在同一层装完就清掉，否则镜像会虚胖。

## 容器到底是什么？

容器的实质是**被隔离的进程**，不是一台轻量虚拟机。

| 对比项     | 容器                          | 虚拟机                   |
| ---------- | ----------------------------- | ------------------------ |
| 虚拟化层级 | 操作系统（进程隔离）          | 硬件（完整 Guest OS）    |
| 内核       | 共享宿主机内核                | 每个 VM 自己的内核       |
| 启动速度   | 秒级甚至更快                  | 通常更慢                 |
| 体积       | 镜像常几十到几百 MB           | 往往 GB 级               |
| 隔离强度   | 进程级，依赖 namespace/cgroup | 更彻底，适合多租户强隔离 |

容器依赖 Linux 的 namespaces（进程、网络、挂载等隔离）和 cgroups（CPU、内存配额）。同一宿主机上多个容器共享内核，所以：

- 比虚拟机轻，密度高；
- 内核漏洞、内核参数往往是整机级影响，不能把它当成「完全独立的小机器」。

容器存储也分层：下面是镜像只读层，最上面是**容器可写层**。容器删了，可写层默认一起没了。业务数据、数据库文件不要指望写在容器文件系统里还活着——该挂卷就挂卷。

## 仓库、标签和「不可变版本」

镜像构建完只在本机有用。要跨机器交付，就需要 Registry：

- 一个 Registry 里可以有多个仓库（Repository）；
- 一个仓库里可以有多个标签（Tag）；
- 每个标签指向某个具体镜像。

常用写法：

```bash
# 拉指定版本
docker pull mysql:8.4

# 打标签并推到私有仓库
docker tag my-app:1.2.0 registry.example.com/team/my-app:1.2.0
docker push registry.example.com/team/my-app:1.2.0
```

`latest` 只是默认标签名，**不保证永远是最新、也不保证内容不变**。有人今天 push 了 `latest`，明天又覆盖成另一份，线上回滚时就说不清跑的是哪一版。

生产更稳妥的做法：

1. 用语义化版本或构建号：`1.2.0`、`1.2.0-build.318`；
2. 需要强不可变时，用镜像 digest：`image@sha256:...`；
3. 私有仓库做权限与扫描，别默认从公网裸拉未知镜像。

国内访问公网 Registry 常慢，企业环境一般会配内部镜像仓库或加速代理；这是工程配置问题，不影响对象模型本身。

## 一条命令里发生了什么

以启动 MySQL 为例：

```bash
docker run -d --name mysql84 \
  -e MYSQL_ROOT_PASSWORD=secret \
  -p 3306:3306 \
  mysql:8.4
```

大致顺序：

1. 本地没有 `mysql:8.4` 就先 pull；
2. 基于镜像创建容器，分配可写层、网络、命名空间；
3. 按镜像默认入口（或你覆盖的 command）启动进程；
4. 端口映射把容器端口接到宿主机。

常见排查入口：

```bash
docker images          # 本地有哪些镜像
docker ps -a           # 容器状态（含已退出）
docker logs mysql84    # 进程日志
docker inspect mysql84 # 网络、挂载、环境变量细节
```

`docker commit` 能把「改过的容器」再打成镜像，适合临时抢救或教学演示；正规交付仍应以 Dockerfile 可复现构建为准，否则镜像来源不可追溯。

## 容易答偏的几点

1. **「容器 = 轻量虚拟机」**：隔离模型不同。容器共享内核，虚拟机模拟硬件。
2. **「删容器数据还在」**：默认可写层随容器走；没挂卷就没了。
3. **「镜像里有动态数据」**：镜像构建后应视为不可变；运行时状态在可写层或卷上。
4. **「pull 的永远是 latest 最新」**：标签可被覆盖；要钉版本或 digest。
5. **「一个容器里塞一堆服务」**：能跑，但不便于扩缩、重启和排障；一进程一容器仍是主流默认。

## 小结

1. 镜像是只读分层模板，容器是镜像上的运行实例，仓库负责分发。
2. 分层换来复用与缓存；上层删除不等于下层文件从镜像里消失。
3. 容器是隔离进程，不是完整虚拟机；共享宿主机内核。
4. 可写层随容器消亡，持久数据必须落到卷或外部存储。
5. 生产钉版本或 digest，不要依赖可被覆盖的 `latest`。
